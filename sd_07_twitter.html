🎙 Phase 1 — Snowflake ID Generation
PHASE 1 · ID GENERATION
IR
INTERVIEWER
Let's start with something foundational — how do you generate tweet IDs? Why can't you just use auto-increment in MySQL?
ME
YOU
Three problems with auto-increment in MySQL at Twitter scale:
1. Single point of failure: One MySQL instance controls the counter. If it's down, you can't post tweets.
2. Not shardable: If you shard tweets across databases, each DB generates IDs independently — duplicates guaranteed.
3. Not time-sortable: Auto-increment IDs tell you nothing about when the tweet was created. You'd need a separate timestamp column to sort by recency.

Twitter's solution: Snowflake IDs — a 64-bit integer with the structure:
  • 1 bit: Sign bit (always 0)
  • 41 bits: Milliseconds since Twitter epoch (Nov 4, 2010). Gives ~69 years before overflow.
  • 10 bits: Machine/datacenter ID (up to 1,024 unique workers)
  • 12 bits: Sequence number per millisecond (up to 4,096 IDs per ms per machine)
Key properties:
  • Time-sortable: higher IDs = newer tweets. No need for separate timestamp sort.
  • Globally unique: no coordination needed between machines
  • 4,096 IDs × 1,024 machines = 4.19 million IDs per millisecond — vastly more than Twitter needs
  • Decentralized: each machine generates its own IDs
IR
INTERVIEWER
What happens if the server clock goes backwards (NTP adjustment)?
ME
YOU
Clock skew backward is the most dangerous Snowflake failure mode. If the timestamp goes backward, you'd generate IDs smaller than previously generated ones — breaking the monotonicity guarantee and potentially causing duplicates.

The standard solution: detect backward clock and refuse to generate IDs until the clock catches up.
if (currentMs < lastTimestamp) { long wait = lastTimestamp - currentMs; Thread.sleep(wait); // wait for clock to catch up }

In practice, NTP adjustments are typically <1ms. Waiting a few milliseconds is acceptable. For larger corrections (server restart after maintenance), the machine ID in the ID structure provides some insurance — same timestamp but different machine ID → no conflict.
💡 Snowflake is a frequently asked standalone interview question at companies like Airbnb, Uber, Twitter. Know the 41/10/12 bit breakdown and clock skew handling by heart. The time-sortability property is what makes it uniquely valuable for Twitter feeds.
📰 Phase 2 — Home Timeline: The Fanout Problem
PHASE 2 · TIMELINE
IR
INTERVIEWER
Design the home timeline. This is the core hard problem — walk me through it in depth.
ME
YOU
The home timeline shows you tweets from people you follow, in reverse chronological order. Two fundamental approaches:

Approach A — Fanout on Write (Push):
When @alice tweets, immediately write her tweet_id to the home timeline of every one of her followers.
Pro: Timeline reads are O(1) — just read from a pre-built sorted list in Redis.
Problem: @cristiano has 160M followers. His tweet creates 160M Redis writes in seconds. That's a write storm that can saturate the entire fanout worker fleet.

Approach B — Fanout on Read (Pull):
When a user requests their timeline, fetch their followee list, query each person's recent tweets, merge and sort.
Pro: No fanout writes needed.
Problem: If you follow 5,000 people, each timeline load requires 5,000 DB queries, then merge-sort of potentially millions of tweets. 300M × 10 opens/day = 3B timeline loads/day → at 5,000 queries each = 15 trillion DB queries/day. Impossible.

Twitter's actual solution: Hybrid
  • Normal users (<10K followers): Fanout on Write. Their tweets push to all followers' Redis timeline caches. Fast reads.
  • Celebrities (≥10K followers): Fanout on Read. Their tweets are NOT pushed to followers' caches. Instead, when a user requests their timeline, we fetch recent tweets from any celebrities they follow separately, then merge with the pre-built cache.
  • The merge at read time: timeline = precomputed_cache + celebrity_tweets, sorted by Snowflake ID (which is already time-sorted).
Storage: Redis Sorted Set per user: key = home_timeline:{user_id}, members = tweet_ids, score = tweet Snowflake ID (time-sortable). ZREVRANGE gives most recent tweets in O(log N).
IR
INTERVIEWER
10K followers is the celebrity threshold? How do you update that classification dynamically?
ME
YOU
Good question. The threshold is configurable and stored as a flag on the user record: is_celebrity = true. The User Service updates this flag when follower count crosses the threshold (either way — account can lose celebrity status if followers drop).

When an account gains celebrity status, we need to handle the transition carefully:
  • Stop pushing new tweets to followers' caches (prevent future write storms)
  • No need to retroactively remove already-pushed tweets from caches — they naturally expire (TTL-based)
  • Mark the account as celebrity in Redis: SADD celebrities {user_id}
  • Fanout workers check this set before deciding push vs skip
When an account loses celebrity status (rare), we can re-enable push for new tweets going forward. Existing followers' caches will backfill through normal activity.
📈 Phase 3 — Trending Topics
PHASE 3 · TRENDING
IR
INTERVIEWER
How do you compute trending topics in real-time?
ME
YOU
Trending is about detecting velocity — topics rising quickly in the last hour, not just topics with the highest absolute count.

Step 1 — Extract topics from tweets:
When a tweet is posted, a background service (Kafka consumer) parses the text, extracts:
- #hashtags
- Named entities (e.g., "Elon Musk", "World Cup") using NLP/regex

Step 2 — Count with sliding window (Redis):
For each extracted topic, use a sliding time bucket:
ZINCRBY trending:{region}:{5min_bucket} 1 "{hashtag}"

Bucket key rotates every 5 minutes. Keep last 12 buckets = 1 hour. TTL on old buckets for auto-cleanup.

Step 3 — Compute velocity (batch every 5 min):
A scheduled job reads the last 12 buckets, computes:
- Raw count of each topic in the last hour
- Velocity = count in last 5min / count in prev 5min (ratio)
- Score = count × velocity (boost rapidly rising topics)
- Write final ranking to trending_final:{region} Redis Sorted Set

Deduplication: "World Cup" and "#WorldCup" and "#worldcup" should count together. Lowercase + normalize before counting.

Spam prevention: Ignore users who have tweeted the same hashtag more than 5 times in 5 minutes (bot behavior). Their counts are discarded.
IR
INTERVIEWER
Trending topics at 500M tweets/day means 5,787 tweets/sec. Counting every hashtag in Redis at this rate — won't that kill Redis?
ME
YOU
Good concern. If every tweet goes directly to Redis ZINCRBY, we have 5,787 ops/sec to trending counters. Redis handles 1M ops/sec, so this is fine.

But the key insight: we don't need exact counts for trending. We can use probabilistic counting. Two options:
1. Count-Min Sketch: A probabilistic data structure that uses O(constant) memory and gives approximate counts with tunable error. Error rate <1% with just 1MB of memory — drastically cheaper than exact counting.
2. Approximate counting with sampling: Count only 10% of tweets (random sample), multiply by 10 for estimated total. Trending topics emerge with 1% accuracy — more than sufficient.

For a system design interview, mentioning Count-Min Sketch shows algorithmic depth beyond Redis ZINCRBY.
🔍 Phase 4 — Search & User Timeline
PHASE 4 · SEARCH + USER TIMELINE
IR
INTERVIEWER
How do you implement full-text tweet search?
ME
YOU
Tweet search is a real-time full-text search problem. The requirements are different from web search:
- Very high write rate (5,787 tweets/sec indexed)
- Freshness critical — new tweets should be searchable within seconds
- Query types: keyword, hashtag, @mention, from:user

Architecture: Elasticsearch
Tweets flow into Elasticsearch via Kafka (async, non-blocking for the post API). Each tweet is indexed with fields: content, user_id, created_at, hashtags[], mentioned_users[].

Index sharding: Shard by tweet creation time (time-based indices). Today's tweets in tweets-2026-03-22, yesterday's in tweets-2026-03-21, etc. Queries for recent tweets only touch the current index — much faster than scanning all time.

Ranking: By default, recent tweets first (Snowflake ID descending = time descending). For "Top" tab, factor in like_count + retweet_count in the relevance score.
IR
INTERVIEWER
What about user timeline — all tweets from a specific user?
ME
YOU
User timeline is simpler than home timeline — it's just all tweets by one person, sorted by Snowflake ID descending.

Storage: Cassandra table partitioned by user_id, clustered by tweet_id DESC:
user_timeline(user_id, tweet_id, created_at, PRIMARY KEY (user_id, tweet_id))

Read: SELECT tweet_id FROM user_timeline WHERE user_id = ? ORDER BY tweet_id DESC LIMIT 20

Since tweet IDs are time-sortable (Snowflake), sorting by tweet_id DESC gives reverse-chronological order without a separate timestamp column — clean and efficient.

Write: When a tweet is posted, write to two places atomically (or via Kafka):
1. tweets table (the actual tweet content)
2. user_timeline table (the index of tweet IDs per user)

The user_timeline table is effectively a sorted index — it contains only tweet_ids (pointers), not the full tweet content. This makes it memory-efficient and fast to paginate.
🔥 Phase 5 — Scale, Delete, and Edge Cases
PHASE 5 · EDGE CASES
IR
INTERVIEWER
A user deletes a tweet. How do you remove it from all 150 followers' cached timelines?
ME
YOU
Tweet deletion is handled lazily for cached timelines — not eagerly:
  1. Mark tweet as deleted = true in the tweets table (soft delete)
  2. Remove from author's user_timeline Cassandra table immediately
  3. Remove from author's own home timeline Redis cache
  4. Publish a tweet_deleted event to Kafka
  5. For follower caches: We do NOT immediately purge from all 150 followers' Redis sorted sets — that would require 150 Redis ZREM operations asynchronously, adding complexity for a rare operation
  6. Instead: when a user's timeline is fetched, we hydrate the tweet_ids from the cache by batch-fetching their content from the tweets table. Deleted tweets return null → filter them out at hydration time
This lazy deletion approach means a deleted tweet may appear in caches for a few minutes, but practically nobody will see it (it won't be in the visible viewport). Within one TTL cycle (24 hours), all cached references are gone.
IR
INTERVIEWER
What if a user's home timeline cache is cold — they haven't opened Twitter in a week?
ME
YOU
Redis home timeline caches have a TTL of 24 hours (inactive users' caches expire). When the user opens Twitter after a week, their cache is empty.

Cache miss flow:
  1. Timeline Service detects empty cache for user
  2. Fetches their followee list (Graph Service)
  3. Queries user_timeline table for each followee's recent tweets (batch Cassandra queries)
  4. For celebrity followees: fetches their recent tweets from celebrity timelines
  5. Merges all tweets sorted by Snowflake ID, takes top 800
  6. Writes to Redis: ZADD home_timeline:{user_id} {tweet_id} {tweet_id} with TTL reset
  7. Returns first page to user
This rebuild is expensive (~100ms+) but happens rarely — only when a user is truly inactive. Active users' caches are continuously refreshed by the fanout worker as new tweets arrive.

🔬 Snowflake ID — Structure Deep Dive

A 64-bit integer encoding time, machine identity, and sequence. No coordination needed between machines.

sign
1 bit
TIMESTAMP
41 bits = ms since epoch (~69 years)
DATACTR
5 bits = 32 DCs
WORKER
5 bits = 32 workers/DC
SEQUENCE
12 bits = 4096/ms/machine
Capacity: 4,096 IDs/ms × 1,024 machines = 4.19 billion IDs/sec (Twitter needs only ~17,000)
Time range: 41-bit ms timestamp → 69 years from Nov 4, 2010 → expires ~2079
Sort order: Larger Snowflake ID = newer tweet — no timestamp sort column needed

🗄 Database Schema
-- ── Cassandra: Tweets (primary content store) ────────
-- Partition by tweet_id → uniform distribution
-- tweet_id is Snowflake (time-sortable) so no separate created_at needed
CREATE TABLE tweets (
    tweet_id        BIGINT    PRIMARY KEY,  -- Snowflake ID
    user_id         BIGINT,
    content         TEXT,
    media_urls      LIST<TEXT>,
    hashtags        LIST<TEXT>,
    retweet_of_id   BIGINT,               -- NULL for original tweets
    reply_to_id     BIGINT,               -- NULL for non-replies
    like_count      COUNTER,
    retweet_count   COUNTER,
    is_deleted      BOOLEAN  DEFAULT FALSE
);

-- ── Cassandra: User Timeline (pointer table) ─────────
-- Partition by user_id → fast lookup "all tweets by user X"
-- Cluster by tweet_id DESC → newest first, sort is free (Snowflake = time order)
CREATE TABLE user_timeline (
    user_id     BIGINT,
    tweet_id    BIGINT,
    PRIMARY KEY (user_id, tweet_id)
) WITH CLUSTERING ORDER BY (tweet_id DESC);

-- ── Cassandra: Follow Graph (adjacency list) ─────────
CREATE TABLE followees (            -- "who does user X follow?"
    follower_id  BIGINT,
    followee_id  BIGINT,
    created_at   TIMESTAMP,
    PRIMARY KEY  (follower_id, followee_id)
);
CREATE TABLE followers (            -- "who follows user X?" (for fanout)
    followee_id  BIGINT,
    follower_id  BIGINT,
    created_at   TIMESTAMP,
    PRIMARY KEY  (followee_id, follower_id)
);

-- ── Redis: Home Timeline Cache ────────────────────────
-- Sorted Set per user. Score = tweet_id (Snowflake = time sortable!)
-- Key: home_timeline:{user_id}
-- ZADD home_timeline:{uid} {tweet_id} {tweet_id}  (score = member = tweet_id)
-- ZREVRANGE home_timeline:{uid} 0 19 → newest 20 tweet_ids
-- TTL: 24 hours (inactive users auto-evict)

-- ── Redis: Celebrity Registry ─────────────────────────
-- Key: celebrities → SET of user_ids with > 10K followers
-- Used by Fanout Workers to skip push for celebrity tweets
-- Used by Timeline Service to know which followees to pull

-- ── Redis: Trending ──────────────────────────────────
-- Key: trending:{region}:{5min_bucket} → Sorted Set: {hashtag: count}
-- Key: trending_final:{region} → Sorted Set: top 50 trending topics
-- TTL on bucket keys: 1 hour (12 buckets × 5 min)

// ── 1. Snowflake ID Generator ─────────────────────────
public class SnowflakeIdGenerator {

    // Bit allocations (total: 63 usable bits in a signed long)
    private static final int  TIMESTAMP_BITS   = 41;
    private static final int  DATACENTER_BITS  = 5;
    private static final int  WORKER_BITS      = 5;
    private static final int  SEQUENCE_BITS    = 12;

    private static final long MAX_SEQUENCE     = ~(-1L << SEQUENCE_BITS);  // 4095
    private static final long WORKER_SHIFT     = SEQUENCE_BITS;
    private static final long DC_SHIFT         = SEQUENCE_BITS + WORKER_BITS;
    private static final long TIMESTAMP_SHIFT  = DC_SHIFT + DATACENTER_BITS;

    // Twitter epoch: Nov 4, 2010, 01:42:54 UTC
    private static final long TWITTER_EPOCH    = 1288834974657L;

    private final long  datacenterId;
    private final long  workerId;
    private long        sequence      = 0L;
    private long        lastTimestamp = -1L;

    public SnowflakeIdGenerator(long datacenterId, long workerId) {
        this.datacenterId = datacenterId;
        this.workerId     = workerId;
    }

    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();

        // Clock went backwards — wait for it to catch up
        if (timestamp < lastTimestamp) {
            long wait = lastTimestamp - timestamp;
            if (wait <= 5) { // Tolerate up to 5ms clock skew
                timestamp = tilNextMillis(lastTimestamp);
            } else {
                throw new RuntimeException(
                    "Clock moved backwards by " + wait + "ms");
            }
        }

        if (timestamp == lastTimestamp) {
            // Same millisecond — use sequence number
            sequence = (sequence + 1) & MAX_SEQUENCE;
            if (sequence == 0) {
                // Sequence overflow — wait for next millisecond
                timestamp = tilNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L; // New millisecond — reset sequence
        }

        lastTimestamp = timestamp;

        return ((timestamp - TWITTER_EPOCH) << TIMESTAMP_SHIFT)
             | (datacenterId               << DC_SHIFT)
             | (workerId                   << WORKER_SHIFT)
             | sequence;
    }

    private long tilNextMillis(long lastTs) {
        long ts = System.currentTimeMillis();
        while (ts <= lastTs) ts = System.currentTimeMillis();
        return ts;
    }

    /** Extract creation time from Snowflake ID */
    public Instant getTimestamp(long snowflakeId) {
        long ts = (snowflakeId >> TIMESTAMP_SHIFT) + TWITTER_EPOCH;
        return Instant.ofEpochMilli(ts);
    }
}

// ── 2. Timeline Service (Hybrid Push + Pull) ──────────
@Service
public class TimelineService {

    @Autowired private RedisTemplate<String,Long>   redis;
    @Autowired private TweetRepository               tweetRepo;
    @Autowired private GraphRepository               graphRepo;

    private static final int  MAX_TIMELINE_SIZE = 800;
    private static final int  TIMELINE_TTL_SECS = 86400; // 24h

    public List<Tweet> getHomeTimeline(long userId,
                                           Long  maxId,
                                           int   count) {
        String cacheKey = "home_timeline:" + userId;
        double maxScore  = maxId != null
            ? (double) (maxId - 1) : Double.MAX_VALUE;

        // Step 1: Get pushed tweet IDs from Redis ZSet
        Set<Long> pushedIds = redis.opsForZSet()
            .reverseRangeByScore(cacheKey, 0, maxScore, 0, count * 2);

        // Step 2: Check if cache is cold → rebuild
        if (pushedIds == null || pushedIds.isEmpty()) {
            rebuildFromCassandra(userId);
            pushedIds = redis.opsForZSet()
                .reverseRangeByScore(cacheKey, 0, maxScore, 0, count * 2);
        }

        // Step 3: Get celebrity followees → pull their tweets
        Set<Long> celebFollowees = graphRepo.getCelebFollowees(userId);
        List<Long> pulledIds = new ArrayList<>();

        for (long celebId : celebFollowees) {
            // Each celeb's user_timeline is also a Redis ZSet
            Set<Long> celebTweets = redis.opsForZSet()
                .reverseRangeByScore("user_timeline:" + celebId,
                                     0, maxScore, 0, 10);
            if (celebTweets != null) pulledIds.addAll(celebTweets);
        }

        // Step 4: Merge and sort by Snowflake ID descending (= time order)
        List<Long> allIds = new ArrayList<>(pushedIds);
        allIds.addAll(pulledIds);
        allIds.sort(Collections.reverseOrder()); // Snowflake = time sort!
        List<Long> pageIds = allIds.stream()
            .distinct().limit(count).collect(Collectors.toList());

        // Step 5: Batch fetch tweet content from Cassandra
        List<Tweet> tweets = tweetRepo.findAllById(pageIds);

        // Step 6: Filter deleted tweets (lazy delete)
        return tweets.stream()
            .filter(t -> !t.isDeleted())
            .sorted((Comparator.comparingLong(Tweet::getTweetId).reversed()))
            .collect(Collectors.toList());
    }

    private void rebuildFromCassandra(long userId) {
        String cacheKey = "home_timeline:" + userId;
        List<Long> followees = graphRepo.getFollowees(userId);

        // Fetch recent tweets from each followee
        List<Long> tweetIds = followees.stream()
            .filter(id -> !graphRepo.isCelebrity(id)) // skip celebrities
            .flatMap(fid -> tweetRepo
                .getUserTimelineIds(fid, 20).stream())
            .sorted(Collections.reverseOrder())
            .limit(MAX_TIMELINE_SIZE)
            .collect(Collectors.toList());

        // Write to Redis ZSet (tweet_id is both score and member)
        tweetIds.forEach(id ->
            redis.opsForZSet().add(cacheKey, id, (double) id));
        redis.opsForZSet().removeRange(cacheKey, 0, -(MAX_TIMELINE_SIZE + 1));
        redis.expire(cacheKey, Duration.ofSeconds(TIMELINE_TTL_SECS));
    }
}

// ── 3. Fanout Worker ──────────────────────────────────
@Component
public class FanoutWorker {

    @Autowired private RedisTemplate<String,Long> redis;
    @Autowired private GraphRepository              graphRepo;
    private static final int MAX_TIMELINE = 800;

    @KafkaListener(topics = "tweets", groupId = "fanout-workers")
    public void onTweet(TweetEvent event) {
        long tweetId  = event.getTweetId();
        long authorId = event.getAuthorId();

        // Skip fanout for celebrities — readers pull their tweets
        if (graphRepo.isCelebrity(authorId)) {
            // Still add to author's own user_timeline Redis cache
            redis.opsForZSet().add("user_timeline:" + authorId,
                tweetId, (double) tweetId);
            return;
        }

        // Get all followers (could be 0–9,999 for non-celebrities)
        List<Long> followers = graphRepo.getFollowers(authorId);

        // Fan out in batches with Redis pipeline (reduces round trips)
        Lists.partition(followers, 1000).forEach(batch ->
            redis.executePipelined(conn -> {
                batch.forEach(followerId -> {
                    String key = "home_timeline:" + followerId;
                    conn.zAdd(key.getBytes(),
                              (double) tweetId,
                              longToBytes(tweetId));
                    conn.zRemRange(key.getBytes(),
                                   0, -(MAX_TIMELINE + 1));
                });
                return null;
            }));

        // Also update author's user_timeline
        redis.opsForZSet().add("user_timeline:" + authorId,
            tweetId, (double) tweetId);
    }
}

import time, threading, logging
from typing import List, Optional

logger = logging.getLogger(__name__)


# ── 1. Snowflake ID Generator ─────────────────────────
class SnowflakeIdGenerator:
    """
    Generates globally unique, time-sortable 64-bit IDs.
    No coordination needed between instances.
    Twitter epoch: Nov 4, 2010, 01:42:54 UTC
    """
    TWITTER_EPOCH   = 1288834974657  # ms
    TIMESTAMP_BITS  = 41
    DC_BITS         = 5
    WORKER_BITS     = 5
    SEQUENCE_BITS   = 12

    MAX_SEQUENCE    = (1 << SEQUENCE_BITS) - 1   # 4095
    WORKER_SHIFT    = SEQUENCE_BITS
    DC_SHIFT        = SEQUENCE_BITS + WORKER_BITS
    TIMESTAMP_SHIFT = SEQUENCE_BITS + WORKER_BITS + DC_BITS

    def __init__(self, datacenter_id: int, worker_id: int):
        self.datacenter_id  = datacenter_id
        self.worker_id      = worker_id
        self.sequence       = 0
        self.last_timestamp = -1
        self._lock          = threading.Lock()

    def next_id(self) -> int:
        """Thread-safe Snowflake ID generation"""
        with self._lock:
            ts = self._current_ms()

            if ts < self.last_timestamp:
                wait_ms = self.last_timestamp - ts
                if wait_ms <= 5:
                    ts = self._wait_next_ms(self.last_timestamp)
                else:
                    raise RuntimeError(
                        f"Clock moved backwards by {wait_ms}ms")

            if ts == self.last_timestamp:
                self.sequence = (self.sequence + 1) & self.MAX_SEQUENCE
                if self.sequence == 0:
                    ts = self._wait_next_ms(self.last_timestamp)
            else:
                self.sequence = 0

            self.last_timestamp = ts

            return (
                ((ts - self.TWITTER_EPOCH) << self.TIMESTAMP_SHIFT)
                | (self.datacenter_id << self.DC_SHIFT)
                | (self.worker_id     << self.WORKER_SHIFT)
                | self.sequence
            )

    def get_timestamp_ms(self, snowflake_id: int) -> int:
        """Extract creation time (ms) from a Snowflake ID"""
        return (snowflake_id >> self.TIMESTAMP_SHIFT) + self.TWITTER_EPOCH

    def _current_ms(self) -> int:
        return int(time.time() * 1000)

    def _wait_next_ms(self, last_ts: int) -> int:
        ts = self._current_ms()
        while ts <= last_ts:
            ts = self._current_ms()
        return ts


# ── 2. Timeline Service ───────────────────────────────
class TimelineService:
    """
    Hybrid push (pre-computed Redis ZSet) + pull (celebrity tweets at read time).
    Snowflake IDs act as both member and score in ZSet — sort by ID = sort by time.
    """
    MAX_TIMELINE_SIZE = 800
    TIMELINE_TTL      = 86400  # 24 hours

    def __init__(self, redis_client, tweet_repo, graph_repo):
        self.redis  = redis_client
        self.tweets = tweet_repo
        self.graph  = graph_repo

    def get_home_timeline(self, user_id: int,
                            max_id: Optional[int] = None,
                            count: int = 20) -> List[dict]:
        cache_key = f"home_timeline:{user_id}"
        max_score = (max_id - 1) if max_id else float('+inf')

        # Step 1: Get pushed tweet IDs from Redis ZSet
        pushed_ids = self.redis.zrevrangebyscore(
            cache_key, max_score, 0,
            start=0, num=count * 2)

        # Step 2: Cold cache → rebuild from Cassandra
        if not pushed_ids:
            self._rebuild_from_cassandra(user_id)
            pushed_ids = self.redis.zrevrangebyscore(
                cache_key, max_score, 0, start=0, num=count * 2)

        pushed_ids = [int(x) for x in (pushed_ids or [])]

        # Step 3: Pull celebrity tweets at read time
        celeb_followees = self.graph.get_celeb_followees(user_id)
        pulled_ids = []
        for celeb_id in celeb_followees:
            celeb_tweets = self.redis.zrevrangebyscore(
                f"user_timeline:{celeb_id}",
                max_score, 0, start=0, num=10)
            pulled_ids.extend([int(x) for x in (celeb_tweets or [])])

        # Step 4: Merge + sort by Snowflake ID desc (= reverse-chronological)
        all_ids = sorted(set(pushed_ids + pulled_ids), reverse=True)[:count]

        # Step 5: Batch fetch tweet content
        tweets = self.tweets.get_many(all_ids)

        # Step 6: Filter deleted (lazy delete pattern)
        return [t for t in tweets if t and not t.get('is_deleted')]

    def _rebuild_from_cassandra(self, user_id: int):
        """Called when a user's cache is cold (inactive > 24h)"""
        followees  = self.graph.get_followees(user_id)
        tweet_ids  = []

        for fid in followees:
            if not self.graph.is_celebrity(fid):
                ids = self.tweets.get_user_timeline_ids(fid, limit=20)
                tweet_ids.extend(ids)

        tweet_ids = sorted(set(tweet_ids), reverse=True)[:self.MAX_TIMELINE_SIZE]

        # Write to Redis ZSet: score=tweet_id (Snowflake = time sort!)
        if tweet_ids:
            cache_key = f"home_timeline:{user_id}"
            mapping   = {str(tid): float(tid) for tid in tweet_ids}
            pipe = self.redis.pipeline()
            pipe.zadd(cache_key, mapping)
            pipe.zremrangebyrank(cache_key, 0, -(self.MAX_TIMELINE_SIZE + 1))
            pipe.expire(cache_key, self.TIMELINE_TTL)
            pipe.execute()


# ── 3. Fanout Worker ──────────────────────────────────
class FanoutWorker:
    """
    Kafka consumer: for each new tweet, push tweet_id to followers' Redis ZSets.
    Skip celebrities — their followers pull tweets at read time.
    """
    MAX_TIMELINE = 800

    def __init__(self, redis_client, graph_repo):
        self.redis = redis_client
        self.graph = graph_repo

    def on_tweet(self, tweet_id: int, author_id: int):
        """Called by Kafka consumer for each new tweet event"""
        # Always update the author's own user_timeline
        self.redis.zadd(f"user_timeline:{author_id}",
                        {str(tweet_id): float(tweet_id)})

        # Skip push fanout for celebrities
        if self.graph.is_celebrity(author_id):
            return

        followers = self.graph.get_followers(author_id)

        # Fan out in batches of 1000 using Redis pipeline
        batch_size = 1000
        for i in range(0, len(followers), batch_size):
            batch = followers[i:i + batch_size]
            pipe  = self.redis.pipeline()
            for follower_id in batch:
                key = f"home_timeline:{follower_id}"
                pipe.zadd(key, {str(tweet_id): float(tweet_id)})
                pipe.zremrangebyrank(key, 0, -(self.MAX_TIMELINE + 1))
            pipe.execute()
        logger.info(
            f"Fanned out tweet {tweet_id} to {len(followers)} followers")


# ── 4. Trending Service ───────────────────────────────
class TrendingService:
    """
    Counts hashtag occurrences in sliding 5-minute buckets.
    Computes velocity-weighted trending score every 5 minutes.
    """
    BUCKET_SIZE_SEC = 300   # 5 minutes
    WINDOW_BUCKETS  = 12    # 1 hour
    BUCKET_TTL      = 3600  # 1 hour

    def __init__(self, redis_client):
        self.redis = redis_client

    def record_tweet(self, hashtags: List[str], region: str = "global"):
        """Called for each tweet. Increments hashtag counters in current bucket."""
        now_bucket = int(time.time()) // self.BUCKET_SIZE_SEC
        key = f"trending:{region}:{now_bucket}"

        pipe = self.redis.pipeline()
        for tag in hashtags:
            normalized = tag.lower().lstrip('#')
            pipe.zincrby(key, 1, normalized)
        pipe.expire(key, self.BUCKET_TTL)
        pipe.execute()

    def compute_trending(self, region: str = "global") -> List[dict]:
        """Run every 5 min: aggregate counts + velocity score"""
        now_bucket = int(time.time()) // self.BUCKET_SIZE_SEC
        counts = {}

        # Aggregate counts from the last WINDOW_BUCKETS buckets
        for i in range(self.WINDOW_BUCKETS):
            key = f"trending:{region}:{now_bucket - i}"
            bucket_data = self.redis.zrevrangebyscore(
                key, '+inf', 0, withscores=True, start=0, num=1000)
            recency_weight = 1.0 - (i / self.WINDOW_BUCKETS)
            for tag, count in (bucket_data or []):
                tag = tag.decode() if isinstance(tag, bytes) else tag
                counts[tag] = counts.get(tag, 0) + count * recency_weight

        # Write final ranking to Redis (for serving API)
        final_key = f"trending_final:{region}"
        if counts:
            self.redis.delete(final_key)
            self.redis.zadd(final_key, counts)
            self.redis.expire(final_key, 600)  # 10 min TTL

        # Return top 10 trending topics
        top = self.redis.zrevrangebyscore(
            final_key, '+inf', 0, withscores=True, start=0, num=10)
        return [
            {"name": f"#{t.decode()}", "score": round(s)}
            for t, s in (top or [])
        ]
⚖️ Design Tradeoffs & Decisions
Twitter's defining tradeoff is optimizing for read (timeline) over write (tweet). Every design decision flows from this — pre-compute at write time, serve from cache at read time.
🆔 ID Generation: Snowflake vs UUID vs Auto-Increment
✅ Snowflake

Time-sortable (no separate timestamp sort), globally unique without coordination, decentralized generation, 64-bit (smaller than UUID). Enables sort-by-ID = sort-by-time — a huge simplification for timeline queries.

❌ UUID v4

Globally unique without coordination. But: random — not time-sortable. Can't use as timeline sort key. 128-bit (2× storage vs Snowflake). Index fragmentation in B-trees due to random insertion order.

❌ Auto-Increment

Simple but: single point of failure (one DB owns the counter), not shardable (each shard generates duplicates), no time-sortability benefit. Cannot scale to Twitter's write rate.

📰 Timeline Generation: Push vs Pull vs Hybrid
✅ Hybrid (push normal, pull celebrities)

Best of both worlds. Normal users get O(1) timeline reads (pre-computed). Celebrity write storms (100M followers) avoided. Merge at read time is cheap — only a handful of celebrity followees per user.

❌ Pure Push

Works for normal users. Breaks catastrophically for celebrities: @Cristiano with 160M followers → 160M Redis writes per tweet. At peak, this saturates the entire fanout fleet for minutes. Timeline may lag hours.

❌ Pure Pull

No fanout writes. But: generating timeline requires fetching from thousands of followees and merge-sorting. 300M × 10 opens/day × 5,000 followees = 15 trillion DB queries/day. Impossible at scale.

💾 Tweet Storage: Cassandra vs MySQL vs DynamoDB
✅ Cassandra

Built for massive time-series append-only writes. Tweets partition naturally by user_id. Snowflake IDs cluster perfectly by time. No complex joins needed. Write throughput: millions/sec. Twitter and Discord use it.

⚠️ MySQL (sharded)

Works at smaller scale. ACID, familiar tooling, complex queries possible. But: manual sharding needed above ~100M rows/table. JOINs across shards are impossible. Snowflake IDs don't help B-tree index locality at this write rate.

💡 Why NOT store full tweets in Redis

Tweets are immutable after posting — not frequently updated. Redis is RAM-based, expensive at 273TB/year scale. Use Cassandra for persistent storage, Redis only for the timeline index (tweet_ids, not content). Never store full tweet content in Redis.

📈 Trending: Redis ZSet vs Kafka Streams vs Count-Min Sketch
✅ Redis ZSet (sliding buckets)

Simple, fast, accurate enough for trending. ZINCRBY is atomic. Bucket rotation handles sliding window automatically. No stream processing infrastructure needed. Refresh every 5 minutes is fine for trending — users don't notice 5-min lag.

⚠️ Kafka Streams / Flink

More accurate real-time stream processing. Can compute exact counts with sub-second latency. But: significant operational complexity (Kafka Streams job, state stores, checkpointing). Overkill if 5-minute refresh is acceptable.

💡 Count-Min Sketch

Probabilistic data structure for approximate frequency counting. O(1) update, O(1) query, fixed memory (~1MB for 1% error). Better than Redis ZSet for very high cardinality (millions of unique hashtags). Mention it to show algorithmic depth beyond naive ZINCRBY.

🗑 Delete Tweet: Eager vs Lazy Removal from Caches
✅ Lazy deletion (filter at hydration)

Soft-delete the tweet (is_deleted=true). Don't purge from follower caches. When timeline is served, batch-fetch tweet content — null/deleted tweets are filtered. No O(followers) fan-out needed for deletes.

❌ Eager cache purge

For each delete, find all followers, ZREM tweet_id from each Redis ZSet. For a user with 9,999 followers, that's 9,999 Redis ZREM operations asynchronously. For celebrities who didn't push, not needed anyway. Complex, slow, rarely important.

💡 Exception: sensitive content

For legal takedowns or content violations (CSAM, court orders), eager purge from all caches is required — can't wait for lazy expiry. These are handled by a dedicated trust & safety pipeline with different SLAs, not the normal delete API.
