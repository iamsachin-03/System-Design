Rate Limiter
Design a distributed rate limiting system — protecting APIs from abuse, DDoS, and quota overuse. Covers all four major algorithms (Token Bucket, Leaky Bucket, Fixed Window, Sliding Window) with distributed implementation using Redis.
🔴 MUST DO 🟢 Medium Token Bucket Sliding Window Redis Lua Script Middleware
🎯 Clarify Scope
Rate limiter is asked differently from other designs. The interviewer usually wants you to: (1) know all the algorithms deeply, (2) implement them correctly in a distributed system, (3) handle edge cases like race conditions and multi-datacenter.

✅ Functional Requirements
CORE
Allow/Deny Decision: For each incoming request, decide ALLOW or DENY in <5ms. Return decision to the caller.
Multiple Limit Dimensions: Limit by user_id, IP, API key, and/or endpoint. Apply the most restrictive matching rule.
Configurable Rules: Rules specify: identifier (who), limit (how many), window (per minute/hour/day), and algorithm.
HTTP 429 Response: Rejected requests get 429 Too Many Requests with Retry-After, X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset headers.
Distributed Accuracy: Correct across multiple app servers — no per-server counters that under-count.
EXTENDED
Rule management API: Create/update/delete rate limit rules without restarting services
Analytics: Dashboard showing blocked requests per user/endpoint, top offenders
Graceful degradation: If Redis is down, fail open (allow all requests) with alerting — don't block legitimate traffic
Burst allowance: Allow short bursts above the sustained rate (Token Bucket handles this naturally)
OUT OF SCOPE
Authentication/authorization · Request queuing (throttle) · Content-based limiting · Geographic routing
⚡ Non-Functional Requirements
PERFORMANCE
Rate check latency: <5ms p99 (must not become the bottleneck)
Throughput: 10M checks/sec (distributed across Redis cluster)
Overhead on normal requests: <1ms added latency
RELIABILITY
Fail open: if Redis unavailable, allow all requests (don't block legitimate traffic)
Accuracy: allow up to ±1% counting error (slightly under-count is acceptable)
No race conditions: counter operations must be atomic
SCALABILITY
Horizontally scalable — add Redis nodes to increase throughput
Works across multiple data centers
Rules stored in config (Redis/DB) — not in code
CORRECTNESS
Atomic check-and-increment: no TOCTOU (Time of Check to Time of Use) race
TTL auto-expire: windows reset automatically, no orphan counters
Idempotent: retry-safe (no double-counting on network retry)
🎙 Phase 1 — Problem Space & Why It's Hard
PHASE 1 · PROBLEM
IR
INTERVIEWER
Design a rate limiter. Start by telling me what the core challenges are.
ME
YOU
A rate limiter sounds simple — count requests, reject if over limit. But at scale, three things make it hard:

1. Distributed correctness: If you have 100 app servers each with a local counter, and the limit is 100 req/min, a user can send 100 requests to each server = 10,000 actual requests with no rejections. Counters must be shared state.

2. Atomicity: Without atomic operations, two requests arriving simultaneously can both read count=99 (under limit), both increment to 100, and both get allowed — resulting in 101 requests. You need atomic check-and-increment.

3. Algorithm choice affects UX: Different algorithms produce very different traffic patterns. A fixed window counter can allow 2× the limit at window boundaries. Token bucket allows bursts. Sliding window is most accurate but most expensive. Choosing the right algorithm matters.

I'll design around Redis for shared state (atomic operations, sub-millisecond latency), with the Token Bucket algorithm as the primary choice, plus Sliding Window Log for endpoints needing precision.
💡 Naming these three challenges upfront — distributed state, atomicity, algorithm — immediately signals you understand this problem at depth. Most candidates jump straight to "store a counter in Redis" without addressing why naïve approaches fail.
🧮 Phase 2 — Algorithm Deep Dive
PHASE 2 · ALGORITHMS
IR
INTERVIEWER
Walk me through each algorithm. What are the tradeoffs?
ME
YOU
Let me walk through all four algorithms — this is the heart of this design:

1. Fixed Window Counter
Split time into fixed windows (e.g., minute 0:00–1:00, 1:00–2:00, ...). Count requests per window per user. Reject when count exceeds limit.
Pros: Simple, O(1) memory, easy to implement.
Fatal flaw: A user can send 100 requests at 0:59 and 100 more at 1:01 — gets 200 requests in 2 seconds while limit is "100 per minute". The window boundary lets through 2× the limit as a burst.

2. Sliding Window Log
Keep a timestamped log of every request. On each new request, remove entries older than 1 window, count remaining, reject if over limit.
Pros: Perfectly accurate — no boundary burst issue.
Fatal flaw: O(N) memory where N = limit × window. At 1000 req/min limit with 1M users, you need to store 1 billion timestamps. Memory prohibitive at scale.

3. Sliding Window Counter (Hybrid)
Track two adjacent fixed windows (current + previous). Interpolate based on how far into the current window we are:
current_count = prev_window_count × (1 - elapsed/window_size) + curr_window_count
Pros: O(1) memory. Eliminates the boundary burst problem (98% accurate).
Cons: Assumes uniform distribution of requests in the previous window — slight inaccuracy.

4. Token Bucket
Each user has a "bucket" with capacity N tokens. Tokens refill at rate R/sec. Each request consumes one token. If bucket is empty, reject.
Pros: Allows short bursts (bucket absorbs them). Smooth sustained rate. Most natural model for APIs.
Cons: Harder to implement correctly (fractional tokens, floating point timing).

5. Leaky Bucket
Requests go into a queue. Queue drains at a fixed rate (one request processed per N ms). If queue is full, reject.
Pros: Perfectly smooth output rate — good for downstream systems that can't handle bursts.
Cons: Queue adds latency. Bursts queue up rather than being served immediately. Doesn't work well for stateless HTTP.
IR
INTERVIEWER
Which would you recommend and why?
ME
YOU
For a general-purpose API rate limiter, I'd recommend Token Bucket as the primary algorithm, with Sliding Window Counter as the backup for simpler use cases.

Token Bucket wins because:
  • It handles real-world traffic naturally — mobile apps that burst on app open are not "abusers", they're just users
  • It has a clear mental model users understand: "I have a quota, it refills over time"
  • Burst allowance is controlled (bucket capacity) without unfairly penalizing users
  • Twitter, Stripe, GitHub all use token bucket for their public APIs
For per-second precision requirements (financial APIs, voting systems), I'd use Sliding Window Counter which gives 98%+ accuracy at O(1) memory cost.
⚙️ Phase 3 — Distributed Implementation
PHASE 3 · IMPLEMENTATION
IR
INTERVIEWER
How do you implement Token Bucket in a distributed system without race conditions?
ME
YOU
The classic Token Bucket implementation requires two pieces of state: tokens_remaining and last_refill_time. The problem: this is a read-modify-write pattern. In Redis, if two requests happen simultaneously:

1. Both read tokens=50
2. Both compute: tokens_after = max(50 - 1 + refill, capacity) = 50
3. Both write tokens=49
4. One request effectively didn't count → race condition

Solution: Redis Lua Script
Lua scripts in Redis run atomically — no other Redis commands execute while the Lua script runs. This gives us atomic read-modify-write:

EVAL lua_script 1 {key} {capacity} {refill_rate} {now_ms}

The Lua script:
  1. HGET tokens and last_refill from Redis hash
  2. Compute elapsed_ms = now - last_refill
  3. Compute new_tokens = min(capacity, stored_tokens + elapsed_ms × refill_rate)
  4. If new_tokens ≥ 1: consume (new_tokens - 1), return ALLOW
  5. Else: return DENY with time until next token
  6. HMSET updated state back to Redis with TTL
The entire sequence is atomic — no race condition possible. And since Redis is single-threaded for command execution, the Lua script blocks no one else for more than ~1ms.
IR
INTERVIEWER
What about Sliding Window Counter in Redis?
ME
YOU
Sliding Window Counter is even simpler in Redis:

We maintain two keys — current window count and previous window count. Each with a TTL aligned to the window boundary.

curr_key = "rl:{user_id}:{endpoint}:{current_minute}"
prev_key = "rl:{user_id}:{endpoint}:{previous_minute}"

On each request:
  1. INCR current window counter (atomic, sets to 1 if key doesn't exist)
  2. EXPIRE current key to 2 × window_size (so it exists for one full extra window)
  3. GET previous window counter
  4. Compute: weighted_count = prev_count × (1 - seconds_elapsed/60) + curr_count
  5. If weighted_count > limit: return DENY (rollback current count with DECR)
  6. Else: ALLOW
We can do steps 1-3 in a Redis pipeline (one round trip). The DECR on reject adds a second round trip in the rejection path — but rejections are rare in well-configured systems.
IR
INTERVIEWER
What if Redis goes down? Should you block all traffic?
ME
YOU
Never block traffic when the rate limiter fails. The rate limiter is a protection mechanism, not the primary system. If it goes down, legitimate users suffer — that's worse than letting a few bad actors through temporarily.

Two options:
1. Fail Open: If Redis is unavailable, allow all requests. Log/alert immediately. This is the standard approach — availability > rate limiting precision.
2. In-process fallback: Keep a local in-memory counter as fallback (per app server, sliding window). Less accurate (under-counts globally) but better than nothing. Activate automatically on Redis failure.

I'd use Fail Open + alert as primary, with in-process fallback as secondary. The monitoring alert should trigger immediate Redis incident response.
🌐 Phase 4 — Multi-Datacenter & Advanced
PHASE 4 · DISTRIBUTED
IR
INTERVIEWER
How do you handle rate limiting across multiple data centers? A user might hit US-West, US-East, and EU simultaneously.
ME
YOU
This is the hardest part of distributed rate limiting. Three approaches:

Option 1 — Global Redis (single source of truth):
All datacenters write to the same Redis cluster (with cross-region replication). Pros: perfectly accurate. Cons: every rate check has a cross-region round trip (~50–100ms latency). Completely unacceptable for a 5ms requirement.

Option 2 — Local Redis per region + async sync:
Each region has its own Redis. Counters increment locally (~0.5ms). A background job syncs aggregated counts across regions every 1–5 seconds. Pros: fast. Cons: slightly over-allows (a user could use the full limit in 3 regions for ~5 seconds).

Option 3 — Rate limit per region + global allowance split:
Global limit = 100/min. Split into per-region sub-limits based on traffic distribution. US-West: 40/min, US-East: 40/min, EU: 20/min. No cross-region state needed. Pros: simplest, fastest. Cons: if user traffic shifts unexpectedly, one region gets exhausted while others have headroom.

My recommendation: Option 2 (local Redis + async sync) for most cases. The brief over-allowing is acceptable for typical rate limiting (protecting against abuse, not billing). For billing-sensitive limits, use a global Redis with async fallback to local on timeout.
IR
INTERVIEWER
How do you handle rate limiting at different granularities — per minute AND per day?
ME
YOU
Multi-window rate limiting: check all applicable rules and reject if any limit is exceeded.

For a rule: "100 req/min AND 10,000 req/day":
We maintain two independent counters:
rl:{user_id}:minute:{current_minute} → INCR, EX 120
rl:{user_id}:day:{current_day} → INCR, EX 172800

Both can be checked in a single Redis pipeline (2 GETs + 2 INCRs in one round trip). If either exceeds its limit, reject and DECR both. Return the most restrictive limit's headers in the response.

The Retry-After header tells the client which window is the binding constraint: if the minute limit is hit, Retry-After = seconds until next minute. If the day limit is hit, Retry-After = seconds until next day.
🔧 Phase 5 — Architecture & Where to Place It
PHASE 5 · PLACEMENT
IR
INTERVIEWER
Where in the request lifecycle do you place the rate limiter?
ME
YOU
Rate limiter can be placed at multiple layers — each with tradeoffs:

1. API Gateway (recommended primary): Before the request reaches any application code. Rejects at the edge, saves application server CPU. Clients get 429 without the request ever hitting your microservices. Centralized rule management.

2. Middleware in each service: Implemented as HTTP middleware (Express, Spring, Django). Runs inside each service. Allows per-service limits. Can access request context (authenticated user, etc.) that the gateway doesn't have. Downside: every service must implement it.

3. SDK/Library: Rate limiting logic embedded in a shared library imported by each service. Consistent behavior, can cache rules locally. But requires all services to use it — hard to enforce across polyglot systems.

Best practice: layered approach.
  • API Gateway: coarse-grained limits (per IP, per API key) — stops DDoS before it hits services
  • Service middleware: fine-grained limits (per user per endpoint) — stops quota abuse
  • Both layers share the same Redis cluster and rule store — consistent policies

📊 Capacity Estimation
The rate limiter must be faster than the services it protects. A 5ms p99 limit means Redis operations must complete in <2ms — well within Redis's typical 0.1–1ms range.
1
Request Rate — What We Need to Check
How many rate limit checks per second?
Given: 10M requests/sec across all services (assumption) Rate limit checks = requests (1:1 ratio, every request checked): Average = 10,000,000 checks/sec Peak (3×) = 30,000,000 checks/sec Redis throughput (pipeline, 1M simple ops/sec per node): For sliding window (2 INCR + 2 GET = 4 ops per check): 10M checks × 4 ops = 40M Redis ops/sec Redis nodes needed = 40M ÷ 1M = 40 Redis nodes at sustained load For token bucket (1 Lua EVAL per check, ~3× heavier than simple ops): 10M × 3 = 30M op-equivalents/sec Redis nodes needed = ~30 nodes With Redis Cluster (key-space sharded by user_id): Each user's counter on same shard → no cross-shard ops → 30-40 Redis nodes for 10M checks/sec. Very manageable.
2
Memory — Counter Storage
How much Redis RAM do we need?
Assumptions: 100M unique users, 10 endpoints rate-limited per user, window = 1 min Sliding Window Counter (2 keys per user per endpoint): Key size: "rl:{user_id}:{endpoint}:{minute}" ≈ 40 bytes Value: integer counter ≈ 8 bytes Redis overhead per key ≈ 100 bytes total Keys = 100M users × 10 endpoints × 2 windows = 2B keys But TTL = 2min → at any time only ~2min worth of active keys Active keys at any time ≈ 100M × 10 × 2 = 2B keys Total memory = 2B × 100B = 200 GB With 32GB Redis nodes: 200GB ÷ 32GB = ~7 nodes for memory (throughput requirement of 40 nodes >> memory requirement) Token Bucket (1 hash per user per endpoint): Hash: 2 fields (tokens + last_refill) ≈ 150 bytes 100M users × 10 endpoints × 150B = 150 GB → Throughput dominates. Need 30-40 Redis nodes for 10M req/sec.
3
Latency Budget
How do we stay under 5ms p99?
Total budget: 5ms p99 for the rate limit check Breakdown: Rule lookup (local cache in app server) = ~0.01ms Key computation (hash user_id + endpoint) = ~0.01ms Redis network round-trip (same DC) = ~0.5ms Redis command execution (Lua or pipeline) = ~0.1ms Result processing + header generation = ~0.1ms ──────── Total p50 ≈ 0.7ms Total p99 (Redis tail + GC jitter) ≈ 2-3ms → Well within 5ms p99 budget. The rate limiter adds ~1ms to p50 latency. For ultra-low latency: add L1 in-process cache for rate limit decisions (cache DENY decisions for 100ms — a denied user stays denied)

💻 Implementation — All Four Algorithms
// ══════════════════════════════════════════════════════
// Rate Limiter — All Algorithms (Java)
// ══════════════════════════════════════════════════════

// ── 1. Token Bucket (Lua atomic, recommended) ─────────
@Component
public class TokenBucketLimiter implements RateLimiter {

    @Autowired private RedisTemplate<String,String> redis;

    // Lua script: atomically refill tokens and consume one
    private static final String TOKEN_BUCKET_LUA = """
        local key = KEYS[1]
        local capacity = tonumber(ARGV[1])
        local refill_rate = tonumber(ARGV[2])  -- tokens per second
        local now_ms = tonumber(ARGV[3])
        local ttl = tonumber(ARGV[4])

        -- Get current state
        local stored = redis.call('HGET', key, 'tokens')
        local last_ms = redis.call('HGET', key, 'last_refill_ms')

        local tokens = capacity  -- default: full bucket for new user
        local last_refill = now_ms

        if stored and last_ms then
            tokens = tonumber(stored)
            last_refill = tonumber(last_ms)
        end

        -- Refill based on elapsed time
        local elapsed_sec = (now_ms - last_refill) / 1000.0
        local new_tokens = math.min(capacity, tokens + elapsed_sec * refill_rate)

        -- Try to consume one token
        if new_tokens >= 1 then
            new_tokens = new_tokens - 1
            redis.call('HSET', key, 'tokens', new_tokens, 'last_refill_ms', now_ms)
            redis.call('EXPIRE', key, ttl)
            -- Return [allowed=1, remaining=floor(tokens), retry_after=0]
            return {1, math.floor(new_tokens), 0}
        else
            -- Not enough tokens — compute wait time
            local wait_sec = math.ceil((1 - new_tokens) / refill_rate)
            redis.call('HSET', key, 'tokens', new_tokens, 'last_refill_ms', now_ms)
            redis.call('EXPIRE', key, ttl)
            return {0, 0, wait_sec}
        end
        """;

    public RateResult check(String key, RateRule rule) {
        long capacity    = rule.getBurstCapacity();
        double refillRate = (double) rule.getMaxRequests() / rule.getWindowSeconds();
        long nowMs       = System.currentTimeMillis();
        long ttl         = rule.getWindowSeconds() * 2;

        List<Long> result = (List<Long>) redis.execute(
            new DefaultRedisScript<>(TOKEN_BUCKET_LUA, List.class),
            Collections.singletonList(key),
            String.valueOf(capacity),
            String.valueOf(refillRate),
            String.valueOf(nowMs),
            String.valueOf(ttl)
        );

        boolean allowed      = result.get(0) == 1;
        long    remaining    = result.get(1);
        int     retryAfter   = result.get(2).intValue();

        return new RateResult(allowed, remaining, capacity,
                               Instant.now().plusSeconds(retryAfter), retryAfter);
    }
}

// ── 2. Sliding Window Counter ──────────────────────────
@Component
public class SlidingWindowLimiter implements RateLimiter {

    @Autowired private RedisTemplate<String,String> redis;

    public RateResult check(String key, RateRule rule) {
        long windowSec    = rule.getWindowSeconds();
        long limit        = rule.getMaxRequests();
        long nowSec       = Instant.now().getEpochSecond();
        long currWindow   = nowSec / windowSec;
        long prevWindow   = currWindow - 1;
        long elapsedInWin = nowSec % windowSec;

        String currKey = key + ":" + currWindow;
        String prevKey = key + ":" + prevWindow;

        // Pipeline: INCR current window + GET previous window (1 round trip)
        List<Object> results = redis.executePipelined(conn -> {
            conn.incr(currKey.getBytes());
            conn.expire(currKey.getBytes(), windowSec * 2);
            conn.get(prevKey.getBytes());
            return null;
        });

        long currCount = (Long) results.get(0);
        long prevCount = results.get(2) != null
            ? Long.parseLong((String) results.get(2)) : 0L;

        // Weighted count: interpolate previous window
        double weight = 1.0 - (double) elapsedInWin / windowSec;
        double weighted = prevCount * weight + currCount;

        if (weighted > limit) {
            // Rollback the increment we just did
            redis.opsForValue().decrement(currKey);
            long retryAfter = windowSec - elapsedInWin;
            return RateResult.denied(0, limit, retryAfter);
        }

        long remaining = (long) (limit - weighted);
        long resetAt   = (currWindow + 1) * windowSec;
        return RateResult.allowed(remaining, limit, resetAt);
    }
}

// ── 3. Sliding Window Log (high-precision, low-limit) ─
@Component
public class SlidingWindowLogLimiter implements RateLimiter {

    @Autowired private RedisTemplate<String,String> redis;

    public RateResult check(String key, RateRule rule) {
        long nowMs       = System.currentTimeMillis();
        long windowMs    = rule.getWindowSeconds() * 1000L;
        long windowStart = nowMs - windowMs;

        // Redis Sorted Set: member=uuid, score=timestamp
        List<Object> results = redis.executePipelined(conn -> {
            // Remove expired entries
            conn.zRemRangeByScore(key.getBytes(), 0, windowStart);
            // Count current window entries
            conn.zCard(key.getBytes());
            return null;
        });

        long count = (Long) results.get(1);

        if (count >= rule.getMaxRequests()) {
            return RateResult.denied(0, rule.getMaxRequests(),
                rule.getWindowSeconds());
        }

        // Add this request to the log
        redis.opsForZSet().add(key, UUID.randomUUID().toString(), nowMs);
        redis.expire(key, Duration.ofSeconds(rule.getWindowSeconds() + 1));

        long remaining = rule.getMaxRequests() - count - 1;
        return RateResult.allowed(remaining, rule.getMaxRequests(), 0);
    }
}

// ── 4. Rate Limit Middleware (Spring) ─────────────────
@Component
public class RateLimitMiddleware implements HandlerInterceptor {

    @Autowired private RateLimiterFactory  limiterFactory;
    @Autowired private RuleStore            ruleStore;     // cached rules
    @Autowired private KafkaTemplate<String,BlockedEvent> kafka;

    @Override
    public boolean preHandle(HttpServletRequest req,
                               HttpServletResponse res,
                               Object handler) {
        String endpoint  = req.getMethod() + " " + req.getRequestURI();
        String userId    = (String) req.getAttribute("userId");
        String ip        = extractClientIp(req);

        // Get all matching rules (sorted by priority)
        List<RateRule> rules = ruleStore.getMatchingRules(userId, ip, endpoint);

        for (RateRule rule : rules) {
            String key = buildKey(rule, userId, ip);
            RateLimiter limiter = limiterFactory.get(rule.getAlgorithm());

            RateResult result;
            try {
                result = limiter.check(key, rule);
            } catch (RedisException e) {
                // FAIL OPEN: Redis unavailable → allow request
                log.warn("Redis unavailable, failing open: {}", e.getMessage());
                continue; // skip this rule, try next
            }

            // Always add informational headers
            res.setHeader("X-RateLimit-Limit",       String.valueOf(result.getLimit()));
            res.setHeader("X-RateLimit-Remaining",  String.valueOf(result.getRemaining()));
            res.setHeader("X-RateLimit-Reset",     String.valueOf(result.getResetAt()));

            if (!result.isAllowed()) {
                res.setStatus(429);
                res.setHeader("Retry-After", String.valueOf(result.getRetryAfter()));
                res.setContentType("application/json");
                res.getWriter().write("""
                    {"error":"RATE_LIMIT_EXCEEDED",
                     "retry_after":%d,
                     "rule":"%s"}
                    """.formatted(result.getRetryAfter(), rule.getName()));

                // Async analytics
                kafka.send("blocked-requests",
                    new BlockedEvent(userId, ip, endpoint, rule.getName()));
                return false; // stop processing
            }
        }
        return true; // all rules passed — proceed
    }
}

import time, uuid, logging
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional

logger = logging.getLogger(__name__)


@dataclass
class RateResult:
    allowed:      bool
    remaining:    int
    limit:        int
    reset_at:     int   # unix timestamp
    retry_after:  int   # seconds to wait if denied


class RateLimiter(ABC):
    @abstractmethod
    def check(self, key: str, limit: int, window_sec: int, **kwargs) -> RateResult:
        pass


# ── 1. Token Bucket (Lua atomic) ──────────────────────
class TokenBucketLimiter(RateLimiter):
    """
    Atomic token bucket via Redis Lua script.
    Allows controlled bursts. Refills continuously.
    """

    TOKEN_BUCKET_LUA = """
        local key = KEYS[1]
        local capacity = tonumber(ARGV[1])
        local refill_rate = tonumber(ARGV[2])
        local now_ms = tonumber(ARGV[3])
        local ttl = tonumber(ARGV[4])

        local stored = redis.call('HGET', key, 'tokens')
        local last_ms = redis.call('HGET', key, 'last_refill_ms')

        local tokens = capacity
        local last_refill = now_ms

        if stored and last_ms then
            tokens = tonumber(stored)
            last_refill = tonumber(last_ms)
        end

        local elapsed_sec = (now_ms - last_refill) / 1000.0
        local new_tokens = math.min(capacity, tokens + elapsed_sec * refill_rate)

        if new_tokens >= 1 then
            new_tokens = new_tokens - 1
            redis.call('HSET', key, 'tokens', new_tokens, 'last_refill_ms', now_ms)
            redis.call('EXPIRE', key, ttl)
            return {1, math.floor(new_tokens), 0}
        else
            local wait_sec = math.ceil((1 - new_tokens) / refill_rate)
            redis.call('HSET', key, 'tokens', new_tokens, 'last_refill_ms', now_ms)
            redis.call('EXPIRE', key, ttl)
            return {0, 0, wait_sec}
        end
    """

    def __init__(self, redis_client):
        self.redis = redis_client
        self.script = redis_client.register_script(self.TOKEN_BUCKET_LUA)

    def check(self, key: str, limit: int, window_sec: int,
               burst_capacity: int = None, **kwargs) -> RateResult:
        capacity    = burst_capacity or int(limit * 1.5)
        refill_rate = limit / window_sec  # tokens/second
        now_ms      = int(time.time() * 1000)
        ttl         = window_sec * 2

        result = self.script(
            keys=[key],
            args=[capacity, refill_rate, now_ms, ttl]
        )
        allowed, remaining, retry_after = result
        reset_at = int(time.time()) + retry_after

        return RateResult(
            allowed=bool(allowed),
            remaining=int(remaining),
            limit=capacity,
            reset_at=reset_at,
            retry_after=int(retry_after)
        )


# ── 2. Sliding Window Counter ──────────────────────────
class SlidingWindowLimiter(RateLimiter):
    """
    O(1) memory. 98% accurate (assumes uniform request distribution).
    Eliminates fixed-window boundary burst problem.
    """

    def check(self, key: str, limit: int, window_sec: int, **kwargs) -> RateResult:
        now        = int(time.time())
        curr_win   = now // window_sec
        prev_win   = curr_win - 1
        elapsed    = now % window_sec

        curr_key = f"{key}:{curr_win}"
        prev_key = f"{key}:{prev_win}"

        # Pipeline: INCR + GET in one round trip
        pipe = self.redis.pipeline()
        pipe.incr(curr_key)
        pipe.expire(curr_key, window_sec * 2)
        pipe.get(prev_key)
        results = pipe.execute()

        curr_count = results[0]
        prev_count = int(results[2]) if results[2] else 0

        # Sliding interpolation
        weight   = 1.0 - elapsed / window_sec
        weighted = prev_count * weight + curr_count

        if weighted > limit:
            # Rollback — this request should not count
            self.redis.decr(curr_key)
            retry_after = window_sec - elapsed
            return RateResult(
                allowed=False, remaining=0, limit=limit,
                reset_at=int(time.time()) + retry_after,
                retry_after=retry_after
            )

        remaining = max(0, int(limit - weighted))
        reset_at  = (curr_win + 1) * window_sec
        return RateResult(
            allowed=True, remaining=remaining, limit=limit,
            reset_at=reset_at, retry_after=0
        )


# ── 3. Sliding Window Log (precision) ─────────────────
class SlidingWindowLogLimiter(RateLimiter):
    """
    Exact accuracy. O(limit) memory — use only for small limits.
    Best for: OTP sends (limit=5), login attempts (limit=10).
    """

    def check(self, key: str, limit: int, window_sec: int, **kwargs) -> RateResult:
        now_ms       = time.time() * 1000
        window_start = now_ms - (window_sec * 1000)

        pipe = self.redis.pipeline()
        pipe.zremrangebyscore(key, 0, window_start)  # remove expired
        pipe.zcard(key)                                 # count current
        results = pipe.execute()

        count = results[1]
        if count >= limit:
            return RateResult(
                allowed=False, remaining=0, limit=limit,
                reset_at=int(time.time()) + window_sec,
                retry_after=window_sec
            )

        # Add this request timestamp to the log
        member = f"{now_ms}:{uuid.uuid4().hex[:8]}"  # unique member
        pipe = self.redis.pipeline()
        pipe.zadd(key, {member: now_ms})
        pipe.expire(key, window_sec + 1)
        pipe.execute()

        return RateResult(
            allowed=True, remaining=limit - count - 1,
            limit=limit, reset_at=int(time.time()) + window_sec,
            retry_after=0
        )

from functools import wraps
from flask import Flask, request, jsonify, g

class RateLimitMiddleware:
    def __init__(self, app: Flask, limiter_factory, rule_store, kafka):
        self.app             = app
        self.limiter_factory = limiter_factory
        self.rule_store      = rule_store
        self.kafka           = kafka
        app.before_request(self.check_rate_limit)

    def check_rate_limit(self):
        endpoint = f"{request.method} {request.path}"
        user_id  = getattr(g, 'user_id', None)
        ip       = request.headers.get('X-Forwarded-For',
                                        request.remote_addr)

        rules = self.rule_store.get_matching_rules(user_id, ip, endpoint)

        for rule in rules:
            limiter = self.limiter_factory.get(rule.algorithm)
            key     = self._build_key(rule, user_id, ip)

            try:
                result = limiter.check(key, rule.max_requests,
                                       rule.window_seconds)
            except Exception as e:
                # FAIL OPEN: Redis down → allow request
                logger.warning(f"Rate limiter error, failing open: {e}")
                continue

            # Add informational headers to every response
            from flask import after_this_request
            rl_headers = {
                "X-RateLimit-Limit":     str(result.limit),
                "X-RateLimit-Remaining": str(result.remaining),
                "X-RateLimit-Reset":     str(result.reset_at),
            }
            g.rate_limit_headers = rl_headers

            if not result.allowed:
                # Async analytics
                self.kafka.send("blocked-requests", {
                    "user_id": user_id, "ip": ip,
                    "endpoint": endpoint, "rule": rule.name
                })
                resp = jsonify({
                    "error": "RATE_LIMIT_EXCEEDED",
                    "retry_after": result.retry_after,
                    "rule": rule.name
                })
                resp.status_code = 429
                resp.headers["Retry-After"] = str(result.retry_after)
                for k, v in rl_headers.items():
                    resp.headers[k] = v
                return resp

    def _build_key(self, rule, user_id, ip) -> str:
        if   rule.identifier_type == "USER_ID": ident = f"u:{user_id}"
        elif rule.identifier_type == "IP":      ident = f"ip:{ip}"
        else:                                    ident = f"k:{user_id}"
        return f"rl:{rule.rule_id}:{ident}"

-- Redis Lua Scripts — Atomic Rate Limit Operations
-- These scripts run atomically in Redis (no race conditions)
-- ══════════════════════════════════════════════════════

-- ── 1. Token Bucket (full implementation) ─────────────
-- KEYS[1] = bucket key (e.g., "rl:tb:user:user_123:rule_abc")
-- ARGV[1] = capacity (max tokens)
-- ARGV[2] = refill_rate (tokens per second)
-- ARGV[3] = now_ms (current timestamp in milliseconds)
-- ARGV[4] = ttl (seconds until key expires)
-- Returns: {allowed(0/1), remaining_tokens, retry_after_seconds}

local key          = KEYS[1]
local capacity      = tonumber(ARGV[1])
local refill_rate   = tonumber(ARGV[2])
local now_ms        = tonumber(ARGV[3])
local ttl           = tonumber(ARGV[4])

-- Read current state (nil if key doesn't exist = new user)
local stored     = redis.call('HGET', key, 'tokens')
local last_ms    = redis.call('HGET', key, 'last_refill_ms')

local tokens      = capacity  -- new user starts with full bucket
local last_refill = now_ms

if stored and last_ms then
    tokens      = tonumber(stored)
    last_refill = tonumber(last_ms)
end

-- Refill tokens based on elapsed time since last refill
local elapsed_sec = (now_ms - last_refill) / 1000.0
local new_tokens  = math.min(capacity, tokens + elapsed_sec * refill_rate)

if new_tokens >= 1 then
    -- ✅ ALLOW: consume one token
    new_tokens = new_tokens - 1
    redis.call('HSET', key,
        'tokens',         new_tokens,
        'last_refill_ms', now_ms)
    redis.call('EXPIRE', key, ttl)
    return {1, math.floor(new_tokens), 0}
else
    -- ❌ DENY: compute wait time until 1 token available
    local deficit    = 1 - new_tokens
    local wait_sec   = math.ceil(deficit / refill_rate)
    redis.call('HSET', key,
        'tokens',         new_tokens,
        'last_refill_ms', now_ms)
    redis.call('EXPIRE', key, ttl)
    return {0, 0, wait_sec}
end


-- ── 2. Fixed Window Counter (simplest) ────────────────
-- KEYS[1] = counter key (e.g., "rl:fw:user:user_123:28516920")
-- ARGV[1] = limit
-- ARGV[2] = window_seconds (for TTL)
-- Returns: {allowed(0/1), new_count, remaining}

local key    = KEYS[1]
local limit  = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local count = tonumber(redis.call('GET', key) or "0")

if count >= limit then
    -- ❌ DENY: already at limit, don't increment
    return {0, count, 0}
end

-- ✅ ALLOW: increment atomically
local new_count = redis.call('INCR', key)
if new_count == 1 then
    -- First request in this window — set TTL
    redis.call('EXPIRE', key, window)
end
return {1, new_count, limit - new_count}


-- ── 3. Multi-limit check (minute + day simultaneously) ─
-- KEYS[1] = minute counter key
-- KEYS[2] = day counter key
-- ARGV[1] = minute limit
-- ARGV[2] = day limit
-- Returns: {allowed, min_remaining, day_remaining, binding_window}

local min_key   = KEYS[1]
local day_key   = KEYS[2]
local min_limit = tonumber(ARGV[1])
local day_limit = tonumber(ARGV[2])

local min_count = tonumber(redis.call('GET', min_key) or "0")
local day_count = tonumber(redis.call('GET', day_key) or "0")

-- Check both limits BEFORE incrementing either
if min_count >= min_limit then
    return {0, 0, day_limit - day_count, "minute"}
end
if day_count >= day_limit then
    return {0, min_limit - min_count, 0, "day"}
end

-- Both under limit — increment both atomically
local new_min = redis.call('INCR', min_key)
local new_day = redis.call('INCR', day_key)
if new_min == 1 then redis.call('EXPIRE', min_key, 120) end
if new_day == 1 then redis.call('EXPIRE', day_key, 172800) end

return {1, min_limit - new_min, day_limit - new_day, "none"}
⚖️ Design Tradeoffs & Decisions
Rate limiter tradeoffs are nuanced. The same algorithm can be right for one use case and wrong for another. Always justify your choice with the specific requirements.
🧮 Algorithm: Token Bucket vs Sliding Window vs Fixed Window
✅ Token Bucket (general APIs)

Natural burst handling. Users don't get punished for opening the app. Configurable bucket size. Used by GitHub, Stripe, Twitter APIs. The right default for most public APIs.

⚠️ Sliding Window (strict)

Near-perfect accuracy. No boundary bursts. O(1) memory. Best for high-limit APIs (>100/min) where precise counting matters. The interpolation error (~2%) is negligible.

❌ Fixed Window (avoid for HTTP)

Simple but the 2× boundary burst problem makes it incorrect for strict rate limiting. Acceptable only for quota systems where the burst doesn't matter (e.g., "1000 API calls per day" — the boundary burst only matters at midnight).

🔒 Atomicity: Lua Script vs Redis Transaction (MULTI/EXEC) vs Optimistic Lock
✅ Lua Script

Runs atomically — no other Redis commands execute concurrently. Can include conditional logic (if tokens >= 1). Single round trip. Lowest latency. Industry standard for rate limiting in Redis.

⚠️ MULTI/EXEC Transaction

Atomic but only if combined with WATCH for optimistic locking. If WATCH detects a concurrent write, the transaction fails and must be retried. Under high concurrency, retry loops add latency. More code, more failure modes.

💡 Why not just GET then INCR?

Race condition: two requests arrive simultaneously. Both GET count=99. Both check 99 < 100 → allowed. Both INCR to 100. Result: 101 requests allowed when limit is 100. This is the TOCTOU bug. Lua script eliminates it entirely.

❌ Redis Failure: Fail Open vs Fail Closed vs Fallback
✅ Fail Open (primary)

If Redis is down, allow all requests. Legitimate users keep working. DDoS protection temporarily lost but legitimate traffic unaffected. Alert immediately for incident response. Standard for user-facing APIs.

❌ Fail Closed

If Redis down, reject all requests. Zero DDoS risk. But: your entire API goes down when Redis hiccups. Users get 429s even though nothing is wrong with their request. Never acceptable for user-facing APIs.

⚠️ Local Fallback

Per-server in-memory counter as backup. Better than fail-open — at least some rate limiting continues. But under-counts globally (each server independently allows up to limit). Suitable for bursty DDoS protection. Combine with fail-open for non-critical APIs.

🌍 Multi-DC: Local Redis vs Global Redis vs Rate Split
✅ Local Redis + Async Sync

Each region has its own Redis. Rate checks are sub-millisecond. Background sync of aggregated counts every 1-5s. Slight over-allowing during sync gap (~5s). Best for most use cases.

❌ Global Redis

Single Redis cluster, cross-region replication. Perfectly accurate but: every rate check has 50-100ms cross-region latency. A 100ms rate check makes your API 100ms slower. Completely unacceptable.

⚠️ Per-region rate split

Global limit 100/min → US-East: 50/min, EU: 30/min, APAC: 20/min. No cross-region state. Simplest. But: if user traffic shifts to EU, they exhaust EU quota while US/APAC have headroom. Works when traffic distribution is stable.

🔑 Key Design: Granular vs Coarse Rate Limit Keys
✅ Per-user per-endpoint

rl:{rule_id}:{user_id}. Most precise. A user abusing one endpoint doesn't affect their access to other endpoints. Industry standard for production APIs. Redis memory is cheap.

⚠️ Per-user only

Single counter per user across all endpoints. Simpler. But: a user who hits one slow endpoint counts against their quota for all fast endpoints. Unfair and confusing UX.

💡 Key TTL matters

Always set TTL on rate limit keys (= 2× window size). Without TTL, keys for inactive users accumulate forever — Redis OOM after months. For Token Bucket, set TTL on every update to reset inactivity timer. Active users' keys stay alive; inactive users auto-clean.
