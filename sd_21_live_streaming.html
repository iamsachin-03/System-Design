Live Streaming (Twitch)
Design a live video streaming platform — RTMP ingest from streamer OBS, real-time transcoding to HLS at multiple bitrates, sub-10-second end-to-end latency, CDN delivery to millions of concurrent viewers, and real-time chat at massive scale. Fundamentally different from on-demand video: no buffering, no seek, every viewer is live.
🔴 MUST DO 🟣 Hard RTMP Ingest HLS Adaptive Bitrate Transcoding Edge CDN Live Chat
🎯 Clarify Scope
Live streaming is fundamentally different from on-demand video (YouTube). Key differences: no seeking (viewers are at the live edge), latency matters (chat interaction requires <10s), transcoding must happen in real-time (not batched), and every CDN PoP must receive the stream continuously.

💬 Phase 1 — Ingest to Delivery Pipeline
PHASE 1 · STREAM PIPELINE
IR
INTERVIEWER
Walk me through what happens from the moment a streamer clicks "Go Live" in OBS to a viewer seeing the video.
ME
YOU
The pipeline has five stages:

1. RTMP Ingest: OBS sends a raw H.264/AAC stream over RTMP to our nearest ingest edge node. The ingest node is selected via DNS GeoDNS — the streamer connects to the closest point of presence. RTMP is TCP-based and handles the streamer's unreliable internet connection with retries.

2. Stream Relay: The ingest edge node relays the RTMP stream to our transcoding cluster (typically in a central data centre). This is a reliable backbone connection — we don't trust the last-mile streamer uplink for quality.

3. Real-time Transcoding: The transcoding server receives the raw stream and generates 3 quality variants simultaneously using GPU-accelerated FFmpeg:
  • 1080p @ 6 Mbps (high quality)
  • 720p @ 3 Mbps (medium)
  • 480p @ 1.5 Mbps (low / mobile)
Each variant is segmented into 2-second HLS chunks (.ts files) and pushed to S3 + CDN origin.

4. HLS Manifest Update: Every 2 seconds, the master .m3u8 playlist is updated pointing to the newest segments. The viewer's player polls this every 1-2s (low-latency HLS).

5. CDN Delivery: Viewers fetch segments from the nearest CDN edge node. The CDN pull-caches segments from S3. Since every viewer watches the same segments at the live edge, the CDN hit rate for a popular stream is ~99% — effectively all viewers share the same cached chunks.
💡 The key insight: Unlike on-demand video where different users want different positions, live streaming means all viewers want the same 2-second segment at the same time. This makes CDN caching extraordinarily effective — one cache miss serves millions of viewers.
⚡ Phase 2 — Latency Sources & Reduction
PHASE 2 · LATENCY
IR
INTERVIEWER
Where does the latency come from? Standard HLS is 30+ seconds. How do you get to sub-10?
ME
YOU
Standard HLS latency budget breakdown:
  • Encoding latency: Time to encode one segment. At 6-second segments → 6s minimum latency. Reduce to 2s segments → 2s encoding latency.
  • Playlist polling: Player polls manifest every segment length. At 6s segments → up to 6s additional. Use HTTP/2 server push or chunked transfer to reduce to ~0.5s.
  • CDN propagation: Segment reaches CDN after upload. With pull-caching: ~0.5s. With push (pre-positioned at edge): ~0s.
  • Player buffer: Player buffers 3 segments by default → 18s of buffer. With Low-Latency HLS (LL-HLS): 1 segment → 2s buffer.
Low-Latency HLS (Apple LL-HLS) approach:
  • 2-second segments instead of 6-second
  • Partial segments (0.5s chunks delivered mid-segment using chunked HTTP)
  • Playlist push using long-polling: player receives manifest update immediately when new segment is ready
  • Result: 3-8s end-to-end latency (vs 30s+ standard HLS)
Total latency budget: encoding (2s) + upload (0.5s) + CDN (0.5s) + player buffer (2s) + network (~1s) = ~6s.
💬 Phase 3 — Chat at 100K Messages/Min
PHASE 3 · CHAT
IR
INTERVIEWER
A popular streamer has 100,000 concurrent viewers all sending chat messages. How does chat work?
ME
YOU
100K viewers × 1 message/minute = ~1,667 messages/sec per channel. Each message must be delivered to all 100K viewers within 1-2 seconds. This is a fan-out problem.

Architecture:
Each viewer maintains a WebSocket connection to a chat server. Chat servers are stateful — they know which channel each connection is watching. We use a pub/sub system (Redis Pub/Sub or Kafka) to fan out messages:
  1. Viewer sends message → chat server validates (rate limit, moderation)
  2. Chat server publishes to Redis channel: PUBLISH chat:{channel_id} {message_json}
  3. All chat servers subscribed to that channel receive the message via Redis Pub/Sub
  4. Each chat server pushes to all WebSocket connections watching that channel
Scale math: 100K viewers across 50 chat servers = 2,000 WebSocket connections per server. Each server subscribes to Redis for their active channels. Redis Pub/Sub easily handles 100K message deliveries/sec per channel.

Slow mode & rate limiting: For popular channels, apply slow mode (1 message per user per 30s). This reduces 100K viewers to at most 3,333 messages/min regardless of how many people try to chat simultaneously.

// ── 1. Stream Ingest Controller ───────────────────────
@RestController
public class StreamIngestController {

    @Autowired private StreamKeyService    keyService;
    @Autowired private TranscoderService  transcoder;
    @Autowired private MetadataService    metadata;

    // Called by RTMP relay when stream connects
    @PostMapping("/internal/stream/on-publish")
    public ResponseEntity<?> onStreamStart(
            @RequestParam String name,  // stream key
            @RequestParam String addr) { // streamer IP

        // 1. Validate stream key
        Streamer streamer = keyService.validate(name);
        if (streamer == null)
            return ResponseEntity.status(403).body("Invalid stream key");

        // 2. Launch transcoding pipeline for this stream
        TranscodingJob job = TranscodingJob.builder()
            .channelId(streamer.getChannelId())
            .inputRtmpUrl("rtmp://localhost/live/" + name)
            .qualities(List.of(
                new Quality(1920, 1080, 6000),
                new Quality(1280, 720,  3000),
                new Quality(854,  480,  1500)))
            .segmentDurationSeconds(2)
            .outputS3Prefix("streams/" + streamer.getChannelId())
            .build();
        transcoder.launch(job);

        // 3. Mark channel as live in metadata store
        metadata.setLive(streamer.getChannelId(), true);

        return ResponseEntity.ok("Stream accepted");
    }

    @PostMapping("/internal/stream/on-done")
    public void onStreamEnd(@RequestParam String name) {
        Streamer s = keyService.validate(name);
        if (s == null) return;
        transcoder.stop(s.getChannelId());
        metadata.setLive(s.getChannelId(), false);
        // Trigger VOD assembly (async)
        vodService.assembleVod(s.getChannelId());
    }
}

// ── 2. HLS Packager — Segment Push ────────────────────
@Service
public class HlsPackager {

    @Autowired private S3Client    s3;
    @Autowired private RedisTemplate<String,String> redis;

    /**
     * Called every 2 seconds with a new completed segment.
     * Uploads to S3 and updates the live HLS manifest.
     */
    public void onSegmentComplete(String channelId, String quality,
                                     int sequenceNum, byte[] tsData) {
        String segKey = String.format("streams/%s/%s/seg_%d.ts",
            channelId, quality, sequenceNum);

        // 1. Upload segment to S3 (CDN pulls from here)
        s3.putObject(PutObjectRequest.builder()
            .bucket("stream-segments").key(segKey)
            .cacheControl("public, max-age=86400") // long TTL — segments don't change
            .build(), RequestBody.fromBytes(tsData));

        // 2. Update manifest in Redis (fast, viewers poll manifest frequently)
        String manifestKey = "manifest:" + channelId + ":" + quality;
        redis.opsForList().rightPush(manifestKey, segKey);
        redis.opsForList().trim(manifestKey, -5, -1); // keep last 5 segments

        // 3. Upload updated manifest to S3 (CDN polls with 1s TTL)
        byte[] manifest = buildManifest(channelId, quality, sequenceNum);
        s3.putObject(PutObjectRequest.builder()
            .bucket("stream-segments")
            .key("streams/" + channelId + "/" + quality + ".m3u8")
            .cacheControl("public, max-age=1") // short TTL — manifest changes every 2s
            .build(), RequestBody.fromBytes(manifest));
    }

    private byte[] buildManifest(String channelId, String quality, int seqNum) {
        List<String> recentSegs = redis.opsForList()
            .range("manifest:" + channelId + ":" + quality, 0, -1);
        StringBuilder sb = new StringBuilder();
        sb.append("#EXTM3U
#EXT-X-VERSION:6
");
        sb.append("#EXT-X-TARGETDURATION:2
");
        sb.append("#EXT-X-MEDIA-SEQUENCE:").append(seqNum - recentSegs.size() + 1).append("
");
        for (String seg : recentSegs) {
            sb.append("#EXTINF:2.0,
").append(seg).append("
");
        }
        return sb.toString().getBytes();
    }
}

// ── 3. Chat Service — WebSocket + Redis Pub/Sub ────────
@Component
public class ChatWebSocketHandler extends TextWebSocketHandler {

    @Autowired private RedisTemplate<String,String> redis;
    @Autowired private RateLimiter    rateLimiter;
    // channelId → set of WebSocket sessions watching that channel
    private ConcurrentHashMap<String, Set<WebSocketSession>> viewers = new ConcurrentHashMap<>();

    @Override
    protected void handleTextMessage(WebSocketSession session, TextMessage msg) {
        ChatMessage chat = JsonUtil.parse(msg.getPayload());

        // Rate limit: 1 message per user per 30s in slow mode
        if (!rateLimiter.tryAcquire("chat:" + chat.getUserId(), 1, 30))
            return;

        // Publish to Redis Pub/Sub → all chat servers for this channel receive it
        redis.convertAndSend("chat:" + chat.getChannelId(),
            JsonUtil.toJson(chat));
    }

    // Redis subscription handler — push to all local WebSocket sessions
    @RedisMessageHandler
    public void onChatMessage(String channelId, String messageJson) {
        Set<WebSocketSession> sessions = viewers.getOrDefault(channelId, Set.of());
        for (WebSocketSession s : sessions) {
            if (s.isOpen()) {
                try { s.sendMessage(new TextMessage(messageJson)); }
                catch (Exception ignored) {}
            }
        }
    }
}

import subprocess, boto3, json, time
from typing import List

s3 = boto3.client('s3')
BUCKET = 'stream-segments'

def launch_transcoder(channel_id: str, rtmp_input: str):
    """
    Launch FFmpeg with GPU acceleration (NVENC) for 3-quality HLS output.
    Segments uploaded to S3 via HLS packager hook.
    """
    cmd = [
        'ffmpeg', '-re',
        '-i', rtmp_input,
        # 1080p output
        '-c:v', 'h264_nvenc', '-b:v', '6000k', '-vf', 'scale=1920:1080',
        '-c:a', 'aac', '-b:a', '192k',
        '-f', 'hls', '-hls_time', '2',
        '-hls_flags', 'delete_segments+append_list',
        '-hls_segment_filename', f'/tmp/{channel_id}/1080p/seg_%d.ts',
        f'/tmp/{channel_id}/1080p.m3u8',
        # 720p output (map same input)
        '-c:v', 'h264_nvenc', '-b:v', '3000k', '-vf', 'scale=1280:720',
        '-f', 'hls', '-hls_time', '2',
        '-hls_segment_filename', f'/tmp/{channel_id}/720p/seg_%d.ts',
        f'/tmp/{channel_id}/720p.m3u8',
    ]
    return subprocess.Popen(cmd)


def upload_segment(channel_id: str, quality: str,
                    seg_num: int, seg_path: str):
    """Upload a completed .ts segment to S3 with long TTL."""
    key = f'streams/{channel_id}/{quality}/seg_{seg_num}.ts'
    s3.upload_file(seg_path, BUCKET, key,
        ExtraArgs={'CacheControl': 'public, max-age=86400',
                   'ContentType': 'video/MP2T'})


def update_manifest(channel_id: str, quality: str,
                      recent_segs: List[int], redis_client):
    """Rebuild and upload the HLS manifest with recent segments."""
    manifest_lines = [
        '#EXTM3U', '#EXT-X-VERSION:6',
        '#EXT-X-TARGETDURATION:2',
        f'#EXT-X-MEDIA-SEQUENCE:{recent_segs[0]}',
    ]
    for n in recent_segs:
        manifest_lines += ['#EXTINF:2.0,',
            f'https://cdn.example.com/streams/{channel_id}/{quality}/seg_{n}.ts']

    manifest = '\n'.join(manifest_lines).encode()
    key = f'streams/{channel_id}/{quality}.m3u8'
    s3.put_object(Bucket=BUCKET, Key=key, Body=manifest,
        CacheControl='public, max-age=1',  # short TTL — changes every 2s
        ContentType='application/x-mpegURL')


# ── Chat Service ──────────────────────────────────────
import asyncio, websockets

class ChatServer:
    def __init__(self, redis):
        self.redis    = redis
        self.channels = {}  # channel_id → set of websocket connections

    async def handle(self, ws, path):
        channel_id = path.strip('/')
        self.channels.setdefault(channel_id, set()).add(ws)
        # Subscribe to Redis pub/sub for this channel
        pubsub = self.redis.pubsub()
        await pubsub.subscribe(f'chat:{channel_id}')
        try:
            async for msg in ws:
                data = json.loads(msg)
                # Validate + rate limit, then publish
                await self.redis.publish(f'chat:{channel_id}', json.dumps({
                    'user': data['user'], 'text': data['text'],
                    'ts': int(time.time() * 1000)
                }))
        finally:
            self.channels.get(channel_id, set()).discard(ws)

    async def broadcast_loop(self, channel_id: str):
        """Receive from Redis, fan out to all local WebSocket connections."""
        pubsub = self.redis.pubsub()
        await pubsub.subscribe(f'chat:{channel_id}')
        async for message in pubsub.listen():
            if message['type'] == 'message':
                payload = message['data']
                dead = set()
                for ws in self.channels.get(channel_id, set()):
                    try:
                        await ws.send(payload)
                    except:
                        dead.add(ws)
                self.channels[channel_id] -= dead
⚖️ Tradeoffs
⏱ Latency vs Reliability: HLS vs WebRTC vs RTMP Playback
✅ HLS / LL-HLS (chosen)

5-10s latency. Works everywhere (every browser, mobile, smart TV). CDN-cacheable — one stream serves millions. Battle-tested. Industry standard. Can't do sub-3s but 5-10s is acceptable for most gaming streams.

⚠️ WebRTC (ultra-low latency)

<500ms latency. P2P or SFU server. Excellent for interactive applications (trivia, co-op games, where viewer reaction time matters). But: doesn't scale to 1M viewers per stream — requires mesh or SFU infrastructure. High server cost. Not CDN-cacheable. Twitch uses for "co-streaming" with hosts.

❌ RTMP Playback

2-3s latency. But: Flash is dead, requires custom player, no CDN caching. Every viewer connection directly to origin server — doesn't scale. Deprecated as a delivery protocol. Fine for ingest (streamer → server) but not for delivery (server → viewer).

🎬 Transcoding: Live on Ingest vs Batched vs Client-Side
✅ Real-time GPU transcoding

GPU-accelerated NVENC processes in under 2s per segment. One GPU server handles 50 streams. Scales linearly with stream count. High GPU cost but no alternative for live content — you can't buffer and transcode offline.

❌ Passthrough (no transcoding)

Forward raw stream to viewers. Zero transcoding cost. But: all viewers must support the streamer's exact codec/bitrate. A 6 Mbps 1080p stream will buffer on mobile. No adaptive bitrate = terrible viewer experience on poor connections. Not viable for consumer platform.

💡 Adaptive vs Fixed Quality

Fixed quality (one bitrate): simpler, 33% of transcoding cost. But viewers on mobile or poor connection either buffer or watch low-res. Adaptive (3 bitrates): HLS.js player automatically selects 480p/720p/1080p based on available bandwidth. Industry standard for consumer platforms.

💬 Chat: Redis Pub/Sub vs Kafka vs WebSocket Direct
❌ WebSocket direct (no broker)

All chat servers must know each other. Viewer sends → server must forward to all other servers. Full mesh O(N²) at 50 servers. Operational nightmare. No persistence.

✅ Redis Pub/Sub (chosen)

Central broker. Any server publishes → all subscribers receive. Simple. <1ms delivery. No persistence (fire and forget). Perfect for chat — messages don't need durability. Redis easily handles 100K messages/sec per channel.

⚠️ Kafka (over-engineered for chat)

Durable, ordered, replayable. Would work. But: adds 50-200ms latency per message (unnecessary for chat). Operational overhead. Chat doesn't need durability — if you miss a message while disconnected, that's fine. Save Kafka for when you need the durability (analytics).
