Ride Matching / Dispatch (Uber)
Design the core of Uber's dispatch system — real-time geospatial indexing of millions of moving drivers, matching riders to the nearest available driver in under 2 seconds, ETA computation via road network graphs, surge pricing from demand/supply imbalance, and handling the thundering herd when a concert ends.

💬 Phase 1 — Geospatial Indexing
PHASE 1 · GEO INDEX
IR
INTERVIEWER
1 million drivers are moving around the world. A rider requests a trip. How do you find nearby available drivers efficiently?
ME
YOU
Naive approach: scan all 1M drivers, calculate distance to each. At O(N) this is 1M distance calculations per request — too slow.

Geospatial indexing with S2 or H3 cells:

The Earth's surface is divided into hierarchical hexagonal/square cells. Uber uses Uber H3 (hexagonal hierarchical index); Google Maps uses S2 (quad cells). The key property: all points within a cell are close together, and adjacent cells cover contiguous geography.

Each driver's location is mapped to a cell ID at a resolution that covers roughly 1km². Driver locations are stored in Redis: HSET cell:{h3_cell_id} {driver_id} {status,lat,lng,timestamp}.

Finding nearby drivers: Convert rider's location to H3 cell ID at the same resolution. Query that cell + its 6 immediate neighbours (H3's hexagonal grid means 7 cells = ~7 km² coverage). This reduces 1M driver scan to typically <500 drivers in nearby cells.

Dynamic radius: If fewer than 3 available drivers found in radius-1, expand to radius-2 (next ring of cells, ~37 cells). Most cities find enough drivers in radius-1. Rural areas expand to radius-3+.
💡 Why H3 over geohash? H3 hexagons have equal area and equidistant neighbours — no edge distortion. Geohash squares have non-uniform distance to corner neighbours vs edge neighbours. For proximity queries, hexagons give more uniform results.
⚡ Phase 2 — Matching Algorithm
PHASE 2 · MATCHING
IR
INTERVIEWER
You found 50 nearby drivers. How do you decide which one gets the trip?
ME
YOU
Simplistic answer: pick the nearest driver. But "nearest" in Euclidean distance ≠ "nearest" in real-world driving time due to road topology, traffic, one-ways. Better:

ETA-based matching: For each candidate driver, compute the estimated time to reach the rider using the road network graph. Select the driver with minimum ETA to pickup.

ETA computation: Uber maintains a road graph for each city (nodes = intersections, edges = road segments with travel time weights). Travel times are updated in real-time from GPS trace data of all active drivers. For each candidate driver, run a truncated Dijkstra/A* from driver location to rider location. With a time limit (e.g., compute for max 2km radius), this runs in <10ms per driver.

Multi-objective scoring: Production matching isn't purely ETA-minimizing. Uber's algorithm balances:
  • ETA to pickup (rider satisfaction)
  • Driver's heading direction (a driver already driving toward the pickup zone gets a bonus)
  • Driver's idle time (fairness — longer-waiting drivers preferred if ETA is similar)
  • Surge zone efficiency (matching within the same surge zone avoids cross-zone arbitrage)
Batched matching: Rather than matching each ride request independently, Uber batches requests every 500ms and runs an optimal matching (linear assignment problem) across all pending requests and available drivers. This allows better global optimisation — avoids two drivers crossing paths to serve two riders when swapping would be more efficient.
💰 Phase 3 — Surge Pricing
PHASE 3 · SURGE
IR
INTERVIEWER
A concert just ended. 10,000 people suddenly want rides simultaneously. How does surge pricing work technically?
ME
YOU
Surge is a demand-to-supply ratio computed per geographic zone, updated every 1-2 minutes:

Supply: Count of available drivers in each H3 zone (queried from the driver location index).
Demand: Count of active ride requests in each zone in the last 5 minutes (time-windowed from the request stream).
Surge multiplier: A piecewise function of demand/supply ratio. At ratio=1 (balanced), multiplier=1.0×. At ratio=2 (twice as many riders as drivers), multiplier=1.5×. At ratio=5+, multiplier=3-4×. The exact function is tuned to bring supply into equilibrium — higher prices incentivise distant drivers to drive into the surge zone.

Technical implementation: A Flink/Spark streaming job reads driver location updates and ride requests, computes the demand/supply ratio per H3 zone every 60 seconds, and writes the surge multiplier to Redis: SET surge:{h3_zone} 2.1 EX 120 (TTL 2 minutes ensures stale surge expires). The pricing service reads this key at trip request time to compute the final fare estimate.

// Ride Matching — Java: Geo lookup + ETA-based match
// ══════════════════════════════════════════════════════

@Service
public class MatchingService {

    @Autowired private RedisGeoOps  geoOps;
    @Autowired private EtaService    etaService;
    @Autowired private H3Index       h3;

    /**
     * Find and dispatch the best driver for a ride request.
     * 1. Get nearby available drivers via H3 + Redis GEO
     * 2. Compute ETA for top candidates (parallel)
     * 3. Score and select best match
     * 4. Send offer to driver (WebSocket)
     */
    public MatchResult matchRider(RideRequest req) {
        Location riderLoc = req.getPickupLocation();

        // 1. H3 cell lookup — find nearby drivers in radius 1
        long riderCell  = h3.geoToH3(riderLoc.getLat(), riderLoc.getLng(), 8); // res 8
        List<long> cells = h3.kRing(riderCell, 1);    // center + 6 neighbors

        List<Driver> candidates = cells.stream()
            .flatMap(cell -> geoOps.getAvailableDrivers(cell).stream())
            .filter(Driver::isAvailable)
            .limit(30)  // cap candidates for ETA computation
            .collect(Collectors.toList());

        // If not enough drivers, expand to radius 2
        if (candidates.size() < 3) {
            cells = h3.kRing(riderCell, 2);
            candidates = cells.stream()
                .flatMap(cell -> geoOps.getAvailableDrivers(cell).stream())
                .filter(Driver::isAvailable).limit(30).collect(Collectors.toList());
        }

        if (candidates.isEmpty()) return MatchResult.noDrivers();

        // 2. Parallel ETA computation for all candidates
        List<CompletableFuture<DriverEta>> etaFutures = candidates.stream()
            .map(driver -> CompletableFuture.supplyAsync(
                () -> etaService.computePickupEta(driver.getLocation(), riderLoc)))
            .collect(Collectors.toList());

        List<DriverEta> etas = etaFutures.stream()
            .map(CompletableFuture::join)
            .collect(Collectors.toList());

        // 3. Score drivers: ETA + heading alignment + wait time
        DriverEta best = etas.stream()
            .min(Comparator.comparingDouble(e -> score(e, req)))
            .orElseThrow();

        // 4. Send offer and wait for acceptance (10s timeout)
        return offerAndAwait(best.getDriver(), req);
    }

    /** Scoring: lower is better. ETA-dominant with adjustments. */
    private double score(DriverEta eta, RideRequest req) {
        double etaSeconds    = eta.getEtaSeconds();
        double headingBonus  = headingAlignment(eta.getDriver(), req.getPickupLocation()) * 30;
        double waitBonus     = eta.getDriver().getIdleSeconds() * 0.1;
        return etaSeconds - headingBonus - waitBonus;
    }

    /** headingAlignment: 1.0 if driver heading toward pickup, 0.0 if away */
    private double headingAlignment(Driver driver, Location pickup) {
        double bearingToPickup = bearing(driver.getLocation(), pickup);
        double diff = Math.abs(driver.getHeading() - bearingToPickup);
        if (diff > 180) diff = 360 - diff;
        return 1.0 - (diff / 180.0);
    }
}

// ── Driver Location Update ────────────────────────────
@Service
public class LocationService {
    @Autowired private RedisTemplate<String,String> redis;
    @Autowired private H3Index h3;

    public void updateDriverLocation(String driverId, double lat, double lng) {
        long cell = h3.geoToH3(lat, lng, 8);

        // Remove from old cell if changed
        String oldCell = redis.opsForValue().get("driver_cell:" + driverId);
        if (oldCell != null && !oldCell.equals(String.valueOf(cell))) {
            redis.opsForHash().delete("cell:" + oldCell, driverId);
        }

        // Update current cell
        redis.opsForValue().set("driver_cell:" + driverId, String.valueOf(cell), 30, TimeUnit.SECONDS);
        redis.opsForHash().put("cell:" + cell, driverId,
            lat + "," + lng + "," + System.currentTimeMillis());
        redis.expire("cell:" + cell, 30, TimeUnit.SECONDS);
    }
}

# Ride Matching — Python: H3 geo lookup + matching
# ══════════════════════════════════════════════════════

import h3, redis, asyncio, math
from dataclasses import dataclass
from typing import List, Optional

r = redis.Redis()

@dataclass
class Driver:
    driver_id: str
    lat:       float
    lng:       float
    heading:   float  # degrees 0-360
    idle_secs: int
    available: bool

@dataclass
class RideRequest:
    rider_id:   str
    pickup_lat: float
    pickup_lng: float
    dest_lat:   float
    dest_lng:   float


def update_driver_location(driver_id: str, lat: float, lng: float):
    """Called every 4s per driver. Update H3 cell assignment."""
    cell = h3.geo_to_h3(lat, lng, resolution=8)
    old_cell = r.get(f\'driver_cell:{driver_id}\')
    if old_cell and old_cell.decode() != cell:
        r.hdel(f\'cell:{old_cell.decode()}\', driver_id)
    r.setex(f\'driver_cell:{driver_id}\', 30, cell)
    r.hset(f\'cell:{cell}\', driver_id, f\'{lat},{lng}\')
    r.expire(f\'cell:{cell}\', 30)


def get_nearby_drivers(lat: float, lng: float, radius: int = 1) -> List[Driver]:
    """H3 k-ring search for available drivers."""
    rider_cell = h3.geo_to_h3(lat, lng, resolution=8)
    cells      = h3.k_ring(rider_cell, radius)
    drivers    = []
    for cell in cells:
        data = r.hgetall(f\'cell:{cell}\')
        for did, pos in data.items():
            driver_lat, driver_lng = map(float, pos.decode().split(\',\'))
            # Check availability flag
            status = r.get(f\'driver_status:{did.decode()}\')
            if status and status.decode() == \'available\':
                drivers.append(Driver(
                    driver_id=did.decode(), lat=driver_lat, lng=driver_lng,
                    heading=0, idle_secs=0, available=True))
    return drivers[:30]


def haversine_km(lat1, lng1, lat2, lng2) -> float:
    R = 6371
    dlat = math.radians(lat2 - lat1)
    dlng = math.radians(lng2 - lng1)
    a = math.sin(dlat/2)**2 + math.cos(math.radians(lat1))*math.cos(math.radians(lat2))*math.sin(dlng/2)**2
    return R * 2 * math.asin(math.sqrt(a))


def estimate_eta_seconds(driver: Driver, pickup_lat: float, pickup_lng: float) -> float:
    """Simplified ETA: distance / avg speed. Production: A* on road graph."""
    dist_km   = haversine_km(driver.lat, driver.lng, pickup_lat, pickup_lng)
    avg_kmh   = 25  # city average speed
    return (dist_km / avg_kmh) * 3600


def match_driver(req: RideRequest) -> Optional[Driver]:
    """Full matching pipeline."""
    drivers = get_nearby_drivers(req.pickup_lat, req.pickup_lng, radius=1)
    if len(drivers) < 3:
        drivers = get_nearby_drivers(req.pickup_lat, req.pickup_lng, radius=2)
    if not drivers:
        return None

    # Score each driver (lower = better)
    def score(driver: Driver) -> float:
        eta   = estimate_eta_seconds(driver, req.pickup_lat, req.pickup_lng)
        wait_bonus = driver.idle_secs * 0.1
        return eta - wait_bonus

    best = min(drivers, key=score)
    # In production: send WebSocket push, wait 10s for acceptance
    return best


# ── Surge pricing ────────────────────────────────────
def compute_surge(zone_cell: str) -> float:
    demand = int(r.get(f\'demand:{zone_cell}\') or 0)
    supply = int(r.get(f\'supply:{zone_cell}\') or 1)
    ratio  = demand / max(supply, 1)
    if   ratio < 1.2: return 1.0
    elif ratio < 2.0: return 1.5
    elif ratio < 3.0: return 2.0
    else:             return min(4.0, 1.0 + ratio * 0.6)
⚖️ Tradeoffs
🗺 Geospatial Index: H3 vs S2 vs Geohash vs PostGIS
✅ H3 Hexagons

Uber's own standard. Equal-area hexagons with uniform neighbour distance. 7-cell k-ring covers ~7 km² at res 8. Excellent for proximity queries. Open source. Python, Java, Go bindings. Best for ride-sharing because hexagons have no "corner" distortion.

S2 Geometry (Google)

Quad-tree subdivision of the sphere. Used by Google Maps, Foursquare. Excellent for arbitrary polygon containment queries. Cells are squares — corner cells are farther from centre than edge cells, making proximity queries slightly less uniform than H3. Robust, battle-tested.

Geohash

Z-order curve encoding lat/lng as a base-32 string. Simple, human-readable (e.g., "u4pruyd"). Adjacent cells can have very different geohash prefixes at boundaries — requires checking 8 neighbours instead of 6. Sufficient for simple use cases. Not uniform area. Avoid for production ride-sharing at scale.

🎯 Matching: Greedy vs Batched Optimal
Greedy (first-come, min-ETA)

Match each request independently as it arrives. Simple, low latency (<200ms). But: can make globally suboptimal choices — two drivers may "cross" to serve two riders when swapping would save 4 minutes total. Works fine at low density.

✅ Batched (500ms window)

Collect all requests in a 500ms window. Run bipartite matching (Hungarian algorithm or linear programming relaxation) to optimise total ETA across all pending requests simultaneously. ~5% better than greedy at city scale. Uber uses this in production. Adds 500ms latency — acceptable given the 2s total budget.

ML-based matching

Train a model on historical data (demand patterns, traffic, driver behaviour). Predict which driver is most likely to accept and complete the trip efficiently. Better than ETA alone. Higher engineering complexity. Uber uses ML for cancellation prediction, not raw matching.
