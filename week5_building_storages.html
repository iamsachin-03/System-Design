Implementing a Single-node Cache

Before scaling to a distributed cache, you must understand the internals of a single-node cache. This is how Redis, Memcached, and DiceDB work at their core — and the foundation for every distributed caching system.

Core Data Structures

Hash Map — O(1) lookup

The primary key-value store. key → {value, expiry, lru_node_ptr}. Collision resolution via chaining or open addressing. Python's dict, Java's HashMap, or a custom hash table with linear probing.

Resize strategy: when load factor exceeds 0.75, double capacity and rehash. Amortized O(1) insert.

Doubly-Linked List — O(1) LRU

Maintain access recency. Every GET/SET moves the node to the list head (MRU). Eviction removes from the tail (LRU). O(1) because we store a pointer to the node in the hash map — no search needed.

The combination of Hash Map + Doubly-Linked List = O(1) LRU cache. This is a canonical interview question — know it cold.

Memory Layout

Eviction Policies

When the cache is full, which entry is removed to make room for a new one? The choice fundamentally affects hit rate and system behaviour.

LRU — Least Recently Used (The Interview Standard)

Evict the entry that was accessed least recently. Assumes temporal locality — recently accessed data is likely to be accessed again soon. Implementation: Hash Map + Doubly-Linked List = O(1) all operations.

lru_cache.py — O(1) LRU with hash map + doubly-linked listPython
from collections import OrderedDict

class LRUCache:
    """
    O(1) GET, PUT, DELETE with LRU eviction.
    Uses OrderedDict (internally a dict + doubly-linked list).
    """
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = OrderedDict()   # key → value, LRU order

    def get(self, key: str):
        if key not in self.cache:
            return None
        self.cache.move_to_end(key)   # mark as MRU (most recently used)
        return self.cache[key]

    def put(self, key: str, value):
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.capacity:
            self.cache.popitem(last=False)   # remove LRU (oldest = first item)

    def delete(self, key: str) -> bool:
        return bool(self.cache.pop(key, None))

Lock-free Concurrency

How do you handle thousands of concurrent requests on a single-node cache without locks degrading throughput? Two approaches dominate production caches.

Approach 1: Single-threaded Event Loop (Redis model)

How Redis Does It

  • Single thread processes ALL commands sequentially
  • No locks needed — only one thread accesses data structures
  • I/O multiplexing (epoll/kqueue) handles thousands of client connections
  • Non-blocking I/O: accept → read command → process → write response → next
  • Throughput: ~1M ops/sec on a modern CPU core

Why It Works

  • Cache operations are CPU-bound, not I/O-bound (data in RAM)
  • Each operation is microseconds — no value in parallelizing
  • No context switching, no lock contention, no cache coherence overhead
  • Simple programming model — linearizable by construction
1 thread × 1M ops/sec > 4 threads × 200K ops/sec each (lock overhead). Single-threaded wins for in-memory operations.

Approach 2: Sharded Lock (Java ConcurrentHashMap model)

Divide the hash map into N segments (shards). Each shard has its own lock. A key maps to one shard. Operations on different shards run in parallel — only same-shard operations contend.

sharded_cache.py — segment-level locking for multi-threaded accessPython
import threading, hashlib

class ShardedCache:
    """
    N independent LRU caches, each with its own lock.
    Contention = 1/N of a single global lock.
    """
    def __init__(self, total_capacity: int, num_shards: int = 16):
        self.num_shards = num_shards
        shard_cap = max(1, total_capacity // num_shards)
        self.shards = [LRUCache(shard_cap) for _ in range(num_shards)]
        self.locks  = [threading.Lock()    for _ in range(num_shards)]

    def _shard(self, key: str) -> int:
        return int(hashlib.md5(key.encode()).hexdigest(), 16) % self.num_shards

    def get(self, key: str):
        idx = self._shard(key)
        with self.locks[idx]:          # only lock one shard
            return self.shards[idx].get(key)

    def put(self, key: str, value):
        idx = self._shard(key)
        with self.locks[idx]:
            self.shards[idx].put(key, value)

Approach 3: Lock-free with CAS (Compare-And-Swap)

Use atomic hardware primitives (CAS, fetch-and-add) to update data structures without acquiring a mutex. More complex to implement correctly but eliminates lock contention entirely. Used in Caffeine (Java's state-of-the-art cache).

Caffeine (Java) uses W-TinyLFU — a lock-free cache with frequency sketch for policy decisions and ring buffers for read/write admission. It achieves 10× higher throughput than ConcurrentHashMap + LRU at high concurrency. This is what Guava Cache was replaced with.
TTL & Expiry Mechanisms

TTL (Time-To-Live) is a first-class feature of any production cache. Understanding how expiry is implemented affects both correctness and memory efficiency.

Three Expiry Strategies

Lazy Expiry (Passive)

Check expiry at read time only. GET key → if expired: delete and return nil. No background work. Simple. Problem: expired keys occupy memory until next access. If keys are never accessed again, memory leaks.

CPU efficient. Memory inefficient. Good for low-TTL, high-access keys.

Active Expiry (Background sweep)

Background thread periodically scans and deletes expired keys. Redis does this every 100ms — samples 20 random volatile keys, deletes expired ones, repeats if >25% were expired.

Memory efficient. CPU cost bounded. Slightly delayed reclamation.

Min-Heap / Priority Queue

Maintain a min-heap ordered by expiry time. Background thread wakes exactly when the next key expires. Precise expiry, efficient for large TTL variance. Used in DiceDB.

Precise timing. O(log N) per insert/delete. Memory overhead for heap.
ttl_cache.py — lazy + active expiry combinationPython
import time, threading, heapq

class TTLCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self._store: dict = {}              # key → (value, expires_at)
        self._heap:  list = []              # min-heap: (expires_at, key)
        self._lock = threading.Lock()
        threading.Thread(target=self._sweep, daemon=True).start()

    def set(self, key: str, value, ttl_seconds: float = None):
        with self._lock:
            exp = time.time() + ttl_seconds if ttl_seconds else None
            self._store[key] = (value, exp)
            if exp:
                heapq.heappush(self._heap, (exp, key))
            if len(self._store) > self.capacity:
                self._evict_one()

    def get(self, key: str):
        with self._lock:
            entry = self._store.get(key)
            if not entry: return None
            value, exp = entry
            if exp and time.time() > exp:   # lazy expiry check
                del self._store[key]
                return None
            return value

    def _sweep(self):
        """Background thread: active expiry using min-heap."""
        while True:
            now = time.time()
            with self._lock:
                while self._heap and self._heap[0][0] <= now:
                    exp, key = heapq.heappop(self._heap)
                    entry = self._store.get(key)
                    if entry and entry[1] == exp:  # not re-set with new TTL
                        del self._store[key]
                sleep_until = self._heap[0][0] - now if self._heap else 1.0
            time.sleep(max(0.001, min(sleep_until, 1.0)))

production_cache.py — thread-safe LRU+TTL cachePython
import time, threading
from dataclasses import dataclass
from typing import Optional, Any
from collections import OrderedDict

@dataclass
class CacheEntry:
    value:      Any
    expires_at: Optional[float] = None   # None = no TTL

    def is_expired(self) -> bool:
        return self.expires_at is not None and time.time() > self.expires_at

class Cache:
    """
    Thread-safe LRU cache with TTL support.
    Commands: GET, SET, DEL, TTL, KEYS, FLUSHALL
    """
    def __init__(self, max_size: int = 1000):
        self._max_size = max_size
        self._cache: OrderedDict[str, CacheEntry] = OrderedDict()
        self._lock  = threading.RLock()   # reentrant — same thread can re-acquire
        self._stats = {"hits": 0, "misses": 0, "evictions": 0}

    def get(self, key: str) -> Optional[Any]:
        with self._lock:
            entry = self._cache.get(key)
            if entry is None:
                self._stats["misses"] += 1; return None
            if entry.is_expired():
                del self._cache[key]
                self._stats["misses"] += 1; return None
            self._cache.move_to_end(key)   # promote to MRU
            self._stats["hits"] += 1
            return entry.value

    def set(self, key: str, value, ttl: Optional[float] = None):
        with self._lock:
            exp = time.time() + ttl if ttl else None
            self._cache[key] = CacheEntry(value, exp)
            self._cache.move_to_end(key)
            if len(self._cache) > self._max_size:
                evicted = self._cache.popitem(last=False)  # remove LRU
                self._stats["evictions"] += 1

    def delete(self, key: str) -> bool:
        with self._lock:
            return bool(self._cache.pop(key, None))

    def hit_rate(self) -> float:
        total = self._stats["hits"] + self._stats["misses"]
        return self._stats["hits"] / total if total > 0 else 0.0

DiceDB — Maximising Single-node Performance

DiceDB is Arpit Bhayani's open-source fork of Valkey (Redis fork) with two killer features: Query Subscriptions (reactive queries) and Multi-tiered Storage. Understanding DiceDB teaches you how to squeeze maximum performance from a single node before going distributed.

What DiceDB Adds Over Redis

  • Query Subscriptions — subscribe to a query, get pushed updates when the result changes. Like a reactive materialized view.
  • Multi-tiered Storage — hot data in RAM, warm data on NVMe SSD, cold data on object storage (S3). Transparent to client.
  • Multi-threaded command execution — Go goroutines instead of Redis's single thread. Scales across CPU cores.
  • RESP2/RESP3 compatible — drop-in Redis replacement for most use cases.

Why These Matter Architecturally

  • Query subscriptions eliminate polling — instead of GET counter every 100ms, subscribe once and receive push when it changes.
  • Multi-tier storage lets you have a 10 TB "hot" dataset on a single node — most data on cheaper storage, hot data in expensive RAM.
  • Multi-threading unlocks all CPU cores — a 32-core server runs 32× concurrent commands, not serialized on one thread.

Single-node Performance Maximisation

How Redis Maximises Single-node Throughput

1
I/O Multiplexing (epoll / kqueue)
A single thread monitors thousands of socket file descriptors simultaneously. When data is readable, it's processed inline. Zero thread context switching per request.
2
Simple, flat data structures
Redis uses int-encoded strings, ziplist (for small hashes/lists), and skiplist (for sorted sets). Each chosen to minimize memory and maximize CPU cache efficiency.
3
RESP protocol — minimal parsing overhead
Redis Serialization Protocol: simple text-based framing. Commands like *3\r\n$3\r\nSET\r\n$5\r\nhello\r\n$5\r\nworld\r\n. Minimal parsing — no JSON overhead.
4
Pipelining
Client sends multiple commands without waiting for responses. Server processes in batch. Reduces RTT overhead by N× for N pipelined commands.
5
Kernel bypass (DiceDB future)
io_uring on Linux lets the application bypass the kernel for I/O operations — submitting and completing I/O in userspace. 10–50% throughput improvement for I/O-heavy workloads.

Memory Optimization Techniques

Integer Encoding

Strings that are integers (e.g., "12345") are stored as 8-byte integers instead of string objects with pointer + length. Saves 16+ bytes per key.

Ziplist / Listpack

For small hashes and lists (<128 entries, <64 bytes per value), Redis uses a flat byte array instead of pointer-linked nodes. 4–10× more memory efficient.

Shared Objects

Integers 0–9999 are pre-allocated and shared. Multiple keys referencing the value "1" point to the same object — no duplication.

jemalloc

Redis uses jemalloc (not system malloc) for memory allocation. Better fragmentation characteristics, explicit arena control, and per-thread caching reduce allocation overhead.

DiceDB Multithreading Model

DiceDB's core innovation over Redis: running commands truly in parallel across CPU cores without sacrificing correctness.

The Challenge: Concurrency + Correctness

The problem with naive multi-threading

If two goroutines concurrently modify the same key: Thread A reads value 5, Thread B reads value 5. Thread A writes 6. Thread B writes 6. Expected: 7. Actual: 6 (lost update). Need serialization for conflicting operations.

DiceDB's approach: Key-level locking

Lock at the key level, not the entire store. Operations on different keys run in parallel. Operations on the same key are serialized. Hash key → lock shard. Lock only before modifying, release immediately after.

Go Goroutine per Command

DiceDB-style concurrent command execution (pseudocode)Python (Go-like)
# Simplified DiceDB command dispatcher
# Each incoming command gets its own goroutine
# Key-level sharded locks prevent data races

class DiceDBCore:
    def __init__(self):
        self.store      = {}                     # global key-value store
        self.key_locks  = ShardedLockManager(64)  # 64 key-level lock shards
        self.executor   = ThreadPoolExecutor(max_workers=os.cpu_count())

    def handle_command(self, conn, command: str, args: list):
        """Each command dispatched to thread pool — true parallelism."""
        future = self.executor.submit(self._execute, conn, command, args)
        future.add_done_callback(lambda f: conn.send_response(f.result()))

    def _execute(self, conn, command: str, args: list):
        key = args[0] if args else None
        with self.key_locks.lock_for(key):    # key-level lock, not global
            if   command == "GET": return self.store.get(key)
            elif command == "SET": self.store[key] = args[1]; return "OK"
            elif command == "INCR":
                self.store[key] = int(self.store.get(key, 0)) + 1
                return self.store[key]
Result: On a 32-core server, DiceDB can process 32 non-conflicting commands simultaneously. For a social network workload (random key distribution), conflict rate is ~1/N where N = unique keys → near-linear scaling with cores.
Query Subscriptions (QWATCH)

The most innovative DiceDB feature: subscribe to a query and receive push notifications when the result changes. Eliminates polling entirely.

The Problem QWATCH Solves

Without QWATCH (polling)

Client polls GET leaderboard:top10 every 100ms to detect changes. 10 clients × 10 polls/sec = 100 reads/sec even when nothing changes. Latency = poll interval (up to 100ms stale).

With QWATCH (reactive)

Client subscribes once: QWATCH "SELECT * FROM leaderboard ORDER BY score DESC LIMIT 10". Server pushes update only when the result set changes. Zero polling, real-time updates, zero wasted reads.

How QWATCH Works Internally

1
Client sends QWATCH with a query
QWATCH "SELECT * FROM users WHERE active=1 ORDER BY score DESC LIMIT 10". Server evaluates it immediately and sends current result.
2
Server parses the query → identifies dependent keys
Determines which keys/patterns can affect this query result. Registers interest in those keys.
3
Any SET/INCR/DEL on a dependent key → re-evaluate
If the new result differs from the last pushed result, push an update to the subscribed client. If result is unchanged, no push — smart deduplication.
4
Client receives updates as they happen
WebSocket or persistent connection. Latency = time for write to propagate → negligible. Client never polls.
Real-world use case: Leaderboard. Instead of 1000 players polling the top-10 every second (1000 reads/sec), they all subscribe once. Every score update triggers one re-evaluation and one push to all subscribers. 1000× less read load.
Multi-tiered Storage

DiceDB's multi-tier storage lets a single node hold 10× more data than would fit in RAM by transparently spilling to faster/cheaper storage tiers.

Eviction & Promotion

Eviction (RAM → SSD)

When RAM exceeds threshold, LRU keys are serialized and written to RocksDB on NVMe. A "spill" pointer replaces the value in RAM. TTLs are preserved — keys still expire correctly even on SSD tier.

Promotion (SSD → RAM)

On access, if key is on SSD, fetch from RocksDB into RAM (~100µs). If key is on S3, fetch from object storage into RAM (~10ms). Transparent to client — slightly higher latency on first access.

Business impact: A workload with 1 TB total dataset but 10 GB working set: with Redis (RAM only) = $5,000/TB RAM. With DiceDB multi-tier = $5/GB × 10 GB + $0.10/GB × 1 TB = $50 + $100 = $150. 33× cost reduction.

Consistent Hashing — Operational Complexities

We covered consistent hashing in Week 3. Here we go deeper: the operational challenges of adding/removing nodes, rebalancing data, handling failures, and the real-world nuances that make it harder than theory suggests.

The Core Problem with Modular Hashing

Standard: key % N

Add 1 node to a cluster of N=5 → N becomes 6 → all keys rehash. Every key potentially moves to a different node. During rehash: massive data migration, cache cold start, potential downtime.

At 1 TB data, adding one node = migrate ~833 GB. Days of rebalancing. Revenue impact.

Consistent Hashing

Add 1 node → only the keys between the new node and its predecessor on the ring move. With N nodes, adding one moves approximately K/N keys (K = total keys). At 5 nodes: only ~20% of data moves, not 100%.

At 1 TB data with 5 nodes: adding one node = migrate ~200 GB. 4× less disruption.

The Ring

Hash the key space (0 to 2³² or 2⁶⁴) into a circular ring. Hash each server to a position on the ring. A key belongs to the first server clockwise from its hash position.

Virtual Nodes

A single server mapped to one ring position creates uneven distribution. With 4 servers at random positions, one might own 40% of the ring and another only 10%. Virtual nodes fix this.

The Skew Problem

Without Virtual Nodes

4 servers hashed to random positions. Server B happens to own a large arc (40% of ring). Server A owns a small arc (10%). Load is uneven by 4×. Adding a new server helps the neighbor, but skew remains.

With Virtual Nodes (vnodes)

Each physical server maps to 150 ring positions (virtual nodes). Server A owns positions: hash("S-A-0"), hash("S-A-1"), ..., hash("S-A-149"). Load distribution converges to uniform. Each server owns ~1/N of the ring.

How Many Virtual Nodes?

Cassandra uses 256 vnodes by default (configurable). Redis Cluster uses 16,384 hash slots pre-assigned to nodes — a discrete ring with fixed granularity rather than continuous consistent hashing with vnodes.
Operational Complexities

The theory is elegant. Production operation is hard. Here's what actually happens when you add/remove nodes from a consistent hash ring.

Adding a Node

1
New node joins the ring at position P
It becomes responsible for all keys between its predecessor and P. These keys currently live on the successor node.
2
Migrate data from successor → new node
Identify all keys in the new node's range on the successor. Copy them to the new node. During copy: both nodes serve reads from their copy (dual read phase).
3
Switch ring to route to new node
Update the ring so clients route to the new node. Old keys on successor will be accessed at new node going forward.
4
Delete migrated keys from successor
Once new node is serving traffic, delete the migrated range from successor. Free the memory. Verify no requests are still going to old node.

Real Operational Challenges

Split-brain during migration

While migrating: a write to key K arrives during migration. Which node gets it — old or new? Solution: "migration lock" on the key range being migrated. Writes queue; drain after node switch. Complex to implement correctly.

Hotspots during migration

The successor node serves its normal load + migration read I/O simultaneously. This can push it to CPU/disk saturation. Solution: rate-limit the migration stream. Accepts slower migration in exchange for stable production traffic.

Replication factor

In production, each key has replication factor RF=3 (Cassandra, Redis Cluster). A new node must receive not just the primary copy but also become a replica for adjacent ranges. Data movement = 3× naive estimate.

Client routing updates

Clients must know the updated ring configuration. During transition, old clients may route to wrong node. Solution: nodes redirect misrouted requests (MOVED error in Redis Cluster), or use a proxy layer. Gossip protocol propagates ring changes.

Operational reality: A full cluster rebalance (adding 1 node to 5-node cluster with 1 TB data) typically takes 4–8 hours with rate limiting. During this window, monitoring, manual verification of key distribution, and rollback planning are essential. Never do this during peak traffic.
Code — Consistent Hash Ring
consistent_hash.py — ring with virtual nodesPython
import hashlib, bisect
from collections import defaultdict

class ConsistentHashRing:
    """
    Consistent hash ring with virtual nodes.
    Thread-safe for concurrent reads (ring is immutable between changes).
    """
    def __init__(self, vnodes_per_server: int = 150):
        self.vnodes_per_server = vnodes_per_server
        self._ring:   dict[int, str] = {}    # hash_position → server_id
        self._sorted: list[int]      = []    # sorted hash positions (for bisect)
        self._servers: set[str]       = set()

    def _hash(self, key: str) -> int:
        """Consistent 32-bit hash."""
        return int(hashlib.md5(key.encode()).hexdigest(), 16) % (2 ** 32)

    def add_server(self, server_id: str) -> list[tuple]:
        """Add server. Returns list of (start, end) hash ranges now owned by it."""
        self._servers.add(server_id)
        for i in range(self.vnodes_per_server):
            pos = self._hash(f"{server_id}#{i}")
            self._ring[pos] = server_id
            bisect.insort(self._sorted, pos)   # O(log N) insertion

    def remove_server(self, server_id: str):
        """Remove server and its vnodes from ring."""
        self._servers.discard(server_id)
        positions_to_remove = [pos for pos, srv in self._ring.items() if srv == server_id]
        for pos in positions_to_remove:
            del self._ring[pos]
            idx = bisect.bisect_left(self._sorted, pos)
            del self._sorted[idx]

    def get_server(self, key: str) -> str:
        """Route key to its responsible server — O(log N)."""
        if not self._ring:
            raise ValueError("No servers in ring")
        key_hash = self._hash(key)
        idx = bisect.bisect(self._sorted, key_hash) % len(self._sorted)
        pos = self._sorted[idx]
        return self._ring[pos]

    def get_servers_for_replication(self, key: str, rf: int = 3) -> list[str]:
        """Return RF unique servers for replication (walk ring clockwise)."""
        key_hash = self._hash(key)
        start    = bisect.bisect(self._sorted, key_hash) % len(self._sorted)
        servers, seen = [], set()
        for i in range(len(self._sorted)):
            pos = self._sorted[(start + i) % len(self._sorted)]
            srv = self._ring[pos]
            if srv not in seen:
                servers.append(srv); seen.add(srv)
            if len(servers) == rf: break
        return servers

No-downtime Data Migration

Moving data from one storage system to another (schema change, engine change, shard rebalance) while production traffic continues — without losing writes or serving stale data. This is one of the hardest operational problems in engineering.

The Problem

  • Old system has existing data. New system is empty.
  • Traffic continues during migration — new writes arrive constantly.
  • You can't stop writes ("maintenance window") at scale — 5 min downtime for 100M users = headlines.
  • You can't read from new system until it has all data — reads would miss migrated keys.
  • Migration itself takes hours/days — it's not atomic.

The Requirements

  • Zero downtime — no service interruption
  • Zero data loss — no write dropped
  • Correctness — reads return consistent, up-to-date data
  • Rollbackable — must be able to abort and revert to old system
  • Progressive — cut over a small % of traffic first, verify, then increase

The Dual-write Pattern

The canonical strategy for zero-downtime migration: write to both old and new systems simultaneously, then gradually shift reads to the new system.

5-Phase Migration Protocol

1
Phase 1: Shadow write (both systems)
Enable dual-write: every write goes to OLD (primary) AND NEW (shadow). Reads still come from OLD. NEW builds up current state. No user impact.
2
Phase 2: Backfill historical data
Bulk copy all existing data from OLD to NEW. Rate-limited to avoid overloading OLD. NEW now has: backfill + live writes. Duration: hours to days.
3
Phase 3: Verify (shadow read)
For a sample of reads, query BOTH old and new. Compare results. Alert on divergence. Fix discrepancies. Only proceed if match rate ≥ 99.9%.
4
Phase 4: Gradual read cutover
Route 1% → 5% → 25% → 50% → 100% of reads to NEW. Monitor error rate and latency at each step. Roll back immediately if anomalies detected.
5
Phase 5: Decommission OLD
After 100% reads on NEW and verified stability for 72h+: stop dual-writes to OLD. Delete OLD data. Done. Total migration: zero downtime, zero data loss.
This is the pattern Facebook, Google, and Stripe use for every major storage migration. The key insight: reads and writes are migrated independently, with reads always lagging writes to ensure the new system is always at least as fresh as the old.

Blue-Green Deployment

Pattern

Maintain two identical environments: Blue (live) and Green (idle). Deploy new version to Green. Shift traffic via LB. Blue becomes standby. Rollback = redirect LB back to Blue. Zero-downtime, instant rollback.

Database Challenge

Blue-green works for stateless app servers. For databases: both Blue and Green share the same DB. Schema changes must be backward-compatible — app v1 and v2 must both work against the same schema during cutover window.

Expand-Contract (Parallel Change) Pattern

E
Expand: add new column/table without removing old
App writes to BOTH old and new columns. Reads from OLD. Backward compatible — v1 of app ignores new column.
M
Migrate: backfill old data into new structure
Background job copies old column values to new column. Verify completeness.
C
Contract: switch reads to new column, drop old
App v2 reads from new column. After all instances updated, drop old column. Atomic, safe.

dual_write.py — shadow write + gradual read cutoverPython
import asyncio, random, logging
from enum import Enum

class MigrationPhase(Enum):
    OLD_ONLY        = "old_only"         # reads + writes to OLD
    DUAL_WRITE      = "dual_write"       # writes to both, reads from OLD
    SHADOW_READ     = "shadow_read"      # dual write + compare reads
    GRADUAL_CUTOVER = "gradual_cutover"  # % of reads from NEW
    NEW_ONLY        = "new_only"         # all traffic to NEW

class MigrationProxy:
    def __init__(self, old_store, new_store):
        self.old           = old_store
        self.new           = new_store
        self.phase         = MigrationPhase.OLD_ONLY
        self.new_read_pct  = 0     # 0–100: % of reads from NEW
        self.divergence_cnt = 0   # count of read mismatches

    async def get(self, key: str):
        if self.phase == MigrationPhase.OLD_ONLY:
            return await self.old.get(key)

        if self.phase == MigrationPhase.SHADOW_READ:
            # Read from both, compare, return OLD result
            old_val, new_val = await asyncio.gather(
                self.old.get(key), self.new.get(key)
            )
            if old_val != new_val:
                self.divergence_cnt += 1
                logging.warning(f"Divergence on {key}: old={old_val} new={new_val}")
            return old_val   # always return OLD during shadow phase

        if self.phase == MigrationPhase.GRADUAL_CUTOVER:
            # Route new_read_pct% of reads to NEW
            if random.randint(1, 100) <= self.new_read_pct:
                return await self.new.get(key)
            return await self.old.get(key)

        return await self.new.get(key)   # NEW_ONLY phase

    async def set(self, key: str, value):
        if self.phase == MigrationPhase.OLD_ONLY:
            return await self.old.set(key, value)

        if self.phase == MigrationPhase.NEW_ONLY:
            return await self.new.set(key, value)

        # Dual-write phases: write to both, return OLD result
        old_res, new_res = await asyncio.gather(
            self.old.set(key, value),
            self.new.set(key, value)
        )
        return old_res   # old is authoritative during transition

    def advance_phase(self, phase: MigrationPhase, new_read_pct: int = 0):
        self.phase = phase
        self.new_read_pct = new_read_pct
        logging.info(f"Migration phase → {phase.value} ({new_read_pct}% new reads)")
Outcome
  • Dual-write is the foundation of every live migration — writes to both systems, reads from old, gradually cut reads over to new. This decouples write migration from read migration, enabling safe progressive rollout.
  • Shadow reads are the validation gate — never advance to gradual cutover until shadow reads show <0.1% divergence. Divergences reveal bugs in data transformation logic before they affect users.
  • gh-ost makes online schema changes possible — creates shadow table, copies rows, tails binlog for live changes, then performs an atomic swap. No table-level locks, no downtime for even billion-row tables.
  • Expand-Contract eliminates the need for synchronized deploys — add new column before switching logic, drop old column after. Both v1 and v2 of the app work during the transition window.
  • Rollback must be planned before starting migration — define the rollback procedure (re-enable old reads, stop dual-write) and test it in staging. A migration without a practiced rollback is an incident waiting to happen.

Design a Word Dictionary Without a DB

Store 500K English words with support for: exact lookup (is "serendipity" a valid word?), prefix search (all words starting with "ser"), definition storage, and offline operation. No database allowed.

Requirements

  • Exact lookup: O(L) where L = word length
  • Prefix search: return all words with a given prefix
  • Persistence: survive restarts without a database
  • Efficient storage: 500K words × avg 10 chars = 5 MB minimum
  • Definition storage: arbitrary text per word
  • Spell check: is this word valid?

Why No DB?

  • Forces you to understand data structure tradeoffs from first principles
  • Teaches disk-based data structure design
  • Applicable to: embedded devices, edge computing, CLI tools
  • The techniques used here are the same ones real DBs are built on

The Trie — Core Data Structure

A trie (prefix tree) stores words character by character. Each node represents one character; paths from root to marked nodes represent words. Perfect for prefix search.

Properties

  • Exact lookup: O(L) — L = word length. Just traverse character by character.
  • Prefix search: O(P + K) — P = prefix length, K = results. Find prefix node, then DFS.
  • Memory: shared prefixes stored once. "sea", "seal", "search" share "sea" nodes.
  • 26 children per node (English alphabet) or 128 (ASCII)

Drawbacks

  • Memory: each node = 26 pointers × 8 bytes + metadata = ~200 bytes. 500K words × avg depth 8 = millions of nodes = GBs in worst case.
  • Fix: Compressed Trie (Patricia/Radix trie) — merge single-child nodes into edges. Reduces node count by 10×.
Bloom Filter for Spell Check

For "is this word valid?" queries, a Bloom filter gives a probabilistic answer in O(k) time with O(n × k / ln2) bits of memory — vastly less than a trie or hash set.

Bloom Filter Properties

How it works

  • Bit array of size m, initialized to 0
  • k hash functions, each maps a word to one bit position
  • Add word: set bits at all k hash positions to 1
  • Query word: if ALL k positions = 1 → "possibly in set" (no false negatives)
  • Query word: if ANY position = 0 → "definitely not in set" (no false positives)

For 500K words, 1% false positive rate

  • Optimal bit array size: m = -n × ln(p) / (ln2)² = ~5 MB
  • Optimal hash functions: k = m/n × ln2 = 7 hash functions
  • False positive rate: ~1% (1 in 100 non-words passes)
  • For spell check: acceptable! Users see "not a word" → check trie for confirmation

Two-level Lookup Strategy

Optimal pattern for spell check:
1. Bloom filter query → O(k=7) hash ops, ~5 MB memory
2. If Bloom says NO → return "invalid word" immediately (definitely not in dictionary)
3. If Bloom says MAYBE → confirm with Trie/HashSet lookup
This eliminates 99%+ of full trie lookups (most random strings fail the Bloom filter).
bloom_filter.py — word dictionary spell checkPython
import hashlib, math

class BloomFilter:
    def __init__(self, n_words: int, fp_rate: float = 0.01):
        self.m = int(-n_words * math.log(fp_rate) / (math.log(2))**2)  # bits
        self.k = int(self.m / n_words * math.log(2))               # hash fns
        self._bits = bytearray(self.m // 8 + 1)

    def _hashes(self, word: str) -> list[int]:
        """Generate k independent hash positions using double hashing."""
        h1 = int(hashlib.md5(word.encode()).hexdigest(), 16)
        h2 = int(hashlib.sha1(word.encode()).hexdigest(), 16)
        return [(h1 + i * h2) % self.m for i in range(self.k)]

    def add(self, word: str):
        for pos in self._hashes(word):
            self._bits[pos // 8] |= (1 << (pos % 8))

    def contains(self, word: str) -> bool:
        """True = possibly in set. False = definitely not in set."""
        return all(
            self._bits[pos // 8] & (1 << (pos % 8))
            for pos in self._hashes(word)
        )
Disk Storage — No Database

How do you persist 500K words with definitions on disk without a database? Build a simple file-based storage with an index — the same approach real databases use internally.

Approach 1: Sorted Flat File + Binary Search

Store all words alphabetically in a flat file. Each line: word\tdefinition\n. For lookup: binary search on the file using file offsets — O(log N × disk seek) time.

flat_file_dict.py — binary search on sorted word filePython
class FlatFileDictionary:
    """Words stored sorted in flat file. Binary search by file offset."""
    def __init__(self, filepath: str):
        self.filepath = filepath
        self._build_index()

    def _build_index(self):
        """Build sparse index: every 1000th word → file offset."""
        self.index = []   # [(word, file_offset)]
        with open(self.filepath, 'rb') as f:
            offset, line_num = 0, 0
            for line in f:
                if line_num % 1000 == 0:
                    word = line.split(b'\t')[0].decode()
                    self.index.append((word, offset))
                offset += len(line); line_num += 1
        self.total_words = line_num

    def lookup(self, word: str) -> str | None:
        """O(log N + page_size) — sparse index + sequential scan in block."""
        # Binary search in sparse index to find nearest offset
        lo, hi = 0, len(self.index) - 1
        while lo < hi - 1:
            mid = (lo + hi) // 2
            if self.index[mid][0] <= word: lo = mid
            else: hi = mid
        start_offset = self.index[lo][1]

        # Sequential scan from nearest index entry
        with open(self.filepath, 'rb') as f:
            f.seek(start_offset)
            for line in f:
                parts = line.decode().strip().split('\t', 1)
                if parts[0] == word: return parts[1] if len(parts) > 1 else ""
                if parts[0] > word: return None   # gone past — word not found
        return None

from dataclasses import dataclass, field
from typing import Optional

@dataclass
class TrieNode:
    children: dict = field(default_factory=dict)
    is_word:  bool = False
    def_offset: Optional[int] = None   # byte offset in definition file

class WordDictionary:
    """
    Three-layer word lookup:
    1. Bloom filter  → O(k) quick existence check (reject obviously invalid)
    2. Trie          → O(L) exact lookup + prefix search
    3. Definition file → O(1) definition fetch by offset
    """
    def __init__(self):
        self.root      = TrieNode()
        self.bloom     = BloomFilter(500_000, fp_rate=0.001)  # 0.1% FP
        self.def_file  = None

    def add_word(self, word: str, definition: str, def_offset: int):
        self.bloom.add(word)
        node = self.root
        for ch in word.lower():
            node = node.children.setdefault(ch, TrieNode())
        node.is_word    = True
        node.def_offset = def_offset

    def search(self, word: str) -> Optional[str]:
        """Fast two-layer lookup: Bloom → Trie → definition file."""
        word = word.lower()
        if not self.bloom.contains(word):
            return None   # definitely not a word
        node = self.root
        for ch in word:
            node = node.children.get(ch)
            if not node: return None
        if not node.is_word: return None
        # Fetch definition from flat file
        return self._fetch_definition(node.def_offset)

    def prefix_search(self, prefix: str) -> list[str]:
        """All words with given prefix — DFS from prefix node."""
        node = self.root
        for ch in prefix.lower():
            node = node.children.get(ch)
            if not node: return []
        results = []
        self._dfs(node, list(prefix), results)
        return results

    def _dfs(self, node, current, results, max_results=20):
        if node.is_word:
            results.append("".join(current))
        if len(results) >= max_results: return
        for ch, child in sorted(node.children.items()):
            self._dfs(child, current + [ch], results, max_results)

Log-Structured KV Store (LSM Tree)

The storage engine behind RocksDB, LevelDB, Cassandra, HBase, and ScyllaDB. LSM trees trade read amplification for dramatically better write throughput by converting random writes into sequential disk I/O.

Why Not B-Trees for Write-heavy Workloads?

B-Tree (InnoDB, PostgreSQL)

  • Updates modify data in-place → random writes to disk pages
  • Random write = seek time + latency on HDD, write amplification on SSD
  • Page splits on insert → read-modify-write the page → expensive
  • Write throughput: ~50K–100K rows/sec per node
  • ✅ Excellent for reads (O(log N)), range scans

LSM Tree (RocksDB, Cassandra)

  • All writes go to an in-memory buffer (MemTable) → sequential disk flush
  • Sequential writes = no seek time, full disk bandwidth utilization
  • Disk writes are immutable (append-only) → no in-place modification
  • Write throughput: 500K–5M rows/sec per node
  • ❌ Reads may check multiple SSTables → read amplification
Write Path

Every write in an LSM tree is a sequential operation — never modifies data in place. This is the fundamental insight that makes write throughput so much higher than B-trees.

Write Flow

1
Write to WAL (Write-Ahead Log)
Before modifying memory, append the operation to the WAL on disk. Sequential append = fast. If crash: replay WAL to recover MemTable. Guarantees durability.
2
Write to MemTable (in-memory sorted structure)
Insert key-value pair into MemTable — an in-memory balanced BST (Red-Black tree) or Skip List. Maintained in sorted key order. O(log M) insert where M = MemTable size.
3
When MemTable is full → flush to SSTable
MemTable becomes immutable. A background thread writes it to disk as an SSTable (Sorted String Table). Keys already sorted in memory → one sequential write to disk = maximum I/O throughput.
4
New MemTable starts for incoming writes
While flushing, a new empty MemTable accepts writes. No blocking. Flush is fully asynchronous.
5
Delete = tombstone write
No key is ever deleted in place. Instead, a tombstone marker (key → DELETED) is written. Old versions persist until compaction removes them.

SSTable Format

Read Path

The price of LSM's write performance: reads must potentially check multiple SSTables. Multiple layers of optimization reduce this cost dramatically.

Read Algorithm

1
Check MemTable (newest data, O(log M))
Key found in MemTable → return immediately. This is the common case for recently written keys.
2
Check Bloom filter for each SSTable (newest → oldest)
For each SSTable, the Bloom filter answers: "Is this key possibly in this file?" If NO → skip entire file. If YES → check index. Bloom filter eliminates 99%+ of unnecessary disk reads.
3
Check SSTable range (min/max key in footer)
SSTable stores min and max key in metadata. If the target key is outside [min, max] → skip this SSTable entirely. Additional pruning before disk I/O.
4
Binary search index block → fetch data block
Sparse index → find the data block containing the key. Read that block from disk. Binary search within block. O(log N) disk operations.
5
Block cache
Frequently accessed data blocks are cached in a block cache (LRU). Hot data serves reads in microseconds. Cold data requires disk I/O.

Read Amplification

Worst case: Key doesn't exist. Must check every SSTable (Bloom filter + range check). With 100 SSTables: 100 Bloom filter lookups (fast, in memory) + potentially many disk reads if Bloom has false positives. This is why compaction is essential — it reduces SSTable count.
With leveled compaction (RocksDB): At each level, SSTables have non-overlapping key ranges. A read at Level N only checks ONE SSTable per level. Total disk reads = number of levels (typically 5–7). Read amplification bounded to O(levels).
Compaction

Compaction merges multiple SSTables, removes obsolete versions, removes tombstones, and reduces the number of SSTables — trading write amplification for better read performance.

Why Compaction is Necessary

Without compaction: 1000 SSTables after 1M writes. Every read checks 1000 files. Space is wasted on old versions and tombstones. Compaction is the background "cleanup" that keeps LSM trees efficient.

Compaction Strategies

Size-Tiered (STCS)

Group SSTables of similar size, merge them. New merged SSTable goes to next tier. Multiple SSTables per level → some key range overlap.

  • ✅ Lower write amplification (good for writes)
  • ❌ Higher read amplification (overlapping key ranges)
  • Used by: Cassandra (default), ScyllaDB

Leveled (LCS) ✓ RocksDB default

Each level has one merged SSTable with non-overlapping key ranges. A key exists in exactly one file per level. Guarantees O(levels) read amplification.

  • ✅ Lower read amplification
  • ❌ Higher write amplification (more merging)
  • Used by: RocksDB, LevelDB, TiKV

FIFO (time-series)

Oldest SSTables deleted when capacity reached. No merging. For time-series data where old data is simply expired.

  • ✅ Lowest write amplification
  • ❌ No deduplication
  • Used by: InfluxDB, Prometheus (tsdb)

Write Amplification

The dark side of compaction: A single user write may be physically written to disk many times as it gets compacted through levels. With 5 levels and 10× size ratio: write amplification = 1 (WAL) + 1 (L0 flush) + 10 (L0→L1) + 10 (L1→L2) + ... ≈ 30–50× write amplification for leveled compaction. This accelerates SSD wear. RocksDB tuning is largely about controlling this trade-off.

lsm_tree.py — MemTable + SSTable + compactionPython
import os, json, time, bisect
from sortedcontainers import SortedDict  # Red-Black tree equivalent

TOMBSTONE   = "__DELETED__"
MEMTABLE_MAX = 1000   # flush when MemTable hits this size

class SSTable:
    """Immutable sorted file on disk."""
    def __init__(self, filepath: str, data: SortedDict):
        self.filepath = filepath
        self.min_key  = data.peekitem(0)[0]    # for range pruning
        self.max_key  = data.peekitem(-1)[0]
        self._bloom   = BloomFilter(len(data))
        # Write sorted data to disk
        with open(filepath, "w") as f:
            for key, val in data.items():
                f.write(json.dumps({"k": key, "v": val}) + "\n")
                self._bloom.add(key)

    def get(self, key: str):
        if key < self.min_key or key > self.max_key: return None
        if not self._bloom.contains(key): return None   # Bloom filter skip
        with open(self.filepath) as f:           # linear scan (simplified)
            for line in f:
                entry = json.loads(line)
                if entry["k"] == key: return entry["v"]
                if entry["k"]  > key: return None
        return None

class LSMTree:
    def __init__(self, data_dir: str = "/tmp/lsm"):
        os.makedirs(data_dir, exist_ok=True)
        self.data_dir  = data_dir
        self.memtable  = SortedDict()   # active MemTable
        self.sstables  = []              # newest first
        self._wal      = open(f"{data_dir}/wal.log", "a")

    def set(self, key: str, value: str):
        self._wal.write(json.dumps({"op":"set","k":key,"v":value}) + "\n")
        self._wal.flush()   # WAL must be flushed before MemTable update
        self.memtable[key] = value
        if len(self.memtable) >= MEMTABLE_MAX:
            self._flush_memtable()

    def delete(self, key: str):
        self.set(key, TOMBSTONE)   # deletion = tombstone write

    def get(self, key: str):
        # 1. Check MemTable (most recent)
        val = self.memtable.get(key)
        if val is not None:
            return None if val == TOMBSTONE else val
        # 2. Check SSTables newest → oldest
        for sst in self.sstables:
            val = sst.get(key)
            if val is not None:
                return None if val == TOMBSTONE else val
        return None   # key not found

    def _flush_memtable(self):
        """Flush MemTable to disk as SSTable (sequential write)."""
        path = f"{self.data_dir}/sst_{int(time.time()*1000)}.sst"
        sst  = SSTable(path, self.memtable)
        self.sstables.insert(0, sst)     # newest first
        self.memtable = SortedDict()     # fresh MemTable
        if len(self.sstables) > 4:       # compact when too many files
            self._compact()

Operations Log

  • All writes are sequential appends — WAL first (durability), then MemTable (speed). When MemTable is full, one sequential disk write creates an SSTable. No random I/O until compaction.
  • Bloom filters prevent 99%+ of unnecessary disk reads — for a key not in an SSTable, the Bloom filter answers "definitely not" in O(k) hash operations, skipping the disk read entirely.
  • Tombstones are the deletion mechanism — never delete in place. Write a special marker. Old versions and tombstones persist until compaction sweeps them away. Critical for understanding "deleted" keys reappearing after crash-recovery.
  • Leveled compaction bounds read amplification to O(levels) — non-overlapping key ranges at each level mean at most one SSTable per level is checked. Size-tiered compaction is faster to write but slower to read.
  • Write amplification is the LSM tax — data is written once (WAL), once more (L0 flush), then rewritten at each compaction level. Total amplification = 30–50× for leveled compaction. SSD endurance is the practical constraint.
  • Used in production everywhere — RocksDB (Facebook/Meta, Kafka, TiKV), LevelDB (Chrome), Cassandra, HBase, ScyllaDB. The LSM tree is the dominant storage engine for write-heavy distributed systems.
