GitHub's File Sync Algorithm

Remote file sync (Dropbox, GitHub, Google Drive) must efficiently synchronise files across devices: only sending changes (deltas), not the entire file. For a 1 GB file with a 1 KB change, transferring 1 KB not 1 GB. This is the core engineering challenge of every file sync system.

The Naive Approach — Why It Fails

Send whole file every time

Simple but wasteful. Edit one line in a 10 GB video file → upload 10 GB. For large binary files (designs, videos, databases), this is completely impractical. Wastes bandwidth, time, and money.

Send only diff (line-by-line)

Works beautifully for text files (Unix diff). But binary files have no line concept. Moving 1 byte at position 0 shifts every subsequent byte → the entire file looks "different" even though only one byte changed.

The rsync Algorithm — Elegant Solution

rsync (1996, Andrew Tridgell) solves the binary file delta problem with rolling checksums. The core insight: find which chunks of the old file still exist in the new file, regardless of their position.

1
Remote has OLD file: split into fixed-size chunks (e.g., 4 KB)
For each chunk: compute weak checksum (Adler-32, 4 bytes) and strong checksum (MD5, 16 bytes). Send these checksums to the local sender — very small overhead (~5 bytes per 4 KB chunk = 0.12%).
2
Local sender has NEW file: rolling window search
For every byte offset in the NEW file, compute weak checksum for the next 4 KB. If it matches a remote chunk's weak checksum → verify with strong checksum (MD5). If both match → this chunk is already on remote — no need to send!
3
Build delta: instructions for remote to reconstruct new file
Delta = sequence of: COPY(chunk_id) — use existing chunk from remote. LITERAL(bytes) — these bytes are new, send them. Only the LITERAL portions transfer over the network.
4
Remote reconstructs new file from delta
Apply instructions: for each COPY → read chunk from old file. For each LITERAL → use the sent bytes. Result: exact copy of new file with minimal data transfer.

Rolling Checksum — The Key Insight

Computing checksum for every 4 KB window (1 byte step) would be O(N × chunk_size) = very slow. A rolling checksum (Adler-32) can be updated in O(1) when the window slides one byte: new_checksum = (old_checksum - old_byte_contribution + new_byte_contribution) % modulus. This makes the entire file scan O(N).

Delta Sync & Chunking Strategies

Fixed-size vs Content-defined Chunking

Fixed-size Chunks (rsync classic)

  • Split file into chunks of exactly N bytes (e.g., 4 KB)
  • Inserting 1 byte shifts ALL subsequent chunks
  • Example: insert "X" at position 0 of 1 GB file → all 256K chunks are now "different"
  • ❌ Bad for files with frequent insertions (text, code)
  • ✅ Simple, fast

Content-Defined Chunking (CDC) ✓

  • Chunk boundaries determined by content, not position
  • Rolling hash: when hash(window) ends in N zeros → chunk boundary
  • Inserting 1 byte only changes the chunks containing/near the change
  • ✅ Used by Dropbox, Bup, restic, Perkeep
  • Average chunk size tunable (Dropbox: 4 MB avg, min 512 KB, max 16 MB)

Content-Defined Chunking Algorithm (Rabin Fingerprinting)

cdc.py — content-defined chunking with Rabin fingerprintPython
class RabinChunker:
    """
    Content-defined chunking using Rabin fingerprinting.
    Chunk boundaries at positions where rolling hash has N trailing zeros.
    Average chunk size = 2^N bytes.
    """
    WINDOW_SIZE = 48
    MASK        = (1 << 22) - 1   # target avg chunk size ~4 MB = 2^22 bytes
    MIN_CHUNK   = 512  * 1024     # 512 KB minimum
    MAX_CHUNK   = 16   * 1024**2  # 16 MB maximum

    def chunk(self, data: bytes) -> list[bytes]:
        """Split data at content-defined boundaries. Returns list of chunks."""
        chunks  = []
        start   = 0
        fp      = 0   # rolling Rabin fingerprint
        window  = bytearray(self.WINDOW_SIZE)

        for i, byte in enumerate(data):
            # Update rolling fingerprint (simplified)
            oldest  = window[i % self.WINDOW_SIZE]
            fp      = (fp * 31 + byte - oldest * (31 ** self.WINDOW_SIZE)) % (2**32)
            window[i % self.WINDOW_SIZE] = byte

            chunk_size = i - start + 1
            if chunk_size < self.MIN_CHUNK: continue

            # Boundary: hash ends in MASK zeros OR chunk hit max size
            if (fp & self.MASK) == 0 or chunk_size >= self.MAX_CHUNK:
                chunks.append(data[start:i+1])
                start = i + 1
                fp    = 0

        if start < len(data):
            chunks.append(data[start:])   # last chunk
        return chunks

Deduplication via Content-Addressing

Content-addressing: Each chunk is identified by SHA-256(chunk_content). If two files have an identical chunk, it's stored once. Dropbox achieved 70%+ deduplication for typical user files. Large benefits for: office templates, firmware updates, video segments with common frames.
Conflict Resolution

When two users (or devices) modify the same file concurrently, their changes must be merged or conflicts detected and surfaced. This is one of the hardest problems in distributed file sync.

Three-way Merge

B
Base version — last common ancestor
The version both devices had before they diverged. Stored in the sync server. This is the "origin" of the branch.
A
Version A — what Device 1 has
Changes made on laptop since last sync. Diff(Base, A) = what Device 1 added/removed.
M
Merge: apply A's changes to B, then B's changes to A
If A changed line 5 and B changed line 20: merge is unambiguous. If A changed line 5 AND B also changed line 5: CONFLICT — human must resolve.

Conflict Strategies

Last-write-wins (LWW)

Whoever saved last overwrites. Simple. May silently discard work. Used by: Dropbox (saves conflicted copy), Google Docs (reverts one).

Operational Transformation (OT)

Real-time collaborative editing. Each operation transformed relative to concurrent operations. Used by: Google Docs, Notion, Figma. Complex but enables real-time collaboration without conflicts.

CRDTs

Conflict-free Replicated Data Types. Math guarantees that all replicas converge to the same state regardless of operation order. Used by: Figma multiplayer, Notion (new), Linear. Works for specific data types (counters, sets, sequences).

Dropbox's Conflict Copy Pattern

Dropbox approach (pragmatic): When a conflict is detected, rather than attempting complex merge: rename the conflicted version to filename (User's conflicted copy 2024-03-26).ext and save it alongside the winner. The user sees both copies and can manually resolve. Simple, never loses data, user-visible. Most sync conflicts are due to accidental concurrent edits, not genuine merge scenarios.
Architecture

Merkle Trees for Directory Sync

To sync a directory tree, you need to efficiently determine which files changed. Comparing every file's hash is O(N) over the network. A Merkle tree reduces this to O(log N) round trips.

Merkle Tree Structure

  • Leaf node = SHA-256(file_content)
  • Interior node = SHA-256(concat of children hashes)
  • Root hash = fingerprint of entire directory tree
  • If root hashes match → nothing changed. Done. 1 comparison.
  • If different → compare children to find the changed subtree
  • O(log N) comparisons to locate any changed file in a tree of N files

Why This Matters

  • 1M files in repo: without Merkle = 1M hash comparisons per sync
  • With Merkle tree = log₂(1M) ≈ 20 comparisons per sync
  • 50,000× fewer comparisons to locate changes
  • Git uses this: every commit has a root Merkle tree hash
  • Two commits with same root hash = identical file trees
Code — Delta Sync Implementation
delta_sync.py — rsync-style delta computationPython
import hashlib, struct
from dataclasses import dataclass

CHUNK_SIZE = 4096   # 4 KB chunks

@dataclass
class ChunkSignature:
    index:      int
    weak_hash:  int   # Adler-32 rolling checksum
    strong_hash: str  # MD5/SHA-256 of chunk content

def adler32(data: bytes) -> int:
    """Adler-32: fast rolling checksum. Supports O(1) window slide."""
    MOD, s1, s2 = 65521, 1, 0
    for b in data: s1 = (s1 + b) % MOD; s2 = (s2 + s1) % MOD
    return (s2 << 16) | s1

def build_signatures(old_data: bytes) -> dict[int, ChunkSignature]:
    """Build chunk signatures for old file. Sent to remote sender."""
    sigs = {}
    for i in range(0, len(old_data), CHUNK_SIZE):
        chunk = old_data[i : i + CHUNK_SIZE]
        weak  = adler32(chunk)
        strong = hashlib.md5(chunk).hexdigest()
        sigs[weak] = ChunkSignature(i // CHUNK_SIZE, weak, strong)
    return sigs

def compute_delta(old_sigs: dict, new_data: bytes) -> list:
    """
    Find which parts of new_data exist in old_data (by chunk matching).
    Return delta: list of COPY(chunk_idx) or LITERAL(bytes) instructions.
    """
    delta, i, unmatched_start = [], 0, 0
    while i <= len(new_data) - CHUNK_SIZE:
        chunk = new_data[i : i + CHUNK_SIZE]
        weak  = adler32(chunk)

        if weak in old_sigs:
            sig = old_sigs[weak]
            strong = hashlib.md5(chunk).hexdigest()
            if strong == sig.strong_hash:   # chunk match confirmed
                if unmatched_start < i:
                    delta.append({"op": "LITERAL",
                                   "data": new_data[unmatched_start:i]})
                delta.append({"op": "COPY", "chunk": sig.index})
                i += CHUNK_SIZE
                unmatched_start = i
                continue
        i += 1   # slide one byte (rolling window)

    if unmatched_start < len(new_data):
        delta.append({"op": "LITERAL", "data": new_data[unmatched_start:]})
    return delta

def apply_delta(old_data: bytes, delta: list) -> bytes:
    """Reconstruct new file from old file + delta instructions."""
    result = bytearray()
    for op in delta:
        if   op["op"] == "COPY":
            start = op["chunk"] * CHUNK_SIZE
            result.extend(old_data[start : start + CHUNK_SIZE])
        elif op["op"] == "LITERAL":
            result.extend(op["data"])
    return bytes(result)

Transfer Efficiency

  • Rolling checksum enables O(N) delta computation — Adler-32 updates in O(1) as the window slides one byte. Without rolling checksum: O(N × chunk_size) = millions of times slower.
  • Content-defined chunking eliminates the shift problem — inserting one byte in a fixed-chunk scheme shifts all subsequent chunks. CDC uses content-derived boundaries, so only the chunk containing the change needs retransmission.
  • Merkle trees reduce directory scan from O(N) to O(log N) — comparing root hashes in one step determines if anything changed at all. Traversing the tree to the changed file requires O(depth) comparisons instead of O(N file hashes).
  • Content-addressing enables deduplication — chunk = SHA256(content). Identical chunks (across files or versions) stored once. Dropbox reported 70%+ storage savings from deduplication.
  • Conflict resolution is a product decision, not a technical one — OT and CRDTs are complex but enable real-time collaboration. For file sync (not real-time editing), conflict copies (Dropbox model) are simpler and just as correct.

Why Standard Indexes Don't Work

Naive: query by lat/lng ranges

WHERE lat BETWEEN 12.9 AND 13.1 AND lng BETWEEN 77.5 AND 77.7. This is a bounding box query, not a radius query. A B-tree index on lat helps for the lat range, but then must scan all rows in that lat band for the lng filter. O(N) scan within the band.

Solution: Space-filling curves

Map 2D coordinates to a 1D curve (Geohash, S2, H3) that preserves spatial locality. Points close in space map to nearby 1D codes. Then index the 1D code in a B-tree for O(log N) proximity queries.

Haversine Formula

distance.py — great-circle distance between two lat/lng pointsPython
import math

def haversine_km(lat1: float, lon1: float, lat2: float, lon2: float) -> float:
    """Great-circle distance in km using Haversine formula."""
    R = 6371.0   # Earth radius in km
    φ1, φ2 = math.radians(lat1), math.radians(lat2)
    Δφ = math.radians(lat2 - lat1)
    Δλ = math.radians(lon2 - lon1)
    a  = (math.sin(Δφ/2)**2 +
          math.cos(φ1) * math.cos(φ2) * math.sin(Δλ/2)**2)
    return R * 2 * math.asin(math.sqrt(a))

Geohash — 1D Encoding of 2D Space

Geohash encodes a lat/lng pair into a short alphanumeric string by recursively bisecting the Earth's surface. Nearby points share a common prefix — enabling O(log N) proximity queries via a standard B-tree index.

How Geohash Works

1
Bisect longitude range → 1 bit
Earth is between -180° and 180°. Point is in West half (0) or East half (1). Repeat, alternating between lat and lng bisections.
2
Interleave lat and lng bits
Bits alternate: lng₀, lat₀, lng₁, lat₁, … → a single bitstring. This interleaving creates the space-filling curve property: nearby points → nearby bitstrings.
3
Encode bitstring as Base-32 string
Every 5 bits → one character. "ezs42" encodes a ~2.4 km × 4.9 km cell. Length 12 → ±1m precision. Length 6 → ±1.2 km.

The Boundary Problem

Geohash boundary problem: Two points very close together but on different sides of a geohash cell boundary may have very different hashes. A restaurant at the edge of cell "ezs42" and a user just inside "ezs43" share no common prefix, yet may be 10m apart. Fix: query the target cell AND all 8 neighboring cells. Redis GEOSEARCH handles this automatically.
QuadTree & R-Tree

QuadTree

Recursively divide a 2D region into 4 quadrants. Each node represents a region. If a quadrant has more than K points, subdivide it. Efficient for sparse, non-uniform spatial distributions.

QuadTree Properties

  • Adapts to point density — densely populated areas subdivide more
  • O(log N) query for uniformly distributed points
  • Used by: Bing Maps, Mapbox, Twitter's search-by-location
  • Leaf node stores points. Internal node stores bounding box.

R-Tree (PostgreSQL PostGIS)

  • Indexes arbitrary rectangles/polygons (not just points)
  • Minimum Bounding Rectangles at each level
  • Supports: contains, intersects, within queries
  • Used by: PostgreSQL (GIST index), SQLite, MySQL Spatial
  • Preferred for complex polygon queries (city boundaries, delivery zones)

Scaling Geospatial Search

Sharding Geospatial Data

Shard by City/Region

All restaurants in Bangalore → shard 1. All in Mumbai → shard 2. Queries are routed to the appropriate shard based on the search location. Uneven distribution — large cities get disproportionate load. Use consistent hashing on city_id.

Shard by Geohash Prefix

All geohash prefixes starting with 'a'-'h' → shard 1, 'i'-'p' → shard 2, etc. Geographic regions naturally map to shards. Cross-shard queries rare (same city usually same prefix).

Hierarchical Indexing

Global → Country → City → Neighborhood hierarchy. Start search at city level (fast). Drill down if needed. Uber uses H3 (hexagonal grid) at multiple resolutions — H3 level 6 for city-wide, level 9 for block-level.

Uber's H3 Hexagonal Grid

Why hexagons? A hexagonal grid has equal distance from center to all 6 neighbors (unlike squares where diagonals are 1.41× farther). This makes distance calculations uniform and eliminates the corner effect. H3 has 16 resolution levels: level 0 = continent (~4M km²), level 15 = ~1 m².

geo_search.py — FastAPI + Redis GEOSETPython
import math
from fastapi import FastAPI
import redis.asyncio as aioredis

app   = FastAPI()
redis = aioredis.Redis(host="redis")

@app.post("/locations/{category}/{entity_id}")
async def add_location(category: str, entity_id: str, lat: float, lng: float):
    """Add or update entity location. category = 'restaurants', 'drivers', etc."""
    geo_key = f"geo:{category}"
    await redis.geoadd(geo_key, {entity_id: (lng, lat)})
    return {"indexed": True}

@app.get("/search/{category}")
async def search_nearby(category: str, lat: float, lng: float,
                         radius_km: float = 5.0, limit: int = 20):
    """Find all entities within radius_km of (lat, lng)."""
    geo_key = f"geo:{category}"
    results = await redis.geosearch(
        geo_key,
        longitude=lng, latitude=lat,
        radius=radius_km, unit="km",
        sort="ASC", count=limit,
        withcoord=True, withdist=True
    )
    return [{
        "id":       r[0].decode(),
        "dist_km": round(r[1], 3),
        "lat":     r[2][1],
        "lng":     r[2][0],
    } for r in results]

@app.put("/driver/{driver_id}/location")
async def update_driver(driver_id: str, lat: float, lng: float):
    """Driver location update (called every 5s from mobile app)."""
    async with redis.pipeline() as pipe:
        pipe.geoadd("geo:drivers", {driver_id: (lng, lat)})
        pipe.zadd("driver:last_seen", {driver_id: int(time.time())})
        await pipe.execute()

@app.get("/drivers/nearby")
async def find_drivers(lat: float, lng: float, radius_km: float = 3):
    """Find available (recently active) drivers near a rider."""
    cutoff = int(time.time()) - 30   # active in last 30s
    nearby = await search_nearby("drivers", lat, lng, radius_km, limit=10)
    active = []
    for driver in nearby:
        last_seen = await redis.zscore("driver:last_seen", driver["id"])
        if last_seen and int(last_seen) > cutoff:
            active.append(driver)
    return active

Affinity Score

Affinity is a numeric score quantifying the strength of the relationship between two users. It's used in feed ranking (show posts from close friends higher) and "suggested friends" recommendations.

Signals that Contribute to Affinity

Explicit signals

  • Follow/friend connection (+100)
  • Direct message exchange (+50/message)
  • Close Friends list membership (+200)
  • Comment on each other's posts (+10/comment)

Implicit signals

  • Like their posts (+1/like)
  • View their stories (+2/view)
  • Visit their profile (+5/visit)
  • Share their content (+15)

Time decay

Recent interactions weigh more heavily than old ones. Score decays exponentially with time: score × e^(-λ × days). A like from yesterday = 10× a like from 30 days ago.

Affinity Score Formula

affinity_score.py — time-decayed weighted interactionsPython
import math, time
from dataclasses import dataclass

WEIGHTS = {
    "follow":     100,
    "dm_sent":     50,
    "close_friend":200,
    "comment":      10,
    "like":          1,
    "story_view":    2,
    "profile_visit": 5,
    "share":        15,
}
DECAY_RATE = 0.05   # λ: half-life ≈ 14 days

@dataclass
class Interaction:
    event_type: str
    timestamp_ms: int

def compute_affinity(interactions: list[Interaction]) -> float:
    """
    Affinity score = Σ (weight × e^(-λ × age_in_days))
    Time decay: interactions decay exponentially with age.
    """
    now = time.time() * 1000
    score = 0.0
    for ix in interactions:
        weight   = WEIGHTS.get(ix.event_type, 0)
        age_days = (now - ix.timestamp_ms) / 86_400_000
        decay    = math.exp(-DECAY_RATE * age_days)
        score   += weight * decay
    return round(score, 2)

# Usage: compare A→B vs B→A (asymmetric affinity)
# If A→B affinity is high but B→A is low: A is a fan, not a close friend
# Mutual high affinity → close friends → boost in feed ranking

Storing Affinity Scores

Storage pattern: Store affinity as a Redis Sorted Set. Key: affinity:{user_id}. Member: other_user_id. Score: affinity score. ZREVRANGE affinity:42 0 9 → user 42's top 10 closest connections. Used to rank the feed, determine notification priority, and sort suggested friends. Recomputed periodically (every few hours) from the interaction event stream.

from fastapi import FastAPI
import redis.asyncio as aioredis, time

app   = FastAPI()
redis = aioredis.Redis(host="redis")

───────────────────────────────────
 FOLLOW OPERATIONS
───────────────────────────────────
@app.post("/follow")
async def follow(follower_id: int, following_id: int):
    """A follows B. Update DB + invalidate Redis cache."""
    now = int(time.time() * 1000)
    await db.execute(
        "INSERT IGNORE INTO follows(follower_id, following_id, created_at, status) VALUES(?,?,'active',?)",
        [follower_id, following_id, now]
    )
    # Update Redis caches atomically
    async with redis.pipeline() as pipe:
        pipe.sadd(f"following:{follower_id}", following_id)
        pipe.sadd(f"followers:{following_id}", follower_id)
        # Update affinity: follow = +100
        pipe.zincrby(f"affinity:{follower_id}", 100, str(following_id))
        await pipe.execute()
    return {"status": "following"}

@app.get("/following/{user_id}")
async def get_following(user_id: int, page: int = 0, size: int = 100):
    """Paginated list of who user_id follows."""
    cache = await redis.smembers(f"following:{user_id}")
    if cache:
        ids = [int(x) for x in cache]
        return {"following": ids[page*size:(page+1)*size], "source": "cache"}
    # Cache miss → query DB
    rows = await db.fetch("SELECT following_id FROM follows WHERE follower_id=? AND status='active'", user_id)
    ids  = [r["following_id"] for r in rows]
    if len(ids) <= 5000:   # only cache non-celebrity following lists
        await redis.sadd(f"following:{user_id}", *map(str, ids))
        await redis.expire(f"following:{user_id}", 3600)
    return {"following": ids[page*size:(page+1)*size], "source": "db"}

───────────────────────────────────
 AFFINITY
───────────────────────────────────
@app.get("/affinity/{user_id}/top")
async def top_affinity(user_id: int, limit: int = 20):
    """Return top N users by affinity score. Used for feed ranking."""
    results = await redis.zrevrange(f"affinity:{user_id}", 0, limit-1, withscores=True)
    return [{"user_id": int(uid), "score": round(score, 2)}
            for uid, score in results]

Interaction Log

  • Primary key (follower_id, following_id) is the optimal schema — O(1) lookup for "does A follow B". The reverse index (following_id, follower_id) handles "who follows B" efficiently via a separate B-tree traversal.
  • Cache following lists in Redis SET for active users — SISMEMBER is O(1). For regular users (<5K following), cache the full set in Redis. For celebrities (1M+ followers), paginate from DB — caching 270M members in one SET is impractical.
  • Affinity score is time-decayed — recent interactions matter more. A like from yesterday outweighs a like from 60 days ago by 20×. Without decay, old interactions dominate and the feed becomes stale.
  • Affinity is directional — A→B affinity ≠ B→A. A super-fan may have high affinity towards a celebrity, but the celebrity has near-zero affinity towards them. This asymmetry reflects real relationship strength.
  • Redis Sorted Set for affinity ranking gives O(log N) per interaction — ZINCRBY updates the score. ZREVRANGE returns top-K by score. All in-memory, all fast. Recompute from scratch periodically for time-decay correctness.

Interview Checklist

For Every System Design Interview

The 5-step Framework

1
Requirements (3–5 min)
Functional + Non-functional. Clarify scale (DAU, QPS, storage). Define what "success" means.
2
Capacity estimation (3–5 min)
QPS, storage, bandwidth. Use round numbers. Identify bottlenecks proactively.
3
High-level design (10 min)
API endpoints, data flow, core components. Start simple, identify the core data structure or algorithm.
4
Deep dive (15 min)
Pick the hardest part: data model, concurrency, scaling strategy. Show depth here.
5
Tradeoffs & extensions
Acknowledge weaknesses in your design. How would it handle 10× scale? What would you change?

Questions to Always Ask

  • What is the expected QPS / DAU?
  • Read-heavy or write-heavy? Ratio?
  • What are the consistency requirements? Can we tolerate eventual?
  • What is the data retention policy?
  • Is there a global deployment requirement?
  • What is the latency SLA? p50, p99?
  • Are there regulatory requirements (GDPR, data residency)?

Common Anti-Patterns to Avoid

Over-engineering

Starting with microservices, Kafka, Redis, Elasticsearch before identifying if they're needed. Start simple. Add components only when you've identified a concrete bottleneck they solve.

Ignoring the data model

Jumping to architecture without defining the schema. The data model often determines the entire architecture. What you store and how drives everything else.

No tradeoffs

Every choice has a tradeoff. Saying "we'll use Cassandra because it scales" without mentioning its eventual consistency, no joins, or tombstone management shows shallow understanding.
