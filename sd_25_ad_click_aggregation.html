Ad Click Aggregation
Design a system that ingests 1 million ad click events per second, deduplicates fraudulent/repeated clicks, aggregates counts per ad per minute using sliding windows, stores results queryable by advertisers in real time — all while handling late-arriving events, watermarks, and the tradeoffs between exactly-once and at-least-once delivery.
🔴 MUST DO 🟣 Hard Stream Processing Flink / Kafka Sliding Windows Deduplication Watermarks OLAP Query
🎯 Clarify Scope
Ad click aggregation is a canonical stream processing problem. The hard parts: deduplication at scale (same click sent multiple times), late-arriving events (click from a user on a slow network arrives 5 minutes after it happened), and making aggregates queryable in real time without scanning raw logs.

🙋 Ask First
Query granularity — last 1 min, last hour, last day?
Exact counts or approximate (HyperLogLog)?
How do we define a duplicate click? Same user + ad within X seconds?
How late can events arrive and still count?
Do advertisers need real-time dashboards or batch reports?
✅ Assumptions
Query: clicks per ad per minute (1-min tumbling window). Also support last 1h, 24h roll-ups.
Exact counts for billing. Approximate OK for dashboards.
Duplicate = same (user_id, ad_id) within 1 minute.
Late events up to 5 minutes accepted. After that, discarded.
Real-time dashboards (seconds lag), batch billing reports (hourly).
✅ Functional + Non-Functional
FUNCTIONAL
Ingest: Receive click events with ad_id, user_id, timestamp, IP
Dedup: Deduplicate repeat clicks same user+ad within 1 min
Aggregate: Count clicks per ad per 1-min window (tumbling)
Query: GET clicks for ad X over time range Y–Z
Billing: Hourly exact count batch report per advertiser
NON-FUNCTIONAL
Ingest: 1M events/sec sustained, 3M peak
Query latency: <1s for aggregated results
Aggregate freshness: <10s lag from click to dashboard
Billing accuracy: exactly-once semantics (no undercounting)
Late events: accepted up to 5 minutes after event time
💬 Phase 1 — Deduplication at 1M events/sec
PHASE 1 · DEDUPLICATION
IR
INTERVIEWER
A mobile app sends a click event, then retries on network failure and sends it again 30 seconds later. How do you deduplicate at 1M events/sec?
ME
YOU
The dedup key is (user_id, ad_id, window_minute) — same user clicking the same ad in the same 1-minute window counts as one click.

Two-tier dedup strategy:

1. Event-level dedup (click_id): Each click event includes a client-generated click_id (UUID). This is the primary dedup key for exact duplicate events (retries, network re-sends). Kafka's idempotent producer + the consumer's enable.idempotence=true handles this at the message layer — same click_id sent twice = processed exactly once.

2. Business-level dedup (user+ad+window): Even after event dedup, we may have multiple genuine clicks from the same user on the same ad within 1 minute (spam, bot, accidental double-click). For this we maintain a Redis set per window: SADD dedup:{ad_id}:{window_min} {user_id}. If SADD returns 0 (user already in set), discard this click. TTL on the set: 5 minutes (covering our late-event window).

Memory calculation: 1M clicks/min × avg 10 bytes per user_id → 10 MB per ad per window. With 100K ads, this is 1 TB — too large for Redis sets. Solution: use HyperLogLog per ad per window for dashboard (approximate, ~12 KB per window) + exact Bloom filter for billing-critical dedup. Alternatively, partition by ad_id across a Redis cluster — each Redis node handles only ~10K ads.
💡 Key insight: Separate the dedup concern from the aggregation concern. Use Bloom filters or HyperLogLog for fast approximate dedup in stream layer, then reconcile against the raw event log in batch for exact billing counts.
⏰ Phase 2 — Late Events & Watermarks
PHASE 2 · LATE DATA
IR
INTERVIEWER
A user is on a plane with no wifi. They click an ad offline. 4 minutes later the plane lands and the event is sent. How does your system handle this?
ME
YOU
This is the late event / watermark problem. The event's event time (when the user clicked) is 4 minutes before the processing time (when we receive it).

Windowing with watermarks (Apache Flink approach):

Watermark: A signal that says "I won't receive any more events with event_time < W". Flink generates watermarks by tracking the maximum event timestamp seen, minus a configurable allowed lateness (our 5 min). When the watermark advances past the end of a window, the window closes and emits its result.

Example: We're at processing time 12:10. Our watermark is 12:05 (5 min allowed lateness). An event arrives with event_time=12:03 → this is < watermark (12:05), so it's accepted as a late event and added to the 12:03–12:04 window. But an event with event_time=11:59 is 6 minutes late → past our allowed lateness → discarded (or routed to a side-output dead-letter queue for manual review).

Implementation: Flink's TumblingEventTimeWindows + withAllowedLateness(Time.minutes(5)). Windows emit an initial result when watermark passes, then emit updated results for each accepted late event. The billing batch job reconciles using the raw log (Kafka retained 7 days) to catch anything the stream missed.
📐 Phase 3 — Query API Design
PHASE 3 · QUERY LAYER
IR
INTERVIEWER
An advertiser queries "how many clicks did ad #12345 get in the last hour?" How does that work?
ME
YOU
Two data stores serve different query patterns:

1. OLAP (ClickHouse / Apache Druid) for time-range queries: Flink writes 1-minute aggregates to ClickHouse: INSERT INTO ad_clicks (ad_id, window_start, click_count). Query: SELECT SUM(click_count) FROM ad_clicks WHERE ad_id=12345 AND window_start BETWEEN now()-3600 AND now(). ClickHouse columnar storage makes this extremely fast — 60 rows for 1 hour, sub-millisecond. Scales to billions of rows for historical analysis.

2. Redis for real-time "last N minutes": Flink also writes to a Redis time-series key: ZADD clicks:{ad_id} {timestamp} {count}. Query last 5 minutes: ZRANGEBYSCORE clicks:{ad_id} (now-300) now → SUM the values. Ultra-fast for dashboard sparklines. TTL: 24h — after that, read from ClickHouse.

Why both? Redis: sub-millisecond for recent real-time data. ClickHouse: seconds for historical analysis over months of data. Neither is optimal alone for both time ranges.

// ══════════════════════════════════════════════════════
// Ad Click Aggregation — Apache Flink (Java)
// ══════════════════════════════════════════════════════

public class AdClickAggregationJob {

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

        // ── 1. Kafka Source ──────────────────────────────────
        KafkaSource<ClickEvent> source = KafkaSource.<ClickEvent>builder()
            .setBootstrapServers("kafka:9092")
            .setTopics("ad_clicks")
            .setGroupId("flink-agg-group")
            .setValueOnlyDeserializer(new ClickEventDeserializer())
            .build();

        DataStream<ClickEvent> clicks = env
            .fromSource(source, WatermarkStrategy
                .<ClickEvent>forBoundedOutOfOrderness(Duration.ofMinutes(5)) // 5-min late allowance
                .withTimestampAssigner((e, ts) -> e.getEventTimeMs()),
            "Kafka Click Source");

        // ── 2. Dedup Stage ───────────────────────────────────
        DataStream<ClickEvent> deduped = clicks
            .keyBy(e -> e.getAdId() + ":" + e.getUserId())
            .process(new DeduplicationFunction());

        // ── 3. Tumbling 1-min window ─────────────────────────
        DataStream<AdClickAggregate> aggregates = deduped
            .keyBy(ClickEvent::getAdId)
            .window(TumblingEventTimeWindows.of(Time.minutes(1)))
            .allowedLateness(Time.minutes(5))  // re-emit on late events
            .aggregate(new CountAggregator(), new WindowMetadataAdder());

        // ── 4. Dual sink: Redis TS + ClickHouse ──────────────
        aggregates.addSink(new RedisSink());
        aggregates.addSink(new ClickHouseSink());

        env.execute("Ad Click Aggregation");
    }
}

// ── Deduplication: Bloom filter per (user, ad, window) ──
public class DeduplicationFunction
    extends KeyedProcessFunction<String, ClickEvent, ClickEvent> {

    private ValueState<Boolean> seenState;

    @Override
    public void open(Configuration params) {
        seenState = getRuntimeContext().getState(
            new ValueStateDescriptor<>("seen", Boolean.class));
        // TTL: auto-clear state after 65 seconds (1-min window + margin)
        seenState.enableTimeToLive(StateTtlConfig
            .newBuilder(Time.seconds(65)).build());
    }

    @Override
    public void processElement(ClickEvent event, Context ctx, Collector<ClickEvent> out) throws Exception {
        if (seenState.value() == null) {
            seenState.update(true);
            out.collect(event); // first click in this window for this key
        }
        // else: duplicate — drop silently
    }
}

// ── Click Count Aggregator ───────────────────────────────
public class CountAggregator
    implements AggregateFunction<ClickEvent, Long, Long> {
    @Override public Long createAccumulator() { return 0L; }
    @Override public Long add(ClickEvent e, Long acc) { return acc + 1; }
    @Override public Long getResult(Long acc) { return acc; }
    @Override public Long merge(Long a, Long b) { return a + b; }
}

// ── Redis Time-Series Sink ───────────────────────────────
public class RedisSink extends RichSinkFunction<AdClickAggregate> {
    private JedisPool pool;
    @Override
    public void invoke(AdClickAggregate agg, Context ctx) {
        try (Jedis j = pool.getResource()) {
            // ZADD clicks:{ad_id} {timestamp_score} {count}
            j.zadd("clicks:" + agg.getAdId(),
                   agg.getWindowStart(),
                   agg.getWindowStart() + ":" + agg.getClickCount());
            j.expire("clicks:" + agg.getAdId(), 86400); // 24h TTL
        }
    }
}

# Ad Click Aggregation — Python (Faust stream library)
# ══════════════════════════════════════════════════════

import faust, time, redis
from datetime import datetime, timedelta

app = faust.App(\'ad-click-agg\',
    broker=\'kafka://localhost:9092\',
    value_serializer=\'json\')


class ClickEvent(faust.Record):
    click_id:   str
    ad_id:      str
    user_id:    str
    event_time: int  # unix ms


class AggResult(faust.Record):
    ad_id:        str
    window_start: int
    click_count:  int


clicks_topic  = app.topic(\'ad_clicks\', value_type=ClickEvent)
agg_topic     = app.topic(\'ad_agg\', value_type=AggResult)
redis_client  = redis.Redis()

# State: dedup table (user+ad+window → seen bool)
seen_table = app.Table(\'dedup\', default=bool, help=\'Dedup state per user+ad+window\')
# State: counts per ad per window
count_table = app.Table(\'counts\', default=int, help=\'Click counts per ad per minute window\')


def window_minute(event_time_ms: int) -> int:
    """Floor event_time to minute boundary."""
    return (event_time_ms // 60000) * 60000

def dedup_key(event: ClickEvent) -> str:
    return f"{event.user_id}:{event.ad_id}:{window_minute(event.event_time)}"

def count_key(event: ClickEvent) -> str:
    return f"{event.ad_id}:{window_minute(event.event_time)}"


@app.agent(clicks_topic)
async def process_clicks(stream):
    async for event in stream:
        # 1. Dedup check
        dk = dedup_key(event)
        if seen_table[dk]:
            continue  # duplicate — skip
        seen_table[dk] = True

        # 2. Late event check (5-min allowed lateness)
        now_ms   = int(time.time() * 1000)
        lateness = now_ms - event.event_time
        if lateness > 300_000:  # 5 minutes in ms
            continue  # too late — discard

        # 3. Increment window counter
        ck = count_key(event)
        count_table[ck] += 1

        # 4. Every minute: emit aggregate to Redis + ClickHouse
        await emit_aggregate(event.ad_id, window_minute(event.event_time), count_table[ck])


async def emit_aggregate(ad_id: str, window_start: int, count: int):
    """Write aggregate to Redis TS and downstream (ClickHouse in production)."""
    redis_key = f\'clicks:{ad_id}\'

    # ZADD with score=timestamp for time-range queries
    redis_client.zadd(redis_key, {f\'{window_start}:{count}\': window_start})
    redis_client.expire(redis_key, 86400)  # 24h TTL

    # In production: also INSERT INTO clickhouse.ad_clicks ...
    print(f\'[AGG] ad={ad_id} window={window_start} count={count}\')


# ── Query API (separate FastAPI service) ────────────
def query_clicks(ad_id: str, start_ms: int, end_ms: int) -> dict:
    """
    Route query to Redis (recent) or ClickHouse (historical).
    """
    cutoff_ms = int(time.time() * 1000) - 86_400_000  # 24h ago

    if start_ms >= cutoff_ms:
        # Recent: serve from Redis TS
        entries = redis_client.zrangebyscore(
            f\'clicks:{ad_id}\', start_ms, end_ms, withscores=False)
        buckets = []
        for entry in entries:
            window_start, count = entry.decode().split(\':\')
            buckets.append({\'window_start\': int(window_start), \'click_count\': int(count)})
        return {\'ad_id\': ad_id, \'buckets\': buckets,
                \'total\': sum(b[\'click_count\'] for b in buckets),
                \'data_source\': \'realtime\'}
    else:
        # Historical: query ClickHouse
        # SELECT window_start, click_count FROM ad_clicks WHERE ad_id=? AND ...
        return {\'ad_id\': ad_id, \'data_source\': \'historical\'}


if __name__ == \'__main__\':
    app.main()
⚖️ Tradeoffs
⚡ Exactly-Once vs At-Least-Once vs At-Most-Once
At-Most-Once

Drop events on failure. Never double-count. Undercounts on crashes. Unacceptable for billing — advertisers would be undercharged but the gap is invisible. Only OK for non-critical metrics where under-reporting is preferred to over-reporting.

⚠️ At-Least-Once (stream layer)

Retry on failure — no events lost. May double-count on retry. Mitigated by application-level dedup (click_id + bloom filter). Acceptable for dashboards. Not sufficient alone for billing (need batch reconciliation).

✅ Exactly-Once (billing via batch)

Stream gives at-least-once (fast, low latency). Batch Spark job reads from Kafka raw log (7-day retention) with exact dedup — produces billing-accurate hourly counts. Two-layer approach: stream for realtime visibility, batch for financial accuracy. Best of both.

📊 Aggregation: Stream-only vs Lambda vs Kappa
Stream-only

Only Flink, no batch. Simple architecture. But: stream state is lossy on failure (if Kafka retention expires, late events are lost permanently). Complex exactly-once handling. Acceptable if Kafka retention covers your reprocessing window.

✅ Lambda (stream + batch)

Stream for real-time (seconds lag). Batch for accuracy (minutes lag). Two separate pipelines. More complex to maintain. But: batch always wins on correctness — can reprocess any time window from the Kafka log. This is the industry standard for billing systems at Google/Meta/Twitter scale.

Kappa (one stream for both)

Single Flink job, re-processes history by rewinding Kafka offsets. Simpler to maintain. Works well when stream processing is idempotent. Risk: slow historical reprocessing blocks real-time if they share resources. Emerging pattern but less battle-tested for billing accuracy.

🗃 Storage: ClickHouse vs TimescaleDB vs Druid
✅ ClickHouse (chosen)

Columnar OLAP. Sub-second queries on billions of rows. Excellent compression (10×). INSERT throughput: 500K rows/sec. Perfect for pre-aggregated 1-minute data. Used by Cloudflare, Uber, eBay for analytics at this scale.

Apache Druid

Real-time OLAP with native stream ingest. More complex to operate than ClickHouse. Better for sub-second ingestion and query. Higher operational cost. Best when you need <1s query on raw (non-pre-aggregated) data.

TimescaleDB

PostgreSQL extension for time-series. Familiar SQL. Great for <100M rows/day. At 1M events/sec → 86B rows/day — even aggregated, TimescaleDB struggles at this scale. Fine for small-scale analytics, not for this problem.
