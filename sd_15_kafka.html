🧠 Why a Message Queue? The Core Problem
Message queues decouple producers from consumers. The payment service doesn't need to wait for the email service, analytics service, and fraud service all to finish processing before it can respond to the user. They each consume the payment event independently at their own pace.

❌ WITHOUT A QUEUE — TIGHT COUPLING
1
Payment service calls Email service (HTTP) → waits 200ms
2
Payment service calls Analytics (HTTP) → waits 500ms
3
Payment service calls Fraud (HTTP) → waits 100ms
4
Total: 800ms before user gets a response!
5
If Email is down → payment fails too. Fatal coupling.
✅ WITH A QUEUE — DECOUPLED
1
Payment service publishes "payment_completed" to Kafka → 5ms
2
Returns 200 OK to user immediately
3
Email, Analytics, Fraud each consume independently
4
If Email is slow → doesn't affect payment latency
5
If Email is down → messages wait in queue, delivered when it recovers
📖 Key Terminology
CORE ENTITIES
Topic: Named stream of messages. E.g., payments, user-events. Like a database table but for events.
Partition: A topic is split into N partitions. Each partition is an ordered, append-only log. Parallelism unit.
Offset: Sequential integer assigned to each message within a partition. 0, 1, 2, 3... Consumer tracks which offset it has processed.
Broker: A Kafka server. Stores partitions and serves producers/consumers.
PRODUCER SIDE
Producer: Publishes messages to a topic. Chooses which partition via key hash or round-robin.
Partition key: Determines which partition a message goes to. Same key → same partition → ordered delivery for that key.
acks: Durability setting. acks=0 (fire-and-forget), acks=1 (leader ack), acks=all (all replicas ack).
Batch: Producer batches messages before sending for throughput.
CONSUMER SIDE
Consumer Group: N consumers sharing a topic. Each partition assigned to exactly one consumer. Parallelism = min(consumers, partitions).
Committed offset: The offset the consumer has durably acknowledged. On restart, resumes from here.
Lag: Latest offset minus committed offset. Measures how far behind a consumer is.
Rebalance: Consumer group reconfigures when a consumer joins or leaves.

⚙️ Replication — ISR (In-Sync Replicas)
Kafka's durability guarantee comes from ISR replication. Every partition has one leader and N-1 followers. Writes go to leader. Followers pull from leader. A replica is "in-sync" if it's within the configured lag threshold.


📝 WRITE PATH (acks=all)
1
Producer sends message to leader broker for partition
2
Leader appends to its local log (sequential disk write)
3
ISR followers pull the new data from leader (fetch request)
4
Each follower appends to its log, sends ACK to leader
5
Leader waits for ALL ISR replicas to ACK, then responds to producer
6
High Water Mark (HWM) advances — message is now visible to consumers
🔁 LEADER ELECTION (broker fails)
1
Broker 1 (leader) crashes. ZooKeeper / KRaft detects it.
2
Controller selects new leader from ISR — only ISR members eligible (guaranteed to have all committed data)
3
Broker 2 becomes new leader. Updates partition metadata in ZooKeeper.
4
Producer/consumer clients refresh metadata, reconnect to new leader
5
If ISR has only 1 member: choose between data loss (elect non-ISR) or availability loss (wait for ISR member to come back). Controlled by unclean.leader.election.enable.
👥 Consumer Groups — Parallel Consumption
The consumer group model is Kafka's load-balancing mechanism. Each partition is assigned to exactly one consumer in the group. To increase parallelism: add consumers (up to the partition count) or add partitions.

Key rule: Each partition assigned to exactly ONE consumer per group. Adding a 7th consumer to the email group (which has 6 partitions) = idle consumer — no partition left to assign. Adding partitions first unlocks more parallelism.
🗜 Log Compaction vs Log Deletion
🗑 LOG DELETION (time/size based)

Default Kafka retention. Delete old log segments after a time or size threshold (e.g., 7 days or 100 GB). Messages older than the window are gone forever.

[0:pay_101] [1:pay_102] [2:pay_103] [3:pay_104] ←keep
[segment deleted: >7 days old]

Use for: event streams, logs, metrics — where old data has no value

🗜 LOG COMPACTION (key-based)

Keep only the latest value per key. Old values for same key are removed. The log always represents the "current state" — like a changelog for a database table.

Before: [user:1:v1][user:2:v1][user:1:v2][user:1:v3]
After: [user:2:v1][user:1:v3] ← latest values only

Use for: database changelogs, user profiles, inventory state — where only the current value matters. Used by Kafka Connect (CDC), Kafka Streams (KTable)

💬 Phase 1 — Core Architecture
PHASE 1 · FUNDAMENTALS
IR
INTERVIEWER
Design a distributed message queue. Start with the architecture.
ME
YOU
A distributed message queue needs to solve four problems: durability (messages survive broker failures), scalability (handle millions of msgs/sec), ordering (at least within a partition), and replay (consumers can re-read old messages).

The core architecture:
  1. Topics + Partitions: A topic is sharded into N partitions. Each partition is an ordered, append-only log on disk. This gives us horizontal write scalability — producers write to different partitions in parallel.
  2. Brokers: Each partition is hosted on a broker. With replication factor 3, each partition has 1 leader and 2 follower replicas on different brokers.
  3. Offsets: Each message within a partition gets a sequential offset. Consumers track which offset they've processed — this is how we enable replay, at-least-once delivery, and independent consumer groups.
  4. Consumer Groups: Each partition assigned to exactly one consumer per group. Adding consumers scales up consumption proportionally to partition count.
Why append-only logs? Disk sequential writes are extremely fast (>500 MB/s on HDD). By writing sequentially, Kafka can sustain millions of messages/sec while maintaining durability.
💡 Mention "append-only log + offsets" early — this is the key insight that makes Kafka different from traditional queues like RabbitMQ. Messages aren't deleted on consumption; the consumer's offset tracks progress.
🔁 Phase 2 — Delivery Guarantees
PHASE 2 · DELIVERY SEMANTICS
IR
INTERVIEWER
What delivery guarantees can you provide? At-most-once, at-least-once, exactly-once?
ME
YOU
These are orthogonal on the producer side and consumer side:

Producer side:
  • acks=0: At-most-once. No acknowledgment. Message may be lost. Fastest. Use for high-throughput metrics where some loss is acceptable.
  • acks=1: At-least-once (from leader only). Leader ACKs before replication. If leader crashes before replication, message lost. Moderate.
  • acks=all: At-least-once (durable). All ISR replicas ACK. No data loss on leader failure. Slowest but safest.
Consumer side (the tricky part):
  • At-most-once: Commit offset BEFORE processing. If the consumer crashes after commit but before processing, the message is skipped.
  • At-least-once: Commit offset AFTER processing. If the consumer crashes after processing but before committing, the message is reprocessed. This is the most common and correct default — make your consumer idempotent.
Exactly-once: Requires the Kafka Transactions API. Producer uses transactional.id. Consumer uses isolation.level=read_committed. Both produce and commit offset in one atomic transaction. Expensive (~2-3× throughput reduction). Only worth it for financial transactions, de-duplication-critical pipelines.
IR
INTERVIEWER
Consumer crashes mid-batch. What happens with at-least-once delivery?
ME
YOU
The consumer processes messages 100-150 without committing. It crashes at message 130. On restart, it resumes from the last committed offset — say 99. It reprocesses messages 100–130.

These messages are processed twice. For at-least-once to be correct, processing must be idempotent — applying the same operation twice gives the same result as applying it once.

Examples of idempotent operations:
  • UPDATE users SET email = 'a@b.com' WHERE id = 123 — safe to run twice
  • INSERT INTO events ... ON CONFLICT DO NOTHING — second insert is ignored
Non-idempotent (dangerous with at-least-once):
  • INSERT INTO payments (amount) VALUES (100) — duplicate payment!
  • Send welcome email — user gets 2 emails
The solution: use a deduplication key. Store the message's offset or a unique event ID in the target database. Check for existence before processing: if already processed, skip.
📐 Phase 3 — Partitioning Strategy
PHASE 3 · PARTITIONING
IR
INTERVIEWER
How do you decide which partition a message goes to? When does ordering matter?
ME
YOU
Partitioning determines both ordering guarantees and parallelism:

Key-based partitioning: partition = hash(key) % num_partitions. All messages with the same key go to the same partition → they're processed in order. Essential when operations on an entity must be sequential.

Example: user_id as key → all events for user 123 go to partition 2 → the consumer processing partition 2 sees user 123's events in order: "account created → email verified → first purchase". If these went to different partitions and different consumers, you might process "first purchase" before "account created".

Round-robin (no key): Messages distributed evenly across all partitions. Best for maximum throughput with no ordering requirements. Example: log messages, metrics.

Custom partitioner: Route VIP users to dedicated partitions for priority processing. Route by region. Implement any business logic.

Critical tradeoff: More partitions = more parallelism but more overhead (more OS file handles, more ZooKeeper nodes, slower rebalancing). Rule of thumb: start with 3× the expected peak consumer count. Don't over-partition.
⚡ Phase 4 — Why Is Kafka Fast?
PHASE 4 · PERFORMANCE
IR
INTERVIEWER
Kafka handles millions of messages per second. What makes it so fast?
ME
YOU
Kafka's performance comes from four design decisions that work together:

1. Sequential disk I/O: Kafka always appends to the end of a log. Sequential writes on a modern HDD: ~500 MB/s. Random writes: ~100 KB/s. By making writes sequential, Kafka makes HDD writes as fast as RAM writes in some scenarios. The OS also aggressively caches sequential disk pages in the page cache.

2. Zero-copy sendfile: Normally to send data to a network socket: disk → kernel buffer → user space → kernel socket buffer → network. Zero-copy: disk → kernel buffer → network (using the sendfile() system call). Eliminates one expensive kernel-to-user-space copy. Kafka uses this for consumer reads.

3. Batching: Producer batches messages before sending. Instead of 1000 network round trips for 1000 messages, one round trip sends all 1000 as a batch. Amortises network and disk overhead massively.

4. Compression: Entire batches are compressed (lz4, gzip, snappy). Text messages compress 5-10×. Less network bandwidth, less disk space. The consumer decompresses the entire batch.

Together, these let Kafka achieve 500K-1M messages/sec on a single commodity broker.
💡 "Sequential disk I/O + zero-copy sendfile" is the answer interviewers are fishing for when they ask about Kafka performance. These two points alone distinguish strong candidates.
🔄 Phase 5 — Consumer Lag & Backpressure
PHASE 5 · OPERATIONS
IR
INTERVIEWER
Consumers are falling behind — consumer lag is growing. What do you do?
ME
YOU
Consumer lag = latest offset − committed offset. A growing lag means the consumer is consuming slower than producers are producing. I'd diagnose and fix in order:

1. Diagnose first: Is the bottleneck CPU-bound (processing), I/O-bound (DB writes), or network-bound? Check consumer CPU, downstream DB latency, network throughput.

2. Add more consumers (if partitions allow it): If we have 6 partitions and 3 consumers, adding 3 more consumers instantly doubles consumption throughput — each consumer handles fewer partitions. This is the first lever.

3. Add more partitions (then consumers): If consumers == partitions (ceiling hit), add more partitions. Caution: adding partitions means previously key-ordered messages may reroute. Do this carefully for key-partitioned topics.

4. Optimise the consumer: Batch database writes instead of one-by-one. Use async processing (fire-and-forget for idempotent operations). Tune max.poll.records (fetch larger batches).

5. Use consumer groups strategically: Separate slow consumers (reporting) from fast consumers (alerting) into different groups. The slow group falls behind without affecting the fast group.

// ── 1. Producer — At-Least-Once with idempotency ──────
@Service
public class PaymentEventProducer {

    private final KafkaTemplate<String, PaymentEvent> kafka;

    // Producer config (set in application.yml or Properties):
    // acks=all, retries=3, enable.idempotence=true (exactly-once at producer)
    // batch.size=16384, linger.ms=5, compression.type=lz4

    public CompletableFuture<SendResult<String, PaymentEvent>>
            publishPaymentCompleted(PaymentEvent event) {

        // Partition key = user_id → all events for same user → same partition
        // Guarantees ordering: debit before credit for same user
        return kafka.send("payments", event.getUserId(), event)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    // Log and alert — producer retries automatically (3 retries)
                    log.error("Failed to publish payment event: {}",
                        event.getPaymentId(), ex);
                } else {
                    // Record metadata: which partition and offset this message landed at
                    log.debug("Published to {}-{} offset {}",
                        result.getRecordMetadata().topic(),
                        result.getRecordMetadata().partition(),
                        result.getRecordMetadata().offset());
                }
            });
    }
}

// ── 2. Consumer — At-Least-Once with manual commit ────
@Component
public class PaymentEventConsumer {

    @Autowired private EmailService   emailSvc;
    @Autowired private DedupeStore    dedupeStore; // Redis/DB for idempotency

    // enable.auto.commit=false — we commit manually AFTER processing
    // This gives at-least-once: if we crash after processing but before commit,
    // we reprocess on restart. Make processing idempotent!
    @KafkaListener(
        topics = "payments",
        groupId = "email-notifications",
        containerFactory = "manualAckContainerFactory"
    )
    public void onPayment(List<ConsumerRecord<String, PaymentEvent>> records,
                           Acknowledgment ack) {

        for (ConsumerRecord<String, PaymentEvent> record : records) {
            String       dedupeKey = record.topic() + ":"
                                      + record.partition() + ":"
                                      + record.offset(); // unique per message

            // IDEMPOTENCY CHECK: skip if already processed
            if (dedupeStore.exists(dedupeKey)) {
                log.info("Skipping duplicate message at offset {}", record.offset());
                continue;
            }

            try {
                // Process the message
                PaymentEvent event = record.value();
                emailSvc.sendPaymentConfirmation(event);

                // Mark as processed BEFORE committing
                dedupeStore.mark(dedupeKey, Duration.ofDays(7));

            } catch (RetryableException e) {
                // Don't ack — message will be retried
                log.warn("Retryable failure, not committing", e);
                return;
            } catch (Exception e) {
                // Non-retryable — send to DLQ and continue
                sendToDlq(record, e);
            }
        }

        // Commit AFTER successful processing of all records in batch
        ack.acknowledge(); // commits the highest offset in this batch
    }

    private void sendToDlq(ConsumerRecord<?, ?> record, Exception e) {
        // Publish to dead letter topic for manual investigation
        kafka.send("payments.dlq",
            new DlqMessage(record.key(), record.value(), e.getMessage(),
                           record.topic(), record.partition(), record.offset()));
    }
}

// ── 3. Consumer Lag Monitor ───────────────────────────
@Component
public class ConsumerLagMonitor {

    @Autowired private KafkaAdmin          admin;
    @Autowired private AlertService        alerts;

    @Scheduled(fixedDelay = 30_000) // check every 30s
    public void checkLag() {
        // Kafka AdminClient: list consumer group offsets vs end offsets
        Map<TopicPartition, Long> endOffsets =
            admin.listEndOffsets("payments");
        Map<TopicPartition, OffsetAndMetadata> committed =
            admin.listConsumerGroupOffsets("email-notifications");

        long totalLag = 0;
        for (var entry : endOffsets.entrySet()) {
            OffsetAndMetadata c = committed.get(entry.getKey());
            long lag = (c != null)
                ? entry.getValue() - c.offset()
                : entry.getValue();
            totalLag += lag;

            if (lag > 10_000) { // threshold: 10K messages behind
                alerts.fire("HIGH_LAG", entry.getKey(), lag);
            }
        }
        metrics.gauge("kafka.consumer.total_lag", totalLag);
    }
}

from confluent_kafka import Producer, Consumer, KafkaError, KafkaException
from confluent_kafka.admin import AdminClient, NewTopic
import json, logging, time
from typing import Callable, Optional

logger = logging.getLogger(__name__)


# ── 1. Producer ───────────────────────────────────────
class PaymentEventProducer:
    """
    Publishes payment events to Kafka.
    Uses acks=all + idempotence for at-least-once with no duplicates at producer.
    Batching (linger.ms=5) + lz4 compression for throughput.
    """

    def __init__(self, brokers: str):
        self._producer = Producer({
            'bootstrap.servers':  brokers,
            'acks':               'all',          # wait for all ISR
            'retries':            3,
            'retry.backoff.ms':   100,
            'enable.idempotence': True,         # no duplicates at producer
            'compression.type':   'lz4',        # compress batch
            'linger.ms':          5,             # wait 5ms to fill batch
            'batch.size':         16384,        # 16 KB batch
        })

    def publish_payment(self, event: dict) -> None:
        """
        Publish payment event. Partition by user_id for ordering.
        Async — delivery_report callback fires when broker ACKs.
        """
        key   = str(event['user_id']).encode()
        value = json.dumps(event).encode()

        try:
            self._producer.produce(
                topic='payments',
                key=key,
                value=value,
                on_delivery=self._delivery_report
            )
            # poll() to trigger delivery callbacks and keep internal queue drained
            self._producer.poll(0)

        except BufferError:
            # Internal queue full — flush and retry
            logger.warning("Producer queue full, flushing")
            self._producer.flush()
            self._producer.produce(topic='payments', key=key, value=value,
                                   on_delivery=self._delivery_report)

    def _delivery_report(self, err, msg) -> None:
        if err:
            logger.error(f"Delivery failed: {err}")
        else:
            logger.debug(f"Delivered to {msg.topic()}-{msg.partition()} "
                         f"offset={msg.offset()}")

    def flush(self):
        """Wait for all messages to be delivered before shutdown."""
        self._producer.flush(timeout=30)


# ── 2. Consumer — At-Least-Once with manual commit ────
class PaymentEventConsumer:
    """
    Consumes payment events with at-least-once semantics.
    Manual offset commit AFTER processing.
    Idempotency check prevents duplicate side effects on retry.
    """

    def __init__(self, brokers: str, group_id: str,
                  process_fn: Callable, dedupe_store):
        self._consumer = Consumer({
            'bootstrap.servers':       brokers,
            'group.id':                group_id,
            'auto.offset.reset':       'earliest',  # start from beginning
            'enable.auto.commit':      False,        # manual commit!
            'max.poll.interval.ms':    300000,       # 5 min max processing time
            'session.timeout.ms':      30000,
        })
        self._process    = process_fn
        self._dedupe     = dedupe_store
        self._running    = True

    def run(self, topics: list[str]) -> None:
        """Main consumer loop. Blocks until shutdown()."""
        self._consumer.subscribe(
            topics,
            on_assign=self._on_assign,
            on_revoke=self._on_revoke
        )

        try:
            while self._running:
                # Poll up to 500ms. Returns None on timeout.
                msg = self._consumer.poll(timeout=0.5)
                if msg is None:
                    continue

                if msg.error():
                    if msg.error().code() == KafkaError.PARTITION_EOF:
                        continue  # caught up — not an error
                    raise KafkaException(msg.error())

                self._process_message(msg)

        finally:
            # Commit any pending offsets before graceful shutdown
            self._consumer.commit(asynchronous=False)
            self._consumer.close()

    def _process_message(self, msg) -> None:
        # Unique key for idempotency: topic:partition:offset
        dedupe_key = f"{msg.topic()}:{msg.partition()}:{msg.offset()}"

        if self._dedupe.exists(dedupe_key):
            logger.info(f"Skipping duplicate at offset {msg.offset()}")
        else:
            try:
                event = json.loads(msg.value())
                self._process(event)
                self._dedupe.mark(dedupe_key, ttl_days=7)

            except Exception as e:
                logger.error(f"Processing failed for offset {msg.offset()}: {e}")
                self._send_to_dlq(msg, e)

        # Commit AFTER processing (at-least-once)
        # store_offsets() stages for next commitSync
        # We commit synchronously here for simplicity; use async in prod
        self._consumer.store_offsets(msg)
        self._consumer.commit(asynchronous=False)

    def _on_assign(self, consumer, partitions) -> None:
        logger.info(f"Assigned partitions: {[p.partition for p in partitions]}")

    def _on_revoke(self, consumer, partitions) -> None:
        # Commit before rebalance to avoid reprocessing
        consumer.commit(asynchronous=False)
        logger.info(f"Revoked partitions: {[p.partition for p in partitions]}")

    def shutdown(self): self._running = False
    def _send_to_dlq(self, msg, err): pass  # publish to payments.dlq

⚖️ Key Tradeoffs
Most Kafka tuning is durability vs throughput vs latency. Every parameter trades off one against the others. Know these tradeoffs cold — interviewers ask them directly.
🔒 acks Setting: 0 vs 1 vs all
acks=all (safest)

All ISR replicas must acknowledge. No data loss even if leader fails. Latency: +10–50ms (waits for follower replication). Use for: payments, orders, any transactional data where losing a message is unacceptable.

acks=1 (default)

Only leader acknowledges. Fast (+1ms). Risk: if leader crashes before replication, message lost. Acceptable for: logs, non-critical notifications, cases where slight data loss is OK.

acks=0 (fire-and-forget)

No acknowledgment. Fastest possible. No durability guarantee at all. Use only for: high-volume metrics, analytics events where some loss is acceptable and throughput is everything.

🔁 Delivery Semantics: At-Most-Once vs At-Least-Once vs Exactly-Once
At-Most-Once

Commit offset BEFORE processing. If crash occurs, message is skipped — never processed. Use for: ephemeral metrics, low-value events where loss is acceptable. Never for transactions.

✅ At-Least-Once (recommended)

Commit offset AFTER processing. On crash, messages are reprocessed. Make processing idempotent. Industry default for most Kafka pipelines. Low complexity, high reliability, works for 95% of use cases.

Exactly-Once (EOS)

Kafka Transactions API. Producer + consumer offset committed atomically. 2–3× throughput reduction. High complexity. Only for: financial ledgers, inventory counts, de-duplication-critical pipelines where idempotent processing is impossible.

📊 Partition Count: Few vs Many
✅ Start with 3× expected peak consumers

Rule: partitions = 3× peak consumer count. Gives headroom to scale consumers 3× without adding partitions. Each partition maps to one file handle, one thread. Start conservative — easier to add partitions than to remove (adding re-routes key-partitioned data).

Too few partitions

Bottleneck if consumer count approaches partition count. Can't increase parallelism without repartitioning. For a topic with 3 partitions and 3 consumers at 100% CPU: only option is add partitions (disruptive) or optimise each consumer.

Too many partitions

More OS file handles, more memory for ISR tracking, slower rebalancing on consumer failure. Kafka recommends 4000 partitions per broker max. At 200K partitions cluster-wide, ZooKeeper/KRaft becomes a bottleneck. Over-partitioning is a common mistake in early designs.

🆚 Kafka vs RabbitMQ vs SQS
✅ Kafka — high throughput + replay

Append-only log. Messages retained for days/weeks. Multiple consumer groups read independently. Replay from offset 0. Best for: event sourcing, analytics pipelines, audit logs, stream processing. 1M+ msgs/sec per cluster.

RabbitMQ — routing flexibility

Push-based. Rich routing (exchanges, bindings). Messages deleted on ACK. Best for: task queues, RPC-style patterns, complex routing rules. Lower throughput (~50K msgs/sec). Easier to set up for simple use cases.

SQS — managed, simple

AWS fully managed. No servers to operate. At-least-once. DLQ built-in. Best for: simple task queues, Lambda triggers, low-ops teams. Limited: 10 msgs per batch, no ordering (FIFO queue has limits), no replay, no consumer groups. Not for analytics pipelines.
