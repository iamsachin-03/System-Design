Web Crawler
Design a distributed web crawler capable of crawling 1 billion pages/day — URL frontier management, BFS traversal, politeness policies (robots.txt, rate limiting), content deduplication, and recrawl scheduling.

✅ Functional Requirements
CORE
URL Frontier: Maintain a priority queue of URLs to crawl next. Prioritise by page importance (PageRank-like) and freshness requirements.
Fetch: Download HTML content of each URL. Handle HTTP redirects, timeouts, and errors gracefully.
Parse: Extract all hyperlinks from downloaded HTML. Normalise and canonicalise extracted URLs.
Deduplicate URLs: Never crawl the same URL twice in a crawl cycle. Use Bloom Filter for memory-efficient tracking.
Content Deduplication: Detect duplicate or near-duplicate content (mirror pages, syndicated articles) using content fingerprinting.
Politeness: Respect robots.txt (disallow rules). Enforce minimum crawl delay per domain. Avoid overwhelming any single server.
Recrawl: Periodically revisit already-crawled pages based on their change frequency and importance.
OUT OF SCOPE
JavaScript rendering (Headless Chrome) · Image crawling · Search index building · PageRank computation · Malware detection · Login-required pages

🎙 Phase 1 — Architecture & Hard Problems
PHASE 1 · SCOPE
IR
INTERVIEWER
Design a web crawler. What are the core challenges?
ME
YOU
A web crawler has five hard sub-problems worth naming upfront:

1. URL Frontier management: The web has ~60 billion pages. We can't hold all pending URLs in memory. The frontier must be persisted, prioritised, and efficiently queued for billions of URLs.

2. URL deduplication: Any page links to other pages. Naively, a crawler would revisit the same URL thousands of times as multiple pages link to it. We need to track which URLs have been seen — efficiently for billions of URLs with minimal memory.

3. Politeness: We can't hammer servers. We need to throttle requests per domain, respect robots.txt, and avoid getting our IP blocked. This limits throughput — every domain gets at most 1 req/sec from us.

4. Content deduplication: Many pages contain identical or near-identical content (mirrors, republished articles). Processing duplicates wastes storage and compute.

5. Recrawl scheduling: Pages change over time. News pages change hourly. Static pages rarely change. We need to recrawl the right pages at the right frequency.

The key data structures are: Priority Queue (URL frontier), Bloom Filter (URL dedup), SimHash (content dedup), and Per-domain politeness queues.
💡 Structure first. Naming the five problems immediately shows you've thought deeply about crawlers. Most candidates just say "download pages, extract links, repeat" — missing the hard parts entirely.
🌐 Phase 2 — URL Frontier & Priority Queue
PHASE 2 · URL FRONTIER
IR
INTERVIEWER
Design the URL frontier. There are 100 million pending URLs. How do you manage them?
ME
YOU
The URL frontier has two conflicting requirements: priority (crawl important pages first) and politeness (don't hammer a single domain). These pull in different directions — a simple priority queue would try to crawl all of Wikipedia simultaneously.

Two-tier frontier architecture:

Front Queue (Priority-based):
Multiple FIFO queues, each for a priority level (e.g., High/Medium/Low based on PageRank of the page). A Prioritizer assigns incoming URLs to queues based on importance. A Selector picks the next queue to draw from, weighted toward higher-priority queues.

Back Queue (Politeness-based):
One queue per domain. URLs from the same domain always go to the same back queue. Each back queue has a last-crawled timestamp and a crawl-delay (from robots.txt or default 1 req/sec). Worker threads pick from back queues whose delay has elapsed.

Mapping: Front → Back is done by a Dispatcher: pops URL from front priority queue, routes to the appropriate domain back queue. Back queues are stored in Redis (domain → sorted set of URLs by priority).

Scale: 100M pending URLs in Redis: 100 bytes per URL × 100M = 10 GB. Manageable. Back queues expire domain entries after crawl window to prevent memory bloat.
IR
INTERVIEWER
How do you handle a domain with millions of URLs (e.g., Wikipedia with 60M articles)?
ME
YOU
Wikipedia gets ~60M URLs but only 1 req/sec politeness limit. At that rate, a full Wikipedia crawl takes ~694 days. Clearly we can't put all 60M in one back queue.

Solution: Domain shard limits. Each domain gets a back queue with a cap of N URLs (e.g., 10,000). New Wikipedia URLs are sampled: only accept a URL if hash(url) % sampling_rate == 0. This ensures we crawl a representative sample of Wikipedia rather than trying to crawl everything.

Alternatively, for major domains, pre-configure a higher concurrency allowance (Wikipedia explicitly allows multiple crawlers in their robots.txt). Google Googlebot gets special treatment.
🔍 Phase 3 — URL Deduplication with Bloom Filter
PHASE 3 · DEDUPLICATION
IR
INTERVIEWER
How do you efficiently check if a URL has already been crawled? You have 1 billion URLs.
ME
YOU
Naïve approach: HashSet of all crawled URLs. 1 billion × 100 bytes = 100 GB just for the set. Too much RAM.

Solution: Bloom Filter
A Bloom filter is a probabilistic data structure: it can definitively say "URL has NOT been seen" (no false negatives) or "URL has PROBABLY been seen" (small false positive rate).

How it works:
  1. A bit array of M bits, initially all 0
  2. K independent hash functions
  3. To add a URL: compute K hash positions, set those M bits to 1
  4. To check: compute K hash positions, if ALL are 1 → "probably seen". If ANY is 0 → "definitely not seen"
Math for our scale:
For 1 billion URLs with 1% false positive rate:
M = -n × ln(p) / ln(2)² = -1B × ln(0.01) / 0.48 = ~9.6 billion bits = ~1.2 GB
K = M/n × ln(2) = 9.6 × ln(2) = ~6.7 → 7 hash functions

1.2 GB fits in RAM for 1 billion URLs with 1% false positive rate. Compare to 100 GB for a HashSet. 80× more memory efficient.

False positive impact: A 1% false positive means 1 in 100 new URLs is incorrectly skipped. For a crawler, this is acceptable — missing 1% of new URLs doesn't significantly hurt crawl coverage.
IR
INTERVIEWER
Bloom filters can't delete entries. What if a URL needs to be recrawled?
ME
YOU
Good catch. Recrawl is handled separately from the "URL seen" Bloom filter. The Bloom filter tracks URLs that have ever entered the frontier — not "currently active in the frontier". Two solutions:

Option 1 — Separate recrawl tracking: The Bloom filter prevents re-adding a URL to the frontier. A separate recrawl scheduler maintains a list of already-crawled URLs with their next recrawl timestamp. When a recrawl is due, the scheduler adds the URL back to the frontier, and we do not check the Bloom filter for scheduled recrawls (they bypass dedup).

Option 2 — Time-windowed Bloom filter: Reset the Bloom filter periodically (e.g., daily). Within a day's crawl window, deduplicate with the filter. When the daily window ends, reset and start fresh. Each URL is recrawled once per day at most — fine for news sites.

Option 2 is simpler operationally. Option 1 gives more control for variable recrawl frequencies.
🤝 Phase 4 — Politeness: robots.txt & Rate Limiting
PHASE 4 · POLITENESS
IR
INTERVIEWER
How do you implement politeness — robots.txt, crawl delay, and not getting blocked?
ME
YOU
Politeness has three components:

1. robots.txt compliance:
Before crawling any domain, fetch https://domain.com/robots.txt. Parse the Disallow rules for User-agent: * and our specific User-Agent. Cache the parsed rules per domain in Redis (TTL: 24 hours). Before fetching any URL, check if it matches any Disallow rule — skip if so.

2. Crawl-delay enforcement:
robots.txt may specify Crawl-delay: 10 (seconds between requests). Store last_crawl_time[domain] in Redis. Before fetching, check if enough time has elapsed. If not, put the URL back in the domain queue and pick a different domain.

3. Per-domain rate limiting:
Our default: 1 request per second per domain. This is implemented per worker: worker thread picks a URL from a domain queue, fetches it, sleeps for max(crawl_delay, 1) seconds before picking another URL from the same domain. During that sleep, the worker can fetch URLs from other domains.

4. Respect HTTP response codes:
- 429 Too Many Requests → back off exponentially, slow down crawl for that domain
- 503 Service Unavailable → re-queue URL, try again in 1 hour
- 404 Not Found → discard URL, remove from frontier
- 301/302 Redirect → follow redirect, add redirect target to frontier (check Bloom filter first)
🧪 Phase 5 — Content Dedup, DNS, & Traps
PHASE 5 · EDGE CASES
IR
INTERVIEWER
How do you detect duplicate content? Two pages with the same article but different URLs.
ME
YOU
Exact deduplication: Compute MD5/SHA-1 of the downloaded content. Store hash in a seen-hashes set. If hash exists → discard (duplicate). This catches exact duplicates (e.g., printer-friendly version of a page).

Near-duplicate detection (SimHash): Many pages are 90% similar (translated pages, lightly edited syndicated content). SimHash produces a 64-bit fingerprint where similar documents produce similar fingerprints (hamming distance <3 = near-duplicate).
How SimHash works:
1. Tokenise page into shingles (n-grams of words)
2. For each shingle, compute a 64-bit hash
3. For each bit position: sum +1 if hash has bit=1, -1 if bit=0, weighted by word frequency
4. Final SimHash: bit[i] = sign(sum[i])
Similar pages → similar SimHash → small hamming distance → mark as near-duplicate.
IR
INTERVIEWER
What's a crawler trap and how do you prevent infinite loops?
ME
YOU
Crawler traps are URLs that infinitely generate new unique URLs, trapping a naive crawler in an infinite loop:
  • Dynamic calendars: clicking "next month" generates /cal?month=1&year=2026, /cal?month=2... → infinitely
  • Session IDs in URLs: /page?sessionid=abc123 generates unique URLs per visitor
  • Infinite parameter permutations: /search?sort=date&order=asc&page=1, &page=2, etc.
Prevention strategies:
  • URL depth limit: Never crawl pages more than N path segments deep (e.g., max depth 10). Most legitimate content is accessible within 5 hops from the homepage.
  • URL canonicalisation: Strip session IDs, tracking parameters. /page?sessionid=abc&utm_source=x → /page. Many different URLs collapse to the same canonical form → Bloom filter catches them.
  • Domain page limit: Max 100,000 pages per domain per crawl cycle. Prevents one site from consuming unlimited crawler resources.
  • Content similarity threshold: If 50+ pages from the same domain are near-duplicates, the domain is likely a trap. Stop crawling it.
IR
INTERVIEWER
What about DNS? Crawling 11,574 pages/sec means 11,574 DNS lookups/sec minimum.
ME
YOU
DNS is a critical bottleneck. Public DNS resolvers have rate limits and latency (~20-100ms per lookup). At 11,574 pages/sec, naïve DNS would add 200ms of latency to every fetch.

Solution: Local DNS cache
Each crawler worker maintains a local DNS cache (TTL 30 minutes): domain → IP. First visit to a domain: DNS lookup. Subsequent visits (up to 30 min): serve from cache. Given that we're crawling 1B pages from a much smaller number of domains (~100M unique domains), cache hit rate is very high.

Additionally: use a local recursive DNS resolver (unbound/bind) on the crawler cluster. Keep it warm with frequent preloaded domains. This gives sub-millisecond DNS resolution for known domains.

// ── 1. Bloom Filter ───────────────────────────────────
public class BloomFilter {

    private final long[]   bits;
    private final int    numHashFunctions;
    private final long   bitArraySize;

    /**
     * @param expectedInsertions 1 billion for our crawler
     * @param falsePositiveRate  0.01 (1%)
     */
    public BloomFilter(long expectedInsertions, double falsePositiveRate) {
        // Optimal bit array size: m = -n*ln(p) / ln(2)^2
        this.bitArraySize = (long) (-expectedInsertions
            * Math.log(falsePositiveRate) / (Math.log(2) * Math.log(2)));
        // Optimal hash functions: k = m/n * ln(2)
        this.numHashFunctions = (int) Math.round(
            (double) bitArraySize / expectedInsertions * Math.log(2));
        // Allocate bit array (long array where each long = 64 bits)
        this.bits = new long[(int) (bitArraySize / 64) + 1];
    }

    public void add(String url) {
        for (int i = 0; i < numHashFunctions; i++) {
            long pos = hash(url, i) % bitArraySize;
            bits[(int)(pos / 64)] |= (1L << (pos % 64));
        }
    }

    public boolean mightContain(String url) {
        for (int i = 0; i < numHashFunctions; i++) {
            long pos = hash(url, i) % bitArraySize;
            if ((bits[(int)(pos / 64)] & (1L << (pos % 64))) == 0) {
                return false; // definitely not seen
            }
        }
        return true; // probably seen (small false positive rate)
    }

    private long hash(String url, int seed) {
        // MurmurHash3: fast, good distribution
        return Math.abs((long) (seed * 31 + url.hashCode()) * 0x9e3779b97f4a7c15L);
    }
}

// ── 2. URL Frontier ───────────────────────────────────
@Service
public class UrlFrontier {

    @Autowired private RedisTemplate<String,String> redis;
    private final BloomFilter bloomFilter = new BloomFilter(1_000_000_000L, 0.01);
    private static final int ACK_TIMEOUT_SEC = 300; // 5 min

    public void addUrl(String rawUrl, Priority priority) {
        String url = canonicalise(rawUrl);
        if (url == null || !isValidUrl(url)) return;

        // Dedup: only add if Bloom filter says "not seen"
        if (bloomFilter.mightContain(url)) return;

        bloomFilter.add(url);
        String domain = extractDomain(url);

        // Domain back queue: sorted set, score = priority (lower = higher)
        redis.opsForZSet().add(
            "frontier:back:" + domain,
            url,
            (double) priority.ordinal()
        );
    }

    public List<CrawlTask> nextBatch(String workerId, int count) {
        List<CrawlTask> tasks = new ArrayList<>();

        // Get domains with URLs ready to crawl (respecting delay)
        Set<String> readyDomains = getReadyDomains(count);
        for (String domain : readyDomains) {
            // Get highest-priority URL for this domain
            Set<String> urls = redis.opsForZSet()
                .range("frontier:back:" + domain, 0, 0);
            if (urls == null || urls.isEmpty()) continue;

            String url     = urls.iterator().next();
            String taskId  = UUID.randomUUID().toString();

            // Remove from domain queue
            redis.opsForZSet().remove("frontier:back:" + domain, url);

            // Track in-progress (for fault tolerance)
            redis.opsForHash().put("frontier:inprogress", taskId,
                url + "|" + workerId + "|" + System.currentTimeMillis());

            // Update domain last-crawl time
            redis.opsForValue().set("frontier:domain_ts:" + domain,
                String.valueOf(System.currentTimeMillis()),
                Duration.ofSeconds(2));

            tasks.add(new CrawlTask(taskId, url, ACK_TIMEOUT_SEC));
        }
        return tasks;
    }

    private Set<String> getReadyDomains(int count) {
        // Get domains with back queues, filter out those still in delay period
        Set<String> allDomains = redis.keys("frontier:back:*");
        return allDomains.stream()
            .map(k -> k.substring("frontier:back:".length()))
            .filter(this::isDomainReady)
            .limit(count)
            .collect(Collectors.toSet());
    }

    private boolean isDomainReady(String domain) {
        String lastTs = redis.opsForValue().get("frontier:domain_ts:" + domain);
        if (lastTs == null) return true; // never crawled
        long elapsed = System.currentTimeMillis() - Long.parseLong(lastTs);
        return elapsed >= 1000; // 1 req/sec default
    }

    private String canonicalise(String rawUrl) {
        try {
            URI uri = new URI(rawUrl.trim().toLowerCase());
            // Strip fragment (#section), tracking params, session IDs
            String path   = uri.getPath() != null ? uri.getPath() : "/";
            String query  = cleanQueryString(uri.getQuery());
            return uri.getScheme() + "://" + uri.getHost() + path
                + (query.isEmpty() ? "" : "?" + query);
        } catch (Exception e) { return null; }
    }

    /** Reassign tasks that workers failed to complete in time */
    @Scheduled(fixedDelay = 60_000)
    public void reassignStaleTasks() {
        long staleThreshold = System.currentTimeMillis()
            - (ACK_TIMEOUT_SEC * 1000L);
        redis.opsForHash().entries("frontier:inprogress")
            .forEach((taskId, value) -> {
                String[] parts = ((String) value).split("\\|");
                long startedAt = Long.parseLong(parts[2]);
                if (startedAt < staleThreshold) {
                    // Re-add URL to frontier and remove from in-progress
                    addUrl(parts[0], Priority.MEDIUM);
                    redis.opsForHash().delete("frontier:inprogress", taskId);
                }
            });
    }
}

// ── 3. Crawler Worker ─────────────────────────────────
@Component
public class CrawlerWorker implements Runnable {

    private final AsyncHttpClient   httpClient;
    private final UrlFrontier       frontier;
    private final RobotsParser      robotsParser;
    private final ContentStore      contentStore;
    private final Map<String,InetAddress> dnsCache;
    private final String             workerId;

    public void run() {
        while (true) {
            List<CrawlTask> tasks = frontier.nextBatch(workerId, 10);
            if (tasks.isEmpty()) {
                Thread.sleep(1000); continue;
            }
            // Fetch all tasks concurrently
            tasks.forEach(task -> CompletableFuture.runAsync(
                () -> processTask(task)));
        }
    }

    private void processTask(CrawlTask task) {
        String url    = task.getUrl();
        String domain = extractDomain(url);

        // Check robots.txt
        if (!robotsParser.isAllowed(url, "MyCrawler/1.0")) {
            frontier.complete(task.getTaskId(), CrawlResult.robotsDisallowed());
            return;
        }

        try {
            Response response = fetchWithDnsCache(url);

            if (response.getStatusCode() == 200) {
                String html        = response.getResponseBody();
                String contentMd5  = md5(html);

                // Exact content dedup
                if (!contentStore.isContentSeen(contentMd5)) {
                    String s3Path = contentStore.store(url, html);
                    Set<String> links = extractLinks(html, url);
                    frontier.complete(task.getTaskId(),
                        CrawlResult.success(s3Path, contentMd5, links));
                } else {
                    frontier.complete(task.getTaskId(),
                        CrawlResult.duplicateContent(contentMd5));
                }
            } else if (response.getStatusCode() == 429) {
                // Back off: double the delay for this domain
                frontier.backoffDomain(domain);
                frontier.requeue(task);
            } else if (response.getStatusCode() >= 400) {
                frontier.complete(task.getTaskId(),
                    CrawlResult.error(response.getStatusCode()));
            }
        } catch (TimeoutException e) {
            frontier.complete(task.getTaskId(), CrawlResult.timeout());
        }
    }
}

import asyncio, aiohttp, hashlib, mmh3, re, time
from urllib.parse import urlparse, urljoin, urlencode, parse_qs
from typing import Optional, Set, List
from bs4 import BeautifulSoup


# ── 1. Bloom Filter ───────────────────────────────────
class BloomFilter:
    """
    Memory-efficient URL deduplication.
    1B URLs at 1% FPR → 1.2 GB RAM vs 100 GB for HashSet.
    """
    def __init__(self, expected_insertions: int = 1_000_000_000,
                  false_positive_rate: float = 0.01):
        # Optimal bit array size
        self.m = int(-expected_insertions * (false_positive_rate.__class__(1)
                  and __import__('math').log(false_positive_rate))
                  / (__import__('math').log(2) ** 2))
        # Optimal number of hash functions
        self.k = round(self.m / expected_insertions * __import__('math').log(2))
        # Bit array as bytearray (each byte = 8 bits)
        self.bits = bytearray(self.m // 8 + 1)

    def add(self, url: str) -> None:
        for i in range(self.k):
            pos = mmh3.hash(url, i, signed=False) % self.m
            self.bits[pos // 8] |= (1 << (pos % 8))

    def might_contain(self, url: str) -> bool:
        """False = definitely not seen. True = probably seen (1% FPR)."""
        for i in range(self.k):
            pos = mmh3.hash(url, i, signed=False) % self.m
            if not (self.bits[pos // 8] & (1 << (pos % 8))):
                return False  # definitely new
        return True  # probably seen


# ── 2. URL Canonicaliser ──────────────────────────────
class UrlCanonicaliser:
    """Normalise URLs for consistent deduplication."""

    # Query parameters to strip (tracking, session)
    STRIP_PARAMS = {'utm_source', 'utm_medium', 'utm_campaign',
                    'sessionid', 'PHPSESSID', 'sid', 'fbclid', 'gclid'}

    def canonicalise(self, url: str, base_url: str = None) -> Optional[str]:
        try:
            # Resolve relative URLs
            if base_url: url = urljoin(base_url, url)

            parsed = urlparse(url.strip().lower())

            # Only crawl HTTP/HTTPS
            if parsed.scheme not in ('http', 'https'):
                return None

            # Strip fragment (#section = client-side only, same page)
            # Clean query params (remove tracking, sessions)
            params = parse_qs(parsed.query, keep_blank_values=True)
            clean  = {k: v for k, v in params.items()
                       if k not in self.STRIP_PARAMS}
            query  = urlencode(sorted(clean.items()))  # sort for consistency

            path = parsed.path.rstrip('/') or '/'
            return f"{parsed.scheme}://{parsed.netloc}{path}" + (
                f"?{query}" if query else "")
        except Exception:
            return None


# ── 3. Async Crawler Worker ───────────────────────────
class CrawlerWorker:
    """
    Async crawler: 10 concurrent HTTP connections per worker.
    At 500ms avg latency → 20 pages/sec per worker.
    1,000 workers → 20,000 pages/sec > our target of 11,574.
    """
    MAX_CONCURRENT = 10
    USER_AGENT     = "MyCrawler/1.0 (+https://example.com/bot)"
    TIMEOUT_SEC    = 30
    MAX_DEPTH      = 10  # trap prevention

    def __init__(self, worker_id: str, frontier, robots_parser,
                  content_store, bloom_filter, canonicaliser):
        self.worker_id     = worker_id
        self.frontier      = frontier
        self.robots        = robots_parser
        self.content_store = content_store
        self.bloom         = bloom_filter
        self.canonicaliser = canonicaliser
        self.dns_cache     = {}  # simple TTL dict
        self.semaphore     = asyncio.Semaphore(self.MAX_CONCURRENT)

    async def run(self):
        """Main crawl loop"""
        async with aiohttp.ClientSession(
            headers={"User-Agent": self.USER_AGENT},
            timeout=aiohttp.ClientTimeout(total=self.TIMEOUT_SEC)
        ) as session:
            self.session = session
            while True:
                tasks = await self.frontier.next_batch(self.worker_id, count=10)
                if not tasks:
                    await asyncio.sleep(1)
                    continue
                # Process all 10 tasks concurrently
                await asyncio.gather(*[self._process(t) for t in tasks],
                                      return_exceptions=True)

    async def _process(self, task: dict):
        url    = task["url"]
        domain = urlparse(url).netloc

        # 1. Politeness: check robots.txt
        if not await self.robots.is_allowed(url, self.USER_AGENT):
            await self.frontier.complete(task["task_id"], "ROBOTS_DISALLOWED")
            return

        # 2. Depth trap prevention
        depth = url.count('/') - 2
        if depth > self.MAX_DEPTH:
            await self.frontier.complete(task["task_id"], "DEPTH_EXCEEDED")
            return

        async with self.semaphore:
            try:
                async with self.session.get(url,
                        allow_redirects=True, max_redirects=5) as resp:

                    if resp.status == 200:
                        html         = await resp.text(errors='ignore')
                        content_hash = hashlib.md5(html.encode()).hexdigest()

                        # 3. Content dedup
                        if not await self.content_store.is_seen(content_hash):
                            s3_path = await self.content_store.store(
                                url, html, content_hash)

                            # 4. Extract + queue new links
                            links = self._extract_links(html, url)
                            new_links = [
                                l for l in links
                                if not self.bloom.might_contain(l)
                            ]
                            for link in new_links:
                                self.bloom.add(link)

                            await self.frontier.complete(
                                task["task_id"], "SUCCESS",
                                extracted_urls=new_links,
                                content_hash=content_hash,
                                storage_path=s3_path)
                        else:
                            await self.frontier.complete(
                                task["task_id"], "DUPLICATE_CONTENT")

                    elif resp.status == 429:
                        # Rate limited: back off this domain
                        await self.frontier.backoff_domain(domain, factor=2)
                        await self.frontier.requeue(task)

                    else:
                        await self.frontier.complete(
                            task["task_id"], f"ERROR_{resp.status}")

            except asyncio.TimeoutError:
                await self.frontier.complete(task["task_id"], "TIMEOUT")

    def _extract_links(self, html: str, base_url: str) -> List[str]:
        """Extract and canonicalise all links from HTML."""
        soup  = BeautifulSoup(html, 'html.parser')
        links = set()
        for tag in soup.find_all('a', href=True):
            canonical = self.canonicaliser.canonicalise(tag['href'], base_url)
            if canonical:
                links.add(canonical)
        return list(links)
⚖️ Design Tradeoffs
Crawlers make inherently tension-filled tradeoffs between completeness (crawl everything), politeness (don't hurt web servers), and freshness (keep content current).
🔍 URL Deduplication: Bloom Filter vs HashSet vs DB
✅ Bloom Filter

1.2 GB for 1B URLs at 1% FPR. No false negatives (never misses a seen URL). 1% false positives (skips ~1% of genuinely new URLs). 80× more memory-efficient than a HashSet. Industry standard for web crawlers (Google, Common Crawl).

❌ In-Memory HashSet

Zero false positives. But 100 GB for 1B URLs — doesn't fit in RAM on a single machine. Would need Redis cluster, adding 1ms latency per URL check. At 11,574 URLs/sec extracted, that's 11,574 Redis lookups/sec. Expensive.

⚠️ Database (Cassandra)

Persistent, can handle any scale. But ~5ms per lookup at 11,574 checks/sec = 57 seconds of DB time per second of crawling — the DB becomes a bottleneck. Use DB for recrawl scheduling (infrequent), not for hot-path URL dedup.

📋 Crawl Ordering: BFS vs DFS vs Priority Queue
✅ Priority Queue (best-first)

Pages ranked by importance (PageRank estimate, link count). High-quality pages crawled first. If crawl is interrupted, the important web has been indexed. The two-tier frontier (priority + politeness queues) is this approach.

⚠️ Breadth-First Search (BFS)

Start from seeds, explore level by level. Naturally discovers "hub" pages first (popular pages get many links → found early). Simple. But doesn't prioritise by quality — wastes time on low-value pages in early rounds.

❌ Depth-First Search (DFS)

Goes deep into one site before moving to others. Terrible for a general crawler: gets stuck crawling all 60M Wikipedia articles before moving to other domains. Also very impolite (hammers one server). Never use for general-purpose crawling.

⏱ Recrawl Strategy: Fixed Interval vs Adaptive vs On-Demand
✅ Adaptive (change frequency-based)

Track how often each page changes between crawls. High-change pages (news) get shorter recrawl intervals. Low-change pages (Wikipedia stable articles) get longer intervals. Maximises freshness per unit crawl bandwidth. Google uses this.

⚠️ Fixed Interval

Every page recrawled every N hours regardless. Simple. But wastes resources on static pages that never change, while potentially missing rapid changes on news sites. Only sensible for small focused crawlers.

💡 Sitemap-based recrawl

Many sites publish XML sitemaps with <lastmod> and <changefreq> hints. Excellent signal for recrawl scheduling — the site tells you directly when it was last modified. Use sitemap data to seed recrawl scheduler instead of guessing from crawl diffs.

🚀 Fetch Concurrency: Async I/O vs Threads vs Processes
✅ Async I/O (aiohttp / Netty)

Single thread, thousands of concurrent HTTP connections. 99% of crawler time is waiting for network — async I/O perfectly matches this pattern. 10 concurrent connections × 1,000 workers = 10,000 concurrent HTTP requests with minimal CPU overhead.

⚠️ Thread Pool

1 thread per connection. With 500ms avg latency and 1,000 req/sec target per machine, you need 500 threads per machine. Context switching overhead. Memory overhead (~1MB stack per thread). Works but less efficient than async I/O.

❌ Process per URL

Fork a new process per URL. Maximum isolation but catastrophic overhead — process creation is 10,000× more expensive than async context switch. Fork overhead alone would dominate crawl time. Never for high-throughput crawlers.
