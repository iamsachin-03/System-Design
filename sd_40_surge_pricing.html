⚡
Surge Pricing Engine
Design Uber's surge pricing system — computing real-time supply/demand ratios across thousands of geospatial zones, applying a piecewise pricing function that brings supply and demand into equilibrium, serving the current multiplier to riders in under 50ms, and handling the thundering herd after a concert ends where 10,000 simultaneous ride requests spike demand in a single hex cell.
🔴 MUST DO 🟢 Medium H3 Hexagons Supply/Demand Streaming (Flink) Pricewise Function Redis TTL Market Clearing
🎯 Requirements
Surge pricing is an equilibrium mechanism, not just a money-grab. The goal is to bring supply (drivers) and demand (riders) into balance in a specific geographic zone. Higher prices (a) suppress some demand (price-sensitive riders cancel), and (b) attract more supply (nearby drivers move into the zone). The system must compute this in real time across thousands of zones simultaneously.

FUNCTIONAL
Compute supply (available drivers) per geographic zone in real time
Compute demand (active ride requests) per zone in real time
Apply pricing function: demand/supply ratio → multiplier
Serve current multiplier for a user's location in <50ms
Display surge map to drivers (which zones have highest multipliers)
Multipliers update every 60 seconds per zone
NON-FUNCTIONAL
50K concurrent zones worldwide (cities active at any time)
Multiplier read latency: <10ms (from Redis)
Multiplier freshness: updated within 60s of supply/demand change
Handle spike: 10K concurrent requests in single zone (post-concert)
Graceful degradation: if surge service unavailable, return multiplier=1.0
💬 Phase 1 — Computing Surge
PHASE 1 · SUPPLY & DEMAND
IR
INTERVIEWER
How do you compute the surge multiplier for a specific geographic zone? Walk me through the exact data flow.
ME
YOU
Supply computation: Every driver sends a GPS heartbeat every 4 seconds. The Location Service converts the GPS coordinate to an H3 hex cell (resolution 6, ~36 km² per cell — coarser than the res-8 cells used for driver matching). Each heartbeat updates a counter in Redis: INCR supply:{h3_cell_r6} EX 120. The count is the number of available drivers in that cell. TTL of 120s ensures drivers who stop heartbeating expire naturally.

Demand computation: Every ride request event (from the ride request service) is published to Kafka. A Flink streaming job consumes this stream. For each request event: extract H3 cell (res 6) of the pickup location. Maintain a tumbling 5-minute window counter per cell. Write to Redis: SETEX demand:{h3_cell_r6} 60 {count}. Count = ride requests in the last 5 minutes in this cell.

Multiplier computation (every 60 seconds): A Flink job reads supply and demand from Redis for all active cells, computes ratio = demand / max(supply, 1), applies the pricing function, writes multiplier back to Redis:
SET surge:{h3_cell_r6} {multiplier} EX 120

Pricing function (piecewise):
ratio < 1.2 → multiplier = 1.0 (no surge)
ratio 1.2–2.0 → multiplier = 1.0 + (ratio-1.2) × 0.625 (linear from 1.0→1.5)
ratio 2.0–4.0 → multiplier = 1.5 + (ratio-2.0) × 0.25 (linear from 1.5→2.0)
ratio > 4.0 → multiplier = min(4.0, 2.0 + (ratio-4.0) × 0.1) (diminishing returns)

The cap at 4.0× is a business decision (PR sensitivity) not a technical constraint.
💡 Why cap the multiplier? Uncapped surge can reach 10× or more (demonstrated during emergencies). This creates a PR disaster and regulatory pushback. Uber caps at ~4× in practice. The real goal is equilibrium — at 2-3× most supply-demand imbalances resolve as drivers flood into high-surge zones.
🎸 Phase 2 — The Concert Thundering Herd
PHASE 2 · SPIKE HANDLING
IR
INTERVIEWER
A sold-out concert just ended. 10,000 people simultaneously request rides from the same 4-block radius. Walk me through what happens second by second.
ME
YOU
T+0s: 10,000 ride request events hit the ride request service simultaneously. These are published to Kafka partitioned by H3 cell ID — all 10,000 land in the same partition for the arena's cell. Kafka absorbs the spike without loss.

T+2s: The Flink streaming job's tumbling window for this cell now shows demand=10,000. Supply (available drivers in the cell before the concert ended) = ~50. Ratio = 10,000/50 = 200. Multiplier = 4.0× (capped). Flink writes SET surge:{arena_cell} 4.0 EX 120 to Redis.

T+5s: Every new ride request for this location checks GET surge:{arena_cell} → 4.0×. Shows the 4× multiplier in the app. Some price-sensitive riders cancel. Demand drops to 7,000. The app also displays the surge map to nearby drivers, who start driving toward the arena.

T+60s: Next Flink computation: demand = 7,000 (some cancelled), supply = 300 (drivers attracted by 4× multiplier). Ratio = 7,000/300 = 23. Multiplier = 4.0× (still capped, ratio very high).

T+10min: Supply = 2,000 (drivers converged from 5km radius), demand = 4,000 (half the crowd got rides or found alternatives). Ratio = 2.0. Multiplier = 1.5×. Equilibrium approaching.

T+30min: Ratio = 0.8. Surge = 1.0×. Concert traffic fully absorbed.

Key design detail: The rider-facing price shown is the multiplier at time of request lock-in, not the current multiplier. If you request at 4.0× and a driver accepts 90 seconds later when surge is 1.5×, you still pay 4.0× (you locked in at that price). This prevents bait-and-switch and is guaranteed by storing the locked multiplier with the trip in Postgres.

// ══════════════════════════════════════════════════════
// Surge Pricing — Java: Flink + Pricing Function + API
// ══════════════════════════════════════════════════════

// ── Pricing function ─────────────────────────────────
public class PricingFunction {

    private static final double CAP            = 4.0;
    private static final double THRESHOLD       = 1.2;
    private static final double TIER1_MAX_RATIO = 2.0;
    private static final double TIER2_MAX_RATIO = 4.0;

    public static double compute(long demand, long supply) {
        double ratio = demand / (double) Math.max(supply, 1);

        if (ratio <= THRESHOLD)        return 1.0;
        if (ratio <= TIER1_MAX_RATIO)  return 1.0 + (ratio - THRESHOLD) * 0.625;
        if (ratio <= TIER2_MAX_RATIO)  return 1.5 + (ratio - TIER1_MAX_RATIO) * 0.25;
        return Math.min(CAP, 2.0 + (ratio - TIER2_MAX_RATIO) * 0.1);
    }
}

// ── Flink Surge Computation Job ───────────────────────
public class SurgeComputationJob {

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

        // Supply stream: driver location heartbeats
        DataStream<DriverLocation> supply = env
            .addSource(new FlinkKafkaConsumer<>("driver-locations", ...))
            .filter(DriverLocation::isAvailable);

        // Supply per H3 cell — 60-second tumbling window
        DataStream<Tuple2<String,Long>> supplyPerCell = supply
            .keyBy(loc -> loc.getH3CellR6())
            .window(TumblingProcessingTimeWindows.of(Time.seconds(60)))
            .aggregate(new CountAggregate<>());

        // Demand stream: ride requests
        DataStream<RideRequest> demand = env
            .addSource(new FlinkKafkaConsumer<>("ride-requests", ...));

        // Demand per H3 cell — 5-minute tumbling window
        DataStream<Tuple2<String,Long>> demandPerCell = demand
            .keyBy(req -> req.getPickupH3CellR6())
            .window(TumblingProcessingTimeWindows.of(Time.minutes(5)))
            .aggregate(new CountAggregate<>());

        // Join supply + demand on cell ID, compute surge multiplier
        supplyPerCell
            .keyBy(t -> t.f0)
            .intervalJoin(demandPerCell.keyBy(t -> t.f0))
            .between(Time.minutes(-5), Time.minutes(0))
            .process((ProcessJoinFunction<Tuple2<String,Long>,Tuple2<String,Long>,SurgeResult>)
                (supplyTuple, demandTuple, ctx, out) -> {
                    String cell        = supplyTuple.f0;
                    double multiplier   = PricingFunction.compute(demandTuple.f1, supplyTuple.f1);
                    out.collect(new SurgeResult(cell, multiplier, demandTuple.f1, supplyTuple.f1));
                })
            .addSink(new RedisSurgeSink()); // writes SET surge:{cell} {mult} EX 120

        env.execute("SurgeComputation");
    }
}

// ── Redis Surge Sink ──────────────────────────────────
public class RedisSurgeSink implements SinkFunction<SurgeResult> {
    private JedisPool jedisPool;

    @Override
    public void invoke(SurgeResult result, Context ctx) {
        try (Jedis jedis = jedisPool.getResource()) {
            jedis.setex("surge:" + result.getCell(), 120, String.valueOf(result.getMultiplier()));
        }
    }
}

// ── Surge API (read path) ─────────────────────────────
@RestController
public class SurgeController {

    @Autowired private RedisTemplate<String,String> redis;

    @GetMapping("/surge")
    public SurgeResponse getSurge(@RequestParam double lat, @RequestParam double lng) {
        long   cell       = h3.geoToH3(lat, lng, 6);  // res 6 for pricing
        String value      = redis.opsForValue().get("surge:" + cell);
        double multiplier = value != null ? Double.parseDouble(value) : 1.0; // default: no surge
        return new SurgeResponse(multiplier, cell);
    }
}

# Surge Pricing — Python: Pricing Function + API
# ══════════════════════════════════════════════════════

import h3, redis, time
from dataclasses import dataclass

r = redis.Redis(decode_responses=True)

CAP              = 4.0
SURGE_THRESHOLD  = 1.2
SURGE_TTL        = 120  # seconds
SURGE_RESOLUTION = 6   # H3 res-6: ~36 km² per cell


@dataclass
class SurgeResult:
    cell:       str
    multiplier: float
    demand:     int
    supply:     int


def pricing_function(demand: int, supply: int) -> float:
    """Piecewise linear surge multiplier. Capped at CAP."""
    ratio = demand / max(supply, 1)
    if   ratio <= 1.2: return 1.0
    elif ratio <= 2.0: return 1.0 + (ratio - 1.2) * 0.625
    elif ratio <= 4.0: return 1.5 + (ratio - 2.0) * 0.25
    else:              return min(CAP, 2.0 + (ratio - 4.0) * 0.1)


def compute_and_store_surge(cell: str, demand: int, supply: int) -> float:
    """Called by Flink job every 60s per active cell."""
    multiplier = pricing_function(demand, supply)
    if multiplier > 1.0:
        r.setex(f"surge:{cell}", SURGE_TTL, f"{multiplier:.4f}")
    else:
        r.delete(f"surge:{cell}")  # no surge = no key (GET returns None → 1.0)
    return multiplier


def get_surge_multiplier(lat: float, lng: float) -> float:
    """Read current surge for a location. <1ms from Redis."""
    cell  = h3.geo_to_h3(lat, lng, SURGE_RESOLUTION)
    value = r.get(f"surge:{cell}")
    return float(value) if value else 1.0  # no key = no surge = 1.0×


def get_surge_map(lat: float, lng: float, radius: int = 2) -> dict:
    """Return surge multipliers for nearby cells (for driver app heat map)."""
    center_cell = h3.geo_to_h3(lat, lng, SURGE_RESOLUTION)
    nearby      = h3.k_ring(center_cell, radius)
    pipe        = r.pipeline()
    for cell in nearby:
        pipe.get(f"surge:{cell}")
    values = pipe.execute()
    return {
        cell: float(val) if val else 1.0
        for cell, val in zip(nearby, values)
    }


# ── Simulated Flink batch: compute surge for all active cells ──
def surge_computation_batch(supply_counts: dict[str, int], demand_counts: dict[str, int]):
    """Called every 60s by Flink job. Writes surge to Redis for all active cells."""
    all_cells = set(supply_counts.keys()) | set(demand_counts.keys())
    results   = []
    for cell in all_cells:
        supply = supply_counts.get(cell, 0)
        demand = demand_counts.get(cell, 0)
        mult   = compute_and_store_surge(cell, demand, supply)
        if mult > 1.0:
            results.append(SurgeResult(cell, mult, demand, supply))
    return results


# ── Concert spike simulation ──────────────────────────
def simulate_concert_spike():
    arena_lat, arena_lng = 37.332, -121.900  # San Jose SAP Center
    cell = h3.geo_to_h3(arena_lat, arena_lng, SURGE_RESOLUTION)

    # T=0: Concert ends. 10,000 requests, 50 nearby drivers
    mult = compute_and_store_surge(cell, demand=10000, supply=50)
    print(f"T=0s:   demand=10000 supply=50  ratio={10000/50:.1f}x  surge={mult:.2f}x")

    # T=5min: Drivers attracted by 4× multiplier flock in
    mult = compute_and_store_surge(cell, demand=7000, supply=500)
    print(f"T=5min: demand=7000  supply=500  ratio={7000/500:.1f}x  surge={mult:.2f}x")

    # T=15min: Market clearing
    mult = compute_and_store_surge(cell, demand=3000, supply=2000)
    print(f"T=15min:demand=3000  supply=2000 ratio={3000/2000:.1f}x  surge={mult:.2f}x")

    # T=30min: Equilibrium
    mult = compute_and_store_surge(cell, demand=500, supply=800)
    print(f"T=30min:demand=500   supply=800  ratio={500/800:.2f}x surge={mult:.2f}x")

⚖️ Tradeoffs
🗺 Zone Granularity: Coarse vs Fine vs Dynamic
Very Fine (H3 res-8, ~0.7 km²)

Highly targeted surge — the exact block has high surge but the next block doesn't. Very accurate but statistically noisy: a single event can completely dominate a tiny zone. Drivers don't benefit much — they can't navigate to a specific 700m² zone. Zone boundaries shift constantly.

✅ Medium (H3 res-6, ~36 km²) — chosen

Large enough to be statistically stable. Small enough to capture neighborhood-level variation. Drivers can meaningfully navigate into a zone. Uber uses res-6 for surge pricing. Easy to display on a map. 50K active zones worldwide is very manageable. The right balance for a city-level marketplace.

Very Coarse (City-level)

Single multiplier for the entire city. Easy to compute. Easy for riders to understand. But: completely misses micro-market imbalances — surge at the airport doesn't help the concert venue 15km away. Useless for demand management in specific areas. Only appropriate for very low-density markets.

⏱ Computation Frequency: Reactive vs Periodic vs Predictive
Reactive (compute on every event)

Re-compute surge on every driver/request event. Maximum freshness. But: 250K+ events/sec → 250K surge computations/sec. Each computation reads all nearby demand/supply data. Extremely expensive. Race conditions between concurrent updates. Not practical at scale.

✅ Periodic (60s batch) — chosen

Flink tumbling window aggregates supply and demand. Every 60 seconds: compute new multiplier for all active cells. Write to Redis. Simple, predictable load. Max staleness: 60 seconds — acceptable. Riders accept surge displayed at request time, which may be slightly stale. Industry standard.

Predictive

ML model predicts future demand (knowing concert ends at 10pm, start surging at 9:55pm). Pre-emptive surge attracts drivers before the spike. Better equilibrium than reactive surge. Used by Uber in practice for known events. Requires event calendar integration and demand forecasting model. Higher complexity, significantly better rider experience.
