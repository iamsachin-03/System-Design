Distributed Task Scheduler
Design a distributed cron / task scheduler that executes millions of scheduled jobs reliably at scale — handling at-least-once execution guarantees, idempotency, clock skew across nodes, missed-run catch-up, exactly-once semantics via distributed locking, and graceful handling of long-running tasks and worker failures.
🔴 MUST DO 🟣 Hard At-Least-Once Idempotency Distributed Lock Cron Expression Missed Run Leader Election
🎯 Clarify Scope
A task scheduler is deceptively hard. Single-machine cron is trivial; distributed cron requires solving: which node fires the job (leader election), what if two nodes both fire it (exactly-once), what if a node crashes mid-job (at-least-once + idempotency), what if a job was missed during downtime (catch-up), and how to scale to millions of scheduled jobs.

💬 Phase 1 — The Double-Trigger Problem
PHASE 1 · EXACTLY-ONCE TRIGGER
IR
INTERVIEWER
You have 3 scheduler nodes for high availability. At 2:00 PM, all three wake up to scan for jobs. How do you prevent the same job firing three times?
ME
YOU
This is the exactly-one-trigger problem. Three strategies:

1. Leader election (single scheduler): Use ZooKeeper or etcd to elect one leader among the scheduler nodes. Only the leader scans and fires jobs. Followers watch the leader's heartbeat — if it dies, they hold a new election in seconds. Simplest to reason about. Weakness: slight gap between leader death and new leader election (10–30s with ZooKeeper).

2. Optimistic locking in the job DB: Each scheduler node tries to claim a job with an atomic compare-and-swap: UPDATE jobs SET status='RUNNING', claimed_by='{node_id}', claimed_at=NOW() WHERE id=? AND status='PENDING' AND next_run_at <= NOW(). Only the node where rowsAffected == 1 actually runs the job. Others get 0 rows affected and skip. Works without a separate coordination service. Requires a DB that supports row-level locking.

3. Distributed lock (Redis SET NX): Before firing, acquire a Redis lock: SET lock:job:{job_id}:{run_window} true NX PX 300000 (300s TTL). Only one node gets the lock; others skip. Clean, stateless. Weakness: if the winner crashes before completing, the lock expires and another node can claim it — this gives at-least-once, not exactly-once.

Production approach: Leader election for normal operation (only one scanner), optimistic DB locking as an additional safety layer. Defense in depth.
💡 At-least-once is the practical choice. Exactly-once requires two-phase commits or distributed transactions. At-least-once + idempotent job logic = correctness without coordination overhead. The job handler must be designed to be safe to run twice.
⏰ Phase 2 — Scaling to 10M Jobs
PHASE 2 · SCALE
IR
INTERVIEWER
How does the scheduler efficiently find which jobs need to fire in the next minute, out of 10M registered jobs?
ME
YOU
A full table scan of 10M rows every minute is too slow. Three efficient approaches:

1. DB index on next_run_at: SELECT * FROM jobs WHERE next_run_at BETWEEN NOW() AND NOW()+60s AND status='ACTIVE' with an index on (status, next_run_at). At 100K jobs firing per minute, this returns 100K rows quickly via index seek. After firing, update next_run_at to the next scheduled time. Simple and effective for moderate scale.

2. Redis Sorted Set as a priority queue: Maintain a ZSet where score = next_run_at (Unix timestamp). Every minute: ZRANGEBYSCORE due_jobs 0 {now_unix} — returns all jobs due up to now. Pop them atomically with ZREMRANGEBYSCORE. Re-add each job with its next_run_at as the new score. Extremely fast — O(log N + K) where K is jobs due. Handles 10M jobs in ~100ms scan. This is how Sidekiq and Celery implement delayed job queues.

3. Time-bucketed sharding: Partition jobs into time buckets: bucket_{YYYY-MM-DD-HH-MM}. Each bucket is a list of jobs due in that minute. The scanner only reads the current bucket — no scanning needed at all, O(1) lookup. Requires pre-computing all future schedules at job creation time (manageable for jobs scheduled months out).
💀 Phase 3 — Worker Failure & Missed Runs
PHASE 3 · FAILURE HANDLING
IR
INTERVIEWER
A worker picks up a job and crashes after 30 seconds — job is half done. Also: the entire scheduler goes down for 2 hours. What happens?
ME
YOU
Worker crash (in-flight job):
The job is in status=RUNNING with a claimed_at timestamp and a heartbeat. Workers must send a heartbeat every 30 seconds: UPDATE executions SET heartbeat_at=NOW() WHERE id=?. A background reaper process scans for jobs where heartbeat_at < NOW() - 120s (2× heartbeat interval) → these are stuck/crashed. The reaper resets them to status=PENDING and increments retry_count. Worker picks them up again. Job must be idempotent to handle re-execution.

Scheduler downtime for 2 hours:
When scheduler restarts, it detects missed runs by scanning: SELECT * FROM jobs WHERE next_run_at < NOW() AND status='ACTIVE' AND missed_catch_up=true. For each missed job, fire once regardless of how many scheduled times were missed (one catch-up run). Record the catch-up execution. Then advance next_run_at to the next future scheduled time.

For jobs where it's critical to fire every missed run (e.g., hourly billing), compute all missed intervals and fire once per interval. This is configurable per job: miss_policy: FIRE_ONCE | FIRE_ALL | SKIP.

// Distributed Task Scheduler — Java (Spring Boot)
// ═══════════════════════════════════════════════════════

// ── Scheduler Tick (leader only, every 10s) ─────────────
@Service
public class SchedulerService {

    @Autowired private RedisTemplate<String,String> redis;
    @Autowired private JobRepository  jobRepo;
    @Autowired private SqsTemplate    sqs;
    @Autowired private LeaderElection leader;
    @Autowired private CronParser     cronParser;

    @Scheduled(fixedDelay = 10_000)
    public void tick() {
        if (!leader.isLeader()) return; // followers skip

        long nowUnix = Instant.now().getEpochSecond();

        // 1. Pull all due jobs from ZSet
        Set<String> dueIds = redis.opsForZSet()
            .rangeByScore("due_jobs", 0, nowUnix);
        if (dueIds == null || dueIds.isEmpty()) return;

        // 2. Atomic claim via CAS — prevents double-trigger
        dueIds.forEach(jobId -> {
            int claimed = jobRepo.claimJob(jobId, Instant.now());
            if (claimed == 0) return; // already claimed by another node

            Job job = jobRepo.findById(jobId).orElse(null);
            if (job == null) return;

            // 3. Enqueue to SQS
            sqs.send("job-execution-queue", new JobPayload(job));

            // 4. Compute next run & re-add to ZSet
            Instant nextRun = cronParser.nextAfter(job.getCronExpr(), Instant.now());
            redis.opsForZSet().add("due_jobs", jobId, nextRun.getEpochSecond());
            jobRepo.updateNextRun(jobId, nextRun);
        });

        // 5. Remove fired jobs from ZSet
        redis.opsForZSet().removeRangeByScore("due_jobs", 0, nowUnix);
    }
}

// ── Job Repository — CAS claim ────────────────────────
public interface JobRepository extends JpaRepository<Job, String> {

    // Atomic: claim only if still PENDING
    @Modifying
    @Query("""
        UPDATE Job j SET j.status = 'RUNNING',
            j.claimedAt = :now, j.claimedBy = :nodeId
        WHERE j.id = :jobId AND j.status = 'PENDING'
        """)
    int claimJob(@Param("jobId") String jobId, @Param("now") Instant now);
}

// ── Worker — execution + heartbeat ──────────────────
@SqsListener("job-execution-queue")
public void executeJob(JobPayload payload) {
    String execId = createExecution(payload.getJobId());

    // Heartbeat thread — keep execution alive
    ScheduledFuture<?> heartbeat = heartbeatExecutor.scheduleAtFixedRate(
        () -> executionRepo.updateHeartbeat(execId, Instant.now()),
        0, 30, TimeUnit.SECONDS);

    try {
        if ("HTTP".equals(payload.getType())) {
            httpClient.post(payload.getCallback(), payload.getBody());
        } else {
            internalHandlers.get(payload.getHandlerClass()).run(payload);
        }
        executionRepo.markCompleted(execId);
    } catch (Exception e) {
        executionRepo.markFailed(execId, e.getMessage());
        throw e; // SQS visibility expires → automatic retry
    } finally {
        heartbeat.cancel(false);
    }
}

// ── Reaper — recover stuck jobs every 60s ─────────────
@Scheduled(fixedDelay = 60_000)
public void reapStuckJobs() {
    Instant staleThreshold = Instant.now().minus(Duration.ofSeconds(120));
    List<Execution> stuck = executionRepo
        .findRunningWithHeartbeatBefore(staleThreshold);

    stuck.forEach(exec -> {
        if (exec.getRetryCount() < exec.getJob().getMaxRetries()) {
            Duration backoff = Duration.ofSeconds(
                (long) Math.pow(2, exec.getRetryCount()) * 30);
            sqs.sendDelayed("job-execution-queue",
                new JobPayload(exec.getJob(), exec.getRetryCount() + 1),
                backoff.getSeconds());
        } else {
            executionRepo.markFailed(exec.getId(), "max retries exceeded");
            alertService.send(exec.getJob().getId(), "Job exhausted retries");
        }
    });
}

import time, threading, logging
from croniter import croniter
from dataclasses import dataclass
from typing import Optional
import redis, boto3

r   = redis.Redis()
sqs = boto3.client(\'sqs\')
DUE_ZSET   = \'due_jobs\'
JOB_QUEUE  = \'https://sqs.../job-execution\'
TICK_SECS  = 10
STALE_SECS = 120


@dataclass
class Job:
    id:          str
    cron_expr:   str
    callback:    str
    max_retries: int = 3
    miss_policy: str = \'FIRE_ONCE\'


def next_run_after(cron_expr: str, after: float) -> float:
    """Return next unix timestamp for cron expression after given time."""
    itr = croniter(cron_expr, after)
    return itr.get_next(float)


class Scheduler:
    """Runs on leader node only — ticks every 10s."""
    def __init__(self, is_leader_fn):
        self.is_leader = is_leader_fn

    def start(self):
        while True:
            if self.is_leader():
                self.tick()
            time.sleep(TICK_SECS)

    def tick(self):
        now = time.time()
        # 1. All due job IDs from ZSet
        due = r.zrangebyscore(DUE_ZSET, 0, now)
        if not due: return

        pipe = r.pipeline(transaction=True)
        for job_id_bytes in due:
            job_id = job_id_bytes.decode()
            # 2. CAS claim: SETNX per (job_id, tick_window)
            window = int(now // TICK_SECS)
            lock_key = f\'claim:{job_id}:{window}\'
            pipe.set(lock_key, \'1\', nx=True, ex=TICK_SECS * 2)
        results = pipe.execute()

        for job_id_bytes, claimed in zip(due, results):
            if not claimed: continue  # another node claimed it
            job_id = job_id_bytes.decode()
            job    = self._load_job(job_id)
            if not job: continue

            # 3. Enqueue to SQS
            sqs.send_message(QueueUrl=JOB_QUEUE,
                MessageBody=f\'{{"job_id": "{job_id}", "retry": 0}}\')

            # 4. Compute next run & re-add to ZSet
            next_ts = next_run_after(job.cron_expr, now)
            r.zadd(DUE_ZSET, {job_id: next_ts})

        # 5. Remove fired entries
        r.zremrangebyscore(DUE_ZSET, 0, now)

    def _load_job(self, job_id: str) -> Optional[Job]:
        # Load from PostgreSQL (stub)
        return Job(id=job_id, cron_expr=\'*/5 * * * *\',
                   callback=\'https://api.example.com/webhook\')


class Worker:
    """Polls SQS, executes jobs, sends heartbeats."""
    def start(self):
        while True:
            msgs = sqs.receive_message(QueueUrl=JOB_QUEUE,
                MaxNumberOfMessages=10, WaitTimeSeconds=20,
                VisibilityTimeout=300)  # 5-min visibility window
            for msg in msgs.get(\'Messages\', []):
                self._execute(msg)

    def _execute(self, msg: dict):
        import json, requests
        payload    = json.loads(msg[\'Body\'])
        receipt    = msg[\'ReceiptHandle\']
        job_id     = payload[\'job_id\']
        exec_id    = f\'exec:{job_id}:{int(time.time())}\'

        # Heartbeat thread
        stop_hb = threading.Event()
        def heartbeat_loop():
            while not stop_hb.wait(30):
                r.setex(f\'heartbeat:{exec_id}\', STALE_SECS, \'alive\')
        threading.Thread(target=heartbeat_loop, daemon=True).start()

        try:
            job = self._load_job(job_id)
            requests.post(job.callback, json={\'job_id\': job_id}, timeout=60)
            # Delete from queue on success
            sqs.delete_message(QueueUrl=JOB_QUEUE, ReceiptHandle=receipt)
            logging.info(f\'Job {job_id} completed\')
        except Exception as e:
            logging.error(f\'Job {job_id} failed: {e}\')
            # Don\'t delete → SQS visibility expires → automatic re-queue
        finally:
            stop_hb.set()

⚖️ Tradeoffs
🔒 Exactly-Once vs At-Least-Once Execution
❌ Exactly-Once (complex)

Requires 2-phase commit between the scheduler DB and the job executor. On worker crash after job completes but before ACK, the coordinator must determine if job actually ran — needs distributed transaction protocol. Extremely complex. Almost never done in practice for task schedulers.

✅ At-Least-Once + Idempotency

Job may run more than once on failure. Job logic must be idempotent: INSERT IGNORE, UPDATE WHERE status=NEW, deduplicated by idempotency key. This is the industry standard. Sidekiq, Celery, AWS Step Functions — all at-least-once. Idempotency in the job is the right layer for this guarantee.

Idempotency patterns

Common approaches: (1) Check-then-act with a unique execution ID: INSERT INTO results (exec_id, ...) ON CONFLICT DO NOTHING. (2) External idempotency key passed to downstream APIs. (3) Conditional write with expected state: UPDATE WHERE status='PENDING'. The job itself owns the idempotency guarantee.

📅 Job Storage: Redis ZSet vs DB Index vs Time Buckets
✅ Redis ZSet (chosen)

O(log N + K) to get all due jobs. 240 MB for 10M jobs. Sub-millisecond scan. Survives scheduler restarts (persistent Redis). Atomic ZRANGEBYSCORE + ZREMRANGEBYSCORE prevents double-trigger. Used by Sidekiq, Celery, Resque for delayed job queues.

DB index on next_run_at

Works well at smaller scale (<1M jobs). No extra dependency. DB is the source of truth. At 10M jobs, the index scan can be slow under write pressure when many jobs update next_run simultaneously (at every-minute mark). Fine for moderate scale.

Time buckets (pre-partitioned)

Pre-compute all future run times at job creation, store in scheduled_runs(run_at, job_id). Scanner reads only current minute's partition — O(1) lookup. No scanning at all. Best for very high scale. Cost: complex pre-computation for complex cron expressions and jobs that run for years.

👑 Leader Election: ZooKeeper vs DB Lock vs Single Node
✅ ZooKeeper / etcd

Industry standard for leader election. Ephemeral nodes auto-expire when node dies. Re-election in 10–30s. Proven at Kafka, HBase, Hadoop scale. Adds operational complexity (ZK cluster to manage) but eliminates split-brain risk.

DB advisory lock

SELECT pg_try_advisory_lock(1) in PostgreSQL. Simple — no extra service. Leader refreshes lock every 30s. On death, lock expires after session disconnect. Works well for smaller deployments. Risk: DB becomes a SPOF for leader election as well as job storage.

Single node (no HA)

One scheduler node, no leader election. Simple. Fast failover via Kubernetes pod restart (~30s). Acceptable if the 30s scheduling gap during restart is OK. Many production schedulers (Quartz, APScheduler) start here and add HA later when needed.
