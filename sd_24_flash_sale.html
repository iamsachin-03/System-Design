Flash Sale / Inventory System
Design a system that handles 100,000 concurrent buyers trying to purchase 1,000 limited-edition items simultaneously — preventing oversell (selling more than inventory), ensuring fairness, absorbing traffic spikes without melting the database, and gracefully handling the burst of 100× normal traffic that hits at sale start.
🔴 MUST DO 🟣 Hard Oversell Prevention Redis Atomic Queue Fairness Rate Limiting Pre-warming
🎯 Clarify Scope
Flash sales are one of the hardest distributed systems problems. The hard part isn't the happy path — it's the thundering herd at exactly T=0, preventing oversell under extreme concurrency, and making sure legitimate buyers aren't starved by bots.

💬 Phase 1 — The Oversell Problem
PHASE 1 · OVERSELL PREVENTION
IR
INTERVIEWER
Inventory is at 1. Two users simultaneously try to buy. How do you guarantee only one succeeds?
ME
YOU
This is a classic race condition. Without proper synchronisation, two concurrent reads both see inventory=1, both decide to proceed, both decrement, and you've oversold.

Three approaches in increasing robustness:

1. Optimistic locking (DB-level): UPDATE inventory SET count = count - 1 WHERE sku_id = ? AND count > 0 — the WHERE count > 0 clause is the guard. DB processes this as an atomic operation. Check rowsAffected == 1; if 0, inventory was 0 when you tried. Works but at 100K QPS this hits the DB directly — expensive.

2. Redis DECR (atomic counter): DECR inventory:{sku_id} is atomic in Redis. Redis single-thread guarantees no race. If result >= 0: success. If result < 0: run INCR to put it back, return sold-out. This moves the hot check out of the DB entirely. Handles hundreds of thousands of ops/sec.

3. Redis Lua script (reservation + dedup in one shot): For production: a Lua script atomically checks inventory, checks user hasn't already bought, decrements, and records the reservation — all in a single Redis command, guaranteed atomic. No interleaving possible.
💡 The canonical answer: Redis DECR for the hot path. The DB is only written to when a reservation is confirmed (asynchronously). This decouples the spike traffic from the DB entirely.
⚡ Phase 2 — Thundering Herd at T=0
PHASE 2 · TRAFFIC SPIKE
IR
INTERVIEWER
100K users hit "Buy" at exactly 12:00:00. How does your system not fall over?
ME
YOU
The key insight: 99,000 of those requests need to be rejected fast and cheaply. We can't let them all reach Redis or the DB.

Layer 1 — Rate limiting at edge (API Gateway): Per-user: 1 purchase request per sale per SKU. A user hammering "Buy" 500 times gets rate-limited immediately at the gateway — zero backend cost for duplicate requests. Use a Redis token bucket per user_id + sku_id.

Layer 2 — Virtual waiting room queue: At T=0, instead of routing all 100K requests directly to the purchase API, route them to a queue admission service. Each request gets enqueued with a timestamp. The queue service issues a position ticket and streams progress updates via SSE or WebSocket. Only N requests per second are dequeued and forwarded to actual purchase processing (N = our processing capacity, e.g. 5,000/sec). This converts the thundering herd into a smooth trickle for the backend.

Layer 3 — Pre-warm Redis inventory: At T minus 5 minutes, load inventory count into Redis. Don't wait for the first request. When sale starts: Redis is already hot, no DB hit needed for the check.

Layer 4 — CDN/edge "sold out" response: Once inventory hits 0 in Redis, set a flag. All subsequent requests get a 200 "Sold Out" response from a lightweight edge service — zero work for the purchase service or DB.
🔄 Phase 3 — Inventory Release on Payment Failure
PHASE 3 · HOLDS & TIMEOUTS
IR
INTERVIEWER
User reserves the item but doesn't complete payment within 10 minutes. How do you release the inventory?
ME
YOU
Reservation creates a temporary hold that must be released if payment doesn't arrive. Two mechanisms:

1. Redis expiring reservation key: When creating a reservation: SET reservation:{user_id}:{sku_id} "held" EX 600 (600 seconds = 10 min). A background job scans for expired reservation keys and increments the Redis inventory counter for each. Simple but polling is imprecise.

2. Delayed job queue (preferred): On reservation, publish a message to a delayed queue (e.g., SQS with visibility timeout of 10 min): {action: "release", sku_id: X, user_id: Y, reservation_id: Z, expires_at: T+600}. The consumer checks: if order status is still "reserved" (payment never came), run INCR inventory:{sku_id} and update order to "expired". If payment arrived first, the consumer sees status="paid" and discards the message.

Delayed queue is more reliable — works even if the reservation Redis key gets evicted under memory pressure (unlikely but possible). The queue message acts as a durability guarantee.

🛡 Anti-Bot & Fairness
ANTI-BOT MEASURES
1
CAPTCHA at sale start: Users must solve CAPTCHA before entering queue. Bots fail, humans pass. Rate limits their repeated attempts.
2
Device fingerprint + account age: New accounts (<7 days) flagged. Multiple purchases from same IP/device within 60 seconds throttled.
3
Pre-registration: Users register interest 24h before sale. Only registered users enter the queue. Bots must register in advance — makes mass registration the bottleneck instead of T=0 spike.
4
Randomised queue (lottery): Instead of strict FCFS, shuffle registered users randomly. Bots gain zero advantage from submitting at T=0.000 vs T=0.010.
QUEUE FAIRNESS MECHANISMS
1
Virtual waiting room: Users see their position (#4,382 of 100,000). Psychological anchoring — users wait instead of spamming refresh.
2
Session token: When dequeued, user gets a time-limited session token (valid 5 min) to complete purchase. Others can't steal the slot.
3
One-per-user enforcement: Redis SET reservation:{uid} with NX flag — only succeeds once per user. Server-side dedup prevents double-buying even if client sends two requests.
4
Idempotency key: Client sends X-Idempotency-Key: {uuid}. Server deduplicates identical requests within 60s — safe to retry on network failure.

// Flash Sale — Java: Atomic Redis Lua + Purchase Service
// ══════════════════════════════════════════════════════

@Service
public class InventoryService {

    @Autowired private RedisTemplate<String,String> redis;
    @Autowired private OrderRepository  orderRepo;
    @Autowired private SqsTemplate      sqs;

    // ── Lua script: atomic check + decrement ─────────────
    // Returns 1=success, -1=sold_out, -2=duplicate_buy
    // Runs as single atomic Redis command. No race condition.
    private static final String RESERVE_SCRIPT = """
        local inv_key  = KEYS[1]
        local res_key  = KEYS[2]
        local user_key = KEYS[3]
        local ttl      = ARGV[1]

        -- Check duplicate purchase
        if redis.call('EXISTS', user_key) == 1 then
          return -2
        end

        -- Atomic check + decrement
        local remaining = redis.call('DECR', inv_key)
        if remaining < 0 then
          redis.call('INCR', inv_key)  -- undo the decrement
          return -1
        end

        -- Record reservation (expires in 10 minutes)
        redis.call('SET', res_key, '1', 'EX', ttl)
        redis.call('SET', user_key, '1', 'EX', ttl)
        return 1
        """;

    private final RedisScript<Long> reserveScript =
        RedisScript.of(RESERVE_SCRIPT, Long.class);

    public PurchaseResult purchase(String userId, String skuId) {
        String invKey  = "inventory:" + skuId;
        String resKey  = "reservation:" + userId + ":" + skuId;
        String userKey = "bought:" + userId + ":" + skuId;

        Long result = redis.execute(reserveScript,
            List.of(invKey, resKey, userKey),
            "600"); // 10-min hold

        switch (result.intValue()) {
            case 1: {
                // Async: persist to DB and enqueue payment
                String orderId = createOrder(userId, skuId);
                scheduleRelease(orderId, skuId, userId);
                return PurchaseResult.success(orderId);
            }
            case -1: return PurchaseResult.soldOut();
            case -2: return PurchaseResult.alreadyPurchased();
            default:  throw new IllegalStateException("Unexpected: " + result);
        }
    }

    @Async
    private String createOrder(String userId, String skuId) {
        return orderRepo.save(new Order(userId, skuId, OrderStatus.RESERVED))
                        .getOrderId();
    }

    @Async
    private void scheduleRelease(String orderId, String skuId, String userId) {
        // Delayed SQS message — fires after 600s
        sqs.sendDelayed("inventory-release-queue",
            new ReleaseJob(orderId, skuId, userId),
            600);
    }
}

// ── Release Consumer ─────────────────────────────────
@SqsListener("inventory-release-queue")
public void handleRelease(ReleaseJob job) {
    Order order = orderRepo.findById(job.orderId()).orElse(null);
    if (order == null || order.getStatus() != OrderStatus.RESERVED) {
        return; // already paid or cancelled — nothing to release
    }
    // Return inventory to Redis
    redis.opsForValue().increment("inventory:" + job.skuId());
    orderRepo.updateStatus(job.orderId(), OrderStatus.EXPIRED);
}

// ── Pre-warm inventory at sale start ─────────────────
@Scheduled(cron = "0 55 11 * * ?") // 5 min before 12:00 sale
public void prewarmInventory() {
    for (SaleItem item : saleItemRepo.findAllActive()) {
        redis.opsForValue().set(
            "inventory:" + item.getSkuId(),
            String.valueOf(item.getInventoryCount()));
    }
}

# Flash Sale — Python: Redis Lua atomic reservation
# ══════════════════════════════════════════════════════

import redis, boto3, json
from dataclasses import dataclass
from enum import Enum
from typing import Optional

class PurchaseStatus(Enum):
    SUCCESS          = 1
    SOLD_OUT         = -1
    ALREADY_PURCHASED = -2

# Lua script: atomic check-and-decrement
# Runs as single Redis command — no race condition possible
RESERVE_SCRIPT = """
local inv_key  = KEYS[1]
local res_key  = KEYS[2]
local user_key = KEYS[3]
local ttl      = tonumber(ARGV[1])

if redis.call('EXISTS', user_key) == 1 then
  return -2
end

local remaining = redis.call('DECR', inv_key)
if remaining < 0 then
  redis.call('INCR', inv_key)
  return -1
end

redis.call('SET', res_key, '1', 'EX', ttl)
redis.call('SET', user_key, '1', 'EX', ttl)
return 1
"""

class InventoryService:
    def __init__(self, redis_client, sqs_client):
        self.r      = redis_client
        self.sqs    = sqs_client
        self.script = self.r.register_script(RESERVE_SCRIPT)

    def purchase(self, user_id: str, sku_id: str) -> dict:
        """Atomically reserve one unit for user. Returns status dict."""
        inv_key  = f\'inventory:{sku_id}\'
        res_key  = f\'reservation:{user_id}:{sku_id}\'
        user_key = f\'bought:{user_id}:{sku_id}\'

        result = self.script(
            keys=[inv_key, res_key, user_key],
            args=[600]  # 10-min hold TTL
        )

        status = PurchaseStatus(result)
        if status == PurchaseStatus.SUCCESS:
            order_id = self._create_order(user_id, sku_id)
            self._schedule_release(order_id, sku_id, user_id)
            return {\'status\': \'success\', \'order_id\': order_id}
        elif status == PurchaseStatus.SOLD_OUT:
            return {\'status\': \'sold_out\'}
        elif status == PurchaseStatus.ALREADY_PURCHASED:
            return {\'status\': \'duplicate\'}

    def _create_order(self, user_id: str, sku_id: str) -> str:
        # Async DB write (in production: via Kafka/SQS)
        order_id = f\'ORD-{user_id}-{sku_id}\'
        return order_id

    def _schedule_release(self, order_id: str, sku_id: str, user_id: str):
        # SQS delayed message — fires after 600s
        self.sqs.send_message(
            QueueUrl=\'https://sqs.../inventory-release\',
            MessageBody=json.dumps({
                \'order_id\': order_id, \'sku_id\': sku_id, \'user_id\': user_id
            }),
            DelaySeconds=600
        )

    def handle_release(self, job: dict):
        """SQS consumer: release inventory if payment never came."""
        order_id = job[\'order_id\']
        sku_id   = job[\'sku_id\']
        # Check DB: if order still RESERVED, release
        order_status = self._get_order_status(order_id)
        if order_status == \'RESERVED\':
            self.r.incr(f\'inventory:{sku_id}\')
            self._update_order_status(order_id, \'EXPIRED\')

    def prewarm(self, sku_id: str, count: int):
        """Pre-load inventory into Redis 5 min before sale."""
        self.r.set(f\'inventory:{sku_id}\', count)
        print(f\'[PREWARM] inventory:{sku_id} = {count}\')
⚖️ Tradeoffs
🔐 Oversell Prevention: Optimistic vs Pessimistic vs Redis
❌ DB Pessimistic Lock

SELECT FOR UPDATE on inventory row. Guarantees correctness. But: each buyer holds a lock for the duration of their transaction (seconds). At 100K concurrent buyers, the lock queue causes massive head-of-line blocking. DB connection pool exhausted. System grinds to halt. Never use for flash sales.

⚠️ DB Optimistic Lock

UPDATE WHERE count > 0. Non-blocking — no lock held. Returns rows_affected; retry on 0. Works at moderate scale (10K QPS). At 100K concurrent: DB still becomes bottleneck (even reads are expensive at this scale). Better than pessimistic but not enough.

✅ Redis Lua Script (chosen)

Atomic single-command execution. Redis processes 500K+ ops/sec. No locks, no contention. DB only written asynchronously (17 writes/sec). The Redis Lua script executes as one indivisible operation — impossible to have a race. Best approach for flash sale scale.

⏳ Queue vs No Queue at T=0
❌ No queue (direct to purchase)

All 100K requests hit the purchase service simultaneously. Even with Redis handling inventory, the purchase service pods need to handle 100K RPS. Auto-scaling can't spin up fast enough (minutes). First 1,000 get through; remaining 99K receive timeouts and errors. Poor UX.

⚠️ Simple throttle

Reject anything over N RPS. Simple to implement. But: no fairness — users who happen to hit at exactly the right millisecond get through. Bot-friendly: fast bots get priority over slow humans. No position feedback.

✅ Virtual waiting room

Enqueue all 100K with timestamps. Dequeue at controlled rate (5K/sec). Users see their position. FCFS is fair. Bots get no timing advantage if pre-registration is required. Backend sees constant 5K RPS instead of 100K spike. Best UX and system stability.
