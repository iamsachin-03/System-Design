Pessimistic Locking on Relational DBs

When multiple transactions compete for the same rows, locking determines who wins and how the others wait. Getting this wrong causes deadlocks, poor throughput, or data corruption.

Why Pessimistic Locking Exists

Imagine two users booking the last seat on a flight simultaneously. Without locking, both read "1 seat available," both decrement to 0, and you've sold one seat twice. Pessimistic locking prevents this by holding a row-level lock from read through commit.

Lost Update Problem: T1 reads value X, T2 reads value X, T1 writes X+1, T2 writes X+1. Result: one increment is lost. This is the classic race condition that locks are designed to prevent.

Pessimistic vs Optimistic Locking

Pessimistic Locking

  • Lock row at read time with SELECT … FOR UPDATE
  • Other transactions block until lock released
  • Guarantees no lost updates
  • Risk: deadlocks, reduced throughput under contention
  • Best for: high contention, short transactions, seat booking, inventory decrement

Optimistic Locking

  • No lock at read; compare-and-swap at write using version column
  • UPDATE … WHERE id=? AND version=?
  • If rowcount=0, conflict — retry the transaction
  • Risk: retry storms under high contention
  • Best for: low contention, long reads, user profile updates

SQL Patterns

Pessimistic locking patternsSQL
-- 1. Standard pessimistic lock — blocks waiting transactions
BEGIN;
SELECT seats_available FROM flights WHERE id = 42 FOR UPDATE;
UPDATE flights SET seats_available = seats_available - 1 WHERE id = 42;
COMMIT;

-- 2. NOWAIT — fail immediately if row is locked (no queuing)
SELECT * FROM seats WHERE id = 5 FOR UPDATE NOWAIT;
-- Raises: ERROR 3572 (HY000): Statement aborted because lock(s) could not be acquired

-- 3. SKIP LOCKED — job queue pattern: grab unclaimed work
SELECT * FROM jobs WHERE status = 'pending'
LIMIT 10 FOR UPDATE SKIP LOCKED;
-- Each worker grabs different rows — no contention, no queuing

-- 4. Optimistic locking with version column
UPDATE accounts
SET balance = balance - 100, version = version + 1
WHERE id = 7 AND version = 3;    -- only succeeds if nobody else updated
-- If rowcount = 0 → conflict detected → retry with fresh read

When Locking Kills Throughput

A single long-running transaction holding a lock on a hot row (e.g., inventory count for a popular product) causes all other transactions to queue. 1000 RPS × 50ms lock hold = queue depth of 50 waiting transactions. Each holds a connection. Pool exhaustion follows.
Fix: Minimize time inside the transaction. Move all non-DB logic (validation, API calls) outside the transaction. Lock for milliseconds, not seconds.
Deadlocks — Detection & Prevention

A deadlock occurs when T1 holds lock A and waits for lock B, while T2 holds lock B and waits for lock A. Neither can proceed.

Prevention Rules

Rule 1 — Consistent Lock Order

Always acquire locks in the same order across all transactions. If all code locks Row A before Row B, a circular wait is impossible by construction.

Sort IDs before locking: ORDER BY id ASC FOR UPDATE

Rule 2 — Keep Transactions Short

The window for deadlock is the transaction duration. Sub-millisecond transactions have near-zero deadlock probability. Long transactions with user interaction are deadlock-prone.

Rule 3 — Use NOWAIT + Retry

Instead of waiting indefinitely, use NOWAIT to fail fast, then retry with exponential backoff. Converts potential deadlock into a controlled retry.

Rule 4 — Let the DB Handle It

MySQL/PostgreSQL auto-detect deadlocks and kill one transaction. Your code must handle the ER_LOCK_DEADLOCK error and retry. Always build deadlock-safe retry logic.

from sqlalchemy import text
import asyncio

async def book_seat(db, flight_id: int, user_id: int) -> bool:
    """Pessimistic lock: SELECT FOR UPDATE then decrement."""
    max_retries, backoff = 3, 0.05

    for attempt in range(max_retries):
        try:
            async with db.begin():   # transaction auto-commits or rolls back
                # Lock the row — all other writers block here
                row = (await db.execute(
                    text("SELECT seats_available FROM flights WHERE id=:id FOR UPDATE"),
                    {"id": flight_id}
                )).fetchone()

                if not row or row.seats_available <= 0:
                    return False   # no seats — no need to lock further

                await db.execute(
                    text("UPDATE flights SET seats_available=seats_available-1 WHERE id=:id"),
                    {"id": flight_id}
                )
                await db.execute(
                    text("INSERT INTO bookings(flight_id,user_id) VALUES(:f,:u)"),
                    {"f": flight_id, "u": user_id}
                )
            return True   # commit on exit from async with

        except Exception as e:
            if "deadlock" in str(e).lower():
                await asyncio.sleep(backoff * (2 ** attempt))   # exp backoff
                continue
            raise
    return False   # exhausted retries


async def dequeue_job(db):
    """SKIP LOCKED — safe for concurrent workers, no contention."""
    async with db.begin():
        rows = (await db.execute(
            text("SELECT id,payload FROM jobs WHERE status='pending' LIMIT 10 FOR UPDATE SKIP LOCKED")
        )).fetchall()
        ids = [r.id for r in rows]
        if ids:
            await db.execute(text(f"UPDATE jobs SET status='processing' WHERE id IN ({','.join(map(str,ids))})"))
        return rows
BookingService.java — Spring + JPA pessimistic lockJava
@Service @Transactional
public class BookingService {
    private final FlightRepository flights;
    private final BookingRepository bookings;

    public boolean bookSeat(long flightId, long userId) {
        // @Lock(PESSIMISTIC_WRITE) triggers SELECT … FOR UPDATE
        Flight flight = flights.findByIdForUpdate(flightId)
            .orElseThrow();

        if (flight.getSeatsAvailable() <= 0) return false;

        flight.setSeatsAvailable(flight.getSeatsAvailable() - 1);
        flights.save(flight);
        bookings.save(new Booking(flightId, userId));
        return true;
    }
}

// Repository
public interface FlightRepository extends JpaRepository<Flight,Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT f FROM Flight f WHERE f.id = :id")
    Optional<Flight> findByIdForUpdate(@Param("id") long id);
}
Outcome
  • SELECT FOR UPDATE is the foundation — it's the primitive that prevents lost updates, double-bookings, and overselling. Every ticket/inventory system uses it.
  • SKIP LOCKED enables parallel job queues — multiple workers can safely dequeue without blocking each other. The DB-backed job queue pattern is simpler than adding Kafka for many use cases.
  • Consistent lock ordering prevents deadlocks by construction — if every transaction acquires locks in the same order (sort IDs ascending), a circular wait cannot form.
  • Keep transactions tight — lock hold time = throughput killer. Move validation, external API calls, and computation outside the transaction. Lock → read → write → unlock in milliseconds.
  • Always retry on deadlock — the DB will detect and kill the victim transaction. Your application must catch the error and retry. Implement exponential backoff to avoid retry storms.

Design a Scalable SQL-backed KV Store

Build a key-value store backed by a relational database — like a lightweight Redis alternative when you need SQL's durability and ACID guarantees but also KV-style access patterns.

Problem Statement

Design a KV store that supports: GET(key), SET(key, value, ttl?), DELETE(key), EXISTS(key), and eventually consistent expiry of TTL keys. The system must handle 100K read ops/sec and 10K write ops/sec with sub-10ms p99.

Core Requirements

  • GET/SET/DEL operations at high throughput
  • Optional TTL per key (auto-expire)
  • Namespace support (logical isolation)
  • Atomic compare-and-set (CAS)

High-Level Requirements

  • Durable — survives restarts (SQL WAL)
  • Consistent — ACID transactions
  • Support range queries on keys (unlike Redis)
  • Serve as persistent layer beneath a cache

Micro Requirements

  • Throughput not affected by locking
  • Expired keys lazily deleted + background cleanup
  • No hotspot on single row / single index page
Schema Design

Core Table Design

kv_store schema — MySQL / PostgreSQLSQL
CREATE TABLE kv_store (
    namespace   VARCHAR(64)   NOT NULL DEFAULT 'default',
    key_name    VARCHAR(512)  NOT NULL,
    value       MEDIUMBLOB   NOT NULL,   -- up to 16MB, binary safe
    version     BIGINT        NOT NULL DEFAULT 1,  -- for CAS / optimistic lock
    expires_at  BIGINT        DEFAULT NULL,   -- epoch ms; NULL = no expiry
    created_at  BIGINT        NOT NULL,
    updated_at  BIGINT        NOT NULL,

    PRIMARY KEY (namespace, key_name),   -- composite PK = O(1) lookup

    -- Index for TTL expiry background job
    INDEX idx_expires (expires_at)
);

-- GET: exact PK lookup — B-tree traversal, O(log N)
SELECT value, expires_at FROM kv_store
WHERE namespace='app' AND key_name='user:42'
  AND (expires_at IS NULL OR expires_at > UNIX_TIMESTAMP()*1000);

-- SET (upsert): one statement, no SELECT + INSERT race
INSERT INTO kv_store (namespace, key_name, value, expires_at, version, created_at, updated_at)
VALUES ('app', 'user:42', :val, :exp, 1, :now, :now)
ON DUPLICATE KEY UPDATE
    value=VALUES(value), expires_at=VALUES(expires_at),
    version=version+1, updated_at=VALUES(updated_at);

-- CAS (compare-and-set): only update if version matches
UPDATE kv_store SET value=:new_val, version=version+1
WHERE namespace=:ns AND key_name=:key AND version=:expected_version;
-- rowcount=0 → CAS failed (someone else updated first)

-- Background TTL cleanup — run every 60s
DELETE FROM kv_store
WHERE expires_at IS NOT NULL AND expires_at < UNIX_TIMESTAMP()*1000
LIMIT 1000;   -- chunk deletes to avoid long lock holds

Why Composite Primary Key?

Clustering benefit

InnoDB clusters rows by PK. With composite PK (namespace, key_name), all keys in the same namespace are physically co-located on disk. Range scans over a namespace become sequential I/O.

Hot page problem

If using a single auto-increment PK, all writes go to the last B-tree page ("right-hand insert"). Under high write load, this page becomes a mutex hotspot. Composite keys distribute writes across the tree.

Architecture & Scaling

System Architecture

Scaling the Write Path

Write Bottleneck: InnoDB Row Locks

At 10K writes/s all hitting the same table, row-level locks on hot keys become contention points. Solution: partition writes by namespace hash across multiple MySQL shards. No cross-shard transactions needed (KV operations are single-key).

Read Scaling: Cache + Replicas

90% of reads served from Redis cache. Remaining 10% hit MySQL read replicas. The primary only receives writes and cache-miss reads for keys requiring strong consistency.

kv_service.py — FastAPI KV store servicePython
import time
from fastapi import FastAPI, HTTPException
from sqlalchemy import text
import redis.asyncio as aioredis

app   = FastAPI()
cache = aioredis.Redis(host="redis", decode_responses=True)

@app.get("/kv/{namespace}/{key}")
async def kv_get(namespace: str, key: str):
    # 1. Check Redis cache first
    cache_key = f"{namespace}:{key}"
    val = await cache.get(cache_key)
    if val is not None:
        return {"value": val, "source": "cache"}

    # 2. Cache miss — hit MySQL
    now_ms = int(time.time() * 1000)
    async with get_db() as db:
        row = (await db.execute(
            text("SELECT value, expires_at FROM kv_store WHERE namespace=:ns AND key_name=:k"
                 " AND (expires_at IS NULL OR expires_at > :now)"),
            {"ns": namespace, "k": key, "now": now_ms}
        )).fetchone()

    if not row:
        raise HTTPException(404)

    # 3. Populate cache with remaining TTL
    ttl = max(1, int((row.expires_at - now_ms) / 1000)) if row.expires_at else 3600
    await cache.setex(cache_key, ttl, row.value)
    return {"value": row.value, "source": "db"}

@app.put("/kv/{namespace}/{key}")
async def kv_set(namespace: str, key: str, value: str, ttl_seconds: int = None):
    now_ms  = int(time.time() * 1000)
    exp_ms  = now_ms + ttl_seconds * 1000 if ttl_seconds else None

    async with get_db() as db:
        await db.execute(text(
            "INSERT INTO kv_store(namespace,key_name,value,expires_at,version,created_at,updated_at)"
            " VALUES(:ns,:k,:v,:exp,1,:now,:now)"
            " ON DUPLICATE KEY UPDATE value=VALUES(value),expires_at=VALUES(expires_at),"
            "version=version+1,updated_at=VALUES(updated_at)"
        ), {"ns":namespace,"k":key,"v":value,"exp":exp_ms,"now":now_ms})

    # Invalidate cache after write
    await cache.delete(f"{namespace}:{key}")
    return {"ok": True}
KVService.java — Spring BootJava
@Service
public class KVService {
    private final JdbcTemplate jdbc;
    private final StringRedisTemplate cache;

    public Optional<String> get(String ns, String key) {
        String cacheKey = ns + ":" + key;
        String cached = cache.opsForValue().get(cacheKey);
        if (cached != null) return Optional.of(cached);

        long now = System.currentTimeMillis();
        return jdbc.query(
            "SELECT value, expires_at FROM kv_store WHERE namespace=? AND key_name=?"
            + " AND (expires_at IS NULL OR expires_at > ?)",
            rs -> {
                if (!rs.next()) return Optional.empty();
                String val = rs.getString("value");
                long exp = rs.getLong("expires_at");
                long ttlSecs = exp > 0 ? (long)Math.max(1,(exp-now)/1000) : 3600L;
                cache.opsForValue().set(cacheKey, val, ttlSecs, TimeUnit.SECONDS);
                return Optional.of(val);
            }, ns, key, now);
    }

    public void set(String ns, String key, String value, Long ttlSecs) {
        long now = System.currentTimeMillis();
        Long exp = ttlSecs != null ? now + ttlSecs * 1000L : null;
        jdbc.update(
            "INSERT INTO kv_store(namespace,key_name,value,expires_at,version,created_at,updated_at)"
            + " VALUES(?,?,?,?,1,?,?) ON DUPLICATE KEY UPDATE"
            + " value=VALUES(value),expires_at=VALUES(expires_at),version=version+1,updated_at=?",
            ns, key, value, exp, now, now, now);
        cache.delete(ns + ":" + key);   // invalidate cache
    }
}
Tradeoffs

✅ SQL KV Store Pros

  • Full ACID durability — survives any crash without data loss
  • Range queries on keys — impossible with Redis
  • CAS atomically via UPDATE WHERE version=N
  • SQL joins — can query KV data alongside relational data
  • Replication, backup, PITR all come for free with MySQL

❌ SQL KV Store Cons

  • Higher latency than Redis (disk I/O vs in-memory)
  • Write throughput ceiling ~10-50K/s per node vs Redis 1M/s
  • TTL expiry is eventual (background job) not precise
  • Schema migrations require careful planning at scale
  • Not suitable for sub-millisecond requirements

Outcome
  • Composite PK (namespace, key) distributes writes — avoids the right-hand insert hotspot of auto-increment PKs. InnoDB's clustered index co-locates namespace data for efficient range scans.
  • UPSERT (INSERT ON DUPLICATE KEY) is atomic — no separate SELECT + conditional INSERT race. One statement handles both create and update correctly.
  • CAS via version column — UPDATE WHERE version=N is an atomic check-and-set. rowcount=0 means conflict detected. Enables optimistic locking without SELECT FOR UPDATE.
  • Lazy TTL expiry + background cleanup — check expires_at > now in every GET query (lazy expiry). Background job deletes expired rows in small chunks to avoid long lock holds.
  • Redis cache in front reduces MySQL load by 90% — the SQL KV becomes a durable backing store; Redis handles the hot read path. This is the read-through cache pattern in practice.

Scaling Relational Databases

A single MySQL node handles ~10K writes/s and ~50K reads/s before becoming the bottleneck. Scaling beyond that requires a deliberate strategy at each axis.

The Scaling Ladder

1
Vertical scaling — bigger machine
Upgrade CPU, RAM, NVMe. Cheapest first step. Ceiling: largest available instance (~96 vCPU, 768 GB). Diminishing returns after 32 cores for OLTP.
2
Read replicas — horizontal read scaling
Add 1–5 MySQL replicas. Route SELECTs to replicas via ProxySQL. Primary handles only writes. Works until writes saturate the primary.
3
Connection pooling + query optimization
ProxySQL, slow query log analysis, proper indexes, EXPLAIN plans. Often doubles effective throughput before infrastructure changes.
4
Caching layer — reduce DB load
Redis for hot reads. Application-level query result caching. Can reduce DB read load by 80–95% for read-heavy workloads.
5
Functional decomposition — split by domain
Split monolith DB into: users DB, orders DB, products DB. Each independently scalable. Cross-DB joins become API calls (denormalization trade-off).
6
Sharding — horizontal write scaling
Split rows across N databases by shard key (user_id % N). Each shard handles 1/N of writes. Last resort — extreme operational complexity.
Read Scaling

Read Replicas

MySQL primary streams binary log to replicas asynchronously. Replicas apply changes with ~50ms lag (semi-sync can reduce to ~5ms). ProxySQL routes reads to replicas, writes to primary.

Replication lag means reads from replica may be stale. Use primary for reads requiring strong consistency (e.g., "read your own writes").

Index Optimization

Wrong: full table scan on 100M rows. Right: covering index = query resolved entirely from index, no row fetch. An index that reduces query time from 5s → 5ms is worth more than any hardware upgrade.

Covering index exampleSQL
-- Without index: full scan
SELECT user_id, created_at FROM orders WHERE status='pending';

-- Covering index: no row fetch needed
CREATE INDEX idx_status_covering
ON orders (status, user_id, created_at);

Write Scaling

Write Scaling Techniques

Batch Writes

Instead of 1000 INSERT statements, use one INSERT INTO … VALUES (…),(…),(…). Reduces transaction overhead, index maintenance calls, and lock acquisitions by 100×.

Batch insertSQL
-- Bad: 1000 roundtrips
INSERT INTO events (uid, ts) VALUES (1, 100);
INSERT INTO events (uid, ts) VALUES (2, 101);
-- Good: 1 roundtrip
INSERT INTO events (uid, ts)
VALUES (1,100),(2,101),(3,102); -- up to 10K rows

Write-ahead with Queue

Decouple write volume from DB throughput. App writes to Kafka/Redis queue. Worker batches and bulk-inserts. DB sees 1 bulk insert per second instead of 10K individual writes per second.

Partitioning (within one DB)

MySQL RANGE/LIST/HASH partitioning splits a table into physical sub-tables. Queries with partition key in WHERE clause only scan the relevant partition. Best for time-series data (partition by month).

Archival / Tiered Storage

Move data older than N months to a cold table or archive DB. Active table stays small → all queries stay fast. Restore archived data only when needed (rare cold path).

Sharding — Last Resort

Sharding splits rows across N independent databases. Each shard holds 1/N of the data and handles 1/N of writes. Extremely powerful — and extremely painful.

Hash Sharding

shard = user_id % N. Even distribution. Cannot easily rebalance — changing N rehashes all keys. Use consistent hashing to mitigate. Good for: user data, orders.

Range Sharding

shard = created_at range. Shard 0: Jan–Mar, Shard 1: Apr–Jun. Easy range queries within a shard. Risk: temporal hotspot (current shard gets all writes). Good for: time-series, logs.

Sharding Costs

What You Lose

  • Cross-shard JOINs require application-level aggregation
  • Global ORDER BY requires merge sort across shards
  • ACID transactions only within a shard — distributed transactions are complex
  • Schema migrations must be applied to all N shards
  • Re-sharding (changing N) requires data migration — weeks of work

Mitigations

  • Denormalize: embed data you'd JOIN into the primary row
  • Vitess / Citus: middleware that handles cross-shard queries automatically
  • Two-phase commit for cross-shard transactions (complex, avoid if possible)
  • Consistent hashing: add virtual nodes to allow gradual rebalancing
Rule: Don't shard until you've exhausted: vertical scaling → read replicas → caching → query optimization → functional decomposition → partitioning. Sharding is the last resort — it introduces irreversible complexity.
Outcome
  • The scaling ladder — exhaust vertical scaling, replicas, caching, and query optimization before sharding. Most systems never need sharding; those that do regret it without proper planning.
  • Read replicas + ProxySQL — route reads to replicas, writes to primary. Standard for 10×–50× read scaling. Replication lag is the trade-off; design your reads accordingly.
  • Covering indexes are the highest-ROI optimization — an index that allows the DB to answer a query without touching the row reduces I/O by orders of magnitude. Always EXPLAIN your slow queries.
  • Batch writes reduce DB overhead by 100× — one INSERT … VALUES (1000 rows) is dramatically more efficient than 1000 individual INSERTs. Standard for event ingestion.
  • Sharding is irreversible complexity — no cross-shard JOINs, no global ORDER BY, schema migrations across N nodes. Use Vitess/Citus to hide this complexity if sharding is unavoidable.

Outcome
  • NoSQL ≠ "no schema" — it means "not only SQL." Every NoSQL DB has a data model and access pattern constraints. Cassandra requires you to know your queries at design time.

  • Start with Postgres, specialize only when proven — Postgres handles most workloads. A premature switch to a specialized DB trades operational simplicity for marginal performance gains.

Graph Databases — When & Why

A graph DB stores data as nodes (entities) and edges (relationships). The killer feature: multi-hop traversals are O(1) per hop — no join penalty regardless of graph depth. SQL joins get exponentially slower with each additional hop.

Graph Model

The Multi-hop Problem

SQL: "Find friends of Alice's friends who work at Acme" = 3 JOINs. Each JOIN multiplies intermediate rows. At 1M users with 200 friends each, 3-hop traversal in SQL = minutes. In Neo4j = milliseconds.

Graph traversal is O(k) where k = number of edges traversed. SQL JOIN is O(N × M) at each hop — exponential blowup.
When to Use Graph DBs

✅ Use a Graph DB When

  • Relationships are first-class: friend-of-friend, path finding, recommendation
  • You need multi-hop traversals (2+ hops is common query)
  • Graph structure is dynamic and irregular
  • Fraud detection: "accounts connected through shared device/IP"
  • Knowledge graphs, recommendation engines, supply chain

❌ Don't Use a Graph DB When

  • Simple parent-child hierarchy (adjacency list in SQL is fine)
  • You need OLAP/aggregations (graph DBs are poor at SUM, GROUP BY)
  • Data is highly structured with simple 1-2 relationship hops
  • Write-heavy workload (Neo4j scales reads well, writes are harder)
  • Need horizontal write scaling (sharding graphs is very hard)

Graph vs SQL — Head to Head
Friend-of-friend query comparisonSQL vs Cypher
── SQL: 2-hop friend-of-friend ──────────────────────────────
SELECT DISTINCT u3.id, u3.name
FROM users u1
JOIN friendships f1 ON u1.id = f1.user_id
JOIN friendships f2 ON f1.friend_id = f2.user_id
JOIN users u3       ON f2.friend_id = u3.id
WHERE u1.id = 42
  AND u3.id != 42
  AND u3.id NOT IN (SELECT friend_id FROM friendships WHERE user_id=42);
-- O(F²) complexity where F = avg friends. 200 friends = 40,000 row join.

── Cypher (Neo4j): equivalent query ─────────────────────────
MATCH (u:User {id: 42})-[:FRIENDS_WITH*2..2]->(foaf:User)
WHERE NOT (u)-[:FRIENDS_WITH]->(foaf) AND foaf <> u
RETURN DISTINCT foaf.id, foaf.name
LIMIT 50;
-- O(F×depth) — graph traversal follows edges directly (pointer chasing)
-- No materialized intermediate join table

Code — Neo4j Cypher Patterns
cypher_patterns.py — py2neo driverPython
from neo4j import GraphDatabase

driver = GraphDatabase.driver("bolt://neo4j:7687", auth=("neo4j", "pass"))

def create_user_and_follow(tx, from_id, to_id):
    tx.run("""
        MERGE (a:User {id: $from_id})
        MERGE (b:User {id: $to_id})
        MERGE (a)-[:FOLLOWS]->(b)
    """, from_id=from_id, to_id=to_id)

def get_recommendations(tx, user_id, limit=10):
    """People followed by people I follow (2-hop), that I don't follow."""
    result = tx.run("""
        MATCH (me:User {id: $uid})-[:FOLLOWS]->(:User)-[:FOLLOWS]->(rec:User)
        WHERE NOT (me)-[:FOLLOWS]->(rec) AND rec.id <> $uid
        RETURN rec.id AS id, rec.name AS name, COUNT(*) AS common_connections
        ORDER BY common_connections DESC
        LIMIT $limit
    """, uid=user_id, limit=limit)
    return [dict(row) for row in result]

def shortest_path(tx, from_id, to_id):
    """Degrees of separation between two users."""
    result = tx.run("""
        MATCH path = shortestPath(
            (a:User {id: $from})-[:FOLLOWS*]-(b:User {id: $to})
        )
        RETURN length(path) AS degrees, [n IN nodes(path) | n.name] AS via
    """, from=from_id, to=to_id)
    return result.single()

# Usage with session
with driver.session() as session:
    recs = session.execute_read(get_recommendations, user_id=42)
Outcome
  • Graph traversal is O(k) per hop — doesn't depend on total nodes/edges. SQL JOIN is O(N×M) at each hop — exponential. This is the decisive advantage for multi-hop relationship queries.
  • Graph DBs are not general-purpose — poor at aggregations, harder to scale writes horizontally, no SQL ecosystem. Use them only when relationships are the primary query pattern.
  • Fraud detection = canonical graph use case — detecting rings of accounts sharing devices/IPs/cards requires multi-hop traversal that's impractical in SQL at production scale.
  • Cypher is declarative like SQL — MATCH (a)-[:FOLLOWS*2]->(b) is elegant and readable. The pattern-matching syntax is one of Neo4j's strongest UX advantages.
  • Adjacency list in Postgres is often enough — for 1–2 hop queries and moderate scale, a self-referential table with recursive CTEs handles it without a graph DB. Reserve Neo4j for 3+ hops at scale.

Architecture — Realtime Chat

Why Redis Pub/Sub for Fan-out?

Alice sends a message to Channel #engineering. Bob and 50 others are in the channel, connected to different chat servers. Server 1 (Alice's server) publishes to Redis channel channel:engineering. All chat servers subscribed to that channel receive the message and push it to their connected clients. Without Redis Pub/Sub, Server 1 would need to know which servers Bob is on — stateful and fragile.

Why Cassandra for Messages?

Message reads are always: "Give me the last N messages in channel X." Cassandra partitions by channel_id and clusters by message_id DESC — this is exactly a Cassandra scan: one partition, descending time order. Reads are O(N) with no JOINs, no cross-partition queries.
Schema Design
Cassandra messages tableCQL
-- Message storage: partition by channel, time-ordered
CREATE TABLE messages (
    channel_id   UUID,
    message_id   TIMEUUID,    -- monotonic, time-sortable
    sender_id    UUID,
    content      TEXT,
    msg_type     TEXT,        -- text | image | file | reaction
    edited_at    TIMESTAMP,
    deleted      BOOLEAN     DEFAULT false,
    PRIMARY KEY (channel_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);

-- Fetch last 50 messages in a channel — single partition read
SELECT * FROM messages
WHERE channel_id = ? LIMIT 50;

-- Pagination: fetch before a given message
SELECT * FROM messages
WHERE channel_id = ? AND message_id < ?   -- cursor-based
LIMIT 50;

-- Delivery receipts (separate table — high write rate)
CREATE TABLE delivery_receipts (
    message_id   UUID,
    user_id      UUID,
    status       TEXT,   -- sent | delivered | read
    updated_at   TIMESTAMP,
    PRIMARY KEY (message_id, user_id)
);

Why TIMEUUID for message_id?

Monotonic ordering

TIMEUUID encodes timestamp + UUID suffix → globally unique, time-sortable without a central ID generator. Two messages created at the same millisecond have different UUIDs. Cassandra can sort them without a separate timestamp column.

No ID generator SPOF

Auto-increment IDs require a single sequence generator — a bottleneck in distributed systems. TIMEUUID = each client generates its own message ID independently. Conflicts are statistically impossible (UUID space = 2^122).

Code — WebSocket Chat Handler
chat_server.py — FastAPI WebSocket + Redis Pub/SubPython
import uuid, asyncio
from fastapi import FastAPI, WebSocket, WebSocketDisconnect
import redis.asyncio as aioredis
from cassandra.cluster import Cluster

app    = FastAPI()
redis  = aioredis.Redis(host="redis")
cass   = Cluster(["cassandra"]).connect("chat")

# Local registry: channel_id → set of active WebSockets on THIS server
local_subs: dict[str, set] = {}

@app.websocket("/ws/{user_id}/{channel_id}")
async def ws_handler(ws: WebSocket, user_id: str, channel_id: str):
    await ws.accept()
    local_subs.setdefault(channel_id, set()).add(ws)

    # Subscribe to Redis channel for this chat channel
    pubsub = redis.pubsub()
    await pubsub.subscribe(f"channel:{channel_id}")
    asyncio.create_task(_forward_redis_to_ws(pubsub, channel_id))

    try:
        while True:
            data = await ws.receive_json()   # {"content": "Hello!"}

            # 1. Persist to Cassandra
            msg_id = uuid.uuid1()           # TIMEUUID
            cass.execute(
                "INSERT INTO messages(channel_id,message_id,sender_id,content) VALUES(%s,%s,%s,%s)",
                (channel_id, msg_id, user_id, data["content"])
            )

            # 2. Publish to Redis — all chat servers get this
            msg = {"id": str(msg_id), "sender": user_id,
                   "content": data["content"], "channel": channel_id}
            await redis.publish(f"channel:{channel_id}", str(msg))

    except WebSocketDisconnect:
        local_subs[channel_id].discard(ws)

async def _forward_redis_to_ws(pubsub, channel_id: str):
    """Forward Redis Pub/Sub messages to all local WebSocket subscribers."""
    async for msg in pubsub.listen():
        if msg["type"] == "message":
            dead = set()
            for ws in local_subs.get(channel_id, set()):
                try:
                    await ws.send_text(msg["data"])
                except:
                    dead.add(ws)
            local_subs[channel_id] -= dead   # clean up disconnected clients
Outcome
  • Redis Pub/Sub solves the cross-server fan-out problem — without it, each chat server would need to know where every user is connected. With it, just publish once; all servers forward to their local connections.
  • Cassandra is the ideal message store — time-series writes, partition by channel, cluster by time. Every "get last N messages" query is a single-partition scan: fast regardless of total message count.
  • TIMEUUID = distributed ID + time sort in one — no ID generator service needed. Clients generate their own message IDs. Collision probability is astronomically small.
  • Sticky sessions needed for WebSocket LB — because WebSocket connections are stateful (Alice's socket exists on Server 1), the load balancer must route all of Alice's traffic to Server 1. ALB supports session stickiness via cookie.
  • Offline delivery via push notifications — if Carol isn't connected, the message is already persisted in Cassandra. On reconnect, Carol fetches history. Meanwhile, a push notification wakes the mobile app.

Scaling WebSockets

WebSockets are persistent, stateful, bidirectional connections. Each connected client holds an open TCP connection. This fundamentally changes how you scale — horizontal scale-out is harder than for stateless REST.

WebSocket vs REST Scaling Model

REST (Stateless)

  • Any server can handle any request
  • Load balancer distributes freely round-robin
  • Server crashes → retry on any other server
  • Scale out: add servers, no coordination
  • Memory: minimal (request/response only)

WebSocket (Stateful)

  • Client is pinned to a specific server for the session
  • Load balancer needs sticky sessions (by cookie or IP)
  • Server crash → all its clients must reconnect
  • Scale out: add servers + re-balance connections
  • Memory: 1MB+ per connection × 10K connections = 10GB
Scaling Challenges & Solutions

Challenge 1 — Message Fan-out Across Servers

Alice (Server 1) sends a message to a channel. Bob and Carol are on Server 2 and Server 3. Server 1 alone cannot deliver to them.

Solution: Redis Pub/Sub or Kafka as message bus. Sending server publishes; all servers subscribe per-channel. Each server delivers to its own locally-connected clients.

Challenge 2 — Sticky Session Load Balancing

Standard round-robin LB breaks WebSocket — client connects to Server 1, next request may hit Server 2 where no connection exists.

Solution: AWS ALB sticky sessions (LB cookie, 1-day duration). Or consistent hashing on user_id to deterministically route to the same server. Or connection-level LB (Layer 4 NLB).

Challenge 3 — Connection Recovery

Server crashes → all its clients disconnect. Naive clients → reconnect storm → new server overwhelmed.

Solution: Exponential backoff with jitter on client reconnect. Stagger reconnect attempts: wait = base * 2^attempt + random(0, 1000ms). Distributes reconnect load across 30–60 seconds instead of all at once.

Challenge 4 — Horizontal Scaling

10K connections per server × 100 servers = 1M concurrent connections. Need to know which server holds which user's connection for targeted delivery (DMs).

Solution: Redis hash: SET user:{user_id}:server {server_id}. Any server can look up where a user is connected and route the message via Redis Pub/Sub to that server.

Challenge 5 — Memory & File Descriptors

Memory per connection

Each WebSocket conn: ~64KB kernel buffers + app-level buffers = ~1MB. At 10K connections: 10 GB RAM needed per server. Use Node.js or Go for lower per-connection overhead (event loop vs thread-per-connection).

File descriptor limit

Linux default: 1024 open file descriptors per process. WebSocket = 1 FD each. Increase: ulimit -n 1000000 and /etc/sysctl.conf: net.core.somaxconn = 65535.

Outcome
  • WebSockets require sticky sessions — unlike REST, the LB must route each client to the same server for the connection lifetime. AWS ALB cookie-based stickiness or consistent hashing on user_id achieves this.
  • Redis Pub/Sub solves cross-server fan-out — the canonical solution for <100K concurrent users. Beyond that, switch to Kafka for higher throughput and durability of the message bus.
  • Connection registry in Redis — user:{id}:server = server_id lets any server look up where a user is connected for direct message routing without broadcasting to all servers.
  • Exponential backoff is mandatory — if a server crashes and 10K clients all reconnect simultaneously, the new server is immediately overwhelmed. Jittered backoff distributes the reconnect storm across 30–60 seconds.
  • Go and Node.js for WS servers — event-loop models handle 10K–100K concurrent connections on one core. Thread-per-connection models (Java blocking I/O, Python sync) cap out at ~1K connections per core. Use async/non-blocking runtimes.
  • File descriptor limits are a real constraint — always set ulimit -n to at least 1,000,000 on WS servers. Default Linux limit of 1024 will cause mysterious failures at production scale.
