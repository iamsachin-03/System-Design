Notification System
Design a scalable notification platform supporting push (APNs/FCM), email, SMS, and in-app notifications — with fan-out for bulk sends, template rendering, user preferences, deduplication, retry with backoff, and at-least-once delivery guarantees.
🔴 MUST DO 🟢 Medium APNs / FCM Fan-out Kafka Templates At-Least-Once Idempotency
🎯 Clarify Scope
Notification systems look simple but hide complexity in fan-out for bulk sends, provider reliability (APNs/FCM can fail), user preferences (opt-outs, quiet hours), deduplication, and guaranteeing every critical notification is delivered exactly once.

Latency: ~100-500ms

💬

Twilio, AWS SNS SMS. 160 chars per segment. International routing. High cost. Best for OTP, critical alerts.
Latency: 1–10 seconds

Stored in DB, shown in notification center. Persisted forever. No 3rd party. Badge count on app icon.
Latency: real-time (WS)
💬
Web Push
Browser push via Web Push Protocol. Works when browser is closed. Requires user permission. Lower CTR than mobile.
Latency: ~500ms
✅ Functional Requirements
CORE
Send Notification: Accept notification request (to user, channel, content), deliver via the appropriate provider(s). Return immediately (async delivery).
Bulk Broadcast: Send notification to N users (up to 100M). Fan-out without blocking the caller.
User Preferences: Respect per-user opt-outs (e.g., "no marketing emails"), quiet hours, and per-channel preferences.
Retry with Backoff: Failed deliveries (provider errors, network issues) are retried with exponential backoff. Max 3 retries.
Deduplication: Each notification has a unique ID. If the same ID is submitted twice, deliver only once.
Delivery Tracking: Track status per notification: QUEUED → SENDING → DELIVERED | FAILED.
EXTENDED
Templates: Pre-defined templates with variable substitution. Different templates for different languages/regions.
Scheduling: Send at a specific time (e.g., "send at 9am user's local time")
Analytics: Open rates, click rates, opt-out rates per notification campaign
Rate Limiting: Don't send more than N notifications per user per hour (spam prevention)
⚡ Non-Functional Requirements
PERFORMANCE
API response: <100ms (async — just queue the notification)
Push delivery p99: <5 seconds for transactional
Bulk of 100M: complete fanout within 30 minutes
RELIABILITY
At-least-once delivery: every notification eventually delivered or moved to DLQ
Idempotent: duplicate sends don't spam users
Provider failure: fallback to secondary provider (e.g., SendGrid → SES)
SCALE
Transactional: 10M/day = ~116/sec avg, ~500/sec peak
Marketing: 100M/day, sent in batches over 30 min = ~55,000/sec
Kafka-backed fan-out: infinitely scalable with consumer groups
COMPLIANCE
GDPR: user can opt out of all marketing communications
CAN-SPAM: commercial emails must include unsubscribe link
Quiet hours respected (10pm–8am in user's timezone)
🎙 Phase 1 — Architecture & Asynchronous Design
PHASE 1 · ASYNC PIPELINE
IR
INTERVIEWER
Design a notification system. Start by explaining the high-level architecture.
ME
YOU
The fundamental principle: the notification system must be fully asynchronous. The caller (e.g., the payment service saying "notify user that payment succeeded") should get a 202 Accepted in <100ms. The actual delivery to APNs/FCM/SMTP happens in the background.

Why async is essential:
  • APNs/FCM can take 100-500ms per notification. At 500/sec, you can't do this synchronously.
  • Providers fail. If we call APNs synchronously and it's down, the payment service call blocks/fails.
  • Bulk sends (100M recipients) would block the caller for hours synchronously.
Architecture:
  1. Notification Service API — accepts notification requests, validates, persists to DB, publishes to Kafka. Returns 202 immediately.
  2. Kafka — durable message queue. Topic per channel: notifications.push, notifications.email, notifications.sms.
  3. Channel Workers — Kafka consumers per channel. Read from queue, check user preferences, render template, call provider.
  4. Providers — APNs (iOS), FCM (Android), SES/SendGrid (email), Twilio (SMS).
The key insight: each channel is independently scalable. If email is slow, add more email workers. Push notifications are fast (APNs is <500ms) while email can take seconds — they shouldn't block each other.
💡 Always lead with "fully asynchronous" for notification systems. The API never blocks on provider calls — this immediately shows you understand the reliability and scalability requirements.
📤 Phase 2 — Fan-out for Bulk Sends
PHASE 2 · FAN-OUT
IR
INTERVIEWER
A marketing campaign sends a push notification to 100 million users. Walk me through the fan-out.
ME
YOU
Sending to 100M users is a fan-out problem. You can't build a single message with 100M recipients and put it in Kafka — that's a 10GB message. The right approach:

Step 1 — Create campaign record:
Caller submits: POST /notifications/campaigns with template_id, segment (all_users or specific cohort), scheduled_at. Campaign record created with status PENDING.

Step 2 — Fan-out Service (Kafka consumer):
Campaign event published to Kafka. Fan-out Service consumes it and:
  1. Queries User Service to get all users in the segment (paginated, 10,000 at a time)
  2. For each batch of 10,000 users, checks user preferences (opt-outs, quiet hours) in batch
  3. Creates individual notification messages in Kafka: one message per user (not one message with 100M users)
  4. At 55,000 messages/sec, completing 100M in 30 minutes
Kafka topic partitioning: Partition by user_id. This ensures notifications for the same user always go to the same partition — preserving ordering and preventing duplicate delivery from race conditions.

Worker parallelism: 100 push worker instances each consuming from different Kafka partitions. Each handles ~550 messages/sec. APNs can handle thousands of concurrent connections with HTTP/2 multiplexing — 100 workers × 550/sec = 55,000/sec total.
IR
INTERVIEWER
What happens if the fan-out service crashes mid-way through a 100M batch?
ME
YOU
The fan-out service tracks its progress in the campaign record:
UPDATE campaigns SET last_processed_user_id = {cursor}, processed_count = {n}

When it restarts (or a replacement consumer picks up the Kafka message), it reads the campaign record, sees the cursor, and resumes from where it left off. No notifications are re-sent to users already processed.

The user batch processing is idempotent: checking if a user has already received this campaign notification (Redis SET with TTL: SADD notif_sent:{campaign_id} {user_id}) before sending. Even if the fan-out runs twice, each user gets exactly one notification.
⚙️ Phase 3 — User Preferences & Quiet Hours
PHASE 3 · PREFERENCES
IR
INTERVIEWER
How do you handle user preferences — opt-outs, quiet hours, per-channel settings?
ME
YOU
User preferences form a preference hierarchy:

1. Global opt-out: User has unsubscribed from all marketing. Never send marketing, regardless of channel.
2. Channel opt-out: User opts out of email but keeps push. Per-channel boolean flags.
3. Notification type opt-out: User wants order updates but not promotional offers. Per-type preferences.
4. Quiet hours: Don't send between 10pm and 8am in the user's timezone.

Storage: User preference record in PostgreSQL: user_preferences(user_id, channel, type, is_opted_in, quiet_start_hr, quiet_end_hr, timezone). Cached in Redis (TTL 5 minutes) for fast reads during high-throughput fan-outs.

Quiet hours enforcement: When a notification is picked up by a channel worker, check if current time is within quiet hours for this user. If yes, delay the notification by scheduling it in a delayed queue (Redis ZSET with score = delivery_timestamp). A scheduler picks up due notifications and re-queues them to the channel worker.

Notification type priority overrides: Critical notifications (account security, OTP, fraud alerts) bypass quiet hours and marketing opt-outs. The notification request includes a priority field: CRITICAL | TRANSACTIONAL | MARKETING. Only CRITICAL bypasses all user preferences.
🔁 Phase 4 — Retry, Idempotency & At-Least-Once
PHASE 4 · RELIABILITY
IR
INTERVIEWER
APNs returns an error for 5% of notifications. How do you handle retries without spamming users?
ME
YOU
Retry strategy depends on the error type — not all errors are retriable:

Retriable errors (transient):
  • APNs HTTP 429 (rate limited) → backoff and retry
  • APNs HTTP 500/503 (provider down) → retry with exponential backoff
  • Network timeout → retry after 1s, 2s, 4s, 8s (max 3 retries)
Non-retriable errors (permanent):
  • APNs HTTP 400 with error code BadDeviceToken → device token is invalid, delete it from our database. User may have uninstalled the app.
  • APNs HTTP 410 Unregistered → token expired. Delete token, user re-registers on next app open.
  • SMS number invalid → don't retry.
Idempotency (no duplicate sends):
Every notification has a unique notification_id (generated by the caller or auto-assigned). Before sending to the provider, the worker checks Redis: SET notif_sent:{notification_id} 1 NX EX 86400. If key already exists → this notification was already sent → skip. The NX ensures only one worker sends even if the Kafka message is processed twice (rebalance, worker restart).
IR
INTERVIEWER
After 3 retries, the notification still fails. What happens?
ME
YOU
The notification is moved to a Dead Letter Queue (DLQ). The DLQ is a separate Kafka topic: notifications.dlq. It includes the original message, all error details, retry count, and timestamps.

The DLQ serves two purposes:
1. On-call alerts: A DLQ monitor fires a PagerDuty/Slack alert when DLQ depth exceeds a threshold (e.g., 1000 messages). This indicates a systemic provider issue needing investigation.
2. Manual replay: Engineers can inspect DLQ messages, fix the underlying issue (e.g., APNs credentials expired), and replay the messages. This prevents permanent notification loss.

For transactional notifications (e.g., OTP), failure to deliver after retries should trigger an alternative channel: if push fails, try SMS. This fallback channel logic is configured per notification type.
📋 Phase 5 — Templates & Analytics
PHASE 5 · TEMPLATES
IR
INTERVIEWER
How do you handle templated notifications with personalisation?
ME
YOU
Templates separate content from code. A marketing team creates templates without deploying code. Key components:

Template definition:
{"id": "order_shipped", "channel": "push", "title": "Your order is on its way! 🚚", "body": "Hi {{user.first_name}}, your order #{{order.id}} shipped. Estimated delivery: {{delivery.date}}"}

Rendering: Template Engine (Handlebars/Jinja2) renders the template with a context object at send time. The context comes from the notification request and is enriched from the User Service and Order Service.

Internationalisation (i18n): Each template has variants per locale: template_id:en-US, template_id:es-MX. The worker fetches the user's preferred language, picks the right template variant. Falls back to en-US if the user's locale isn't translated.

A/B testing: Templates support variants. Marketing can run A/B tests on subject lines or body copy. The Fan-out Service randomly assigns users to variant A or B (bucketed by hash(user_id) % 2). Analytics tracks open/click rates per variant.

-- ── PostgreSQL: Notifications ─────────────────────────
CREATE TABLE notifications (
    notification_id  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    idempotency_key  VARCHAR(256) UNIQUE,        -- caller dedup key
    user_id          BIGINT       NOT NULL,
    channel          ENUM('push','email','sms','in_app'),
    priority         ENUM('CRITICAL','TRANSACTIONAL','MARKETING'),
    template_id      VARCHAR(100),
    template_data    JSONB,
    rendered_title   TEXT,
    rendered_body    TEXT,
    status           ENUM('QUEUED','SENDING','DELIVERED','FAILED','SKIPPED'),
    retry_count      INT          DEFAULT 0,
    last_error       TEXT,
    scheduled_at     TIMESTAMP,
    queued_at        TIMESTAMP    DEFAULT NOW(),
    delivered_at     TIMESTAMP,
    ttl_seconds      INT          DEFAULT 86400,
    campaign_id      UUID                             -- null for transactional
);
CREATE INDEX idx_notif_user    ON notifications (user_id, queued_at DESC);
CREATE INDEX idx_notif_status  ON notifications (status) WHERE status != 'DELIVERED';

-- ── PostgreSQL: User Preferences ──────────────────────
CREATE TABLE user_notification_prefs (
    user_id           BIGINT      PRIMARY KEY,
    global_opt_out    BOOLEAN     DEFAULT FALSE,
    push_enabled      BOOLEAN     DEFAULT TRUE,
    email_enabled     BOOLEAN     DEFAULT TRUE,
    sms_enabled       BOOLEAN     DEFAULT FALSE,
    marketing_enabled BOOLEAN     DEFAULT TRUE,
    quiet_start_hr    SMALLINT    DEFAULT 22,    -- 10pm
    quiet_end_hr      SMALLINT    DEFAULT 8,     -- 8am

    token_id     UUID         PRIMARY KEY,
    user_id      BIGINT       NOT NULL,
    token        TEXT         NOT NULL UNIQUE,
    platform     ENUM('ios','android','web'),
    is_active    BOOLEAN      DEFAULT TRUE,
    registered   TIMESTAMP,
    last_used    TIMESTAMP
);

-- ── Redis Data Structures ─────────────────────────────
-- Idempotency key (prevents duplicates)
-- Key: notif_dedup:{idempotency_key}  →  notification_id  EX 86400

-- User preference cache
-- Key: user_prefs:{user_id}  →  HASH (all pref fields)  EX 300

// Notification System — Java Implementation
// ══════════════════════════════════════════════════════

// ── 1. Notification Service (API layer) ───────────────
@RestController
public class NotificationController {

    @Autowired private NotificationService notifSvc;

    @PostMapping("/api/v1/notifications")
    public ResponseEntity<NotifResponse> send(
            @RequestBody NotifRequest req) {
        return ResponseEntity.status(202)
            .body(notifSvc.send(req));
    }
}

@Service
public class NotificationService {

    @Autowired private NotificationRepository  repo;
    @Autowired private KafkaTemplate<String,NotifMessage> kafka;
    @Autowired private RedisTemplate<String,String>  redis;
    @Autowired private TemplateEngine              templates;

    public NotifResponse send(NotifRequest req) {
        // 1. Idempotency check: return existing if key already seen
        String dedupKey = "notif_dedup:" + req.getIdempotencyKey();
        String existing  = redis.opsForValue().get(dedupKey);
        if (existing != null) {
            return new NotifResponse(existing, "ALREADY_QUEUED");
        }

        // 2. Render template (if using template_id)
        RenderedContent content = req.getTemplateId() != null
            ? templates.render(req.getTemplateId(), req.getTemplateData(),
                               req.getUserLocale())
            : RenderedContent.fromRaw(req.getContent());

        // 3. Persist notification record
        Notification notif = Notification.builder()
            .notificationId(UUID.randomUUID().toString())
            .idempotencyKey(req.getIdempotencyKey())
            .userId(req.getUserId())
            .channel(req.getChannel())
            .priority(req.getPriority())
            .renderedTitle(content.getTitle())
            .renderedBody(content.getBody())
            .status("QUEUED")
            .ttlSeconds(req.getTtlSeconds())
            .scheduledAt(req.getScheduledAt())
            .build();
        repo.save(notif);

        // 4. Store idempotency key in Redis (24h TTL)
        redis.opsForValue().set(dedupKey, notif.getNotificationId(),
            Duration.ofHours(24));

        // 5. Publish to appropriate Kafka topic
        String topic = "notifications." + req.getChannel().toLowerCase();
        kafka.send(topic,
            req.getUserId().toString(), // key = user_id (same partition = ordering)
            new NotifMessage(notif));

        return new NotifResponse(notif.getNotificationId(), "QUEUED");
    }
}

// ── 2. Push Worker ────────────────────────────────────
@Component
public class PushWorker {

    @Autowired private ApnsClient          apns;
    @Autowired private FcmClient           fcm;
    @Autowired private PreferenceService   prefs;
    @Autowired private DeviceTokenRepo     tokens;
    @Autowired private NotificationRepo    repo;
    @Autowired private KafkaTemplate<String,NotifMessage> kafka;

    private static final int    MAX_RETRIES = 3;
    private static final long[] BACKOFF_MS  = {1000, 4000, 16000}; // 1s, 4s, 16s

    @KafkaListener(topics = "notifications.push",
                   groupId = "push-workers", concurrency = "60")
    public void onMessage(NotifMessage msg) {
        Notification notif = msg.getNotification();

        // 1. Check TTL — discard stale notifications (e.g., old OTPs)
        if (Instant.now().isAfter(
                notif.getQueuedAt().plusSeconds(notif.getTtlSeconds()))) {
            repo.updateStatus(notif.getNotificationId(), "EXPIRED");
            return;
        }

        // 2. Check user preferences (cached in Redis)
        if (!prefs.isAllowed(notif.getUserId(), "push",
                             notif.getPriority())) {
            repo.updateStatus(notif.getNotificationId(), "SKIPPED");
            return;
        }

        // 3. Check quiet hours (CRITICAL bypasses)
        if (notif.getPriority() != Priority.CRITICAL
                && prefs.isQuietHours(notif.getUserId())) {
            // Delay: put in Redis delayed queue, scheduler will re-queue
            scheduleForAfterQuietHours(notif);
            return;
        }

        // 4. Get device tokens for this user
        List<DeviceToken> userTokens = tokens.getActiveTokens(notif.getUserId());
        if (userTokens.isEmpty()) {
            repo.updateStatus(notif.getNotificationId(), "SKIPPED");
            return;
        }

        // 5. Send to each device with retry
        for (DeviceToken token : userTokens) {
            sendWithRetry(notif, token, 0);
        }
    }

    private void sendWithRetry(Notification notif,
                                  DeviceToken token,
                                  int attempt) {
        try {
            PushResult result = token.getPlatform().equals("ios")
                ? apns.send(token.getToken(), notif)
                : fcm.send(token.getToken(), notif);

            if (result.isSuccess()) {
                repo.updateStatus(notif.getNotificationId(), "DELIVERED");

            } else if (result.isPermanentFailure()) {
                // BadDeviceToken, Unregistered — delete token, don't retry
                tokens.deactivate(token.getTokenId());
                repo.updateStatus(notif.getNotificationId(), "FAILED");

            } else if (attempt < MAX_RETRIES) {
                // Transient error — retry with exponential backoff
                Thread.sleep(BACKOFF_MS[attempt]);
                sendWithRetry(notif, token, attempt + 1);

            } else {
                // Max retries exhausted — send to DLQ
                repo.updateStatus(notif.getNotificationId(), "FAILED",
                    result.getErrorMessage(), MAX_RETRIES);
                kafka.send("notifications.dlq",
                    new DlqMessage(notif, result.getErrorMessage()));
            }
        } catch (Exception e) {
            if (attempt < MAX_RETRIES) {
                sendWithRetry(notif, token, attempt + 1);
            } else {
                kafka.send("notifications.dlq",
                    new DlqMessage(notif, e.getMessage()));
            }
        }
    }

    private void scheduleForAfterQuietHours(Notification notif) {
        Instant quietEnd = prefs.getQuietHoursEnd(notif.getUserId());
        redis.opsForZSet().add("notif_delayed",
            notif.getNotificationId(),
            (double) quietEnd.toEpochMilli());
    }
}

// ── 3. Fan-out Service ────────────────────────────────
@Component
public class FanoutService {

    @Autowired private UserService         userSvc;
    @Autowired private PreferenceService   prefs;
    @Autowired private KafkaTemplate<String,NotifMessage> kafka;
    @Autowired private CampaignRepository campaigns;
    @Autowired private RedisTemplate<String,String>  redis;

    private static final int BATCH_SIZE = 10_000;

    @KafkaListener(topics = "notifications.fanout")
    public void onCampaign(CampaignEvent event) {
        Campaign campaign = campaigns.findById(event.getCampaignId());
        String   cursor   = campaign.getLastProcessedUserId(); // resume on crash

        while (true) {
            // Fetch next batch of users (paginated)
            List<long> userIds = userSvc.getUsersInSegment(
                event.getSegment(), cursor, BATCH_SIZE);
            if (userIds.isEmpty()) break; // done

            // Batch check preferences (one Redis pipeline call)
            Map<Long, UserPrefs> prefMap = prefs.batchGet(userIds);

            List<Long> eligibleUsers = userIds.stream()
                .filter(uid -> {
                    UserPrefs p = prefMap.get(uid);
                    return p != null
                        && !p.isGlobalOptOut()
                        && p.isChannelEnabled(event.getChannel())
                        && p.isMarketingEnabled();
                })
                .collect(Collectors.toList());

            // Create individual Kafka messages for eligible users
            eligibleUsers.forEach(uid -> {
                // Skip if already sent in this campaign (resume safety)
                if (redis.opsForSet().isMember(
                        "campaign_sent:" + event.getCampaignId(),
                        uid.toString())) {
                    return;
                }
                kafka.send("notifications." + event.getChannel(),
                    uid.toString(),
                    new NotifMessage(buildNotif(event, uid)));
                redis.opsForSet().add(
                    "campaign_sent:" + event.getCampaignId(),
                    uid.toString());
            });

            // Save cursor for fault tolerance
            cursor = String.valueOf(userIds.get(userIds.size() - 1));
            campaigns.updateCursor(event.getCampaignId(), cursor,
                campaign.getProcessedCount() + userIds.size());
        }
        campaigns.updateStatus(event.getCampaignId(), "COMPLETED");
    }
}

# ── 1. Notification Service ───────────────────────────
class NotificationService:
    """
    Accepts notification requests, checks idempotency,
    persists, and publishes to Kafka. Always returns fast.
    """

    def __init__(self, db, kafka, redis, template_engine):
        self.db        = db
        self.kafka     = kafka
        self.redis     = redis
        self.templates = template_engine

    async def send(self, req: dict) -> dict:
        """POST /api/v1/notifications handler"""
        # 1. Idempotency check
        dedup_key = f"notif_dedup:{req['idempotency_key']}"
        existing  = await self.redis.get(dedup_key)
        if existing:
            return {"notification_id": existing, "status": "ALREADY_QUEUED"}

        # 2. Render template
        if req.get("template_id"):
            content = await self.templates.render(
                req["template_id"], req.get("template_data", {}),
                req.get("locale", "en-US"))
        else:
            content = req.get("content", {})

        # 3. Persist
        notif_id = str(uuid.uuid4())
        notif = {
            "notification_id": notif_id,
            "idempotency_key": req["idempotency_key"],
            "user_id":         req["user_id"],
            "channel":         req.get("channel", "push"),
            "priority":        req.get("priority", "TRANSACTIONAL"),
            "title":           content.get("title"),
            "body":            content.get("body"),
            "status":          "QUEUED",
            "ttl_seconds":     req.get("ttl_seconds", 86400),
            "queued_at":       datetime.now(timezone.utc).isoformat(),
        }
        await self.db.insert("notifications", notif)

        # 4. Cache idempotency key
        await self.redis.setex(dedup_key, 86400, notif_id)

        # 5. Publish to Kafka
        topic = f"notifications.{req.get('channel', 'push')}"
        await self.kafka.send(
            topic,
            key=str(req["user_id"]),  # partition by user_id
            value=notif
        )
        return {"notification_id": notif_id, "status": "QUEUED"}


# ── 2. Push Worker ────────────────────────────────────
class PushWorker:
    """
    Kafka consumer for notifications.push topic.
    Handles APNs (iOS) and FCM (Android) delivery.
    Retry with exponential backoff. Dead letter queue on exhaustion.
    """
    MAX_RETRIES = 3
    BACKOFF_SEC = [1, 4, 16]  # exponential backoff

    def __init__(self, apns_client, fcm_client, prefs_service,
                  token_repo, notif_repo, kafka):
        self.apns      = apns_client
        self.fcm       = fcm_client
        self.prefs     = prefs_service
        self.tokens    = token_repo
        self.notifs    = notif_repo
        self.kafka     = kafka

    async def on_message(self, msg: dict):
        notif = msg
        user_id = notif["user_id"]

        # 1. TTL check — discard stale notifications
        queued_at = datetime.fromisoformat(notif["queued_at"])
        ttl_sec   = notif.get("ttl_seconds", 86400)
        age_sec   = (datetime.now(timezone.utc) - queued_at).total_seconds()
        if age_sec > ttl_sec:
            await self.notifs.update_status(notif["notification_id"], "EXPIRED")
            return

        # 2. User preferences check
        allowed = await self.prefs.is_allowed(
            user_id, "push", notif.get("priority"))
        if not allowed:
            await self.notifs.update_status(notif["notification_id"], "SKIPPED")
            return

        # 3. Quiet hours (CRITICAL bypasses)
        if notif.get("priority") != "CRITICAL":
            in_quiet = await self.prefs.is_quiet_hours(user_id)
            if in_quiet:
                await self._delay_until_quiet_end(notif)
                return

        # 4. Get device tokens
        device_tokens = await self.tokens.get_active(user_id)
        if not device_tokens:
            await self.notifs.update_status(notif["notification_id"], "SKIPPED")
            return

        # 5. Send to each device
        tasks = [self._send_with_retry(notif, t) for t in device_tokens]
        await asyncio.gather(*tasks, return_exceptions=True)

    async def _send_with_retry(self, notif: dict, token: dict,
                                  attempt: int = 0):
        platform = token.get("platform")
        try:
            if platform == "ios":
                result = await self.apns.send(token["token"], notif)
            else:
                result = await self.fcm.send(token["token"], notif)

            if result["success"]:
                await self.notifs.update_status(
                    notif["notification_id"], "DELIVERED")

            elif result.get("permanent_failure"):
                # BadDeviceToken / Unregistered — delete token
                await self.tokens.deactivate(token["token_id"])
                await self.notifs.update_status(
                    notif["notification_id"], "FAILED")

            elif attempt < self.MAX_RETRIES:
                await asyncio.sleep(self.BACKOFF_SEC[attempt])
                await self._send_with_retry(notif, token, attempt + 1)

            else:
                await self._move_to_dlq(notif, result.get("error"))

        except Exception as e:
            if attempt < self.MAX_RETRIES:
                await asyncio.sleep(self.BACKOFF_SEC[attempt])
                await self._send_with_retry(notif, token, attempt + 1)
            else:
                await self._move_to_dlq(notif, str(e))

    async def _move_to_dlq(self, notif: dict, error: str):
        await self.notifs.update_status(
            notif["notification_id"], "FAILED", error=error)
        await self.kafka.send("notifications.dlq",
            key=str(notif["user_id"]),
            value={**notif, "error": error, "failed_at": time.time()})


# ── 3. Preference Service with Cache ──────────────────
class PreferenceService:
    """
    Checks user notification preferences.
    Cached in Redis (TTL 5 min) to handle bulk campaigns.
    Batch GET for fan-out efficiency.
    """
    CACHE_TTL = 300

    def __init__(self, db, redis):
        self.db    = db
        self.redis = redis

    async def is_allowed(self, user_id: int, channel: str,
                           priority: str) -> bool:
        prefs = await self._get_cached(user_id)
        if not prefs:
            return True  # fail open

        # CRITICAL always goes through
        if priority == "CRITICAL":
            return True

        if prefs.get("global_opt_out"):
            return False

        if not prefs.get(f"{channel}_enabled", True):
            return False

        if priority == "MARKETING" and not prefs.get("marketing_enabled", True):
            return False

        return True

    async def batch_get(self, user_ids: List[int]) -> Dict[int, dict]:
        """Efficient batch preference lookup for fan-out. One pipeline call."""
        pipe = self.redis.pipeline()
        for uid in user_ids:
            pipe.hgetall(f"user_prefs:{uid}")
        results = await pipe.execute()

        prefs_map = {}
        missing   = []
        for uid, prefs in zip(user_ids, results):
            if prefs:
                prefs_map[uid] = prefs
            else:
                missing.append(uid)

        # Batch-fetch missing from DB
        if missing:
            db_prefs = await self.db.batch_get_prefs(missing)
            for uid, p in db_prefs.items():
                prefs_map[uid] = p
                # Cache in Redis
                self.redis.hmset(f"user_prefs:{uid}", p)
                self.redis.expire(f"user_prefs:{uid}", self.CACHE_TTL)

        return prefs_map

    async def _get_cached(self, user_id: int) -> Optional[dict]:
        cached = await self.redis.hgetall(f"user_prefs:{user_id}")
        if cached:
            return cached
        prefs = await self.db.get_prefs(user_id)
        if prefs:
            await self.redis.hmset(f"user_prefs:{user_id}", prefs)
            await self.redis.expire(f"user_prefs:{user_id}", self.CACHE_TTL)
        return prefs
⚖️ Design Tradeoffs
Notification systems hide subtle tradeoffs between delivery guarantees, user experience, and cost. Over-sending annoys users; under-sending loses conversions.
📬 Delivery Guarantee: At-Least-Once vs Exactly-Once vs Best-Effort
✅ At-Least-Once + Idempotency

Use Kafka (durable, replayable) + Redis SET NX dedup check before each provider call. Worker failure means Kafka redelivers → dedup prevents double send. Industry standard. Simple to implement correctly.

⚠️ Exactly-Once (distributed txn)

Two-phase commit between DB and Kafka. Massive complexity. Kafka's exactly-once semantics help but don't cover provider failures. A provider can accept the call and then fail before confirming — you never know if the notification was actually sent. In practice, at-least-once + idempotency is "exactly-once enough."

❌ Best-Effort (fire and forget)

Write to DB, call provider synchronously, don't retry. Simple. But: 5% of APNs calls fail transiently. At 10M/day, that's 500K lost notifications. For OTPs and payment confirmations, this is completely unacceptable. Only viable for truly unimportant notifications (marketing impressions).

📤 Fan-out: Pre-compute vs Lazy
✅ Fan-out at send time (Kafka)

Campaign triggers fan-out service that creates individual per-user Kafka messages. Worker pool scales independently. Natural backpressure via Kafka consumer lag. Resumable on failure (cursor tracking). Standard pattern for all major notification platforms.

⚠️ Pre-computed targeting lists

Before campaign launch, pre-compute the full list of recipients (filtered for opt-outs) and store as a list. Workers consume from the list. Pro: faster start. Con: list is stale (user may opt out between list generation and delivery), takes more storage upfront.

❌ Synchronous fan-out in API

The caller's API request blocks while you create 100M Kafka messages. At 100K messages/sec, that's 1000 seconds (16 minutes) of blocking. Request times out. Caller retries. Duplicate campaign. Never synchronously fan-out in the request path.

🚫 Provider Failure: Single Provider vs Multi-Provider Fallback
✅ Primary + Fallback provider

Primary: SendGrid for email. Fallback: SES. If SendGrid returns 5xx for 30s, circuit breaker opens, all email routes to SES automatically. Recovery: circuit half-opens after 60s, test with a small % of traffic. Full recovery when error rate normalises.

⚠️ Single provider

Simpler. SendGrid has 99.9% uptime → ~8.7 hours downtime/year. For email, this is acceptable. For SMS (OTP), a 1-hour outage during a popular login window is catastrophic. Single provider is only acceptable for non-critical channels (marketing email).

💡 Per-channel fallback logic

Push: APNs has no real fallback — if APNs is down, try FCM only for dual-platform users, or fall back to email/SMS for the message. SMS: Twilio primary → AWS SNS fallback. Email: SendGrid → SES. Design the fallback per channel's criticality and what alternatives are available.

😴 Quiet Hours: Delay vs Drop vs Override
✅ Delay to window end (scheduled queue)

Notifications sent during quiet hours are scheduled to deliver at 8am user local time. Stored in Redis ZSET: score = delivery_timestamp. Scheduler picks up due notifications. User gets notification when they wake up — better UX than receiving it at 2am.

⚠️ Drop silently

If notification arrives during quiet hours, discard. Simple, no delay queue needed. But: user misses potentially important transactional notifications (flight delay, package delivery). Only acceptable for pure marketing where the notification is time-sensitive (flash sale at 2am — irrelevant by 8am anyway).

💡 TTL + quiet hours interaction

Edge case: notification has TTL=1h, but quiet hours mean it won't be delivered for 6 hours. Should we drop it? Check TTL against quiet hours end time: if TTL expires before quiet hours end, either override quiet hours (if TRANSACTIONAL) or drop (if MARKETING). Always check TTL at actual delivery time, not just enqueue time.
