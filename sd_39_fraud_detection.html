🚨
Fraud Detection System
Design a real-time fraud detection system for a payment processor like Stripe or PayPal — evaluating every transaction in under 100ms before it clears, combining rule-based velocity checks with ML scoring models, maintaining a real-time feature store for user/merchant behavior, and balancing false positive rates (blocking legitimate transactions) against false negatives (missing fraud).
🔴 MUST DO 🟣 Hard Real-time ML Feature Store Velocity Checks Rules Engine False Positive Model Serving
🎯 Requirements
Fraud detection is a real-time ML problem with an asymmetric cost function. Missing fraud (false negative) costs money directly. Blocking a legitimate transaction (false positive) loses a customer permanently. Typical target: 0.1% fraud rate, <0.5% false positive rate. Every millisecond matters — the model must run synchronously before the transaction clears.

FUNCTIONAL
Score every transaction before it clears: APPROVE / DECLINE / REVIEW
Velocity checks: transactions per hour, per day, per merchant
Rule-based filters: country mismatch, unusual amount, new device
ML model scoring: gradient-boosted tree or neural network on 200+ features
Feedback loop: confirmed fraud updates model and rules
Case management: flagged transactions queued for analyst review
NON-FUNCTIONAL
Latency: <100ms p99 per transaction decision
Throughput: 100K transactions/sec peak (Black Friday)
False negative rate: <0.1% (miss <1 in 1000 fraudulent txns)
False positive rate: <0.5% (block <5 in 1000 legitimate txns)
Availability: 99.999% — fraud check must never block payment flow
💬 Phase 1 — The Three-Layer Decision System
PHASE 1 · DECISION LAYERS
IR
INTERVIEWER
A transaction arrives. Walk me through every step from receipt to the APPROVE/DECLINE decision — and explain why you have multiple layers instead of just running the ML model.
ME
YOU
Three layers, each progressively more expensive but more accurate:

Layer 1 — Hard Rules (1ms): Instant DECLINE with no ML needed. Examples: transaction amount exceeds user's account limit, card reported stolen (blocklist lookup in Redis O(1)), country of transaction ≠ card's registered country by more than 2 hops, IP is in known fraud VPN/proxy list. These are high-precision signals — almost never wrong. If triggered: DECLINE immediately, no further processing.

Layer 2 — Velocity Checks (5ms): Sliding window counters in Redis. Retrieve and compare:
  • Transactions from this card in the last 1 hour (normal: 1-5, suspicious: >20)
  • Transactions from this IP in the last 10 minutes (card testing attack: 50+ attempts)
  • Unique merchants this card visited in the last hour (normal: 1-2, suspicious: 10+)
  • Amount distribution: is this amount 10× larger than the user's average transaction?
Each check produces a "velocity feature" consumed by Layer 3.

Layer 3 — ML Model Scoring (20-50ms): Gradient-boosted tree (XGBoost) or small neural network. Input: 200+ features combining transaction attributes (amount, merchant category, time of day) + user profile features (average spend, typical locations, device history) + real-time velocity features from Layer 2. Output: fraud probability 0.0-1.0. Threshold: >0.8 → DECLINE, 0.4-0.8 → REVIEW (send to analyst queue), <0.4 → APPROVE.

Why not just ML? ML model startup cost is 20-50ms. Hard rules catch 30% of fraud in <1ms. This reduces model invocations by 30% and prevents obvious fraud without ML's minimum latency overhead.
💡 The key insight interviewers want to hear: You must always approve by default if the fraud system is unavailable (circuit breaker open). A payment processor that blocks ALL transactions when the fraud system crashes is worse than one that temporarily allows transactions through. Availability of the payment flow is the top priority; fraud detection is a best-effort safeguard.
🗄 Phase 2 — Real-Time Feature Store
PHASE 2 · FEATURE STORE
IR
INTERVIEWER
Your ML model needs 200 features per transaction. Where do you get them from fast enough to hit 100ms?
ME
YOU
Features come from two sources with very different latency profiles:

Batch features (precomputed, refreshed hourly/daily): User's 30-day average transaction amount, typical merchant categories, historical fraud rate for this merchant, user's home country, device fingerprint trust score. These are computed offline by Spark jobs and stored in Redis (hot) with a TTL. On each transaction: HGETALL user_features:{user_id} → returns 100+ features in <2ms.

Real-time features (computed on the fly): Current velocity (transactions in last 1h), time since last transaction, whether this merchant is new for this user, location distance from last transaction. Computed via Redis sorted sets and counters. Each feature: 1 Redis command, <1ms. Total for 50 real-time features: 50 Redis pipeline commands = 5ms.

Feature store architecture: All features (batch + real-time) are materialised into a single flat feature vector before calling the ML model. This "feature lookup" phase takes 10ms total. The ML model then runs inference on the preloaded vector in 20ms. Combined: 30ms for feature+inference — well within 100ms budget.

Feature freshness: Batch features updated every 4 hours via Spark. Real-time features updated on every transaction (when a transaction completes, update the user's velocity counters). This means the model always has up-to-date velocity data even for features that require historical aggregation.
🔄 Phase 3 — Feedback Loop & Model Retraining
PHASE 3 · FEEDBACK LOOP
IR
INTERVIEWER
Fraudsters adapt to your model. How do you keep the model up to date?
ME
YOU
The label generation problem: you only know a transaction was fraudulent after the chargeback dispute is resolved (days to weeks). This creates a label delay — the model is always training on slightly stale ground truth.

Label pipeline: Transaction logs → chargeback system (when user disputes a charge) → fraud label written to the training dataset with the original transaction's features. Lag: 7-14 days average for chargebacks. Immediate labels available from: user reporting fraud, card network fraud signals (Visa/MC share confirmed fraud events), and internal analyst decisions on REVIEW queue.

Retraining cadence: Daily retraining on the last 90 days of labeled data (rolling window — older data is less relevant as fraud patterns evolve). Retrain is a Spark/offline job: runs on a GPU cluster, produces a new model artifact, deployed via A/B test (10% traffic to new model, 90% to existing) before full rollout.

Drift detection: Monitor model score distribution daily. If the distribution shifts significantly (KS test), trigger an early retrain. Also monitor: fraud rate week-over-week, false positive rate (via user complaints about declined transactions). Sudden spike = new fraud vector → analysts investigate → write new hard rules immediately (faster than model retraining).

// Fraud Detection — Java: Decision Engine
// ══════════════════════════════════════════════════════

public record Transaction(
    String txnId, String userId, String cardHash,
    double amount, String merchantCategory,
    String ipCountry, String cardCountry,
    String deviceFingerprint, long timestampMs
) {}

public enum FraudDecision { APPROVE, DECLINE, REVIEW }

@Service
public class FraudDecisionEngine {

    @Autowired private RulesEngine        rules;
    @Autowired private FeatureStoreClient  featureStore;
    @Autowired private MLModelClient       mlModel;
    @Autowired private KafkaTemplate<String,Object> kafka;

    public FraudDecision evaluate(Transaction txn) {
        long start = System.currentTimeMillis();

        // Layer 1: Hard rules — instant decision, no ML needed
        RuleResult ruleResult = rules.evaluate(txn);
        if (ruleResult.isDefiniteFraud()) {
            logAsync(txn, FraudDecision.DECLINE, 1.0, "hard_rule:" + ruleResult.getRuleName());
            return FraudDecision.DECLINE;
        }

        // Layer 2: Fetch features (batch + real-time velocity in parallel)
        FeatureVector features;
        try {
            features = featureStore.fetchWithTimeout(txn.userId(), txn.cardHash(), 15); // 15ms timeout
        } catch (TimeoutException e) {
            // Feature store unavailable — fail open (approve) to protect payment flow
            logAsync(txn, FraudDecision.APPROVE, -1.0, "feature_store_timeout");
            return FraudDecision.APPROVE;
        }

        // Layer 3: ML model scoring
        double fraudScore;
        try {
            fraudScore = mlModel.score(features, 50); // 50ms timeout
        } catch (TimeoutException e) {
            // ML model unavailable — fall back to rules-only decision
            fraudScore = ruleResult.getRuleScore(); // conservative estimate
        }

        // Layer 4: Decision threshold
        FraudDecision decision;
        if      (fraudScore >= 0.80) decision = FraudDecision.DECLINE;
        else if (fraudScore >= 0.40) decision = FraudDecision.REVIEW;
        else                          decision = FraudDecision.APPROVE;

        // Async: update velocity counters + log (NOT in critical path)
        final double fs = fraudScore;
        CompletableFuture.runAsync(() -> {
            featureStore.incrementVelocity(txn.userId(), txn.cardHash());
            logAsync(txn, decision, fs, "ml_model");
            if (decision == FraudDecision.REVIEW) kafka.send("fraud-review", txn);
        });

        Metrics.timer("fraud.decision.latency").record(System.currentTimeMillis() - start, MILLISECONDS);
        return decision;
    }
}

// ── Velocity Checks via Redis Sliding Window ──────────
@Service
public class VelocityChecker {

    @Autowired private RedisTemplate<String,String> redis;

    /** Record a transaction and return velocity features. */
    public VelocityFeatures recordAndFetch(String userId, String cardHash, long nowMs) {
        String cardKey = "vel:card:"  + cardHash;
        String userKey = "vel:user:"  + userId;
        long   oneHourAgo = nowMs - 3_600_000L;

        // Add current timestamp to sorted set, then count in window — atomic via Lua
        String luaScript = """
            redis.call('ZADD', KEYS[1], ARGV[1], ARGV[1])
            redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', ARGV[2])
            redis.call('EXPIRE', KEYS[1], 7200)
            return redis.call('ZCARD', KEYS[1])
            """;

        Long cardTxns1h = (Long) redis.execute(
            new DefaultRedisScript<>(luaScript, Long.class),
            List.of(cardKey), String.valueOf(nowMs), String.valueOf(oneHourAgo));

        return VelocityFeatures.builder()
            .cardTxns1h(cardTxns1h != null ? cardTxns1h : 0)
            .build();
    }
}

// ── Rules Engine ──────────────────────────────────────
@Service
public class RulesEngine {

    @Autowired private RedisTemplate<String,String> redis;

    public RuleResult evaluate(Transaction txn) {
        // Rule 1: Card blocklist (O(1) Redis lookup)
        if (redis.hasKey("blocklist:card:" + txn.cardHash()))
            return RuleResult.decline("card_blocked", 1.0);

        // Rule 2: Country mismatch
        if (!txn.ipCountry().equals(txn.cardCountry()))
            return RuleResult.suspicious("country_mismatch", 0.6);

        // Rule 3: Amount exceeds hard limit
        if (txn.amount() > 10_000.0)
            return RuleResult.suspicious("high_amount", 0.5);

        return RuleResult.ok();
    }
}

# Fraud Detection — Python: Decision Engine + Velocity
# ══════════════════════════════════════════════════════

import time, redis, numpy as np
from dataclasses import dataclass
from enum import Enum
from typing import Optional
import onnxruntime as ort

r = redis.Redis(decode_responses=True)


class Decision(Enum):
    APPROVE = "approve"
    DECLINE = "decline"
    REVIEW  = "review"

@dataclass
class Transaction:
    txn_id:             str
    user_id:            str
    card_hash:          str
    amount:             float
    merchant_category:  str
    ip_country:         str
    card_country:       str
    device_fingerprint: str
    timestamp_ms:       int


# ── Velocity check via Redis sorted set ───────────────
VELOCITY_LUA = """
local key    = KEYS[1]
local now    = tonumber(ARGV[1])
local cutoff = tonumber(ARGV[2])
redis.call('ZADD', key, now, now)
redis.call('ZREMRANGEBYSCORE', key, '-inf', cutoff)
redis.call('EXPIRE', key, 7200)
return redis.call('ZCARD', key)
"""
_vel_script = r.register_script(VELOCITY_LUA)


def get_velocity(card_hash: str, window_ms: int = 3_600_000) -> int:
    now     = int(time.time() * 1000)
    cutoff  = now - window_ms
    key     = f"vel:card:{card_hash}"
    return int(_vel_script(keys=[key], args=[now, cutoff]) or 0)


# ── Feature fetch from Redis ──────────────────────────
def fetch_features(txn: Transaction) -> Optional[np.ndarray]:
    pipe = r.pipeline()
    pipe.hgetall(f"user_features:{txn.user_id}")
    pipe.zscore(f"card_last_txn:{txn.card_hash}", "ts")
    results = pipe.execute()

    user_f  = results[0] or {}
    last_ts = float(results[1] or 0)

    velocity_1h = get_velocity(txn.card_hash, 3_600_000)
    velocity_24h = get_velocity(txn.card_hash, 86_400_000)
    time_since_last = (txn.timestamp_ms - last_ts) / 1000  # seconds

    avg_amount = float(user_f.get("avg_amount_30d", txn.amount))
    amount_zscore = (txn.amount - avg_amount) / max(avg_amount * 0.5, 1)

    is_international = 1 if txn.ip_country != txn.card_country else 0
    is_blocked = 1 if r.exists(f"blocklist:card:{txn.card_hash}") else 0

    return np.array([
        txn.amount,
        amount_zscore,
        velocity_1h,
        velocity_24h,
        time_since_last,
        is_international,
        is_blocked,
        float(user_f.get("historical_fraud_count", 0)),
        float(user_f.get("user_age_days", 0)),
        float(user_f.get("chargeback_rate", 0)),
    ], dtype=np.float32)


# ── ONNX Model for fast inference ────────────────────
_sess: Optional[ort.InferenceSession] = None

def load_model(model_path: str):
    global _sess
    _sess = ort.InferenceSession(model_path, providers=["CPUExecutionProvider"])

def score_transaction(features: np.ndarray) -> float:
    if _sess is None: return 0.5  # model not loaded — conservative
    result = _sess.run(None, {"features": features.reshape(1, -1)})
    return float(result[0][0][1])  # P(fraud)


# ── Main decision function ────────────────────────────
def evaluate(txn: Transaction) -> Decision:
    start = time.time()

    # Layer 1: Hard rules
    if r.exists(f"blocklist:card:{txn.card_hash}"):
        return Decision.DECLINE

    # Layer 2+3: Feature fetch + ML scoring
    try:
        features = fetch_features(txn)
        if features is None: return Decision.APPROVE  # fail open

        # Early hard-rule exit on extreme velocity
        if features[2] > 50:  # velocity_1h > 50 = definite card testing
            return Decision.DECLINE

        fraud_score = score_transaction(features)

    except Exception:
        return Decision.APPROVE  # fail open on any error

    # Layer 4: Threshold decision
    latency_ms = (time.time() - start) * 1000
    print(f"[FRAUD] txn={txn.txn_id} score={fraud_score:.3f} latency={latency_ms:.1f}ms")

    if   fraud_score >= 0.80: return Decision.DECLINE
    elif fraud_score >= 0.40: return Decision.REVIEW
    else:                     return Decision.APPROVE
⚖️ Tradeoffs
🤖 Model Choice: Rules Only vs ML vs Hybrid
Rules Only

Hand-coded if/then rules. Explainable: "declined because of country mismatch + high amount." Instant to update: add a rule in hours. Problem: can't detect novel fraud patterns not yet encoded as rules. Fraudsters adapt quickly — rules become stale. False positive rate is high without fine-tuning.

✅ Hybrid (chosen)

Rules for obvious fraud (blocklists, hard limits) + ML for nuanced pattern detection. Rules execute first — ML only runs if rules don't trigger. Best of both: fast obvious-case handling, adaptive pattern detection. Used by Stripe, PayPal, Visa. Rules catch ~30%, ML catches the remaining 70%.

ML Only

Gradient-boosted tree or neural network on all 200+ features. Highest accuracy. Handles complex non-linear patterns rules can't express. Problem: black box — hard to explain why a transaction was declined (required for customer service and regulatory compliance). Slower to update (requires retrain cycle).

⚖️ Threshold Tuning: Precision vs Recall
High threshold (0.9+)

Only decline very high-confidence fraud. Fewer false positives — most legitimate transactions go through. High false negative rate — miss more fraud. Cost: fraud losses increase. Good for premium merchants where false positives destroy trust (luxury goods, enterprise).

✅ Adaptive threshold + REVIEW band

0.8+ = DECLINE, 0.4-0.8 = REVIEW (human analyst), <0.4 = APPROVE. The REVIEW band handles uncertain cases without blocking or allowing blindly. Analysts decisions feed back into model training. Extra friction (3D Secure challenge) can be added for REVIEW band instead of full DECLINE. Used by Stripe Radar.

Low threshold (0.3)

Decline at lower confidence. Catches more fraud. But false positive rate spikes — legitimate customers are blocked. At scale, even 1% false positives means 1000 declined good transactions per 100K. Each declined legitimate transaction risks losing that customer permanently. Never use without a REVIEW band.
