💬 Phase 1 — LSM Tree Write Path
PHASE 1 · WRITE PATH
IR
INTERVIEWER
How does a write (PUT) work? Take me from the API call to the data being durably stored.
ME
YOU
The LSM (Log-Structured Merge) write path has three stages:

1. Write-Ahead Log (WAL): Before anything else, the write is appended to a WAL file on disk. The WAL is an append-only log — sequential disk writes, extremely fast (>500 MB/s). Only a single fsync() call is needed. If the server crashes after the WAL write but before the next step, the WAL lets us replay and recover the write. Durability guaranteed.

2. Memtable (in-memory write): After WAL, the key-value pair is written to the memtable — an in-memory sorted data structure (typically a red-black tree or skip list). The memtable is ordered by key, enabling O(log N) inserts and range queries. Reads check the memtable first — it has the freshest data. Memory: the memtable is typically 64MB–256MB per shard.

3. SSTable flush (when memtable full): When the memtable reaches capacity, it's flushed to disk as an SSTable (Sorted String Table) — an immutable, sorted file. The memtable is then cleared. A new empty memtable + WAL segment is created. The SSTable is never modified — only compacted (merged with other SSTables). This makes writes fast because we're always appending, never doing random I/O.
💡 The LSM tree insight: Writes are fast because they're sequential (WAL) + in-memory (memtable). We trade write performance (fast) for read performance (must check multiple SSTables). Compaction is the background process that reclaims space and improves read performance.
🔍 Phase 2 — Read Path + Bloom Filters
PHASE 2 · READ PATH
IR
INTERVIEWER
How does a GET work? There are multiple SSTables on disk — do you check all of them?
ME
YOU
The read path checks data sources in order from newest to oldest:

1. Memtable: Check in-memory sorted tree first. O(log N). If found: return immediately. (freshest data)

2. Immutable memtables: Recently flushed but not yet written to SSTable. O(log N).

3. SSTables (newest to oldest): Each SSTable must be checked in reverse chronological order because newer SSTables may have overwritten values in older ones.

The problem: With many SSTables, a "not found" key would require checking every single one. At 1,000 SSTables, that's 1,000 disk reads for a missing key. Catastrophic.

Bloom Filters to the rescue: Each SSTable has a Bloom filter — a compact probabilistic data structure (bit array) that answers "is this key definitely NOT in this SSTable?" with zero false negatives. If the Bloom filter says NO: skip this SSTable entirely. If YES: the key might be here (false positive rate ~1%), so actually check.

With Bloom filters: a missing key check costs 1 Bloom filter query per SSTable (in memory, <1µs each) instead of 1 disk seek per SSTable (4-10ms each). A 1,000-SSTable lookup goes from 10 seconds to 1 millisecond.
🗜 Phase 3 — Compaction
PHASE 3 · COMPACTION
IR
INTERVIEWER
Over time you accumulate thousands of SSTables. What is compaction and why is it necessary?
ME
YOU
Without compaction, three problems compound over time:
  • Read amplification: more SSTables to check per read
  • Space amplification: deleted keys and overwritten values still occupy space in older SSTables
  • Too many file handles
Compaction process: A background job merges multiple SSTables into one. During merge, it:
  • Keeps only the newest value for each key (older overwrites discarded)
  • Removes tombstones (DELETE markers) that are older than all other data
  • Produces one sorted, de-duplicated SSTable from N input SSTables
Two main compaction strategies:
Size-tiered: When N SSTables of similar size accumulate, merge them into one larger SSTable. Simple, fast writes. But: temporary space amplification (you need 2× space during compaction) and reads can still hit many levels.

Leveled: SSTables organised into levels (L0, L1, L2...). Each level has a size budget (e.g., 10MB, 100MB, 1GB, 10GB). When L0 is full, compact into L1. Non-overlapping key ranges in L1+. Better read performance (1 SSTable per level per key). Higher write amplification (data written and rewritten through levels). RocksDB, LevelDB use leveled compaction.
🌲 LSM Tree — Complete Write + Read + Compaction
🔵 B-Tree vs LSM Tree
B-trees (used by PostgreSQL, MySQL InnoDB) and LSM trees (used by RocksDB, Cassandra, LevelDB) make opposite tradeoffs. B-trees optimise reads. LSM trees optimise writes.

PropertyB-Tree (PostgreSQL)LSM Tree (RocksDB)
WriteRandom I/O (page update in-place) → slowSequential I/O (append WAL + memtable) → fast
ReadSingle tree traversal O(log N) → fastCheck memtable + multiple SSTables → slower
Space amplification~1× (in-place update)1.1–10× (multiple versions during compaction)
Write amplification~3–5× (WAL + B-tree page)~10–30× (data moves through levels)
Read amplification~3–5× (root → leaf)~10× (multiple SSTables checked)
Range queriesEfficient (tree traversal)Efficient (sorted SSTables, merge)
Best forRead-heavy, OLTP, range queriesWrite-heavy, append workloads, time-series
Used byPostgreSQL, MySQL, SQLiteCassandra, RocksDB, Kafka (segment files), HBase

Key insight for interviews: Cassandra and DynamoDB use LSM trees because they're optimised for write-heavy workloads. MySQL/PostgreSQL use B-trees because they need fast reads. When asked "which storage engine would you use?" — ask about the read/write ratio first.

// Simplified LSM-style KV Store — Java
// ══════════════════════════════════════════════════════

public class SimpleLsmStore {

    // Memtable: sorted in-memory structure
    private final ConcurrentSkipListMap<String, String> memtable = new ConcurrentSkipListMap<>();
    private static final int MEMTABLE_THRESHOLD = 1000; // flush after 1K entries
    private static final String TOMBSTONE = "__DELETED__";

    // SSTables (ordered newest to oldest)
    private final List<SSTable> sstables = new CopyOnWriteArrayList<>();

    // WAL for durability
    private final WalWriter wal;

    /**
     * PUT: Write to WAL first (durability), then memtable (speed).
     * O(log N) where N = memtable size.
     */
    public void put(String key, String value) {
        wal.append("PUT", key, value);    // persist to disk first
        memtable.put(key, value);          // then update in-memory
        if (memtable.size() >= MEMTABLE_THRESHOLD) {
            flushMemtable();
        }
    }

    /**
     * DELETE: Write tombstone marker.
     * Actual deletion happens during compaction.
     */
    public void delete(String key) {
        wal.append("DELETE", key, null);
        memtable.put(key, TOMBSTONE);      // tombstone in memtable
    }

    /**
     * GET: Check memtable first, then SSTables newest to oldest.
     * Bloom filter skips SSTables that definitely don't have the key.
     */
    public Optional<String> get(String key) {
        // 1. Check memtable
        String memVal = memtable.get(key);
        if (memVal != null) {
            return TOMBSTONE.equals(memVal)
                ? Optional.empty() // deleted
                : Optional.of(memVal);
        }

        // 2. Check SSTables newest first
        for (SSTable sst : sstables) {
            // Bloom filter: skip if key definitely not in this SSTable
            if (!sst.bloomFilter.mightContain(key)) continue;

            Optional<String> val = sst.get(key);
            if (val.isPresent()) {
                return TOMBSTONE.equals(val.get()) ? Optional.empty() : val;
            }
        }

        return Optional.empty();
    }

    /**
     * Flush memtable to a new SSTable.
     * Called when memtable hits threshold.
     */
    private synchronized void flushMemtable() {
        if (memtable.isEmpty()) return;
        // Snapshot the current memtable (concurrent writes go to new one)
        Map<String, String> snapshot = new TreeMap<>(memtable);
        memtable.clear();

        // Build SSTable from sorted snapshot
        SSTable sst = SSTable.build(snapshot);
        sstables.add(0, sst);  // newest first
        wal.rotate();           // new WAL segment, old segment no longer needed

        // Trigger compaction if too many SSTables
        if (sstables.size() > 10) {
            compact();
        }
    }

    /**
     * Size-tiered compaction: merge all SSTables into one.
     * In production: leveled compaction with L0-L2+ tiers.
     */
    private synchronized void compact() {
        TreeMap<String, String> merged = new TreeMap<>();

        // Iterate oldest to newest — newer values overwrite older ones
        for (int i = sstables.size() - 1; i >= 0; i--) {
            sstables.get(i).entries().forEach((k, v) -> {
                if (TOMBSTONE.equals(v)) {
                    merged.remove(k);    // honour tombstones
                } else {
                    merged.put(k, v);    // newer overwrites older
                }
            });
        }

        // Replace all SSTables with one compacted SSTable
        sstables.clear();
        if (!merged.isEmpty()) {
            sstables.add(SSTable.build(merged));
        }
    }
}

// Bloom Filter — simple bit array implementation
public class BloomFilter {
    private final long[] bits;
    private final int   numHashFunctions;

    public BloomFilter(int expectedInsertions, double falsePositiveRate) {
        int m = optimalBitSize(expectedInsertions, falsePositiveRate);
        numHashFunctions = optimalHashCount(m, expectedInsertions);
        bits = new long[m / 64 + 1];
    }

    public void put(String key) {
        long[] hashes = getHashes(key);
        for (long hash : hashes) {
            int bit = (int) (hash % (bits.length * 64));
            bits[bit / 64] |= (1L << (bit % 64));
        }
    }

    public boolean mightContain(String key) {
        for (long hash : getHashes(key)) {
            int bit = (int) (hash % (bits.length * 64));
            if ((bits[bit / 64] & (1L << (bit % 64))) == 0) {
                return false; // definitely not present
            }
        }
        return true; // might be present (could be false positive)
    }
}

# Simplified LSM KV Store — Python

import sortedcontainers, hashlib, math
from sortedcontainers import SortedDict
from typing import Optional

TOMBSTONE = b\'__DELETED__\'

class BloomFilter:
    """Probabilistic set membership. 0 false negatives, ~1% false positives."""
    def __init__(self, capacity=10000, error_rate=0.01):
        m = int(-capacity * math.log(error_rate) / (math.log(2)**2))
        self.bits = bytearray(m // 8 + 1)
        self.size = m
        self.k    = max(1, int(m / capacity * math.log(2)))

    def _hashes(self, key: str):
        h1 = int(hashlib.md5(key.encode()).hexdigest(), 16)
        h2 = int(hashlib.sha1(key.encode()).hexdigest(), 16)
        return [(h1 + i * h2) % self.size for i in range(self.k)]

    def add(self, key: str):
        for pos in self._hashes(key):
            self.bits[pos >> 3] |= (1 << (pos & 7))

    def might_contain(self, key: str) -> bool:
        return all(self.bits[pos >> 3] & (1 << (pos & 7))
                   for pos in self._hashes(key))


class SSTable:
    """Immutable sorted file. In production: written to disk."""
    def __init__(self, sorted_data: dict):
        self._data        = dict(sorted_data)  # sorted by key
        self.bloom_filter = BloomFilter(len(sorted_data))
        for k in sorted_data: self.bloom_filter.add(k)

    def get(self, key: str) -> Optional[bytes]:
        if not self.bloom_filter.might_contain(key): return None
        return self._data.get(key)

    def items(self): return self._data.items()


class LSMStore:
    MEMTABLE_MAX = 1000

    def __init__(self, wal_path: str = "/tmp/wal.log"):
        self.memtable = SortedDict()          # sorted in-memory
        self.sstables = []                    # newest first
        self.wal      = open(wal_path, \'ab\')  # append-only WAL

    def put(self, key: str, value: bytes):
        # WAL first (durability), then memtable (speed)
        self.wal.write(f\'PUT {key}\\n\'.encode() + value + b\'\\n\')
        self.wal.flush()
        self.memtable[key] = value
        if len(self.memtable) >= self.MEMTABLE_MAX:
            self._flush_memtable()

    def delete(self, key: str):
        self.wal.write(f\'DELETE {key}\\n\'.encode())
        self.wal.flush()
        self.memtable[key] = TOMBSTONE

    def get(self, key: str) -> Optional[bytes]:
        # 1. Memtable (newest data)
        if key in self.memtable:
            val = self.memtable[key]
            return None if val == TOMBSTONE else val

        # 2. SSTables newest first (bloom filter → skip if not present)
        for sst in self.sstables:
            val = sst.get(key)
            if val is not None:
                return None if val == TOMBSTONE else val
        return None

    def _flush_memtable(self):
        if not self.memtable: return
        sst = SSTable(dict(self.memtable))
        self.sstables.insert(0, sst)  # newest first
        self.memtable.clear()
        if len(self.sstables) > 10:
            self._compact()

    def _compact(self):
        """Merge all SSTables. Newer values overwrite older."""
        merged = {}
        for sst in reversed(self.sstables):  # oldest first
            for k, v in sst.items():
                if v == TOMBSTONE: merged.pop(k, None)
                else: merged[k] = v
        self.sstables = [SSTable(merged)] if merged else []
⚖️ Tradeoffs
🌲 LSM Tree: Write vs Read Amplification
✅ Write fast (sequential I/O)

All writes are sequential (WAL + memtable). No random disk seeks. Write throughput: >500 MB/s on HDD. Critical for write-heavy workloads (time-series, logging, Cassandra). The fundamental insight: make writes sequential at the cost of read complexity.

⚠️ Read amplification

A read may touch memtable + multiple SSTables at multiple levels. With leveled compaction (RocksDB): worst case 1 read per level. With 5 levels: 5 disk reads for a missing key (mitigated by Bloom filters). In-memory databases (Redis) solve this by keeping everything in RAM.

🌸 Bloom Filter: Size vs False Positive Rate
1% false positive rate

Memory: 9.6 bits per element. For 10M keys: ~12 MB per SSTable. In practice each SSTable has <1M keys → ~1.2 MB Bloom filter. Very manageable. At 1% FPR: 99% of "not found" reads skip the SSTable entirely. The other 1% do an unnecessary disk read.

0.1% false positive rate

Memory: 14.4 bits per element. 50% more memory for 10× better performance on "not found" queries. Worth it if your workload has many reads for non-existent keys (negative lookups). RocksDB lets you tune FPR per column family.

⚡ In-Memory vs On-Disk Storage
✅ In-memory (Redis)

All data in RAM. Sub-microsecond reads. No storage engine complexity — simple hash map. Persistence via RDB snapshots + AOF (Append-Only File, similar to WAL). Limit: dataset must fit in RAM. Cost: RAM is 10-50× more expensive than SSD per GB. Best for cache layers, session stores, rate limiting.

On-disk + cache (RocksDB)

Dataset can exceed RAM. Hot data cached in block cache (LRU). Cold data on SSD. LSM tree enables fast writes (WAL + memtable). Compaction runs in background. Best for: databases, blob storage, large datasets. DynamoDB, Cassandra, MongoDB all use this model.
