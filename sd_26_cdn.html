CDN — Content Delivery Network
Design a CDN like Cloudflare or Akamai — global PoP placement using anycast routing, pull vs push caching strategy, origin shield, TLS termination at edge, cache invalidation propagation, and how a CDN serves 100 Tbps from thousands of edge nodes while the origin sees almost zero traffic.
🔴 MUST DO 🟣 Hard Anycast PoP Placement Origin Shield Cache Invalidation TLS at Edge Pull vs Push
🎯 Requirements
A CDN solves two problems simultaneously: latency (serve content from the nearest edge, not the origin across the world) and scale (origin can't handle 100M concurrent users — edge nodes absorb the load).

FUNCTIONAL
Cache and serve static assets (images, JS, CSS, video segments) from edge PoPs globally
Pull caching: on cache miss, fetch from origin and cache at edge
Push caching: pre-populate edges with known hot content
Cache invalidation: propagate purge to all edges within 5 seconds
TLS termination at edge (no TLS cost to origin for cached content)
Origin shield: funnel all cache misses through a single shield region
NON-FUNCTIONAL
TTFB (Time To First Byte): <20ms for cached content globally
Cache hit rate: >95% for static assets
100 Tbps aggregate egress capacity across all PoPs
Invalidation propagation: <5 seconds globally
99.99% availability (4 nines) — users can always reach an edge
💬 Phase 1 — Routing: How does a User Hit the Nearest PoP?
PHASE 1 · ANYCAST ROUTING
IR
INTERVIEWER
The CDN has 200 PoPs worldwide. How does a user in Tokyo automatically connect to the Tokyo PoP and not the New York one?
ME
YOU
Two mechanisms — CDNs typically use both:

1. DNS-based routing (GeoDNS): The CDN's authoritative DNS server resolves the CDN hostname (e.g., cdn.example.com) to different IP addresses depending on the resolver's geographic location. Tokyo resolver → Tokyo PoP IP. Simple, works everywhere. Limitation: DNS TTL means routing decisions are sticky for 30-60 seconds — a PoP going down isn't instantly avoided.

2. Anycast routing (preferred for large CDNs): All PoPs announce the same IP prefix (e.g., 104.20.0.0/20) into BGP. The internet's routing protocol naturally delivers packets to the topologically nearest router announcing that prefix. A user in Tokyo reaches 104.20.0.0 at the Tokyo PoP; a user in London reaches the same IP at the London PoP. No DNS tricks needed. Failover is automatic — if Tokyo PoP fails, BGP reconverges within seconds and Tokyo users route to the next-nearest PoP.

Cloudflare uses anycast for all traffic. Akamai uses a mix of both. Anycast has one limitation: it routes by BGP hop count, not latency — occasionally a geographically "near" PoP is further in BGP hops. CDNs compensate with their own backbone to bypass sub-optimal internet routing.
💡 Anycast is the key differentiator. It provides instant failover with no DNS TTL wait, handles DDoS mitigation automatically (attacks are spread across all PoPs rather than overwhelming one), and enables global traffic routing without configuration per user location.
🏛 Phase 2 — Origin Shield
PHASE 2 · ORIGIN SHIELD
IR
INTERVIEWER
You have 200 PoPs. On a cache miss, all 200 could simultaneously hit your origin. How do you protect the origin?
ME
YOU
Without an origin shield: if a new popular asset is requested simultaneously from 200 PoPs, 200 concurrent requests hit the origin. For a large content library (millions of assets), this becomes a serious origin load problem.

Origin Shield: Designate one intermediate caching layer (the "shield") between edge PoPs and the origin. Typically one shield region per continent (e.g., us-east-1 serves all North American PoPs). All edge PoPs send cache misses to the shield, not directly to the origin. The shield collapses those 200 requests into 1 origin request. The origin now sees at most 1 request per unique asset per shield region, regardless of how many PoPs are requesting it.

Request collapsing: At the shield, multiple concurrent requests for the same URL are collapsed into one outgoing request to the origin (using a technique called "request coalescing"). While the origin request is in flight, all waiting PoPs are held until the response arrives, then served simultaneously.

Cost: One extra network hop (edge → shield → origin) on cache miss. Adds 20-50ms. Acceptable since cache hits (95%+) have zero extra hops.
🗑 Phase 3 — Cache Invalidation in <5 Seconds
PHASE 3 · INVALIDATION
IR
INTERVIEWER
A customer pushes a bad JS file. They call you demanding the CDN stops serving it immediately. How does invalidation propagate globally in under 5 seconds?
ME
YOU
Cache invalidation is the hardest part of CDN design. Two approaches:

1. Soft purge (mark stale): Don't delete the cached file immediately. Mark it as stale. The next request for this URL fetches the fresh version from origin and updates the cache. Old version continues being served until then (seconds to minutes). Works well for graceful rollouts — no traffic spike to origin.

2. Hard purge (delete immediately): The control plane sends a purge command to all 200 PoPs simultaneously. Each PoP deletes the cached file from disk. After deletion, the next request for that URL is a cache miss → fetches fresh from origin. This causes a brief traffic spike to origin as 200 PoPs simultaneously re-fetch the new asset.

Propagation mechanism for <5s globally:
  • Customer calls Purge API → control plane publishes purge event to a global message bus (Kafka, or a proprietary system)
  • Each PoP has a purge agent subscribing to this bus. On receipt: delete the cached file atomically
  • 200 PoPs × 5ms average purge processing = all done within 1-2 seconds
  • Purge is acknowledged only after all PoPs confirm deletion
Surrogate keys (tag-based invalidation): Advanced feature — tag assets with keys (e.g., product:12345). A single purge command for that tag invalidates all cached variations of that product (different image sizes, locales, etc.) simultaneously across all PoPs.

📥 Pull vs Push Caching Strategy
PULL CACHING (default)
1
User requests asset. Edge checks local cache.
2
Cache miss → edge fetches from origin (via shield). Caches response.
3
Subsequent requests for same asset served from cache.
4
When to use: Long-tail content — millions of unique assets, most rarely accessed. No wasted cache storage on unused assets.
5
Problem: First user after deployment or cache expiry pays the miss penalty. Cold-start problem on new deployments.
PUSH CACHING (pre-warm)
1
Before deploying new content, customer calls CDN Push API.
2
CDN control plane distributes asset to all PoPs proactively.
3
First user hit is also a cache hit — zero cold-start latency.
4
When to use: Hot, known content — new software releases, movie trailers, game patches. Content that will be immediately viral.
5
Problem: Wastes storage on PoPs where content may never be requested. Requires explicit push call per deployment.

// CDN Edge Node — Java: Cache lookup + origin fetch
// ══════════════════════════════════════════════════════

@Service
public class EdgeCacheService {

    @Autowired private DiskCacheStore    diskCache;
    @Autowired private OriginShieldClient shieldClient;

    /**
     * Main edge request handler.
     * 1. Check local SSD cache
     * 2. On miss: fetch from origin shield (which may fetch origin)
     * 3. Cache response locally
     */
    public CacheResponse handle(HttpRequest req) {
        String cacheKey = buildCacheKey(req);

        // 1. Local cache lookup
        CacheEntry entry = diskCache.get(cacheKey);
        if (entry != null && !entry.isExpired()) {
            return CacheResponse.hit(entry);  // ~1ms
        }

        // 2. Stale-while-revalidate: serve stale, refresh in background
        if (entry != null && entry.isStaleButRevalidatable()) {
            CompletableFuture.runAsync(() -> refresh(cacheKey, req));
            return CacheResponse.stale(entry);  // serve stale immediately
        }

        // 3. Cache miss: fetch from shield (request coalescing handled at shield)
        OriginResponse originResp = shieldClient.fetch(req);
        CacheEntry fresh = new CacheEntry(
            cacheKey,
            originResp.getBody(),
            originResp.getHeaders(),
            parseTtl(originResp.getHeader("Cache-Control"))
        );
        diskCache.put(cacheKey, fresh);
        return CacheResponse.miss(fresh);
    }

    /** Cache key includes URL + relevant Vary headers */
    private String buildCacheKey(HttpRequest req) {
        StringBuilder sb = new StringBuilder(req.getUrl());
        // Vary: Accept-Encoding means gzip/br responses are cached separately
        if (req.getHeader("Accept-Encoding") != null) {
            sb.append("|enc:").append(normalizeEncoding(req.getHeader("Accept-Encoding")));
        }
        return sb.toString();
    }

    private int parseTtl(String cacheControl) {
        if (cacheControl == null) return 3600; // default 1h
        return CacheControlParser.maxAge(cacheControl);
    }
}

// ── Origin Shield — Request Coalescing ────────────────
@Service
public class OriginShieldService {

    // inflight: URL → pending future (request coalescing)
    private final ConcurrentHashMap<String, CompletableFuture<OriginResponse>> inflight
        = new ConcurrentHashMap<>();

    public OriginResponse fetch(HttpRequest req) {
        return inflight.computeIfAbsent(req.getUrl(), url -> {
            // Only the first thread fires the origin request.
            // All others join the same CompletableFuture.
            return CompletableFuture
                .supplyAsync(() -> fetchFromOrigin(url))
                .whenComplete((resp, ex) -> inflight.remove(url));
        }).join(); // all concurrent requests wait for the same origin call
    }

    private OriginResponse fetchFromOrigin(String url) {
        // HTTP/2 to origin for multiplexing
        return originHttpClient.get(url);
    }
}

// ── Purge Handler — invalidate cached asset ───────────
@KafkaListener(topics = "cdn-purge-events")
public void onPurgeEvent(PurgeEvent event) {
    for (String url : event.getUrls()) {
        String key = buildCacheKey(url, event.getVaryHeaders());
        diskCache.delete(key);
    }
    purgeAckService.acknowledge(event.getPurgeId(), POP_ID);
}

import asyncio, time, hashlib
from dataclasses import dataclass
from typing import Optional

@dataclass
class CacheEntry:
    key:        str
    body:       bytes
    headers:    dict
    expires_at: float
    etag:       str = ""

    def is_expired(self) -> bool:
        return time.time() > self.expires_at

    def ttl_remaining(self) -> float:
        return max(0, self.expires_at - time.time())


class EdgeCache:
    """In-memory cache (production: disk-backed SSD cache)."""
    def __init__(self):
        self._store: dict[str, CacheEntry] = {}

    def get(self, key: str) -> Optional[CacheEntry]:
        entry = self._store.get(key)
        if entry and not entry.is_expired(): return entry
        return None

    def put(self, entry: CacheEntry): self._store[entry.key] = entry

    def delete(self, key: str): self._store.pop(key, None)


class OriginShield:
    """Request coalescing: N concurrent misses → 1 origin request."""
    def __init__(self, origin_url: str):
        self.origin_url = origin_url
        self._inflight: dict[str, asyncio.Future] = {}

    async def fetch(self, path: str) -> dict:
        if path in self._inflight:
            return await self._inflight[path]  # join existing request

        loop = asyncio.get_event_loop()
        future = loop.create_future()
        self._inflight[path] = future

        try:
            result = await self._fetch_from_origin(path)
            future.set_result(result)
            return result
        except Exception as e:
            future.set_exception(e)
            raise
        finally:
            self._inflight.pop(path, None)

    async def _fetch_from_origin(self, path: str) -> dict:
        # HTTP/2 request to origin
        import httpx
        async with httpx.AsyncClient(http2=True) as client:
            r = await client.get(f"{self.origin_url}{path}")
            return {"body": r.content, "headers": dict(r.headers),
                    "status": r.status_code}


class EdgeNode:
    def __init__(self, shield: OriginShield):
        self.cache  = EdgeCache()
        self.shield = shield

    async def handle_request(self, path: str, headers: dict) -> dict:
        key = self._cache_key(path, headers)

        # 1. Cache HIT
        entry = self.cache.get(key)
        if entry:
            return {"body": entry.body, "headers": entry.headers,
                    "x-cache": "HIT"}

        # 2. Cache MISS: fetch from shield (request coalescing)
        origin = await self.shield.fetch(path)

        # 3. Parse TTL from Cache-Control
        cc    = origin["headers"].get("cache-control", "max-age=3600")
        ttl   = self._parse_max_age(cc)
        fresh = CacheEntry(key=key, body=origin["body"],
                             headers=origin["headers"],
                             expires_at=time.time() + ttl)
        self.cache.put(fresh)

        return {"body": origin["body"], "headers": origin["headers"],
                "x-cache": "MISS"}

    def purge(self, urls: list[str]):
        """Handle purge command from control plane."""
        for url in urls:
            self.cache.delete(self._cache_key(url, {}))
        print(f"[PURGE] deleted {len(urls)} entries")

    def _cache_key(self, path: str, headers: dict) -> str:
        enc = headers.get("accept-encoding", "identity")
        return hashlib.md5(f"{path}|{enc}".encode()).hexdigest()

    def _parse_max_age(self, cc: str) -> int:
        for part in cc.split(","):
            part = part.strip()
            if part.startswith("max-age="):
                return int(part.split("=")[1])
        return 3600
⚖️ Tradeoffs
🗺 Routing: Anycast vs GeoDNS vs IP Anycast + GeoDNS
GeoDNS only

DNS resolver location → map to PoP IP. Simple to implement. Works everywhere. But: DNS TTL means slow failover (60s). DNS-based routing can be inaccurate when resolver is far from client (corporate proxy). Can't react to real-time congestion.

✅ Anycast BGP (chosen)

Single IP announced from all PoPs. BGP routes to topologically nearest. Sub-second failover when PoP goes down (BGP withdrawal). DDoS automatically spread across all PoPs. Industry standard for large CDNs. Requires BGP peering with ISPs at every PoP — significant operational investment.

Hybrid (both)

Anycast for primary routing. GeoDNS as fallback for clients behind NAT/VPN that break anycast. Anycast for attack traffic (it distributes automatically). GeoDNS for business continuity when specific PoPs need to be bypassed. Cloudflare uses this combination.

🗑 Cache Invalidation: TTL vs Purge vs Versioned URLs
Short TTL only

Set Cache-Control: max-age=60. Content auto-expires every 60s. No purge needed. Simple. Cost: origin gets 5% miss traffic every 60 seconds for all assets. At 100 Tbps scale, 5 Tbps hits origin every minute. Expensive. Also: content may be stale for up to 60s after update — poor UX for breaking news or product price changes.

✅ Long TTL + explicit purge

Cache-Control: max-age=86400 (24h). Most traffic served from cache. When content changes: call Purge API → propagates to all PoPs in <5s. Best of both: high cache hit rate + instant invalidation on demand. Industry standard for static assets.

✅ Versioned URLs (immutable cache)

/static/app.abc123.js — URL contains content hash. Cache-Control: max-age=31536000, immutable. Never expires. New deploys use new hash → new URL. Zero invalidation needed. The gold standard for JS/CSS assets. Requires build system integration but eliminates all cache invalidation complexity.
