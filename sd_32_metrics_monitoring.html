Metrics & Monitoring System (Prometheus/Datadog)
Design a metrics and monitoring platform — ingesting billions of time-series data points per day from thousands of services, storing them efficiently using columnar compression, querying with PromQL-style range functions, evaluating alerting rules every 15 seconds, and handling the high-cardinality explosion that kills most naive implementations.
🔴 MUST DO 🟣 Hard Time-Series DB High Cardinality Delta Encoding Alerting Downsampling Pull vs Push
🎯 Requirements
The hardest problem in metrics at scale is high cardinality. A metric like http_requests_total{service="api", status="200", pod="api-7d9f-k8s-pod-abc123"} creates one time series per unique label combination. With 10K pods × 500 status codes × 200 services = 1 billion unique series. Most TSDB implementations silently die here.

FUNCTIONAL
Ingest metrics from thousands of services (push/pull)
Store time-series data with labels (metric name + key=value pairs)
Query: range queries, aggregation, rate calculations
Alerting: evaluate rules every 15s, fire on threshold breach
Dashboards: visualise time-series over custom ranges
Data retention: 15s resolution for 2 weeks; 1m for 1 year
NON-FUNCTIONAL
10M time-series (active), 1B+ total ever seen
Ingest: 1M samples/sec (10M series × 1 sample/10s avg)
Query latency: <1s for last 1h; <10s for last 30d
Alert evaluation: exactly every 15 seconds, <5s to notify
Write availability: 99.9% — monitoring must not lose data
💬 Phase 1 — Time-Series Storage
PHASE 1 · STORAGE INTERNALS
IR
INTERVIEWER
1M samples/sec, each is a (timestamp, float64) pair. How do you store this efficiently? Raw storage would be 12 bytes × 1M/sec = 12 MB/sec = 1 TB/day.
ME
YOU
Prometheus TSDB and Gorilla (Facebook/Meta's TSDB) use a technique called double-delta encoding for timestamps and XOR-based compression for values, reducing storage by ~10×.

Timestamp compression: Metrics arrive at regular intervals (e.g., every 15s). Instead of storing full 8-byte timestamps, store deltas. For a 15s interval: delta=15000ms fits in 1 byte. For irregular intervals: store delta-of-deltas. A perfectly regular time series needs ~1 bit per sample for timestamps. Real-world: ~3-4 bytes per sample vs 8 bytes raw.

Value compression (Gorilla XOR encoding): Consecutive float64 values in a metric tend to be similar (CPU at 45%, 46%, 45%). XOR the current value with the previous one — the leading and trailing zeros tell you how many bits differ. Store only the changing bits with a prefix code. Real-world: many metrics store in <1 byte per sample vs 8 bytes raw. Overall: 10-12× compression typical.

Chunk format: Each time series is stored as a sequence of fixed-size chunks (typically 2h of data). Each chunk is compressed independently. Prometheus uses 128-byte chunks. Chunks are immutable once closed — a new chunk starts every 2h. This enables columnar-style access: to query a time range, load only the relevant chunks.
💡 Gorilla paper: Facebook's 2015 VLDB paper "Gorilla: A Fast, Scalable, In-Memory Time Series Database" describes this compression. Prometheus adopted it. Reduced their data by 12× with <1ms decompression overhead.
🔥 Phase 2 — High Cardinality
PHASE 2 · CARDINALITY EXPLOSION
IR
INTERVIEWER
Engineer adds a label "request_id" to a metric. Every HTTP request gets a unique request_id. System falls over in 30 minutes. Why and how do you prevent it?
ME
YOU
Why it happens: Each unique label set creates a new time series. Adding "request_id" to a metric with 10K req/sec = 10K new time series per second. After 10 minutes: 6M new time series. The TSDB's in-memory index (inverted index mapping labels → series IDs) balloons. Memory exhausted. Write path slows. Cascading failure.

Prevention — cardinality limits:
  • Per-metric label value limit: If any label has >10K unique values, reject new series creation. Alert the team. Prometheus has --storage.tsdb.max-block-chunk-segment-size and per-metric limits via recording rules.
  • Label value allow-listing: Only allow enumerated label values (e.g., status: {200, 400, 500, other}) — catch-all "other" bucket prevents explosion.
  • Cardinality dashboard: Monitor the rate of new series creation. Alert on >1K new series/min. Catch the problem before it escalates.
  • Drop high-cardinality labels at ingest: A relabelling rule drops the offending label before storing: labelmap drop request_id.
Detection: Track prometheus_tsdb_head_series metric. If it grows faster than baseline, trigger an alert to the metric owner.
🚨 Phase 3 — Alerting
PHASE 3 · ALERTING
IR
INTERVIEWER
You need to evaluate 10K alert rules every 15 seconds across 10M series. How?
ME
YOU
Alert evaluation at scale:

Each alert rule is a PromQL query with a threshold: rate(http_errors_total[5m]) > 0.01. Running this across all 10M series every 15s would be expensive — O(10M) work 4× per minute = 40M evaluations/minute.

Key optimisation — pre-computed recording rules: Instead of evaluating complex queries at alert time, pre-compute them continuously: record: job:http_errors:rate5m = rate(http_errors_total[5m]). This creates a new (lower-cardinality) time series that's already computed. Alert rules just threshold the pre-computed value — O(1) per alert rule. Recording rules run every 15s as streaming jobs.

Alert manager flow: Evaluation → Alertmanager → route by severity/team → deduplicate (same alert firing repeatedly = 1 notification) → group (30 alerts from same service = 1 pagerduty) → silence (known issue, suppress for 4h) → notify (PagerDuty/Slack/email).

Avoid alert storms: Alert on symptoms not causes. One alert "user-facing error rate > 1%" is better than 50 alerts "service X returning 5xx" + "DB connection timeout" + etc. This is Google's SRE "symptom-based alerting" principle.

            .description("Total orders created")
            .register(registry);

        // Timer: latency histogram with standard buckets
        orderLatency = Timer.builder("order_processing_seconds")
            .tag("service", "order-svc")
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);

        // Gauge: current value (not cumulative)
        pendingOrders = Gauge.builder("orders_pending", orderQueue, Queue::size)
            .tag("service", "order-svc")
            .register(registry);

        // Histogram: distribution of order values
        orderValue = DistributionSummary.builder("order_value_dollars")
            .serviceLevelObjectives(10, 50, 100, 500, 1000)
            .register(registry);
    }

    @PostMapping("/orders")
    public Order createOrder(@RequestBody OrderRequest req) {
        return orderLatency.record(() -> {
            Order order = orderService.create(req);
            ordersTotal.increment();
            orderValue.record(req.getTotalAmount());
            return order;
        });
    }
}

# compression concept + alerting engine

import time, struct, threading
from collections import defaultdict
from dataclasses import dataclass, field
from typing import List, Dict, Optional, Callable

@dataclass
class Sample:
    timestamp: int   # unix ms
    value:     float


class TimeSeries:
    """In-memory time series with delta-encoded timestamp storage."""
    def __init__(self, labels: dict):
        self.labels    = labels
        self.samples:  List[Sample] = []
        self._last_ts  = 0
        self._last_val = 0.0

    def append(self, ts: int, val: float):
        self.samples.append(Sample(ts, val))
        # In real Gorilla: store delta(ts) and XOR(val) in compressed chunk
        self._last_ts  = ts
        self._last_val = val

    def query_range(self, start_ms: int, end_ms: int) -> List[Sample]:
        return [s for s in self.samples if start_ms <= s.timestamp <= end_ms]


class TSDB:
    """Minimal time-series database with label-based lookup."""
    def __init__(self):
        self._series: Dict[str, TimeSeries] = {}
        self._index:  Dict[str, Dict[str, List[str]]] = defaultdict(lambda: defaultdict(list))
        self._lock = threading.RLock()

    def _labels_to_key(self, labels: dict) -> str:
        return ",".join(f"{k}={v}" for k, v in sorted(labels.items()))

    def write(self, labels: dict, ts_ms: int, value: float):
        key = self._labels_to_key(labels)
        with self._lock:
            if key not in self._series:
                self._series[key] = TimeSeries(labels)
                # Update inverted index: label_name=value → series_keys
                for name, val in labels.items():
                    self._index[name][val].append(key)
            self._series[key].append(ts_ms, value)

    def select(self, label_matchers: dict) -> List[TimeSeries]:
        """Find all series matching ALL label matchers."""
        with self._lock:
            candidate_keys = None
            for name, val in label_matchers.items():
                matches = set(self._index[name].get(val, []))
                candidate_keys = matches if candidate_keys is None else candidate_keys & matches
            return [self._series[k] for k in (candidate_keys or set())]


class AlertEngine:
    """Evaluates PromQL-style threshold alerts every 15s."""
    def __init__(self, tsdb: TSDB, notify: Callable):
        self.tsdb    = tsdb
        self.notify  = notify
        self.rules:  list = []
        self.firing: set  = set()

    def add_rule(self, name: str, matchers: dict, threshold: float,
                 window_s: int = 300, comparator: str = ">"):
        self.rules.append({"name": name, "matchers": matchers,
                           "threshold": threshold, "window_s": window_s,
                           "comparator": comparator})

    def evaluate_all(self):
        now_ms = int(time.time() * 1000)
        for rule in self.rules:
            series_list = self.tsdb.select(rule["matchers"])
            for ts in series_list:
                window_start = now_ms - rule["window_s"] * 1000
                samples = ts.query_range(window_start, now_ms)
                if not samples: continue
                avg_val = sum(s.value for s in samples) / len(samples)
                alert_key = (rule["name"], str(ts.labels))
                firing_now = (rule["comparator"] == ">" and avg_val > rule["threshold"])
                if firing_now and alert_key not in self.firing:
                    self.firing.add(alert_key)
                    self.notify(rule["name"], ts.labels, avg_val)
                elif not firing_now and alert_key in self.firing:
                    self.firing.discard(alert_key)  # resolved

    def run(self, interval_s: int = 15):
        while True:
            self.evaluate_all()
            time.sleep(interval_s)
⚖️ Tradeoffs
📥 Collection: Pull (Prometheus) vs Push (Datadog/InfluxDB)
✅ Pull (Prometheus)

Prometheus scrapes each service's /metrics endpoint. Scraper knows which targets are up — if a target doesn't respond, that's itself a signal (down). Service discovery via Kubernetes SD / Consul / EC2 tags. No firewall rules needed from service to monitoring. Standard for Kubernetes-native infra.

✅ Push (Datadog/StatsD)

Services push metrics to a local agent, which batches and forwards. Works for short-lived processes (serverless, cron jobs) that may not be alive when the scraper polls. Works across NAT/firewalls. Scales better at high metric volume. Datadog, InfluxDB, OpenTelemetry all support push. Better for heterogeneous environments.

Hybrid (both)

Prometheus for long-running services (pull), OTel collector for push from short-lived services, lambdas, batch jobs. OTel has become the industry standard for vendor-neutral metric/trace/log collection. Use Prometheus as the storage backend regardless of collection method.

🗄 TSDB: Prometheus vs Thanos vs VictoriaMetrics vs Grafana Mimir
Prometheus (single node)

Single binary, easy to run. 2-week storage, 10-20M series max. No HA — if node dies, metrics are unavailable. The starting point for every monitoring system. Needs Thanos/Cortex for HA and long-term.

✅ Thanos / Grafana Mimir

HA Prometheus clusters with global query view. Thanos: fan-out query across multiple Prometheus instances. Mimir: fully horizontally scalable TSDB (Cortex successor). Both use object storage (S3) for long-term. Industry standard for large-scale production monitoring.

✅ VictoriaMetrics

Drop-in Prometheus replacement with 2-3× better compression and lower RAM. Single binary handles 100M+ series. Prometheus-compatible query API. Better at high cardinality than Prometheus. Gaining adoption for cost-conscious teams. Used by Grammarly, Gitlab, Adidas.
