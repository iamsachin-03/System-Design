Stock Exchange / Trading System
Design the core matching engine of a stock exchange — price-time priority order book, order types (market/limit/cancel), microsecond-latency matching, market data distribution (Level 2 feeds), risk checks, and the sequencer that guarantees total ordering of trades.
🔴 MUST DO 🟣 Hard Order Book Matching Engine Sequencer LMAX Disruptor Price-Time Priority Market Data Feed
🎯 Clarify Scope
Stock exchange is the most latency-sensitive system design question. The hard constraints are microsecond-order latency, strict ordering guarantees (every trade must be deterministic — same input always produces same output), and zero tolerance for data loss or duplicate trades.

✅ Functional Requirements
CORE
Place Order: Submit a buy or sell order (market or limit). Order receives a unique ID and is immediately acknowledged. Limit orders that don't match rest on the book.
Cancel Order: Cancel a resting limit order by order ID. Must be atomic — can't cancel an order that just matched.
Matching Engine: Match buy and sell orders using Price-Time Priority (FIFO within same price level). Produce trade records when orders match.
Order Book: Maintain the live order book per symbol: sorted bids (buy orders, descending price) and asks (sell orders, ascending price).
Market Data Feed: Publish order book updates and trade executions in real-time to subscribers.
Order Status: Query current status of an order: PENDING, OPEN, PARTIALLY_FILLED, FILLED, CANCELLED.
EXTENDED
Pre-trade Risk Checks: Validate order before matching — sufficient funds/shares, price within circuit breaker limits, position limits.
Circuit Breakers: Halt trading for a symbol if price moves >10% in 5 minutes.
Trade History: Immutable audit log of all trades (regulatory requirement). Every trade timestamped to nanosecond.
Market Open/Close: Pre-market auction, regular session, after-hours handling.
⚡ Non-Functional Requirements
LATENCY (CRITICAL)
Order acknowledgment: <10ms p99 (retail exchange)
Matching engine internal: <1µs per order (in-memory, single-threaded)
Market data feed lag: <1ms from match to subscribers
CORRECTNESS (NON-NEGOTIABLE)
No duplicate trades — each order matched at most once per fill
Strict price-time priority — no order jumping the queue
Deterministic: replay of the same order sequence always produces same trades
DURABILITY
All orders persisted to WAL before acknowledgment
Crash recovery: replay WAL to reconstruct order book state
Trade records immutable, append-only, never deleted
THROUGHPUT

🎙 Phase 1 — Why Single-Threaded Matching Engine?
PHASE 1 · ARCHITECTURE
IR
INTERVIEWER
Design a stock exchange matching engine. Start with the core architecture decision.
ME
YOU
The most important architectural decision in a matching engine is: the matching engine must be single-threaded.

This sounds counterintuitive — why not use a thread pool for throughput? Because:

1. Correctness requires total ordering. Two orders arriving simultaneously must be matched in a deterministic order. If thread A and thread B both try to match against the same resting order, you get a race condition — both could claim the same fill. A mutex "solves" this but introduces lock contention, which is expensive at microsecond latencies.

2. No locking = much faster. A single thread operating on in-memory data structures with no locks can process 5-10 million orders/second on modern hardware. Lock contention on a multi-threaded approach typically achieves less throughput than a well-optimized single-threaded design.

3. Deterministic replay. The matching engine is essentially a deterministic state machine: same input sequence → same output. This makes crash recovery trivial: replay the WAL through the matching engine from the last snapshot to rebuild state.

The architecture:
  1. Sequencer — assigns a global monotonic sequence number to every incoming order. This defines total ordering.
  2. Matching Engine — single-threaded, processes orders in sequence number order from a ring buffer. No locks, no network I/O, just in-memory operations.
  3. Outbound channels — trade events and order book updates are written to output queues for downstream systems.
This is the LMAX Disruptor pattern, used by the LMAX Exchange (London), NASDAQ, and many others.
💡 "Single-threaded for correctness" is the key insight. This immediately distinguishes you from candidates who suggest a distributed matching engine or thread pool. The matching engine is intentionally sequential — parallelism happens at the level of multiple independent matching engines (one per symbol or symbol group).
📚 Phase 2 — Order Book & Matching Algorithm
PHASE 2 · ORDER BOOK
IR
INTERVIEWER
Walk me through the order book data structure and the matching algorithm.
ME
YOU
The order book is a sorted collection of outstanding limit orders. Two sides:
  • Bids (buy orders): Sorted by price descending (highest bid first), then by time ascending (FIFO within same price)
  • Asks (sell orders): Sorted by price ascending (lowest ask first), then by time ascending
Data structure: TreeMap<Price, Queue<Order>>
- TreeMap: O(log N) price level insert/lookup. Price levels are the keys. This gives us sorted iteration by price.
- Queue<Order> at each price level: Orders at the same price form a FIFO queue. The front of the queue is matched first (price-time priority).
- We also maintain a HashMap<OrderId, Order> for O(1) cancel lookups.

Matching algorithm for a new buy limit order at price P:
  1. Check if best ask ≤ P (there's a matching sell order)
  2. If yes: fill against the front order in the lowest ask queue
  3. If the ask is fully filled, remove it from the queue; remove empty queue from TreeMap
  4. If the new order is partially filled, continue matching against the next ask
  5. If all asks at price ≤ P are exhausted, remaining quantity rests on the bid side
Market order: Same as limit order but with no price limit — keeps matching until fully filled or the book is empty (rare). If the book empties before the market order is filled, the remainder is typically cancelled (depends on exchange rules).
IR
INTERVIEWER
What's the time complexity of adding an order? Cancelling?
ME
YOU
Add limit order (no match): O(log N) for TreeMap insert at a new price level, or O(1) if the price level already exists (just enqueue).

Match limit/market order: O(K × log N) where K is the number of fills (usually 1-2), and log N is the TreeMap operation to check/remove exhausted price levels.

Cancel order: O(1) average with the HashMap: lookup order by ID → O(1). Remove from the price level queue: O(1) for doubly-linked list (if order stores a pointer to its position). Remove price level if empty: O(log N) TreeMap remove.

In practice, the matching engine processes hundreds of thousands of orders per second because all operations are O(1) or O(log N) on small N (the order book typically has <1000 active price levels per symbol), and all data fits in L3 cache.
🔢 Phase 3 — The Sequencer: Total Ordering
PHASE 3 · SEQUENCER
IR
INTERVIEWER
How do you guarantee total ordering when orders arrive from thousands of brokers simultaneously?
ME
YOU
This is the sequencer's job. Every order, before it enters the matching engine, must be assigned a globally unique, monotonically increasing sequence number. Orders are then processed in sequence number order — this establishes total ordering.

Sequencer design:
The sequencer is a single bottleneck by design. It receives all incoming orders and stamps them with a sequence number (1, 2, 3, ...). It then writes the sequenced order to a ring buffer (LMAX Disruptor pattern) from which the matching engine reads.

Why a ring buffer?
The LMAX Disruptor ring buffer avoids garbage collection, lock contention, and is cache-friendly (all data in a pre-allocated array). The publisher (sequencer) writes to position N, the consumer (matching engine) reads from position N-1. One cache-line distance separation prevents false sharing. This gives ~6 million events/second on commodity hardware.

Durability:
Before the sequencer publishes to the ring buffer, it writes the order to a Write-Ahead Log (WAL) on a local SSD (or replicated). The WAL is append-only, so it achieves low latency even with durability. On crash, replay the WAL through a fresh matching engine to reconstruct state from the last snapshot.
IR
INTERVIEWER
The sequencer is a single bottleneck. What's your strategy if it fails?
ME
YOU
The sequencer must be highly available. Three approaches:

1. Hot standby: A secondary sequencer receives all orders in parallel but doesn't publish sequence numbers. If the primary fails, the standby takes over at the last sequence number + 1. Failover time: ~100ms.

2. Raft-based sequencer: 3 sequencer nodes form a Raft cluster. All writes go through the Raft leader. On leader failure, new leader elected in <500ms. Higher latency (+1-2ms) due to Raft consensus. NYSE uses a similar multi-copy sequencing approach.

3. Accept the downtime: Exchanges actually take brief maintenance windows. A 10-second outage for failover is acceptable (vs microseconds of latency degradation from Raft). Most real exchanges use hot standby with deterministic failover.
📡 Phase 4 — Market Data Feed
PHASE 4 · MARKET DATA
IR
INTERVIEWER
How do you distribute market data (order book updates and trade prints) to 1 million subscribers?
ME
YOU
Market data is a fan-out problem — one event (a trade at AAPL) must be delivered to 1M+ subscribers with <1ms latency. The key constraint is that you cannot update the order book atomically for all subscribers — you must fan out.

Two tiers of market data:
Level 1 (BBO — Best Bid and Offer): Just the best price and quantity on each side. Low bandwidth. All 1M subscribers can receive Level 1 updates via UDP multicast. UDP multicast sends one packet to a multicast group, the network switch copies it to all subscribers — massive bandwidth savings vs TCP unicast to 1M clients.

Level 2 (Full Depth): All price levels in the book. Higher bandwidth. Typically sold as a premium data feed to a smaller set of subscribers (algorithmic traders, market makers). Published via TCP to dedicated data servers that re-distribute.

Sequence numbers on market data: Every market data message has a sequence number. If a subscriber misses message 547 (gap between 546 and 548), it requests a retransmission from the market data replay server. This handles the unreliability of UDP multicast.

Feed architecture:
Matching Engine → output ring buffer → Market Data Publisher → UDP multicast (Level 1) + TCP unicast (Level 2 premium) → subscribers
🛡 Phase 5 — Risk Checks & Circuit Breakers
PHASE 5 · RISK
IR
INTERVIEWER
Where do you perform risk checks? Before or after the sequencer?
ME
YOU
There are two layers of risk checks, each with different tradeoffs:

Pre-sequencer (soft checks at the gateway):
Each broker gateway (the entry point for broker connections) performs fast, stateless checks:
  • Order format validation (valid symbol, price, quantity)
  • Account authentication
  • Gross notional limit check: order value ≤ max allowed (e.g., single order can't be >$1M for a retail account)
These run before the sequencer and can reject orders quickly, reducing sequencer load.

Pre-matching (hard checks inside the sequencer/engine):
  • Position checks: does the seller actually own the shares they're selling?
  • Buying power check: does the buyer have sufficient funds?
  • Circuit breaker: is this symbol currently halted?
These must be synchronous with the matching engine to be authoritative. A position check at the gateway can be stale (another order just used up the position).

Circuit breakers: If a stock moves >10% in 5 minutes, halt trading. The matching engine checks the current price against the last circuit breaker snapshot on every match. If triggered, all new orders are rejected for 5 minutes, then trading resumes. This is a regulatory requirement (SEC Rule 48, LULD bands).

📊 Capacity Estimation
Stock exchanges are NOT storage or bandwidth problems — they are latency problems. All active order books fit in a few GB of RAM. The challenge is processing 500K orders/sec with sub-millisecond latency.
1
Order Rate & Matching Engine Capacity
How fast must the matching engine be?
Given: 500,000 orders/sec peak, 10,000 symbols Average orders per symbol per second: 500,000 ÷ 10,000 = 50 orders/sec avg per symbol Hot symbols (top 100, e.g., AAPL, TSLA): ~10× average 500 orders/sec for hot symbols Single-threaded matching engine throughput: LMAX Disruptor-style: 5-10 million ops/sec per core One matching engine per symbol group (100 symbols) 100 matching engine threads for 10,000 symbols Each handles: 500,000 ÷ 100 = 5,000 orders/sec comfortable Time per order (each matching engine): 1,000,000 µs/sec ÷ 5,000 orders/sec = 200 µs/order budget per order Actual matching: ~1 µs → 200× headroom
2
Order Book Memory
How much RAM for all active order books?
Per resting limit order size: order_id (8B) + symbol (4B) + side (1B) + price (8B) + quantity (8B) + timestamp (8B) + client_id (8B) = 45 bytes Active resting orders per symbol (typical liquid stock): ~1,000 price levels × 5 orders/level = 5,000 orders/symbol Total active orders across all symbols: 10,000 symbols × 5,000 orders = 50 million orders Memory for all order books: 50M × 45B = ~2.25 GB RAM Plus TreeMap overhead (~100B per price level): 10,000 × 1,000 levels × 100B = ~1 GB Total order book RAM ≈ ~3.5 GB → All order books fit comfortably in 16 GB RAM. This is intentional — everything in-memory = fast.
3
Trade Storage & Market Data
Persistent storage requirements
Trades executed per day (10% of orders result in a trade fill): 500,000 orders/sec × 0.1 × 23,400 sec (trading day) = ~1.17 billion fills/day Per trade record size: trade_id (8B) + buy_order_id (8B) + sell_order_id (8B) + symbol (4B) + price (8B) + quantity (8B) + timestamp (8B) = 52 bytes Daily trade storage: 1.17B × 52B = ~61 GB/day Annual trade storage (immutable audit log): 61 GB × 252 trading days = ~15 TB/year (Cassandra append-only) WAL throughput (orders + events, pre-matching): 500,000 orders/sec × 64 bytes = ~32 MB/s NVMe SSD can sustain this easily Market data feed bandwidth (Level 1 multicast): 500,000 events/sec × 50B = ~25 MB/s UDP multicast

// Stock Exchange — Matching Engine (Java)
// ══════════════════════════════════════════════════════

// ── 1. Order Book with Price-Time Priority ────────────
public class OrderBook {

    private final String symbol;

    // Bids: highest price first → descending TreeMap
    private final TreeMap<BigDecimal, ArrayDeque<Order>> bids =
        new TreeMap<>(Comparator.reverseOrder());

    // Asks: lowest price first → natural ascending TreeMap
    private final TreeMap<BigDecimal, ArrayDeque<Order>> asks =
        new TreeMap<>();

    // O(1) cancel lookup by orderId
    private final HashMap<Long, Order> orderIndex = new HashMap<>();

    public List<Trade> addOrder(Order incoming) {
        List<Trade> trades = new ArrayList<>();

        if (incoming.getSide() == Side.BUY) {
            matchBuy(incoming, trades);
        } else {
            matchSell(incoming, trades);
        }

        // If not fully filled and it's a limit order, add to book
        if (!incoming.isFullyFilled()
                && incoming.getType() == OrderType.LIMIT) {
            TreeMap<BigDecimal, ArrayDeque<Order>> side =
                incoming.getSide() == Side.BUY ? bids : asks;
            side.computeIfAbsent(incoming.getPrice(), p -> new ArrayDeque<>())
                .addLast(incoming);  // FIFO: new orders at the back
            orderIndex.put(incoming.getOrderId(), incoming);
        }
        return trades;
    }

    private void matchBuy(Order buy, List<Trade> trades) {
        // Check each ask level from lowest price upward
        while (!buy.isFullyFilled() && !asks.isEmpty()) {
            Map.Entry<BigDecimal, ArrayDeque<Order>> bestAskEntry =
                asks.firstEntry();
            BigDecimal bestAskPrice = bestAskEntry.getKey();

            // Check if price matches (limit: ask ≤ buy limit; market: always)
            boolean priceMatches = buy.getType() == OrderType.MARKET
                || bestAskPrice.compareTo(buy.getPrice()) <= 0;

            if (!priceMatches) break; // no more matching asks

            ArrayDeque<Order> askQueue = bestAskEntry.getValue();
            Order sellOrder = askQueue.peekFirst(); // FIFO: front of queue

            // Calculate fill quantity
            long fillQty = Math.min(buy.getRemainingQty(),
                                       sellOrder.getRemainingQty());

            // Execute fill at the ask price (price of the resting order)
            buy.fill(fillQty);
            sellOrder.fill(fillQty);

            trades.add(Trade.builder()
                .buyOrderId(buy.getOrderId())
                .sellOrderId(sellOrder.getOrderId())
                .price(bestAskPrice)   // price of the maker (resting) order
                .quantity(fillQty)
                .symbol(symbol)
                .timestampNs(System.nanoTime())
                .build());

            // Remove fully filled sell orders from book
            if (sellOrder.isFullyFilled()) {
                askQueue.pollFirst();
                orderIndex.remove(sellOrder.getOrderId());
                if (askQueue.isEmpty()) asks.remove(bestAskPrice);
            }
        }
    }

    public boolean cancelOrder(long orderId) {
        Order order = orderIndex.remove(orderId);
        if (order == null) return false; // not found / already filled

        TreeMap<BigDecimal, ArrayDeque<Order>> side =
            order.getSide() == Side.BUY ? bids : asks;

        ArrayDeque<Order> queue = side.get(order.getPrice());
        if (queue != null) {
            queue.remove(order); // O(N) for deque but N is usually tiny
            if (queue.isEmpty()) side.remove(order.getPrice());
        }
        return true;
    }

    public Optional<BigDecimal> getBestBid() {
        return bids.isEmpty()
            ? Optional.empty()
            : Optional.of(bids.firstKey());
    }

    public Optional<BigDecimal> getBestAsk() {
        return asks.isEmpty()
            ? Optional.empty()
            : Optional.of(asks.firstKey());
    }
}

// ── 2. Matching Engine Event Loop ─────────────────────
public class MatchingEngine implements Runnable {

    private final Map<String, OrderBook> books = new HashMap<>();
    private final RingBuffer<OrderEvent> ringBuffer; // LMAX Disruptor
    private final TradePublisher           publisher;
    private final CircuitBreakerMonitor     circuitBreaker;
    private volatile boolean               running = true;

    @Override
    public void run() {
        long nextSeq = ringBuffer.getCursor() + 1;

        while (running) {
            long available = ringBuffer.waitFor(nextSeq);

            while (nextSeq <= available) {
                OrderEvent event = ringBuffer.get(nextSeq);
                processEvent(event);
                nextSeq++;
            }
            ringBuffer.publish(nextSeq - 1); // advance consumer cursor
        }
    }

    private void processEvent(OrderEvent event) {
        switch (event.getType()) {
            case NEW_ORDER -> processNewOrder(event.getOrder());
            case CANCEL   -> processCancelOrder(event.getOrderId());
        }
    }

    private void processNewOrder(Order order) {
        // Circuit breaker check
        if (circuitBreaker.isHalted(order.getSymbol())) {
            publisher.publishRejection(order, "CIRCUIT_BREAKER_HALT");
            return;
        }

        OrderBook book = books.computeIfAbsent(order.getSymbol(),
            sym -> new OrderBook(sym));

        List<Trade> trades = book.addOrder(order);

        // Publish all generated trades (market data + audit log)
        for (Trade trade : trades) {
            publisher.publishTrade(trade);
            circuitBreaker.onTrade(trade); // update for circuit breaker
        }

        // Publish order status update
        publisher.publishOrderUpdate(order);
    }
}

// ── 3. Circuit Breaker Monitor ────────────────────────
public class CircuitBreakerMonitor {

    private final Map<String, CircuitState> states = new HashMap<>();
    private static final double HALT_THRESHOLD   = 0.10; // 10% move
    private static final long   WINDOW_NANOS     = 5 * 60 * 1_000_000_000L;
    private static final long   HALT_DURATION_NS = 5 * 60 * 1_000_000_000L;

    public void onTrade(Trade trade) {
        CircuitState state = states.computeIfAbsent(
            trade.getSymbol(), s -> new CircuitState());

        if (state.referencePrice == null) {
            state.referencePrice = trade.getPrice();
            state.windowStart    = trade.getTimestampNs();
            return;
        }

        // Reset reference price every 5-minute window
        if (trade.getTimestampNs() - state.windowStart > WINDOW_NANOS) {
            state.referencePrice = trade.getPrice();
            state.windowStart    = trade.getTimestampNs();
        }

        // Check percentage move from window open
        double move = trade.getPrice().subtract(state.referencePrice)
            .abs().divide(state.referencePrice, 4, RoundingMode.HALF_UP)
            .doubleValue();

        if (move >= HALT_THRESHOLD && !state.isHalted) {
            state.isHalted   = true;
            state.haltedAt   = trade.getTimestampNs();
            log.warn("Circuit breaker triggered: {} moved {}%",
                     trade.getSymbol(), move * 100);
        }
    }

    public boolean isHalted(String symbol) {
        CircuitState state = states.get(symbol);
        if (state == null || !state.isHalted) return false;
        // Auto-resume after HALT_DURATION_NS
        if (System.nanoTime() - state.haltedAt > HALT_DURATION_NS) {
            state.isHalted = false;
            return false;
        }
        return true;
    }
}

from sortedcontainers import SortedDict
from collections import deque
from dataclasses import dataclass, field
from decimal import Decimal
from enum import Enum
from typing import List, Dict, Optional
import time


class Side(Enum):
    BUY  = "BUY"
    SELL = "SELL"

class OrderType(Enum):
    MARKET = "MARKET"
    LIMIT  = "LIMIT"


@dataclass
class Order:
    order_id:        int
    client_order_id: str
    symbol:          str
    side:            Side
    order_type:      OrderType
    quantity:        int
    price:           Optional[Decimal] = None
    filled_qty:      int                  = 0
    sequence_number: int                  = 0
    timestamp_ns:    int                  = field(default_factory=time.time_ns)

    def remaining_qty(self) -> int:
        return self.quantity - self.filled_qty

    def is_fully_filled(self) -> bool:
        return self.filled_qty >= self.quantity

    def fill(self, qty: int):
        self.filled_qty += qty


@dataclass
class Trade:
    buy_order_id:  int
    sell_order_id: int
    symbol:        str
    price:         Decimal
    quantity:      int
    timestamp_ns:  int = field(default_factory=time.time_ns)


# ── Order Book (Price-Time Priority) ──────────────────
class OrderBook:
    """
    Maintains bids and asks for one symbol.
    Price-Time Priority: highest bid first, lowest ask first.
    Within same price: FIFO (oldest order matched first).

    Data structures:
    - SortedDict for O(log N) price level operations
    - deque at each price level for O(1) FIFO
    - dict for O(1) cancel by order_id
    """

    def __init__(self, symbol: str):
        self.symbol = symbol
        # Bids: negate price so SortedDict gives highest-first ordering
        self.bids: Dict[Decimal, deque] = SortedDict(lambda p: -p)
        # Asks: natural ascending order (lowest ask first)
        self.asks: Dict[Decimal, deque] = SortedDict()
        # O(1) lookup for cancel
        self.order_index: Dict[int, Order] = {}

    def add_order(self, order: Order) -> List[Trade]:
        trades = []
        if order.side == Side.BUY:
            self._match_buy(order, trades)
        else:
            self._match_sell(order, trades)

        # Remaining unmatched quantity → rest on book (limit orders only)
        if not order.is_fully_filled() and order.order_type == OrderType.LIMIT:
            side = self.bids if order.side == Side.BUY else self.asks
            if order.price not in side:
                side[order.price] = deque()
            side[order.price].append(order)  # FIFO: append to back
            self.order_index[order.order_id] = order
        return trades

    def _match_buy(self, buy: Order, trades: List[Trade]):
        """Match a buy order against resting asks (lowest price first)."""
        while not buy.is_fully_filled() and self.asks:
            best_ask_price = next(iter(self.asks))  # lowest ask

            # Price check: for limit orders, ask must be ≤ buy limit
            if (buy.order_type == OrderType.LIMIT
                    and best_ask_price > buy.price):
                break

            ask_queue = self.asks[best_ask_price]
            sell_order = ask_queue[0]  # front of queue (FIFO)

            fill_qty = min(buy.remaining_qty(), sell_order.remaining_qty())
            buy.fill(fill_qty)
            sell_order.fill(fill_qty)

            trades.append(Trade(
                buy_order_id=buy.order_id,
                sell_order_id=sell_order.order_id,
                symbol=self.symbol,
                price=best_ask_price,   # maker price (resting order)
                quantity=fill_qty
            ))

            # Clean up fully filled sell orders
            if sell_order.is_fully_filled():
                ask_queue.popleft()
                self.order_index.pop(sell_order.order_id, None)
                if not ask_queue:
                    del self.asks[best_ask_price]

    def _match_sell(self, sell: Order, trades: List[Trade]):
        """Match a sell order against resting bids (highest price first)."""
        while not sell.is_fully_filled() and self.bids:
            best_bid_price = next(iter(self.bids))  # highest bid (SortedDict negated)

            if (sell.order_type == OrderType.LIMIT
                    and best_bid_price < sell.price):
                break

            bid_queue = self.bids[best_bid_price]
            buy_order = bid_queue[0]

            fill_qty = min(sell.remaining_qty(), buy_order.remaining_qty())
            sell.fill(fill_qty)
            buy_order.fill(fill_qty)

            trades.append(Trade(
                buy_order_id=buy_order.order_id,
                sell_order_id=sell.order_id,
                symbol=self.symbol,
                price=best_bid_price,
                quantity=fill_qty
            ))

            if buy_order.is_fully_filled():
                bid_queue.popleft()
                self.order_index.pop(buy_order.order_id, None)
                if not bid_queue:
                    del self.bids[best_bid_price]

    def cancel_order(self, order_id: int) -> bool:
        order = self.order_index.pop(order_id, None)
        if not order:
            return False  # already filled or not found

        side = self.bids if order.side == Side.BUY else self.asks
        queue = side.get(order.price)
        if queue:
            try:
                queue.remove(order)  # O(N) but queue is tiny
            except ValueError:
                pass
            if not queue:
                del side[order.price]
        return True

    def best_bid(self) -> Optional[Decimal]:
        return next(iter(self.bids), None) if self.bids else None

    def best_ask(self) -> Optional[Decimal]:
        return next(iter(self.asks), None) if self.asks else None

    def spread(self) -> Optional[Decimal]:
        b, a = self.best_bid(), self.best_ask()
        return (a - b) if b and a else None
⚖️ Design Tradeoffs
Stock exchanges make explicit tradeoffs between latency, correctness, and availability. Unlike most web systems, correctness is more important than availability — it's better to halt trading than to produce incorrect trades.
⚙️ Matching Engine: Single-Threaded vs Multi-Threaded vs Distributed
✅ Single-Threaded per Symbol Group

No locking overhead. Cache-friendly (all data in L3 cache). Deterministic — same input always produces same output (essential for audit and replay). LMAX Exchange achieved 6M orders/sec on a single core. Multiple matching engines (one per symbol group) give horizontal scaling.

❌ Multi-Threaded (shared order book)

Requires locks or CAS for order book operations. Lock contention at high throughput is catastrophically slow — worse than single-threaded. Non-deterministic — impossible to reconstruct state from WAL if thread scheduling affects outcomes. Never used in production matching engines.

⚠️ Distributed Matching Engine

Partition order book across nodes. But: matching requires atomic access to the entire book for a symbol — you can't split one symbol's book. Distributed works at the symbol partition level (different nodes handle different symbols), not within a symbol. NASDAQ uses per-symbol matching engine threads.

📝 Durability: Synchronous WAL vs Async WAL vs No Persistence
✅ Synchronous WAL (fsync before ACK)

Order acknowledged only after written to WAL. On crash, replay WAL to reconstruct state. Zero data loss. This is the standard for real exchanges — regulatory requirement to have immutable audit trail. NVMe SSD fsync latency: ~100µs, acceptable.

⚠️ Async WAL (batch fsync every 1ms)

Acknowledgment before fsync. Sub-microsecond ACK latency. But: on crash, lose last ~1ms of orders. Acceptable for some HFT systems with guaranteed retransmission from brokers. Tradeoff: lower latency, small data loss window.

❌ In-memory only

Zero I/O overhead. But: any crash loses all outstanding orders — unacceptable for a regulated exchange. Regulatory requirement (SEC, FINRA) is to maintain a complete audit trail with nanosecond timestamps. In-memory only violates this.

📡 Market Data: UDP Multicast vs TCP Unicast vs Kafka
✅ UDP Multicast (Level 1)

Single packet to multicast group, network hardware delivers to all subscribers. 1M subscribers at essentially zero marginal cost. Sub-millisecond latency. Unreliable — handles gaps with sequence numbers + retransmission from replay server. NYSE, NASDAQ use this.

⚠️ TCP Unicast (Level 2 premium)

Reliable delivery guaranteed by TCP. Used for full order book depth (higher bandwidth). Suitable for hundreds/thousands of premium subscribers. Higher latency than UDP multicast. Cannot scale to millions of subscribers.

⚠️ Kafka (delayed data)

Great for analytics, backtesting, downstream processing. But adds ~1ms+ latency. Not suitable for live trading where milliseconds matter. Use Kafka for persisting market data for historical queries, not for real-time trading feeds.

🔢 Sequencer HA: Hot Standby vs Raft vs Accept Downtime
✅ Hot Standby + Deterministic Failover

Primary sequences all orders; standby receives all orders in parallel but doesn't publish. On primary failure, standby takes over at next sequence number. Failover: ~100ms. Zero added latency in normal operation. Most real exchanges use this.

⚠️ Raft Consensus

3-node Raft cluster. All writes confirmed by majority before acknowledgment. Strong consistency guarantee. But: +1-2ms latency for every order (Raft round trip). At 500K orders/sec, this extra latency significantly impacts HFT firms who co-locate at the exchange.

💡 Accept Planned Downtime

Exchanges operate during defined market hours. Brief maintenance windows (10-30s) during low-traffic periods are acceptable. NYSE opens at 9:30am ET — the 2 minutes before open can be used for failover/restart. Not all downtime scenarios are equal.
