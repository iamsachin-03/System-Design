Web-Scale Search Engine (Google)
Design a web-scale search engine — a crawler that discovers and indexes billions of pages, an inverted index that stores which pages contain which words, a ranking system using PageRank and hundreds of signals, a query processor that answers 100K queries/second in under 200ms, and a pipeline to keep the index fresh with continuous recrawling.
🔴 MUST DO 🟣 Hard Inverted Index PageRank Web Crawler MapReduce TF-IDF Index Sharding
🎯 Requirements
A search engine is a pipeline, not a single system. Crawl → parse → index → rank → serve. Each phase is independently scalable and has different latency requirements. The indexing pipeline can take days; the query path must answer in milliseconds. The key design insight: precompute everything you can during indexing so the query path is as cheap as possible.

FUNCTIONAL
Crawl the web: discover and download billions of web pages
Index pages: build an inverted index (word → list of pages)
Rank results: most relevant pages first (PageRank + content signals)
Query processing: given a query, return top-10 relevant pages
Continuous recrawl: keep index fresh as pages change
NON-FUNCTIONAL
50 billion pages indexed (Google scale)
100K search queries/sec, <200ms p99
Crawler: 1 billion pages/day throughput
Index freshness: major pages recrawled within hours
Crawl politeness: respect robots.txt, crawl-delay
💬 Phase 1 — The Crawler
PHASE 1 · WEB CRAWLER
IR
INTERVIEWER
Design a web crawler that can crawl 1 billion pages per day. What are the key design challenges?
ME
YOU
1 billion pages/day = ~11,600 pages/sec. At avg page size 200KB: 2.3 GB/sec bandwidth. The challenges:

URL Frontier (priority queue): Start with seed URLs. Parse each page for links, add new URLs to the frontier. Use a distributed priority queue (Kafka + Redis sorted set) prioritising: (1) high PageRank domains, (2) URLs not recently crawled, (3) URLs from sitemaps. Dedup with a Bloom filter to avoid re-crawling known URLs.

Politeness: Never hammer a single host. Per-domain rate limiting: at most 1 request every 5 seconds per domain. Store "next_crawl_time" per domain in Redis. Respect robots.txt — cache each site's robots.txt and check before every crawl.

Distributed crawling: Partition URL frontier by domain hash → each crawler worker owns a subset of domains. This naturally provides politeness (one worker per domain partition) and prevents duplicate crawling.

Deduplication: Web has massive duplicate content (mirrors, boilerplate). SimHash (locality-sensitive hash) detects near-duplicates efficiently — two documents with SimHash Hamming distance < 3 are near-duplicates.

Scale: 10,000 crawler workers × 1.16 pages/sec each = 11,600 pages/sec. Each worker is a simple HTTP client — stateless, horizontally scalable.
💡 Crawler traps: Infinite URL spaces (e.g., calendars with infinitely deep "next month" links), recursive redirects, very large pages. Mitigate with: max URL path depth, URL normalization, max page size limit (5MB), redirect loop detection.
📑 Phase 2 — Inverted Index Construction
PHASE 2 · INDEXING
IR
INTERVIEWER
You have 50 billion pages. How do you build an inverted index at this scale?
ME
YOU
An inverted index maps: word → [(doc_id, tf, positions), ...] — "tf" = term frequency (how often the word appears), positions = where in the document (needed for phrase queries).

Building with MapReduce:
Map phase: Input = (URL, page_content). For each word in the page (after tokenization, stemming, stop-word removal): emit (word, (doc_id, tf, positions)).
Reduce phase: For each word, collect all (doc_id, tf, positions) tuples, sort by PageRank score descending, write out the posting list.

Index sharding: 50B pages × avg 1000 unique words × 6 bytes per posting = ~300 TB of posting lists. Shard by word hash across 300+ index servers (term-partitioned index). A query for "machine learning" hits the shard containing "machine" and the shard containing "learning" in parallel, then intersects the result sets.

Tiered index: Sort documents by PageRank within each posting list. During query serving, scan only the top-K docs in each list (typically top 10,000 per term). Only extend the scan if not enough results found. This makes queries fast — most queries hit only the first tier of each posting list.
🏆 Phase 3 — Ranking with PageRank
PHASE 3 · RANKING
IR
INTERVIEWER
How does PageRank work and how do you compute it for 50 billion pages?
ME
YOU
PageRank models the web as a directed graph. A page's importance = how many important pages link to it. Formally:

PR(A) = (1-d) + d × Σ (PR(B) / outlinks(B)) for all B linking to A

Where d=0.85 (damping factor = probability of following a link vs jumping randomly). This is an iterative algorithm — start all pages with PR=1/N, iterate until convergence (typically 50-100 iterations).

Computing at scale: The web graph for 50B pages has trillions of edges. Too large for a single machine. Solution: sparse matrix-vector multiplication with MapReduce (or Spark). Each iteration: Map: for each page A with links to [B, C, D]: emit (B, PR(A)/3), (C, PR(A)/3), (D, PR(A)/3). Reduce: for each page, sum all incoming contributions.

Practical ranking: PageRank is one of ~200 ranking signals Google uses. Others include: TF-IDF relevance, page freshness (recency), click-through rate (CTR) from search logs, HTTPS (security signal), page load speed (Core Web Vitals), mobile-friendliness. Modern search uses ML models (neural ranking, BERT for semantic understanding) that combine hundreds of features.

// ── Posting List Entry ────────────────────────────────
public record Posting(
    long     docId,
    int      termFreq,
    double   pageRank,
    int[]    positions    // for phrase queries
) implements Comparable<Posting> {
    @Override public int compareTo(Posting o) {
        return Double.compare(o.pageRank(), this.pageRank()); // sorted descending by PageRank
    }
}

// ── In-memory Inverted Index ──────────────────────────
public class InvertedIndex {

    // term → sorted posting list (sorted by pageRank desc)
    private final Map<String, List<Posting>> index = new HashMap<>();
    private final Map<Long, Double>          idf   = new HashMap<>(); // term → IDF
    private long totalDocs;

    public void addDocument(long docId, String content, double pageRank) {
        Map<String,Integer> termFreqs = tokenize(content);
        termFreqs.forEach((term, tf) -> {
            List<Posting> list = index.computeIfAbsent(term, __ -> new ArrayList<>());
            list.add(new Posting(docId, tf, pageRank, new int[0]));
            list.sort(Comparator.naturalOrder());
        });
        totalDocs++;
    }

    /** Query: AND semantics — find documents containing ALL query terms. */
    public List<ScoredDoc> query(String queryText, int topK) {
        List<String> terms = tokenizeQuery(queryText);
        if (terms.isEmpty()) return List.of();

        // 1. Get posting list for each term (scan only top 10K)
        List<Set<Long>> sets = terms.stream()
            .map(t -> index.getOrDefault(t, List.of()).stream()
                .limit(10_000).map(Posting::docId).collect(Collectors.toSet()))
            .collect(Collectors.toList());

        // 2. AND intersection
        Set<Long> candidates = sets.get(0);
        for (int i = 1; i < sets.size(); i++) candidates.retainAll(sets.get(i));

        // 3. Score each candidate: TF-IDF × PageRank
        return candidates.stream()
            .map(docId -> new ScoredDoc(docId, score(docId, terms)))
            .sorted(Comparator.comparingDouble((ScoredDoc s) -> s.score()).reversed())
            .limit(topK)
            .collect(Collectors.toList());
    }

    private double score(long docId, List<String> terms) {
        double tfidf = terms.stream().mapToDouble(term -> {
            Posting p = getPosting(term, docId);
            if (p == null) return 0.0;
            double tf  = Math.log(1 + p.termFreq());
            double idf = Math.log(totalDocs / (double) index.getOrDefault(term, List.of()).size());
            return tf * idf;
        }).sum();
        double pageRank = getPageRank(docId);
        return tfidf * Math.log(1 + pageRank); // combine TF-IDF and PageRank
    }

    private Map<String,Integer> tokenize(String text) {
        return Arrays.stream(text.toLowerCase().split("\\W+"))
            .filter(w -> !STOP_WORDS.contains(w) && w.length() > 2)
            .collect(Collectors.toMap(w -> stem(w), w -> 1, Integer::sum));
    }
}

// ── MapReduce Mapper ──────────────────────────────────
public class IndexMapper extends Mapper<LongWritable, Text, Text, PostingWritable> {

    @Override
    protected void map(LongWritable key, Text value, Context context) throws IOException, InterruptedException {
        PageRecord page = PageRecord.parse(value.toString());
        Map<String,Integer> terms = tokenize(page.getContent());
        for (var entry : terms.entrySet()) {
            // Emit: key=term, value=(docId, tf, pageRank)
            context.write(new Text(entry.getKey()),
                new PostingWritable(page.getDocId(), entry.getValue(), page.getPageRank()));
        }
    }
}

// ── MapReduce Reducer ─────────────────────────────────
public class IndexReducer extends Reducer<Text, PostingWritable, Text, PostingListWritable> {

    @Override
    protected void reduce(Text term, Iterable<PostingWritable> postings, Context ctx) {
        List<PostingWritable> list = new ArrayList<>();
        postings.forEach(list::add);
        // Sort by PageRank descending so query can scan top results first
        list.sort((a, b) -> Double.compare(b.pageRank, a.pageRank));
        ctx.write(term, new PostingListWritable(list));
    }
}

# Search Engine — Python: Inverted Index + Crawler

import math, re, heapq, hashlib
from collections import defaultdict, Counter
from dataclasses import dataclass, field
from typing import Dict, List, Tuple, Set

STOP_WORDS = {"the", "a", "an", "is", "in", "on", "at", "to", "for", "of", "and"}


@dataclass
class Posting:
    doc_id:    int
    term_freq: int
    page_rank: float
    positions: List[int] = field(default_factory=list)


class InvertedIndex:
    def __init__(self):
        self.index:      Dict[str, List[Posting]] = defaultdict(list)
        self.total_docs: int                        = 0
        self.page_ranks: Dict[int, float]           = {}

    def add_document(self, doc_id: int, content: str, page_rank: float = 1.0):
        self.page_ranks[doc_id] = page_rank
        terms = self._tokenize(content)
        tf    = Counter(terms)
        for term, freq in tf.items():
            positions = [i for i, t in enumerate(terms) if t == term]
            self.index[term].append(Posting(doc_id, freq, page_rank, positions))
        self.total_docs += 1
        # Keep posting lists sorted by pagerank descending
        for term in tf:
            self.index[term].sort(key=lambda p: -p.page_rank)

    def query(self, query_text: str, top_k: int = 10) -> List[Tuple[int, float]]:
        terms = self._tokenize(query_text)
        if not terms: return []

        # AND semantics: find docs containing all terms (scan top 10K per term)
        sets = [
            {p.doc_id for p in self.index.get(t, [])[:10_000]}
            for t in terms
        ]
        candidates = sets[0].intersection(*sets[1:]) if len(sets) > 1 else sets[0]

        # Score: TF-IDF × log(1 + PageRank)
        scored = [(doc_id, self._score(doc_id, terms)) for doc_id in candidates]
        return sorted(scored, key=lambda x: -x[1])[:top_k]

    def _score(self, doc_id: int, terms: List[str]) -> float:
        tfidf = 0.0
        for term in terms:
            posting = next((p for p in self.index.get(term, []) if p.doc_id == doc_id), None)
            if not posting: continue
            tf  = math.log1p(posting.term_freq)
            df  = len(self.index.get(term, []))
            idf = math.log(self.total_docs / max(df, 1))
            tfidf += tf * idf
        pr = self.page_ranks.get(doc_id, 1.0)
        return tfidf * math.log1p(pr)

    def _tokenize(self, text: str) -> List[str]:
        words = re.findall(r'\b[a-zA-Z]{3,}\b', text.lower())
        return [w for w in words if w not in STOP_WORDS]


# ── PageRank (iterative) ──────────────────────────────
def compute_pagerank(graph: Dict[int, List[int]], d: float = 0.85, iters: int = 50) -> Dict[int, float]:
    """Iterative PageRank: graph[page] = [outlink_pages]."""
    N  = len(graph)
    pr = {node: 1.0 / N for node in graph}

    for _ in range(iters):
        new_pr: Dict[int, float] = {}
        for node in graph:
            # Random jump component
            new_pr[node] = (1.0 - d) / N
        # Link contribution
        for src, outlinks in graph.items():
            if not outlinks: continue
            share = pr[src] / len(outlinks)
            for dest in outlinks:
                new_pr[dest] = new_pr.get(dest, (1-d)/N) + d * share
        pr = new_pr

    return pr


# ── Minimal crawler skeleton ──────────────────────────
import heapq, time

class URLFrontier:
    def __init__(self):
        self.queue: list       = []
        self.seen:  Set[str]   = set()
        self.next_crawl: Dict[str, float] = {}

    def push(self, url: str, priority: float = 1.0):
        url_hash = hashlib.md5(url.encode()).hexdigest()
        if url_hash in self.seen: return  # Bloom filter substitute
        self.seen.add(url_hash)
        heapq.heappush(self.queue, (-priority, time.time(), url))

    def pop(self) -> str:
        _, _, url = heapq.heappop(self.queue)
        domain = url.split("/")[2]
        # Politeness: backoff if we crawled this domain recently
        next_ok = self.next_crawl.get(domain, 0)
        if time.time() < next_ok:
            time.sleep(next_ok - time.time())
        self.next_crawl[domain] = time.time() + 5  # 5s per-domain delay
        return url
⚖️ Tradeoffs
📊 Index Partitioning: Term-Partitioned vs Doc-Partitioned
✅ Term-Partitioned (chosen)

Each server owns a subset of terms. A query for "machine learning" hits 2 servers. Advantage: complete posting list for a term is on one server — efficient sorted intersection. Disadvantage: query fan-out scales with query length. A 10-term query hits 10 servers. Used by Google's original architecture.

Doc-Partitioned

Each server owns a subset of documents. All terms for a document are co-located on the same server. A query for "machine learning" hits ALL servers (each checks if its documents match). Each server returns top-K local results. Aggregator picks global top-K. Advantage: no large posting lists to transfer. Disadvantage: query hits all servers every time. Used by Bing's architecture for some components.

Hybrid

Two-level hierarchy. Global index: top-K documents per term (serves 99% of queries). Local index: full posting list per shard for rare/long-tail queries. Combines fast common-case with correct rare-case handling. Best for query latency but most complex to implement and maintain.

🔄 Index Freshness: Full Rebuild vs Incremental
Full Rebuild

Rebuild the entire index from scratch via MapReduce. Clean, no stale data, correct PageRank. Problem: takes days for 50B pages. Unacceptable for news (needs hourly freshness). Practical only for the base "deep web" index.

✅ Tiered Crawl

Separate the index into tiers: (1) Real-time: news sites recrawled every hour, serving freshest content; (2) Recent: high-traffic sites every day; (3) Static: slow-changing sites weekly. Query blends results from all tiers. This is what Google does — separate freshness and quality signals.

✅ Incremental Index

Stream changed documents from the crawler → delta indexer → merge into existing index. Elasticsearch uses this pattern (segment merging). Allows near-realtime indexing (<1min) for important pages. Complexity: must handle deletions/updates correctly. Required for competitive freshness.
