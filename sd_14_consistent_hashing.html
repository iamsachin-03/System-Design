🔴 The Core Problem — Why Modulo Hashing Fails
Interview context: Consistent hashing is asked both as a standalone algorithm question ("implement it") and as the solution to the distributed cache key routing problem. Know the motivation, the algorithm, and the implementation cold.

❌ NAIVE: MODULO HASHING
node = hash(key) % N
N = number of nodes

With N=3 nodes:
"user:123" → 456 % 3 = 0 → Node A
"session:x" → 789 % 3 = 0 → Node A
"order:99" → 100 % 3 = 1 → Node B
💥 Add node D (N=4):
"user:123" → 456 % 4 = 0 → Node A ✓ same
"session:x" → 789 % 4 = 1 → Node B ✗ moved!
"order:99" → 100 % 4 = 0 → Node A ✗ moved!

~75% of all keys remapped — entire cache invalidated simultaneously → thundering herd → DB dies
✅ CONSISTENT HASHING
Both keys and nodes map to a circle (0..2³²)
Key routed to the next node clockwise

With N=3 nodes on ring:
"user:123" → Node A (clockwise next)
"session:x" → Node B (clockwise next)
"order:99" → Node C (clockwise next)
✅ Add node D (between B and C):
"user:123" → Node A ✓ unaffected
"session:x" → Node B ✓ unaffected
"order:99" → Node D ← only keys in D's new arc moved

Only ~1/N keys remapped = ~25% with 4 nodes. Everything else stays.
📐 Key Properties
GUARANTEES
1
Minimal remapping: Adding or removing a node remaps only ~1/N keys, not all N keys
2
Uniform distribution: With virtual nodes, each physical node handles ~1/N of all keys
3
No central coordinator: Every client independently computes the same mapping
4
Monotonic: A key always maps to the same node unless the ring changes

💍 The Hash Ring — Step by Step

The ring is a conceptual circle from 0 to 2³²−1. Every node and key gets hashed onto this ring. A key is served by the first node encountered going clockwise.

🔍 Lookup Algorithm — O(log N)
// Ring stored as sorted map: hash_position → node
// ceilingEntry() = first entry with key >= hash = "clockwise next"

TreeMap<Long, Node> ring = new TreeMap<>();

Node getNode(String key) {
    long hash = murmur3(key);
    // Find first ring position >= hash (clockwise)
    Map.Entry<Long, Node> entry = ring.ceilingEntry(hash);
    // Wrap: if hash is past last node, use first node
    if (entry == null) entry = ring.firstEntry();
    return entry.getValue();
}
// O(log N) where N = total virtual node count
// 150 vnodes × 10 nodes = 1500 ring points → log(1500) ≈ 11 comparisons
🧩 The Problem with Basic Consistent Hashing
Basic consistent hashing has two problems that virtual nodes fix: uneven load distribution, and poor load spreading when a node fails.

❌ PROBLEM 1: UNEVEN DISTRIBUTION

With 4 nodes placed randomly on the ring, arcs are not equal. One node might own 40% of the ring, another only 5%. That node receives 8× the traffic and becomes a hotspot.

Node A: 50% of ring ← overloaded!
Node B: 9% of ring ← underused
Node C: 29% of ring
Node D: 12% of ring
✅ VIRTUAL NODES FIX THIS

Each physical node placed on ring K=150 times using different hash seeds: hash("Node_A#0"), hash("Node_A#1")... The 150 small arcs interleave and converge to ~25% each.

Node A: 25.1% ← near-perfect
Node B: 24.9% ← near-perfect
Node C: 25.0% ← near-perfect
Node D: 25.0% ← near-perfect
🧮 Virtual Node Math — Choosing K
Physical nodes: N = 10 | Virtual nodes per physical: K Expected key share per node: 1/N = 10% Standard deviation in share vs K: K=1 → σ ≈ ±50% (some nodes get 30%, others 2% — broken) K=10 → σ ≈ ±16% (still noticeable imbalance) K=100 → σ ≈ ±5% (near-uniform, acceptable) K=150 → σ ≈ ±4% ← Cassandra default K=256 → σ ≈ ±3% (diminishing returns vs memory) Memory cost (K=150, 10 nodes): 150 × 10 = 1,500 ring entries × 64 bytes = ~96 KB — trivial → K=150 is the sweet spot: near-uniform distribution at negligible cost
⚖️ Heterogeneous Clusters — Weighted Virtual Nodes
Virtual nodes unlock weighted distribution: assign more vnodes to more powerful machines. A 2× larger server gets 2× the vnodes → naturally receives 2× the traffic. No manual configuration needed.
Cluster with different capacities: Server A (8GB RAM): 100 vnodes → 25% traffic Server B (16GB RAM): 200 vnodes → 50% traffic Server C (8GB RAM): 100 vnodes → 25% traffic Total: 400 vnodes → weight = (capacity / min_capacity) × base_vnodes
🔀 Node Addition — Exactly Which Keys Move
➕ ADDING NODE E
1
E's 150 virtual nodes computed: hash("E#0")...hash("E#149"), inserted into ring
2
For each of E's 150 positions: keys between (E's predecessor) and (E's position) previously went to E's clockwise successor — they now go to E
3
Only keys in E's new arcs are affected: ~1/N ≈ 20% (1 of 5 nodes)
4
For a cache: these become cache misses, rebuilt from DB naturally. For a DB: data streams from successor to E before E goes live.
5
All other keys — ~80% — are completely unaffected
➖ REMOVING NODE B (fails)
1
B's 150 virtual node positions removed from ring
2
Without vnodes: ALL of B's keys dump onto one successor node — it gets 2× load, may fail too (cascade!)
3
With vnodes: B's 150 ring positions each have different successors, spread across many physical nodes
4
B's load spreads across ~N-1 nodes proportionally. Load increase per surviving node: ~1.25%
5
Virtual nodes are the critical difference between graceful degradation and cascading failure

💬 Interview Simulation
IR
INTERVIEWER
What is consistent hashing and why do we need it?
ME
YOU
Consistent hashing solves the key routing problem in distributed systems: given a key, which node stores or serves it?

The naive solution is node = hash(key) % N. This works perfectly with a fixed N, but the moment N changes — node added or removed — virtually every key remaps. For a cache with 10M entries when N goes from 10 to 11, ~91% of cache entries suddenly miss simultaneously. This thundering herd of cache misses can kill the backend database.

Consistent hashing solves this by placing both keys and nodes on a circular hash ring. A key is assigned to the first node clockwise from its position. When a node is added, only the keys in its new arc are remapped (~1/N). Everything else is unaffected.
IR
INTERVIEWER
Walk me through adding a node. Exactly which keys move?
ME
YOU
Say we have nodes A, B, C at positions 100, 200, 300 on the ring (0–360).

- Keys in (300..100] → Node A
- Keys in (100..200] → Node B
- Keys in (200..300] → Node C

We add Node D at position 150 (between A and B):
- Keys in (100..150] were going to Node B. Now they route to D.
- Everything else unchanged.

The only movement: keys between D's predecessor (100) and D's position (150). That's 50/360 ≈ 14% of ring keys — roughly 1/N.
💡 This precise answer is what interviewers want. Not just "1/N keys move" but which specific keys — those in the arc between D's predecessor and D's own position, previously belonging to D's clockwise successor.
IR
INTERVIEWER
Without virtual nodes, what happens when a node fails?
ME
YOU
Without virtual nodes, each node occupies exactly one position. When Node B fails, all of B's keys route to its one clockwise successor — Node C. If B was handling 25% of load, C jumps from 25% to 50% instantly. C may become overloaded and fail too — cascading failure.

With K=150 virtual nodes: B's 150 ring positions each have different successors scattered across all other physical nodes. B's load distributes proportionally across all surviving nodes. Load increase per surviving node: from 25% to ~33% — manageable, not catastrophic.
IR
INTERVIEWER
Implement consistent hashing. What data structure?
ME
YOU
TreeMap<Long, Node> — keys are hash ring positions, values are physical nodes. TreeMap keeps them sorted, giving O(log N) for all operations.

Add node: For each of K virtual nodes, compute hash(nodeId + "#" + i), insert into TreeMap.

Remove node: Reverse — remove all K positions.

Lookup key: Compute hash(key). Call TreeMap.ceilingEntry(hash) to get first position ≥ hash (clockwise). If null, wrap around with TreeMap.firstEntry().

Thread safety: ReadWriteLock — many concurrent reads during normal operation, exclusive write only when ring topology changes (rare). Or use ConcurrentSkipListMap for lock-free reads.
IR
INTERVIEWER
What hash function would you use and why?
ME
YOU
We need uniform distribution, determinism, and speed. We do NOT need cryptographic security.

MurmurHash3 — my first choice. Non-cryptographic, 3–5× faster than MD5, excellent avalanche effect, no known clustering patterns. Used by Cassandra, Redis, Elasticsearch. Available as mmh3 in Python.

MD5 — acceptable for legacy systems (libmemcached uses it). Well-studied distribution but slower. Overkill cryptographically.

Never use Java's default hashCode() — too many collisions, not stable across JVM restarts, and NOT deterministic across JVM versions. Keys would route to different nodes after a restart.

// ══════════════════════════════════════════════════════
// Consistent Hashing — Production Java
// ══════════════════════════════════════════════════════

public class ConsistentHashRing<T> {

    // Sorted ring: position (long) → physical node
    // ReadWriteLock: concurrent reads, exclusive writes on topology change
    private final TreeMap<Long, T>   ring = new TreeMap<>();
    private final ReadWriteLock      lock = new ReentrantReadWriteLock();
    private final int                  virtualNodesPerNode;

    public ConsistentHashRing(int virtualNodesPerNode) {
        this.virtualNodesPerNode = virtualNodesPerNode; // 150 recommended
    }

    /** Add a physical node with K virtual positions on the ring. */
    public void addNode(String nodeId, T node) {
        lock.writeLock().lock();
        try {
            for (int i = 0; i < virtualNodesPerNode; i++) {
                // Each virtual node uses a different hash seed via nodeId#i
                long pos = hash(nodeId + "#" + i);
                ring.put(pos, node);
            }
        } finally { lock.writeLock().unlock(); }
    }

    /** Remove all virtual positions for this node. */
    public void removeNode(String nodeId) {
        lock.writeLock().lock();
        try {
            for (int i = 0; i < virtualNodesPerNode; i++) {
                ring.remove(hash(nodeId + "#" + i));
            }
        } finally { lock.writeLock().unlock(); }
    }

    /**
     * Route a key to its responsible node.
     * O(log N) — binary search on sorted ring.
     *
     * 1. Hash the key to a ring position
     * 2. ceilingEntry: first position >= hash (clockwise)
     * 3. If none, wrap around to ring's first entry
     */
    public T getNode(String key) {
        lock.readLock().lock();
        try {
            if (ring.isEmpty()) return null;
            Map.Entry<Long, T> entry = ring.ceilingEntry(hash(key));
            if (entry == null) entry = ring.firstEntry(); // wrap
            return entry.getValue();
        } finally { lock.readLock().unlock(); }
    }

    /**
     * Get N distinct physical nodes for a key (for replication).
     * Walk ring clockwise, collecting unique physical nodes.
     */
    public List<T> getNodes(String key, int count) {
        lock.readLock().lock();
        try {
            if (ring.isEmpty()) return List.of();
            Set<T> seen   = new LinkedHashSet<>();
            long   start  = hash(key);

            // Iterate from ceiling, wrap around via concat of tail+head
            NavigableMap<Long, T> tail = ring.tailMap(start, true);
            for (T node : Iterables.concat(tail.values(), ring.values())) {
                seen.add(node);
                if (seen.size() == count) break;
            }
            return new ArrayList<>(seen);
        } finally { lock.readLock().unlock(); }
    }

    /**
     * MurmurHash3-inspired mixing — fast, excellent distribution.
     * For production use the real mmh3 library or use MD5.
     */
    private long hash(String key) {
        byte[] data = key.getBytes(StandardCharsets.UTF_8);
        long h = 0xdeadbeefcafeL;
        for (byte b : data) {
            h ^= (b & 0xFFL);
            h *= 0xc4ceb9fe1a85ec53L;
            h ^= (h >>> 33);
        }
        h *= 0xff51afd7ed558ccdL;
        h ^= (h >>> 33);
        return h;
    }
}

# Consistent Hashing — Python (uses sortedcontainers)
# ══════════════════════════════════════════════════════

import hashlib
from sortedcontainers import SortedList
from typing import TypeVar, Generic, Optional, List
from threading import RLock

T = TypeVar('T')


class ConsistentHashRing(Generic[T]):
    """
    Production-grade consistent hash ring with virtual nodes.
    K=150 virtual nodes per physical node gives ±4% load variance.
    Thread-safe via RLock.
    """

    def __init__(self, virtual_nodes: int = 150):
        self.virtual_nodes = virtual_nodes
        # Sorted list of (position, node) tuples
        self._ring: List[tuple] = SortedList(key=lambda x: x[0])
        self._node_positions: dict[str, list[int]] = {}
        self._lock = RLock()

    def add_node(self, node_id: str, node: T) -> None:
        """Place K virtual nodes for this physical node on the ring."""
        with self._lock:
            positions = []
            for i in range(self.virtual_nodes):
                pos = self._hash(f"{node_id}#{i}")
                self._ring.add((pos, node_id, node))
                positions.append(pos)
            self._node_positions[node_id] = positions

    def remove_node(self, node_id: str) -> None:
        """Remove all virtual positions for this node."""
        with self._lock:
            positions = self._node_positions.pop(node_id, [])
            # Remove entries matching this node_id
            self._ring = SortedList(
                [e for e in self._ring if e[1] != node_id],
                key=lambda x: x[0])

    def get_node(self, key: str) -> Optional[T]:
        """
        Route key to responsible node. O(log N).
        1. Hash key to ring position
        2. Find first ring entry >= position (clockwise)
        3. If none, wrap to first entry
        """
        with self._lock:
            if not self._ring:
                return None

            key_pos = self._hash(key)

            # Binary search for first position >= key_pos
            idx = self._ring.bisect_left((key_pos,))

            if idx == len(self._ring):
                idx = 0  # wrap around

            return self._ring[idx][2]  # (pos, node_id, node) → node

    def get_nodes(self, key: str, count: int) -> List[T]:
        """Get N distinct physical nodes (for replication)."""
        with self._lock:
            if not self._ring:
                return []

            key_pos = self._hash(key)
            idx     = self._ring.bisect_left((key_pos,)) % len(self._ring)
            seen    = []
            seen_ids = set()

            for i in range(len(self._ring)):
                entry   = self._ring[(idx + i) % len(self._ring)]
                node_id = entry[1]
                node    = entry[2]
                if node_id not in seen_ids:
                    seen.append(node)
                    seen_ids.add(node_id)
                    if len(seen) == count:
                        break
            return seen

    def _hash(self, key: str) -> int:
        """MD5-based 64-bit hash. Swap for mmh3.hash(key, signed=False) in prod."""
        return int.from_bytes(
            hashlib.md5(key.encode()).digest()[:8], 'big')

    def __len__(self) -> int: return len(self._ring)
    def node_count(self) -> int: return len(self._node_positions)

🌍 How Real Systems Use Consistent Hashing
🐍 Apache Cassandra — Token Ring (256 vnodes)

Cassandra uses a token ring with 256 virtual nodes per physical node. Each node owns 256 non-contiguous token ranges. Adding a node: Cassandra automatically picks 256 token ranges from across the cluster — data migrates via streaming. The nodetool ring command shows the full ring. Uses Murmur3 partitioner by default. The original Dynamo paper's token ring concept, operationalised.

⚡ Redis Cluster — 16,384 Hash Slots

Redis Cluster uses 16,384 pre-defined hash slots (0–16,383) instead of a continuous ring. A key maps via CRC16(key) % 16384. Slots are distributed across nodes. Adding a node = move some slots from existing nodes. This is consistent hashing with discrete numbered slots — more explicit and easier to reason about and administer than a continuous ring.

🛒 Amazon DynamoDB — The Dynamo Paper (2007)

DynamoDB's foundational Dynamo paper popularised consistent hashing with virtual nodes in production. Key insights: virtual nodes for heterogeneous hardware, N replicas at the next N clockwise nodes (fault tolerance), and transparent node add/remove for applications. Also introduced vector clocks, sloppy quorum, and hinted handoff in the same paper — all still used today.

🌐 CDN & Load Balancers

Akamai published the original consistent hashing paper (Karger et al., 1997) to distribute web content across thousands of cache servers. Nginx (hash $request_uri consistent), HAProxy (balance uri), and AWS ALB all support consistent hash load balancing for sticky sessions — route all requests from the same user to the same backend without a centralised session store.

⚖️ Tradeoffs & Variants
Consistent hashing is not magic. It solves key routing gracefully but virtual node count, hash function choice, and hot key handling all require careful thought.
🔢 Virtual Node Count K: 1 vs 150 vs 256+
✅ K=150 (recommended)

Standard production. ±4% load variance. 96 KB memory for 10 nodes. log(1500) ≈ 11 comparisons per lookup. Cassandra's default. Good balance of uniformity, memory, and speed. Diminishing returns beyond 150.

❌ K=1 (basic ring)

One position per node. High load variance — one node may own 40% of ring. When a node fails, ALL its keys dump onto ONE successor (cascading failure risk). Never use in production. Academic demonstration only.

⚠️ K=256+ (Redis: 16,384 slots)

Near-perfect uniformity. But topology changes become complex. Redis solves this with discrete numbered slots (not a hash ring) — more explicit but less flexible. Justified for very large clusters where uniformity matters above all else.

🔀 Consistent Hashing vs Rendezvous Hashing vs Jump Hash
✅ Consistent Hashing (ring)

Industry standard. O(log N) lookup. Trivial add/remove. Virtual nodes for even distribution. Supports weighted heterogeneous clusters. Best for dynamic clusters where nodes come and go unpredictably.

⚠️ Rendezvous Hashing (HRW)

Score each node with hash(key, node), pick highest score. Minimal remapping. But: O(N) per lookup (check all nodes). Only practical for N ≤ 20. Simple to implement, no virtual nodes needed — naturally uniform.

💡 Jump Consistent Hash

Google 2014. O(log N), perfect uniformity, 5 lines. But: only supports adding buckets (can't remove arbitrary ones). Great for batch processing / MapReduce sharding. Terrible for dynamic caches where nodes fail unpredictably.

🔥 Hot Key Problem — Consistent Hashing Doesn't Solve It
✅ Accept the limitation

CH distributes keys evenly, not request load. If one key gets 1M req/sec, all go to one node — CH can't help. Solutions are orthogonal: client-side L1 cache (zero network calls for hot keys), key replication, or CDN caching. CH is necessary but not sufficient.

⚠️ Key replication

Store K copies: hot_key_0, hot_key_1, ..., hot_key_K-1. Reads pick a random copy (spread load). Writes update all K. Application logic, not CH. Each copy hashes to a different node via CH. Turns 1M req/sec on one key into 200K req/sec on 5 nodes.

💡 Key sharding

Split a counter: counter:user:123:shard_0...shard_N. Writes increment a random shard. Reads aggregate across shards. Consistent hashing routes each shard to a different node. Turns one hot-write key into N cool-write keys.

🏷 Hash Function: MD5 vs MurmurHash3 vs SHA-256
✅ MurmurHash3

Non-cryptographic. 3–5× faster than MD5. Excellent avalanche effect, no clustering. Used by Cassandra, Redis, Elasticsearch. The default choice for new systems. Python: pip install mmh3. Java: guava's Hashing.murmur3_128().

⚠️ MD5 (legacy)

128-bit, well-studied distribution. Used by libmemcached. Slower than MurmurHash3. Cryptographic (overkill for routing). Acceptable for backward compatibility — existing systems using MD5 should not change (would remap all keys).

❌ SHA-256 or Java hashCode()

SHA-256 is 10–50× slower — cryptographic security you don't need. Java hashCode() is not deterministic across JVM restarts or versions — keys would route to different nodes after a restart. Never use either for distributed routing.
