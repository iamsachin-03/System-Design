Distributed File System (GFS / HDFS)
Design a distributed file system like Google File System or HDFS — storing exabytes across thousands of commodity machines, with a single master/namenode tracking all file-to-chunk mappings, chunk servers streaming data directly to clients, a 3-way replication pipeline for durability, and automatic recovery when machines fail daily at this scale.
🔴 MUST DO 🟣 Hard (Staff/Principal) Namenode Chunk Replication Heartbeat Lease Erasure Coding SPOF Mitigation
🎯 Requirements
GFS (2003) was Google's foundational insight: at scale, machine failure is the norm, not the exception. Design the system to expect and tolerate failure. Commodity hardware, 3-way replication, automatic recovery — not expensive SAN arrays. This paper changed how everyone builds distributed storage.

FUNCTIONAL
Store and retrieve large files (GBs to TBs each)
Files split into fixed-size chunks (64MB in GFS)
Each chunk replicated 3× across different rack/nodes
Support: create, read, append, delete (no random write)
Automatic re-replication when chunk falls below 3 replicas
Namespace management (directory tree, permissions)
NON-FUNCTIONAL
Exabyte-scale storage across 10,000+ nodes
Optimised for large sequential reads (MapReduce workloads)
Throughput over latency — 100s of MB/s read bandwidth
Durability: survive simultaneous failure of any 2 replicas
Automatic recovery from node failures (1-3 nodes/day at scale)
💬 Phase 1 — Master / Namenode Design
PHASE 1 · MASTER NODE
IR
INTERVIEWER
The master tracks all file-to-chunk mappings. It's a single node. Isn't that a SPOF?
ME
YOU
Yes — and GFS acknowledges it. The master is intentionally simple: it holds all metadata in memory (fast), writes a journal (operation log) to disk and to a shadow master for durability. If the master fails, a shadow master takes over (manual or automated failover in GFS; HDFS v2+ uses Zookeeper-based automatic HA).

Why is it acceptable?
  • Metadata is small: 10 billion chunks × 64 bytes = 640 GB — fits in RAM of a single powerful server.
  • Master is never on the data path: clients contact the master only for chunk locations, then stream data directly to/from chunk servers. So master failure doesn't immediately interrupt in-progress data transfers.
  • Master is stateless beyond the journal: recovery = replay the journal from the last checkpoint. Fast recovery (<60 seconds).
  • HDFS addressed this with NameNode HA: two namenodes in active-standby, Zookeeper for leader election, journal nodes for shared edit log.
Master responsibilities: File namespace (tree), chunk-to-server mapping (rebuilt from heartbeats — not persisted!), chunk placement, garbage collection, re-replication. Chunk server locations are NOT persisted — on startup, master polls all chunk servers for their chunk lists (heartbeat-based rebuild).
💡 Key insight: Chunk-to-server mapping is rebuilt from heartbeats, not persisted. This is elegant — it avoids the master and chunk servers getting out of sync. The chunk servers are the ground truth of what they hold.
📝 Phase 2 — Write Path & Lease
PHASE 2 · WRITE PATH
IR
INTERVIEWER
Client wants to write data to a file. Walk me through the full write path.
ME
YOU
1. Client asks master: "I want to write chunk 42 of file /logs/access.log." Master returns: primary chunk server (has lease) + 2 secondary replicas. The lease is given to one chunk server — the primary — for 60 seconds. The primary serialises all mutations to this chunk during its lease period.

2. Client pipelines data to all 3 replicas: Client pushes data to the closest replica (by network topology). That replica forwards to the next, which forwards to the third. This pipeline uses the full bandwidth of each link — all three transfers happen in parallel. 64MB chunk: sent once from client, forwarded twice internally = only 1 RTT of latency for the chain (once all bytes arrive at replica 1, it starts forwarding to replica 2 simultaneously).

3. Client sends WRITE RPC to primary: After all replicas acknowledge data receipt (buffered, not yet applied), client tells primary to apply the write. Primary applies it at a specific offset, assigns a serial number, sends WRITE to secondaries with the same serial number.

4. Secondaries acknowledge to primary, primary acks client. If any secondary fails: client retries from step 2. Chunks may have diverged partially — handled at file close with garbage collection.

Why separate data flow from control flow? Data (the actual bytes) flows client → replica → replica (pipeline). Control (where to write) flows client → primary → secondaries. This decoupling lets us optimise data transfer independently of coordination.
💥 Phase 3 — Failure Recovery
PHASE 3 · FAILURE & RECOVERY
IR
INTERVIEWER
3 nodes per day fail out of 10,000. How does the system detect and recover?
ME
YOU
Detection — heartbeats: Each chunk server sends a heartbeat to the master every 3 seconds. Heartbeat includes: list of chunks it holds (used to rebuild chunk-to-server mapping on restart). If master receives no heartbeat for 30s: mark that chunk server as dead.

Recovery — re-replication: Master scans all chunks that have a replica on the dead server. For each chunk now with <3 replicas: add to re-replication queue. Priority: chunks with only 1 replica first (highest risk). Master instructs a chunk server with a copy to send a replica to another chunk server. Chunk servers copy directly — master only orchestrates, doesn't transfer data.

Placement strategy: Replicas placed across different racks (rack-aware placement). One replica per unique rack. Ensures even a rack power failure (kills all nodes in a rack) can't lose data.

Re-replication bandwidth: 3 node failures/day × 10TB per node × 3 replicas to rebuild / 86400 seconds = ~1 GB/sec sustained re-replication traffic. Each failed node's chunks spread across all surviving nodes — parallel recovery finishes in minutes not hours.
🏗 Architecture Deep Dive
MASTER / NAMENODE
1
In-memory namespace tree: directory hierarchy, file metadata, access control. All fits in RAM (640 GB for 10B chunks).
2
Chunk mapping table: file → list of (chunk_handle, chunk_version, [server1, server2, server3]). Chunk-to-server NOT persisted — rebuilt from heartbeats.
3
Operation log (journal): All namespace mutations written to log before taking effect. Checkpointed periodically. Recovery = last checkpoint + replay log.
4
Shadow master: Replays journal asynchronously. Can serve read-only metadata queries if primary master fails. Failover: promote shadow to primary.
5
Background tasks: garbage collect orphaned chunks, re-balance chunk placement, re-replicate under-replicated chunks, load balance.
CHUNK SERVERS / DATANODES
1
Store chunks as flat files: Each chunk is a local file on the chunk server's disk. Chunk handle = filename. No RAID — replication provides durability.
2
Heartbeat to master every 3s: Reports chunk list, disk usage, health. Master uses this to rebuild chunk-to-server mapping.
3
Direct client streaming: After master tells client which servers hold the chunk, client connects directly to chunk server. Master is out of the data path.
4
Checksum each block: 64KB blocks within a chunk have CRC32 checksums. Verified on every read. Silent corruption detected and reported to master for re-replication.
5
Lease holder (primary): One chunk server per chunk holds the lease. Serialises concurrent writes. Lease expires after 60s or is explicitly released.

// ── Master: namespace + chunk location tracking ──────
public class Master {
    // File namespace: path → list of (chunkHandle, version)
    private final Map<String, List<ChunkInfo>> namespace = new ConcurrentHashMap<>();
    // Chunk location: chunkHandle → [server1, server2, server3]
    // NOTE: rebuilt from heartbeats, NOT persisted to disk
    private final Map<Long, List<String>> chunkLocations = new ConcurrentHashMap<>();
    // Lease holder: chunkHandle → (primary_server, expiry_time)
    private final Map<Long, Lease> leases = new ConcurrentHashMap<>();

    /**
     * Client lookup: return chunk servers for a file range.
     * Client can then contact chunk servers directly.
     */
    public ChunkLocationResponse getChunkLocations(String filePath, int chunkIndex) {
        List<ChunkInfo> chunks = namespace.get(filePath);
        if (chunks == null || chunkIndex >= chunks.size())
            throw new FileNotFoundException(filePath);
        ChunkInfo chunk   = chunks.get(chunkIndex);
        List<String> servers = chunkLocations.get(chunk.handle());
        return new ChunkLocationResponse(chunk.handle(), chunk.version(), servers);
    }

    /** Grant or renew a lease for writing a chunk (client asks before write). */
    public LeaseResponse grantLease(long chunkHandle) {
        List<String> servers = chunkLocations.get(chunkHandle);
        Lease lease = new Lease(servers.get(0), System.currentTimeMillis() + 60_000);
        leases.put(chunkHandle, lease);
        return new LeaseResponse(lease.primary(), servers.subList(1, servers.size()));
    }

    /** Heartbeat from chunk server — updates chunk location index. */
    public void heartbeat(String serverId, List<Long> chunkHandles) {
        chunkHandles.forEach(handle ->
            chunkLocations.computeIfAbsent(handle, k -> new ArrayList<>())
                          .add(serverId));
        checkUnderReplicated(chunkHandles);
    }

    /** Find chunks with <3 replicas and trigger re-replication. */
    private void checkUnderReplicated(List<Long> handles) {
        handles.stream()
            .filter(h -> chunkLocations.getOrDefault(h, List.of()).size() < 3)
            .forEach(this::scheduleReplication);
    }
}

// ── GFS Client ────────────────────────────────────────
public class GFSClient {
    private final Master        master;
    private final Map<String, ChunkLocationResponse> locationCache = new HashMap<>();

    public byte[] read(String file, long offset, int length) {
        int chunkIndex = (int) (offset / GFSConstants.CHUNK_SIZE);
        long chunkOffset = offset % GFSConstants.CHUNK_SIZE;

        // 1. Ask master for chunk location (cache for 5 minutes)
        ChunkLocationResponse loc = master.getChunkLocations(file, chunkIndex);

        // 2. Pick closest replica (by network distance / RTT)
        String closestServer = pickClosestReplica(loc.servers());

        // 3. Stream data directly from chunk server — master not involved
        return chunkServerClient.read(closestServer, loc.handle(), chunkOffset, length);
    }

    public void write(String file, long offset, byte[] data) {
        LeaseResponse lease = master.grantLease(getChunkHandle(file, offset));

        // Push data to all replicas via pipeline: primary → sec1 → sec2
        chunkServerClient.pushData(lease.primary(), data);  // cascades to secondaries

        // Send WRITE to primary; it serialises and acks secondaries
        chunkServerClient.write(lease.primary(), getChunkHandle(file, offset), offset % GFSConstants.CHUNK_SIZE);
    }
}

# DFS — Python: Chunk Server + Heartbeat + Checksum

import os, hashlib, time, threading, struct
from pathlib import Path
from dataclasses import dataclass
from typing import List

CHUNK_SIZE   = 64 * 1024 * 1024  # 64 MB
BLOCK_SIZE   = 64 * 1024          # 64 KB (for checksumming)


class ChunkServer:
    """Stores chunks as local files. Checksums each 64KB block."""

    def __init__(self, server_id: str, data_dir: str, master_stub):
        self.server_id  = server_id
        self.data_dir   = Path(data_dir)
        self.master     = master_stub
        self.chunks:    dict[int, str] = {}  # handle → filepath
        self.checksums: dict[int, list] = {}  # handle → [crc32 per block]
        self.data_dir.mkdir(parents=True, exist_ok=True)
        threading.Thread(target=self._heartbeat_loop, daemon=True).start()

    def write_chunk(self, handle: int, offset: int, data: bytes):
        path = self.data_dir / str(handle)
        with open(path, \'r+b\' if path.exists() else \'w+b\') as f:
            f.seek(offset)
            f.write(data)
        self.chunks[handle] = str(path)
        self._update_checksums(handle, path)

    def read_chunk(self, handle: int, offset: int, length: int) -> bytes:
        path = self.chunks.get(handle)
        if not path:
            raise FileNotFoundError(f\'Chunk {handle} not found\')
        with open(path, \'rb\') as f:
            f.seek(offset)
            data = f.read(length)
        self._verify_checksums(handle, path, offset, len(data))
        return data

    def _update_checksums(self, handle: int, path: str):
        """Compute CRC32 for each 64KB block."""
        import zlib
        blocks = []
        with open(path, \'rb\') as f:
            while True:
                block = f.read(BLOCK_SIZE)
                if not block: break
                blocks.append(zlib.crc32(block))
        self.checksums[handle] = blocks

    def _verify_checksums(self, handle: int, path: str, offset: int, length: int):
        """Verify CRC32 for blocks covering the read range."""
        import zlib
        start_block = offset // BLOCK_SIZE
        end_block   = (offset + length - 1) // BLOCK_SIZE
        stored      = self.checksums.get(handle, [])
        with open(path, \'rb\') as f:
            for i in range(start_block, end_block + 1):
                f.seek(i * BLOCK_SIZE)
                block = f.read(BLOCK_SIZE)
                actual = zlib.crc32(block)
                if i < len(stored) and stored[i] != actual:
                    self.master.report_corruption(self.server_id, handle, i)
                    raise IOError(f\'Checksum mismatch chunk={handle} block={i}\')

    def _heartbeat_loop(self):
        while True:
            self.master.heartbeat(self.server_id, list(self.chunks.keys()))
            time.sleep(3)

    def replicate_to(self, handle: int, target_server: str):
        """Send chunk to another server (during re-replication)."""
        data = open(self.chunks[handle], \'rb\').read()
        target_stub = get_chunk_server_stub(target_server)
        target_stub.write_chunk(handle, 0, data)
⚖️ Tradeoffs
💾 Durability: 3× Replication vs Erasure Coding
3× Replication (GFS/HDFS default)

Simple: 3 full copies. 200% storage overhead. Tolerates 2 simultaneous failures. Read from any replica — 3× read bandwidth. Fast recovery (copy one replica). Used for hot data and new writes. HDFS default since the beginning.

✅ Erasure Coding (cold data)

Reed-Solomon (6+3): split file into 6 data fragments + 3 parity. Tolerate any 3 failures. Only 50% storage overhead vs 200% for 3× replication. CPU cost to encode/decode. Best for cold, infrequently accessed data (archival). HDFS 3.0+ supports EC natively. Used by Facebook/Meta for warm/cold tiers.

✅ Tiered (hot + cold)

Hot data: 3× replication (fast writes, simple recovery). Cold data (>30 days unaccessed): automatically migrate to erasure coded storage. Halves storage cost with minimal latency impact on cold reads. Industry standard (Google Colossus, AWS S3).

🧠 Master HA: Single Master vs Federated vs HDFS NameNode HA
Single Master (GFS original)

Simple. All metadata in one RAM. 320 GB for 1 EB. Manual failover to shadow master (minutes of downtime). Acceptable for batch workloads (MapReduce). Not acceptable for interactive or latency-sensitive workloads.

✅ HDFS NameNode HA

Active + standby NameNode. Shared edit log via Journal Nodes (3+). Zookeeper for leader election. Automatic failover in <30s. Standard for production HDFS since Hadoop 2.x. Both namenodes keep namespace in RAM — identical state.

✅ Federated NameNode

Multiple NameNodes each responsible for a subtree of the namespace. Horizontal scaling of metadata. Needed when namespace exceeds 320 GB RAM. Used by very large Hadoop clusters (Yahoo, Alibaba). More complex routing but eliminates the single-master bottleneck.
