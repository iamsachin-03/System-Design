✅ Functional Requirements
CORE OPERATIONS
GET(key) → value | null: Return cached value or null (cache miss). O(1). Should not block.
SET(key, value, ttl): Store key-value pair with optional TTL. O(1). Overwrite if exists.
DELETE(key): Remove a key. O(1). No-op if not found.
TTL/EXPIRE: Keys auto-expire after specified duration. Expired keys return null.
SYSTEM BEHAVIORS
Consistent Hashing: Keys distributed across nodes using consistent hashing. Adding/removing a node only remaps ~1/N keys (minimal disruption).
Eviction: When a node is full, evict keys using the configured policy (LRU, LFU, etc.) to make room for new entries.
Replication: Each shard has 1 primary (writes) + 1 replica (reads + failover). Replica promoted on primary failure.
Hot key handling: A single popular key doesn't overwhelm one node. Detect and distribute hot keys.
⚡ Non-Functional Requirements
PERFORMANCE
GET/SET latency: p99 < 1ms
Throughput: 10M ops/sec across cluster
Cache hit rate target: >90%
SCALABILITY
Horizontally scalable: add nodes without restart
Adding a node remaps only ~1/N keys, not all keys
Linear throughput scaling with node count
RELIABILITY
Node failure: replica promotes, no data loss for replicated keys
Cache loss on total failure: rebuild from source DB (acceptable — it's a cache)
No cascading failures (cache stampede protection)
CONSISTENCY
Eventually consistent: replica may lag primary by ~ms
Stale reads acceptable (cache semantics)
Atomic operations: INCR, SETNX for specific use cases
🎙 Phase 1 — Why Consistent Hashing?
PHASE 1 · MOTIVATION
IR
INTERVIEWER
Design a distributed cache. Let's start with how you distribute keys across nodes.
ME
YOU
The naive approach: hash the key, take modulo N nodes:
node = hash(key) % N

This works. But the moment you add or remove a node, N changes. Every single key remaps to a different node. If you have 10M cached keys and add an 11th node, all 10M cache entries become invalid simultaneously — a thundering herd of 10M DB queries. Your database gets destroyed.

This is why consistent hashing exists. With consistent hashing, adding/removing a node only remaps approximately 1/N of the keys instead of all keys. Adding the 11th node to a 10-node cluster only remaps ~10% of keys — the rest stay valid.

The key insight: instead of mapping keys to nodes, we map both keys and nodes to a circular hash ring (0 to 2³²-1). A key is served by the first node encountered going clockwise from the key's position on the ring.
IR
INTERVIEWER
If adding a node only remaps 1/N keys, what happens to those 1/N cache entries that move to the new node?
ME
YOU
Those cache entries are cache misses — the client looks up the key, hits the new node, and finds nothing there. It then fetches from the origin database, stores in the new node, and future requests hit the cache normally.

The disruption is bounded and temporary. If you have a 10-node cluster and add a node, ~10% of keys miss once. That's a 10% increase in DB load for a short period — completely manageable. Compare to modulo hashing where 100% of keys miss simultaneously.
🎯 Phase 2 — Virtual Nodes (The Critical Enhancement)
PHASE 2 · VIRTUAL NODES
IR
INTERVIEWER
Basic consistent hashing has a problem — key distribution is uneven. How do you fix that?
ME
YOU
With N nodes placed on the ring, each node occupies a random arc. Some nodes might occupy 30% of the ring, others only 5%. The more popular arcs get more requests — hotspot nodes that are overloaded while others are idle.

Virtual Nodes (vnodes) fix this. Instead of placing each physical node once on the ring, we place it K times (K virtual nodes per physical node). For example, with 10 nodes × 100 virtual nodes = 1,000 points on the ring.

Why this works: with 1,000 points uniformly distributed (via different hash seeds), each physical node's total arc coverage converges toward 10% of the ring — even distribution by the law of large numbers. No single physical node gets disproportionately more keys.

Benefits beyond even distribution:
  • Graceful node loss: If a node dies, its K virtual nodes are distributed across all other nodes rather than all going to one neighbor — spreading the load
  • Heterogeneous clusters: A larger node can have more virtual nodes (proportional to its capacity), naturally receiving more traffic
  • K = 100–150 virtual nodes per physical node is typical in production (Cassandra uses 256)
🔥 Phase 3 — Cache Stampede & Hot Keys
PHASE 3 · FAILURE MODES
IR
INTERVIEWER
What's a cache stampede and how do you prevent it?
ME
YOU
A cache stampede (also called thundering herd) happens when a popular cache entry expires simultaneously, and many concurrent requests all see a cache miss at the same time. All of them query the database simultaneously — potentially thousands of queries for the same key, crushing the DB.

Example: A product page for the iPhone 16 is cached with TTL 60 minutes. At 11am, the TTL expires. In the next 50 milliseconds, 10,000 users request that page — all see cache miss, all hit the DB.

Prevention strategies:
  1. Mutex/Lock (Cache aside with lock): First request to detect miss acquires a Redis distributed lock (SET NX). While lock is held, other requests for the same key return stale value or wait. Lock holder fetches from DB, updates cache, releases lock. Only 1 DB query instead of 10,000.
  2. Probabilistic Early Expiration (PER): Each request probabilistically decides to refresh the cache before it expires, using the formula: refresh if now + β × log(random()) > expiry. More likely to refresh as TTL approaches. Smooths the refresh over time instead of a spike.
  3. Stale-while-revalidate: Serve the stale value immediately (cache hit), then asynchronously refresh in the background. Zero-latency response, no DB spike. The 1-2s staleness is usually acceptable.
  4. Jitter on TTL: Instead of TTL = 3600, use TTL = 3600 + random(0, 300). Prevents mass expiration of entries set at the same time.
I'd use stale-while-revalidate + TTL jitter for most cases. Add distributed lock for situations where stale data is truly unacceptable.
IR
INTERVIEWER
What about hot keys — a single key receiving millions of requests per second?
ME
YOU
Hot keys are a fundamental problem in consistent hashing — all requests for a key always go to the same node. If one key gets 1M requests/sec and the node handles 100K, it's overwhelmed regardless of how many other nodes you have.

Three solutions:
1. Client-side hot key replication: Detect hot keys (threshold: >10K req/sec). Store N copies of the hot key: hot_key_0, hot_key_1, ..., hot_key_N-1. Client reads from a random copy: hot_key_{random(0, N)}. Writes go to all N copies. Spreads read load across N nodes.

2. Local in-process (L1) cache: Each application server maintains a tiny in-memory cache (e.g., Caffeine, ~10K entries). Hot keys stay in L1 for 1–5 seconds. The distributed cache (L2) only sees cache misses. Effectively 0 network calls for the hottest keys.

3. CDN for truly viral content: If a key is public and cacheable (e.g., a viral video's metadata), put it in a CDN. CDN edge nodes absorb the traffic before it even reaches your caching layer.
💡 Hot keys + cache stampede are the two most-tested cache failure modes in FAANG interviews. Be able to explain both from first principles and give multiple prevention strategies for each.
🔄 Phase 4 — Cache Invalidation Strategies
PHASE 4 · INVALIDATION
IR
INTERVIEWER
How do you keep the cache consistent with the database? Walk through the main strategies.
ME
YOU
Cache invalidation is famously one of the hardest problems in computer science. There are four main patterns, each with distinct tradeoffs:

1. Cache Aside (Lazy Loading)
Application code: On read miss → fetch from DB → store in cache. On write → write to DB first, then DELETE cache entry.
Pro: Cache only contains data that's actually been requested. Works with any DB. Simple.
Con: First read after write is always a miss. Race condition: two writes can leave stale data if interleaved.

2. Write Through
On every write → write to cache AND DB simultaneously (synchronous). No cache-behind lag.
Pro: Cache is always up-to-date. No stale reads.
Con: Write latency = DB latency + cache latency. Cache fills with data that may never be read.

3. Write Behind (Write Back)
Write to cache immediately, return success. Asynchronously flush to DB in batches.
Pro: Extremely fast writes (in-memory speed). Great for write-heavy workloads.
Con: Data loss risk if cache node crashes before DB flush. Complex to implement safely.

4. Refresh Ahead
Background process predicts which keys will expire soon and pre-fetches them. Cache always warm.
Pro: No cache misses for predicted-popular keys. Low read latency.
Con: Requires prediction logic. May refresh data nobody ends up requesting.

My recommendation: Cache Aside for most read-heavy workloads. Write Through for critical data (user sessions, payment state). Write Behind for metrics/counters where DB write throughput matters.
🔁 Phase 5 — Replication & Failure Handling
PHASE 5 · REPLICATION
IR
INTERVIEWER
How does Redis replication and failover work?
ME
YOU
Redis uses asynchronous primary-replica replication:
  1. Primary accepts all writes. After writing to its own memory, it asynchronously streams the write commands to replicas via a replication backlog buffer.
  2. Replicas apply the commands and stay in sync (typically <1ms lag).
  3. Reads can be served from replicas (read scaling) or primary only (stronger consistency).
Failover with Redis Sentinel:
Redis Sentinel is a distributed monitoring system that runs alongside Redis:
  1. Multiple Sentinel processes monitor the primary. If primary doesn't respond for >30s, Sentinel consensus (quorum) declares it down.
  2. Sentinel promotes the replica with the most up-to-date replication offset as the new primary.
  3. Other replicas and application clients are notified of the new primary address.
  4. Application clients using the Sentinel-aware client library reconnect automatically.
Redis Cluster (alternative): Built-in sharding + replication. 16,384 hash slots distributed across nodes. Each master has at least one replica. No separate Sentinel needed. The standard for production Redis at scale.

⚡ Add/Remove Node — What Gets Remapped
➕ ADDING A NODE (Node D)
1
Node D's virtual nodes are placed on the ring at computed positions
2
For each of D's positions on the ring, the keys that previously mapped to the next clockwise node now map to D
3
Only the keys in D's new segments get remapped — ~K/ring_size fraction = ~1/N keys total
4
These keys are cache misses on first access → fetched from DB → stored on D
5
All other keys are unaffected — no disruption
➖ REMOVING A NODE (Node B fails)
1
B's virtual nodes are removed from the ring
2
Keys that were on B's segments now map to the next node clockwise from each of B's positions
3
With 100 virtual nodes, B's keys scatter to ~many different nodes (not all to one neighbor)
4
These keys are cache misses initially — refilled from DB over time
5
Load distributes evenly across remaining nodes (virtual node benefit)
🗑 Eviction Policies — When the Cache is Full
Know all 8 Redis eviction policies by name. Interviewers ask "which would you use for a session cache vs a product catalog cache vs a leaderboard?"

noeviction
Never evict. Return error when full. Use only when cache is also the source of truth (not typical).
allkeys-lru
LRU across all keys. Evict the least recently accessed key. Best for general caches where recency predicts future access.
allkeys-lfu
LFU across all keys. Evict the least frequently accessed. Better than LRU for skewed access patterns (popular items stay longer).
volatile-lru
LRU only among keys with TTL. Never evicts keys without TTL. Use when some keys are critical and must never be evicted.
volatile-lfu
LFU only among keys with TTL. LFU variant of volatile-lru.
volatile-ttl
Evict soonest-to-expire keys first. When full, removes keys with the shortest remaining TTL. Useful when you want maximum fresh data.
volatile-random
Random eviction among TTL keys. Low overhead. Use when access pattern is uniform (no hot keys).
allkeys-random
Fully random eviction. Lowest overhead, no access tracking. Only useful for truly random access patterns.
🎯 Which Policy for Which Use Case
Use CaseRecommended PolicyWhy
General-purpose cache (product catalog, user profiles)allkeys-lruRecency predicts future use well. Simple, effective.
Viral content (videos, tweets with skewed popularity)allkeys-lfuPopular items stay in cache longer. LRU evicts them after a single access gap.
Session cache (user auth tokens)volatile-lru or volatile-ttlSessions have TTLs. Volatile policies respect explicit expiry, never evict permanent keys.
Rate limiting countersnoevictionCounters must not be silently dropped. Pre-size the cache appropriately.
Leaderboard / sorted scoresallkeys-lruAlways accessed, always recent. LRU keeps the data you need.
Pre-computed reports (batch results)volatile-ttlReports have natural expiry. Evict soon-to-expire reports first — they're least valuable.
⚙️ LRU Implementation — O(1) GET and O(1) SET
Classic interview question: "Implement an LRU cache in O(1) time for both GET and SET." The key insight: HashMap + Doubly Linked List.

Data Structures
HashMap<key, DListNode>
→ O(1) lookup: given key, find node in list

Doubly Linked List (DLL)
→ Most recently used = head
→ Least recently used = tail
→ O(1) move to front (with pointer to node)
→ O(1) remove from tail (eviction)
Operations
GET(key):
1. Lookup node in HashMap → O(1)
2. Move node to head of DLL → O(1)
3. Return value

SET(key, value):
1. If exists: update + move to head
2. If new: create node, add to head
3. If full: remove tail node (LRU victim), delete from HashMap
→ Both GET and SET: O(1)

🛡 Cache Aside with Stampede Protection
// The recommended pattern: Cache Aside + Distributed Lock (anti-stampede)
async function getWithCache(key, fetchFromDB, ttl) {
  // 1. Try cache first (fast path)
  const cached = await cache.get(key);
  if (cached !== null) return cached;

  // 2. Cache miss — try to acquire lock (prevents stampede)
  const lockKey = `lock:${key}`;
  const acquired = await cache.set(lockKey, "1", { nx: true, ttl: 5 });

  if (!acquired) {
    // Another request is fetching — wait briefly and retry
    await sleep(50); // ms
    const retried = await cache.get(key);
    if (retried !== null) return retried;
  }

  try {
    // 3. Lock holder: fetch from DB
    const value = await fetchFromDB(key);

    // 4. Store in cache with TTL jitter (prevents mass expiry)
    const jitter = Math.random() * ttl * 0.1;
    await cache.set(key, value, { ttl: ttl + jitter });

    return value;
  } finally {
    await cache.delete(lockKey); // always release lock
  }
}

// ── 1. Consistent Hash Ring ───────────────────────────
public class ConsistentHashRing {

    // Sorted map: hash position → cache node
    private final TreeMap<Long, CacheNode> ring = new TreeMap<>();
    private final int  virtualNodesPerNode;
    private final ReadWriteLock lock = new ReentrantReadWriteLock();

    public ConsistentHashRing(int virtualNodesPerNode) {
        this.virtualNodesPerNode = virtualNodesPerNode;
    }

    public void addNode(CacheNode node) {
        lock.writeLock().lock();
        try {
            for (int i = 0; i < virtualNodesPerNode; i++) {
                // Each virtual node gets a unique hash seed
                long hash = hash(node.getNodeId() + "#" + i);
                ring.put(hash, node);
            }
        } finally {
            lock.writeLock().unlock();
        }
    }

    public void removeNode(CacheNode node) {
        lock.writeLock().lock();
        try {
            for (int i = 0; i < virtualNodesPerNode; i++) {
                long hash = hash(node.getNodeId() + "#" + i);
                ring.remove(hash);
            }
        } finally {
            lock.writeLock().unlock();
        }
    }

    public CacheNode getNode(String key) {
        lock.readLock().lock();
        try {
            if (ring.isEmpty()) return null;
            long hash = hash(key);
            // Find the first node clockwise (>= hash position)
            Map.Entry<Long, CacheNode> entry = ring.ceilingEntry(hash);
            // Wrap around the ring if at the end
            if (entry == null) entry = ring.firstEntry();
            return entry.getValue();
        } finally {
            lock.readLock().unlock();
        }
    }

    /** Get N distinct nodes for a key (for replication or hot key copies) */
    public List<CacheNode> getNodes(String key, int n) {
        lock.readLock().lock();
        try {
            Set<String> seen = new HashSet<>();
            List<CacheNode> nodes = new ArrayList<>();
            long hash = hash(key);

            // Walk ring clockwise, collecting unique physical nodes
            NavigableMap<Long, CacheNode> tail = ring.tailMap(hash);
            Iterable<CacheNode> iter = Iterables.concat(
                tail.values(), ring.values());

            for (CacheNode node : iter) {
                if (seen.add(node.getNodeId())) {
                    nodes.add(node);
                    if (nodes.size() == n) break;
                }
            }
            return nodes;
        } finally {
            lock.readLock().unlock();
        }
    }

    /** MD5-based hash: consistent, well-distributed */
    private long hash(String key) {
        try {
            MessageDigest md = MessageDigest.getInstance("MD5");
            byte[] digest = md.digest(key.getBytes(StandardCharsets.UTF_8));
            return ((long) (digest[3] & 0xFF) << 24)
                 | ((long) (digest[2] & 0xFF) << 16)
                 | ((long) (digest[1] & 0xFF) << 8)
                 | ((long) (digest[0] & 0xFF));
        } catch (Exception e) { throw new RuntimeException(e); }
    }
}

// ── 2. LRU Cache (O(1) GET + SET) ────────────────────
public class LRUCache<K, V> {

    private final int capacity;
    private final LinkedHashMap<K, V> cache;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        // LinkedHashMap with accessOrder=true: LRU order maintained automatically
        this.cache = new LinkedHashMap<>(capacity, 0.75f, true) {
            @Override
            protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
                return size() > capacity; // auto-evict LRU when full
            }
        };
    }

    public synchronized V get(K key) {
        return cache.getOrDefault(key, null); // updates access order (LRU)
    }

    public synchronized void put(K key, V value) {
        cache.put(key, value); // removes eldest if over capacity
    }

    public synchronized void remove(K key) {
        cache.remove(key);
    }

    public synchronized int size() { return cache.size(); }
}

// ── 3. Cache Client (routing + stampede protection) ───
public class DistributedCacheClient {

    private final ConsistentHashRing ring;
    private final LRUCache<String, byte[]> localL1;  // in-process hot key cache
    private final ConcurrentHashMap<String, AtomicLong> reqCounts; // hot key detection

    private static final int L1_SIZE       = 10_000;
    private static final int HOT_THRESHOLD = 10_000; // req/sec
    private static final int TIMEOUT_MS    = 10;

    public DistributedCacheClient(List<CacheNode> nodes) {
        ring     = new ConsistentHashRing(150);
        localL1  = new LRUCache<>(L1_SIZE);
        reqCounts= new ConcurrentHashMap<>();
        nodes.forEach(ring::addNode);
    }

    public byte[] get(String key) {
        // L1 check: hot keys served from in-process cache (0 network calls)
        byte[] l1Value = localL1.get(key);
        if (l1Value != null) return l1Value;

        // Route to correct node via consistent hash
        CacheNode node = ring.getNode(key);
        if (node == null) return null;

        try {
            byte[] value = node.get(key, TIMEOUT_MS);

            // Hot key detection: store in L1 if accessed frequently
            if (isHotKey(key) && value != null) {
                localL1.put(key, value);
            }
            return value;
        } catch (TimeoutException | NodeUnavailableException e) {
            // Fail open: treat as cache miss, let caller fall back to DB
            log.warn("Cache get failed, failing open: {}", key);
            return null;
        }
    }

    public boolean set(String key, byte[] value, int ttlSeconds) {
        // Invalidate L1 on write
        localL1.remove(key);

        CacheNode node = ring.getNode(key);
        if (node == null) return false;

        // TTL jitter: prevents mass expiry at same time (stampede prevention)
        int jitter = (int) (Math.random() * ttlSeconds * 0.1);
        int finalTtl = ttlSeconds + jitter;

        try {
            return node.set(key, value, finalTtl, TIMEOUT_MS);
        } catch (Exception e) {
            log.warn("Cache set failed: {}", key);
            return false;
        }
    }

    private boolean isHotKey(String key) {
        return reqCounts
            .computeIfAbsent(key, k -> new AtomicLong())
            .incrementAndGet() > HOT_THRESHOLD;
    }
}

# Distributed Cache — Python Implementation

import hashlib, bisect, threading, time
from collections import OrderedDict
from typing import List, Optional, Dict, Any


# ── 1. Consistent Hash Ring ───────────────────────────
class ConsistentHashRing:
    """
    Consistent hashing with virtual nodes.
    Adding/removing a node remaps only ~1/N keys.
    """
    def __init__(self, virtual_nodes: int = 150):
        self.virtual_nodes  = virtual_nodes
        self._ring: Dict[int, str] = {}     # hash → node_id
        self._sorted_keys: List[int] = []   # sorted hash positions
        self._nodes: Dict[str, any] = {}    # node_id → node object
        self._lock = threading.RWLock()

    def add_node(self, node_id: str, node: any):
        with self._lock.writer():
            self._nodes[node_id] = node
            for i in range(self.virtual_nodes):
                key = self._hash(f"{node_id}#{i}")
                self._ring[key] = node_id
            self._sorted_keys = sorted(self._ring.keys())

    def remove_node(self, node_id: str):
        with self._lock.writer():
            for i in range(self.virtual_nodes):
                key = self._hash(f"{node_id}#{i}")
                self._ring.pop(key, None)
            self._nodes.pop(node_id, None)
            self._sorted_keys = sorted(self._ring.keys())

    def get_node(self, cache_key: str) -> Optional[any]:
        with self._lock.reader():
            if not self._ring:
                return None
            h = self._hash(cache_key)
            # Find first ring position >= h (clockwise)
            idx = bisect.bisect_left(self._sorted_keys, h)
            # Wrap around
            idx = idx % len(self._sorted_keys)
            node_id = self._ring[self._sorted_keys[idx]]
            return self._nodes[node_id]

    def _hash(self, key: str) -> int:
        """MD5-based hash for good distribution"""
        h = hashlib.md5(key.encode()).digest()
        return int.from_bytes(h[:4], 'big')


# ── 2. LRU Cache (O(1) GET and SET) ──────────────────
class LRUCache:
    """
    O(1) LRU cache using Python OrderedDict.
    OrderedDict maintains insertion order + O(1) move-to-end.
    """
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache    = OrderedDict()  # key → value, LRU order
        self._lock    = threading.Lock()

    def get(self, key: str) -> Optional[Any]:
        with self._lock:
            if key not in self.cache:
                return None
            # Move to end (most recently used)
            self.cache.move_to_end(key)
            return self.cache[key]

    def put(self, key: str, value: Any) -> None:
        with self._lock:
            if key in self.cache:
                self.cache.move_to_end(key)
            self.cache[key] = value
            if len(self.cache) > self.capacity:
                # Remove least recently used (first item)
                self.cache.popitem(last=False)

    def delete(self, key: str) -> None:
        with self._lock:
            self.cache.pop(key, None)


# ── 3. Cache Aside with Stampede Protection ───────────
class CacheClient:
    """
    Distributed cache client with:
    - Consistent hash routing
    - L1 (in-process) + L2 (distributed) two-tier cache
    - Cache stampede protection via distributed lock
    - Fail-open on node unavailability
    """
    L1_SIZE       = 10_000
    LOCK_TTL_SEC  = 5
    TIMEOUT_MS    = 10

    def __init__(self, nodes: list):
        self.ring   = ConsistentHashRing(virtual_nodes=150)
        self.l1     = LRUCache(capacity=self.L1_SIZE)
        self._hit_counts: Dict[str, int] = {}

        for node in nodes:
            self.ring.add_node(node.node_id, node)

    def get(self, key: str) -> Optional[bytes]:
        """Two-tier GET: L1 (in-process) → L2 (distributed)"""
        # L1 check (in-process, zero network)
        l1_val = self.l1.get(key)
        if l1_val is not None:
            return l1_val

        # L2: distributed cache
        node = self.ring.get_node(key)
        if not node:
            return None

        try:
            value = node.get(key, timeout_ms=self.TIMEOUT_MS)
            if value is not None:
                self._maybe_promote_to_l1(key, value)
            return value
        except Exception as e:
            # Fail open: treat as cache miss
            return None

    def get_with_loader(self, key: str, loader_fn,
                          ttl: int = 300) -> Any:
        """
        Cache aside with stampede protection.
        Only ONE request fetches from DB when cache misses.
        Others wait briefly or return stale.
        """
        value = self.get(key)
        if value is not None:
            return value

        lock_key = f"lock:{key}"
        node = self.ring.get_node(lock_key)

        try:
            # Try to acquire lock (NX = only if not exists)
            acquired = node.set(lock_key, "1",
                                ttl=self.LOCK_TTL_SEC, nx=True)
        except:
            acquired = False

        if not acquired:
            # Another request is loading — wait and retry
            time.sleep(0.05)
            retried = self.get(key)
            if retried is not None:
                return retried
            # Still no value — fall through to load (lock may have expired)

        try:
            value = loader_fn(key)
            jitter = int(ttl * 0.1 * (0.5 + 0.5 * hash(key) % 1000 / 1000))
            self.set(key, value, ttl + jitter)
            return value
        finally:
            try: node.delete(lock_key)  # always release
            except: pass

    def set(self, key: str, value: bytes, ttl: int = 300) -> bool:
        self.l1.delete(key)  # invalidate L1 on write
        node = self.ring.get_node(key)
        if not node:
            return False
        try:
            return node.set(key, value, ttl=ttl)
        except:
            return False

    def _maybe_promote_to_l1(self, key: str, value: bytes):
        count = self._hit_counts.get(key, 0) + 1
        self._hit_counts[key] = count
        if count > 10:  # promote hot keys to L1
            self.l1.put(key, value)
⚖️ Design Tradeoffs
Cache design is full of tradeoffs. Every caching decision trades off consistency, availability, and performance in different ways.
🔄 Key Distribution: Consistent Hashing vs Modulo vs Rendezvous Hashing
✅ Consistent Hashing + Virtual Nodes

Adding/removing nodes only remaps ~1/N keys. Virtual nodes ensure even distribution. Industry standard (Redis Cluster, Cassandra, DynamoDB). Minimal cache invalidation on topology changes.

❌ Modulo Hashing

Simple: node = hash(key) % N. Works perfectly with a fixed N. Fails completely on N change: all keys remap → cache becomes empty simultaneously → DB gets 100% traffic. Never use with dynamic clusters.

⚠️ Rendezvous Hashing

For each key, compute score for each node, pick highest score. Minimal remapping on change (same as consistent hashing). But: O(N) per lookup (check all nodes). Only practical with few nodes. Consistent hashing with virtual nodes scales better.

💾 Cache Write Pattern: Cache Aside vs Write-Through vs Write-Behind
✅ Cache Aside (Lazy Loading)

Application code: read miss → DB fetch → SET cache. Write: DB write → DELETE cache key. Most flexible. Cache only stores actually-requested data. Works with any backend. Potential for brief stale reads after write.

⚠️ Write-Through

Write to cache AND DB synchronously. Always consistent. But: write latency = DB latency. Every write goes through cache even if the data is never read again. Cache fills with cold data. Use for user sessions, payment state — low write rate, must be consistent.

⚠️ Write-Behind

Write to cache, async to DB. Blazing fast writes. But: data loss risk if cache crashes before flush. Ordering challenges for multi-key transactions. Use only for analytics counters or non-critical metrics where some loss is acceptable.

🏃 Eviction: LRU vs LFU vs TTL-only
✅ LRU (allkeys-lru)

Evict least recently used. O(1) with HashMap + DLL. Handles temporal locality well — recently accessed items tend to be accessed again. Good for most general caches: product pages, user profiles.

⚠️ LFU (allkeys-lfu)

Evict least frequently used. Keeps popular items longer. Better for viral/skewed workloads (top 20% items get 80% traffic). But: newly popular items start with low frequency — slow to "promote". Redis 4.0+ implements approximate LFU efficiently.

💡 TTL-only

No active eviction — keys expire by TTL. Predictable: each key lives exactly N seconds. Simplest to reason about. Bad for highly variable access patterns — popular items expire as frequently as unpopular ones. Often combine with LRU/LFU: TTL for correctness, LRU/LFU for memory pressure.

🔥 Cache Stampede: Lock vs Stale-While-Revalidate vs Early Expiration
✅ Stale-While-Revalidate

Serve stale data immediately. Refresh asynchronously in background. Zero extra latency for caller. Eliminates stampede completely. Acceptable staleness window (1-2s). Best for non-critical data (product catalog, blog posts).

⚠️ Distributed Lock (NX)

First miss acquires Redis NX lock. Others wait. Only one DB query per cache miss. Correct. But: adds latency for waiters (50ms wait loop). Lock holder failure requires TTL-based auto-release. Use for data where staleness is truly unacceptable (inventory counts).

💡 TTL Jitter

Add randomness to TTL: base_ttl + random(0, base_ttl*0.1). Prevents N items cached at the same time all expiring simultaneously. Simplest prevention. Combine with stale-while-revalidate for comprehensive stampede protection.
