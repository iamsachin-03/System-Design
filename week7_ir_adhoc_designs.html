Designing Recent Searches

When a user clicks the search bar, they see a personalised list of their recent searches. Design a system that stores, retrieves, deduplicates, and ranks recent queries for hundreds of millions of users — efficiently and at low latency.

100M+
DAU with search activity
<50ms
Recent searches load latency
5–10
Queries shown per user
30 days

Requirements

Functional

  • When user opens search box → show last N (5–10) unique searches
  • When user submits a search → save it to their history
  • User can delete individual searches or clear all history
  • Deduplicate: if "iPhone" searched 5 times, show it once (most recent timestamp)
  • Personalised: each user sees only their own history

Non-Functional

  • Read latency <50ms (opened on every search interaction)
  • Write is async-ok — slight delay in saving is fine
  • Data should persist across devices (server-side, not just local)
  • Privacy: user can delete all history at any time (GDPR right to erasure)
  • Scale: 100M users × avg 5 searches/day = 500M writes/day

Data Model & Storage Design

Redis Sorted Set Pattern

Redis operations for recent searchesRedis
-- Key per user: recent_searches:{user_id}
-- Score = Unix timestamp ms (higher = more recent)
-- Member = search query string

-- User searches "iPhone 16"
ZADD recent_searches:user_42  1711000000000  "iPhone 16"
-- Same query searched again → updates score (dedup automatic!)
ZADD recent_searches:user_42  1711001000000  "iPhone 16"

-- Trim to last 20 (keep latest 20, delete older)
ZREMRANGEBYRANK recent_searches:user_42  0  -21
-- Rank 0 = lowest score (oldest). -21 means: keep top 20 (ranks -20 to -1)

-- Fetch last 10 searches (most recent first)
ZREVRANGE recent_searches:user_42  0  9  WITHSCORES

-- Delete one search
ZREM recent_searches:user_42  "iPhone 16"

-- Clear all history (GDPR right to erasure)
DEL recent_searches:user_42

-- Set TTL to auto-expire old history (30 days)
EXPIRE recent_searches:user_42  2592000  -- 30 days in seconds

DB Persistence Layer

Redis is the fast serving layer. But Redis can lose data on restart (unless AOF/RDB). For durability, also persist to MySQL/Postgres asynchronously via Kafka.

Search Ranking & Suggestions

Beyond Simple Recency: Personalised Ranking

A naive recent-searches list shows queries in order of recency. More sophisticated systems rank by a combination of recency + frequency + result quality. This is how Google's "Search history" and YouTube's "Recent searches" work.

Recency Score

Queries searched more recently score higher. Simple: use timestamp as-is. Time-decayed: score = timestamp × exp(-λ × age). Older searches decay in importance over hours/days.

Frequency Boost

If "MacBook Pro" was searched 10 times and "keyboard shortcut" once, MacBook Pro ranks higher even if keyboard shortcut was more recent. Combine: score = recency × log(frequency + 1).

Contextual Ranking

Boost queries matching the current prefix. User types "ip" → "iPhone 16" ranks higher than "iPad case" in recent searches because "ip" is a stronger prefix match for "iPhone". Combine prefix affinity with recency.

async def get_recent_searches_for_prefix(redis, user_id: int, prefix: str) -> list[str]:
    """Return recent searches matching prefix, ordered by recency."""
    # Fetch all recent searches (up to 50)
    all_recent = await redis.zrevrange(f"recent_searches:{user_id}", 0, 49)

    # Filter to prefix matches (case-insensitive)
    prefix_lower = prefix.lower()
    matches = [
        q.decode() for q in all_recent
        if q.decode().lower().startswith(prefix_lower)
    ]
    return matches[:5]   # return top 5 prefix-matching recent searches

Code — Full Recent Search Service
recent_search_service.py — FastAPI + RedisPython
import time
from fastapi import FastAPI
import redis.asyncio as aioredis

app   = FastAPI()
redis = aioredis.Redis(host="redis")

MAX_HISTORY = 20     # keep at most 20 searches per user
SHOW_TOP    = 10     # show top 10 to user
TTL_DAYS    = 30     # auto-expire after 30 days of inactivity

def _key(user_id: int): return f"recent_searches:{user_id}"

@app.post("/searches")
async def record_search(user_id: int, query: str):
    """Record a search. Deduplicates and trims automatically."""
    q    = query.strip().lower()[:200]   # normalise, cap at 200 chars
    if not q: return
    key  = _key(user_id)
    now  = int(time.time() * 1000)

    async with redis.pipeline() as pipe:
        pipe.zadd(key, {q: now})          # add or update score
        pipe.zremrangebyrank(key, 0, -MAX_HISTORY-1)  # trim oldest
        pipe.expire(key, TTL_DAYS * 86400)  # reset TTL on activity
        await pipe.execute()

    return {"recorded": True}

@app.get("/searches/{user_id}")
async def get_recent(user_id: int, prefix: str = ""):
    """Return recent searches, optionally filtered by prefix."""
    raw = await redis.zrevrange(_key(user_id), 0, MAX_HISTORY-1)
    queries = [q.decode() for q in raw]
    if prefix:
        queries = [q for q in queries if q.startswith(prefix.lower())]
    return {"searches": queries[:SHOW_TOP]}

@app.delete("/searches/{user_id}/{query}")
async def delete_search(user_id: int, query: str):
    await redis.zrem(_key(user_id), query.lower())
    return {"deleted": True}

@app.delete("/searches/{user_id}")
async def clear_all(user_id: int):
    """GDPR right to erasure — delete entire history."""
    await redis.delete(_key(user_id))
    return {"cleared": True}

  • Redis Sorted Set is the canonical data structure for recent searches — ZADD with timestamp as score provides O(log N) insert, O(log N) deduplicate (update score), O(log N) range query. All operations in one call.
  • Deduplication is automatic — ZADD on the same member updates the score. "iPhone 16" searched 10 times = one entry with the most recent timestamp. No extra logic needed.
  • Write path is fully async — search results are returned immediately; the history record is written to Kafka → Redis in the background. No latency added to the critical path.
  • 30-day TTL with reset-on-activity — EXPIRE is reset to 30 days on every search. Inactive users' data auto-expires. Dramatically reduces memory footprint without manual cleanup jobs.
  • GDPR compliance is a single DEL command — DEL recent_searches:{user_id} wipes all history instantly. Combined with Kafka event deletion for the durable MySQL layer, this completes the right-to-erasure requirement.

Cricbuzz Text Commentary System

Cricbuzz delivers ball-by-ball text commentary to millions of users during a cricket match. Each delivery generates a commentary event (runs, wicket, wide, boundary) that must reach all connected users in real-time with sub-second latency — even during India vs Pakistan matches with 50M+ concurrent viewers.

50M+
Peak concurrent users (IND-PAK)
~1
Update per ball (every 3–5 minutes)
<1s
Commentary delivery latency
900+
Deliveries per match (20-over)

Why This Is Hard

Fan-out problem

Each ball: 1 writer (the scorekeeper) must notify 50M readers simultaneously. A naïve push model: 50M WebSocket pushes per delivery. Server can handle ~10K pushes/sec per node → 5000 nodes. Too expensive.

Pull vs Push trade-off

Pure polling (every user polls every second): 50M × 1 req/s = 50M RPS. Unacceptable. Pure push (WebSocket to all): 50M persistent connections. Need a hybrid model: push to connected users, pull for reconnecting.

Architecture: SSE + Redis PubSub + Kafka

Real-time Delivery

Server-Sent Events (SSE) — The Right Protocol

SSE ✓ (Best for commentary)

  • One-directional: server → client (commentary is read-only)
  • Browser built-in: EventSource API, auto-reconnect, retry
  • Over HTTP/1.1 or HTTP/2 — CDN-friendly
  • Much lighter than WebSocket (no bidirectional overhead)
  • Graceful reconnect: sends Last-Event-ID → server resends missed events

WebSocket (Overkill for commentary)

  • Bidirectional — both directions add overhead for read-only data
  • Not CDN-cacheable at the protocol level
  • More complex keepalive and reconnect logic
  • Better for: chat, collaborative editing, games

Handling Reconnects — The Last-Event-ID Pattern

Redis PubSub Fan-out

Architecture: Each SSE server subscribes to match:{id} Redis channel. When a new ball commentary arrives, the fan-out worker publishes to this channel. All SSE servers receive it and push to their connected clients. This scales horizontally — add more SSE servers without changing the fan-out logic.

Scorecard Calculation

Every ball updates the live scorecard. Recomputing from scratch on each ball is expensive. Instead, maintain a cached scorecard in Redis, update atomically with Lua scripts:

score_updater.py — atomic scorecard updatePython
async def update_scorecard(redis, match_id: str, event: dict):
    """Atomically update all scorecard fields in one Redis pipeline."""
    async with redis.pipeline(transaction=True) as pipe:
        score_key = f"match:{match_id}:score"

        if event["event_type"] == "wicket":
            pipe.hincrby(score_key, "wickets", 1)
        else:
            pipe.hincrby(score_key, "total_runs", event.get("runs_scored", 0))

        pipe.hset(score_key, "last_ball", event["id"])
        pipe.expire(score_key, 86400)  # expire score after 24h
        await pipe.execute()

  • SSE is better than WebSocket for one-directional live data — commentary, scores, news feeds, live dashboards. Auto-reconnect with Last-Event-ID means zero missed events even on flaky mobile connections.

Instagram Live Reactions

During an Instagram Live stream, viewers tap reaction buttons (❤️ 😂 🔥 👏) and see a burst of floating emoji animations on screen. Millions of reactions per second must be aggregated, sampled, and delivered in near-real-time to all viewers. Not every single reaction needs to be shown — just enough to create the feeling of high energy.

<500ms

Why Not Naive Push?

1M reactions/sec → 1M WebSocket messages/sec

If 1M users each tap once per second, and the server must push each reaction to all other connected viewers, that's: 1M reactions × 500K viewers = 500 billion messages/sec. Impossible. Need aggressive aggregation.

The solution: Aggregate, Sample, Burst

Instead of pushing individual reactions, aggregate them in windows (50–100ms). Push the aggregate: {heart: 4,200, fire: 1,800, laugh: 900} to all viewers. Each viewer's client renders an appropriate density of floating emojis based on the count.

Architecture

Aggregation Pipeline

1
Viewer taps reaction
Client sends POST /live/{stream_id}/react {type: "heart"}. Fire-and-forget — client doesn't wait for response. The animation starts immediately on the local client (optimistic UI).
2
Reaction API batches writes in Redis
HINCRBY on a time-bucketed key: HINCRBY react:{stream_id}:{ts_100ms} heart 1. Bucket window = 100ms. EXPIRE = 5s on each bucket (auto-cleanup).
3
Aggregator reads buckets every 100ms
Background worker: HGETALL the current bucket → compute aggregated counts → publish to Redis channel → clear bucket. This is the fan-out event.
4
Fan-out via Redis PubSub → WebSocket servers
WebSocket/SSE servers subscribed to the channel push the aggregate to all connected viewers: {window_ms:100, heart:4200, fire:1800, laugh:900}.
5
Client renders proportional animation
Client calculates emoji spawn rate from counts. 4200 hearts in 100ms = 42 hearts/sec floating up the screen. Capped at ~10 visible simultaneously for visual clarity.

Why 100ms Aggregation Windows?

Visual perception

Humans can't distinguish individual emoji appearing at >20/sec. Aggregating over 100ms windows loses no perceptible granularity. The energy feeling comes from density, not individual emoji identity.

Infrastructure reduction

At 1M reactions/sec: individual push = 1M messages/sec. 100ms aggregation = 10 messages/sec to all subscribers. 100,000× reduction in fan-out message volume while preserving the live feel.

Batching & Aggregation

Redis Time-Bucketed Counter

reaction_aggregator.py — 100ms bucket aggregationPython
import asyncio, time, json

BUCKET_MS = 100   # 100ms aggregation window

def bucket_key(stream_id: str) -> str:
    """Current 100ms bucket key."""
    ts_bucket = int(time.time() * 1000) // BUCKET_MS
    return f"react:{stream_id}:{ts_bucket}"

async def record_reaction(redis, stream_id: str, reaction_type: str):
    """Atomic increment in current time bucket. O(1)."""
    key = bucket_key(stream_id)
    async with redis.pipeline() as pipe:
        pipe.hincrby(key, reaction_type, 1)
        pipe.expire(key, 5)   # auto-cleanup after 5s
        await pipe.execute()

async def aggregation_loop(redis, stream_id: str):
    """Background worker: reads bucket every 100ms, publishes aggregate."""
    while True:
        await asyncio.sleep(BUCKET_MS / 1000)

        # Read and atomically clear previous bucket
        prev_ts   = (int(time.time() * 1000) // BUCKET_MS) - 1
        prev_key  = f"react:{stream_id}:{prev_ts}"
        counts    = await redis.hgetall(prev_key)
        await redis.delete(prev_key)   # consume the bucket

        if counts:
            aggregate = {
                k.decode(): int(v) for k, v in counts.items()
            }
            aggregate["window_ms"] = BUCKET_MS

            # Fan-out to all SSE/WS servers for this stream
            await redis.publish(
                f"reactions:{stream_id}",
                json.dumps(aggregate)
            )

Client-side Rendering Logic

reaction_renderer.js — emoji burst animationJavaScript
const MAX_VISIBLE = 8;   // max simultaneous floating emojis
const EMOJI_MAP = { heart:'❤️', fire:'🔥', laugh:'😂', clap:'👏', wow:'😮' };

function renderReactions(aggregate) {
    const total = Object.values(aggregate)
        .filter(v => typeof v === 'number')
        .reduce((a,b) => a+b, 0);

    if (total === 0) return;

    // Scale to max visible: if 10000 reactions, still show ≤8 emojis
    const scale = Math.min(1, MAX_VISIBLE / total);

    for (const [type, count] of Object.entries(aggregate)) {
        if (!EMOJI_MAP[type]) continue;
        const numToShow = Math.max(1, Math.round(count * scale));
        for (let i=0; i<numToShow; i++) {
            setTimeout(() => floatEmoji(EMOJI_MAP[type]), i * 50);
        }
    }
}

function floatEmoji(emoji) {
    const el = document.createElement('span');
    el.textContent = emoji;
    el.style = `position:fixed;bottom:100px;left:${10+Math.random()*80}%;
        font-size:${24+Math.random()*16}px;animation:float-up 2s ease-out forwards;
        pointer-events:none;z-index:1000`;
    document.body.appendChild(el);
    setTimeout(() => el.remove(), 2000);
}

reactions_api.py — ingestion endpointPython
from fastapi import FastAPI
from pydantic import BaseModel
from typing import Literal

app = FastAPI()
VALID_TYPES = {"heart","fire","laugh","clap","wow"}

class ReactionRequest(BaseModel):
    reaction_type: str
    batch_count:   int = 1   # client can batch N reactions if tapping rapidly

@app.post("/live/{stream_id}/react")
async def react(stream_id: str, req: ReactionRequest, user_id: int):
    """Record reaction. Returns immediately (fire-and-forget write)."""
    if req.reaction_type not in VALID_TYPES:
        return {"ok": False}

    # Rate limit per user: max 5 reactions/second
    rl_key = f"rl:react:{user_id}"
    count  = await redis.incr(rl_key)
    if count == 1: await redis.expire(rl_key, 1)  # 1-second window
    if count > 5: return {"ok": False, "reason": "rate_limited"}

    await record_reaction(redis, stream_id, req.reaction_type)
    return {"ok": True}

@app.get("/live/{stream_id}/reactions/stream")
async def reaction_stream(stream_id: str):
    """SSE endpoint: stream aggregated reaction counts every 100ms."""
    async def generator():
        pubsub = redis.pubsub()
        await pubsub.subscribe(f"reactions:{stream_id}")
        async for msg in pubsub.listen():
            if msg["type"] == "message":
                yield f"data: {msg['data'].decode()}\n\n"
    return StreamingResponse(generator(), media_type="text/event-stream")

Designing a Distributed Task Scheduler

A distributed task scheduler allows clients to register tasks (one-time or recurring) with a scheduled execution time. The system must execute each task exactly once within 10 seconds of its scheduled time, even if scheduler nodes fail.

<10s
Execution delay SLA
Exactly once
Guarantee (not at-least-once)
Cron + one-time
Two task types supported
Millions
Tasks registered at any time

Use Cases

One-time tasks

  • Send order confirmation email in 5 minutes
  • Cancel unconfirmed reservation after 15 minutes
  • Send birthday greeting on user's birthday at 9am
  • Retry failed payment after 24 hours

Recurring (Cron) tasks

  • Daily backup at 2am: 0 2 * * *
  • Generate weekly report every Monday: 0 9 * * 1
  • Flush cache every 5 minutes: */5 * * * *
  • Check invoice payment daily: 0 10 * * *

Challenges

  • Exactly-once: task must not run twice (double email, double charge)
  • No-miss: task must not be skipped even if scheduler crashes
  • Late execution: what if scheduler was down at execution time?
  • Scale: millions of tasks with second-level precision
Architecture

Two-layer Design

1
Client registers task
POST /tasks with payload: {type, schedule, handler_url, payload, max_retries}. Scheduler assigns task_id, stores in persistent DB (MySQL), returns task_id to client.
2
Polling workers scan for due tasks
Multiple scheduler workers query DB: SELECT * FROM tasks WHERE next_run_at <= NOW() AND status='PENDING' LIMIT 100. They "claim" tasks by locking them (pessimistic lock or CAS update).
3
Worker claims task atomically
CAS update: UPDATE tasks SET status='RUNNING', worker_id=me WHERE task_id=X AND status='PENDING'. Affected rows = 0 → another worker got it (race lost). Affected rows = 1 → claim succeeded.
4
Execute the task
Call the handler URL (HTTP callback) or execute the registered handler function. Set a heartbeat timestamp to detect stuck tasks. On success: mark DONE. On failure: increment retry count, reschedule.
5
Reschedule recurring tasks
After successful execution of a cron task, compute next_run_at using cron expression, reset status to PENDING. Task recurs automatically.
Why not Redis for task storage? Redis is ephemeral — if it crashes without persistence, all scheduled tasks are lost. MySQL (or any relational DB with ACID guarantees) ensures tasks survive server restarts. Redis can be used as a secondary index for fast "due tasks" scanning.

Cron Expression Parsing

cron_parser.py — next run time computationPython
from croniter import croniter
from datetime import datetime, timezone

def next_run_at_ms(cron_expr: str) -> int:
    """Compute next execution timestamp from cron expression."""
    # croniter: https://pypi.org/project/croniter/
    base = datetime.now(timezone.utc)
    cron = croniter(cron_expr, base)
    next_run = cron.get_next(datetime)
    return int(next_run.timestamp() * 1000)

# Examples:
# "0 2 * * *"      → next 2:00 AM UTC
# "*/5 * * * *"    → next 5-minute mark
# "0 9 * * 1"      → next Monday 9:00 AM UTC
# "0 0 1 * *"      → first day of next month at midnight
Fault Tolerance

Exactly-once Execution — The Hard Part

Problem: Worker crash after claim but before completion

Worker W1 claims task T, begins execution. W1 crashes mid-execution. Task is stuck in RUNNING state forever. No other worker will pick it up. Task is effectively lost (never completes).

Solution: Heartbeat + timeout-based stuck task recovery

Running workers write a heartbeat timestamp every 30s: UPDATE tasks SET heartbeat_at=NOW() WHERE task_id=X. A separate watchdog query finds tasks where: status=RUNNING AND heartbeat_at < NOW()-60s. These are stuck → reset to PENDING for retry.

Exactly-Once via Idempotent Handlers

Important: Even with heartbeat recovery, a task might execute TWICE: worker starts, completes, then crashes before marking DONE. Watchdog resets it. It runs again. To prevent double execution: make the handler idempotent. Include task_id in the handler. On second call: check if already executed, skip.
idempotent_handler.py — safe double-executionPython
@app.post("/handlers/send-email")
async def send_email_handler(task_id: int, email: str, subject: str):
    """Idempotent email sender: safe to call multiple times."""
    # Check if already sent (using task_id as dedup key)
    already_sent = await redis.get(f"task_done:{task_id}")
    if already_sent:
        return {"status": "already_completed"}

    # Send the email
    await email_service.send(to=email, subject=subject)

    # Mark as done (with 7-day TTL for the dedup key)
    await redis.setex(f"task_done:{task_id}", 604800, "1")
    return {"status": "sent"}

task_scheduler_worker.py — polling workerPython
import asyncio, uuid

class SchedulerWorker:
    def __init__(self, worker_id: str = None):
        self.worker_id = worker_id or str(uuid.uuid4())[:8]
        self.running = False

    async def start(self):
        self.running = True
        while self.running:
            await self._poll_and_execute()
            await asyncio.sleep(5)   # poll every 5 seconds

    async def _poll_and_execute(self):
        now_ms = int(time.time() * 1000)

        # 1. Find due tasks
        tasks = await db.fetch(
            "SELECT * FROM scheduled_tasks WHERE status='PENDING' "
            "AND next_run_at <= $1 ORDER BY next_run_at LIMIT 10", now_ms
        )

        for task in tasks:
            # 2. Try to claim (CAS update — prevents double execution)
            claimed = await db.execute(
                "UPDATE scheduled_tasks SET status='RUNNING', "
                "worker_id=$1, heartbeat_at=$2 "
                "WHERE task_id=$3 AND status='PENDING'",
                self.worker_id, now_ms, task["task_id"]
            )
            if not claimed: continue  # another worker got it

            # 3. Execute in background (don't block polling loop)
            asyncio.create_task(self._execute(task))

    async def _execute(self, task: dict):
        try:
            async with httpx.AsyncClient(timeout=30) as client:
                await client.post(task["handler_url"],
                    json={"task_id": task["task_id"], **task["payload_json"]})

            # 4. Mark done (or reschedule if recurring)
            if task["task_type"] == "recurring":
                next_run = next_run_at_ms(task["cron_expr"])
                await db.execute(
                    "UPDATE scheduled_tasks SET status='PENDING', next_run_at=$1 WHERE task_id=$2",
                    next_run, task["task_id"])
            else:
                await db.execute("UPDATE scheduled_tasks SET status='DONE' WHERE task_id=$1", task["task_id"])

        except Exception as e:
            attempts = task["attempts"] + 1
            status   = "FAILED" if attempts >= task["max_attempts"] else "PENDING"
            next_run = int(time.time() * 1000) + (60_000 * attempts)  # exponential backoff
            await db.execute("UPDATE scheduled_tasks SET status=$1, attempts=$2, next_run_at=$3 WHERE task_id=$4",
                            status, attempts, next_run, task["task_id"])

Outcome
  • CAS update is the distributed mutex for task claiming — UPDATE WHERE status='PENDING' atomically claims a task. If 10 workers try simultaneously: only 1 gets affected_rows=1. The rest get 0 and skip. Zero distributed lock overhead.
  • Heartbeat + watchdog handles worker crashes — worker writes heartbeat every 30s. Watchdog finds tasks where heartbeat is >60s old → reset to PENDING. Combination guarantees no task is permanently lost even if the worker dies.
  • Exactly-once requires idempotent handlers — the scheduler can guarantee at-most-once claiming, but the handler may still execute twice (crash after execute, before marking DONE). Make handlers idempotent using task_id as dedup key.
  • INDEX (status, next_run_at) is the critical query optimization — the polling query scans only PENDING tasks sorted by execution time. Without this index: full table scan every 5 seconds. With it: O(log N) per poll.
  • Exponential backoff for retries prevents thundering herds — failed tasks retry after 1min, 2min, 4min. Prevents hammering a failing external service. Combined with max_attempts cap prevents infinite retries.

Designing & Implementing Flash Sale

A flash sale (Flipkart Big Billion Day, Xiaomi Mi sales) offers limited quantity products at steep discounts for a short window. The system must handle a massive traffic spike at t=0, prevent overselling, ensure fairness, and protect the rest of the platform from being overwhelmed.

Why Flash Sales Are Hard

Thundering herd at t=0

5 million users click "Buy" simultaneously at noon. Every request hits the checkout API, database, payment service at the same instant. Most systems collapse under 100× normal load. Need to absorb the spike before it reaches the backend.

Inventory correctness

5000 units available. If 10M requests hit the DB simultaneously, naive implementation sells 10M units. Must atomically decrement inventory and refuse requests once stock is exhausted. No race conditions.

Core Architecture: Queue-based Admission

Key insight: Don't let all 10M users hit the checkout flow. Gate them with a virtual queue. Only 5000 users actually need to go through checkout. The rest wait in queue and are told "unavailable" gracefully when inventory runs out. This is exactly how Xiaomi, Flipkart, and Ticketmaster handle flash sales.
Queue-based Admission Architecture

Virtual Queue Design

1
All users join a virtual queue at t=0
Instead of hitting checkout: POST /flash-sale/join. Redis LPUSH to queue:sale_id. Returns queue position and estimated wait time. Client shows queue page — no backend pressure.
2
Rate-controlled admission to checkout
A dispatcher process pulls N users from queue every second. N = checkout_capacity (e.g., 200/sec based on inventory remaining and checkout completion rate). Issues a time-limited token (Redis key, 10-minute TTL).
3
Admitted users checkout with token
Checkout page loads only for users with valid token. Token checked on entry: if valid and unused → proceed to inventory check and payment. Invalid token → redirect back to queue.
4
Inventory check: Redis atomic decrement
DECR inventory:product:X. If result ≥ 0 → reserved. If result < 0 → INCR back + return "sold out". This is atomic — no lock needed, no race condition possible.
5
Complete purchase and confirm
Payment processed, order created. Inventory decremented in Redis confirmed in MySQL. Queue position holders beyond inventory limit → notify "Sold out, sorry!"

Protecting the Main Platform

Circuit Breaker

Flash sale services run on isolated infrastructure (separate Redis, separate DB shard, separate app servers). If the flash sale crushes its own infra, circuit breakers prevent it from cascading to the main product catalog, user auth, or payment systems.

CDN for Static Content

Product page, countdown timer, queue waiting page — all served from CDN, not from origin servers. 99% of pre-sale traffic is reading the product page. Move that to CDN → remove it from origin entirely.

inventory.py — atomic reserve with Redis LuaPython
import redis.asyncio as aioredis

redis = aioredis.Redis(host="redis")

-- Lua script: atomic check + decrement (no race condition possible)
RESERVE_SCRIPT = """
local current = redis.call('GET', KEYS[1])
if current == false or tonumber(current) <= 0 then
    return 0  -- sold out
end
redis.call('DECR', KEYS[1])
return 1  -- reserved
"""

async def reserve_unit(product_id: str) -> bool:
    """Atomically reserve one unit. Returns True if reserved, False if sold out."""
    key    = f"inventory:flash:{product_id}"
    result = await redis.eval(RESERVE_SCRIPT, 1, key)
    return bool(result)

async def release_unit(product_id: str):
    """Release reserved unit (payment failed, timeout)."""
    await redis.incr(f"inventory:flash:{product_id}")

Fairness — Preventing Bot Advantage

Rate limiting per user

Max 1 queue-join attempt per user per sale. Max 1 purchase per user per sale. Enforced via Redis SET with sale_id+user_id as key. Prevents one user from grabbing multiple slots.

Bot detection

Require CAPTCHA before joining queue. Fingerprint browser (user-agent, canvas hash, TLS fingerprint). Flag suspicious patterns (joins from 1000 accounts on the same IP). Block known bot farms pre-sale.

Purchase reservation timeout

Once admitted to checkout, user has 10 minutes to complete purchase. After 10 minutes: reservation released, unit returns to pool, next person in queue is admitted. Prevents holding inventory without purchase intent.

Code — Flash Sale System
flash_sale.py — queue join + checkout admissionPython
import time, uuid, asyncio
from fastapi import FastAPI, HTTPException

app = FastAPI()

@app.post("/flash-sale/{sale_id}/join")
async def join_queue(sale_id: str, user_id: int):
    """User clicks Buy → joins virtual queue. Fast, O(1)."""
    # 1. Prevent duplicate queue join
    already_in = await redis.get(f"queued:{sale_id}:{user_id}")
    if already_in: return {"status": "already_queued", "position": int(already_in)}

    # 2. Check if inventory is already exhausted
    remaining = await redis.get(f"inventory:flash:{sale_id}")
    if remaining and int(remaining) <= 0:
        return {"status": "sold_out"}

    # 3. Add to queue (RPUSH = append to right = FIFO queue)
    queue_key = f"queue:{sale_id}"
    await redis.rpush(queue_key, user_id)
    position  = await redis.llen(queue_key)

    # 4. Record that this user is queued
    await redis.setex(f"queued:{sale_id}:{user_id}", 3600, position)

    return {"status": "queued", "position": position,
            "estimated_wait_secs": position // 200}

@app.post("/flash-sale/{sale_id}/admit")
async def admit_next_batch(sale_id: str, batch_size: int = 200):
    """Called by dispatcher every second. Admits batch_size users to checkout."""
    queue_key = f"queue:{sale_id}"
    admitted  = []
    for _ in range(batch_size):
        user_id = await redis.lpop(queue_key)   # LPOP = FIFO
        if not user_id: break
        token = str(uuid.uuid4())
        await redis.setex(
            f"checkout_token:{sale_id}:{user_id.decode()}",
            600, token   # 10 minute checkout window
        )
        admitted.append((int(user_id), token))
    return {"admitted": len(admitted)}

@app.post("/flash-sale/{sale_id}/purchase")
async def purchase(sale_id: str, user_id: int, checkout_token: str):
    """Complete purchase. Must have valid checkout token."""
    token_key = f"checkout_token:{sale_id}:{user_id}"
    stored    = await redis.get(token_key)
    if not stored or stored.decode() != checkout_token:
        raise HTTPException(403, "Invalid or expired checkout token")

    if not await reserve_unit(sale_id):
        raise HTTPException(409, "Sold out")

    await redis.delete(token_key)   # single-use token
    return {"status": "reserved", "proceed_to": "payment"}
