Database Replication & Failover
Design a robust MySQL/PostgreSQL replication system — WAL/binlog streaming to replicas, GTID-based position tracking, replication lag monitoring, read replica routing, semi-synchronous vs async replication tradeoffs, and automatic failover using Orchestrator/Patroni that elects a new primary and updates routing in under 30 seconds without data loss.
🔴 MUST DO 🟣 Hard Binlog / WAL GTID Replication Lag Failover Split-Brain Read Replicas
🎯 Requirements
Database replication is about two things: scaling reads and surviving failures. The hard problems are: how much data can you afford to lose when the primary dies (RPO), how quickly must the system recover (RTO), and how do you prevent split-brain — two nodes both thinking they're the primary and accepting writes independently?

FUNCTIONAL
Stream writes from primary to N replicas in near-realtime
Route read traffic to replicas; write traffic to primary
Monitor replication lag; route away from lagging replicas
Detect primary failure; elect new primary; update routing
Prevent split-brain: only one primary accepting writes at a time
NON-FUNCTIONAL
RPO (data loss): 0 seconds with semi-sync; <1s with async
RTO (recovery time): <30 seconds to elect new primary
Replication lag: <1 second under normal load
Read scale: 5 replicas × primary write throughput
Zero false failovers (split-brain prevention mandatory)
💬 Phase 1 — Replication Mechanics
PHASE 1 · BINLOG / WAL STREAMING
IR
INTERVIEWER
How does MySQL replication actually work under the hood?
ME
YOU
MySQL replication uses the binary log (binlog) — a sequential log of all changes made to the database. Three components:

1. Primary binlog thread: Every committed write is recorded to the binlog as an event: row-based (exact before/after image of every row changed), statement-based (the SQL statement), or mixed. Row-based is safer — no ambiguity about what changed.

2. Replica I/O thread: Connects to the primary, requests events starting from its last known position. Primary sends binlog events as a stream. Replica writes them to its relay log.

3. Replica SQL thread: Reads from the relay log, applies events to the replica's data. The gap between the primary's current binlog position and the replica's applied position is the replication lag.

GTID (Global Transaction ID): Each transaction gets a globally unique ID: {server_uuid}:{seq_no}. This replaces binlog file+offset position tracking. Benefit: if the primary fails and you promote a replica, you don't need to calculate the new replication position — every transaction has a unique ID, and the new replicas simply replay transactions they haven't seen yet. This makes failover much simpler and safer.

PostgreSQL equivalent: WAL (Write-Ahead Log) streaming replication. Same concept — the WAL is the authoritative log of all changes; replicas stream from the primary's WAL sender process. PostgreSQL uses LSN (Log Sequence Number) instead of GTID.
💡 Always use GTID-based replication in production. Binlog position-based replication requires careful manual position tracking during failover — easy to make errors that corrupt replication. GTID makes failover safe by making transaction identity globally unique.
⚡ Phase 2 — Failover Without Data Loss
PHASE 2 · FAILOVER
IR
INTERVIEWER
The primary dies at 2am. Walk me through exactly what happens over the next 30 seconds.
ME
YOU
Using Orchestrator (MySQL) or Patroni (PostgreSQL):

T+0s: Primary crashes. Health checks from Orchestrator to the primary start timing out.
T+5s: After 3 consecutive missed health checks (to avoid false positives from network blips), Orchestrator declares the primary dead. Cross-checks with multiple Orchestrator agents to prevent split-brain detection.
T+10s: Candidate selection. Orchestrator evaluates all replicas: which has the most up-to-date GTID set? Most replicated = elected candidate. In semi-sync setups, the replica that acknowledged the last transaction is always current.
T+15s: Candidate promotion. Orchestrator stops remaining replicas from reading from the old primary (to prevent divergence), tells the candidate to stop replication and promote itself as primary (SET GLOBAL read_only=OFF).
T+20s: Other replicas are re-pointed to the new primary: CHANGE MASTER TO MASTER_HOST='new-primary' USING GTID. GTID automatically figures out where to start replication — no manual position tracking.
T+25s: DNS/ProxySQL routing updated. Application connections that retry automatically hit the new primary. Applications that don't retry get an error and must reconnect.
T+30s: New primary fully operational. Read replicas lagging slightly (re-syncing from new primary). Alert fired to on-call engineer.
🧠 Phase 3 — Split-Brain Prevention
PHASE 3 · SPLIT-BRAIN
IR
INTERVIEWER
The primary becomes unreachable due to a network partition — it's still running fine but the replicas can't see it. They elect a new primary. Now we have two primaries accepting writes. How do you prevent this?
ME
YOU
Split-brain is the worst failure mode in databases — you end up with two diverged write streams that are nearly impossible to reconcile. Multiple defenses:

1. Quorum-based election: A new primary is only elected if a majority of known nodes agree the primary is dead. With 1 primary + 4 replicas = 5 nodes, need 3 votes. If the primary is partitioned but alive, it can't get 3 votes from replicas it can't see — so no split election possible. Orchestrator uses a quorum of Orchestrator agents across different availability zones.

2. STONITH (Shoot The Other Node In The Head): Before promoting a new primary, fence the old one — cut its network access via cloud security group modification, or power it off via IPMI. This ensures the old primary cannot accept any more writes. More aggressive but eliminates split-brain entirely.

3. Semi-synchronous replication: With semi-sync, the primary waits for at least one replica to acknowledge each transaction before committing. If all replicas are unreachable for >rpl_semi_sync_master_timeout, the primary reverts to async mode. This means the primary can't commit during a network partition to replicas — it essentially blocks writes. The system degrades gracefully rather than silently diverging.

4. Lease-based primary lock: Patroni uses etcd/ZooKeeper leases. The primary holds a lease and must renew it every 10s. If it can't renew (network partition to etcd), it voluntarily demotes itself. No other node can become primary without acquiring the lease. This requires the primary to have connectivity to etcd — adding a dependency but guaranteeing no split-brain.
⚠ The golden rule: It is safer to have no primary (cluster refuses writes) than to have two primaries (data diverges silently). Always choose availability of reads over availability of writes when facing uncertainty about whether the primary is alive.

// DB Replication Monitor — Java: Lag check + failover
// ══════════════════════════════════════════════════════

@Service
public class ReplicationMonitor {

    @Autowired private DataSource         primaryDs;
    @Autowired private List<DataSource>    replicaDsList;
    @Autowired private ProxySQLClient      proxySQL;
    @Autowired private AlertService        alerts;

    private static final int  LAG_THRESHOLD_S  = 5;
    private static final int  HEALTH_FAIL_LIMIT = 3;
    private Map<String,Integer> consecutiveFails = new ConcurrentHashMap<>();

    /** Check replication lag for all replicas every 5 seconds. */
    @Scheduled(fixedRate = 5000)
    public void checkReplicationLag() {
        replicaDsList.forEach(ds -> {
            try (Connection conn = ds.getConnection();
                 Statement  st   = conn.createStatement();
                 ResultSet  rs   = st.executeQuery("SHOW SLAVE STATUS")) {

                if (!rs.next()) return;

                String replicaId = rs.getString("Master_Host");
                int    lagSeconds = rs.getInt("Seconds_Behind_Master");
                String ioRunning  = rs.getString("Slave_IO_Running");
                String sqlRunning = rs.getString("Slave_SQL_Running");
                String gtidExec   = rs.getString("Executed_Gtid_Set");

                // Metric to Prometheus
                Metrics.gauge("db_replica_lag_seconds",
                    Tags.of("replica", replicaId)).record(lagSeconds);

                // Route away from lagging replicas
                if (lagSeconds > LAG_THRESHOLD_S || !"Yes".equals(ioRunning)) {
                    proxySQL.setHostGroupWeight(replicaId, 0); // remove from read pool
                    alerts.warn("Replica lagging: " + replicaId + " lag=" + lagSeconds + "s");
                } else {
                    proxySQL.setHostGroupWeight(replicaId, 100); // restore to pool
                }

                consecutiveFails.put(replicaId, 0); // health restored

            } catch (SQLException e) {
                String replicaId = getReplicaId(ds);
                int fails = consecutiveFails.merge(replicaId, 1, Integer::sum);
                if (fails >= HEALTH_FAIL_LIMIT) {
                    proxySQL.setHostGroupWeight(replicaId, 0);
                    alerts.critical("Replica unreachable: " + replicaId);
                }
            }
        });
    }

    /** Check primary health; alert Orchestrator on failure. */
    @Scheduled(fixedRate = 1000)
    public void checkPrimaryHealth() {
        try (Connection conn = primaryDs.getConnection();
             PreparedStatement ps = conn.prepareStatement("SELECT 1")) {
            ps.executeQuery();
            consecutiveFails.put("primary", 0);
        } catch (SQLException e) {
            int fails = consecutiveFails.merge("primary", 1, Integer::sum);
            if (fails >= HEALTH_FAIL_LIMIT) {
                alerts.critical("PRIMARY UNREACHABLE — triggering failover");
                // In production: Orchestrator handles the actual failover
                // We just alert; never manually failover from application code
            }
        }
    }
}

@Aspect
@Component
public class DataSourceRoutingAspect {

    @Around("@annotation(readOnly)")
    public Object routeToReplica(ProceedingJoinPoint pjp, ReadOnly readOnly) throws Throwable {
        DataSourceContextHolder.setReadOnly(true);   // route to replica pool
        try { return pjp.proceed(); }
        finally { DataSourceContextHolder.setReadOnly(false); }
    }
}

# DB Replication Monitor — Python

import time, threading, pymysql
from dataclasses import dataclass
from typing import Dict, List, Optional

LAG_THRESHOLD_S  = 5
HEALTH_FAIL_LIMIT = 3


@dataclass
class ReplicaStatus:
    host:        str
    lag_seconds: Optional[int]
    io_running:  bool
    sql_running: bool
    gtid_set:    str
    healthy:     bool


def check_replica(host: str, port: int = 3306) -> ReplicaStatus:
    """Run SHOW SLAVE STATUS on a replica and return structured status."""
    try:
        conn = pymysql.connect(host=host, port=port, user="monitor",
                               password="secret", connect_timeout=3)
        with conn.cursor(pymysql.cursors.DictCursor) as cur:
            cur.execute("SHOW SLAVE STATUS")
            row = cur.fetchone()
        conn.close()

        if not row:
            return ReplicaStatus(host=host, lag_seconds=None, io_running=False,
                                    sql_running=False, gtid_set="", healthy=False)

        lag = row.get("Seconds_Behind_Master")
        return ReplicaStatus(
            host=host,
            lag_seconds=int(lag) if lag is not None else None,
            io_running=row["Slave_IO_Running"] == "Yes",
            sql_running=row["Slave_SQL_Running"] == "Yes",
            gtid_set=row.get("Executed_Gtid_Set", ""),
            healthy=row["Slave_IO_Running"] == "Yes"
                    and row["Slave_SQL_Running"] == "Yes"
                    and (lag is None or int(lag) <= LAG_THRESHOLD_S)
        )
    except Exception as e:
        print(f"[ERROR] Cannot reach replica {host}: {e}")
        return ReplicaStatus(host=host, lag_seconds=None, io_running=False,
                                sql_running=False, gtid_set="", healthy=False)


class ReplicationMonitor:
    def __init__(self, replicas: List[str]):
        self.replicas      = replicas
        self.fail_counts:  Dict[str, int]  = {h: 0 for h in replicas}
        self.healthy_pool: List[str]       = list(replicas)
        self._lock = threading.Lock()

    def monitor_loop(self, interval: int = 5):
        while True:
            for host in self.replicas:
                status = check_replica(host)
                with self._lock:
                    if status.healthy:
                        self.fail_counts[host] = 0
                        if host not in self.healthy_pool:
                            self.healthy_pool.append(host)
                            print(f"[RESTORED] {host} back in read pool")
                    else:
                        self.fail_counts[host] += 1
                        if self.fail_counts[host] >= HEALTH_FAIL_LIMIT:
                            if host in self.healthy_pool:
                                self.healthy_pool.remove(host)
                                lag_str = f"lag={status.lag_seconds}s" if status.lag_seconds else "unreachable"
                                print(f"[REMOVED] {host} from read pool ({lag_str})")
            time.sleep(interval)

    def get_read_replica(self) -> Optional[str]:
        """Round-robin across healthy replicas."""
        with self._lock:
            return self.healthy_pool[int(time.time()) % len(self.healthy_pool)] if self.healthy_pool else None


# ── Patroni-style failover using etcd ────────────────
def acquire_primary_lease(etcd_client, node_id: str, ttl: int = 10) -> bool:
    """Acquire the primary lease. Returns True if this node is primary."""
    try:
        # etcd compare-and-set: only set if key doesn't exist
        resp = etcd_client.transaction(
            compare=[etcd_client.transactions.version("/primary") == 0],
            success=[etcd_client.transactions.put("/primary", node_id, lease=ttl)],
            failure=[]
        )
        return resp.succeeded
    except: return False


def primary_heartbeat_loop(etcd_client, node_id: str, lease_id: int, interval: int = 3):
    """Renew the primary lease every 3s. If we can't, voluntarily demote."""
    while True:
        try:
            etcd_client.refresh_lease(lease_id)  # renew TTL
        except Exception:
            print("[DEMOTE] Cannot renew lease — network issue. Stepping down as primary.")
            # Set read-only: prevent new writes
            break
        time.sleep(interval)
⚖️ Tradeoffs
🔄 Replication Mode: Async vs Semi-sync vs Sync
Async (default MySQL)

Primary commits without waiting for replica ack. Zero write latency overhead. RPO = up to seconds of data loss on primary failure. Standard for write-heavy workloads where some data loss is acceptable (e.g., analytics, logging). Most commonly deployed due to simplicity.

✅ Semi-sync (1 ACK)

Primary waits for at least one replica to write to its relay log before committing. Guarantees zero data loss on failover (at least 1 replica always has the latest transaction). Write latency +0.5-2ms (one network RTT). Fallback to async if no replica acks within timeout. Best balance of durability and performance.

Fully Sync (all replicas)

Primary waits for ALL replicas to ack. Zero data loss guaranteed. Write latency = slowest replica × overhead. Impractical at scale — a single slow replica blocks all writes. Used by MySQL Cluster (NDB) and only when absolute durability is required (financial transactions). Usually replaced by semi-sync + monitoring.

🛡 Failover Tool: Orchestrator vs Patroni vs AWS RDS
✅ Orchestrator (MySQL)

Purpose-built for MySQL HA. Understands replication topology. Automatic failover with quorum detection. Splits-brain safe via Raft-based agent quorum. Used by GitHub, Booking.com. Open source. Requires operational expertise. Best for on-prem or self-managed MySQL at scale.

✅ Patroni (PostgreSQL)

Template for PostgreSQL HA. Uses etcd/ZooKeeper/Consul for distributed consensus (primary lease). Automatic failover in <30s. REST API for manual operations. Industry standard for self-managed Postgres. Used by Zalando, GitLab. Requires etcd cluster as a dependency.

AWS RDS Multi-AZ

Fully managed failover. Synchronous replica in another AZ. Automatic failover in 60-120s (slower than Orchestrator/Patroni). No operational overhead — AWS handles everything. Cost: 2× instance cost for standby. Best for teams without DB operations expertise or low tolerance for operational complexity.
