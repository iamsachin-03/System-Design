Multi-Region Active-Active
Design a system active in 3 regions simultaneously — all regions serve reads and writes, cross-region replication propagates changes in under 5 seconds, concurrent writes to the same record are resolved via CRDTs or LWW with hybrid logical clocks, and an entire region can fail without user-visible downtime beyond a brief DNS failover window.
🔴 MUST DO 🟣 Hard CRDT LWW + HLC Conflict Resolution GeoDNS Data Sovereignty CAP Theorem
🎯 Requirements
Active-active means every region serves both reads AND writes. The core challenge: two users in different regions can simultaneously modify the same record. You must choose — prevent this (coordinate globally = high latency), detect and resolve (eventual consistency + conflict resolution), or design data models that make conflicts impossible (shard by user/region).

FUNCTIONAL
Users in any region get <50ms read/write latency (local DB)
Data replicates to all regions asynchronously (<5s lag)
Concurrent writes to same record: deterministic conflict resolution
Region failure: traffic rerouted to surviving regions within 30s
EU user data stays in EU region (GDPR data sovereignty)
NON-FUNCTIONAL
3 regions: us-east-1, eu-west-1, ap-southeast-1
Availability: 99.999% (5 nines) — no region is SPOF
CAP choice: AP (availability + partition tolerance), eventual consistency
Conflict rate: <0.1% of writes in typical workloads
Cross-region replication: dedicated backbone, not public internet
💬 Phase 1 — Conflict Resolution
PHASE 1 · WRITE CONFLICTS
IR
INTERVIEWER
User A in London and User B in Singapore both update the same record at exactly the same millisecond. Walk me through what happens end-to-end.
ME
YOU
Both writes land at their local region within milliseconds. EU-west accepts User A's write, AP-southeast accepts User B's write. Both replicate toward each other asynchronously. When EU-west receives the AP write (and vice versa), they detect a conflict: same record, different values, overlapping timestamps.

Strategy 1 — Last-Write-Wins with Hybrid Logical Clock (LWW+HLC):
Each write carries a Hybrid Logical Clock timestamp: HLC = max(physical_clock, last_known_HLC) + epsilon. This guarantees causality even when clocks drift. On conflict: the write with the higher HLC timestamp wins. Loser is discarded. Simple. Works well for scalar fields (user name, product status). The losing write is gone forever — acceptable for most fields.

Strategy 2 — CRDTs (Conflict-free Replicated Data Types):
For commutative operations, design a data structure where merge is automatic and lossless. Example: shopping cart uses an OR-Set CRDT — "add item A" and "add item B" from different regions both survive, merged to {A, B}. View counters use PN-Counter CRDT — each region has its own increment register, merged value is the sum. No conflict possible by construction.

Strategy 3 — Home region per record:
Route writes to the record's "home" region (e.g., hash(user_id) → region). Only one region ever writes a given record — conflicts are structurally impossible. Other regions hold read replicas. Trade-off: a US user editing a record homed in EU has 100ms extra latency for the write. This is what most real-world systems do for critical records (financial transactions, inventory).
💡 In practice, multi-region active-active systems use all three strategies simultaneously: home region for critical mutable data (payments, inventory), CRDTs for counters and sets, LWW+HLC for user profile fields. The art is choosing the right strategy for each data type.
🌐 Phase 2 — Geo Routing & Failover
PHASE 2 · FAILOVER
IR
INTERVIEWER
EU-west goes completely dark at 3am. Walk me through what happens to EU users.
ME
YOU
T+0s: EU-west region stops responding. Load balancer health checks start timing out.
T+10s: After 3 failed health checks, Route53 (or Cloudflare) marks EU endpoints unhealthy.
T+30s: DNS TTL expires for EU users. Their next DNS lookup returns US-east IP instead of EU IP. EU users now connect to US-east.
T+30s-60s: US-east serves EU users with the last-replicated data — at most 5 seconds stale (our replication SLA). New writes from EU users go to US-east and replicate to AP-southeast.
Data sovereignty concern: EU user data is now flowing through US-east, potentially violating GDPR. For PII, the correct choice is to fail closed — return 503 rather than serve from a non-EU region. For non-PII data, US-east can serve. This is a business/legal decision, not a technical one.
EU recovery: When EU-west comes back, it needs to replay all writes that happened while it was down (from US-east's replication log). DynamoDB Global Tables and Cassandra multi-DC both handle this automatically via their replication protocols.
📊 Phase 3 — CAP Theorem in Practice
PHASE 3 · CAP TRADEOFFS
IR
INTERVIEWER
CAP theorem says you can only have 2 of 3 — consistency, availability, partition tolerance. What does this mean for your design?
ME
YOU
Network partitions between regions are a fact of life — transatlantic cable cuts happen. So partition tolerance (P) is mandatory. We choose between C and A:

CP (consistency + partition tolerance): When a partition occurs, refuse writes until you can confirm the record's state across regions. This means global coordination on every write — 100-200ms cross-region latency added to every write. Correct but slow. Used by Google Spanner via TrueTime (GPS-synchronized clocks allowing globally consistent transactions). Appropriate for financial systems where consistency is non-negotiable.

AP (availability + partition tolerance): Continue accepting writes in all regions even during a partition. Resolve conflicts when partition heals. This is what DynamoDB Global Tables, Cassandra, and CockroachDB with async replication provide. Write latency stays low (<50ms local). Correctness depends on conflict resolution strategy. Appropriate for most internet-scale applications where some staleness is acceptable.

Our choice: AP with careful conflict-avoidance design. Home-region routing eliminates most real-world conflicts. CRDTs handle counters and sets perfectly. LWW+HLC handles the rare remaining conflicts. For the truly critical operations (payments, seat reservations), we route to a single authoritative region — accepting 100ms extra latency for those specific transactions.

// ── PN-Counter CRDT ───────────────────────────────────
public class PNCounter {
    private final Map<String,Long> increments = new ConcurrentHashMap<>();
    private final Map<String,Long> decrements = new ConcurrentHashMap<>();
    private final String              regionId;

    public PNCounter(String regionId) { this.regionId = regionId; }

    public void increment(long delta) {
        increments.merge(regionId, delta, Long::sum);
    }

    public void decrement(long delta) {
        decrements.merge(regionId, delta, Long::sum);
    }

    public long value() {
        long inc = increments.values().stream().mapToLong(Long::longValue).sum();
        long dec = decrements.values().stream().mapToLong(Long::longValue).sum();
        return inc - dec;
    }

    /** Merge another region's counter into this one (element-wise max). */
    public void merge(PNCounter other) {
        other.increments.forEach((k, v) -> increments.merge(k, v, Math::max));
        other.decrements.forEach((k, v) -> decrements.merge(k, v, Math::max));
    }
}

// ── Hybrid Logical Clock ──────────────────────────────
public class HybridLogicalClock {

    private volatile long  logicalTime = 0;
    private volatile int   counter     = 0;

    public synchronized long[] tick() {
        long physical = System.currentTimeMillis();
        if (physical > logicalTime) {
            logicalTime = physical; counter = 0;
        } else {
            counter++;
        }
        return new long[] { logicalTime, counter };
    }

    public synchronized void update(long msgL, int msgC) {
        long physical = System.currentTimeMillis();
        long maxL     = Math.max(physical, Math.max(logicalTime, msgL));
        if (maxL == logicalTime && maxL == msgL)      counter = Math.max(counter, msgC) + 1;
        else if (maxL == logicalTime)                 counter++;
        else if (maxL == msgL)                        counter = msgC + 1;
        else                                          counter = 0;
        logicalTime = maxL;
    }
}

// ── LWW Register — Last-Write-Wins field ─────────────
public class LWWRegister<T> {

    private T      value;
    private long   hlcL;
    private int    hlcC;
    private String siteId; // region ID for tie-breaking

    public void set(T newValue, long[] hlcTimestamp, String region) {
        value = newValue; hlcL = hlcTimestamp[0]; hlcC = (int) hlcTimestamp[1]; siteId = region;
    }

    /** Merge: the write with the higher HLC timestamp wins. */
    public void merge(LWWRegister<T> other) {
        int cmp = Long.compare(other.hlcL, this.hlcL);
        if (cmp > 0) { accept(other); return; }  // other is newer
        if (cmp < 0) return;                        // we are newer
        // Same logical time: compare counter, then siteId for determinism
        if (other.hlcC > this.hlcC || (other.hlcC == this.hlcC && other.siteId.compareTo(this.siteId) > 0)) {
            accept(other);
        }
    }

    private void accept(LWWRegister<T> other) {
        value = other.value; hlcL = other.hlcL; hlcC = other.hlcC; siteId = other.siteId;
    }

    public T get() { return value; }
}

// ── Replication: change stream → other regions ────────
@Service
public class CrossRegionReplicator {

    @Autowired private KafkaTemplate<String,ChangeEvent> kafka;

    /** Publish every write to a change topic consumed by other regions. */
    public void publishChange(String entityId, Object newValue, long[] hlc) {
        kafka.send("global-changes", entityId,
            new ChangeEvent(entityId, newValue, hlc, REGION_ID));
    }

    @KafkaListener(topics = "global-changes")
    public void onRemoteChange(ChangeEvent event) {
        if (event.getOriginRegion().equals(REGION_ID)) return; // skip own events
        mergeIntoLocalStore(event); // apply CRDT merge or LWW resolution
    }
}

# Multi-Region — Python: PN-Counter CRDT + LWW+HLC
# ══════════════════════════════════════════════════════

import time
from dataclasses import dataclass, field
from typing import Dict, TypeVar, Generic, Optional

T = TypeVar('T')

# ── PN-Counter CRDT ───────────────────────────────────
class PNCounter:
    """Per-region increment + decrement counters. Merge = element-wise max."""
    def __init__(self, region_id: str):
        self.region_id  = region_id
        self.increments: Dict[str, int] = {region_id: 0}
        self.decrements: Dict[str, int] = {region_id: 0}

    def increment(self, delta: int = 1):
        self.increments[self.region_id] = self.increments.get(self.region_id, 0) + delta

    def decrement(self, delta: int = 1):
        self.decrements[self.region_id] = self.decrements.get(self.region_id, 0) + delta

    def value(self) -> int:
        return sum(self.increments.values()) - sum(self.decrements.values())

    def merge(self, other: 'PNCounter'):
        """Element-wise max merge — always safe to call in any order."""
        for k, v in other.increments.items():
            self.increments[k] = max(self.increments.get(k, 0), v)
        for k, v in other.decrements.items():
            self.decrements[k] = max(self.decrements.get(k, 0), v)


# ── Hybrid Logical Clock ──────────────────────────────
class HLC:
    """Hybrid Logical Clock: max(physical, last_seen) + logical counter."""
    def __init__(self):
        self.l = 0  # logical time (ms)
        self.c = 0  # logical counter

    def tick(self) -> tuple:
        p = int(time.time() * 1000)
        if p > self.l: self.l = p; self.c = 0
        else:          self.c += 1
        return (self.l, self.c)

    def update(self, msg_l: int, msg_c: int):
        p   = int(time.time() * 1000)
        max_l = max(p, self.l, msg_l)
        if max_l == self.l == msg_l: self.c = max(self.c, msg_c) + 1
        elif max_l == self.l:         self.c += 1
        elif max_l == msg_l:          self.c = msg_c + 1
        else:                          self.c = 0
        self.l = max_l


# ── LWW Register ─────────────────────────────────────
@dataclass
class LWWRegister(Generic[T]):
    value:   Optional[T] = None
    hlc_l:   int         = 0
    hlc_c:   int         = 0
    site_id: str         = ""

    def set(self, value: T, hlc: tuple, site: str):
        self.value, self.hlc_l, self.hlc_c, self.site_id = value, hlc[0], hlc[1], site

    def merge(self, other: 'LWWRegister[T]'):
        """Higher HLC wins. Tie-break: higher hlc_c, then lexicographic site_id."""
        other_tuple = (other.hlc_l, other.hlc_c, other.site_id)
        self_tuple  = (self.hlc_l, self.hlc_c, self.site_id)
        if other_tuple > self_tuple:
            self.value, self.hlc_l, self.hlc_c, self.site_id = (
                other.value, other.hlc_l, other.hlc_c, other.site_id)


# ── OR-Set CRDT (shopping cart) ──────────────────────
import uuid

class ORSet:
    """Observed-Remove Set: concurrent add+remove = add wins."""
    def __init__(self): self.elements: Dict[str, set] = {}

    def add(self, item: str):
        tag = str(uuid.uuid4())
        self.elements.setdefault(item, set()).add(tag)

    def remove(self, item: str):
        self.elements.pop(item, None)

    def contains(self, item: str) -> bool:
        return bool(self.elements.get(item))

    def merge(self, other: 'ORSet'):
        for item, tags in other.elements.items():
            self.elements.setdefault(item, set()).update(tags)

    def items(self) -> list:
        return [k for k, v in self.elements.items() if v]


# ── Demo: concurrent cart adds from 2 regions ─────────
if __name__ == "__main__":
    us_cart = ORSet(); eu_cart = ORSet()
    us_cart.add("shoes"); us_cart.add("hat")
    eu_cart.add("jacket")
    eu_cart.remove("hat")  # EU didn't see "hat" yet — add wins after merge

    us_cart.merge(eu_cart); eu_cart.merge(us_cart)  # both regions merge
    print("US cart:", us_cart.items())  # ['shoes', 'hat', 'jacket']
    print("EU cart:", eu_cart.items())  # same — converged ✓
⚖️ Tradeoffs
🏛 Architecture: Active-Active vs Active-Passive vs Single-Region
Single-Region

Simple to build and operate. All data in one place — no conflict resolution. Latency: users far from the region see 100-300ms. Availability: region outage = total outage. Only acceptable for small-scale or latency-insensitive workloads. Never appropriate for global products.

Active-Passive

One primary region, other regions are read-only replicas. Reads are fast (local). Writes route to primary — 100-300ms overhead for non-primary users. Failover: promote a replica if primary fails (30-120s). Data sovereignty: primary is the only writable store — doesn't help EU writes. Simpler than active-active but still has write latency problem.

✅ Active-Active (chosen)

All regions serve reads AND writes. Low write latency for all users globally. Requires conflict resolution (CRDT / LWW / home-region). Highest complexity — the conflict resolution design is the hardest part. Used by Amazon (DynamoDB Global Tables), Cassandra, CockroachDB. Required for true global 5-nine availability.

📊 Database Choice: DynamoDB Global Tables vs Cassandra vs CockroachDB
✅ DynamoDB Global Tables

Fully managed multi-region active-active. AWS handles all replication, conflict resolution (LWW), and failover. Global secondar index support. No operational overhead. LWW conflict resolution is built-in — no custom CRDT code. High cost at scale. Proprietary — lock-in to AWS. Best for teams without operational expertise.

✅ Cassandra multi-DC

Open source, highly configurable. Each DC operates independently. Replication across DCs is async. Tunable consistency (LOCAL_QUORUM for low latency, EACH_QUORUM for stronger guarantees). LWT (lightweight transactions) for single-row linearizability. Used by Apple, Netflix, Discord. High operational complexity — requires expertise.

CockroachDB

Distributed SQL with serializability across regions. No conflict resolution needed — transactions are linearizable globally (like Spanner). Slower than Cassandra/DynamoDB for local writes (150ms+ cross-region latency for strong consistency). Best when you need SQL semantics globally and can accept higher write latency. Younger ecosystem.
