Distributed Task Scheduler
Design a system that runs millions of scheduled jobs reliably — cron expressions, one-time delayed tasks, recurring jobs — with exactly-once execution guarantees, failure recovery, at-least-once delivery fallback, distributed worker coordination, and the ability to scale to 1M concurrent scheduled tasks without a single point of failure.
🔴 MUST DO 🟣 Hard Exactly-Once Idempotency Leader Election Distributed Lock Cron at Scale Failure Recovery
🎯 Requirements
Distributed task scheduling is harder than it looks. The core challenge: with many scheduler nodes, how do you prevent the same job from firing twice at the same second? With worker failures, how do you ensure a job that was "in flight" gets re-executed without running twice in total? These are fundamentally distributed systems problems.

Schedule one-time jobs (execute at T+600s)

💬 Phase 1 — Preventing Double Execution
PHASE 1 · EXACTLY-ONCE
IR
INTERVIEWER
You have 5 scheduler nodes. All 5 check the jobs table at 12:00:00. All 5 see job X is due. How do you prevent all 5 from executing it?
ME
YOU
This is the thundering herd + double execution problem. Three solutions:

1. Database optimistic locking (SELECT FOR UPDATE SKIP LOCKED): All scheduler nodes run:
SELECT * FROM jobs WHERE next_run_at <= NOW() AND status = 'PENDING' FOR UPDATE SKIP LOCKED LIMIT 100
The FOR UPDATE takes a row lock; SKIP LOCKED means other transactions skip already-locked rows rather than waiting. Each scheduler node picks a disjoint set of jobs. First node to lock job X "wins" — others skip it. This is the simplest and most reliable approach. PostgreSQL and MySQL both support this pattern.

2. Distributed lock (Redis SETNX): Before executing a job, acquire a distributed lock:
SET lock:job:{job_id}:{run_id} 1 NX EX 300
Only one node acquires this lock (NX = only if not exists). The node that succeeds executes the job; others get a lock failure and skip. Lock expires after 300s (cleanup if executor dies). TTL must be longer than max job execution time.

3. Leader election (single scheduler): Elect one scheduler node as leader (via ZooKeeper or etcd). Only the leader scans and triggers jobs. Followers watch for leader failure and re-elect. Simplest correctness guarantee — no concurrent scheduling. Downside: single point of throughput (mitigated by leader batching and pushing work to a queue).
💡 Production approach: SKIP LOCKED for job claiming at the DB level + Redis distributed lock as a second guard before execution. Belt and suspenders — DB lock prevents most duplicates, Redis lock catches any edge cases.
🔄 Phase 2 — At-Least-Once + Idempotency
PHASE 2 · FAILURE RECOVERY
IR
INTERVIEWER
A worker picks up a job and crashes halfway through. How do you recover without executing the job twice?
ME
YOU
When a worker picks up a job, it sets status = RUNNING, locked_by = worker_id, lock_expires_at = NOW() + 300s.

Heartbeat mechanism: The worker sends a heartbeat every 30s: UPDATE jobs SET lock_expires_at = NOW() + 300s WHERE job_id = ?. If the worker dies, heartbeats stop and the lock expires.

Reaper process: A background reaper runs every 60s: SELECT * FROM jobs WHERE status = 'RUNNING' AND lock_expires_at < NOW(). For each found job: reset to status = PENDING for re-execution. The job will be picked up by another worker on the next scan.

The problem: This gives at-least-once execution — the job may have partially completed before the crash. If we re-run it, we might double-charge a customer, send two emails, etc.

Solution — idempotency keys: Jobs include an idempotency_key = {job_id}:{attempt_number}. The target service (e.g., payment processor) stores a record of this key and rejects duplicate requests: if idempotency_key was already processed, return the original result. For HTTP callbacks: pass the key in the X-Idempotency-Key header and have the target service handle dedup.

This combination gives us effective exactly-once semantics: the scheduler is at-least-once, but with idempotent targets, re-execution is safe.
⏳ Phase 3 — Cron at Scale
PHASE 3 · CRON SCHEDULING
IR
INTERVIEWER
You have 1M cron jobs, each with a different schedule. Some fire every minute (1M/min at worst). How do you store and efficiently query "which jobs are due in the next 10 seconds?"
ME
YOU
Key insight: we store the next computed run time as a concrete timestamp, not the cron expression. When a job fires, we compute the next run time from the cron expression and update the DB. Queries become simple timestamp comparisons.

Data model: jobs(job_id, cron_expr, next_run_at [INDEXED], status, last_run_at, ...). Index on next_run_at WHERE status = 'PENDING'.

Scanning pattern: Scheduler polls every 5s: SELECT ... WHERE next_run_at BETWEEN NOW() AND NOW() + 10s AND status = 'PENDING' FOR UPDATE SKIP LOCKED LIMIT 500. With a B-tree index on next_run_at, this query is O(log N + K) where K=results — extremely fast even at 1M jobs.

The thundering herd at minute boundaries: Every minute, potentially millions of "run every minute" jobs all have next_run_at = :00. With SKIP LOCKED, workers compete fairly. But at extreme scale (10M+ jobs firing at once), partition the jobs table: hash job_id modulo number of scheduler partitions. Each scheduler pod is responsible for only its partition. Linear scaling.

// Distributed Task Scheduler — Java: Scheduler + Reaper

@Service
public class SchedulerPod {

    @Autowired private JobRepository    jobRepo;
    @Autowired private RedisTemplate<String,String> redis;
    @Autowired private JobExecutor      executor;

    /**
     * Poll for due jobs every 5 seconds.
     * SKIP LOCKED ensures this pod gets a disjoint set of jobs
     * even when multiple scheduler pods run concurrently.
     */
    @Scheduled(fixedDelay = 5000)
    public void pollAndExecute() {
        List<Job> dueJobs = jobRepo.claimDueJobs(Instant.now(), 100);
        dueJobs.forEach(job -> CompletableFuture.runAsync(() -> executeJob(job)));
    }

    private void executeJob(Job job) {
        String lockKey = "lock:job:" + job.getId() + ":" + job.getRunAttempt();

        // Double-check with Redis distributed lock (belt and suspenders)
        Boolean acquired = redis.opsForValue()
            .setIfAbsent(lockKey, "locked", Duration.ofSeconds(300));
        if (!Boolean.TRUE.equals(acquired)) {
            return; // another pod already executing this run
        }

        try {
            // Heartbeat thread: keep lock alive during execution
            ScheduledFuture<?> heartbeat = startHeartbeat(job.getId(), lockKey);

            ExecutionResult result = executor.execute(job);
            heartbeat.cancel(true);

            if (result.isSuccess()) {
                Instant nextRun = computeNextRun(job);
                jobRepo.markSucceeded(job.getId(), result, nextRun);
            } else {
                handleFailure(job, result);
            }
        } catch (Exception e) {
            handleFailure(job, ExecutionResult.error(e.getMessage()));
        } finally {
            redis.delete(lockKey);
        }
    }

    private void handleFailure(Job job, ExecutionResult result) {
        if (job.getAttempts() < job.getMaxRetries()) {
            // Exponential backoff: 30s, 90s, 270s, ...
            long backoffSecs = (long) (30 * Math.pow(3, job.getAttempts()));
            Instant retryAt  = Instant.now().plusSeconds(backoffSecs);
            jobRepo.markRetrying(job.getId(), result, retryAt);
        } else {
            jobRepo.markFailed(job.getId(), result);
        }
    }

    private Instant computeNextRun(Job job) {
        if (job.getCronExpr() == null) return null; // one-time job
        return CronSequenceGenerator.nextExecution(job.getCronExpr(), Instant.now());
    }
}

// ── Job Repository (SKIP LOCKED) ──────────────────────
public interface JobRepository {

    @Query(value = """
        SELECT * FROM jobs
        WHERE next_run_at <= :now AND status = 'PENDING'
        ORDER BY next_run_at
        LIMIT :limit
        FOR UPDATE SKIP LOCKED
        """, nativeQuery = true)
    @Transactional
    List<Job> claimDueJobs(@Param("now") Instant now, @Param("limit") int limit);
}

// ── Reaper process ────────────────────────────────────
@Scheduled(fixedDelay = 60_000)
public void reaperScan() {
    List<Job> staleJobs = jobRepo.findStaleRunning(Instant.now());
    staleJobs.forEach(job -> handleFailure(job, ExecutionResult.error("Executor timed out")));
}

# Distributed Task Scheduler — Python

import time, threading, requests, redis, math
from datetime import datetime, timedelta
from croniter import croniter
from dataclasses import dataclass, field
from typing import Optional
import psycopg2

r = redis.Redis()


def claim_due_jobs(conn, limit: int = 100) -> list:
    """SKIP LOCKED: grab a disjoint batch of due jobs atomically."""
    with conn.cursor() as cur:
        cur.execute("""
            SELECT id, job_type, payload, cron_expr, attempts, max_retries,
                   run_attempt, idempotency_key, callback_url
            FROM jobs
            WHERE next_run_at <= NOW() AND status = \'PENDING\'
            ORDER BY next_run_at
            LIMIT %s
            FOR UPDATE SKIP LOCKED
        """, (limit,))
        return cur.fetchall()


def acquire_lock(job_id: str, run_attempt: int, ttl: int = 300) -> bool:
    key = f\'lock:job:{job_id}:{run_attempt}\'
    return r.set(key, \'1\', nx=True, ex=ttl) is not None

def release_lock(job_id: str, run_attempt: int):
    r.delete(f\'lock:job:{job_id}:{run_attempt}\')

def refresh_lock(job_id: str, run_attempt: int, ttl: int = 300):
    r.expire(f\'lock:job:{job_id}:{run_attempt}\', ttl)


def execute_job(conn, job: tuple):
    job_id, _, _, cron_expr, attempts, max_retries, run_attempt, idem_key, url = job

    # Distributed lock guard
    if not acquire_lock(job_id, run_attempt):
        return  # another pod got it

    # Start heartbeat thread
    stop_heartbeat = threading.Event()
    def heartbeat():
        while not stop_heartbeat.wait(timeout=30):
            refresh_lock(job_id, run_attempt)
            update_lock_expiry(conn, job_id)
    threading.Thread(target=heartbeat, daemon=True).start()

    try:
        # Execute: HTTP callback with idempotency key
        resp = requests.post(url, timeout=60,
                             headers={\'X-Idempotency-Key\': idem_key},
                             json={\'job_id\': job_id})
        stop_heartbeat.set()

        if resp.status_code < 300:
            next_run = compute_next_run(cron_expr)
            mark_succeeded(conn, job_id, next_run)
        else:
            handle_failure(conn, job_id, attempts, max_retries, resp.text)

    except Exception as e:
        stop_heartbeat.set()
        handle_failure(conn, job_id, attempts, max_retries, str(e))
    finally:
        release_lock(job_id, run_attempt)


def handle_failure(conn, job_id: str, attempts: int, max_retries: int, error: str):
    if attempts < max_retries:
        backoff = 30 * (3 ** attempts)  # 30s, 90s, 270s...
        retry_at = datetime.utcnow() + timedelta(seconds=backoff)
        mark_retrying(conn, job_id, retry_at, error)
    else:
        mark_failed(conn, job_id, error)


def compute_next_run(cron_expr: Optional[str]) -> Optional[datetime]:
    if not cron_expr: return None  # one-time job
    return croniter(cron_expr, datetime.utcnow()).get_next(datetime)


def scheduler_loop(conn):
    """Main poll loop — run this on each scheduler pod."""
    while True:
        jobs = claim_due_jobs(conn, limit=100)
        threads = [threading.Thread(target=execute_job, args=(conn, j)) for j in jobs]
        [t.start() for t in threads]
        [t.join() for t in threads]
        if not jobs: time.sleep(5)  # back off if nothing due


def reaper_loop(conn):
    """Scan for stale RUNNING jobs every 60s and reset them."""
    while True:
        time.sleep(60)
        with conn.cursor() as cur:
            cur.execute("""
                SELECT id, attempts, max_retries FROM jobs
                WHERE status = \'RUNNING\' AND lock_expires_at < NOW()
            """)
            stale = cur.fetchall()
        for job_id, attempts, max_retries in stale:
            handle_failure(conn, job_id, attempts, max_retries, \'Lock expired: executor timed out\')
⚖️ Tradeoffs
🔒 Coordination: SKIP LOCKED vs Leader Election vs Partitioning
✅ SKIP LOCKED (chosen)

PostgreSQL/MySQL feature: multiple workers compete for rows; each gets a disjoint set. No coordination layer needed. Works beautifully up to ~50K jobs/sec. Simple — no ZooKeeper, no leader election protocol. The database does the coordination. Airflow, Celery, and Quartz use this pattern.

Leader election (etcd/ZK)

One scheduler node is the leader and does all scheduling. Followers are on standby. Eliminates all race conditions — serial execution. Bottleneck: single node throughput. Mitigated by pushing execution to worker queues. Used by Kubernetes CronJob controller. Simpler reasoning about correctness.

✅ Hash partitioning (at scale)

jobs split by hash(job_id) % N. Each scheduler pod owns exactly 1/N of jobs. No overlap possible — zero coordination needed. Scales linearly. Complexity: pod assignment must be stable; resharding is painful. Best for >1M jobs/sec. Uber's Cadence/Temporal uses workflow partitioning.

⚡ Execution: Exactly-Once vs At-Least-Once vs At-Most-Once
At-Most-Once

Job fires, any failure → never retried. Suitable for: metric aggregation, non-critical cleanup jobs. Loses executions on worker crash. Simple but unreliable for business-critical work.

At-Least-Once + Idempotency

Retry on failure. Target service must be idempotent (safe to call multiple times with same idempotency_key). Ideal for: emails (check before send), payments (payment processor handles dedup), notifications. This is the practical "exactly-once" for most real-world systems.

✅ True Exactly-Once (Workflow engines)

Temporal, Apache Airflow: track every workflow step in a state machine. Re-execution replays from last checkpoint. True exactly-once even across crashes. 10× more complex than simple scheduler. Worth it for multi-step financial workflows. Overkill for simple cron jobs.
