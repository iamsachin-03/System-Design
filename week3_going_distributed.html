Designing Load Balancers

A load balancer distributes incoming traffic across a pool of backend servers, preventing any single server from becoming a bottleneck — while also detecting and bypassing unhealthy servers. The challenge: the LB itself must not become a Single Point of Failure (SPoF).

Rule: Use L7 (ALB/nginx) for most web services — path-based routing, rich health checks, TLS termination. Use L4 (NLB) for WebSockets, gRPC streaming, or when you need <1ms overhead.
LB Algorithms

Round Robin

Requests cycle through servers in order: S1, S2, S3, S1, S2… Simple, even distribution. Ignores server load or response time. Best when servers are identical and requests are uniform.

Default for most LBs. Use when all servers are the same spec.

Weighted Round Robin

Assign weights: S1=3, S2=1. S1 handles 75% of requests. Use when servers have different capacities (e.g., c5.xlarge vs c5.2xlarge). Weights can be static or auto-tuned.

Least Connections

Route to the server with fewest active connections. Better than round-robin when requests have variable processing time (e.g., some requests take 10ms, some take 2s). Prevents overloading slow servers.

Least Response Time

Route to server with lowest avg response time AND fewest connections. More sophisticated — tracks real performance. Used by Envoy (P2C: Power of Two Choices), nginx plus.

IP Hash (Sticky)

server = hash(client_ip) % N. Same client always goes to same server. Enables server-side session state. Problem: hot IPs (NAT → thousands of users behind one IP). Breaks on server add/remove.

Random with Two Choices (P2C)

Pick 2 random servers, route to the less loaded one. Provably near-optimal distribution. Low overhead. Used by Envoy, Finagle. Better than least-connections at scale (avoids herd effect).

Health Checks

The LB must know which backends are healthy. If it routes to a dead server, requests fail. Health checks are the mechanism that keeps the routing table accurate.

Passive Health Checks

Monitor real request outcomes. If a server returns 5xx errors or times out N times, mark it unhealthy. No extra traffic. Latency: may route a few real requests to a failing server before detection.

Active Health Checks ✓

LB sends periodic probe requests (GET /health). Server must respond 200. If it fails K consecutive checks, mark unhealthy. Faster detection. Use short intervals (5–10s) with 2–3 failure threshold.

If your LB itself fails, everything goes down. A single LB is itself a single point of failure. The solution: Active-Active or Active-Passive LB pairs with a shared virtual IP.

load_balancer.py — Round-robin + health checkPython
import asyncio, httpx, itertools
from fastapi import FastAPI, Request
from fastapi.responses import Response

app = FastAPI()

BACKENDS = [
    {"url": "http://app1:8080", "healthy": True, "failures": 0},
    {"url": "http://app2:8080", "healthy": True, "failures": 0},
    {"url": "http://app3:8080", "healthy": True, "failures": 0},
]
_rr = itertools.cycle(range(len(BACKENDS)))  # round-robin pointer
FAIL_THRESHOLD = 3

def next_healthy():
    """Pick next healthy backend (skip unhealthy ones)."""
    for _ in range(len(BACKENDS)):
        idx  = next(_rr)
        back = BACKENDS[idx]
        if back["healthy"]:
            return back
    raise RuntimeError("No healthy backends")

@app.api_route("/{path:path}", methods=["GET","POST","PUT","DELETE"])
async def proxy(request: Request, path: str):
    backend = next_healthy()
    url = f"{backend['url']}/{path}"
    async with httpx.AsyncClient(timeout=5.0) as client:
        try:
            resp = await client.request(
                method=request.method, url=url,
                headers=dict(request.headers),
                content=await request.body()
            )
            backend["failures"] = 0   # reset on success
            return Response(content=resp.content,
                            status_code=resp.status_code,
                            headers=dict(resp.headers))
        except (httpx.RequestError, httpx.TimeoutException):
            backend["failures"] += 1
            if backend["failures"] >= FAIL_THRESHOLD:
                backend["healthy"] = False   # passive health check
            raise

async def active_health_check():
    """Background task: probe each backend every 10s."""
    async with httpx.AsyncClient(timeout=2.0) as client:
        while True:
            for b in BACKENDS:
                try:
                    r = await client.get(f"{b['url']}/health")
                    b["healthy"] = r.status_code == 200
                    b["failures"] = 0
                except:
                    b["failures"] += 1
                    if b["failures"] >= 2:
                        b["healthy"] = False
            await asyncio.sleep(10)
Outcome
  • L4 vs L7 is a routing capability decision — L7 for HTTP path routing, header manipulation, TLS termination; L4 for WebSocket, raw TCP, ultra-low overhead.
  • Active health checks are mandatory in production — passive (observing errors) is too slow. Probing /health every 5–10s with fail-fast ensures unhealthy backends are bypassed before users see errors.
  • Active-Active LB pairs eliminate SPoF — deploy two LB instances, use DNS failover or Anycast. Each handles 50% normally; survivor takes 100% on failure. VRRP/Keepalived for bare-metal.
  • Power of Two Choices (P2C) — pick 2 random servers, route to the less loaded one. Provably near-optimal distribution at scale. Envoy uses this. Better than least-connections for high-throughput services.
  • The LB itself needs monitoring — track: request rate, error rate, active connections per backend, health check failures. Alert on: backend pool <50% healthy, error rate >1%, p99 > threshold.

Leader Election & Self-healing Distributed Systems

In a distributed cluster, exactly one node must be the "leader" — responsible for coordination tasks like log replication, lock granting, or primary DB writes. Leader election is the algorithm by which nodes agree on who that leader is.

Why a Leader?

  • Avoids split-brain — two nodes thinking they're primary
  • Simplifies coordination — one decision-maker
  • Enables Paxos/Raft consensus for distributed log replication
  • Kafka: partition leader handles all reads/writes

Properties a Leader Election Must Have

  • Safety — at most one leader at any time
  • Liveness — if the leader dies, a new one is elected eventually
  • Fault tolerance — survives minority node failures

Real-world Uses

  • etcd — Kubernetes control plane (Raft)
  • Kafka — partition leader election (ZooKeeper/KRaft)
  • MySQL Group Replication — primary election
  • Elasticsearch — master node election
Split-brain: When a network partition causes two nodes to each think they are the leader and accept writes independently. The result: diverged state, data corruption, two conflicting primaries. Preventing split-brain is the #1 requirement of any leader election algorithm.
The Raft Algorithm

Raft is the most widely used consensus algorithm today. It's designed to be understandable (unlike Paxos) while providing the same safety guarantees. Used in etcd, CockroachDB, TiKV, Consul.

Raft Node States

Follower

Default state. Receives log entries from leader. If no heartbeat from leader within election timeout (150–300ms random), transitions to Candidate.

Candidate

Increments term, votes for itself, sends RequestVote RPCs to all peers. If majority grant votes → becomes Leader. If another leader emerges → reverts to Follower.

Leader

Sends periodic heartbeats (AppendEntries with empty payload) to prevent elections. Replicates log entries. Handles all client writes. At most one per term.

Raft Election Flow

1
Election timeout fires (no heartbeat received)
Each follower has a random timeout (150–300ms). Randomization prevents split votes — one node almost always fires first.
2
Node increments term and becomes Candidate
Sends RequestVote(term, candidateId, lastLogIndex, lastLogTerm) to all other nodes.
3
Peers grant or deny vote
Grant if: haven't voted this term AND candidate's log is at least as up-to-date as theirs. Each node votes for at most one candidate per term.
4
Majority wins → becomes Leader
Needs ⌊N/2⌋ + 1 votes (majority). With 5 nodes: needs 3. This prevents split-brain: two candidates cannot both get majority simultaneously.
5
New leader sends heartbeats immediately
Suppresses new elections. Begins log replication. All followers reset their election timers on receiving the heartbeat.
Why randomized timeouts prevent split votes: If all nodes had the same timeout, they'd all become candidates simultaneously and split the votes. Random timeouts (150–300ms range) ensure one node almost always starts election first and wins before others even fire.

The Bully Algorithm

A simpler election algorithm for small clusters. The node with the highest ID that is alive "bullies" its way to leadership. Simpler than Raft but doesn't provide log replication safety.

How Bully Works

1
Node P detects leader is dead (heartbeat timeout)
Sends ELECTION message to all nodes with higher ID than itself.
2
Higher-ID nodes respond with OK
Any node receiving ELECTION from a lower-ID node sends back OK (taking over the election process) and starts their own election.
3
If no OK received → declare self as leader
P sends COORDINATOR message to all: "I am the new leader." If P later receives an OK, it steps down.

✅ Bully Pros

  • Simple to implement
  • Fast election (O(N²) messages in worst case)
  • Deterministic: highest ID always wins

❌ Bully Cons

  • No log replication safety
  • Message complexity O(N²)
  • The "best" node (highest ID) may not be the most appropriate leader
  • Fragile with frequent failures (election storms)
Bully is used in simple coordination systems. For production distributed databases and consensus systems, always use Raft or Paxos — they provide both election AND safe log replication.
Self-healing Distributed Systems

Self-healing means the cluster automatically recovers from node failures without human intervention. Leader election is the core mechanism.

Self-healing Patterns

Auto re-election on leader failure

When leader stops sending heartbeats, followers detect the timeout and trigger election. New leader elected within 2× election timeout (typically <1 second for well-tuned clusters). Zero manual intervention.

Cluster membership changes

When a new node joins, the leader adds it to the cluster via a special "joint consensus" configuration change — a 2-phase approach that ensures safety during membership transitions. No downtime.

Log catch-up on rejoin

When a previously crashed follower rejoins, the leader detects its log is behind and sends missing entries (AppendEntries). The follower catches up without manual resync.

etcd / ZooKeeper in practice

Don't implement election yourself. Use etcd (Raft-based) or ZooKeeper (ZAB-based). These battle-tested systems handle election, log replication, membership changes, and client failover transparently.

Using etcd for Leader Election

etcd_election.py — leader election via etcdPython
import asyncio
import etcd3

etcd = etcd3.client(host='etcd', port=2379)

async def run_as_leader_or_follower(node_id: str):
    """
    Campaign for leadership. If we win, run leader duties.
    If we lose, run follower duties and watch for leader change.
    """
    election = etcd.elections.Campaign("/service/leader", node_id)

    while True:
        print(f"[{node_id}] Campaigning for leadership...")
        # Blocks until this node becomes leader (TTL-based lease)
        await asyncio.to_thread(election.campaign)   # etcd handles the Raft election
        print(f"[{node_id}] *** BECAME LEADER ***")

        try:
            await do_leader_work(node_id)   # perform primary duties
        except Exception as e:
            print(f"[{node_id}] Leader work failed: {e}, resigning")
            election.resign()               # graceful handoff

async def do_leader_work(node_id: str):
    """Runs only on the leader node."""
    while True:
        print(f"[{node_id}] Running scheduled job (leader only)")
        await asyncio.sleep(5)

Outcome
  • Safety requires majority (quorum) — Raft requires ⌊N/2⌋+1 votes to elect a leader. This makes split-brain mathematically impossible: two candidates cannot both achieve majority simultaneously.
  • Randomized timeouts prevent split votes — the single most elegant insight in Raft. Different timeouts ensure one node almost always starts and finishes an election before others even begin.

Implementing Remote & Distributed Locks

In distributed systems, multiple processes across different machines may need exclusive access to a shared resource. A distributed lock coordinates this — ensuring only one process holds the lock at a time across the network.

When You Need a Distributed Lock

  • Cron job deduplication — only one instance runs a scheduled task
  • Inventory reservation — prevent overselling across app nodes
  • Rate limiting per user — atomic increment across servers
  • Leader election — who writes to the primary DB

Requirements

  • Mutual exclusion — only one lock holder at a time
  • Deadlock-free — lock always eventually released (TTL)
  • Fault tolerant — lock holder crash doesn't freeze the system
  • No false releases — only the lock owner can release it

Lock Implementations

  • Redis SETNX + TTL — simplest, single Redis node
  • Redlock — Redis multi-node, quorum-based
  • MySQL / Postgres advisory locks — DB-backed, ACID
  • etcd / ZooKeeper — consensus-based, strongest guarantees
Redis Redlock Algorithm

Redlock is Redis's multi-node distributed lock algorithm. It acquires a lock on a majority of N Redis nodes independently, ensuring safety even if some nodes fail.

Single-Node Redis Lock (Simple, Fragile)

Simple Redis lock — works until Redis restartsRedis
-- Atomic acquire: SET if Not eXists + EXpiry
SET lock:resource-id {owner-uuid} NX PX 30000
-- NX = only set if key doesn't exist (atomic check-and-set)
-- PX 30000 = expire in 30s (prevents deadlock if holder crashes)
-- Returns OK = acquired, nil = already locked

-- Release: Lua script ensures atomic check-and-delete
-- ONLY delete if the value matches our UUID (safety!)
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then
       return redis.call('del', KEYS[1])
     else
       return 0
     end" 1 lock:resource-id {owner-uuid}

Redlock — Multi-Node Algorithm

1
Get current time T1
Record start time in milliseconds. Used to calculate effective lock TTL after acquisition.
2
Try to acquire lock on all N Redis nodes sequentially
Send SET key uuid NX PX ttl to each of N=5 nodes with a short timeout (e.g., 50ms) to avoid blocking on a slow node.
3
Count successful acquisitions
If acquired on ≥ ⌊N/2⌋+1 nodes (majority = 3 of 5), proceed. Otherwise, release all acquired locks and fail.
4
Validate remaining TTL
Effective TTL = original_TTL − (T2 − T1) − clock_drift_margin. If effective TTL ≤ 0, the lock took too long — release and fail.
5
Lock acquired — proceed with critical section
On completion or crash, release by deleting key on all nodes with the Lua check-and-delete script.
Redlock controversy (Martin Kleppmann): Even Redlock has edge cases with clock drift and process pauses that can lead to multiple lock holders. For critical correctness (financial transactions), use ZooKeeper/etcd-based locks with fencing tokens instead.

Fencing Tokens — The Safe Pattern

The problem: lock holder A gets paused (GC, slow network). Lock expires. B acquires lock and starts writing. A resumes and writes too — now both are writing. Solution: fencing tokens.

fencing_token.py — monotonic token prevents stale writesPython
# Lock service returns a monotonically increasing token
# Every write to the storage service includes the token
# Storage rejects writes with tokens lower than the highest seen

async def acquire_lock_with_fence(resource: str):
    """Returns (success, token) — token is a monotonic counter."""
    token = await etcd.increment(f"fence:{resource}")  # atomic increment
    ok = await redis.set(f"lock:{resource}", token, nx=True, ex=30)
    return ok, token

async def write_with_fence(storage, key: str, value: str, token: int):
    """Only applies if our token >= storage's current token."""
    await storage.conditional_write(key, value, min_token=token)
    # If our token is stale → storage rejects → we know lock expired
Code — Production Redis Lock
distributed_lock.py — production-grade Redis lock with retryPython
import uuid, asyncio, time
import redis.asyncio as aioredis
from contextlib import asynccontextmanager

class DistributedLock:
    RELEASE_SCRIPT = """
    if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
    else return 0 end"""

    def __init__(self, redis: aioredis.Redis, name: str,
                 ttl_ms: int = 30000, retry_times: int = 3,
                 retry_delay_ms: int = 200):
        self.redis       = redis
        self.name        = f"lock:{name}"
        self.ttl_ms      = ttl_ms
        self.retry_times  = retry_times
        self.retry_delay  = retry_delay_ms / 1000
        self._owner      = None

    @asynccontextmanager
    async def acquire(self):
        owner = str(uuid.uuid4())   # unique per acquisition attempt

        for attempt in range(self.retry_times):
            ok = await self.redis.set(
                self.name, owner,
                px=self.ttl_ms, nx=True   # NX = only if not exists
            )
            if ok:
                self._owner = owner
                try:
                    yield self             # critical section runs here
                finally:
                    await self._release(owner)
                return

            # Backoff before retry: base + jitter
            jitter = asyncio.get_event_loop().call_later
            await asyncio.sleep(self.retry_delay * (1.5 ** attempt))

        raise TimeoutError(f"Could not acquire lock {self.name!r}")

    async def _release(self, owner: str):
        """Atomic: only release if WE hold the lock (UUID match)."""
        await self.redis.eval(self.RELEASE_SCRIPT, 1, self.name, owner)

# Usage
lock = DistributedLock(redis_client, "inventory:product:42", ttl_ms=5000)

async def reserve_inventory():
    async with lock.acquire():
        # Only ONE process runs this at a time across all app servers
        count = await db.fetchval("SELECT qty FROM inventory WHERE id=42")
        if count > 0:
            await db.execute("UPDATE inventory SET qty=qty-1 WHERE id=42")
            return True
    return False

Common mistake: Using GET then SET separately to implement a lock. These two operations are not atomic — there's a race between them. Always use SET NX PX (one atomic command) for Redis locks.
Lock TTL sizing: TTL must be longer than your critical section (including worst-case network latency), but short enough to recover quickly from crashes. For 100ms critical section: TTL = 5–10s. Monitor lock acquisition failures — growing failures = contention problem.
Outcome
  • SET NX PX is the atomic primitive — not GET+SET. The NX flag makes the set conditional: succeed only if key doesn't exist. This is atomic at the Redis command level.
  • Always use UUID as lock value — the Lua release script checks the value before deleting. Without this check, a crashed lock holder's TTL-expired key could be deleted from another holder, causing two concurrent lock holders.
  • Fencing tokens are the safest pattern — GC pauses can cause a process to sleep through a lock expiry and wake up thinking it still holds the lock. Fencing tokens (monotonic counters checked by the storage layer) defeat this.
  • For critical correctness: use etcd/ZooKeeper — Redis locks have edge cases (clock drift, network pauses). For financial transactions or anything where correctness is non-negotiable, use consensus-backed locks.
  • Lock TTL = safety net, not the primary release mechanism — code should release locks explicitly. TTL is the deadlock prevention fallback for crash scenarios.

Distributed ID Generators

When a database is sharded across N nodes, auto-increment IDs no longer work — each shard would generate the same IDs. You need a system that generates globally unique, ideally time-sortable IDs without coordination between nodes.

Requirements for a Good Distributed ID

Must have

  • Globally unique — no two IDs are the same across all shards
  • Scalable — generated locally, no central coordinator
  • High throughput — millions of IDs per second

Nice to have

  • Time-sortable — IDs generated later compare greater (enables index efficiency)
  • Compact — 64-bit integer is much smaller than 128-bit UUID
  • Debuggable — can decode timestamp from ID
UUID vs Snowflake

UUID v4 (Random)

  • 128-bit random — astronomically unique
  • No coordination needed — generate anywhere
  • Available in all languages natively
  • ❌ Not time-sortable — random byte order
  • ❌ B-tree fragmentation — random insertions scatter across index pages, causing page splits and poor cache utilization
  • ❌ Large — 36 chars as string, 16 bytes binary
  • ❌ Can't extract timestamp or node from ID

Snowflake (Twitter, 64-bit) ✓

  • 64-bit integer — fits in BIGINT
  • Time-sortable — most significant bits = timestamp
  • ✅ Sequential within same millisecond on same node
  • ✅ Generates 4096 unique IDs per millisecond per node
  • ✅ No coordination — each node generates independently
  • ✅ Can decode: when was this created? by which machine?
  • ❌ Clock skew risk — NTP adjustment can cause duplicate IDs

B-tree Index Performance

Why sequential IDs matter for DB performance: With auto-increment or Snowflake IDs, new rows are always inserted at the rightmost leaf of the B-tree. This is called a "right-hand insert pattern" and is cache-friendly — only the last few pages are hot. UUID v4 inserts are random across the entire tree, causing expensive page splits and cache misses at scale. At 100M rows, UUID insert throughput is ~5× worse than sequential IDs.
Snowflake Deep Dive

Bit Layout

Throughput

1024 nodes × 4096 IDs/ms = 4,194,304 IDs/ms = ~4.2 billion IDs/second across the entire cluster. Twitter (with far fewer ID workers) generates billions of tweets without ever running out.

Clock Skew Problem

Clock moves backward (NTP correction): If the system clock jumps backward by 1ms, the Snowflake generator could produce IDs with the same timestamp as previous IDs → duplicate IDs. Fix: detect clock regression, wait until current time catches up past the last timestamp before generating new IDs.
snowflake.py — 64-bit ID generatorPython
import time, threading

class SnowflakeGenerator:
    EPOCH        = 1288834974657  # Twitter epoch: 2010-11-04
    WORKER_BITS  = 5
    DC_BITS      = 5
    SEQ_BITS     = 12
    MAX_SEQ      = (1 << SEQ_BITS) - 1   # 4095

    def __init__(self, worker_id: int, datacenter_id: int):
        self.worker_id      = worker_id
        self.datacenter_id  = datacenter_id
        self.sequence       = 0
        self.last_timestamp = -1
        self._lock           = threading.Lock()

    def generate(self) -> int:
        with self._lock:
            now = self._ms()

            if now < self.last_timestamp:   # clock went backward!
                raise RuntimeError(f"Clock moved backward by {self.last_timestamp - now}ms")

            if now == self.last_timestamp:
                self.sequence = (self.sequence + 1) & self.MAX_SEQ
                if self.sequence == 0:   # sequence exhausted in this ms
                    now = self._wait_next_ms(self.last_timestamp)
            else:
                self.sequence = 0

            self.last_timestamp = now
            return (
                ((now - self.EPOCH) << 22)      |  # 41-bit timestamp
                (self.datacenter_id << 17)       |  # 5-bit datacenter
                (self.worker_id << 12)            |  # 5-bit worker
                self.sequence                       # 12-bit sequence
            )

    def _ms(self): return int(time.time() * 1000)
    def _wait_next_ms(self, ts):
        now = self._ms()
        while now <= ts: now = self._ms()
        return now

UUID v7 — The Modern Choice

UUID v7 (RFC 9562, 2024) encodes a Unix timestamp in the most significant bits, making it time-sortable like Snowflake while keeping the 128-bit UUID format. It's now supported natively in PostgreSQL 17. If you're starting a new project today, UUID v7 is often the best default: globally unique, time-sortable, no machine ID assignment needed.

Outcome
  • UUID v4 kills B-tree performance at scale — random insertions cause page splits across the entire index. Snowflake or UUID v7 (time-sorted) keep inserts sequential, preserving cache locality and write throughput.
  • Snowflake = 41-bit timestamp + 10-bit machine + 12-bit sequence — no coordination needed. Each machine generates its own IDs. 4096 IDs/ms/node = 4.2B IDs/s across 1024 nodes.
  • Machine ID assignment is the one coordination needed — assign machine IDs from a central store (etcd, ZooKeeper, or a simple DB table) at node startup. This is a one-time operation per node lifetime.
  • Handle clock skew explicitly — detect when clock goes backward, wait until current time passes last-seen timestamp. NTP corrections can cause silent ID collisions in naive implementations.
  • UUID v7 is the modern default — time-sortable, globally unique, no machine ID assignment, available in PostgreSQL 17. For new projects without strict 64-bit constraints, prefer UUID v7 over Snowflake.

Handling the Hot-Shard Problem

After sharding your database, you expect even load distribution. Reality: 80% of traffic hits 20% of keys — and those keys may cluster on the same shard. One shard becomes a bottleneck while others sit idle. This is the hot-shard problem.

Why Hot Shards Happen

Natural skew

Some entities are inherently more popular: Justin Bieber vs average user. Drake's new album vs an unknown artist. The World Cup vs a local league game. You can't eliminate natural popularity skew.

Bad shard key choice

Sharding by country where 40% of users are in the US → US shard is 40× hotter. Sharding by creation date → recent shard gets all writes. Key selection is critical and hard to change later.

Temporal patterns

Flash sale at 12:00: all requests for product:123 → one shard. Even distribution normally, extreme skew at peak. Impossible to predict and pre-shard for.

Celebrity / influencer effect

10M followers check Taylor Swift's status simultaneously. All requests hit the same Redis key on the same shard. Perfect hashing doesn't help — it's one real entity.

Detecting Hot Shards

You can't fix what you can't see. Hot shard detection requires the right metrics at the right granularity.

Solutions for Hot Shards

1. Key Salting (Virtual Keys)

Spread one hot key across N virtual keys. Distribute writes round-robin, aggregate on reads.

key saltingPython
SALT_COUNT = 10

# Write: distribute to one of N virtual keys
def incr_counter(base_key: str):
    salt = random.randint(0, SALT_COUNT - 1)
    redis.incr(f"{base_key}:{salt}")

# Read: aggregate all virtual keys
def get_counter(base_key: str) -> int:
    keys = [f"{base_key}:{i}" for i in range(SALT_COUNT)]
    return sum(int(v or 0) for v in redis.mget(keys))

2. Local Cache for Hot Keys

Cache the hottest keys (celebrity profiles, trending items) in application memory with a very short TTL (1–5s). Never hits the DB/cache at all for the hottest reads.

local cachePython
from cachetools import TTLCache

# In-process cache: 1000 keys, 2s TTL
local_cache = TTLCache(maxsize=1000, ttl=2)

async def get_user_status(uid: str):
    if uid in local_cache:
        return local_cache[uid]   # zero network hops
    val = await redis.get(f"user:{uid}:ts")
    local_cache[uid] = val
    return val

3. Read Replicas for Hot Shards

Add extra read replicas specifically for the hot shard. Route reads to the replica pool, writes to the primary. Works when the hot key is read-heavy (most celebrity/trending scenarios are reads).

4. Re-shard with Better Key

If the shard key itself is bad (e.g., all writes go to one shard by design), the only real fix is re-sharding with a different key. Expensive one-time cost; prevents chronic skew. Use consistent hashing to minimize data movement during re-shard.

5. Application-level Rate Limiting

For truly extreme hot entities, rate-limit queries per second. Return cached (slightly stale) responses for requests beyond the limit. Prevents a single celebrity from taking down your entire database.

6. Move Hot Entities to Dedicated Infrastructure

VIP tier: celebrities, trending items get their own Redis nodes or DB partition. Normal users share the standard pool. Used by Twitter (fat shard for VIPs), YouTube (dedicated serving for viral videos).

Consistent Hashing

Standard sharding (key % N) requires rehashing ALL keys when N changes. Consistent hashing minimizes rehashing — only ~1/N of keys move when a node is added or removed.

Virtual Nodes

A single physical node maps to multiple "virtual nodes" (vnodes) on the ring. Each physical node owns ~150 positions. Benefits: even distribution even with heterogeneous hardware, graceful rebalancing when nodes join/leave.

Consistent hashing in the wild: Redis Cluster uses it internally. Cassandra uses it for partition key routing. DynamoDB uses it. Memcached clients use consistent hashing for key-to-server mapping.

Outcome
  • Hot shards are inevitable with natural data skew — 80/20 rules, celebrity effects, and temporal bursts cannot be prevented. Design your system to detect and handle them, not just prevent them.
  • Key salting distributes one hot key across N — splitting user:bieber into user:bieber:0…user:bieber:9 and aggregating on read is the canonical solution for hot counters/writes.
  • Local in-process cache is the most aggressive solution — zero network hops. 2s TTL + LRU eviction of top 1000 keys eliminates hot-shard reads almost entirely for read-heavy endpoints.
  • Consistent hashing minimizes rehashing — when adding/removing shards, only ~1/N keys move. Essential for operational sanity at scale. Standard sharding rehashes all keys.
  • Monitor per-key hit rates, not just per-shard totals — a hot shard won't show up in aggregate metrics until it's already causing latency. Per-key tracking (Redis HOTKEYS command) catches it early.
