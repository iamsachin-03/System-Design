URL Shortener
Design a production-grade URL shortening service like bit.ly or TinyURL — handling 100M+ URL creations and 10B+ redirects daily, with analytics, custom aliases, and sub-10ms redirect latency.
🔴 MUST DO 🟢 Easy–Medium Base62 Cache NoSQL Kafka Counter Service
🎯 Step 1 — Clarify Scope First (Always!)
Interview strategy: Never jump into design. Spend 3–5 minutes asking clarifying questions. This shows you think before you act, and helps avoid designing the wrong system. The interviewer may give hints through their answers.

✅ Functional Requirements
Functional = What the system does. These are user-facing features that the system must support.

CORE (Must Have)
URL Creation: User provides a long URL → system returns a unique short URL (e.g., short.ly/abc123)
Redirect: When a user visits the short URL → system redirects them to the original long URL
URL Expiry: Short URLs automatically expire after a configurable TTL (default: 1 year). Return 410 Gone after expiry.
Custom Alias: User can optionally provide their own short key (e.g., short.ly/my-campaign). Must be unique.
EXTENDED (Should Have)
Analytics: Track click count, timestamp, country, device type, referrer per short URL
URL Deduplication: If same long URL is submitted twice, return the same short URL
Delete / Deactivate: URL owner can deactivate a short URL
Bulk Creation: API endpoint to create multiple short URLs in one request (for enterprise)
OUT OF SCOPE (Nice to Have)
A/B testing (split traffic between two URLs)
Password-protected links
QR code generation
Link preview / metadata (Open Graph)
⚡ Non-Functional Requirements
Non-Functional = How well the system performs. These are quality attributes — speed, reliability, scale, security.

PERFORMANCE
Redirect latency: p99 < 10ms (p50 < 2ms via cache)
Creation latency: p99 < 200ms
Throughput: Handle 116,000 redirect req/sec average, 350,000 peak
RELIABILITY
Availability: 99.99% uptime = max ~52 min downtime/year
Durability: Zero URL data loss — 3× replication minimum
No key collision: Every short key must be globally unique
SCALABILITY
Horizontal scale: No single point of failure — all services stateless and scalable
Storage: ~18 TB/year URL data — must scale to 90 TB over 5 years
Read-heavy: System is 100:1 read-to-write — optimise read path
SECURITY
Malware protection: Scan URLs against Google Safe Browsing API at creation time
Rate limiting: Max 100 URL creations/min per IP to prevent abuse
Private key space: Short keys must not be guessable/sequential (prevent enumeration)

🔑 Phase 2 — Key Generation Deep Dive
PHASE 2 · CORE ALGORITHM
IR
INTERVIEWER
Let's start with the most interesting part — how do you generate the short key? Walk me through your options.
ME
YOU
There are three main approaches. Let me walk through each with their tradeoffs:

Option 1 — Hash the long URL (MD5/SHA256):
Take MD5 of the long URL → get a 128-bit hash → base62-encode it → take first 6 characters. The problem is hash collisions — two different URLs could produce the same first 6 chars. You'd need a retry loop (take next 6 chars), which adds complexity and unpredictability.

Option 2 — Base62 of an auto-incrementing ID:
A counter service generates unique sequential IDs. We convert each ID to Base62. 6 characters = 62⁶ ≈ 56.8 billion unique keys. Zero collision risk. Predictable. My preferred approach.

Option 3 — Pre-generated Key Service (KGS):
A dedicated service pre-generates billions of random keys offline and stores them in a "not-used" table. When a URL is created, grab one key and move it to a "used" table. Simple for the app layer, but adds DB overhead and an extra service to maintain.

I'll go with Option 2 (Base62 + counter) as the primary strategy. It's simple, collision-free, and fast.
IR
INTERVIEWER
Interesting. But sequential IDs mean attackers can enumerate all URLs — abc001, abc002, etc. How do you handle that?
ME
YOU
Great point. Sequential IDs are guessable. A few solutions:

1. Shuffle the ID before encoding: XOR the ID with a secret mask before Base62 encoding. E.g., id XOR 0x3F4A9B2C. The mapping is reversible for lookups, but output looks random.
2. Use a random salt per ID: Generate a UUID for each URL, take the first 64 bits, encode in Base62. Not sequential.
3. Feistel Cipher / Format-Preserving Encryption: Encrypt the sequential ID using a secret key while preserving the output length. This is the cleanest approach — IDs are internally sequential but externally look random and are non-enumerable.

For simplicity, I'll go with Option 1 (XOR masking) — easy to implement, no performance overhead.
IR
INTERVIEWER
The counter service sounds like a single point of failure. What if it goes down?
ME
YOU
Right. The counter service needs to be highly available. Here's my approach:

Range-based allocation: Each app server requests a block of 1 million IDs from the counter service at startup (e.g., Server A: IDs 1–1M, Server B: IDs 1M–2M). The app server then allocates IDs locally from its range — no network call per request. When the range is nearly exhausted, it pre-fetches the next range in the background.

The counter service itself is backed by ZooKeeper (strong consistency, leader election) or a simple Redis INCR operation with persistence. Even if the counter service has a brief outage, existing app servers can continue serving from their pre-allocated ranges for ~1M requests.
IR
INTERVIEWER
What happens if an app server crashes after getting a range but before using all the IDs?
ME
YOU
Those IDs are simply wasted — they're never allocated to any URL. That's acceptable. With 62⁶ ≈ 56.8 billion total keys, wasting a few million IDs per server crash is negligible. We'd need ~1000 servers crashing every day for 100 years to exhaust even 1% of the key space. The simplicity trade-off is worth it.
💡 Pattern: When an interviewer points out a flaw (sequential IDs are guessable), they want to see how you think under pressure. Don't get defensive — present multiple solutions and justify your choice.
🗄 Phase 3 — Storage & Database Choice
PHASE 3 · DATA LAYER
IR
INTERVIEWER
What database would you use, and why?
ME
YOU
Let me think about the access patterns first, then pick the right tool:

Access patterns:
  • Primary: GET short_key → long_url — simple key lookup
  • Secondary: GET long_url → short_key — dedup check on creation
  • Low write rate (1,200/sec), very high read rate (116,000/sec)
  • No complex joins, no multi-row transactions needed

This access pattern is perfectly suited for a key-value store. I'd use Cassandra:
  • Designed for massive scale (handles 18+ TB easily across a few nodes)
  • Auto-sharding via consistent hashing on short_key
  • Configurable replication (RF=3 for 3× data redundancy)
  • Tunable consistency — we can use eventual consistency for reads (fast) and quorum for writes (safe)
  • Built-in TTL support — we can set expires_at and Cassandra auto-deletes expired rows
IR
INTERVIEWER
Why not MySQL or PostgreSQL? It works fine for many systems.
ME
YOU
MySQL could work at small scale. The problem comes at 90 TB over 5 years. SQL databases require manual sharding (painful to operate), and their ACID guarantees add overhead we don't need for a simple key-value pattern. The cost of not needing ACID in this system but paying its performance price is unnecessary.

That said — for a startup with low scale (<10M URLs), I'd absolutely start with PostgreSQL and add Cassandra later. Never over-engineer early.
IR
INTERVIEWER
How does your caching strategy work?
ME
YOU
URL access follows the Zipf/Pareto distribution — roughly 20% of URLs get 80% of traffic (the viral / popular ones). So caching just the hot URLs gives massive benefit.

Two-level caching:
L1 — In-process cache (per app server): A small LRU cache of ~1,000 entries in the app server's heap. Zero network hop — serves in microseconds. For viral links with millions of hits/hour, this is critical.
L2 — Redis cluster: Distributed cache with ~10GB capacity (enough for ~20M URLs at 500B each). Cache hit returns the long URL in ~1ms. LRU eviction policy. TTL aligned with URL expiry.

Cache write strategy — Write-through: When a new URL is created, we immediately write to both Cassandra AND Redis. This ensures the cache is warm from the very first redirect request.
IR
INTERVIEWER
What's your expected cache hit rate?
ME
YOU
With 20M hot URLs cached (20% of daily creates), and assuming Pareto distribution, I'd estimate ~90–95% cache hit rate. This means only 5–10% of redirects hit Cassandra, reducing DB read load from 116K req/sec to ~7K req/sec. Cassandra handles that trivially.
↪️ Phase 4 — Redirect Flow & 301 vs 302
PHASE 4 · CORE FLOWS
IR
INTERVIEWER
Walk me through the exact redirect flow end-to-end. And should you use HTTP 301 or 302?
ME
YOU
Redirect flow step by step:
  1. User visits https://short.ly/abc123 in browser
  2. DNS resolves to our Anycast Load Balancer (closest edge)
  3. LB routes request to a stateless Redirect Service instance
  4. Service checks L1 (in-process cache) → hit: returns redirect immediately
  5. L1 miss → checks Redis (L2 cache) → hit (~95% of time): returns redirect
  6. Redis miss → queries Cassandra, warms both caches, returns redirect
  7. In parallel (non-blocking): emit ClickEvent to Kafka for analytics
  8. Browser follows redirect to the long URL

301 vs 302 — this is a classic tradeoff question:

301 Permanent Redirect: Browser caches the redirect permanently. Subsequent visits go directly to the long URL — never hitting our server again. Pros: saves server load. Cons: we lose all analytics after the first visit. We can't update the destination URL. We can't deactivate it.

302 Temporary Redirect: Browser does NOT cache it. Every redirect hits our server. Pros: we see every click, can update destination, can deactivate. Cons: slightly more load on us — but with Redis, each redirect is just a ~1ms cache lookup. Totally worth it.

Decision: Use 302. The analytics requirement makes 301 a non-starter.
IR
INTERVIEWER
What if there's a sudden viral spike — a celebrity tweets your short link and 1 million people click it in 10 minutes?
ME
YOU
This is the "thundering herd" / hot key problem. Even with Redis, if all 1M requests converge on a single Redis key in a single shard, that shard can become a bottleneck.

Solutions I'd layer together:
1. Local in-process cache (L1): Each app server caches the top ~1,000 URLs in-process. For a viral link, after the first request, subsequent requests on that same server never even touch Redis — served from RAM in microseconds.
2. Redis key replication: For detected hot keys, replicate the value across multiple Redis slots: abc123:0, abc123:1, ..., abc123:9. Requests randomly pick a slot. 10× read distribution.
3. Auto-scaling: Our redirect services are stateless and behind an auto-scaling group. Kubernetes HPA can scale from 50 → 200 instances in under 2 minutes based on CPU/RPS metrics.
📈 Phase 5 — Analytics, Expiry & Edge Cases
PHASE 5 · EDGE CASES
IR
INTERVIEWER
How do you handle analytics without slowing down the redirect?
ME
YOU
The redirect must be fast — we should never write analytics synchronously in the redirect path. Here's the async design:

1. Redirect service fires a non-blocking event to a local in-memory queue after sending the 302 response (fire-and-forget — the user already got their redirect)
2. A background thread batches events and publishes them to Kafka (topic: click-events)
3. A separate Analytics Consumer service reads from Kafka, aggregates click counts per URL per hour, and writes to ClickHouse (columnar DB optimised for aggregations)
4. Analytics dashboard queries ClickHouse

If Kafka is down temporarily, events are buffered in the local queue. We lose at most a few seconds of analytics data — acceptable for eventual consistency.
IR
INTERVIEWER
How do you handle expired URLs?
ME
YOU
Two-pronged strategy:

Lazy expiry (primary): When a redirect request comes in, check expires_at < current_time. If expired, return 410 Gone. No background scan needed for the hot path. Also set Redis TTL equal to URL TTL — expired keys auto-evict from cache.

Active cleanup (background): A nightly Spark/batch job scans Cassandra for expired URLs and hard-deletes them, reclaiming storage and freeing short keys for potential reuse. Cassandra's built-in TTL feature can also handle this automatically at the storage layer.
IR
INTERVIEWER
What about malicious URLs — someone shortening a phishing link?
ME
YOU
Abuse prevention is important. Layered approach:

At creation time:
1. Check URL against Google Safe Browsing API — free, covers known phishing, malware, deceptive sites
2. Blocklist of known malicious domains in Redis (O(1) set lookup)
3. Rate limit by IP: max 100 URL creations/min (Redis token bucket)

At redirect time:
4. Show a safe browsing interstitial warning page for flagged URLs (like Google does) before redirecting
5. URLs reported by users → reviewed by trust & safety team → deactivated if confirmed malicious
IR
INTERVIEWER
Last one — how would you make this globally available with low latency worldwide?
ME
YOU
For global deployment, I'd use a multi-region active-active setup:

1. Deploy redirect services in multiple regions (US, EU, APAC) behind Anycast routing — users automatically hit the nearest region
2. Each region has its own Redis cluster — redirect latency is now <5ms globally
3. Cassandra replication is cross-region (multi-datacenter replication) — eventual consistency between regions
4. URL creation can go to any region — Cassandra's LWW (Last Write Wins) conflict resolution handles concurrent creates
5. For custom aliases, use a global lock (Zookeeper / Paxos) to prevent duplicate alias creation across regions

The key insight: reads don't need global consistency — if a user in India sees a URL 100ms after creation by a user in the US, that's totally fine. But we must never serve stale data for a deactivated/expired URL.
💡 Strong finish: When wrapping up, proactively mention what you'd do with more time: monitoring (Datadog dashboards), circuit breakers (Hystrix), chaos engineering tests, SLA dashboards. This shows production mindset.

🔌 API Design — REST Contract
Good API design shows: You think about the caller's experience, error handling, versioning, and backward compatibility. Always include response codes and error formats.
POST /api/v1/urls Create a new short URL
Request Body — application/json
long_urlstringrequiredThe full destination URL. Must be a valid URL (http/https). Max 2,048 chars. Example: "https://example.com/very/long/path"
custom_aliasstringoptionalUser-defined short key. 3–20 chars, alphanumeric + hyphens. Returns 409 if already taken. Example: "my-promo"
ttl_daysintegeroptionalTime-to-live in days. Range: 1–3650. Default: 365 (1 year). Pass -1 for never-expire.
user_iduuidoptionalAssociate URL with a registered user account for analytics dashboard access.
Response Codes
201Created. Returns short URL details. {"short_key":"abc123","short_url":"https://short.ly/abc123","expires_at":"2027-03-21T00:00:00Z","created_at":"2026-03-21T10:00:00Z"}
400Bad Request. Invalid URL format, missing required field, or URL contains unsafe content (malware/phishing). {"error":"INVALID_URL","message":"The provided URL is not a valid http/https URL"}
409Conflict. Custom alias is already taken. {"error":"ALIAS_CONFLICT","message":"The alias 'my-promo' is already in use"}
429Too Many Requests. Rate limit exceeded (100 creations/min per IP). Includes Retry-After: 30 header.
500Internal Server Error. Counter service failure, DB write failure. Client should retry with exponential backoff.

👥 Actors in the System

Before drawing the architecture, identify who uses the system. Each actor eventually becomes a client or a service class.

🧑‍💻
End User
Clicks a short URL link (e.g., on Twitter). Expects instant redirect to destination. Never creates URLs.
🏢
URL Creator
Submits a long URL to create a short link. Optionally provides custom alias and TTL. May be a developer via API or a web user.
📊
Analytics Consumer
Views click analytics for their short URLs — total clicks, geo breakdown, referrers, device type over time.
🤖
Admin / Trust & Safety
Reviews flagged URLs for abuse/malware. Can deactivate any URL. Manages blocklists.
🔧
Developer (API User)
Uses the REST API to programmatically create/manage short URLs, integrate into their own apps or marketing tools.
⏰
Cleanup Scheduler
Automated background job that deletes expired URLs from storage nightly. Not a human actor.
🏗 High-Level Architecture Diagram

Read path (blue): User → LB → Redirect Service → Redis (95% hit) → 302 back. Miss: Redis → Cassandra → warm cache → 302. Async: emit click to Kafka.

Write path (green): Creator → Rate Limiter → Write Service → Safe Browsing check → Counter Service (get ID) → Cassandra (persist) → Redis (warm cache) → return short URL.
🔄 Detailed Read & Write Flow
📥 REDIRECT (READ) FLOW
1️⃣ GET /abc123 hits Load Balancer
2️⃣ LB routes to any Redirect Service instance
3️⃣ Check L1 in-process cache → hit? → return 302
4️⃣ L1 miss → check Redis L2 cache → hit? → return 302
5️⃣ Redis miss → query Cassandra
6️⃣ Check URL active + not expired
7️⃣ Warm Redis + L1 caches
8️⃣ Return HTTP 302 Location: long_url
9️⃣ 🔀 Async: publish ClickEvent to Kafka
📤 CREATE URL (WRITE) FLOW
1️⃣ POST /api/v1/urls hits Rate Limiter
2️⃣ Validate request (URL format, fields)
3️⃣ Check Safe Browsing API (malware/phishing)
4️⃣ Check dedup: does long_url already exist?
5️⃣ Custom alias? → check uniqueness → 409 if conflict

// ── 1. Key Generation Service ───────────────────────
@Service
public class KeyGenerationService {

    private static final String BASE62 =
        "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789";
    private static final int  KEY_LENGTH  = 6;
    private static final long BLOCK_SIZE  = 1_000_000L;
    private static final long XOR_MASK    = 0x3F4A9B2CL; // prevents sequential enumeration

    private volatile long rangeStart = 0;
    private volatile long rangeEnd   = 0;
    @Autowired private ZookeeperCounterClient zkCounter;

    /**
     * Returns next unique 6-char Base62 key.
     * Thread-safe: synchronized on range allocation only.
     */
    public String nextKey() {
        long id = nextId();
        long masked = id ^ XOR_MASK; // obfuscate sequential IDs
        return encodeBase62(masked);
    }

    private synchronized long nextId() {
        if (rangeStart >= rangeEnd) {
            allocateNewRange(); // fetch next 1M block from ZooKeeper
        }
        // Pre-fetch next range when 90% exhausted (non-blocking)
        if (rangeStart > rangeEnd - (BLOCK_SIZE * 0.1)) {
            CompletableFuture.runAsync(this::prefetchNextRange);
        }
        return rangeStart++;
    }

    private void allocateNewRange() {
        CounterRange range = zkCounter.allocateRange(BLOCK_SIZE);
        rangeStart = range.getStart();
        rangeEnd   = range.getEnd();
    }

    public String encodeBase62(long num) {
        if (num < 0) num = -num; // handle XOR overflow
        StringBuilder sb = new StringBuilder();
        do {
            sb.append(BASE62.charAt((int)(num % 62)));
            num /= 62;
        } while (num > 0);
        while (sb.length() < KEY_LENGTH) sb.append('a'); // left-pad
        return sb.reverse().toString();
    }
}

// ── 2. Two-Level Cache Service ───────────────────────
@Service
public class CacheService {

    // L1: in-process LRU cache (per app server, ~1000 entries)
    private final Map<String, String> l1Cache =
        Collections.synchronizedMap(
            new LinkedHashMap<>(1000, 0.75f, true) {
                @Override protected boolean removeEldestEntry(Map.Entry<?, ?> e) {
                    return size() > 1000; // LRU eviction at 1000 entries
                }
            });

    @Autowired private RedisTemplate<String, String> redis;

    /**
     * Two-level lookup: L1 (in-process) → L2 (Redis)
     * Returns empty if not found in either level.
     */
    public Optional<String> get(String key) {
        // L1 — zero network hop
        String l1Hit = l1Cache.get(key);
        if (l1Hit != null) return Optional.of(l1Hit);

        // L2 — Redis (~1ms)
        String l2Hit = redis.opsForValue().get(key);
        if (l2Hit != null) {
            l1Cache.put(key, l2Hit); // promote to L1
            return Optional.of(l2Hit);
        }
        return Optional.empty();
    }

    public void put(String key, String value, Duration ttl) {
        l1Cache.put(key, value);
        redis.opsForValue().set(key, value, ttl);
    }

    public void evict(String key) {
        l1Cache.remove(key);
        redis.delete(key);
    }
}

// ── 3. Main Shortener Service ────────────────────────
@Service
@Slf4j
public class UrlShortenerService {

    @Autowired private KeyGenerationService         keyGen;
    @Autowired private UrlRepository                 urlRepo;
    @Autowired private CacheService                  cache;
    @Autowired private KafkaTemplate<String,ClickEvent> kafka;
    @Autowired private SafeBrowsingClient             safeBrowsing;

    /** Create a short URL from a long URL */
    public ShortenResponse shorten(ShortenRequest req) {
        // Step 1: validate URL format
        validateUrl(req.getLongUrl());

        // Step 2: check safe browsing (malware/phishing)
        if (safeBrowsing.isMalicious(req.getLongUrl()))
            throw new MaliciousUrlException();

        // Step 3: deduplication — same long URL gets same short key
        String longUrlHash = md5(req.getLongUrl());
        String existingKey = urlRepo.findShortKeyByHash(longUrlHash);
        if (existingKey != null)
            return buildResponse(existingKey);

        // Step 4: resolve short key (custom alias or generated)
        String shortKey;
        if (req.hasCustomAlias()) {
            if (urlRepo.exists(req.getCustomAlias()))
                throw new AliasConflictException(req.getCustomAlias());
            shortKey = req.getCustomAlias();
        } else {
            shortKey = keyGen.nextKey();
        }

        // Step 5: persist to Cassandra
        UrlEntity entity = UrlEntity.builder()
            .shortKey(shortKey).longUrl(req.getLongUrl())
            .longUrlHash(longUrlHash).userId(req.getUserId())
            .createdAt(Instant.now())
            .expiresAt(Instant.now().plusSeconds(req.getTtlDays() * 86400L))
            .isActive(true).build();
        urlRepo.save(entity);

        // Step 6: write-through cache (warm Redis immediately)
        cache.put(shortKey, req.getLongUrl(),
            Duration.ofSeconds(req.getTtlDays() * 86400L));

        return buildResponse(shortKey);
    }

    /** Resolve short key → original URL (redirect) */
    public String resolve(String shortKey, ClickContext ctx) {
        // Fast path: two-level cache lookup
        Optional<String> cached = cache.get(shortKey);
        if (cached.isPresent()) {
            publishClickAsync(shortKey, ctx);
            return cached.get();
        }

        // Slow path: Cassandra
        UrlEntity entity = urlRepo.findById(shortKey)
            .orElseThrow(() -> new UrlNotFoundException(shortKey));

        if (!entity.isAccessible()) // checks isActive && !isExpired
            throw new UrlExpiredException(shortKey);

        // Warm both cache levels
        Duration remaining = Duration.between(Instant.now(), entity.getExpiresAt());
        cache.put(shortKey, entity.getLongUrl(), remaining);

        publishClickAsync(shortKey, ctx);
        return entity.getLongUrl();
    }

    /** Deactivate a URL (sets is_active = false) */
    public void deactivate(String shortKey, UUID requestingUserId) {
        UrlEntity entity = urlRepo.findById(shortKey)
            .orElseThrow(() -> new UrlNotFoundException(shortKey));
        if (!entity.getUserId().equals(requestingUserId))
            throw new UnauthorizedException("Not owner");
        entity.setIsActive(false);
        urlRepo.save(entity);
        cache.evict(shortKey); // invalidate both L1 and L2
    }

    @Async
    private void publishClickAsync(String shortKey, ClickContext ctx) {
        try {
            kafka.send("click-events", new ClickEvent(shortKey, ctx));
        } catch (Exception e) {
            log.warn("Click event publish failed for {}: {}", shortKey, e.getMessage());
            // Don't fail the redirect — analytics loss is acceptable
        }
    }
}

import string, hashlib, time, threading, logging
from typing import Optional
from functools import lru_cache
from collections import OrderedDict
from concurrent.futures import ThreadPoolExecutor

logger = logging.getLogger(__name__)
BASE62 = string.ascii_lowercase + string.ascii_uppercase + string.digits  # 62 chars


# ── 1. Key Generation Service ───────────────────────
class KeyGenerationService:
    """
    Generates unique short keys using range-based counter allocation.
    Each instance owns a block of 1M IDs from ZooKeeper.
    XOR masking prevents sequential key enumeration.
    """
    KEY_LENGTH  = 6
    BLOCK_SIZE  = 1_000_000
    XOR_MASK    = 0x3F4A9B2C

    def __init__(self, zk_counter):
        self.zk = zk_counter
        self.range_start = 0
        self.range_end   = 0
        self.lock = threading.Lock()
        self.executor = ThreadPoolExecutor(max_workers=1)

    def next_key(self) -> str:
        """Thread-safe: returns next unique Base62 key"""
        with self.lock:
            if self.range_start >= self.range_end:
                self._allocate_range()
            # Pre-fetch next range when 90% exhausted
            if self.range_start > self.range_end - (self.BLOCK_SIZE * 0.1):
                self.executor.submit(self._prefetch_range)
            key_id = self.range_start
            self.range_start += 1

        masked = key_id ^ self.XOR_MASK  # obfuscate sequential pattern
        return self._encode_base62(abs(masked))

    def _encode_base62(self, num: int) -> str:
        if num == 0:
            return 'a' * self.KEY_LENGTH
        chars = []
        while num > 0:
            chars.append(BASE62[num % 62])
            num //= 62
        result = ''.join(reversed(chars))
        return result.rjust(self.KEY_LENGTH, 'a')  # left-pad to KEY_LENGTH

    def _allocate_range(self):
        r = self.zk.allocate_range(self.BLOCK_SIZE)
        self.range_start, self.range_end = r.start, r.end


# ── 2. Two-Level Cache Service ──────────────────────
class CacheService:
    """
    L1: in-process LRU (per server, 1000 entries, ~0ms lookup)
    L2: Redis cluster (shared, ~1ms lookup)
    """
    L1_SIZE = 1000

    def __init__(self, redis_client):
        self.redis = redis_client
        self.l1 = OrderedDict()  # LRU via OrderedDict
        self.lock = threading.Lock()

    def get(self, key: str) -> Optional[str]:
        # L1 lookup
        with self.lock:
            if key in self.l1:
                self.l1.move_to_end(key)  # mark as recently used
                return self.l1[key]

        # L2 lookup
        val = self.redis.get(key)
        if val:
            decoded = val.decode()
            self._l1_put(key, decoded)  # promote to L1
            return decoded
        return None

    def put(self, key: str, value: str, ttl_seconds: int):
        self._l1_put(key, value)
        self.redis.setex(key, ttl_seconds, value)

    def evict(self, key: str):
        with self.lock:
            self.l1.pop(key, None)
        self.redis.delete(key)

    def _l1_put(self, key: str, value: str):
        with self.lock:
            self.l1[key] = value
            self.l1.move_to_end(key)
            if len(self.l1) > self.L1_SIZE:
                self.l1.popitem(last=False)  # evict LRU


# ── 3. Main Shortener Service ────────────────────────
class UrlShortenerService:

    def __init__(self, key_gen, db, cache, kafka, safe_browsing):
        self.key_gen       = key_gen
        self.db            = db
        self.cache         = cache
        self.kafka         = kafka
        self.safe_browsing = safe_browsing
        self.executor      = ThreadPoolExecutor(max_workers=20)

    def shorten(self,
              long_url: str,
              custom_alias: Optional[str] = None,
              ttl_days: int = 365,
              user_id: Optional[str] = None) -> dict:
        """Create a short URL — full validation + dedup + persist"""

        # Step 1: Validate URL
        self._validate_url(long_url)

        # Step 2: Safe browsing check
        if self.safe_browsing.is_malicious(long_url):
            raise ValueError("URL_FLAGGED_MALICIOUS")

        # Step 3: Deduplication
        long_url_hash = hashlib.md5(long_url.encode()).hexdigest()
        existing = self.db.find_short_key_by_hash(long_url_hash)
        if existing:
            return self._build_response(existing)

        # Step 4: Resolve short key
        if custom_alias:
            if self.db.exists(custom_alias):
                raise ValueError(f"ALIAS_CONFLICT:{custom_alias}")
            short_key = custom_alias
        else:
            short_key = self.key_gen.next_key()

        # Step 5: Persist to Cassandra
        record = {
            "short_key":     short_key,
            "long_url":      long_url,
            "long_url_hash": long_url_hash,
            "user_id":       user_id,
            "created_at":    time.time(),
            "expires_at":    time.time() + ttl_days * 86400,
            "is_active":     True
        }
        self.db.put(record)

        # Step 6: Write-through cache
        self.cache.put(short_key, long_url, ttl_days * 86400)

        return self._build_response(short_key)

    def resolve(self, short_key: str, click_meta: dict = None) -> str:
        """Resolve short key to original URL — cache-first"""

        # Fast path: two-level cache
        cached = self.cache.get(short_key)
        if cached:
            self.executor.submit(self._publish_click, short_key, click_meta)
            return cached

        # Slow path: DB
        record = self.db.get(short_key)
        if not record:
            raise KeyError("URL_NOT_FOUND")
        if not record["is_active"]:
            raise ValueError("URL_DEACTIVATED")
        if time.time() > record["expires_at"]:
            raise ValueError("URL_EXPIRED")

        # Warm caches
        ttl = int(record["expires_at"] - time.time())
        self.cache.put(short_key, record["long_url"], ttl)
        self.executor.submit(self._publish_click, short_key, click_meta)
        return record["long_url"]

    def deactivate(self, short_key: str, requesting_user_id: str):
        """Soft-delete: set is_active=False, evict from cache"""
        record = self.db.get(short_key)
        if not record:
            raise KeyError("URL_NOT_FOUND")
        if record["user_id"] != requesting_user_id:
            raise PermissionError("NOT_OWNER")
        self.db.update(short_key, {"is_active": False})
        self.cache.evict(short_key)

    def _publish_click(self, short_key: str, meta: dict):
        """Async click event — never fails the redirect"""
        if not meta: return
        try:
            event = {
                "short_key":   short_key,
                "timestamp":   time.time(),
                "ip_hash":     hashlib.md5(
                    meta.get("ip","").encode()).hexdigest(),
                "country":     meta.get("country"),
                "device":      meta.get("device"),
                "referrer":    meta.get("referrer"),
            }
            self.kafka.send("click-events", key=short_key, value=event)
        except Exception as e:
            logger.warning(f"Click publish failed {short_key}: {e}")

    def _validate_url(self, url: str):
        from urllib.parse import urlparse
        parsed = urlparse(url)
        if parsed.scheme not in ('http', 'https'):
            raise ValueError("INVALID_URL: must be http or https")
        if not parsed.netloc:
            raise ValueError("INVALID_URL: missing domain")

    def _build_response(self, short_key: str) -> dict:
        return {"short_key": short_key,
                "short_url": f"https://short.ly/{short_key}"}

⚖️ Design Tradeoffs & Decisions
Interview tip: Every design decision is a tradeoff. When you say "I chose X", always follow with "because Y, and the cost is Z". This shows maturity.
🔄 Redirect Type: HTTP 301 vs 302
✅ Chosen: 302 Temporary

Every redirect hits our server. We see every click, can track analytics, can update destination URL, can deactivate instantly.

❌ Alternative: 301 Permanent

Browser caches permanently. After first visit, user goes directly to long URL — no analytics, no deactivation possible. Saves server load but breaks all tracking.

💡 Why 302 wins here

We have an explicit analytics requirement. 302 load overhead is negligible because each redirect is just a Redis cache lookup — sub-millisecond. If analytics were out of scope, 301 would be better.

🔑 Key Generation: Base62(Counter) vs MD5 Hash vs KGS
✅ Chosen: Base62 + Counter Range

Zero collision guarantee. Predictable key length. Range allocation eliminates central bottleneck. XOR masking hides sequential nature.

❌ Alternative: MD5 Hash of URL

No counter service needed. But collision-prone at scale — two URLs with same MD5 prefix need retry loops. Non-deterministic performance.

💡 Why counter wins

At 100M URLs/day, collision handling adds unpredictable latency. Counter is O(1), deterministic, and range allocation makes it fully distributed. KGS is viable but adds operational complexity of a separate key DB.

🗄 Storage: Cassandra (NoSQL) vs PostgreSQL (SQL)
✅ Chosen: Cassandra

Built for massive key-value scale. Auto-sharding. Tunable consistency. Native TTL. 54+ TB handled across a few nodes. No manual sharding needed.

⚠️ Alternative: PostgreSQL

Great for smaller scale (<10M URLs). ACID transactions, easy dedup queries. But requires manual sharding at 90TB+, and ACID overhead is unnecessary for simple key lookups.

💡 When to use SQL

For a startup with <10M URLs, start with PostgreSQL — simpler, cheaper, easier to operate. Add Cassandra when you hit sharding limits. Never over-engineer early.

⚡ Cache Strategy: Write-Through vs Read-Through vs Write-Behind
✅ Chosen: Write-Through

On URL creation, write to both Cassandra AND Redis simultaneously. Cache is warm from the very first redirect. No cold-start latency spike for new URLs.

❌ Read-Through

Cache populated only on first cache miss. First redirect always goes to Cassandra (~5–10ms). For viral links, this means the first million hits are slower until cache warms up.

💡 Write-Behind risk

Write-behind (async write to DB) would be fastest but risks data loss if server crashes before DB write completes. For URL mappings, we never want to lose data — write-through is the safe choice.

📊 Analytics: Sync Write vs Async via Kafka
✅ Chosen: Async via Kafka

Analytics event is fire-and-forget after sending 302. Kafka consumer aggregates clicks independently. Analytics outage never affects redirect latency.

❌ Sync DB Write

Write click to DB in the redirect path. Simpler architecture but adds 10–50ms per redirect. At 116K req/sec, a slow analytics DB can cascade and kill redirect performance.

💡 Tradeoff accepted

Async means we accept eventual consistency for analytics — clicks may be delayed by seconds. For analytics dashboards, this is completely fine. For real-time billing, it would not be acceptable.

🌍 Consistency: Strong vs Eventual
✅ Chosen: Eventual (reads), Strong (writes)

Reads use eventual consistency (LOCAL_ONE in Cassandra) — fast. Writes use quorum (QUORUM) — ensures durability. Multi-region reads may be slightly stale (100ms), acceptable for URL lookups.

❌ Strong Consistency Everywhere

All reads would require quorum (slower, higher latency). Requires cross-region coordination. Overkill for URL redirects where millisecond-stale data is fine.

💡 Exception: deactivation

When a URL is deactivated (e.g., reported malicious), we need strong consistency — we must ensure ALL regions stop serving it immediately. This specific operation uses QUORUM writes + cache invalidation to all replicas.
