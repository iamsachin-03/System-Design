Distributed Locking (Chubby / ZooKeeper)
Design a distributed lock service like Google Chubby or Apache ZooKeeper — enabling mutual exclusion across thousands of distributed processes, providing lease-based locks with automatic expiry, supporting leader election, and handling the hard problem of "fencing tokens" to prevent a slow lock holder from corrupting shared state after its lease expires.
🔴 MUST DO 🟣 Hard Lease Fencing Token Raft Consensus Leader Election Ephemeral Node Redlock
🎯 Requirements
Distributed locking is one of the most subtle problems in distributed systems. The naive approach — Redis SETNX — seems simple but is dangerously wrong without fencing tokens. A process can acquire a lock, pause (GC, network blip), have its lease expire, and then resume — writing to the protected resource while another process also holds the lock. This is the ghost lock problem.

FUNCTIONAL
Acquire exclusive lock on a named resource
Release lock explicitly or via lease expiry (TTL)
Leader election: exactly one leader at a time
Watch/notify: alert when lock becomes available
Fencing token: monotonically increasing token to detect stale lock holders
NON-FUNCTIONAL
Safety: at most one lock holder at any time
Liveness: if lock holder dies, lock is eventually released
Fault tolerant: survive minority node failures
Acquire latency: <10ms in same region
10K concurrent lock holders across the system
💬 Phase 1 — The Core Mechanism
PHASE 1 · LEASE-BASED LOCK
IR
INTERVIEWER
How do you implement a distributed lock that releases automatically if the lock holder dies?
ME
YOU
A lease-based lock with heartbeat:

Acquire: The client sends an acquire request. The lock server grants the lock with a TTL (e.g., 30 seconds) and returns a lease ID and a fencing token (a monotonically increasing number).

Heartbeat: The lock holder must send a heartbeat (renew request) every 10 seconds. The server extends the TTL on each heartbeat.

Expiry: If the lock holder crashes or becomes network-partitioned and can't send heartbeats, the TTL expires. The lock is automatically released. Another waiter can then acquire it.

ZooKeeper implementation: Uses ephemeral nodes — a ZooKeeper node (znode) that is automatically deleted when the creating client's session expires (detected via heartbeat timeout). Client creates /locks/resource-X/lock-001 as an ephemeral sequential node. The client with the lowest sequence number holds the lock. Others watch the next-lower node — when it's deleted, the next client gets the lock. No thundering herd (each client watches exactly one predecessor, not all watch the lock node).

Redis SETNX implementation: SET lock:resource-X {client-id} NX EX 30. Only one client can set this key. On success: lock acquired. To release: Lua script if GET key == client-id then DEL key end (must be atomic to avoid deleting another client's lock).
⚠️ Phase 2 — The Fencing Token Problem
PHASE 2 · FENCING
IR
INTERVIEWER
Client A holds the lock. It gets hit by a long GC pause (30+ seconds). Its lease expires. Client B acquires the lock. Client A's GC finishes — it now writes to the database. Two clients are writing simultaneously. How do you prevent this?
ME
YOU
This is the "zombie lock holder" problem. The solution is a fencing token.

When Client A acquires the lock, the lock service returns token=1. When Client A's lease expires and Client B acquires the lock, the service returns token=2. The token is a strictly monotonically increasing number.

The protocol: Every write request to the protected resource (e.g., database, file storage) must include the fencing token. The resource server rejects any write with a token lower than the highest token it has seen.

Timeline:
  • Client A acquires lock, gets token=1. Starts writing. GC pause hits.
  • A's lease expires. Client B acquires lock, gets token=2. Writes with token=2. Resource records highest_seen=2.
  • Client A wakes from GC. Tries to write with token=1. Resource server sees 1 < 2 → rejects A's write.
This prevents the zombie write even though we can't stop Client A from trying. The resource itself enforces the fencing.

Where to generate the token: The lock service increments a counter on every lock grant and embeds it in the lease. ZooKeeper's sequential znodes implicitly provide this — the sequence number is the fencing token. etcd uses the lease ID or a revision number.
⚠ Critical insight: A distributed lock alone cannot guarantee mutual exclusion unless the protected resource also validates fencing tokens. No distributed lock service can guarantee perfect mutual exclusion against process pauses (GC, VM suspension) — the fencing token pattern is the only correct solution.
👑 Phase 3 — Leader Election
PHASE 3 · LEADER ELECTION
IR
INTERVIEWER
How do you use a distributed lock service for leader election?
ME
YOU
Leader election is just a distributed lock with a specific protocol:

All candidates try to acquire the same lock (e.g., /election/primary). The one that succeeds is the leader. While leader, it holds the lock and sends heartbeats. If it fails, the lock expires and another candidate acquires it — becoming the new leader.

ZooKeeper leader election: All candidates create ephemeral sequential znodes at /election/candidate-. The one with the lowest number is the leader. Others watch the node just below them. When the leader's session expires (its znode is deleted), the next-lowest node gets notified and becomes leader.

etcd-based election: Candidates compete for a key with PUT /election/leader {candidate-id} NX TTL=10. The winner sends keepalives. Followers watch the key — when it's deleted, they compete to set it again.

Important protocol: When a process loses its lock (lease expired), it must stop acting as leader before the new leader starts. This requires the leader to actively stop its work on lease expiry, not wait to be informed. This is why etcd's Patroni works — the old primary sets itself read-only when it can't renew the lease.

// ══════════════════════════════════════════════════════
// Distributed Lock — Java: Redis-based with fencing token
// ══════════════════════════════════════════════════════

public class DistributedLock {

    private static final String ACQUIRE_SCRIPT = """
        local key   = KEYS[1]
        local fence = KEYS[2]     -- fencing token counter key
        local owner = ARGV[1]
        local ttl   = tonumber(ARGV[2])

        if redis.call('SET', key, owner, 'NX', 'EX', ttl) then
            -- Increment and return the fencing token
            return redis.call('INCR', fence)
        end
        return nil
        """;

    private static final String RELEASE_SCRIPT = """
        if redis.call('GET', KEYS[1]) == ARGV[1] then
            return redis.call('DEL', KEYS[1])
        end
        return 0
        """;

    private final RedisTemplate<String,String> redis;
    private final DefaultRedisScript<Long> acquireScript;
    private final DefaultRedisScript<Long> releaseScript;

    /**
     * @return Optional.of(fencingToken) if acquired, Optional.empty() if not.
     * The fencing token MUST be passed to every write operation on the protected resource.
     */
    public Optional<Long> tryAcquire(String resource, String ownerId, int ttlSeconds) {
        String lockKey   = "lock:" + resource;
        String fenceKey  = "fence:" + resource;

        Long token = redis.execute(acquireScript,
            List.of(lockKey, fenceKey), ownerId, String.valueOf(ttlSeconds));

        return token != null ? Optional.of(token) : Optional.empty();
    }

    public boolean release(String resource, String ownerId) {
        Long result = redis.execute(releaseScript,
            List.of("lock:" + resource), ownerId);
        return Long.valueOf(1L).equals(result);
    }

    /**
     * Renew the lock lease (heartbeat). Call every TTL/3 seconds.
     * Returns false if the lock was already taken by someone else.
     */
    public boolean renew(String resource, String ownerId, int ttlSeconds) {
        String renewScript = """
            if redis.call('GET', KEYS[1]) == ARGV[1] then
                return redis.call('EXPIRE', KEYS[1], ARGV[2])
            end
            return 0
            """;
        Long result = redis.execute(new DefaultRedisScript<>(renewScript, Long.class),
            List.of("lock:" + resource), ownerId, String.valueOf(ttlSeconds));
        return Long.valueOf(1L).equals(result);
    }
}

// ── Usage with fencing token ──────────────────────────
@Service
public class FileProcessor {

    @Autowired private DistributedLock lock;
    @Autowired private FileStorage       storage;

    public void processFile(String fileId) {
        String owner = UUID.randomUUID().toString();
        Optional<Long> tokenOpt = lock.tryAcquire("file:" + fileId, owner, 30);

        if (tokenOpt.isEmpty()) {
            throw new LockNotAcquiredException("File locked by another worker");
        }

        long fencingToken = tokenOpt.get();
        try {
            // Pass fencing token to every write — storage validates it
            storage.write(fileId, process(fileId), fencingToken);
        } finally {
            lock.release("file:" + fileId, owner);
        }
    }
}

// ── ZooKeeper lock (ephemeral sequential) ─────────────
public class ZkDistributedLock {
    private final CuratorFramework curator;

    public void withLock(String path, Runnable task) throws Exception {
        InterProcessMutex lock = new InterProcessMutex(curator, "/locks/" + path);
        if (lock.acquire(10, TimeUnit.SECONDS)) {
            try { task.run(); }
            finally { lock.release(); }
        } else throw new TimeoutException("Failed to acquire lock");
    }
}

# Distributed Lock — Python: Redis with fencing token

import uuid, time, threading, contextlib
from dataclasses import dataclass
from typing import Optional
import redis

r = redis.Redis()

ACQUIRE_SCRIPT = """
local key   = KEYS[1]
local fence = KEYS[2]
local owner = ARGV[1]
local ttl   = tonumber(ARGV[2])
if redis.call('SET', key, owner, 'NX', 'EX', ttl) then
    return redis.call('INCR', fence)
end
return false
"""

RELEASE_SCRIPT = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
end
return 0
"""

_acquire = r.register_script(ACQUIRE_SCRIPT)
_release = r.register_script(RELEASE_SCRIPT)


@dataclass
class LockContext:
    resource:      str
    owner_id:      str
    fencing_token: int
    ttl:           int
    _stop:         threading.Event = None

    def __post_init__(self):
        self._stop = threading.Event()
        self._start_heartbeat()

    def _start_heartbeat(self):
        def heartbeat():
            while not self._stop.wait(self.ttl / 3):
                renew_script = """
                    if redis.call('GET', KEYS[1]) == ARGV[1] then
                        return redis.call('EXPIRE', KEYS[1], ARGV[2])
                    end
                    return 0
                """
                result = r.eval(renew_script, 1, f"lock:{self.resource}",
                                self.owner_id, self.ttl)
                if not result:
                    print(f"[WARN] Lock lost for {self.resource}! TTL expired.")
                    self._stop.set()

        threading.Thread(target=heartbeat, daemon=True).start()

    def release(self):
        self._stop.set()
        _release(keys=[f"lock:{self.resource}"], args=[self.owner_id])


def try_acquire(resource: str, ttl: int = 30) -> Optional[LockContext]:
    owner = str(uuid.uuid4())
    token = _acquire(
        keys=[f"lock:{resource}", f"fence:{resource}"],
        args=[owner, ttl])
    if not token:
        return None
    return LockContext(resource=resource, owner_id=owner,
                        fencing_token=int(token), ttl=ttl)


@contextlib.contextmanager
def distributed_lock(resource: str, ttl: int = 30, retries: int = 3):
    ctx = None
    for attempt in range(retries):
        ctx = try_acquire(resource, ttl)
        if ctx: break
        time.sleep(0.1 * (2 ** attempt))  # exponential backoff

    if not ctx:
        raise RuntimeError(ff"Could not acquire lock on {resource} after {retries} attempts")
    try:
        yield ctx  # ctx.fencing_token must be passed to all writes
    finally:
        ctx.release()

⚖️ Tradeoffs
🔐 Lock Service: Redis SETNX vs ZooKeeper vs etcd
Redis SETNX + Fencing

Extremely fast (<1ms). Simple to implement. Single Redis node = SPOF. Redlock (multi-node) is controversial — Martin Kleppmann proved it's unsafe due to timing assumptions. Use only when some lock violations are acceptable (idempotent operations). Not recommended for critical mutual exclusion.

✅ ZooKeeper

Consensus-based (Zab protocol). Ephemeral sequential nodes are purpose-built for distributed locking. Session expiry = automatic lock release. Watch mechanism for efficient waiting (no polling). Proven at LinkedIn, Twitter scale. Operationally complex — requires dedicated ZK cluster. Strong consistency guarantee.

✅ etcd

Raft-based consensus. Lease-based locking with TTL. Used by Kubernetes for all distributed coordination. Simpler to operate than ZooKeeper. gRPC API. Watch support. Used by Patroni (DB failover) and Kubernetes scheduler. The modern choice for new systems. Linearizable reads/writes.
