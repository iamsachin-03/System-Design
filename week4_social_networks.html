Functional

  • Serve images to clients of any device/resolution

Upload Architecture

Direct vs Pre-Signed URL Upload

❌ Upload through API server

  • Client → API server → S3. API server becomes the bottleneck at 4.2 GB/s
  • API server must buffer the entire file in memory
  • Network bandwidth cost: API server pays egress fees to re-upload to S3
  • Single point of failure in the upload path

✅ Pre-Signed URL (Direct Upload)

  • Client → API server: "I want to upload" → server generates pre-signed S3 URL
  • Client uploads directly to S3 — API server never touches the bytes
  • S3 handles 4.2 GB/s without any API server involvement
  • Pre-signed URL expires in 15 min — security without proxying

Upload Flow Architecture

Resumable Uploads — TUS Protocol

For large files on unreliable mobile connections, standard HTTP uploads fail and must restart from scratch. TUS (resumable upload protocol) saves the upload offset, allowing clients to resume from where they left off.

resumable_upload.py — TUS-style chunked uploadPython
import hashlib, os
from fastapi import FastAPI, Header, UploadFile, HTTPException
from fastapi.responses import Response
import aiofiles

app    = FastAPI()
UPLOAD_DIR = "/tmp/uploads"

@app.post("/upload/init")
async def init_upload(user_id: str, file_size: int, sha256: str):
    """Step 1: Client announces intent. Server reserves upload slot."""
    upload_id = hashlib.sha256(f"{user_id}{file_size}{sha256}".encode()).hexdigest()[:16]
    os.makedirs(UPLOAD_DIR, exist_ok=True)
    meta_path = f"{UPLOAD_DIR}/{upload_id}.meta"
    async with aiofiles.open(meta_path, "w") as f:
        await f.write(f"{file_size}:{sha256}:0")   # total:hash:offset
    return {"upload_id": upload_id, "offset": 0}

@app.patch("/upload/{upload_id}")
async def upload_chunk(upload_id: str, chunk: UploadFile,
                        content_range: str = Header(None)):
    """Step 2: Client sends chunks. Can be called multiple times."""
    meta = await read_meta(upload_id)
    offset = meta["offset"]

    # Parse Content-Range: bytes 0-65535/3145728
    start, end, total = parse_range(content_range)
    if start != offset:
        raise HTTPException(409, f"Expected offset {offset}, got {start}")

    # Append chunk to partial file
    part_path = f"{UPLOAD_DIR}/{upload_id}.part"
    async with aiofiles.open(part_path, "ab") as f:
        await f.write(await chunk.read())

    new_offset = end + 1
    await update_meta(upload_id, new_offset)

    if new_offset >= total:   # upload complete
        await finalize_upload(upload_id, part_path)
        return {"status": "complete"}

    return Response(status_code=204,
                    headers={"Upload-Offset": str(new_offset)})
Image Processing Pipeline

Raw uploaded images must be transformed into multiple device-optimised variants before serving. This happens asynchronously via a Kafka-driven worker pipeline.

What Processing Happens

Resize Variants

  • thumbnail — 150×150px (profile grid, search results)
  • medium — 400×400px (feed on mobile)
  • large — 1080×1080px (detail view on desktop)
  • Maintain aspect ratio with smart cropping

Format Conversion

  • Convert HEIC/PNG → WebP (30% smaller than JPEG at same quality)
  • Convert to AVIF for modern browsers (50% smaller than JPEG)
  • Keep original for download requests
  • Strip EXIF metadata (privacy: GPS coordinates)

Content Moderation

  • NSFW detection via ML model (async, non-blocking)
  • Object/scene detection for feed ranking signals
  • Perceptual hash (pHash) for duplicate detection
  • Face detection for tagging suggestions

from PIL import Image
import io, boto3, aiokafka
from dataclasses import dataclass

VARIANTS = [
    {"suffix": "thumb",  "size": (150, 150)},
    {"suffix": "medium", "size": (400, 400)},
    {"suffix": "large",  "size": (1080, 1080)},
]
s3 = boto3.client("s3")

async def process_photo(photo_id: str, s3_key: str):
    """Download raw, generate all variants, upload processed versions."""
    # 1. Download original from S3
    raw_bytes = s3.get_object(Bucket="raw-uploads", Key=s3_key)["Body"].read()
    img = Image.open(io.BytesIO(raw_bytes))

    # 2. Strip EXIF (privacy — removes GPS coordinates)
    img_clean = Image.frombytes(img.mode, img.size, img.tobytes())

    # 3. Generate each size variant in WebP
    for v in VARIANTS:
        resized = img_clean.copy()
        resized.thumbnail(v["size"], Image.LANCZOS)   # high-quality downscale
        buf = io.BytesIO()
        resized.save(buf, format="WEBP", quality=85, method=6)
        buf.seek(0)

        dest_key = f"processed/{photo_id}/{v['suffix']}.webp"
        s3.put_object(
            Bucket="cdn-origin", Key=dest_key,
            Body=buf, ContentType="image/webp",
            CacheControl="public, max-age=31536000, immutable"  # 1-year cache
        )

    # 4. Update DB: mark photo as processed
    await db.execute("UPDATE photos SET status='processed' WHERE id=?", photo_id)

    # 5. Publish completion event (triggers feed update, notifications)
    await kafka.publish("photo.processed", {"photo_id": photo_id})
Serving & CDN Strategy

Images must be served with <100ms p99 latency globally. This requires a Content Delivery Network (CDN) with intelligent edge caching, responsive image serving, and cache-busting on updates.

CDN Architecture

Cache-Control Strategy

Immutable URLs via content hash

Store images at content-addressable URLs: cdn.example.com/photos/{sha256_of_content}.webp. Since URL = content hash, the file never changes. Set Cache-Control: max-age=31536000, immutable → CDN caches forever, never needs to revalidate.

Responsive Image URL pattern

Instead of pre-generating every size variant, use query params + edge Lambda for on-the-fly transform: cdn.example.com/photos/abc.webp?w=400&q=85. Lambda processes and caches at the edge. Infinite size variants with zero pre-computation.

import time, uuid, boto3
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()
s3  = boto3.client("s3")

class UploadRequest(BaseModel):
    user_id: int;  filename: str;  content_type: str;  size_bytes: int

@app.post("/photos/upload-url")
async def get_upload_url(req: UploadRequest):
    """Return pre-signed S3 URL. Client uploads directly — never touches our servers."""
    photo_id = generate_snowflake_id()
    s3_key   = f"raw/{req.user_id}/{photo_id}/{req.filename}"

    # Validate: content type and size
    if req.content_type not in {"image/jpeg","image/png","image/heic","image/webp"}:
        raise ValueError("Unsupported type")
    if req.size_bytes > 8 * 1024 * 1024:  # 8 MB max
        raise ValueError("File too large")

    # Generate pre-signed URL valid for 15 minutes
    presigned = s3.generate_presigned_url(
        "put_object",
        Params={
            "Bucket": "raw-uploads",
            "Key": s3_key,
            "ContentType": req.content_type,
            "ContentLength": req.size_bytes,
        },
        ExpiresIn=900  # 15 minutes
    )

    # Store metadata in DB (status = uploading)
    now = int(time.time() * 1000)
    await db.execute(
        "INSERT INTO photos(id,user_id,s3_key,status,created_at) VALUES(?,?,?,'uploading',?)",
        [photo_id, req.user_id, s3_key, now]
    )

    return {
        "photo_id": photo_id,
        "upload_url": presigned,      # client PUT directly to this URL
        "expires_in": 900
    }

@app.post("/photos/{photo_id}/uploaded")
async def confirm_upload(photo_id: int, caption: str = ""):
    """Client calls after S3 PUT succeeds. Triggers async processing."""
    await db.execute(
        "UPDATE photos SET status='processing', caption=? WHERE id=?",
        [caption, photo_id]
    )
    # Publish to Kafka — triggers resize, format conversion, CDN warm
    await kafka.publish("photo.uploaded", {
        "photo_id": photo_id,
        "user_id":  await get_user_id(photo_id)
    })
    return {"status": "processing", "photo_id": photo_id}

    private final KafkaTemplate<String,PhotoEvent> kafka;

        validateRequest(req);

        long photoId = generateSnowflakeId();
        String s3Key = "raw/" + req.getUserId() + "/" + photoId + "/" + req.getFilename();

        // Generate pre-signed PUT URL

            .signatureDuration(Duration.ofMinutes(15))

Implementing Private Photos (Instagram Model)

Instagram allows users to set their account private — photos are visible only to approved followers. This changes the serving architecture dramatically: you can no longer cache images at the CDN without access control.

The Core Problem

Public photos: URL is the permission — anyone with the URL can view. CDN cache forever.

Private photos: URL is NOT the permission — you must check at serve time whether the requesting user is an approved follower. This kills naive CDN caching.

If you store private photos at a guessable URL (e.g., /photos/12345.jpg), anyone who finds the URL can view it — regardless of account privacy. This is the IDOR vulnerability.

IDOR — Insecure Direct Object Reference

The vulnerability: GET /photos/photo_id=42 — attacker changes 42 to 43 and sees someone else's private photo. Prevention requires:

  • Non-guessable IDs (Snowflake or UUID)
  • Authorization check on every request: "Can user X see photo Y?"
  • Never expose sequential or predictable IDs

Access Control Models

Option A: Check-before-redirect (ACL at serve time)

1
Client requests: GET /photos/view/photo_id=42
Request goes to your API server (not directly to CDN/S3).
2
API server checks ACL
Is requesting user a follower of the photo owner? Query: SELECT 1 FROM follows WHERE follower_id=? AND following_id=? AND status='approved'
3
If authorized: generate time-limited S3 signed URL
Redirect client to s3.amazonaws.com/bucket/photos/42.webp?Signature=xxx&Expires=1711000000. Valid for 1 hour. Client's browser caches it locally.
4
If unauthorized: return 403 Forbidden
Never expose the underlying S3 URL to unauthorized users. The signed URL itself is the access token.

Option B: Token-in-URL with CDN (Instagram's approach)

Instagram embeds a short-lived cryptographic token in the CDN URL. The CDN edge validates the token (via Lambda@Edge or Cloudflare Worker) without calling back to the auth server — preserving CDN cache efficiency while enforcing access control.

Signed CDN URL structurePattern
URL structure: https://cdn.instagram.com/photos/photo_42.webp
               ?user_id=1234&expires=1711000000&sig=HMAC_SHA256(secret, "1234:1711000000:photo_42")

CDN edge (Lambda@Edge / CF Worker) validates:
1. sig = HMAC(secret, user_id + expires + path)  ← prevents tampering
2. expires > now  ← prevents replay after expiry
3. user_id matches authenticated session  ← prevents sharing URLs

If valid → serve image from CDN cache
If invalid → 403 (never reveals the image)
Benefit: Most requests are CDN cache hits — no call back to origin. Only first access per user generates the token via the API server. Token is scoped to user_id, so the same CDN-cached image can be served to multiple authorized users with different tokens.

ACL Lookup Performance

Problem: ACL lookup on every photo serve

If every image request requires a DB query to check "is user X a follower of photo owner Y?", and you serve 70,000 images/sec, that's 70,000 DB queries/sec just for ACL checks.

Solution: Cache follower relationships

Cache follows:{user_id}:set in Redis as a Set of following IDs. ACL check = Redis SISMEMBER (O(1)). Cache invalidated when follow/unfollow happens. Typical follower list = 500 IDs = ~4 KB in Redis.

SISMEMBER follows:owner_id follower_id — O(1), ~0.5ms. Replace 70K DB queries/sec with 70K Redis ops/sec.

private_photos.py — ACL + Signed URL generationPython
import hmac, hashlib, time
from fastapi import FastAPI, HTTPException, Depends
import redis.asyncio as aioredis

app    = FastAPI()
redis  = aioredis.Redis(host="redis")
SECRET = b"super-secret-key"   # rotate periodically

async def check_photo_access(requester_id: int, photo_id: int) -> bool:
    """Returns True if requester can view this photo."""
    photo = await get_photo_meta(photo_id)

    if photo["user_id"] == requester_id:
        return True   # owner always can view own photos

    if photo["privacy"] == "public":
        return True

    if photo["privacy"] == "followers":
        # O(1) Redis Set lookup — avoid DB query
        is_follower = await redis.sismember(
            f"follows:{photo['user_id']}:approved",
            requester_id
        )
        return bool(is_follower)

    if photo["privacy"] == "close_friends":
        return await redis.sismember(
            f"close_friends:{photo['user_id']}",
            requester_id
        )

    return False   # only_me

def generate_signed_cdn_url(photo_id: int, user_id: int, ttl_seconds: int = 3600) -> str:
    """Generate CDN URL with HMAC token. Token = HMAC(secret, user:expires:photo_id)."""
    expires = int(time.time()) + ttl_seconds
    message = f"{user_id}:{expires}:{photo_id}".encode()
    sig     = hmac.new(SECRET, message, hashlib.sha256).hexdigest()[:16]
    path    = f"processed/{photo_id}/large.webp"
    return f"https://cdn.example.com/{path}?uid={user_id}&exp={expires}&sig={sig}"

@app.get("/photos/{photo_id}/url")
async def get_photo_url(photo_id: int, current_user_id: int = Depends(get_current_user)):
    """
    Returns a time-limited signed CDN URL.
    ACL checked here — CDN edge re-validates the HMAC token.
    """
    if not await check_photo_access(current_user_id, photo_id):
        raise HTTPException(403, "Access denied")

    url = generate_signed_cdn_url(photo_id, current_user_id, ttl_seconds=3600)
    return {"url": url, "expires_in": 3600}
Outcome
  • IDOR is the #1 private photo vulnerability — sequential IDs + no ACL check = anyone can view any photo by guessing IDs. Always use non-guessable IDs (Snowflake/UUID) and enforce ACL on every access.
  • CDN + private photos requires signed URLs — you can't use standard CDN caching for private content. Signed CDN URLs (CloudFront/CF) allow CDN caching while enforcing per-user access control at the edge.
  • Cache follower relationships in Redis Sets — SISMEMBER is O(1), ~0.5ms. Replace 70K DB queries/sec (for ACL checks) with 70K Redis ops/sec. Cache invalidated on follow/unfollow events.
  • HMAC-signed tokens prevent URL sharing — token encodes user_id + expiry. Sharing a signed URL to an unauthorized user won't work — the CDN edge validates the token and rejects mismatched user sessions.
  • Token expiry length is a security/UX tradeoff — short TTL (5 min) = very secure but requires frequent re-generation. Long TTL (24h) = better UX but revocation is slower. Instagram uses ~6h for stories.

Designing Gravatar — Global Avatar Service

Gravatar (Globally Recognised Avatar) is a service where users register an avatar tied to their email. Any website that knows the user's email can show their avatar without storing it locally. How do you design a system that generates and serves millions of personalised default avatars?

How Gravatar Works

1
Hash the user's email with MD5
hash = MD5(email.strip().lower()) → deterministic, privacy-preserving. No email stored on the serving server.
2
Construct CDN URL with hash
https://www.gravatar.com/avatar/{hash}?s=200&d=identicon. Size (s) and default (d) in query params.
3
Serve custom avatar or generate default
If user registered a custom avatar → serve it. If not → generate a deterministic default (identicon, monsterid, robohash) based on the hash.

Default Avatar Generation

Identicon (GitHub style)

Deterministic pixel pattern generated from the email hash. Same hash → always same pattern. 5×5 pixel grid, mirrored for symmetry, colored by hash bits.

Initial-based (Slack style)

Take first letter of name, pick background color from hash. Simple, clean. color = COLORS[hash[0] % len(COLORS)]. Infinitely scalable — generated on-the-fly with zero storage.

Robohash / Monsterid

Composite avatar: combine image parts (head + body + eyes) by hash → unique "monster" per user. Parts stored as sprite sheets. Assembled by hash at serve time.

Why MD5 for Gravatar (not SHA-256)?

MD5 was chosen for backwards compatibility and URL length (32 chars vs 64 for SHA-256). While MD5 is cryptographically broken for security purposes, for Gravatar it's used purely as a consistent hash (not for security). The email is never transmitted — only its hash. If you're building a new system, use SHA-256.

Dynamic OG Image Generation

Open Graph (OG) images are the preview images shown when a link is shared on Slack, Twitter, WhatsApp, iMessage. Dynamic OG images generate a customised preview for every piece of content — at request time.

What is an OG Image?

When you share a URL, platforms fetch <meta property="og:image" content="https://..."/> and display that image in the link preview. Static OG images are the same for all users. Dynamic OG images generate a custom preview per content item (article, profile, post).

Examples of Dynamic OG Images

  • GitHub repo preview — shows stars, language, description on a card
  • Vercel deployment preview — shows project name, status, deploy time
  • Dev.to article preview — author avatar, title, tags on styled background
  • Twitter/X card — user profile with follower count and recent activity
  • YouTube thumbnail — video title, channel, view count

Why It Matters

  • 80% of engagement on shared links comes from the preview card
  • Custom OG images increase CTR by 3–5×
  • Enables marketing information in the preview without changing the page
  • A/B testable without code deploys

Architecture: Two Approaches

On-the-fly with CDN caching

  • First request: Edge Lambda renders image (100–300ms)
  • Result cached at CDN edge for 24h
  • Subsequent requests: ~5ms CDN cache hit
  • Cache key = URL path + query params (content_id, version)
  • ✅ Zero pre-computation, infinite variants
  • ✅ Immediately reflects content changes (after cache TTL)

Pre-generated on content publish

  • Kafka consumer triggers OG generation when content is published
  • Generated image stored in S3 → served via CDN
  • First serve is always fast (already in CDN)
  • ❌ Requires storage for every content item
  • ❌ Must regenerate if template or content changes
  • Used by: Twitter cards, YouTube thumbnails

Designing a Concurrent HashTag Counter

Every Instagram post can contain hashtags like #travel. The system must count how many posts use each hashtag, support trending hashtags in real-time, and handle millions of concurrent updates without race conditions or performance degradation.

The Challenge

Naive DB approach

For every post with #travel, do: UPDATE hashtags SET count = count + 1 WHERE tag = 'travel'

At 100M posts/day × 7 hashtags = 700M UPDATEs/day = 8,000 UPDATEs/sec on a single hot row. InnoDB max = ~50K simple writes/sec. Locks on the travel row create massive contention at peak.

A viral post with #worldcup can receive 100,000 posts per minute → all hitting the same DB row → lock queue → timeouts → system failure.

What we need

  • Lock-free or minimal-contention counter increment
  • Exact count for display (or close enough — users don't notice ±100)
  • Real-time trending: top-N hashtags in the last 1 hour
  • Persistent count that survives Redis restarts
  • Efficient bulk extraction of trending tags

Architecture: Two-tier Counter System

Hot tier (Redis): Real-time approximate counts for display and trending. Redis INCR + ZINCRBY. Fast, in-memory. Potentially lossy on crash (acceptable).

Cold tier (DB + Kafka): Exact counts for reporting, analytics, historical trending. Kafka events batch-aggregated and flushed to MySQL every 60 seconds. Durable.

Pattern 2: Counter Salting (for extreme hot tags)

For viral hashtags like #worldcup during a major event (millions of increments per minute), even Redis INCR on a single key becomes a bottleneck. Use salting: distribute across 100 virtual keys, aggregate on read.

salted_counter.py — hot hashtag distributed counterPython
import random

SALT_COUNT = 100   # 100 virtual shards per hot tag

async def increment_hashtag(redis, tag: str):
    """Distribute increment across SALT_COUNT keys to avoid hot-key."""
    salt = random.randint(0, SALT_COUNT - 1)
    await redis.incr(f"htag:{tag}:{salt}")

async def get_hashtag_count(redis, tag: str) -> int:
    """Aggregate all salt shards — O(SALT_COUNT) keys but batched via MGET."""
    keys = [f"htag:{tag}:{i}" for i in range(SALT_COUNT)]
    values = await redis.mget(keys)
    return sum(int(v or 0) for v in values)

# At 1M increments/sec on #worldcup:
# Without salting: 1M INCR on 1 key → Redis bottleneck
# With 100 salts: 10K INCR/sec on each key → 100× below Redis limit
Approximate Counting — Count-Min Sketch

When you need to count millions of distinct hashtags and exact counts aren't required, probabilistic data structures can reduce memory by 99% while maintaining ~95% accuracy.

Count-Min Sketch

A 2D array of counters with D hash functions. To increment tag T: for each row i, increment table[i][hash_i(T) % width]. To read count: take the minimum across all rows (hence "min"). Overestimates (never underestimates) by a bounded amount.

Parameters & Accuracy

  • Width w = ⌈e/ε⌉ where ε = acceptable error fraction
  • Depth d = ⌈ln(1/δ)⌉ where δ = failure probability
  • For 1% error, 99% confidence: w=272, d=7 → ~13KB for any N
  • Exact counting 100M hashtags at 8 bytes each = 800 MB
  • Count-Min for same = 13 KB — 60,000× less memory

When to Use

  • Top-N trending where approximate rank is fine
  • DDoS detection: approximate request count per IP
  • Real-time analytics where exact counts aren't needed
  • NOT for billing or financial counts — overestimation is unacceptable
  • Redis has native Count-Min Sketch: CMS.INITBYDIM

HyperLogLog for Unique Users per Hashtag

    """Extract lowercase hashtags from post caption."""
    return [tag.lower() for tag in re.findall(r'#(\w+)', caption)]

async def process_post(post_id: int, user_id: int, caption: str):
    """Called on post creation. Updates counters and sorted set for trending."""
    tags = extract_hashtags(caption)
    if not tags: return

    hour_bucket = int(time.time()) // 3600   # e.g., 473618
    trending_key = f"htag:trending:{hour_bucket}"

    async with redis.pipeline() as pipe:   # batch all Redis commands
        for tag in tags:
            # 1. Increment total count
            pipe.incr(f"htag:count:{tag}")
            # 2. Increment trending score for current hour
            pipe.zincrby(trending_key, 1, tag)
            # 3. Track unique users (HLL) — for analytics
            pipe.pfadd(f"htag:users:{tag}", user_id)
        # 4. Auto-expire trending bucket after 2 hours
        pipe.expire(trending_key, 7200)
        await pipe.execute()   # single round-trip for all commands

    # 5. Store hashtag→post mapping (for feed/search)
    for tag in tags:
        await db.execute(
            "INSERT INTO photo_hashtags(photo_id,tag) VALUES(?,?)", [post_id, tag]
        )

@app.get("/trending")
async def trending(window_hours: int = 3, top_n: int = 10):
    """Trending hashtags over last N hours via ZUNIONSTORE."""
    now = int(time.time()) // 3600
    buckets = [f"htag:trending:{now-i}" for i in range(window_hours)]
    merged  = f"htag:merged:{now}:{window_hours}"

    await redis.zunionstore(merged, buckets)
    await redis.expire(merged, 300)   # cache merged result 5 min

    results = await redis.zrevrange(merged, 0, top_n-1, withscores=True)
    return [{"tag": tag, "posts": int(score)} for tag, score in results]

Designing Message Indicators

Message indicators are the "1 new message from 3 people" badge on the chat icon. The system must track: how many unread messages exist per sender, show real-time updates, and mark messages as read when viewed.

What Needs to Be Tracked

  • Unread count — how many unread messages from each sender
  • Total unread — aggregate across all senders (badge number)
  • Delivery status — sent / delivered / read (double tick)
  • Last message preview — first N chars for conversation list
  • Real-time update — badge increments as new messages arrive

Scale

  • WhatsApp: 100B messages/day across 2B users
  • Each message generates 1–3 receipt events (sent/delivered/read)
  • Unread count per user can be 0 to thousands
  • Badge must update within 1 second of new message
  • Read receipts must propagate to sender within 2 seconds

Data Model for Indicators

Delivery Receipts — Sent, Delivered, Read

Delivery receipts (the ticks on WhatsApp/iMessage) track the journey of a message from sent to delivered to read. Each state transition must be propagated back to the sender in real-time.

Three-state Receipt System

Sent (1 tick)Server persists message to Cassandra✓Server ACKs the send to sender's WS
Delivered (2 ticks)Recipient's device receives message via WS or push notification✓✓Client sends POST /receipt/{msg_id}/delivered

Group Message Receipts

Complication: In a group of 100 people, message is "delivered" only when delivered to all 100, "read" when read by all 100. Need to track N receipts per message. Show individual per-user receipt status. System must not mark as "delivered to all" until last member receives it. Implement with a counter: delivered_count, read_count — increment atomically until they hit group size.

async def on_message_sent(sender_id: int, receiver_id: int,
                           msg_id: int, preview: str):
