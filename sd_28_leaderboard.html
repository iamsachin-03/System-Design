Leaderboard / Gaming System
Design a real-time global leaderboard for a game with 10 million active players — storing scores, ranking players globally and within friend groups, handling 100K score updates per second, answering "what is my rank?" in under 10ms, and resetting leaderboards at season boundaries.
🔴 MUST DO 🟢 Medium Redis ZSet ZADD / ZRANK Sharding Score Aggregation Season Reset
🎯 Requirements
The leaderboard problem is deceptively simple until you hit scale. Redis sorted sets are the canonical answer — O(log N) inserts and O(log N) rank queries. The challenge is what to do when 10M players don't fit in one Redis instance, and how to support "friends leaderboard" without scanning all friends' scores.

FUNCTIONAL
Submit score update for a player
Get global top-N players (leaderboard page)
Get player's own global rank + score
Get players around me (±5 ranks)
Friends leaderboard (rank among my friends)
Weekly/seasonal reset with archiving
NON-FUNCTIONAL
10M active players, 100K score updates/sec
Rank query: <10ms p99
Top-N query: <20ms for N≤100
Eventual consistency for rank (±few seconds lag is OK)
Season reset: complete within 5 minutes
💬 Phase 1 — Core Data Structure
PHASE 1 · REDIS SORTED SET
IR
INTERVIEWER
What data structure would you use and why?
ME
YOU
Redis Sorted Set (ZSet) is purpose-built for this problem:

ZADD leaderboard {score} {player_id}: Insert or update a player's score. O(log N). If the player already exists, updates the score and re-sorts. Idempotent — safe to call repeatedly.

ZREVRANK leaderboard {player_id}: Get player's rank (0-indexed, descending by score). O(log N). This is the core operation — it's O(log N) not O(N) because ZSets are backed by a skip list internally.

ZREVRANGE leaderboard 0 99: Get top 100 players. O(log N + 100). Extremely fast.

ZREVRANGEBYSCORE leaderboard +inf {min_score} WITHSCORES: Get all players with score ≥ threshold. Used for finding players around me: get my score, then find players within score±delta.

ZSCORE leaderboard {player_id}: Get raw score. O(1).

The ZSet is backed by two structures: a hash map for O(1) score lookup per member, and a skip list for O(log N) rank queries. It's the perfect data structure for leaderboards — no need to reinvent it.
💡 The key thing to know: ZREVRANK is O(log N), not O(N). Redis's skip list gives you rank queries in logarithmic time. This is why Redis ZSet is the standard answer for leaderboards.
🔧 Phase 2 — Scaling to 10M Players
PHASE 2 · SCALE
IR
INTERVIEWER
10M players × ~20 bytes per entry = 200MB. That fits in one Redis node. But 100K updates/sec — does one Redis handle that?
ME
YOU
200MB data fits fine. The bottleneck is throughput. Redis single-threaded: ~100-150K commands/sec ceiling. At 100K ZADD/sec we're at the limit — any additional load (reads, monitoring, replication) pushes over the edge.

Three approaches:

1. Batch writes (best for throughput): Instead of a ZADD per score update, buffer updates in the application layer for 1 second, then apply as a pipeline: PIPELINE [ZADD leaderboard s1 p1, ZADD s2 p2, ...] EXEC. Reduces Redis commands by 100× (100K individual → 1K batches of 100). Redis pipeline avoids round-trip latency overhead. Best option — reduces Redis load while maintaining near-real-time freshness.

2. Read replicas for rank queries: Score writes go to primary. Rank reads (ZREVRANK) can go to read replicas. Typical ratio: 80% reads, 20% writes. 4 read replicas → 80% of traffic handled by replicas, primary handles 20%.

3. Sharding for extreme scale (100M+ players): Hash player_id modulo N shards: player goes to shard=hash(player_id)%N. Score updates go to their assigned shard. But: getting global rank now requires querying all N shards, getting count of players with higher score on each shard, summing them. Adds complexity but enables virtually unlimited scale.
👥 Phase 3 — Friends Leaderboard
PHASE 3 · FRIENDS
IR
INTERVIEWER
How do you show a player their rank among their friends? They might have 500 friends.
ME
YOU
Two approaches:

1. Dynamic friends ZSet (per-user, ephemeral): At query time, fetch player's friend list from the social graph DB. Get their scores via ZMSCORE leaderboard [friend1, friend2, ...] (O(N) where N=friend count). Create an ephemeral ZSet with just friend scores, run ZREVRANK on it. Compute in application memory rather than Redis. For 500 friends × 1ms each = 500ms — too slow as written, but ZMSCORE is a single command that returns all 500 scores at once.

2. Per-user friends ZSet in Redis: Maintain a separate ZSet per player: friends_lb:{player_id}. When a friend updates their score, also update all their friends' friend-ZSets (fan-out on write). If a player has 500 friends, one score update writes to 501 ZSets. At 100K updates/sec × avg 200 friends = 20M ZSet writes/sec — this is write amplification, and it's expensive. Better for read-heavy patterns (player checks friends leaderboard frequently).

Recommended: Hybrid. ZMSCORE approach for players querying friends leaderboard (read-time). Pre-computed fan-out only for top influencers (players with >10K friends) because they are queried constantly. Simple approach first, optimise when profiling shows it's needed.

// Leaderboard — Java: Redis ZSet operations
// ══════════════════════════════════════════════════════

@Service
public class LeaderboardService {

    private static final String LB_KEY    = "leaderboard";
    private static final int    BATCH_SIZE = 200; // buffer before pipeline flush

    @Autowired private RedisTemplate<String,String> redis;
    @Autowired private SocialGraphService socialGraph;

    // Pending updates buffer (flushed every 1s by scheduler)
    private final ConcurrentLinkedQueue<ScoreUpdate> pending = new ConcurrentLinkedQueue<>();

    /** Add score update to buffer. Thread-safe, non-blocking. */
    public void submitScore(String playerId, double score) {
        pending.offer(new ScoreUpdate(playerId, score));
    }

    /** Flush buffered updates to Redis via pipeline. Called every 1s. */
    @Scheduled(fixedRate = 1000)
    public void flushPending() {
        List<ScoreUpdate> batch = new ArrayList<>();
        ScoreUpdate update;
        while ((update = pending.poll()) != null && batch.size() < BATCH_SIZE) {
            batch.add(update);
        }
        if (batch.isEmpty()) return;

        // Redis pipeline: send all ZADDs in one round-trip
        redis.executePipelined(conn -> {
            batch.forEach(u -> conn.zSetCommands()
                .zAdd(LB_KEY.getBytes(), u.score(), u.playerId().getBytes()));
            return null;
        });
    }

    /** Get player's global rank (1-indexed). O(log N). */
    public long getGlobalRank(String playerId) {
        Long rank = redis.opsForZSet().reverseRank(LB_KEY, playerId);
        return rank == null ? -1 : rank + 1;
    }

    /** Get top N players with scores. O(log N + N). */
    public List<PlayerRank> getTopN(int n) {
        Set<ZSetOperations.TypedTuple<String>> top =
            redis.opsForZSet().reverseRangeWithScores(LB_KEY, 0, n - 1);
        int[] rank = {1};
        return top.stream()
            .map(t -> new PlayerRank(t.getValue(), t.getScore(), rank[0]++))
            .collect(Collectors.toList());
    }

    /** Get players surrounding a player (±count positions). */
    public List<PlayerRank> getAround(String playerId, int count) {
        Long rank = redis.opsForZSet().reverseRank(LB_KEY, playerId);
        if (rank == null) return List.of();
        long start = Math.max(0, rank - count);
        long end   = rank + count;
        Set<ZSetOperations.TypedTuple<String>> around =
            redis.opsForZSet().reverseRangeWithScores(LB_KEY, start, end);
        int[] r = {(int) start + 1};
        return around.stream()
            .map(t -> new PlayerRank(t.getValue(), t.getScore(), r[0]++))
            .collect(Collectors.toList());
    }

    /** Friends leaderboard: rank among friends. */
    public List<PlayerRank> getFriendsLeaderboard(String playerId) {
        List<String> friends = socialGraph.getFriends(playerId);
        friends.add(playerId); // include self

        // Bulk score fetch — single Redis command
        List<Double> scores = redis.opsForZSet().score(LB_KEY, friends.toArray());
        List<PlayerScore> friendScores = new ArrayList<>();
        for (int i = 0; i < friends.size(); i++) {
            if (scores.get(i) != null) {
                friendScores.add(new PlayerScore(friends.get(i), scores.get(i)));
            }
        }
        // Sort in-memory by score descending
        friendScores.sort((Comparator.comparingDouble(PlayerScore::score).reversed()));
        int[] r = {1};
        return friendScores.stream()
            .map(ps -> new PlayerRank(ps.playerId(), ps.score(), r[0]++))
            .collect(Collectors.toList());
    }

    /** Season reset: atomic rename, async archive. */
    public void resetSeason(int seasonNumber) {
        String archiveKey = "leaderboard:season:" + seasonNumber;
        // Atomic RENAME — zero downtime, new leaderboard starts empty
        redis.rename(LB_KEY, archiveKey);
        // Async: export archive to PostgreSQL for history
        CompletableFuture.runAsync(() -> archiveService.export(archiveKey, seasonNumber));
    }
}

# Leaderboard — Python: Redis ZSet + friends
# ══════════════════════════════════════════════════════

import redis, threading, time
from collections import deque
from dataclasses import dataclass
from typing import List, Optional

r = redis.Redis(decode_responses=True)
LB_KEY = "leaderboard"

@dataclass
class PlayerRank:
    player_id: str
    score:     float
    rank:      int


class LeaderboardService:
    """Buffered score writes + all leaderboard queries."""

    def __init__(self):
        self._buffer = deque()
        self._lock   = threading.Lock()
        # Flush buffer every 1 second
        t = threading.Thread(target=self._flush_loop, daemon=True)
        t.start()

    def submit_score(self, player_id: str, score: float):
        """Non-blocking: add to buffer."""
        with self._lock:
            self._buffer.append((player_id, score))

    def _flush_loop(self):
        while True:
            time.sleep(1)
            self._flush()

    def _flush(self):
        with self._lock:
            batch = list(self._buffer)
            self._buffer.clear()
        if not batch: return
        # Pipeline: all ZADDs in one round-trip
        pipe = r.pipeline()
        for player_id, score in batch:
            pipe.zadd(LB_KEY, {player_id: score})
        pipe.execute()

    def get_rank(self, player_id: str) -> Optional[int]:
        rank = r.zrevrank(LB_KEY, player_id)
        return rank + 1 if rank is not None else None

    def get_score(self, player_id: str) -> Optional[float]:
        return r.zscore(LB_KEY, player_id)

    def get_top_n(self, n: int = 100) -> List[PlayerRank]:
        entries = r.zrevrange(LB_KEY, 0, n - 1, withscores=True)
        return [PlayerRank(pid, score, i + 1)
                for i, (pid, score) in enumerate(entries)]

    def get_around(self, player_id: str, count: int = 5) -> List[PlayerRank]:
        rank = r.zrevrank(LB_KEY, player_id)
        if rank is None: return []
        start = max(0, rank - count)
        end   = rank + count
        entries = r.zrevrange(LB_KEY, start, end, withscores=True)
        return [PlayerRank(pid, score, start + i + 1)
                for i, (pid, score) in enumerate(entries)]

    def get_friends_leaderboard(self, player_id: str, friends: List[str]) -> List[PlayerRank]:
        """Rank among friends. ZMSCORE for bulk score fetch."""
        all_players = friends + [player_id]
        scores_raw  = r.zmscore(LB_KEY, all_players)  # O(N) single command
        ranked = sorted(
            [(pid, score) for pid, score in zip(all_players, scores_raw)
             if score is not None],
            key=lambda x: -x[1])
        return [PlayerRank(pid, score, i + 1)
                for i, (pid, score) in enumerate(ranked)]

    def reset_season(self, season_number: int):
        """Atomic rename → archive. Zero downtime."""
        archive_key = f\'leaderboard:season:{season_number}\'
        r.rename(LB_KEY, archive_key)  # O(1), atomic
        # Background: export archive_key to PostgreSQL, then delete
        print(f\'[RESET] Season {season_number} archived. New season started.\')
⚖️ Tradeoffs
📊 Score Storage: ZSet vs DB + Sort vs Segment Tree
✅ Redis ZSet (chosen)

O(log N) ZADD + O(log N) ZREVRANK. 640MB for 10M players. Ops team knows Redis. Built-in persistence. Production-proven at Riot Games, Discord, Roblox. Zero custom code needed for the core data structure.

DB with ORDER BY

SELECT COUNT(*) FROM scores WHERE score > ? — O(N) table scan or O(log N) with index. Index helps but doesn't give O(log N) rank like ZSet. Also: 100K writes/sec will overwhelm a relational DB's write path. Only works at small scale (<100K players).

Segment Tree / Fenwick Tree

Custom in-memory tree. O(log N) updates and rank queries. Used when score range is bounded (e.g., 0-1,000,000). More memory-efficient than ZSet for dense score ranges. Requires custom implementation. Not necessary when Redis already solves the problem perfectly.

⏱ Write Strategy: Immediate vs Batched vs Queue
Immediate (per-event ZADD)

Every score update instantly hits Redis. Rank always current. But: 100K individual commands/sec → single-threaded Redis at limit. Any other Redis work starves. Not recommended at this scale.

✅ Batched (1s pipeline)

Buffer updates, flush every second via pipeline. 100K/sec → 1K pipelines/sec. Redis easily handles. Rank lag: up to 1 second — imperceptible to users. Best throughput/freshness tradeoff. Used by virtually all high-scale game leaderboards.

Kafka queue

Score events → Kafka → consumer writes to Redis. Adds ~100ms latency but decouples ingest from write pressure. Useful when score sources are distributed across many services. Overkill for single-service scenarios — adds operational complexity without benefit over local batching.
