E-Commerce Search (Amazon/Shopify)
Design product search for an e-commerce platform with 500M products — full-text search with relevance ranking, faceted filters (brand, price, rating, category), spell correction, personalised result re-ranking, and near-real-time product index updates from a catalog of constantly changing prices and inventory.
🔴 MUST DO 🟣 Hard Elasticsearch Inverted Index Faceted Filters BM25 Ranking Personalisation Near-realtime Index
🎯 Requirements
Search is deceptively hard at e-commerce scale. The inverted index gives you relevance matching. Facets filter results. Personalisation re-ranks for each user. Near-realtime index updates require decoupling catalog changes from search availability. All this must happen in <100ms.

FUNCTIONAL
Full-text keyword search across product title, description, brand, category
Faceted filters: price range, brand, rating, in-stock, category
Spell correction ("iphone" → suggested "iPhone 15")
Personalised ranking: boost products user has viewed/bought in past
Autocomplete / typeahead for search bar
Near-realtime catalog updates (new products, price changes) visible in <30s
NON-FUNCTIONAL
500M products in the index
50K search queries/sec peak
Search latency: <100ms p99
Index update lag: <30s from catalog change to searchable
99.9% availability — search must stay up during indexing
💬 Phase 1 — Inverted Index & Relevance
PHASE 1 · SEARCH FUNDAMENTALS
IR
INTERVIEWER
User searches "wireless headphones under $50". Walk me through exactly what happens.
ME
YOU
Step 1 — Query analysis: Tokenise and analyse the query. "wireless headphones" → tokens: ["wireless", "headphone" (stemmed)]. The price filter "under $50" is extracted as a structured filter: price < 50. Stop words ("under") are dropped.

Step 2 — Inverted index lookup: For each token, the inverted index maps token → [doc_id, frequency, field, positions]. "wireless" → [prod_1234, prod_5678, ...]. "headphone" → [prod_5678, prod_9012, ...]. The intersection gives candidate products containing both tokens. Elasticsearch handles this automatically.

Step 3 — Relevance scoring (BM25): BM25 is Elasticsearch's default. Score depends on: (a) term frequency in this document (TF), (b) inverse document frequency — rare words in the corpus get higher weight (IDF), (c) field length normalization — a short title match outscores a long description match. "wireless headphones" in the product title scores much higher than in a long description.

Step 4 — Filter application: Apply filter: price < 50 AND in_stock: true. Filters don't affect score — they simply include/exclude documents. Elasticsearch evaluates filters before scoring for efficiency (cached as bitsets).

Step 5 — Return top-K with facets: Return top 20 results plus facet counts: "brand: {Sony: 142, JBL: 87, ...}", "rating: {4+stars: 234, ...}". User can now drill down.
💡 BM25 vs TF-IDF: BM25 is BM (Best Matching) 25. It improves on TF-IDF by adding term saturation (doubling occurrences doesn't double the score — useful for spam prevention) and document length normalization. It's been the gold standard since 1994 and is still what Elasticsearch uses by default.
🎯 Phase 2 — Facets & Personalisation
PHASE 2 · FACETS + RANKING
IR
INTERVIEWER
How do you show brand filters with counts like "Sony (142), JBL (87)" without a separate query per brand?
ME
YOU
Aggregation queries (Elasticsearch Aggs): In a single query, alongside the main search, attach an aggregation on the brand field: "aggs": {"brands": {"terms": {"field": "brand.keyword", "size": 20}}}. Elasticsearch runs the full-text search AND counts per-brand simultaneously — one round-trip. The result includes both the top-20 product hits AND the brand distribution for all matching products. Same for price ranges (histogram aggregation) and ratings (range aggregation).

Filter vs Post-filter: A subtle but important distinction. A "filter" applied to the query excludes documents from both results and aggregation counts. A "post_filter" filters the result documents but the aggregation still counts ALL matching documents before the filter. This gives the "showing results for Sony, but here's how many other brands also match" behaviour — standard on Amazon.
IR
INTERVIEWER
User A bought Sony headphones 3 times. User B bought Apple AirPods. Same search "wireless headphones" — should they see the same results?
ME
YOU
No — personalisation should re-rank results. Two approaches:

1. Score boosting at query time: Extend the Elasticsearch query with a function_score block that adds a personalisation boost. For user A: brands they've purchased from get +2.0 boost. For user B: Apple gets +3.0 boost. This is computed per-request from a user preference profile stored in a fast KV store (Redis). Adds ~5ms to query time.

2. Two-phase retrieval + re-ranking: Phase 1: Elasticsearch returns top-200 by BM25 relevance. Phase 2: A ranking service re-orders those 200 using an ML model with user features, item features, and interaction history. Show top-20 to user. More expensive but better quality. Used by Amazon and ASOS for their ranking models.
⚡ Phase 3 — Near-realtime Index Updates
PHASE 3 · INDEX FRESHNESS
IR
INTERVIEWER
A seller changes a price. How quickly does this appear in search, and how do you ensure indexing doesn't degrade search performance?
ME
YOU
Catalog changes → Kafka topic → Elasticsearch indexer service. The indexer consumes events and calls PUT /products/_doc/{product_id} on Elasticsearch. Elasticsearch's default refresh interval is 1 second — new documents become searchable within 1s of being written. End-to-end: catalog change → Kafka (sub-100ms) → indexer → ES refresh (~1s) = under 5 seconds.

Protecting search during heavy indexing: Bulk indexing (e.g., adding 10M new products) uses the Bulk API and runs against a separate indexing thread pool. Elasticsearch separates read (search) and write (indexing) thread pools, so heavy indexing doesn't block search queries. Additionally: index new documents to a "write alias" while search uses a "read alias" — during a full re-index, the new index is built in the background, then aliases are atomically swapped. Zero-downtime re-indexing.

Routing: Price and inventory change extremely frequently (millions of updates/day). To avoid re-indexing the full document, use partial updates: POST /products/_update/{id} {"doc": {"price": 49.99}}. Elasticsearch merges this into the existing document. Much cheaper than full document re-index.

@Service
public class SearchService {

    @Autowired private ElasticsearchClient esClient;
    @Autowired private RedisTemplate<String,Object> redis;
    @Autowired private UserProfileService userProfiles;

    public SearchResult search(SearchRequest req) {
        // 1. Check cache for common queries
        String cacheKey = "search:" + req.getCacheKey();
        SearchResult cached = (SearchResult) redis.opsForValue().get(cacheKey);
        if (cached != null) return cached;

        // 2. Get user preference profile for personalisation
        UserProfile profile = userProfiles.get(req.getUserId());

        // 3. Build ES query
        SearchResponse<Product> response = esClient.search(s -> s
            .index("products")
            .size(20)
            .query(q -> q
                .functionScore(fs -> fs
                    .query(inner -> inner
                        .bool(b -> b
                            .must(m -> m.multiMatch(mm -> mm
                                .query(req.getQuery())
                                .fields("title^3", "brand^2", "description")
                                .type(TextQueryType.BestFields)
                                .fuzziness("AUTO"))) // spell correction
                            .filter(f -> f
                                .range(r -> r
                                    .field("price")
                                    .lte(JsonData.of(req.getMaxPrice())))
                                .filter(f2 -> f2
                                    .term(t -> t.field("in_stock").value(true))))
                        )
                    )
                    // Popularity boost from pre-computed score
                    .functions(fn -> fn.fieldValueFactor(fvf -> fvf
                        .field("popularity_score").modifier(FieldValueFactorModifier.Log1p).factor(1.5)))
                    // Personalisation: boost brands user has purchased
                    .functions(fn -> buildPersonalBoosts(fn, profile))
                    .scoreMode(FunctionScoreMode.Sum)
                )
            )
            // Facet aggregations
            .aggregations("brands", a -> a.terms(t -> t.field("brand.keyword").size(20)))
            .aggregations("price_hist", a -> a.histogram(h -> h.field("price").interval(10.0)))
            .aggregations("avg_rating", a -> a.range(r -> r
                .field("rating_avg")
                .ranges(rv -> rv.from("4.0").to("5.0"), rv -> rv.from("3.0").to("4.0"))))
            , Product.class);

        SearchResult result = toSearchResult(response);

        // 4. Cache results for 60s (only for non-personalised queries)
        if (!req.isPersonalised()) {
            redis.opsForValue().set(cacheKey, result, 60, TimeUnit.SECONDS);
        }
        return result;
    }

    private FunctionScoreQuery.Builder buildPersonalBoosts(FunctionScoreQuery.Builder b, UserProfile p) {
        p.getFavouriteBrands().forEach(brand ->
            b.functions(fn -> fn
                .filter(f -> f.term(t -> t.field("brand.keyword").value(brand)))
                .weight(2.0)));  // 2× weight for preferred brands
        return b;
    }
}

// ── Catalog change indexer ────────────────────────────
@KafkaListener(topics = "catalog-changes")
public void indexProduct(CatalogEvent event) {
    switch (event.getType()) {
        case CREATED, UPDATED -> esClient.index(i -> i
            .index("products")
            .id(event.getProductId())
            .document(event.getProduct()));
        case PRICE_CHANGED -> esClient.update(u -> u
            .index("products")
            .id(event.getProductId())
            .doc(Map.of("price", event.getNewPrice()))); // partial update
        case DELETED -> esClient.delete(d -> d
            .index("products").id(event.getProductId()));
    }
}

from elasticsearch import Elasticsearch
import redis, json
from dataclasses import dataclass
from typing import Optional, List

es = Elasticsearch([\'http://es-cluster:9200\'])
r  = redis.Redis()


def search_products(query: str, max_price: float, user_id: str = None,
                    brands: List[str] = None, min_rating: float = 0,
                    page: int = 0, page_size: int = 20) -> dict:
    """Full e-commerce search with facets, filters, personalisation."""

    # 1. Check Redis cache
    cache_key = f\'search:{query}:{max_price}:{brands}:{min_rating}:{page}\'
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)

    # 2. Build filters (cached as bitsets by ES — very fast)
    filters = [
        {\'range\': {\'price\': {\'lte\': max_price}}},
        {\'term\':  {\'in_stock\': True}},
    ]
    if brands:
        filters.append({\'terms\': {\'brand.keyword\': brands}})
    if min_rating > 0:
        filters.append({\'range\': {\'rating_avg\': {\'gte\': min_rating}}})

    # 3. Personalisation boosts
    personal_functions = []
    if user_id:
        fav_brands = get_user_favourite_brands(user_id)  # from Redis
        for brand in fav_brands:
            personal_functions.append({
                \'filter\': {\'term\': {\'brand.keyword\': brand}},
                \'weight\': 2.0
            })

    # 4. Build ES query
    body = {
        \'from\': page * page_size, \'size\': page_size,
        \'query\': {
            \'function_score\': {
                \'query\': {\'bool\': {
                    \'must\': {\'multi_match\': {
                        \'query\': query,
                        \'fields\': [\'title^3\', \'brand^2\', \'description\'],
                        \'fuzziness\': \'AUTO\'  # spell correction
                    }},
                    \'filter\': filters
                }},
                \'functions\': [
                    {\'field_value_factor\': {\'field\': \'popularity_score\',
                                            \'modifier\': \'log1p\', \'factor\': 1.5}},
                    *personal_functions
                ],
                \'score_mode\': \'sum\'
            }
        },
        # Facet aggregations
        \'aggs\': {
            \'brands\':     {\'terms\': {\'field\': \'brand.keyword\', \'size\': 20}},
            \'price_hist\': {\'histogram\': {\'field\': \'price\', \'interval\': 10}},
            \'ratings\':    {\'range\': {\'field\': \'rating_avg\',
                \'ranges\': [{\'from\': 4}, {\'from\': 3, \'to\': 4}, {\'to\': 3}]}}
        }
    }

    resp   = es.search(index=\'products\', body=body)
    result = _format_result(resp)

    # 5. Cache non-personalised results 60s
    if not user_id:
        r.setex(cache_key, 60, json.dumps(result))

    return result


def index_product(product: dict):
    """Full document index on create/update."""
    es.index(index=\'products\', id=product[\'id\'], body=product)


def update_price(product_id: str, new_price: float):
    """Partial update — only re-indexes the price field."""
    es.update(index=\'products\', id=product_id,
              body={\'doc\': {\'price\': new_price}})


def _format_result(resp: dict) -> dict:
    hits   = resp[\'hits\'][\'hits\']
    total  = resp[\'hits\'][\'total\'][\'value\']
    brands = resp[\'aggregations\'][\'brands\'][\'buckets\']
    return {
        \'total\':   total,
        \'results\': [h[\'_source\'] for h in hits],
        \'facets\':  {\'brands\': [{\'key\': b[\'key\'], \'count\': b[\'doc_count\']} for b in brands]}
    }
⚖️ Tradeoffs
🔍 Search Engine: Elasticsearch vs Solr vs Typesense vs Custom
✅ Elasticsearch (chosen)

Industry standard for e-commerce search. Rich query DSL, aggregations, function_score for personalisation. Cloud-managed (Elastic Cloud, AWS OpenSearch). 500M documents: well within Elasticsearch's capability. Used by eBay, Shopify, Etsy, Netflix.

Typesense / Algolia

Easier to operate, great DX, fast for <100M documents. Algolia is fully managed. Both lack the depth of Elasticsearch's aggregation and function_score capabilities. Algolia is very expensive at scale (per-search pricing). Good for startups, not 500M products at 50K QPS.

Custom inverted index

Amazon, Google built their own. Gives maximum control over ranking, indexing speed, cost. 18+ months to match Elasticsearch's feature set. Only justified at >1B products or when standard tools' limitations block business objectives. Not appropriate for this scale.

🎯 Ranking: BM25 vs ML-based (LambdaMART) vs Hybrid
BM25 only

Elasticsearch default. Excellent for textual relevance. No learning from user behaviour — treats all users equally. Misses signals like purchase history, seasonal trends, inventory. Good starting point; inadequate for a mature e-commerce search at Amazon scale.

✅ BM25 + function_score

Layer signals (popularity, rating, recency) on top of BM25 via function_score. Add personalisation boosts per user. 80% of the ML benefit with 20% of the complexity. Works in Elasticsearch natively. Appropriate for 1M-500M products.

✅ Two-stage (BM25 + LTR)

Phase 1: BM25 retrieves top-200. Phase 2: Learning-to-Rank (LambdaMART, XGBoost) model re-ranks using 200+ features (user/item/context). Amazon, JD.com, eBay use this. 20-30% conversion rate improvement over BM25 alone. Requires A/B testing infrastructure and offline feature pipelines. Worth it at scale.
