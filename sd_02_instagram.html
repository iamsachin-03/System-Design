🎙 Phase 1 — Requirements & High-Level Direction
PHASE 1 · SCOPE
IR
INTERVIEWER
Design Instagram. Where do you start?
ME
YOU
Before diving in, I want to clarify scope. Instagram is huge — Stories, Reels, DMs, Shopping. I'd like to focus on the core social feed system: photo/video upload, follow graph, news feed, likes, and comments. Can I confirm that's the scope?
IR
INTERVIEWER
Yes. 500M DAU, 100M uploads/day, news feed, follow, like, comment.
ME
YOU
Perfect. At 500M DAU, I immediately see four distinct hard problems:

1. Media Storage: 100M uploads × ~200KB average = ~20TB of new media every day. We need distributed object storage with CDN delivery.
2. News Feed Generation: When 500M people open the app, we need to serve each person a personalised feed in under 200ms. This is the fanout problem.
3. The Celebrity Problem: Kylie Jenner has 400M followers. When she posts, do we write to 400M feed caches simultaneously? That's a write storm.
4. Like Counters at Scale: A viral post gets millions of likes per minute. How do we count accurately without DB contention?

I'll break the system into 4 subsystems: Media Service, User Graph Service, Feed Service, and Interaction Service. Shall I proceed?
💡 Pro tip: Immediately identifying the hard problems (fanout, celebrity problem) before drawing any diagram shows senior-level thinking. The interviewer wants to see that you know where the dragons are.
📸 Phase 2 — Media Upload & Storage
PHASE 2 · MEDIA PIPELINE
IR
INTERVIEWER
Walk me through the photo upload flow end-to-end.
ME
YOU
The upload flow has two phases — upload and processing:

Upload Phase:
  1. Client sends POST /api/v1/posts with caption and metadata (no media yet)
  2. API Server creates a post record in DB with status PROCESSING, returns a post_id
  3. API Server generates a pre-signed S3 URL valid for 15 minutes, returns it to client
  4. Client uploads the raw media file directly to S3 using the pre-signed URL — bypasses our servers entirely

Why pre-signed URL? If the client uploads through our app server, it becomes a bandwidth bottleneck — 100M × 200KB = 20TB/day through our servers. Unacceptable. Pre-signed URLs offload this directly to S3.

Processing Phase:
  1. S3 upload triggers an S3 Event → pushes a message to Kafka (topic: media-uploads)
  2. Media Processing Service consumes the event, downloads raw file from S3
  3. Generates multiple resolutions: thumbnail (150px), medium (640px), full (1080px), and video transcoding (360p, 720p, 1080p)
  4. Uploads processed files back to S3 under CDN-friendly paths
  5. Updates post status to ACTIVE in DB
  6. Triggers fanout to followers' news feeds (via Feed Service)
IR
INTERVIEWER
Why not process images synchronously during upload?
ME
YOU
Async processing gives much better UX. The user sees their post instantly with a "processing" state. Resizing a 50MB video can take 10–30 seconds — if we did this synchronously, the user would wait. With async:
  • Upload API responds in <200ms
  • Post appears in the user's profile immediately (in processing state)
  • When processing completes, the post becomes visible to followers
  • We can retry failed processing jobs without re-uploading
  • Processing cluster scales independently from the upload API
IR
INTERVIEWER
How does media reach users globally with low latency?
ME
YOU
We put a CDN (Content Delivery Network) — like CloudFront or Fastly — in front of S3. The CDN has edge servers in 200+ locations worldwide. When a user in Mumbai requests a photo, it serves from the nearest Mumbai edge node, not our US S3 bucket. First request: cache miss → fetch from S3, cache at edge. Subsequent requests: <20ms from edge cache.

CDN URL format: https://cdn.instagram.com/{user_id}/{post_id}/640.jpg — this gives CDN cache-key uniqueness per image per resolution.
📰 Phase 3 — News Feed: The Fanout Problem
PHASE 3 · FEED GENERATION
IR
INTERVIEWER
This is the core hard problem — how do you generate the news feed? Walk me through it in detail.
ME
YOU
There are two fundamental approaches to news feed generation:

Approach 1 — Fanout on Write (Push Model):
When User A posts, we immediately write the post_id to the feed cache of every follower. Feed reads are O(1) — just read from pre-computed cache.
Problem: If Selena Gomez (450M followers) posts, we do 450M Redis writes. That's a write storm that can take minutes and overload the system.

Approach 2 — Fanout on Read (Pull Model):
When User B opens their feed, we fetch all people they follow, query each person's post timeline, merge and sort. Feed reads are expensive — O(N) where N = number of followees.
Problem: If User B follows 2,000 people, each feed load requires 2,000 DB queries. At 500M users loading feeds — impossible.

My approach — Hybrid (what Instagram/Twitter actually do):
  • Normal users (< 10K followers): Use Push (fanout on write). Their posts fan out to followers' Redis sorted sets immediately. Fast reads.
  • Celebrities (≥ 10K followers): Use Pull. Don't pre-compute. At read time, fetch their latest posts and merge with the pre-computed feed. Celebrity posts are pulled lazily.
  • Feed storage: Redis Sorted Set per user — key: feed:{user_id}, members: post_ids, score: timestamp. O(log N) insert, O(1) range read.
IR
INTERVIEWER
What's the threshold for "celebrity"? How do you detect it?
ME
YOU
Good question. The threshold is configurable — I'd start at 10,000 followers. This is stored as a flag on the user record: is_high_follower_count = true. The User Graph Service updates this flag when a user crosses the threshold (and removes it if they drop below).

There's also a middle ground — for users with 10K–500K followers, we can still fanout but asynchronously with lower priority, so their followers get the post within a few seconds rather than instantly. Only truly massive accounts (Kylie, Cristiano) are fully pull-based.
IR
INTERVIEWER
How do you handle a user who follows both celebrities and normal users? The feed merge seems complex.
ME
YOU
Here's the feed assembly algorithm at read time:

  1. Fetch pre-computed feed from Redis: ZREVRANGE feed:{userId} 0 99 — gives 100 most recent post IDs from normal followees (already pushed)
  2. Get list of celebrity accounts this user follows (stored separately: celeb_followees:{userId})
  3. For each celebrity, query their post timeline (also a Redis sorted set: timeline:{celebId}) for last 20 posts — this is fast since we maintain a timeline per user
  4. Merge the two sets in-memory by timestamp (simple k-way merge sort)
  5. Return top 20 posts, cache merged result for 60 seconds
The merge happens in the Feed Service, in-memory, in <10ms. Since we're doing this at read time only for celebrities, and a user follows at most a handful of celebrities, it's not expensive.
💡 Key insight: Most users follow 200–500 people. Of those, maybe 5–10 are celebrities. So the "pull" set is tiny and fast to query. The "push" set covers 190–490 normal users and is pre-computed. Hybrid gets you the best of both worlds.
❤️ Phase 4 — Likes, Comments & Counters
PHASE 4 · INTERACTIONS
IR
INTERVIEWER
A viral post gets 10 million likes in 10 minutes. How do you count accurately without killing your database?
ME
YOU
This is a classic counter update hotspot problem. If we do UPDATE posts SET like_count = like_count + 1 WHERE post_id = X for every like — at 10M likes/10min = 16,666 updates/sec on a single row — that's a write lock disaster.

Solution — Redis + periodic flush:
  1. When user likes a post: SADD post_likes:{post_id} {user_id} (set, ensures one like per user) + INCR like_count:{post_id} in Redis
  2. Redis handles millions of INCR/sec trivially — it's atomic and single-threaded internally
  3. A background job (every 30 seconds) reads the Redis counter and flushes it to Cassandra
  4. When serving a post, read like_count from Redis (real-time) not from Cassandra
One like per user: We use a Redis Set per post to track who liked it: SADD post_likes:{post_id} {user_id} returns 0 if already liked. Also store in a separate Cassandra table for permanent record.
IR
INTERVIEWER
What if Redis goes down and we lose all the counters?
ME
YOU
Two mitigations:
1. Redis persistence (AOF + RDB): Redis with Append-Only File logging means on restart it replays all operations. We lose at most the last few seconds of likes (AOF fsync every second).
2. Redis Cluster with replication: Master + 2 replicas. If master fails, a replica is promoted in seconds (Sentinel or Cluster mode).

We also accept that like counts can be eventually consistent — showing 999,823 vs 1,000,001 likes is fine for users. It's not financial data.
IR
INTERVIEWER
How do comments work at scale? And what about nested comments?
ME
YOU
Comments are stored in Cassandra with schema: (post_id, comment_id, parent_comment_id, user_id, text, created_at). Partitioned by post_id, clustered by created_at DESC.

For nested comments (replies), parent_comment_id is null for top-level and set for replies. We load top-level comments first (20 at a time) and lazy-load replies on demand — Instagram's "View X replies" UX.

For a viral post with 500K comments — we don't load all 500K. We load the top 20 most recent in the feed, and paginate further on post detail view. Cassandra's partition-level pagination with LIMIT + created_at < cursor makes this efficient.
🌐 Phase 5 — Follow Graph & Scaling
PHASE 5 · GRAPH + EDGE CASES
IR
INTERVIEWER
How do you store the follow graph at this scale?
ME
YOU
The follow graph is a directed graph: A follows B doesn't mean B follows A. Access patterns are:
  • "Who does User A follow?" (for feed assembly)
  • "Who follows User A?" (for fanout on post)
  • "Does User A follow User B?" (for feed visibility check)
A Graph DB like Neo4j is elegant but doesn't scale to Instagram's level. Instead I'd use a sharded adjacency list in Cassandra:

Table 1 (followees): (follower_id, followee_id, created_at) — "who does user X follow?" — partition by follower_id
Table 2 (followers): (followee_id, follower_id, created_at) — "who follows user X?" — partition by followee_id

Both tables are written simultaneously on follow action. O(1) lookup for both directions. The duplication is intentional — this is the standard denormalization pattern for graph storage at scale.
IR
INTERVIEWER
What happens if the Feed Service goes down during fanout? Are posts lost?
ME
YOU
This is handled by Kafka's durability guarantees. The fanout job is triggered by a Kafka message. If the Feed Service crashes mid-fanout, Kafka retains the message (with offset). When the service recovers, it resumes from where it left off — idempotent processing ensures we don't double-write.

For idempotency: before writing post_id to a follower's feed, check if it already exists: ZSCORE feed:{followerId} {postId}. If score exists, skip. This makes the operation safe to retry.
📊 Capacity Planning — Step by Step
Always state assumptions aloud. "I'm assuming average photo size is 200KB after compression. Average user has 500 followers. 10% of users are active at any given moment."
1
Traffic — Daily Active Users & Requests
Translate DAU into actual request rates
Given: DAU = 500 Million Uploads per day = 100 Million Read:Write ratio = 100:1 Upload TPS (write): 100M uploads ÷ 86,400 sec/day = ~1,157 uploads/sec (avg) Peak (3× avg) = ~3,500 uploads/sec Feed load TPS (read): Assume each user opens app 10×/day = 5B feed loads/day 5B ÷ 86,400 = ~57,870 feed reads/sec (avg) Peak (3×) = ~175,000 feed reads/sec Like TPS: Assume 10 likes/user/day × 500M = 5B likes/day 5B ÷ 86,400 = ~57,870 likes/sec
2
Media Storage — Photos & Videos
How much storage do we need for media files?
Photo sizes (3 resolutions per photo): thumbnail (150px) = 10 KB medium (640px) = 100 KB full (1080px) = 300 KB Total per photo = 410 KB ≈ 500 KB (round up) Assume 80% photos, 20% videos (videos much larger): Photo uploads/day = 80M × 500 KB = 40 TB/day Video avg size (3 res) ≈ 50 MB Video uploads/day = 20M × 50 MB = 1,000 TB/day = 1 PB/day ⚠️ Videos dominate storage. This is why Instagram/TikTok use aggressive compression. With 10:1 video compression = 100 TB/day for videos Total media storage/day = ~140 TB/day Storage/year (raw) = ~51 PB/year Stored in S3 (11 nines durability, 3× replication internally) → Use S3 Intelligent-Tiering: hot content stays on SSD, cold moves to Glacier
3
Metadata Storage — Posts, Users, Follow Graph
Database sizing for non-media data
Post metadata record size: post_id (8B) + user_id (8B) + caption (500B avg) + media_urls (200B) + timestamps (16B) + misc (68B) ≈ 800 bytes per post Posts/day = 100M × 800B = 80 GB/day Posts/year = 80GB × 365 = ~29 TB/year User record size ≈ 200 bytes × 2B users: User table total = 400 GB (fits in memory!) Follow graph (edges): Avg follows per user = 400 Total edges = 2B users × 400 = 800B edges Per edge: 16 bytes (2 × user_id) = ~12.8 TB for follow graph Like records (for "did user like this?" check): 5B likes/day × 24 bytes/record = 120 GB/day — keep 30 days = ~3.6 TB
4
Feed Cache Sizing — Redis
How much memory for pre-computed feeds?
Feed cache per user (Redis Sorted Set): Keep last 100 post IDs per user Each entry: 8 bytes (post_id) + 8 bytes (score/timestamp) = 16 bytes Per user feed size = 100 × 16B = 1,600 bytes ≈ 2 KB Total feed cache for all DAU (500M): 500M × 2 KB = 1 TB for feed cache But not all 500M are active simultaneously — ~10% peak concurrency: Active feeds in cache = 50M × 2KB = 100 GB Redis for active users Redis cluster sizing: 100 GB data + 50% overhead (Redis data structures) = 150 GB With 3× replication = 450 GB total Redis cluster RAM ~15 nodes × 32 GB RAM each = 480 GB capacity → 15–20 Redis nodes for feed cache
5
Fanout Cost — Writing to Follower Feeds
How expensive is the push fanout?
Fanout writes per second: Uploads/sec = 1,157 Avg followers/user = 500 Fanout writes/sec = 1,157 × 500 = 578,500 Redis writes/sec This is peak fanout load — Redis can handle ~1M writes/sec per node cluster → Fanout is feasible for normal users. Celebrities (500K+ followers) are excluded from push fanout. Celebrity exception — Kylie Jenner (450M followers): If pushed: 450M Redis writes → takes ~450 seconds at 1M writes/sec That's 7.5 minutes of lag! → UNACCEPTABLE → Must use pull model for celebrities
6
CDN Bandwidth
How much bandwidth does media delivery consume?
Feed loads per second = 57,870 Each feed shows ~10 photos in viewport: Photos served/sec = 57,870 × 10 = 578,700 photo requests/sec Avg photo size (medium, 640px) = 100 KB CDN bandwidth = 578,700 × 100KB = ~57 GB/s outbound Peak (3×) = ~170 GB/s CDN cache hit rate (popular photos cached at edge): ~95% Origin bandwidth = 170 GB/s × 5% = ~8.5 GB/s to S3 origin → CDN is absolutely critical. Without CDN, ~170 GB/s would hit S3 directly — cost-prohibitive.

🏗 High-Level Architecture
🔄 Key Flows — Write & Read
📤 PHOTO UPLOAD FLOW
1
Client POST /posts with caption + media_type to API Gateway
2
Post Service creates record in Cassandra (status: AWAITING_UPLOAD), generates pre-signed S3 URL
3
Client uploads raw file directly to S3 via pre-signed URL (bypasses app servers)
4
S3 upload triggers S3 Event → Kafka topic media-uploads
5
Media Processor resizes (3 sizes), transcodes video (3 bitrates), uploads to CDN-optimised S3 paths
6
Post status updated to ACTIVE. CDN-ready URLs stored in post record.
7
Feed Service triggered → fanout to followers' Redis feeds (push for <10K followers)
8
Notification Service sends push notifications to close followers
📰 FEED LOAD FLOW
1
Client GET /feed?cursor=... to API Gateway
2
Feed Service reads pre-computed feed from Redis: ZREVRANGE feed:{userId} 0 99
3
Get celebrity followees list → fetch their recent posts from timeline Redis sorted sets
4
In-memory k-way merge of pushed + pulled posts → sorted by timestamp
5
Hydrate post metadata from Cassandra (caption, user info, comment count)
6
Check Redis like sets → SISMEMBER post_likes:{id} {userId} → is_liked_by_me flag
7
Return enriched feed. CDN URLs embedded — media loads directly from CDN, not app servers.

🗄 Database Schema (Cassandra)
-- ── Posts Table ──────────────────────────────────────
CREATE TABLE posts (
    post_id        BIGINT    PRIMARY KEY,  -- Snowflake ID (time-sortable)
    user_id        BIGINT,
    caption        TEXT,
    media_urls     MAP<TEXT,TEXT>,       -- {thumbnail, medium, full} → S3/CDN URL
    media_type     TEXT,               -- photo | video
    hashtags       LIST<TEXT>,
    status         TEXT,               -- AWAITING_UPLOAD | PROCESSING | ACTIVE | DELETED
    location_name  TEXT,
    created_at     TIMESTAMP
) WITH compaction = {'class': 'LeveledCompactionStrategy'};

-- ── User Timeline (user's own posts, for profile page) ──
CREATE TABLE user_timeline (
    user_id    BIGINT,
    post_id    BIGINT,
    created_at TIMESTAMP,
    PRIMARY KEY (user_id, created_at, post_id)
) WITH CLUSTERING ORDER BY (created_at DESC, post_id DESC);

-- ── Follow Graph (two tables for bidirectional O(1) lookup) ──
CREATE TABLE followees (  -- "who does user X follow?"
    follower_id  BIGINT,
    followee_id  BIGINT,
    created_at   TIMESTAMP,
    PRIMARY KEY  (follower_id, followee_id)
);
CREATE TABLE followers (  -- "who follows user X?"
    followee_id  BIGINT,
    follower_id  BIGINT,
    created_at   TIMESTAMP,
    PRIMARY KEY  (followee_id, follower_id)
);

-- ── Comments (partitioned by post, time-clustered) ──────
CREATE TABLE comments (
    post_id           BIGINT,
    comment_id        BIGINT,             -- Snowflake ID
    parent_comment_id BIGINT,             -- null for top-level
    user_id           BIGINT,
    text              TEXT,
    created_at        TIMESTAMP,
    PRIMARY KEY       (post_id, created_at, comment_id)
) WITH CLUSTERING ORDER BY (created_at DESC, comment_id DESC);

-- ── Likes (permanent record, for "who liked this post") ──
CREATE TABLE likes (
    post_id    BIGINT,
    user_id    BIGINT,
    created_at TIMESTAMP,
    PRIMARY KEY (post_id, user_id)
) WITH default_time_to_live = 2592000;  -- 30 days (keep recent likes)

@Service
@Slf4j
public class FeedService {

    private static final int  CELEBRITY_THRESHOLD = 10_000;
    private static final int  MAX_FEED_SIZE       = 500;
    private static final long FEED_TTL_SECONDS    = 86400; // 24hr

    @Autowired private RedisTemplate<String, Long> redis;
    @Autowired private PostRepository          postRepo;
    @Autowired private UserGraphRepository     graphRepo;
    @Autowired private ExecutorService          fanoutPool;

    /**
     * FANOUT ON WRITE — called when a user creates a post
     * Push to followers' feeds if author has < CELEBRITY_THRESHOLD followers
     */
    @Async
    public void fanoutPost(long postId, long authorId, long timestamp) {
        long followerCount = graphRepo.getFollowerCount(authorId);

        if (followerCount > CELEBRITY_THRESHOLD) {
            // Celebrity: only store in their own timeline, pull at read time
            redis.opsForSet().add("celebrity_authors", authorId);
            addToUserTimeline(authorId, postId, timestamp);
            return;
        }

        // Normal user: push to all followers' feed caches
        List<Long> followers = graphRepo.getFollowers(authorId);
        List<List<Long>> batches = partition(followers, 1000);

        batches.forEach(batch -> fanoutPool.submit(() -> {
            RedisCallback<Object> pipelined = conn -> {
                batch.forEach(followerId -> {
                    String feedKey = "feed:" + followerId;
                    // ZADD with timestamp as score → time-sorted
                    conn.zAdd(feedKey.getBytes(), timestamp, 
                              String.valueOf(postId).getBytes());
                    // Trim feed to MAX_FEED_SIZE to bound memory
                    conn.zRemRange(feedKey.getBytes(), 0, 
                                   -(MAX_FEED_SIZE + 1));
                });
                return null;
            };
            redis.execute(pipelined); // batch via Redis pipeline
        }));

        addToUserTimeline(authorId, postId, timestamp);
    }

    private void addToUserTimeline(long userId, long postId, long ts) {
        redis.opsForZSet().add("timeline:" + userId, postId, ts);
        redis.opsForZSet().removeRange("timeline:" + userId, 0, -1001);
    }

    /**
     * FEED ASSEMBLY — hybrid push + pull merge
     */
    public FeedPage getFeed(long userId, String cursor, int limit) {
        long cursorTimestamp = cursor != null 
            ? decodeCursor(cursor) : Long.MAX_VALUE;

        // Step 1: Read pre-computed pushed feed from Redis
        Set<TypedTuple<Long>> pushed = redis.opsForZSet()
            .reverseRangeByScoreWithScores(
                "feed:" + userId, 0, cursorTimestamp, 0, limit * 2);

        // Step 2: Pull celebrity posts
        Set<Long> celebFollowees = redis.opsForSet()
            .intersect("celeb_followees:" + userId, "celebrity_authors");

        List<ScoredPost> celebPosts = new ArrayList<>();
        if (celebFollowees != null && !celebFollowees.isEmpty()) {
            celebFollowees.forEach(celebId -> {
                redis.opsForZSet()
                    .reverseRangeByScoreWithScores(
                        "timeline:" + celebId, 0, cursorTimestamp, 0, 10)
                    .forEach(t -> celebPosts.add(
                        new ScoredPost(t.getValue(), t.getScore())));
            });
        }

        // Step 3: K-way merge by score (timestamp) descending
        List<Long> mergedIds = kWayMerge(pushed, celebPosts, limit);

        // Step 4: Hydrate post metadata from Cassandra (batch fetch)
        List<Post> posts = postRepo.findAllById(mergedIds);

        // Step 5: Enrich with like status (pipeline Redis checks)
        enrichWithLikeStatus(posts, userId);

        return new FeedPage(posts, 
            posts.isEmpty() ? null : encodeCursor(posts));
    }

    private void enrichWithLikeStatus(List<Post> posts, long userId) {
        // Redis pipeline: check all "did user like this?" in one round trip
        List<Object> results = redis.executePipelined(conn -> {
            posts.forEach(p -> conn.sIsMember(
                ("post_likes:" + p.getId()).getBytes(),
                String.valueOf(userId).getBytes()));
            return null;
        });
        for (int i = 0; i < posts.size(); i++) {
            posts.get(i).setLikedByMe((Boolean) results.get(i));
        }
    }
}

@Service
public class InteractionService {

    @Autowired private RedisTemplate<String, String> redis;
    @Autowired private LikeRepository   likeRepo;
    @Autowired private KafkaTemplate<String, LikeEvent> kafka;

    /** Idempotent like — safe to call multiple times */
    public LikeResult likePost(long postId, long userId) {
        String likeSetKey    = "post_likes:"  + postId;
        String likeCountKey  = "like_count:"  + postId;

        // SADD returns 1 if new member, 0 if already existed
        Boolean isNewLike = redis.opsForSet()
            .add(likeSetKey, String.valueOf(userId)) == 1;

        long newCount;
        if (isNewLike) {
            newCount = redis.opsForValue().increment(likeCountKey);
            // Async: persist to Cassandra for durability
            likeRepo.saveAsync(new Like(postId, userId, Instant.now()));
            // Notify post author
            kafka.send("like-events", new LikeEvent(postId, userId));
        } else {
            newCount = Long.parseLong(
                redis.opsForValue().get(likeCountKey));
        }
        return new LikeResult(newCount, true);
    }

    public LikeResult unlikePost(long postId, long userId) {
        Boolean removed = redis.opsForSet()
            .remove("post_likes:" + postId, 
                    String.valueOf(userId)) == 1;
        long newCount = removed
            ? redis.opsForValue().decrement("like_count:" + postId)
            : Long.parseLong(redis.opsForValue().get("like_count:" + postId));
        if (removed) likeRepo.deleteAsync(postId, userId);
        return new LikeResult(newCount, false);
    }
}

            PutObjectPresignRequest.builder()
                .signatureDuration(Duration.ofMinutes(15))
                .putObjectRequest(b -> b.bucket("ig-media-raw").key(rawKey))
                .build());
        return new PresignedUploadResult(presigned.url().toString(), rawKey);
    }

    /** Called by Kafka consumer after S3 upload triggers event */
    @KafkaListener(topics = "media-uploads")
    public void processMedia(MediaUploadEvent event) {
        byte[] rawBytes = downloadFromS3(event.getS3Key());
        Map<String, String> cdnUrls = new HashMap<>();

        // Resize and upload each resolution in parallel
        PHOTO_SIZES.entrySet().parallelStream().forEach(entry -> {
            byte[] resized = resizeImage(rawBytes, entry.getValue());
            String cdnKey = uploadProcessed(event.getPostId(), 
                                              entry.getKey(), resized);
            cdnUrls.put(entry.getKey(), 
                "https://cdn.instagram.com/" + cdnKey);
        });

        // Update post record with CDN URLs and mark ACTIVE
        postRepo.updateMediaUrls(event.getPostId(), cdnUrls, "ACTIVE");
        log.info("Media processing complete for post {}", event.getPostId());
    }

import time, heapq, threading, logging
from typing import List, Optional
from concurrent.futures import ThreadPoolExecutor
from dataclasses import dataclass

logger = logging.getLogger(__name__)

CELEBRITY_THRESHOLD = 10_000
MAX_FEED_SIZE       = 500

# ── Feed Service ─────────────────────────────────────
class FeedService:
    """
    Hybrid push-pull feed system.
    Push: normal users (<10K followers) → fanout on write to Redis ZSets
    Pull: celebrities (≥10K followers) → pull at read time, merge with pushed
    """

    def __init__(self, redis_client, post_repo, graph_repo):
        self.redis  = redis_client
        self.posts  = post_repo
        self.graph  = graph_repo
        self.pool   = ThreadPoolExecutor(max_workers=50)

    def fanout_post(self, post_id: int, author_id: int, timestamp: float):
        """Called async after a post is created and processed."""
        follower_count = self.graph.get_follower_count(author_id)

        if follower_count >= CELEBRITY_THRESHOLD:
            # Celebrity: mark + store only in their own timeline
            self.redis.sadd("celebrity_authors", author_id)
            self._add_to_timeline(author_id, post_id, timestamp)
            return

        # Normal user: fan out to all followers in batches
        followers = self.graph.get_followers(author_id)
        batches = [followers[i:i+500] for i in range(0, len(followers), 500)]

        def push_batch(batch):
            pipe = self.redis.pipeline()
            for follower_id in batch:
                feed_key = f"feed:{follower_id}"
                pipe.zadd(feed_key, {str(post_id): timestamp})
                pipe.zremrangebyrank(feed_key, 0, -(MAX_FEED_SIZE + 1))
            pipe.execute()

        # Submit batch fanout to thread pool (non-blocking)
        for batch in batches:
            self.pool.submit(push_batch, batch)

        self._add_to_timeline(author_id, post_id, timestamp)

    def _add_to_timeline(self, user_id: int, post_id: int, ts: float):
        pipe = self.redis.pipeline()
        pipe.zadd(f"timeline:{user_id}", {str(post_id): ts})
        pipe.zremrangebyrank(f"timeline:{user_id}", 0, -1001)
        pipe.execute()

    def get_feed(self, user_id: int, cursor: Optional[str] = None,
                 limit: int = 20) -> dict:
        """Hybrid feed assembly: pushed + pulled celeb posts merged by time."""
        max_score = float(self._decode_cursor(cursor)) if cursor else float('inf')

        # Step 1: pushed feed (normal followees)
        pushed_raw = self.redis.zrevrangebyscore(
            f"feed:{user_id}", max_score, 0,
            withscores=True, start=0, num=limit * 2)
        pushed = [(int(pid), score) for pid, score in pushed_raw]

        # Step 2: celebrity followees for this user
        celeb_followees = self.redis.sinter(
            f"celeb_followees:{user_id}", "celebrity_authors")

        celeb_posts = []
        if celeb_followees:
            for celeb_id in celeb_followees:
                raw = self.redis.zrevrangebyscore(
                    f"timeline:{celeb_id}", max_score, 0,
                    withscores=True, start=0, num=10)
                celeb_posts.extend([(int(pid), score) for pid, score in raw])

        # Step 3: k-way merge by score (timestamp) descending
        merged = sorted(pushed + celeb_posts, key=lambda x: -x[1])
        post_ids = [pid for pid, _ in merged[:limit]]

        # Step 4: batch fetch post metadata
        posts = self.posts.get_many(post_ids)

        # Step 5: enrich with like status via Redis pipeline
        pipe = self.redis.pipeline()
        for post in posts:
            pipe.sismember(f"post_likes:{post['post_id']}", str(user_id))
        like_results = pipe.execute()
        for i, post in enumerate(posts):
            post['is_liked_by_me'] = bool(like_results[i])

        next_cursor = self._encode_cursor(posts[-1]) if posts else None
        return {"posts": posts, "next_cursor": next_cursor}

    def _decode_cursor(self, cursor: str) -> float:
        import base64, json
        return json.loads(base64.b64decode(cursor))['ts']

    def _encode_cursor(self, last_post: dict) -> str:
        import base64, json
        payload = {'ts': last_post['created_at'],
                   'id': last_post['post_id']}
        return base64.b64encode(json.dumps(payload).encode()).decode()


# ── Interaction Service ───────────────────────────────
class InteractionService:
    """Like/unlike posts. Uses Redis for fast counters and duplicate prevention."""

    def __init__(self, redis_client, like_repo, kafka_producer):
        self.redis = redis_client
        self.likes = like_repo
        self.kafka = kafka_producer
        self.pool  = ThreadPoolExecutor(max_workers=10)

    def like_post(self, post_id: int, user_id: int) -> dict:
        """Idempotent — calling twice has same effect as once."""
        like_set_key   = f"post_likes:{post_id}"
        like_count_key = f"like_count:{post_id}"

        # SADD returns 1 if added, 0 if already in set
        is_new_like = self.redis.sadd(like_set_key, str(user_id)) == 1

        if is_new_like:
            new_count = self.redis.incr(like_count_key)
            # Async persist + notify (don't block the response)
            self.pool.submit(self._persist_like, post_id, user_id)
            self.pool.submit(self._notify_like, post_id, user_id)
        else:
            new_count = int(self.redis.get(like_count_key) or 0)

        return {"like_count": new_count, "is_liked": True}

    def unlike_post(self, post_id: int, user_id: int) -> dict:
        like_set_key   = f"post_likes:{post_id}"
        like_count_key = f"like_count:{post_id}"

        removed = self.redis.srem(like_set_key, str(user_id)) == 1
        new_count = self.redis.decr(like_count_key) if removed else \
                    int(self.redis.get(like_count_key) or 0)
        if removed:
            self.pool.submit(self.likes.delete_async, post_id, user_id)
        return {"like_count": max(0, new_count), "is_liked": False}

    def _persist_like(self, post_id: int, user_id: int):
        try:
            self.likes.save({"post_id": post_id, "user_id": user_id,
                             "created_at": time.time()})
        except Exception as e:
            logger.warning(f"Like persist failed: {e}")

    def _notify_like(self, post_id: int, liker_id: int):
        try:
            self.kafka.send("like-events",
                key=str(post_id),
                value={"post_id": post_id, "liker_id": liker_id,
                       "timestamp": time.time()})
        except Exception as e:
            logger.warning(f"Like notify failed: {e}")

class MediaService:
    PHOTO_SIZES = {"thumbnail": 150, "medium": 640, "full": 1080}

    def __init__(self, s3_client, post_repo, bucket="ig-media"):
        self.s3     = s3_client
        self.posts  = post_repo
        self.bucket = bucket
        self.pool   = ThreadPoolExecutor(max_workers=10)

    def generate_upload_url(self, post_id: int) -> dict:
        """Generate pre-signed S3 URL — client uploads directly (bypasses servers)"""
        raw_key = f"raw/{post_id}/original"
        url = self.s3.generate_presigned_url(
            "put_object",
            Params={"Bucket": "ig-media-raw", "Key": raw_key},
            ExpiresIn=900  # 15 minutes
        )
        return {"upload_url": url, "s3_key": raw_key}

    def process_media(self, post_id: int, raw_s3_key: str):
        """Called by Kafka consumer after S3 upload. Resize + store all sizes."""
        from PIL import Image
        import io
        raw_bytes = self.s3.get_object(
            Bucket="ig-media-raw", Key=raw_s3_key)["Body"].read()

        cdn_urls = {}

        def resize_and_upload(size_name, width):
            img = Image.open(io.BytesIO(raw_bytes))
            img.thumbnail((width, width))
            buf = io.BytesIO()
            img.save(buf, format="JPEG", quality=85)
            cdn_key = f"processed/{post_id}/{size_name}.jpg"
            self.s3.put_object(Bucket=self.bucket,
                               Key=cdn_key, Body=buf.getvalue(),
                               ContentType="image/jpeg")
            cdn_urls[size_name] = f"https://cdn.instagram.com/{cdn_key}"

        # Process all sizes in parallel via thread pool
        futures = [
            self.pool.submit(resize_and_upload, name, size)
            for name, size in self.PHOTO_SIZES.items()
        ]
        for f in futures: f.result()  # wait for all

        # Update post record: CDN URLs + status = ACTIVE
        self.posts.update_media_urls(post_id, cdn_urls, "ACTIVE")
        logger.info(f"Media processed for post {post_id}")
⚖️ Design Tradeoffs & Decisions
For every decision, say: "I chose X because Y. The cost is Z. In situation W, I'd pick the alternative instead."
📰 Feed Generation: Push vs Pull vs Hybrid
✅ Chosen: Hybrid

Push for normal users (<10K followers): O(1) feed reads. Pull for celebrities: avoids write storms. Best of both worlds for mixed follower graphs.

❌ Pure Push

Works until celebrities. Kylie Jenner (450M followers) posting triggers 450M Redis writes — takes 7+ minutes. Total write storm. Unacceptable latency for followers.

💡 Trade-off accepted

Hybrid adds code complexity: must track celebrity status, merge two data sources at read time. Accepted because the alternative (pure push/pull) breaks at scale. Read complexity is bounded — at most a handful of celebrity followees per user.

📁 Media Upload: Direct-to-S3 (Pre-signed URL) vs Through App Server
✅ Chosen: Pre-signed URL

Client uploads directly to S3. App servers never touch binary data. Eliminates bandwidth bottleneck: 100M × 200KB = 20TB/day would overwhelm app servers.

❌ Upload via App Server

App server proxies the upload to S3. Simpler client logic, but every byte passes through app servers. At scale, requires massive upload-specific server fleet — expensive and complex to scale separately.

💡 When to use proxy upload

For very small files (<1MB) or when you need server-side validation before S3 (e.g., virus scanning must happen before storage). For user-generated content at Instagram scale, pre-signed is always the right choice.

❤️ Like Counters: Redis Atomic vs DB Row Lock vs Cassandra Counter
✅ Chosen: Redis INCR

Atomic, ~1M ops/sec single-threaded. No row locks. Like count served from Redis in <1ms. Periodic flush to Cassandra for durability. Handles viral post spikes trivially.

⚠️ Cassandra COUNTER

Cassandra has built-in COUNTER type. Distributed, durable. But: counter reads are expensive (multi-replica quorum). Can't reset easily. Mixing counter + non-counter columns in same table is disallowed in Cassandra — architectural constraint.

💡 Durability concern

Redis INCR is fast but volatile without AOF persistence. Mitigation: Redis AOF (fsync every second) + async flush to Cassandra every 30s. Worst case: 30 seconds of like counts lost during Redis failure — acceptable for a social platform.

🗄 Follow Graph Storage: Graph DB vs Cassandra Adjacency List
✅ Chosen: Cassandra Adjacency List

Two denormalised tables: followees + followers. O(1) lookup both directions. Scales to billions of edges (12.8TB). Cassandra's partitioning handles massive graphs naturally.

❌ Neo4j (Graph DB)

Elegant for graph traversal (mutual friends, recommendations). But doesn't scale to Instagram's 800B+ edges. Single machine limits. Sharding graph DBs is extremely complex.

💡 Graph DB use case

Neo4j is great for deep graph traversal (6-degrees-of-separation, friend-of-friend recommendations). For Instagram's simpler use case (1-hop: who do I follow?), the complexity of a graph DB isn't justified. Cassandra wins on scale.

🎞 Media Processing: Sync vs Async
✅ Chosen: Async (Kafka-triggered)

Upload API responds in <200ms. Processing (resize/transcode) happens async. Post visible to uploader immediately (PROCESSING state). Followers see it when processing completes. Processing cluster scales independently.

❌ Sync Processing

Simpler — process on upload, return CDN URLs immediately. But resizing a 4K video takes 30–120 seconds. User waits on upload spinner. Upload API servers become CPU-bound. Can't scale upload and processing independently.

💡 Retry safety

Async processing is idempotent by design — if processing fails, Kafka retries. Offset-based consumption means no message lost. For sync, a timeout = failed upload = user frustration. Async is more resilient.

📄 Feed Pagination: Offset vs Cursor-based
✅ Chosen: Cursor-based

Cursor encodes last seen (timestamp + post_id). Stable: new posts don't shift results. Efficient: Redis ZRANGEBYSCORE skips to exact position. Works with Redis sorted sets naturally.

❌ Offset-based (page=2&size=20)

Simple but unstable: new posts being added to feed while user scrolls shifts all offsets. Page 2 may re-show posts from page 1. Also requires scanning from beginning to offset position — expensive at large offsets.

💡 Cursor design

Encode cursor as base64(JSON{ts, post_id}). The post_id breaks ties when multiple posts have same timestamp. Cursor is opaque to clients — they never parse it, just pass it back. This lets us change cursor format without API breaking changes.
