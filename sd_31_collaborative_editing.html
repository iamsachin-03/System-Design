Collaborative Document Editing (Google Docs)
Design a system where multiple users can simultaneously edit the same document and see each other's changes in real time — covering Operational Transformation vs CRDTs for conflict resolution, cursor synchronisation, presence awareness, offline editing with eventual consistency, and how Google Docs actually works under the hood.

🎯 Requirements
The core challenge: two users type simultaneously. User A inserts "Hello" at position 5. User B deletes character at position 3. Both operations are generated concurrently — each based on the same document state. If applied naively in any order, the result will be wrong for one user. This is the conflict resolution problem.

FUNCTIONAL
Real-time collaborative text editing (multiple users, same doc)
Conflict-free merge of concurrent edits
Cursor & selection presence (see where others are typing)
Offline editing — reconnect and sync without data loss
Undo/redo per user (doesn't undo other users' changes)
Persistent document storage with version history
NON-FUNCTIONAL

Operation propagation: <200ms to all collaborators
Strong eventual consistency — all users converge to same state

💬 Phase 1 — The Conflict Problem
PHASE 1 · CONFLICT RESOLUTION
IR
INTERVIEWER
Document: "Hello". User A inserts "!" at position 5 → "Hello!". User B simultaneously inserts " World" at position 5 → "Hello World". Server receives A's op first, then B's. What happens?
ME
YOU
Without conflict resolution: apply A's insert(pos=5, "!") → "Hello!". Then apply B's insert(pos=5, " World") → "Hello World!" (B's op inserts at 5, pushing ! to end). But on B's machine: B applied their op first → "Hello World", then receives A's insert(pos=5, "!") → "Hello!World". The two clients now have different documents. Diverged state.

Operational Transformation (OT) fixes this: When the server receives B's op (insert at pos=5), it knows A's op (also insert at pos=5) was applied first. OT transforms B's op against A's: since A inserted 1 character before position 5, B's position must shift by +1 → transformed to insert at pos=6. Result: "Hello! World". Both users converge to "Hello! World".

The transform function:
transform(insert(pos, text), insert(pos2, text2))
If pos2 <= pos: pos += len(text2) // shift right
else: pos unchanged

Every pair of operation types (insert/insert, insert/delete, delete/delete) needs a transform function. This is manageable for text, but complex for rich documents (tables, images, formulas).
💡 Google Docs uses OT — specifically the Jupiter/Wave algorithm. Figma uses CRDTs. Both achieve strong eventual consistency but with different tradeoffs.
🔗 Phase 2 — CRDT Alternative
PHASE 2 · CRDT
IR
INTERVIEWER
What's the alternative to OT, and when would you pick it?
ME
YOU
CRDT (Conflict-free Replicated Data Type) — instead of transforming operations, design the data structure itself to be inherently conflict-free.

For text: LSEQ or RGA (Replicated Growable Array). Instead of positions (which shift when others insert), each character gets a globally unique, stable identifier — typically a fractional position between its left and right neighbours. Character "H" gets ID 0.1, "e" gets 0.2, etc. When A inserts "!" between "o" (ID 0.5) and end (ID 1.0), it gets ID 0.75. When B inserts " " at the same spot, it gets ID 0.62. The document is just the characters sorted by ID — no transformation needed. A's "!" at 0.75 and B's " " at 0.62 can be merged by any node independently: sorted order gives " !" or "! " depending on ID comparison. Both users see the same result — convergence without coordination.

Why OT might still be chosen: OT is more mature, better studied for rich text. CRDTs have tombstone problem (deleted characters leave ghost IDs, document state grows forever). OT requires a central server to order operations; CRDTs can work fully peer-to-peer. CRDTs are better for offline-first/P2P (Notion, Linear, Figma). OT is better when you have a central server and want simpler client code.
🌐 Phase 3 — Offline Sync
PHASE 3 · OFFLINE
IR
INTERVIEWER
User edits a document offline for 30 minutes. They reconnect. How do you merge their changes without losing anything?
ME
YOU
The client stores a local operation log with a vector clock (or simply the server's last acknowledged revision number). While offline, the client applies operations locally and queues them with a monotonically increasing local sequence number.

On reconnect: client sends all queued operations to the server, tagged with the revision it was based on: [{op: insert(5,"x"), based_on_rev: 142}, ...]. The server knows the current document is at rev 160. It replays all ops from rev 143-160 through the transform function against each of the client's ops, producing transformed ops that apply cleanly on top of rev 160. Client receives the server's ops (rev 143-160) and applies them, resulting in an identical document state on both sides.

Key design choice: Server persists the full operation log, not just snapshots. This enables replay-based merge. Snapshots are taken periodically (every N ops) for efficiency — merge replays from the nearest snapshot, not from document creation.

// Collaborative Editing — Java: OT Engine (server-side)
// ══════════════════════════════════════════════════════

public class OTEngine {

    private final List<Operation> history = new ArrayList<>();
    private StringBuilder document = new StringBuilder();
    private int serverRevision = 0;

    /**
     * Apply an op from a client. The client may be based on an older revision.
     * Transform the op against all ops since client's revision, then apply.
     */
    public synchronized OperationAck applyClientOp(Operation clientOp, int clientRevision) {
        // Transform against all ops that happened after client's known revision
        Operation transformed = clientOp;
        for (int i = clientRevision; i < history.size(); i++) {
            transformed = transform(transformed, history.get(i));
        }
        // Apply transformed op to document
        applyToDocument(transformed);
        history.add(transformed);
        serverRevision++;
        return new OperationAck(transformed, serverRevision);
    }

    /**
     * OT Transform: adjust opA as if opB was already applied.
     * Both ops start from the same base state.
     */
    private Operation transform(Operation opA, Operation opB) {
        if (opA instanceof Insert insA && opB instanceof Insert insB) {
            return transformInsertInsert(insA, insB);
        } else if (opA instanceof Insert insA && opB instanceof Delete delB) {
            return transformInsertDelete(insA, delB);
        } else if (opA instanceof Delete delA && opB instanceof Insert insB) {
            return transformDeleteInsert(delA, insB);
        } else {
            return transformDeleteDelete((Delete) opA, (Delete) opB);
        }
    }

    /** Insert vs Insert: if opB inserted before opA, shift opA right */
    private Insert transformInsertInsert(Insert a, Insert b) {
        if (b.pos < a.pos || (b.pos == a.pos && b.userId.compareTo(a.userId) < 0)) {
            return new Insert(a.pos + b.text.length(), a.text, a.userId);
        }
        return a; // no shift needed
    }

    /** Insert vs Delete: if delete removed chars before insert position, shift left */
    private Insert transformInsertDelete(Insert a, Delete b) {
        if (b.pos + b.length <= a.pos) {
            return new Insert(a.pos - b.length, a.text, a.userId);
        } else if (b.pos >= a.pos) {
            return a; // delete after insert, no shift
        } else {
            return new Insert(b.pos, a.text, a.userId); // delete overlaps, clamp
        }
    }

    private void applyToDocument(Operation op) {
        if (op instanceof Insert ins) {
            document.insert(ins.pos, ins.text);
        } else if (op instanceof Delete del) {
            document.delete(del.pos, del.pos + del.length);
        }
    }
}

// ── WebSocket session handler ─────────────────────────
@Component
public class DocumentSessionHandler extends TextWebSocketHandler {
    // One OTEngine per active document (held in memory, persisted async)
    private final Map<String, OTEngine> engines = new ConcurrentHashMap<>();
    private final Map<String, Set<WebSocketSession>> docSessions = new ConcurrentHashMap<>();

    @Override
    public void handleTextMessage(WebSocketSession session, TextMessage msg) {
        ClientMessage cm = parse(msg.getPayload());
        OTEngine engine = engines.get(cm.docId());
        OperationAck ack = engine.applyClientOp(cm.op(), cm.clientRevision());

        // Broadcast to all sessions on this doc
        String broadcast = toJson(ack);
        docSessions.get(cm.docId()).forEach(s -> {
            try { s.sendMessage(new TextMessage(broadcast)); }
            catch (Exception e) { /* handle disconnect */ }
        });
    }
}

# Collaborative Editing — Python: OT Transform functions
# ══════════════════════════════════════════════════════

from dataclasses import dataclass
from typing import Union, List
import threading

@dataclass
class Insert:
    pos:     int
    text:    str
    user_id: str

@dataclass
class Delete:
    pos:    int
    length: int

Op = Union[Insert, Delete]


def transform(a: Op, b: Op) -> Op:
    """Transform op a assuming op b has already been applied."""
    if isinstance(a, Insert) and isinstance(b, Insert):
        return _ti_ti(a, b)
    elif isinstance(a, Insert) and isinstance(b, Delete):
        return _ti_td(a, b)
    elif isinstance(a, Delete) and isinstance(b, Insert):
        return _td_ti(a, b)
    else:
        return _td_td(a, b)


def _ti_ti(a: Insert, b: Insert) -> Insert:
    """Insert vs Insert. Tiebreak by user_id for determinism."""
    if b.pos < a.pos or (b.pos == a.pos and b.user_id < a.user_id):
        return Insert(a.pos + len(b.text), a.text, a.user_id)
    return a


def _ti_td(a: Insert, b: Delete) -> Insert:
    """Insert vs Delete."""
    if b.pos + b.length <= a.pos:
        return Insert(a.pos - b.length, a.text, a.user_id)
    elif b.pos >= a.pos:
        return a
    else:
        return Insert(b.pos, a.text, a.user_id)  # clamp to delete start


def _td_ti(a: Delete, b: Insert) -> Delete:
    """Delete vs Insert."""
    if b.pos <= a.pos:
        return Delete(a.pos + len(b.text), a.length)
    elif b.pos >= a.pos + a.length:
        return a
    else:
        return Delete(a.pos, a.length + len(b.text))  # insert inside delete range


def _td_td(a: Delete, b: Delete) -> Delete:
    """Delete vs Delete — overlap handling."""
    a_end = a.pos + a.length
    b_end = b.pos + b.length
    if b_end <= a.pos:
        return Delete(a.pos - b.length, a.length)
    elif b.pos >= a_end:
        return a
    else:  # overlapping deletes — shrink a's range
        new_pos    = min(a.pos, b.pos)
        new_length = max(0, a.length - (min(a_end, b_end) - max(a.pos, b.pos)))
        return Delete(new_pos, new_length)


class OTServer:
    def __init__(self, initial_doc: str = ""):
        self.doc      = list(initial_doc)
        self.history: List[Op] = []
        self.rev      = 0
        self._lock    = threading.Lock()

    def apply(self, op: Op, client_rev: int) -> tuple[Op, int]:
        """Transform op and apply. Returns (transformed_op, server_rev)."""
        with self._lock:
            transformed = op
            for historical_op in self.history[client_rev:]:
                transformed = transform(transformed, historical_op)
            self._apply_to_doc(transformed)
            self.history.append(transformed)
            self.rev += 1
            return transformed, self.rev

    def _apply_to_doc(self, op: Op):
        if isinstance(op, Insert):
            self.doc[op.pos:op.pos] = list(op.text)
        elif isinstance(op, Delete):
            del self.doc[op.pos:op.pos + op.length]

    def text(self) -> str:
        return "".join(self.doc)
⚖️ Tradeoffs
🔀 OT vs CRDT vs Last-Write-Wins
✅ OT (Operational Transform)

Mature, battle-tested (Google Docs, Etherpad). Correct for text. Requires central server to order operations — no true P2P. Transform functions are complex to implement correctly for rich content. Used where a central server is acceptable.

✅ CRDT (Yjs, Automerge)

Peer-to-peer capable — no central server needed. Offline-first. Correct by construction. Tombstone memory overhead for deleted chars. Harder to implement undo/redo. Used by Notion, Linear, Figma, Loom. Growing adoption.

Last-Write-Wins

Lock the document: one user edits at a time (Google Sheets cell locking). Simple but destroys collaboration experience. Or: just overwrite on conflict — leads to data loss. Never acceptable for real-time collaborative text.

🔄 Persistence: Op-log vs Snapshot-only vs Event Sourcing
Snapshot-only

Save the full document every N seconds. Simple. Can't replay history. Conflicts during concurrent saves require full-doc merge. Not suitable for collaborative editing — loses all granular operation history needed for offline sync.

✅ Op-log + periodic snapshots

Full operation history in Kafka (7d). Snapshots every 1000 ops for fast load. Offline sync replays ops since last snapshot. Version history is free — just replay to any revision. Google Docs uses this model.

Snapshot + delta compression

Store only diffs between snapshots. Smaller storage. Complex to implement. Needed only at Google-scale (billions of documents with rich history). Standard op-log is fine for most use cases up to 100M documents.
