API Gateway / Service Mesh
Design an API gateway that handles 1M requests/sec — token-bucket rate limiting per client, JWT authentication, request routing to 500 microservices, circuit breaking for failing upstreams, TLS termination, and the operational difference between a gateway (north-south traffic) and a service mesh (east-west traffic with sidecar proxies).
🔴 MUST DO 🟣 Hard Rate Limiting Circuit Breaker JWT Auth Service Discovery mTLS Envoy / Istio
🎯 Requirements
The API gateway is the single entry point for all client traffic. It's the system that must never go down, must handle any load spike, and must not become a bottleneck. Every millisecond of latency it adds multiplies across every request in the system. The design challenge is making all these cross-cutting concerns (auth, rate limiting, routing, circuit breaking) cheap, correct, and horizontally scalable.

FUNCTIONAL
TLS termination at gateway; upstream services use HTTP internally
Authentication: validate JWT; reject invalid tokens
Rate limiting: per client key (API key or user ID)
Request routing: map URL patterns to upstream services
Circuit breaking: stop sending to a failing upstream
Load balancing across upstream service instances
NON-FUNCTIONAL
1M requests/sec peak throughput
Added latency: <5ms p99 (gateway overhead only)
Rate limiting accuracy: >99.9% (minimal false positives/negatives)
Zero downtime deploys; horizontally scalable
500 upstream services; 10K service instances
💬 Phase 1 — Rate Limiting Algorithms
PHASE 1 · RATE LIMITING
IR
INTERVIEWER
You need to rate limit at 1000 req/sec per API key. Walk me through the algorithms and what you'd pick.
ME
YOU
Four main algorithms:

1. Fixed window counter: Count requests per minute bucket. Simple — INCR key:user:2024010112 in Redis. Problem: boundary burst — a client can fire 1000 req in the last second of minute 1 and 1000 req in the first second of minute 2, getting 2× the intended limit.

2. Sliding window log: Store timestamp of each request in a sorted set. Count entries in the last 60s. Accurate but high memory — storing every timestamp for every user.

3. Sliding window counter (hybrid): Track the current window count + the previous window count. Use a weighted average: rate = prev_count × (1 - overlap) + curr_count. Approximately accurate, low memory (2 counters per user). Good practical choice.

4. Token bucket (my choice): Each API key has a bucket with capacity C. Tokens refill at rate R/sec. Each request consumes 1 token. If bucket empty → reject 429. Allows burst up to C while enforcing average rate R.

Implementation: Redis Lua script for atomicity:
tokens = min(capacity, tokens + rate × elapsed); if tokens >= 1: tokens -= 1; return ALLOW; else: return REJECT

Why token bucket: allows legitimate bursts (e.g., app startup fetching 50 configs at once), enforces long-term average. Natural for API use cases. The Lua script executes atomically — no race conditions between gateway instances.
💡 For distributed rate limiting across 50 gateway nodes: Each node uses Redis as the shared counter store. The Lua script is the critical atomicity primitive. Local caching of token counts (sync every 100ms) reduces Redis calls by 99% while maintaining <10% error rate on limits.
⚡ Phase 2 — Circuit Breaker
PHASE 2 · CIRCUIT BREAKER
IR
INTERVIEWER
The payments service is returning 503s. How do you prevent these failures from cascading to the gateway and taking down the entire system?
ME
YOU
Without a circuit breaker: the gateway keeps sending requests to the failing payments service. Each request waits for the timeout (e.g., 30s). With 1M req/sec, thread pools exhaust and the gateway itself becomes unavailable — cascading failure.

Circuit breaker pattern (3 states):

CLOSED (normal): All requests flow through. Gateway tracks error rate in a sliding window. If error rate exceeds threshold (e.g., 50% failures in last 10s), trip the circuit → OPEN.

OPEN (failing): Gateway immediately returns 503 to clients without contacting the upstream. Fails fast in <1ms instead of waiting for upstream timeout. This protects the gateway's thread pool and prevents timeout storm. A timer is set (e.g., 30s).

HALF-OPEN (probing): After the timer expires, allow a small percentage of requests through (e.g., 5%). If they succeed, close the circuit — service has recovered. If they fail, reopen for another 30s.

Implementation: each upstream service has its own circuit breaker state machine, stored in the gateway process (not Redis — we want ultra-low latency for the OPEN check). Health state replication between gateway nodes via gossip or a shared registry is optional — a brief period where some nodes have a stale circuit state is acceptable.
🔀 Phase 3 — Gateway vs Service Mesh
PHASE 3 · GATEWAY vs MESH
IR
INTERVIEWER
When would you use a service mesh instead of (or in addition to) an API gateway?
ME
YOU
API Gateway handles north-south traffic: client → gateway → service. One per cluster or per API surface. Handles external-facing concerns: auth, rate limiting, TLS termination, API versioning, client-facing routing.

Service Mesh handles east-west traffic: service → service communication inside the cluster. Each service gets a sidecar proxy (Envoy). The sidecar intercepts all outbound/inbound traffic. Provides: mTLS between services, service-to-service rate limiting, retries with backoff, distributed tracing, circuit breaking — without any code changes to services.

When to use a mesh:
  • 100+ microservices where implementing retry/timeout/circuit-breaker in each service's code is impractical
  • Security requirement: zero-trust networking — every service-to-service call must be mutually authenticated (mTLS)
  • Observability: automatic distributed tracing across all services without instrumentation
  • Traffic splitting for canary deployments: route 5% of payments traffic to v2 without code changes
When not to use a mesh: <20 services, teams comfortable implementing resilience patterns in code, or cost-sensitive (sidecar proxies add 5-10ms latency and CPU overhead per hop).

// ── Token Bucket Rate Limiter (Redis Lua) ─────────────
@Service
public class TokenBucketRateLimiter {

    private static final String LUA_SCRIPT = """
        local key      = KEYS[1]
        local capacity = tonumber(ARGV[1])
        local rate     = tonumber(ARGV[2])  -- tokens per second
        local now      = tonumber(ARGV[3])  -- current time (ms)

        local data     = redis.call('HMGET', key, 'tokens', 'last_refill')
        local tokens   = tonumber(data[1]) or capacity
        local last     = tonumber(data[2]) or now

        -- Refill tokens based on elapsed time
        local elapsed  = math.max(0, now - last)
        tokens = math.min(capacity, tokens + rate * elapsed / 1000)

        if tokens >= 1 then
            tokens = tokens - 1
            redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
            redis.call('EXPIRE', key, 3600)
            return 1  -- ALLOW
        else
            redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
            return 0  -- REJECT
        end
        """;

    @Autowired private RedisTemplate<String,String> redis;
    private final DefaultRedisScript<Long> script;

    public boolean allow(String clientKey, double capacity, double ratePerSec) {
        Long result = redis.execute(script,
            List.of("ratelimit:" + clientKey),
            String.valueOf(capacity),
            String.valueOf(ratePerSec),
            String.valueOf(System.currentTimeMillis()));
        return Long.valueOf(1L).equals(result);
    }
}

// ── Circuit Breaker ───────────────────────────────────
public class CircuitBreaker {

    public enum State { CLOSED, OPEN, HALF_OPEN }

    private volatile State  state      = State.CLOSED;
    private AtomicInteger   failures   = new AtomicInteger();
    private AtomicInteger   successes  = new AtomicInteger();
    private volatile long   openedAt   = 0;

    private final int    failureThreshold = 50;  // % errors to trip
    private final int    windowSize       = 100; // requests in window
    private final long   openDuration     = 30_000; // ms before HALF_OPEN

    public boolean canRequest() {
        return switch (state) {
            case CLOSED    -> true;
            case OPEN      -> {
                if (System.currentTimeMillis() - openedAt > openDuration) {
                    state = State.HALF_OPEN; yield true; // probe
                }
                yield false; // fast reject
            }
            case HALF_OPEN -> Math.random() < 0.05; // 5% probe rate
        };
    }

    public void recordSuccess() {
        if (state == State.HALF_OPEN) { state = State.CLOSED; reset(); return; }
        successes.incrementAndGet();
    }

    public void recordFailure() {
        if (state == State.HALF_OPEN) { state = State.OPEN; openedAt = System.currentTimeMillis(); return; }
        int f = failures.incrementAndGet();
        int total = f + successes.get();
        if (total >= windowSize && (f * 100 / total) >= failureThreshold) {
            state = State.OPEN; openedAt = System.currentTimeMillis(); reset();
        }
    }

    private void reset() { failures.set(0); successes.set(0); }
}

// ── Gateway Request Handler ───────────────────────────
@RestController
public class GatewayHandler {

    @Autowired TokenBucketRateLimiter rateLimiter;
    @Autowired JwtValidator            jwtValidator;
    @Autowired ServiceRegistry         registry;
    Map<String, CircuitBreaker>        breakers = new ConcurrentHashMap<>();

    @RequestMapping("/**")
    public ResponseEntity<?> proxy(HttpServletRequest req) {
        String clientId = req.getHeader("X-API-Key");

        // 1. Rate limit
        if (!rateLimiter.allow(clientId, 100, 1000.0/60))
            return ResponseEntity.status(429).body("Rate limit exceeded");

        // 2. JWT auth
        String token = req.getHeader("Authorization");
        if (!jwtValidator.validate(token))
            return ResponseEntity.status(401).body("Unauthorized");

        // 3. Route
        String   upstream = registry.route(req.getRequestURI());
        CircuitBreaker cb = breakers.computeIfAbsent(upstream, __ -> new CircuitBreaker());

        // 4. Circuit breaker
        if (!cb.canRequest())
            return ResponseEntity.status(503).body("Service unavailable");

        // 5. Forward request
        try {
            ResponseEntity<?> resp = forward(req, upstream);
            if (resp.getStatusCode().is2xxSuccessful()) cb.recordSuccess();
            else if (resp.getStatusCode().is5xxServerError()) cb.recordFailure();
            return resp;
        } catch (Exception e) { cb.recordFailure(); throw e; }
    }
}

import time, math, random, threading, requests
from enum import Enum
from dataclasses import dataclass, field
import redis

r = redis.Redis()

TOKEN_BUCKET_LUA = """
local key      = KEYS[1]
local capacity = tonumber(ARGV[1])
local rate     = tonumber(ARGV[2])
local now      = tonumber(ARGV[3])
local data     = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens   = tonumber(data[1]) or capacity
local last     = tonumber(data[2]) or now
local elapsed  = math.max(0, now - last)
tokens = math.min(capacity, tokens + rate * elapsed / 1000)
if tokens >= 1 then
    tokens = tokens - 1
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    redis.call('EXPIRE', key, 3600)
    return 1
else
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    return 0
end
"""

_rl_script = r.register_script(TOKEN_BUCKET_LUA)

def allow_request(client_key: str, capacity: float = 100, rate: float = 16.67) -> bool:
    """Token bucket via Redis Lua. rate = tokens/sec (e.g. 1000/min = 16.67/sec)."""
    result = _rl_script(
        keys=[f"ratelimit:{client_key}"],
        args=[capacity, rate, int(time.time() * 1000)])
    return result == 1


class CBState(Enum):
    CLOSED    = "closed"
    OPEN      = "open"
    HALF_OPEN = "half_open"

class CircuitBreaker:
    def __init__(self, failure_threshold: float = 0.5, window: int = 100, open_secs: int = 30):
        self.state     = CBState.CLOSED
        self.failures  = 0
        self.successes = 0
        self.opened_at = 0
        self.threshold = failure_threshold
        self.window    = window
        self.open_secs = open_secs
        self._lock     = threading.Lock()

    def can_request(self) -> bool:
        with self._lock:
            if self.state == CBState.CLOSED: return True
            if self.state == CBState.OPEN:
                if time.time() - self.opened_at > self.open_secs:
                    self.state = CBState.HALF_OPEN; return True
                return False
            return random.random() < 0.05  # 5% probe in HALF_OPEN

    def record_success(self):
        with self._lock:
            if self.state == CBState.HALF_OPEN: self._close()
            else: self.successes += 1

    def record_failure(self):
        with self._lock:
            if self.state == CBState.HALF_OPEN: self._open(); return
            self.failures += 1
            total = self.failures + self.successes
            if total >= self.window and self.failures / total >= self.threshold:
                self._open()

    def _open(self):
        self.state = CBState.OPEN; self.opened_at = time.time()
        self.failures = self.successes = 0

    def _close(self):
        self.state = CBState.CLOSED; self.failures = self.successes = 0


# ── Gateway Request Pipeline ──────────────────────────
breakers: dict[str, CircuitBreaker] = {}
route_table = {"/payments": "http://payments-svc", "/orders": "http://orders-svc"}

def handle_request(method: str, path: str, headers: dict, body: bytes):
    client_key = headers.get("x-api-key", "anonymous")

    # 1. Rate limit
    if not allow_request(client_key):
        return 429, "Rate limit exceeded"

    # 2. JWT auth (simplified)
    token = headers.get("authorization", "")
    if not token.startswith("Bearer "):
        return 401, "Unauthorized"

    # 3. Route
    prefix  = "/" + path.split("/")[1]
    upstream = route_table.get(prefix)
    if not upstream: return 404, "No route"

    # 4. Circuit breaker
    cb = breakers.setdefault(upstream, CircuitBreaker())
    if not cb.can_request():
        return 503, "Service unavailable (circuit open)"

    # 5. Forward
    try:
        resp = requests.request(method, upstream + path, headers=headers,
                                data=body, timeout=5)
        if resp.status_code >= 500: cb.record_failure()
        else: cb.record_success()
        return resp.status_code, resp.content
    except Exception as e:
        cb.record_failure()
        return 502, str(e)
⚖️ Tradeoffs
🚪 Gateway Implementation: Nginx + Lua vs Envoy vs Custom
Nginx + Lua (OpenResty)

Extremely high performance (1M+ req/sec). Lua scripts for custom logic. Used by Kong (API gateway) as the underlying engine. Operationally complex — Lua debugging is painful. No native service mesh integration. Good for pure API gateway use cases where performance is paramount.

✅ Envoy Proxy

Designed specifically for service proxy use cases. Native xDS config API (used by Istio). Excellent observability out-of-box (Prometheus metrics, distributed tracing). Circuit breaker, retry, timeout, rate limiting built in. Written in C++ — fast. The standard choice for both API gateway and service mesh sidecars.

Custom (Go/Rust)

Full control over behaviour. Used by companies with very specific requirements (e.g., Cloudflare's Pingora in Rust). 6-18 months to match Envoy's feature set. Only justified at extremely large scale (>10M req/sec per node) or when standard proxies have fundamental limitations for your use case.

⏱ Rate Limiting: Local vs Global vs Hybrid
Purely Local

Each gateway node maintains its own counter. Fast — no network call. But: with 20 gateway nodes, a client gets 20× the limit (each node allows 1000 req/min independently). Only correct if you have sticky sessions routing each client to the same node. Unacceptable for API rate limits that must be globally accurate.

Purely Global (Redis)

All nodes check a shared Redis counter. Perfectly accurate. But: every request = 1 Redis RTT (~0.5ms) = 50% of your 1ms latency budget. At 1M req/sec = 1M Redis ops/sec. Redis can handle it, but it's an extra dependency in the hot path. A Redis outage = rate limiting fails open or closed.

✅ Hybrid (local cache + async sync)

Each node maintains local token bucket. Sync consumed tokens to Redis every 100ms. Local node allows requests using local count; Redis reconciles the global view. Error: ±10% on limits (acceptable). Eliminates Redis from the hot path. Used by Cloudflare, Kong in production. Best throughput/accuracy tradeoff.
