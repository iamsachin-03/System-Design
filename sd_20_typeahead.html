💬 Phase 1 — Data Structure Choice
PHASE 1 · TRIE VS ALTERNATIVES
IR
INTERVIEWER
What data structure would you use? Why not a SQL LIKE query or a hash map?
ME
YOU
Three options — each with fatal flaws at scale:

SQL LIKE 'pre%': Even with a B-tree index, this requires a range scan over all strings starting with the prefix. With 300M unique queries and 10M QPS, this becomes seconds per lookup. Non-starter.

Hash map (prefix → top-5 list): O(1) lookup. But pre-computing every prefix for every query string causes memory explosion — a 20-char query generates 20 prefix entries. At 300M queries × avg 20 chars = 6 billion entries. Too large.

Trie (prefix tree): Characters as nodes. "amazon" and "apple" share the "a" root. Each node stores top-5 suggestions for that prefix — precomputed at build time. Lookup: O(L) where L = prefix length ≈ constant. Memory-efficient because shared prefixes share nodes. Winner.

Key optimisation — cache top-5 at each node: Naïve trie lookup would require traversing the entire subtree to find top-5. Instead, during build we run a DFS and store the top-5 at every node. Lookups become O(L) — just follow the prefix, read the stored list.
💡 Always explain why the naïve trie is too slow (subtree traversal per lookup), then immediately pivot to the optimised version with cached top-K at each node. This distinction is what separates strong candidates.
🔄 Phase 2 — Data Pipeline & Freshness
PHASE 2 · PIPELINE
IR
INTERVIEWER
"World Cup 2026" starts trending. How does it appear in suggestions within minutes?
ME
YOU
Two pipelines with different freshness guarantees:

1. Hourly batch rebuild: Every search is logged to Kafka. A Spark/Flink job aggregates query counts over a sliding 7-day window with exponential decay on older data. Every hour: generate top queries per prefix → build new trie → serialise to snapshot → hot-swap. Old trie stays live during build; atomic pointer swap when done. Zero downtime.

2. 5-minute trending overlay: A separate Flink job counts queries in a 5-minute sliding window and detects velocity spikes (query count in last 5min > 3× average). Spikes written to Redis ZSet: ZINCRBY trending:{prefix} {velocity} {query}. At serve time the API merges trie results with trending scores — trending queries can bump their way into the top-5 even if their absolute frequency is lower. This gives us breaking-news sensitivity within 5 minutes.
⚡ Phase 3 — Scaling to 10M QPS
PHASE 3 · SCALE
IR
INTERVIEWER
10M QPS is enormous. How do you handle it without the trie service becoming a bottleneck?
ME
YOU
Multi-layer caching eliminates nearly all load from the trie service:

Layer 1 — Client cache: Browser/app caches results for the current session. If user types "am", gets results, then backspaces to "a" — served from local cache. Handles ~30% of keystrokes.

Layer 2 — CDN edge (5-sec TTL for 1-3 char prefixes): GET /autocomplete?q=am cached at CDN edge globally. 5s stale is acceptable. CDN absorbs ~80% of QPS. 1-3 char prefixes have massive hit rates since millions of users type the same short prefixes simultaneously.

Layer 3 — App server in-process LRU (100K prefixes): ~95% hit rate for top 100K prefixes. Cache miss costs one trie lookup (<1ms).

Layer 4 — Trie service: Only ~5% of 10M QPS = 500K QPS reaches here. The 1.85 GB trie is loaded entirely in RAM. 20 read replicas each handle 25K QPS — trivial for an in-memory lookup.
🌲 Trie Structure & Top-K Caching

BUILD: DFS PROPAGATION
1
Insert all (query, freq) pairs into trie
2
DFS from root; at each node collect all descendant (query, freq) pairs
3
Keep only top-5 by frequency at each node — discard the rest
4
Build time: O(N × avg_len). Done offline hourly.
5
Atomic swap: new trie built while old serves traffic. Pointer swap = instant cutover.
LOOKUP: O(PREFIX_LENGTH)
1
Navigate from root, one char at a time
2
At each node: children.get(c) → O(1)
3
After L steps: reach the node for the full prefix
4
Return node.top_k — already precomputed
5
Merge with Redis trending overlay → final top-5
🗜 Patricia (Compressed) Trie
Production systems compress single-child chains. Instead of nodes a→m→a→z→o→n, store one node "amazon". Reduces memory 3-5×.
STANDARD: 7 nodes for "amazon"
root → a → m → a → z → o → n ✓
Memory: 7 × 350B = 2.45 KB per query path
PATRICIA: 1 node for "amazon"
root → "amazon" ✓
Memory: 1 × 380B = 380 B per query path

// Typeahead — Java: Trie with cached top-K at each node
// ══════════════════════════════════════════════════════

public class TrieNode {
    public Map<Character, TrieNode> children = new HashMap<>();
    public boolean                    isEnd     = false;
    public long                        frequency = 0;
    public List<Suggestion>           topK      = new ArrayList<>();
}

public class AutocompleteTrie {

    private final TrieNode root = new TrieNode();
    private final int       K    = 5;

    /** O(L): navigate to prefix node, return cached top-K */
    public List<Suggestion> suggest(String prefix) {
        TrieNode node = root;
        for (char c : prefix.toLowerCase().toCharArray()) {
            node = node.children.get(c);
            if (node == null) return List.of();
        }
        return node.topK; // pre-computed at build time, O(1)
    }

    /** Insert query and its frequency */
    public void insert(String query, long frequency) {
        TrieNode node = root;
        for (char c : query.toLowerCase().toCharArray()) {
            node = node.children.computeIfAbsent(c, k -> new TrieNode());
        }
        node.isEnd     = true;
        node.frequency = frequency;
    }

    /** DFS: propagate top-K up to every ancestor node */
    public List<Suggestion> buildTopK(TrieNode node, String path) {
        List<Suggestion> results = new ArrayList<>();
        if (node.isEnd) results.add(new Suggestion(path, node.frequency));
        for (var entry : node.children.entrySet()) {
            results.addAll(buildTopK(entry.getValue(), path + entry.getKey()));
        }
        results.sort(Comparator.comparingLong(Suggestion::score).reversed());
        node.topK = results.stream().limit(K).collect(Collectors.toList());
        return node.topK;
    }

    /** Hot-swap: build new trie offline, swap pointer atomically */
    private volatile AutocompleteTrie activeTrie;

    public void hotSwap(AutocompleteTrie newTrie) {
        this.activeTrie = newTrie; // volatile write — instantly visible to all threads
    }
}

// ── Autocomplete Service — trending merge ──────────────
@Service
public class AutocompleteService {

    @Autowired private AutocompleteTrie trie;
    @Autowired private RedisTemplate<String,String> redis;

    public List<Suggestion> autocomplete(String prefix, int limit) {
        prefix = prefix.toLowerCase().strip();

        // 1. Base suggestions from trie (O(L), pre-computed)
        List<Suggestion> base = trie.suggest(prefix);

        // 2. Trending overlay from Redis ZSet (5-min velocity)
        Set<ZSetOperations.TypedTuple<String>> trending =
            redis.opsForZSet().reverseRangeWithScores(
                "trending:" + prefix, 0, limit - 1);

        // 3. Merge: boost trending items, deduplicate
        Map<String, Suggestion> merged = new LinkedHashMap<>();
        for (Suggestion s : base) merged.put(s.query(), s);
        for (var t : trending) {
            merged.merge(t.getValue(),
                new Suggestion(t.getValue(), t.getScore().longValue(), true),
                (existing, trending_) -> new Suggestion(
                    existing.query(), existing.score() + trending_.score(), true));
        }

        return merged.values().stream()
            .sorted(Comparator.comparingLong(Suggestion::score).reversed())
            .limit(limit)
            .collect(Collectors.toList());
    }
}

# Typeahead — Python: Trie + trending overlay
# ══════════════════════════════════════════════════════

from dataclasses import dataclass, field
from typing import Dict, List, Optional


@dataclass
class TrieNode:
    children: Dict[str, 'TrieNode'] = field(default_factory=dict)
    is_end:   bool                     = False
    frequency: int                     = 0
    top_k:    List[tuple]              = field(default_factory=list)  # [(query, score)]


class AutocompleteTrie:
    K = 5

    def __init__(self):
        self.root = TrieNode()

    def insert(self, query: str, frequency: int):
        node = self.root
        for c in query.lower():
            if c not in node.children:
                node.children[c] = TrieNode()
            node = node.children[c]
        node.is_end    = True
        node.frequency = frequency

    def suggest(self, prefix: str) -> List[tuple]:
        """O(L) lookup — returns pre-cached top_k list."""
        node = self.root
        for c in prefix.lower():
            if c not in node.children:
                return []
            node = node.children[c]
        return node.top_k

    def build_top_k(self, node: TrieNode = None, path: str = "") -> List[tuple]:
        """DFS: build and cache top-K at every node. Call once after all inserts."""
        if node is None: node = self.root
        results = []
        if node.is_end:
            results.append((path, node.frequency))
        for char, child in node.children.items():
            results.extend(self.build_top_k(child, path + char))
        results.sort(key=lambda x: -x[1])
        node.top_k = results[:self.K]
        return results


class AutocompleteService:
    def __init__(self, trie: AutocompleteTrie, redis):
        self.trie  = trie
        self.redis = redis

    def autocomplete(self, prefix: str, limit: int = 5) -> list:
        prefix = prefix.lower().strip()

        # 1. Trie base results
        base     = {q: s for q, s in self.trie.suggest(prefix)}

        # 2. Trending overlay from Redis (5-min velocity scores)
        trending = self.redis.zrevrangebyscore(
            f"trending:{prefix}", "+inf", "-inf",
            start=0, num=limit, withscores=True)

        # 3. Merge: trending items boost or inject new results
        merged = dict(base)
        for query, velocity in trending:
            merged[query] = merged.get(query, 0) + velocity

        return sorted(merged.items(), key=lambda x: -x[1])[:limit]
⚖️ Tradeoffs
📊 Data Structure: Trie vs Redis ZSet vs Elasticsearch
✅ Trie (chosen)

O(L) lookup, ~5M prefixes in 1.85 GB RAM. Best for exact prefix match at Google scale. Requires offline build + hot-swap. Optimal for this problem.

⚠️ Redis ZSet (ZRANGEBYLEX)

Simple, no custom data structure. Works great for <1M queries, <100K QPS. At Google scale: one massive ZSet becomes a single-node bottleneck. Fine for small-scale autocomplete (product search, internal tools).

❌ Elasticsearch

Excellent for fuzzy search and complex ranking. But: too much overhead for simple prefix completion. Overkill operationally. Use if you also need full-text search over the same dataset.

🔄 Freshness: Hourly batch vs Real-time vs Hybrid
❌ Real-time trie updates

Every search updates trie nodes. Requires distributed write locks. With 11.6K searches/sec, lock contention kills throughput. Trie becomes unsafe to read during updates. Not practical.

⚠️ Hourly batch only

Deterministic, safe. But breaking news ("earthquake in X") takes up to 1 hour to surface. Terrible UX for trending queries.

✅ Hourly batch + 5-min trending overlay

Best of both. Stable base from batch; real-time responsiveness from Redis velocity overlay. Two independent systems — trending can fail without breaking core autocomplete.
