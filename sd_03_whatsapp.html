🎙 Phase 1 — Connection Architecture
PHASE 1 · CONNECTIONS
IR
INTERVIEWER
Design WhatsApp. Start with the most fundamental question — how do you deliver messages in real-time to billions of users?
ME
YOU
The foundation of any chat system is the persistent connection. Unlike HTTP (request-response), messaging requires the server to push messages to clients at any time. Three options:

Option 1 — Long Polling: Client repeatedly asks "any new messages?" every few seconds. Simple but wasteful — most polls return empty. High server load from constant polling.

Option 2 — Server-Sent Events (SSE): One-way server→client stream. Good for notifications, bad for chat — we also need client→server in the same channel.

Option 3 — WebSocket: Full-duplex bidirectional persistent TCP connection. One connection per client. Client and server can send at any time. This is what WhatsApp, Slack, and Discord use.

I'll use WebSocket. Each client opens a WebSocket to a Chat Server and keeps it alive with periodic heartbeats (ping/pong every 30 seconds). The server tracks which user is connected to which Chat Server.
IR
INTERVIEWER
At 2B users with ~30% online at peak, that's 600M concurrent WebSocket connections. A single server can handle maybe 100K connections. How do you manage this?
ME
YOU
600M ÷ 100K connections/server = 6,000 Chat Server instances minimum. That's a large fleet, but manageable with Kubernetes auto-scaling.

The real challenge is: when Alice wants to send a message to Bob, how does Alice's server know which of the 6,000 servers Bob is connected to?

I'd use a Connection Registry — a fast lookup service (Redis) that maps user_id → chat_server_id. When Bob connects, his Chat Server registers: HSET connections bob server_42. When Alice sends to Bob, her server does: HGET connections bob → gets server_42 → routes the message there → Server 42 pushes to Bob's WebSocket.

If Bob is offline, HGET connections bob returns nil → message goes to the offline message queue instead.
IR
INTERVIEWER
How does Alice's Chat Server communicate with Bob's Chat Server? They're different machines.
ME
YOU
Server-to-server communication uses an internal message broker — Kafka. Here's the flow:
  1. Alice sends message over WebSocket to Chat Server A (her server)
  2. Server A writes message to Kafka topic chat-messages (partitioned by conversation_id for ordering)
  3. Server A also persists message to Cassandra with status SENT
  4. A Message Delivery Service consumes from Kafka
  5. It looks up Bob's server from the Connection Registry
  6. It calls Bob's Chat Server via gRPC: DeliverMessage(bobId, message)
  7. Bob's server pushes to Bob's WebSocket
  8. Bob's client ACKs → Delivery Service updates status to DELIVERED
Why Kafka? It decouples the send and deliver steps, provides durability (messages survive server crashes), and enables retries.
💡 Key insight: The Connection Registry is the nerve center of the system. It must be fast (Redis, <1ms lookup) and consistent (if stale, message goes to wrong server). When a Chat Server crashes, it must quickly clean up stale registrations.
📴 Phase 2 — Offline Message Delivery
PHASE 2 · OFFLINE HANDLING
IR
INTERVIEWER
What happens when Bob is offline for 3 days and then comes back? How do you deliver all the messages he missed?
ME
YOU
This is the store-and-forward problem. When the delivery service detects Bob is offline (not in Connection Registry):
  1. Message is persisted to Cassandra in the message store (regardless of online/offline)
  2. Message is also added to Bob's offline message queue in Cassandra: table offline_messages(user_id, message_id, conversation_id, created_at)
  3. Bob's unread badge count is incremented in Redis: HINCRBY unread_counts bob conversation_123 1

When Bob reconnects:
  1. Bob's client opens WebSocket → Chat Server registers Bob in Connection Registry
  2. Server immediately queries Bob's offline queue: SELECT * FROM offline_messages WHERE user_id = bob ORDER BY created_at LIMIT 500
  3. All queued messages are pushed to Bob's WebSocket in order
  4. Bob's client ACKs each message → status updated to DELIVERED
  5. Delivered messages are removed from the offline queue

What if Bob has 10,000 offline messages (he was gone for a month)? We push the most recent 500 immediately, and paginate the rest as he scrolls up. We don't overwhelm the connection.
IR
INTERVIEWER
What if the server crashes right after Alice sends but before Bob's offline queue is written? Is the message lost?
ME
YOU
This is the durability guarantee question. The key is write-ahead-logging + Kafka's durability:

Step 1: Alice's message is written to Kafka first (durable, replicated log). Only after Kafka ACKs does the server respond ✓ to Alice.
Step 2: The Message Delivery Service reads from Kafka and writes to Cassandra + offline queue. If the service crashes mid-way, Kafka retains the message at the last committed offset. The service restarts and re-processes from that offset.
Idempotency: Each message has a unique message_id (Snowflake ID). If a message is re-processed, the Cassandra insert uses IF NOT EXISTS — duplicate is ignored. Bob's client also deduplicates by message_id.

So the guarantee is: once Alice gets ✓ (sent to server/Kafka), the message will be delivered — even if servers crash. Kafka is the source of truth.
💡 "At-least-once" delivery: With Kafka retries, a message might be delivered twice. Deduplication at the client (message_id set) converts "at-least-once" to "exactly-once" from the user's perspective.
👥 Phase 3 — Group Chat Fanout
PHASE 3 · GROUP MESSAGING
IR
INTERVIEWER
Group chat with up to 1,024 members. Walk me through the fanout.
ME
YOU
When Alice sends a message to Group G (1,024 members):
  1. Alice's Chat Server receives the message, assigns a message_id
  2. Writes to Kafka topic: group-messages with key=group_id (ensures all group messages are ordered in one partition)
  3. Persists message to Cassandra group_messages table once
  4. A Group Fanout Service consumes from Kafka
  5. Looks up all 1,024 members of Group G from the Group Member Cache (Redis Set: group_members:{group_id})
  6. For each member: look up their Chat Server in Connection Registry
  7. For online members: push directly to their Chat Server via gRPC
  8. For offline members: write to their offline queue in Cassandra

Is 1,024 fanout expensive? Let's check: if 1M groups send messages simultaneously, and each has 100 active members on average, that's 100M delivery operations per second. Spread across many fanout workers consuming from Kafka in parallel — completely manageable.
IR
INTERVIEWER
For group read receipts — how does the sender know when all 1,024 members have read the message?
ME
YOU
Group read receipts are expensive to track precisely for 1,024 members. WhatsApp handles this pragmatically:

Delivered status: We don't wait for all 1,024 — ✓✓ shows when the first member receives it. Full delivery tracking is available in the group "info" screen (who received, who didn't).

Storage: Instead of tracking per-member per-message (1,024 rows per message — explodes at scale), I'd store a last_read_message_id per member per group: table group_read_status(group_id, user_id, last_read_msg_id). To know who's read a message, check who has last_read_msg_id ≥ target_msg_id. Much more efficient — one row per user per group rather than one row per user per message.
🟢 Phase 4 — Presence & Typing Indicators
PHASE 4 · PRESENCE
IR
INTERVIEWER
How do you implement online presence (last seen) and typing indicators efficiently?
ME
YOU
Online Presence:
When Bob opens the app, his Chat Server writes: SETEX presence:bob 60 online in Redis (TTL = 60 seconds). Bob's client sends a heartbeat ping every 30 seconds. Each ping refreshes the TTL. If Bob closes the app, no more pings → TTL expires → Redis key disappears → Bob is "offline".

When Alice opens a chat with Bob, her client requests Bob's presence: GET presence:bob. If key exists → "Online". If not → fetch Bob's last_seen timestamp from the User DB.

Typing Indicator:
When Alice starts typing, her client sends a TYPING_START event over WebSocket to the Chat Server. The server relays this directly to Bob's Chat Server (via gRPC — bypasses Kafka, too latency-sensitive for a queue). Bob sees "Alice is typing...".

When Alice stops typing (or after 5 seconds of inactivity), a TYPING_STOP event is sent. Typing events are NOT stored — they're ephemeral, fire-and-forget. If the event is lost, Bob just doesn't see "typing..." — that's fine.

Scalability concern: Presence and typing events could be noisy. If you have 600M users each sending heartbeats every 30 seconds, that's 20M events/second just for presence. Redis with a cluster handles this, but we'd also shard: presence:{user_id % 100} shard key.
IR
INTERVIEWER
What about message ordering? How do you ensure Alice's messages arrive to Bob in the order she sent them?
ME
YOU
Message ordering has two levels:

Client-to-server ordering: Alice's WebSocket is a single TCP connection — messages from Alice arrive at her Chat Server in the order she sent them. TCP guarantees ordering within a connection.

Server-side ordering: Kafka topics are partitioned by conversation_id. All messages in the same conversation go to the same partition → strictly ordered. The Message Delivery Service processes each partition sequentially → ordering preserved end-to-end.

Message IDs for ordering: Each message gets a Snowflake ID (timestamp-based, monotonically increasing). Even if Bob receives messages out of order (e.g., two from different senders in a group chat at the same millisecond), the client sorts by Snowflake ID to display in correct order.

The hard case: Alice sends Message 1 and Message 2 in quick succession. Message 2 is delivered to Bob first (network fluke). Client sorts by message_id → displays 1, 2 correctly. Gap in sequence detected → client knows Message 1 is still in transit, shows a "loading" indicator.
🔒 Phase 5 — End-to-End Encryption
PHASE 5 · SECURITY
IR
INTERVIEWER
How does end-to-end encryption work at this scale? The server stores messages — doesn't that break E2E?
ME
YOU
Great question. E2E encryption means the server stores ciphertext, not plaintext. The server acts as an encrypted message relay — it never has the decryption keys.

WhatsApp uses the Signal Protocol (Double Ratchet Algorithm). Here's the simplified flow:

Key Exchange (one-time setup):
  1. Alice generates a key pair: Identity Key, Signed Pre-Key, One-Time Pre-Keys
  2. Alice uploads her public keys to the Key Distribution Server (KDS)
  3. Bob does the same
Sending a message:
  1. Alice fetches Bob's public keys from KDS
  2. Alice performs a key exchange (X3DH algorithm) to derive a shared secret — a symmetric key that only Alice and Bob can compute
  3. Alice encrypts the message with this shared key using AES-256-GCM
  4. Alice sends the ciphertext to the server
  5. Server stores and relays the ciphertext — it's just bytes, no meaning
  6. Bob decrypts using his private key and the same derived shared secret
Forward Secrecy: The Double Ratchet generates a new encryption key for each message. Even if one key is compromised, past and future messages remain secure. This is why WhatsApp calls it "forward secrecy."

Group E2E: For groups, Alice generates a random group session key, encrypts it separately for each member using their public key (Signal's Sender Keys), and sends the encrypted group key. Members decrypt the group key, then use it to decrypt messages.
💡 Interview tip: You don't need to memorize Signal Protocol math. What matters is: (1) server never has plaintext, (2) keys are on devices only, (3) forward secrecy means past messages stay safe even if current key is compromised. That's enough depth for most interviewers.

🗄 Database Schema
-- ── Cassandra: Messages (hot store, 30-day retention) ──
CREATE TABLE messages (
    conversation_id  BIGINT,
    message_id       BIGINT,        -- Snowflake ID (time-sortable)
    sender_id        BIGINT,
    ciphertext       BLOB,          -- E2E encrypted content (server sees only bytes)
    media_url        TEXT,          -- S3 URL for images/voice (also encrypted key)
    msg_type         TEXT,          -- text | image | voice | system
    status           TEXT,          -- SENT | DELIVERED | READ | DELETED
    client_msg_id    TEXT,          -- client-generated UUID for dedup
    created_at       TIMESTAMP,
    PRIMARY KEY      (conversation_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC)
  AND default_time_to_live = 2592000  -- 30-day TTL auto-delete
  AND compaction = {'class': 'TimeWindowCompactionStrategy',
                    'compaction_window_size': 1,
                    'compaction_window_unit': 'DAYS'};
-- TimeWindowCompactionStrategy: optimised for time-series, fast TTL cleanup

-- ── Cassandra: Offline Message Queue ───────────────────
CREATE TABLE offline_messages (
    user_id          BIGINT,
    message_id       BIGINT,
    conversation_id  BIGINT,
    created_at       TIMESTAMP,
    PRIMARY KEY      (user_id, message_id)
) WITH CLUSTERING ORDER BY (message_id ASC)  -- ASC for ordered delivery
  AND default_time_to_live = 2592000;  -- 30 days

-- ── Cassandra: Group Members ────────────────────────────
CREATE TABLE group_members (
    group_id         BIGINT,
    user_id          BIGINT,
    role             TEXT,          -- admin | member
    joined_at        TIMESTAMP,
    last_read_msg_id BIGINT,       -- for read receipts: efficient per-member tracking
    PRIMARY KEY      (group_id, user_id)
);

-- ── Cassandra: Conversations ────────────────────────────
CREATE TABLE conversations (
    conversation_id  BIGINT  PRIMARY KEY,
    type             TEXT,   -- one_to_one | group
    participant_ids  LIST<BIGINT>,  -- [alice_id, bob_id] for 1:1
    group_id         BIGINT,        -- for group conversations
    created_at       TIMESTAMP,
    last_message_id  BIGINT         -- for conversation list preview
);

-- ── Redis Keys ──────────────────────────────────────────
-- Connection Registry: user_id → server_id
-- HSET connections {user_id} {server_id}   (TTL refreshed on heartbeat)

-- Presence: user online status
-- SETEX presence:{user_id} 60 "online"

-- Offline queue (fast access, falls back to Cassandra)
-- LPUSH offline:{user_id} {message_id}     (list, newest first)
-- LTRIM offline:{user_id} 0 999            (cap at 1000 msgs)

-- Group member cache (for fast fanout)

// ── 1. WebSocket Chat Server ─────────────────────────
@Component
@Slf4j
public class ChatWebSocketHandler implements WebSocketHandler {

    private final ConcurrentHashMap<Long, WebSocketSession> sessions
        = new ConcurrentHashMap<>();

    @Autowired private ConnectionRegistry   registry;
    @Autowired private KafkaTemplate<String, ChatMessage> kafka;
    @Autowired private OfflineQueueService  offlineQueue;
    @Autowired private PresenceService       presence;
    @Autowired private SnowflakeIdGenerator  idGen;
    @Value("${server.id}") private String serverId;

    @Override
    public void afterConnectionEstablished(WebSocketSession session) {
        long userId = extractUserId(session);
        sessions.put(userId, session);
        // Register: "userId is connected to this server"
        registry.register(userId, serverId);
        presence.setOnline(userId);
        // Flush offline queue immediately
        offlineQueue.flush(userId, session);
        log.info("User {} connected to server {}", userId, serverId);
    }

    @Override
    public void handleMessage(WebSocketSession session, WebSocketMessage<?> msg) {
        WsEvent event = parse(msg.getPayload().toString());
        long userId = extractUserId(session);

        switch (event.getType()) {
            case "SEND_MESSAGE"  -> handleSend(userId, event);
            case "MESSAGE_READ"  -> handleRead(userId, event);
            case "TYPING_START"  -> presence.broadcastTyping(
                                    event.getConversationId(), userId, true);
            case "TYPING_STOP"   -> presence.broadcastTyping(
                                    event.getConversationId(), userId, false);
            case "PING"          -> {
                presence.refreshHeartbeat(userId);
                send(session, "{\"type\":\"PONG\"}");
            }
        }
    }

    private void handleSend(long senderId, WsEvent event) {
        ChatMessage message = ChatMessage.builder()
            .messageId(idGen.nextId())           // Snowflake ID
            .conversationId(event.getConversationId())
            .senderId(senderId)
            .recipientId(event.getRecipientId())
            .ciphertext(event.getCiphertext())   // E2E: server sees only bytes
            .clientMsgId(event.getClientMsgId()) // for dedup
            .timestamp(Instant.now())
            .status(MessageStatus.SENT)
            .build();

        // Publish to Kafka (durable, ordered by conversationId partition)
        kafka.send("chat-messages",
            String.valueOf(message.getConversationId()),
            message);

        // ACK sender immediately — ✓ (sent to server)
        sendAck(senderId, event.getClientMsgId(),
                message.getMessageId(), "SENT");
    }

    /** Called by Delivery Service via gRPC to push to this server's user */
    public boolean pushToUser(long userId, ChatMessage message) {
        WebSocketSession session = sessions.get(userId);
        if (session == null || !session.isOpen()) return false;
        try {
            session.sendMessage(new TextMessage(toJson(message)));
            return true;
        } catch (Exception e) {
            log.warn("Push failed for user {}: {}", userId, e.getMessage());
            return false;
        }
    }

    @Override
    public void afterConnectionClosed(WebSocketSession session, CloseStatus status) {
        long userId = extractUserId(session);
        sessions.remove(userId);
        registry.deregister(userId);
        presence.setOffline(userId);
    }
}

// ── 2. Message Delivery Service (Kafka Consumer) ─────
@Service
@Slf4j
public class MessageDeliveryService {

    @Autowired private ConnectionRegistry   registry;
    @Autowired private ChatServerGrpcClient  grpcClient;
    @Autowired private MessageRepository     msgRepo;
    @Autowired private OfflineQueueService   offlineQueue;

    @KafkaListener(topics = "chat-messages",
                   groupId = "delivery-service",
                   concurrency = "20")  // 20 parallel consumers
    public void onMessage(ChatMessage message,
                          @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
                          Acknowledgment ack) {
        try {
            // Step 1: Persist to Cassandra (idempotent via IF NOT EXISTS)
            msgRepo.saveIfAbsent(message);

            // Step 2: Deliver to recipient
            deliverToRecipient(message.getRecipientId(), message);

            ack.acknowledge(); // commit offset only after successful processing
        } catch (Exception e) {
            log.error("Delivery failed for msg {}: {}",
                      message.getMessageId(), e.getMessage());
            // Don't ack → Kafka retries
        }
    }

    private void deliverToRecipient(long recipientId, ChatMessage message) {
        String targetServerId = registry.getServer(recipientId);

        if (targetServerId == null) {
            // User offline: add to offline queue
            offlineQueue.enqueue(recipientId, message.getMessageId());
            log.debug("User {} offline, queued msg {}",
                      recipientId, message.getMessageId());
            return;
        }

        // User online: deliver via gRPC to their Chat Server
        boolean delivered = grpcClient.push(targetServerId, recipientId, message);

        if (delivered) {
            msgRepo.updateStatus(message.getMessageId(),
                                 MessageStatus.DELIVERED);
            // Notify sender via Kafka event → sender sees ✓✓
            notifyDelivered(message.getSenderId(), message.getMessageId());
        } else {
            // Push failed (server crashed, user disconnected mid-flight)
            // Fall back to offline queue
            offlineQueue.enqueue(recipientId, message.getMessageId());
        }
    }
}

// ── 3. Group Fanout Service ───────────────────────────
@Service
public class GroupFanoutService {

    @Autowired private GroupRepository        groupRepo;
    @Autowired private MessageDeliveryService deliveryService;
    @Autowired private RedisTemplate<String,Long> redis;
    private final ExecutorService pool =
        Executors.newFixedThreadPool(50);

    @KafkaListener(topics = "group-messages", groupId = "fanout-service")
    public void fanout(ChatMessage message, Acknowledgment ack) {
        long groupId = message.getGroupId();

        // Get group members (Redis cache first, fallback to DB)
        Set<Long> members = getGroupMembers(groupId);

        // Save message once (not per-member)
        // deliveryService persists via Kafka consumer above

        // Fan out to each member (excluding sender)
        members.stream()
            .filter(memberId -> memberId != message.getSenderId())
            .forEach(memberId -> pool.submit(() -> {
                // Create per-member copy (same message_id, different recipient)
                ChatMessage copy = message.toBuilder()
                    .recipientId(memberId).build();
                deliveryService.deliverToRecipient(memberId, copy);
            }));

        ack.acknowledge();
    }

    private Set<Long> getGroupMembers(long groupId) {
        // Redis SET cache: SMEMBERS group_members:{groupId}
        Set<Long> cached = redis.opsForSet()
            .members("group_members:" + groupId);
        if (cached != null && !cached.isEmpty()) return cached;
        // Cache miss: load from DB, populate cache
        Set<Long> members = groupRepo.getMembers(groupId);
        redis.opsForSet().add("group_members:" + groupId,
            members.toArray(new Long[0]));
        redis.expire("group_members:" + groupId,
            Duration.ofHours(1));
        return members;
    }
}

// ── 4. Connection Registry (Redis-backed) ────────────
@Service
public class ConnectionRegistry {

    @Autowired private RedisTemplate<String,String> redis;
    private static final String REGISTRY_KEY = "connections";
    private static final int    TTL_SECONDS  = 90; // 3× heartbeat interval

    public void register(long userId, String serverId) {
        redis.opsForHash().put(REGISTRY_KEY,
            String.valueOf(userId), serverId);
        // Also set expiry per user key for auto-cleanup on crash
        redis.expire(REGISTRY_KEY + ":" + userId,
            Duration.ofSeconds(TTL_SECONDS));
    }

    public String getServer(long userId) {
        return (String) redis.opsForHash()
            .get(REGISTRY_KEY, String.valueOf(userId));
    }

    public void deregister(long userId) {
        redis.opsForHash().delete(REGISTRY_KEY,
            String.valueOf(userId));
    }

    /** Called on heartbeat — keeps connection alive */
    public void refresh(long userId) {
        redis.expire(REGISTRY_KEY + ":" + userId,
            Duration.ofSeconds(TTL_SECONDS));
    }
}

# ── 1. Connection Registry ───────────────────────────
class ConnectionRegistry:
    """
    Redis-backed map: user_id → server_id
    Entries expire automatically if heartbeat stops (TTL=90s)
    """
    TTL = 90

    def __init__(self, redis_client, server_id: str):
        self.redis = redis_client
        self.server_id = server_id

    async def register(self, user_id: int):
        await self.redis.hset("connections", str(user_id), self.server_id)
        await self.redis.setex(f"conn_ttl:{user_id}", self.TTL, "1")

    async def get_server(self, user_id: int) -> Optional[str]:
        # Check TTL key first (auto-cleanup on crash)
        alive = await self.redis.exists(f"conn_ttl:{user_id}")
        if not alive:
            await self.deregister(user_id)
            return None
        server = await self.redis.hget("connections", str(user_id))
        return server.decode() if server else None

    async def deregister(self, user_id: int):
        await self.redis.hdel("connections", str(user_id))
        await self.redis.delete(f"conn_ttl:{user_id}")

    async def refresh(self, user_id: int):
        """Called on heartbeat to keep connection alive"""
        await self.redis.setex(f"conn_ttl:{user_id}", self.TTL, "1")


# ── 2. Chat WebSocket Handler ─────────────────────────
class ChatServer:
    """
    Manages WebSocket connections for this server instance.
    Sessions held in memory — server is stateful per connection.
    """
    def __init__(self, registry, kafka_producer, offline_queue,
                  presence_svc, snowflake_gen):
        self.sessions: Dict[int, WebSocket] = {}
        self.registry   = registry
        self.kafka      = kafka_producer
        self.offline_q  = offline_queue
        self.presence   = presence_svc
        self.id_gen     = snowflake_gen

    async def connect(self, user_id: int, ws: WebSocket):
        await ws.accept()
        self.sessions[user_id] = ws
        await self.registry.register(user_id)
        await self.presence.set_online(user_id)
        # Flush offline queue on reconnect
        await self.offline_q.flush(user_id, ws)
        logger.info(f"User {user_id} connected")

    async def disconnect(self, user_id: int):
        self.sessions.pop(user_id, None)
        await self.registry.deregister(user_id)
        await self.presence.set_offline(user_id)

    async def handle_message(self, user_id: int, raw: str):
        event = json.loads(raw)
        etype = event.get("type")

        if etype == "SEND_MESSAGE":
            await self._handle_send(user_id, event)
        elif etype == "MESSAGE_READ":
            await self._handle_read(user_id, event)
        elif etype in ("TYPING_START", "TYPING_STOP"):
            is_typing = (etype == "TYPING_START")
            await self.presence.broadcast_typing(
                event["conversation_id"], user_id, is_typing)
        elif etype == "PING":
            await self.registry.refresh(user_id)
            await self._send(user_id, {"type": "PONG"})

    async def _handle_send(self, sender_id: int, event: dict):
        message = {
            "message_id":      self.id_gen.next_id(),  # Snowflake
            "conversation_id": event["conversation_id"],
            "sender_id":       sender_id,
            "recipient_id":    event.get("recipient_id"),
            "ciphertext":      event["ciphertext"],   # E2E encrypted
            "client_msg_id":   event["client_msg_id"], # for dedup
            "timestamp":       time.time(),
            "status":          "SENT"
        }
        # Publish to Kafka (durable, ordered)
        await self.kafka.send(
            topic="chat-messages",
            key=str(message["conversation_id"]),  # partition by conv
            value=message
        )
        # ACK sender: ✓ sent to server
        await self._send(sender_id, {
            "type":          "MESSAGE_ACK",
            "client_msg_id": event["client_msg_id"],
            "server_msg_id": message["message_id"],
            "status":        "SENT"
        })

    async def push_to_user(self, user_id: int, message: dict) -> bool:
        """Called by Delivery Service (gRPC) to push to this server's user"""
        ws = self.sessions.get(user_id)
        if not ws or ws.client_state != WebSocketState.CONNECTED:
            return False
        try:
            await ws.send_json({
                "type":            "NEW_MESSAGE",
                "message_id":      message["message_id"],
                "conversation_id": message["conversation_id"],
                "sender_id":       message["sender_id"],
                "ciphertext":      message["ciphertext"],
                "timestamp":       message["timestamp"]
            })
            return True
        except Exception as e:
            logger.warning(f"Push failed user {user_id}: {e}")
            return False

    async def _send(self, user_id: int, data: dict):
        ws = self.sessions.get(user_id)
        if ws: await ws.send_json(data)


# ── 3. Message Delivery Service (Kafka Consumer) ──────
class MessageDeliveryService:
    """
    Consumes from Kafka 'chat-messages' topic.
    Delivers to online users via gRPC or queues for offline users.
    """
    def __init__(self, registry, grpc_clients, msg_repo, offline_queue):
        self.registry    = registry
        self.grpc        = grpc_clients   # dict: server_id → gRPC stub
        self.msg_repo    = msg_repo
        self.offline_q   = offline_queue

    async def process_message(self, message: dict):
        """Process a single message from Kafka — deliver or queue."""
        # Step 1: Persist (idempotent — skip if message_id exists)
        await self.msg_repo.save_if_absent(message)

        # Step 2: Find recipient's server
        recipient_id = message["recipient_id"]
        server_id = await self.registry.get_server(recipient_id)

        if not server_id:
            # Offline: queue for later delivery
            await self.offline_q.enqueue(recipient_id, message["message_id"])
            return

        # Online: deliver via gRPC to target Chat Server
        stub = self.grpc.get(server_id)
        if stub:
            delivered = await stub.push_message(recipient_id, message)
            if delivered:
                await self.msg_repo.update_status(
                    message["message_id"], "DELIVERED")
                # Notify sender: ✓✓ delivered
                await self._notify_sender(
                    message["sender_id"],
                    message["message_id"], "DELIVERED")
            else:
                # Push failed (user disconnected mid-flight)
                await self.offline_q.enqueue(
                    recipient_id, message["message_id"])

    async def _notify_sender(self, sender_id: int, msg_id: int, status: str):
        server_id = await self.registry.get_server(sender_id)
        if server_id:
            stub = self.grpc.get(server_id)
            if stub:
                await stub.push_delivery_update(sender_id, msg_id, status)

⚖️ Tradeoffs & Design Decisions
For messaging systems, reliability > performance. A slightly slower delivery is acceptable. A lost message is never acceptable. Always bias toward durability.
🔌 Real-time Transport: WebSocket vs Long Polling vs SSE
✅ Chosen: WebSocket

Full-duplex. One persistent TCP connection for both send and receive. Industry standard for chat (Slack, Discord, WhatsApp). ~10KB RAM per connection. Supports all event types: messages, typing, ACKs.

❌ Long Polling

Client polls every few seconds. High server load (millions of empty polls). No true real-time — 1-3s latency baseline. Wasteful bandwidth. Only viable if WebSocket is blocked by firewall.

💡 SSE for notifications

Server-Sent Events (SSE) works for one-way push (notifications, alerts). But chat needs bidirectional — SSE would require a separate HTTP connection for client→server. WebSocket does both over one connection. Use SSE only if you need simple push notifications without bidirectional ACKs.

📬 Message Delivery: At-least-once vs Exactly-once vs At-most-once
✅ Chosen: At-least-once + Client Dedup

Kafka guarantees at-least-once delivery. Deduplication at client using message_id converts to effectively exactly-once. Simple to implement, strong durability.

❌ At-most-once

Fire-and-forget — no retries. Fast but loses messages if server crashes. Completely unacceptable for a chat system where message loss = broken trust. Only suitable for ephemeral events like typing indicators.

💡 True exactly-once

Kafka 0.11+ supports exactly-once semantics with idempotent producers + transactions. More complex to configure. The simpler approach (at-least-once + client dedup by message_id) achieves the same user-visible outcome with less operational complexity.

🗄 Message Storage: Cassandra vs MySQL vs MongoDB
✅ Chosen: Cassandra

Access pattern is pure time-series: "give me messages in conversation X before timestamp T". Cassandra excels at this with partition-by-conversation, cluster-by-time. Handles 9 PB with horizontal scale. Built-in TTL for 30-day auto-delete.

❌ MySQL / PostgreSQL

Good for small scale. But 100TB/day of new messages makes SQL painful — sharding is manual and complex. Relational ACID is unnecessary overhead for chat (no multi-table transactions). Limited TTL support.

💡 MongoDB?

MongoDB is a reasonable alternative — flexible schema, good write throughput. Facebook Messenger actually uses HBase (similar to Cassandra architecture). The key insight is all these options work; what matters is partitioning by conversation_id for locality, and time-clustering for range queries.

🔒 E2E Encryption: Signal Protocol vs Custom vs No E2E
✅ Chosen: Signal Protocol

Battle-tested. Open source. Forward secrecy (new key per message). Used by WhatsApp, Signal, Google Messages. Prevents server from reading messages even if hacked. Industry gold standard.

❌ Server-side encryption

Server encrypts messages with its own key. Server can decrypt — trust the server model. Vulnerable to insider attacks, government subpoenas, server breaches. No forward secrecy. Simpler to implement but weaker security guarantee.

💡 Cost of E2E

E2E has real tradeoffs: no server-side spam detection (can't read content), no message backup unless client manages keys, message search requires client-side index. WhatsApp accepts these tradeoffs. For enterprise chat (Slack), E2E is often sacrificed for compliance/search features.

⚡ Presence System: Redis TTL vs Push/Subscribe vs Dedicated Service
✅ Chosen: Redis TTL + Heartbeat

Simple and self-healing. Client sends heartbeat every 30s. Redis key TTL = 60s. If heartbeat stops → key expires → user shown offline. No explicit "logout" needed. Handles crashes gracefully.

⚠️ Push Subscribe (Redis Pub/Sub)

Real-time presence updates pushed to contacts. More responsive but complex. 600M users each with ~200 contacts = 120B subscription relationships. Redis Pub/Sub can't scale to this — need a dedicated presence fanout system.

💡 Polling for presence

For "last seen", clients poll on-demand when opening a chat — not pushed proactively. This dramatically reduces presence update volume. Only "online right now" needs near-real-time. "Last seen 2 hours ago" can tolerate 30s stale — poll on chat open, not push subscription.

👥 Group Fanout: Server-side vs Client-side
✅ Chosen: Server-side Fanout

Server delivers to each member. Client sends once. Works for all group sizes. Offline members handled transparently. Sender doesn't need to know other members' connection status.

❌ Client-side Fanout

Client sends the message to each member individually. Simple server. But: client must know all member connection states, handles offline logic itself, N×bandwidth from sender's mobile device. Impractical for 1,024-member groups.

💡 Hybrid for large groups

For small groups (<50 members), direct server routing is fine. For large groups (50–1,024), a dedicated fanout worker pool with Kafka is more efficient — processes batches asynchronously, scales independently from Chat Servers.
