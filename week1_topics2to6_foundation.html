3 Mental Models for System Design

Before drawing a single box, you need the right intuition. These three models are a repeatable thinking framework applicable to every system design problem in this course and beyond.

Why Mental Models Matter

Most engineers jump to components — "put a Kafka here, a Redis there." That leads to over-engineered, fragile designs. Mental models force you to ask why before what. They turn ambiguous problems into structured reasoning paths.

🔬

Model 1

Decompose

Break any system into: read path, write path, and storage. Solve each independently.

🔍

Model 2

Bottleneck First

Find the single slowest/hottest component. Optimize it first. Everything else is premature optimization.

⚖️

Model 3

Trade-off Surfacing

For every decision: state what you gain, what you give up, and under what conditions you'd flip the choice.

Arpit's core principle: A good design is always simple and elegant. If you're adding a component, ask — "would I build and operate this myself?" If not, reconsider. Complexity should be justified by a concrete requirement, not by making the diagram look impressive.
Model 1 — Decompose

Every system, no matter how complex, can be split into three paths. Design each path independently, then compose them.

The Three Paths

Write Path

How does data enter the system? Direct DB write? Queue → async worker? Batch ingest? The write path defines your durability and write throughput constraints.

  • Sync write → immediate durability, lower throughput
  • Async via queue → higher throughput, eventual durability
  • Batch → highest throughput, highest latency

Read Path

How does data leave the system? Direct DB query? Cache hit? CDN? The read path defines your read latency and consistency guarantees.

  • Cache-aside → fast reads, potential staleness
  • Read-through → transparent caching, cache miss penalty
  • Read replicas → horizontal read scaling

Storage Layer

What is the source of truth? Relational, NoSQL, in-memory, blob? The storage layer defines your data model, query patterns, and scale ceiling.

  • Relational → ACID, complex queries, vertical scale ceiling
  • NoSQL → horizontal scale, eventual consistency
  • In-memory → highest speed, ephemeral

Applied to Online/Offline Indicator

Rule: Optimize Write and Read Paths Independently

Write path can be async

Heartbeat writes to Redis (fast, in-memory) and asynchronously to MySQL via Kafka. The caller doesn't wait for MySQL. Write latency = Redis latency only (~1 ms).

Read path can be stale

Status query reads from Redis. If key is gone (user offline), falls back to MySQL last_seen. Slight staleness (up to TTL) is acceptable by requirement.

Model 2 — Bottleneck First

Find the single component that will fail first under load. Solve that. Don't pre-optimize everything — that leads to premature complexity.

Bottleneck Identification Process

1
Calculate load per component
Start with your capacity estimate. Divide total load by component capacity. The one closest to 100% is your bottleneck.
2
Identify the critical path
Trace a single request end-to-end. The slowest hop in the chain is your latency bottleneck. The component with highest concurrency is your throughput bottleneck.
3
Fix only the bottleneck — not everything
Adding Redis won't help if your DB is not the bottleneck. Adding Kafka won't help if the consumer is the bottleneck. One fix at a time.
4
Re-calculate after each fix
When you fix one bottleneck, the next one surfaces. This is called "shifting the bottleneck." Design iterates, not explodes.

The "Good Enough" Principle

A system that handles 10× current load with simple components is better than one that handles 100× with 20 components. Design for the next 18 months, not for hypothetical 10B users. Complexity has a cost: ops overhead, debugging time, hiring.
Model 3 — Trade-off Surfacing

Every design decision is a trade-off. The skill is not picking the "right" answer — it's articulating what you give up for what you gain.

The Trade-off Framework

Structure

  • Option A — what you gain, what you lose
  • Option B — what you gain, what you lose
  • When to flip — conditions under which you'd change your choice
  • Default choice — for your current scale/requirements

The CAP Triangle (applied)

  • Consistency + Availability → gives up Partition Tolerance
  • Availability + Partition Tolerance → gives up Consistency (most web systems)
  • Consistency + Partition Tolerance → gives up Availability
  • Most systems choose AP — accept eventual consistency

Core Trade-off Pairs You'll Encounter Every Week

Applying to Any Interview Question

"I'll use Redis for the hot path because it gives us sub-millisecond reads and atomic operations. The trade-off is that it's in-memory — data is lost on crash unless we persist with RDB/AOF. Given that our requirement is eventual consistency and we have MySQL as a fallback for last_seen, this trade-off is acceptable at our scale."

→ This is the response pattern that gets you senior-level credit.
Applying All 3 Models Together

A walkthrough of how to apply all three models sequentially to a new problem — using a URL shortener as an example.

Example: Design a URL Shortener

1
Decompose (Model 1)
Write path: POST /shorten → generate short code → store in DB. Read path: GET /<code> → lookup → 301 redirect. Storage: key-value (code → long URL). These are independent — read is 100× heavier than write.
2
Bottleneck First (Model 2)
1B redirects/day ÷ 86400 = 11,600 reads/s. DB max ~5K reads/s/node. Bottleneck = DB read. Fix: add Redis cache for top URLs. 95% cache hit rate → DB sees only 580 reads/s. ✅
3
Trade-off Surfacing (Model 3)
Cache with 1h TTL: fast, but if URL is deleted, stale for 1h. Trade-off: simplicity + speed vs. instant consistency on delete. At scale, acceptable. For enterprise (link invalidation critical), use shorter TTL or active cache invalidation.

The Mental Model Cheat Sheet

Start here

Decompose: What are the read path, write path, and storage for this system? Draw them separately before connecting them.

Then here

Bottleneck: Given the capacity numbers, which component hits 100% first? Solve only that. Resist adding components for hypothetical scale.

Always do this

Trade-off: For every major component you add, state: gain, loss, and flip condition. Never add anything without articulating the trade-off.

Outcome

What you've internalized from the 3 Mental Models topic.

  • Decompose into read/write/storage paths — every system can be split this way; design each path for its own constraints before connecting them.
  • Bottleneck-first design — calculate load per component, find the one closest to capacity, fix only that. Premature optimization is the root of over-engineered systems.
  • Trade-off articulation is the skill — knowing Redis is fast is table stakes. Knowing what you give up (memory, cost, persistence) and under what conditions you'd switch — that's senior engineering.
  • Shifting bottleneck pattern — fixing one bottleneck surfaces the next. This is normal and expected. Design iterates toward simplicity, not away from it.
  • Simple is harder than complex — adding components is easy. Justifying each one with a concrete requirement is the discipline that separates good designs from cargo-cult architectures.

Connection Pools & DB Proxies

Every database has a hard limit on concurrent connections. Without pooling, your app servers will exhaust it under moderate load. Understanding this is foundational to every data-intensive system.

The Problem

Without Connection Pooling

Every request opens a new DB connection → executes query → closes connection. TCP handshake + TLS + DB auth = 5–50ms overhead per query. At 10K RPS: 10K new connections/s → DB crashes.

MySQL default max_connections = 151. Your app has 10 servers × 100 threads = 1000 connections. 💥

With Connection Pooling

Pool maintains N persistent connections. Requests borrow a connection, use it, return it. Overhead = queue wait time (microseconds). Same 10K RPS with 50 pool connections per server = fine.

10 servers × 50 pool conns = 500 total DB connections. MySQL handles this easily.

Connection Pools — Deep Dive

A connection pool is a cache of pre-established DB connections. Requests borrow from the pool and return after use.

How a Pool Works

1
Pool initialises min_connections at startup
Opens min_size (e.g., 5) connections to DB during app startup. No per-request overhead from here on.
2
Request borrows a connection
If a free connection exists → borrow immediately (microseconds). If pool exhausted → wait in queue or raise PoolTimeout exception.
3
Query executes, connection returned
After query, connection is returned to pool — NOT closed. Pool can reuse it for the next request. Connection stays warm (authenticated, session vars set).
4
Pool scales up to max_connections
If all connections busy, pool opens new ones up to max_size. After burst, idle connections are closed back down to min_size.

Pool Size Formula

Practical formula: pool_size = (core_count × 2) + effective_spindle_count
For a 4-core app server: pool_size = (4 × 2) + 1 = 9 connections.

Counter-intuitively, fewer connections = better throughput up to a point. More connections = more context switching on the DB server = degraded performance. HikariCP's default is 10.

DB Proxies

When you have 100s of app servers each with their own pool, the DB still sees 100s × 50 = 5000 connections. A DB Proxy multiplexes all app connections into a smaller set of actual DB connections.

PgBouncer Pooling Modes

Session Mode

1 client = 1 server conn for the session duration. Same as no proxy for connection count. Only helps with auth overhead.

Transaction Mode ✓

Server conn held only during transaction. Most efficient — 1 conn shared by many clients. Most common choice. Breaks session-level features (prepared statements, advisory locks).

Statement Mode

Conn returned after each statement. Ultra-aggressive reuse. Only works if queries are all single-statement (no transactions). Rarely used.

Issues at Scale

What goes wrong when you get connection pooling wrong — and how to detect and fix each issue.

Common Failure Patterns

Connection Pool Exhaustion

All pool connections are busy. New requests queue or get PoolTimeout. Symptom: p99 latency spikes, errors like "connection pool exhausted."

Cause: Long-running queries holding connections. Fix: Set query timeout; increase pool size; optimize slow queries.

Connection Leak

App checks out a connection, throws an exception, never returns it. Pool slowly drains. Symptom: gradual latency increase, restart fixes it.

Fix: Always use try-finally or context managers. Pool max_lifetime as safety net.

Thundering Herd

App restarts; all pool connections initialize at once → DB gets 10 servers × 50 = 500 new connections in 1 second. DB connection setup is expensive.

Fix: Staggered startup, lazy pool initialization, exponential backoff.

Stale Connections

DB restarts or firewall drops idle connections. Pool still thinks connections are alive → first query on stale conn fails.

Fix: Set max_lifetime < DB wait_timeout. Use testOnBorrow=true or keepalive.

How Locking Affects Pool Throughput

Key insight: A long-running query holding a DB row lock also holds a pool connection. If 50 requests are waiting for the lock, they also hold 50 pool connections doing nothing. This is how a single slow query can cascade into a pool exhaustion event that takes down the entire service.
Lock TypeScopePool ImpactMitigation
Row lock (SELECT FOR UPDATE)Single rowConn held until transaction commitsKeep transactions short; use optimistic locking
Table lockEntire tableAll concurrent conns blockedAvoid; use row-level locks only
DeadlockTwo rowsBoth conns held until timeoutAlways acquire locks in same order
Long transactionAll modified rowsConn held for transaction durationSet innodb_lock_wait_timeout; avoid long transactions

db_pool.py — SQLAlchemy async poolPython
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import sessionmaker
from contextlib import asynccontextmanager

# Connection string with pool params
DATABASE_URL = "postgresql+asyncpg://user:pass@localhost/mydb"

engine = create_async_engine(
    DATABASE_URL,
    pool_size=10,          # steady-state connections
    max_overflow=20,       # extra conns allowed during burst
    pool_timeout=30,       # wait 30s for a free connection
    pool_recycle=1800,     # force-close conns after 30 min
    pool_pre_ping=True,    # test conn before handing out (stale check)
    echo=False
)

AsyncSessionLocal = sessionmaker(
    engine, class_=AsyncSession, expire_on_commit=False
)

@asynccontextmanager
async def get_db():
    """Context manager — guarantees connection return even on exceptions."""
    session = AsyncSessionLocal()
    try:
        yield session
        await session.commit()
    except Exception:
        await session.rollback()
        raise
    finally:
        await session.close()   # returns conn to pool, never closes it

# Usage — connection auto-returned even if exception occurs
async with get_db() as db:
    result = await db.execute("SELECT 1")

import com.zaxxer.hikari.*;

@Configuration
public class DataSourceConfig {

    @Bean
    public DataSource dataSource() {
        HikariConfig config = new HikariConfig();

        config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb");
        config.setUsername("user");
        config.setPassword("pass");

        // Pool sizing — (cpu_cores * 2) + disk_spindles
        config.setMaximumPoolSize(10);     // max connections per app node
        config.setMinimumIdle(10);         // keep pool fully warmed
        config.setConnectionTimeout(30000); // 30s: fail fast if pool exhausted
        config.setIdleTimeout(600000);    // 10m: close idle extra conns
        config.setMaxLifetime(1800000);   // 30m: recycle to prevent stale
        config.setKeepaliveTime(60000);   // 1m: keep-alive ping

        // Validate connections before use (prevents stale conn errors)
        config.setConnectionTestQuery("SELECT 1");

        // Pool name for metrics/logging
        config.setPoolName("MainPool");

        // Register with Micrometer for monitoring
        config.setMetricRegistry(meterRegistry);

        return new HikariDataSource(config);
    }
}
Outcome

What you've learned about connection pools and DB proxies.

  • Why pools exist — TCP connection setup is 5–50ms; at 10K RPS that's the entire latency budget. Pools amortize this cost across thousands of requests.
  • Pool sizing formula — (cores × 2) + spindles. Counterintuitively, more connections ≠ more throughput above a threshold. HikariCP defaults to 10 for a reason.
  • PgBouncer transaction mode — the sweet spot for PostgreSQL. 1 DB connection serves many clients as long as each transaction is short. Breaks session-level features — understand the tradeoff.
  • ProxySQL R/W splitting — route SELECT to replicas, INSERT/UPDATE to primary. Multiplexes 5000 app connections → 50 real DB connections. Essential at 100+ app servers.
  • Connection leak → pool exhaustion — always use context managers / try-finally. A single leaked connection can cascade to a full service outage.
  • Locks hold connections — a row lock in a transaction holds a pool connection. Long transactions under load = pool exhaustion cascade. Keep transactions tight and fast.

Caching and their Issues at Scale

Caching is the single most impactful performance optimization in distributed systems — and the source of the most subtle bugs. Understanding both sides is essential.

Where Caches Live

Client-side Cache

In-process memory (HashMap, Guava Cache). Zero network latency. Private per instance — can diverge across servers. Good for static/rarely-changed data (config, feature flags).

Distributed Cache

Redis, Memcached. Shared across all servers. Network hop (1–5ms). Can scale horizontally. Source of truth for shared mutable state like sessions, presence, rate limits.

CDN / Edge Cache

Cloudflare, Fastly. Geographically distributed. Caches static assets, HTML, API responses near users. Best for content that changes infrequently and is user-agnostic.

Cache Strategies

Different strategies suit different read/write patterns. Choosing the wrong one is a common design mistake.

StrategyHow It WorksBest ForDownside
Cache-aside (Lazy)App checks cache → miss → fetch DB → populate cache → returnRead-heavy, irregular access patternsCache miss penalty (2 hops); stale data possible
Read-throughCache intercepts read; on miss, fetches DB automaticallyUniform read patterns; simpler app codeCold start misses; cache library dependency
Write-throughWrite goes to cache AND DB synchronouslyConsistent reads immediately after writeWrite latency = cache + DB; stores data never read
Write-behind (Write-back)Write to cache only; async flush to DB laterWrite-heavy; throughput > durabilityData loss risk on cache crash before flush
Refresh-aheadProactively refresh cache before TTL expiresPredictable access (e.g., homepage)Wastes resources refreshing unused keys
Most common in practice: Cache-aside for reads + Write-through for writes on critical paths. Write-behind for analytics/counters where eventual persistence is acceptable.
Scale Issues

These are the failure modes that bite you in production. Every one of them has a name because it has bitten someone famous first.

🌊 Cache Stampede (Thundering Herd)

A popular key's TTL expires. Hundreds of concurrent requests miss the cache simultaneously → all hit the DB at once → DB overwhelmed → cascade failure.

Fix: Probabilistic Early Expiration (PER), mutex lock on first miss ("cache warming lock"), or background refresh before TTL expires.

1M RPS on a key that expires every 60s = all 1M requests potentially hitting DB in a 100ms window.

🎯 Cache Penetration

Attacker (or buggy client) requests keys that don't exist in cache OR DB. Every request hits DB. Cache is bypassed entirely.

Fix: Cache negative results (SET key "NULL" EX 60). Or use a Bloom filter to reject non-existent keys before any lookup.

🌊 Cache Avalanche

Many cache keys expire at the same time (e.g., set with identical TTL during app restart). Flood of DB requests simultaneously.

Fix: Add jitter to TTLs: TTL = base_ttl + random(0, base_ttl * 0.1). Staggers expiry across time window.

⭐ Hot Key (Celebrity Problem)

One key (Justin Bieber's profile) gets 100K reads/s. Even with Redis cluster, that key lives on one shard → that shard becomes a hotspot.

Fix: Replicate hot keys to multiple shards (user:bieber:1, user:bieber:2…). Read from random shard. Or local in-process cache for VIP keys.

🔄 Cache Inconsistency (Stale Reads)

DB is updated but cache isn't invalidated. Reads return stale data. Worse: two cache nodes have different values during replication lag.

Fix: Short TTL. Active invalidation (publish invalidation event to all nodes). "Cache-aside + write-through" pattern for critical data.

💾 Cache Warm-up on Restart

After cache restart, all requests miss → DB suddenly sees 10× normal load. Systems can fail or be very slow for minutes post-restart.

Fix: Redis persistence (RDB/AOF). Pre-warm cache from DB snapshot before switching traffic. Blue-green cache deployment.

Outcome
  • Cache strategies map to patterns — cache-aside for irregular reads, write-through for consistency, write-behind for throughput. Knowing which to use when is a senior-level skill.
  • Cache stampede is the most dangerous failure — 1M concurrent cache misses on TTL expiry → DB overwhelmed. Mitigate with probabilistic refresh or mutex locks.

Why Kafka and Not a Simple Queue?

Traditional Queue (RabbitMQ)

  • Message consumed → deleted
  • One consumer group only (by default)
  • Push-based delivery
  • Good for: task queues, RPC, job scheduling

Kafka

  • Message consumed → retained (by time/size)
  • Multiple independent consumer groups
  • Pull-based; consumers control pace
  • Good for: event streaming, log aggregation, decoupling at scale
Kafka Internals

Understanding the data model is key to understanding why Kafka is fast and what its guarantees are.

Core Concepts

Topic

A named stream of events. Think of it as a database table but append-only. Producers write to topics; consumers read from them.

Partition

A topic is split into N partitions. Each partition is an ordered, append-only log stored on disk. Partitions enable parallelism — one consumer per partition max.

Offset

Each message in a partition has a monotonically increasing offset. Consumers track their offset — this is how Kafka knows what each consumer has read.

Consumer Group

Multiple consumers sharing a group ID split partitions among themselves (each partition → one consumer). Multiple groups = independent reads of the same data.

Broker

A Kafka server. Each broker holds a subset of partitions. Leader partition handles reads/writes; follower partitions replicate for HA.

Replication Factor

How many copies of each partition exist across brokers. RF=3 → 1 leader + 2 followers. Survives 2 broker failures before data loss.

Partitioning Strategy

Key-based (Hash)

Same key always goes to same partition. Guarantees ordering per key. Use for: user events (key=user_id → all events for user in order).

Round-robin

Distribute evenly across partitions. Max throughput. No ordering guarantee across partitions. Use for: high-volume logs where ordering doesn't matter.

Custom Partitioner

Write your own logic. E.g., route hot users to dedicated partitions, or shard by region. Adds complexity — use only when needed.

Kafka Consumers

Consumer design determines scalability, fault tolerance, and processing guarantees.

Consumer Group Scaling

The maximum parallelism of a consumer group = number of partitions. Adding more consumers than partitions results in idle consumers. Rule: partitions ≥ max consumer instances you'll ever want.
PartitionsConsumersResult

Offset Management & Delivery Semantics

At-most-once

Commit offset before processing. If consumer crashes mid-processing, message is lost (offset already committed).

At-least-once ✓

Process first, commit after. Crash before commit → message re-processed. Your handler must be idempotent. Most common choice.

Exactly-once

Use Kafka transactions + idempotent producer. Highest overhead. Needed for financial systems. Avoid unless truly required.

Consumer Lag — The Key Metric

Consumer lag = latest partition offset − consumer's committed offset. If lag is growing, your consumers can't keep up. Fix: add partitions + consumers, or optimize processing logic. Alert on lag > 10K messages.
Kafka Guarantees & When to Use

Ordering Guarantees

Within a Partition

Messages are strictly ordered by offset. If you publish events with the same key, they land on the same partition → processed in order. This is Kafka's primary ordering guarantee.

Across Partitions

No ordering guarantee. Two messages published milliseconds apart may be processed in any order if they land on different partitions. Design accordingly.

Outcome
  • Kafka is a log, not a queue — messages persist after consumption; multiple consumer groups read independently; this is the key architectural insight.
  • Partitions = parallelism ceiling — max consumers in a group = num partitions. Set partitions generously at topic creation; reducing partitions is painful.
  • Key-based partitioning guarantees order per key — all heartbeats for user_id=42 land on the same partition, consumed in order, enabling correct last_seen tracking.
  • acks=1 for high-throughput, acks=all for durability — choose based on what you can afford to lose. For presence heartbeats, acks=1 is fine; for financial events, acks=all.
  • At-least-once + idempotent handlers — the practical default. Design your consumer (e.g., UPSERT with MAX) to handle duplicate delivery without side effects.
  • Consumer lag is the health metric — monitor it, alert on growth. Growing lag = consumer can't keep up → add partitions and consumers.
  • Don't use Kafka for everything — simple job queues → RabbitMQ. Low-volume sequential tasks → DB queue. Kafka overhead is justified only at scale or with multiple consumer groups.

Communication Paradigms & Deployment Log Streamer

How services talk to each other — synchronously or asynchronously, push or pull, request-response or event-driven — fundamentally shapes your system's scalability, coupling, and failure modes.

🔄

Synchronous

REST / gRPC

Caller waits for response. Tight coupling. Simple to reason about. Fails if callee is down.

📡

Asynchronous

Kafka / Queue

Caller publishes and moves on. Loose coupling. Callee processes at its own pace. Handles bursts.

🌊

Streaming

WebSocket / SSE / gRPC Stream

Persistent connection. Real-time bidirectional or push. Long-lived connection overhead.

Communication Paradigms

REST (HTTP/1.1, HTTP/2)

Characteristics

  • Request-response; stateless; text or JSON
  • Human-readable; widely supported
  • HTTP/2 multiplexes multiple requests over one TCP conn
  • Easy to debug, cache, and proxy

Limitations

  • One request = one response (no streaming by default)
  • Higher overhead than binary protocols (headers, text encoding)
  • No built-in schema enforcement (OpenAPI is optional)

gRPC (HTTP/2 + Protobuf)

Why gRPC

  • Binary serialization (Protobuf) — 3–10× smaller than JSON
  • Strongly typed schema (.proto files) — compile-time contract
  • Supports unary, server-streaming, client-streaming, bidirectional streaming
  • HTTP/2 multiplexing — multiple streams over one connection
  • Built-in deadline propagation and cancellation

When Not to Use gRPC

  • Browser clients (limited gRPC-web support)
  • Public APIs (REST is more universally understood)
  • Teams unfamiliar with Protobuf tooling

WebSocket

How it works

HTTP upgrade → persistent bidirectional TCP connection. Client and server can send messages at any time. Used for: chat, live feeds, multiplayer games, presence indicators.

Scaling challenge: WebSocket connections are stateful. Load balancers need sticky sessions. Each server holds N open connections — harder to scale horizontally than stateless REST.

SSE (Server-Sent Events)

One-way: server pushes events to browser over HTTP. Simpler than WebSocket for unidirectional push (notifications, live dashboards, log streaming).

  • Works over plain HTTP (proxies, CDNs work)
  • Automatic reconnect built into browser
  • One direction only (client can't send)
  • Perfect for: deployment log streaming ✓

Design: Deployment Log Streamer

Stream deployment logs in real-time to the browser — like watching kubectl logs -f or Heroku build logs. The canonical SSE use case.

Requirements

Functional

  • User initiates a deployment; browser opens a log stream
  • Log lines appear in real-time as the build/deploy runs
  • Reconnect automatically on connection drop
  • Stream terminates when deployment succeeds or fails

Non-functional

  • Latency: log line visible in browser <500ms of being written
  • Scale: 10K concurrent deployments, ~100 browser watchers each
  • History: serve last N lines to newly connected viewers
  • No WebSocket (works through corporate proxies)

Architecture

Why SSE over WebSocket Here

SSE Advantages

  • Works over plain HTTP — no proxy issues in corporate networks
  • Browser EventSource API has built-in auto-reconnect with Last-Event-ID resume
  • Easier load balancing — just sticky session or consistent hash on deploy_id

Why Redis Pub/Sub

  • Build agent may be on a different server than the SSE server the browser is connected to
  • Redis Pub/Sub broadcasts the log line to all SSE server instances subscribed to that deploy_id channel
  • LRANGE provides history for reconnecting browsers

log_streamer.py — FastAPI + Redis Pub/Sub SSEPython
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import redis.asyncio as aioredis
import json, asyncio

app    = FastAPI()
redis  = aioredis.Redis(host="redis", decode_responses=True)

@app.get("/stream/{deploy_id}")
async def stream_logs(deploy_id: str, last_id: int = 0):
    """
    SSE endpoint. Returns text/event-stream.
    last_id: resume offset for reconnecting clients.
    """
    async def event_generator():
        # 1. Replay historical log lines for reconnect / new viewer
        history_key = f"deploy:{deploy_id}:logs"
        history = await redis.lrange(history_key, last_id, -1)
        for idx, line in enumerate(history, start=last_id):
            yield f"id:{idx}\ndata:{line}\n\n"
            await asyncio.sleep(0)   # yield control to event loop

        # 2. Subscribe to live updates via Redis Pub/Sub
        pubsub = redis.pubsub()
        await pubsub.subscribe(f"deploy:{deploy_id}")
        offset = last_id + len(history)

        async for message in pubsub.listen():
            if message["type"] == "message":
                data = message["data"]
                if data == "DONE":
                    yield f"event:done\ndata:deployment complete\n\n"
                    return
                yield f"id:{offset}\ndata:{data}\n\n"
                offset += 1

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",   # disable nginx buffering
        }
    )

@app.post("/log/{deploy_id}")
async def ingest_log(deploy_id: str, line: str):
    """Build agent calls this for each log line."""
    await redis.lpush(f"deploy:{deploy_id}:logs", line)   # persist
    await redis.publish(f"deploy:{deploy_id}", line)        # push to live watchers
    return {"ok": True}

LogStreamerController.java — Spring WebFlux SSEJava
@RestController
public class LogStreamerController {

    private final ReactiveRedisTemplate<String,String> redis;

    @GetMapping(value="/stream/{deployId}", produces=MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> streamLogs(
            @PathVariable String deployId,
            @RequestParam(defaultValue="0") int lastId) {

        String histKey = "deploy:" + deployId + ":logs";
        String channel = "deploy:" + deployId;

        // Replay history + stream live events
        Flux<String> history = redis.opsForList()
            .range(histKey, lastId, -1);

        Flux<String> live = redis.listenToChannel(channel)
            .map(Message::getMessage);

        return Flux.concat(history, live)
            .index()
            .map(t -> ServerSentEvent.<String>builder()
                .id(String.valueOf(lastId + t.getT1()))
                .data(t.getT2())
                .build());
    }

    @PostMapping("/log/{deployId}")
    public Mono<String> ingest(@PathVariable String deployId,
                                  @RequestBody String line) {
        String listKey = "deploy:" + deployId + ":logs";
        return redis.opsForList().leftPush(listKey, line)
            .then(redis.convertAndSend("deploy:" + deployId, line))
            .thenReturn("ok");
    }
}
Outcome
  • Sync vs async is a coupling decision — synchronous REST couples caller and callee; if callee is down, caller fails. Async (Kafka/queue) decouples them; callee processes when it can.
  • SSE is underrated — for server-to-client streaming, SSE beats WebSocket in simplicity: works through proxies, built-in reconnect with Last-Event-ID, plain HTTP.
  • WebSocket when bidirectional is needed — chat, collaborative editing, live gaming. Requires sticky sessions on load balancer. Stateful servers are harder to scale.
  • gRPC for internal service communication — 3–10× smaller payload, compile-time type safety, streaming support. Not for public APIs (browser support limited).
  • Redis Pub/Sub as a fan-out layer — when multiple SSE servers need to push events from one source, Redis Pub/Sub broadcasts to all subscribed instances. Simple and effective.
  • The Log Streamer pattern — write path (agent → Redis LPUSH + PUBLISH) and read path (browser → SSE server → Redis SUB + LRANGE) are independent. History for late joiners from LRANGE; live updates from Pub/Sub.
  • X-Accel-Buffering: no — always set this header to prevent Nginx from buffering your SSE stream. A common gotcha that makes SSE appear to not work in production behind a reverse proxy.
