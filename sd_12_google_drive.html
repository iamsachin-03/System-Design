🎙 Phase 1 — Why Chunked Uploads?
PHASE 1 · UPLOAD DESIGN
IR
INTERVIEWER
Why not just do a single HTTP POST to upload a file? Why do we need chunked uploads?
ME
YOU
Single HTTP POST has three fatal problems for large files:

1. No resumability: If the connection drops at byte 4.9 GB of a 5 GB file, the entire upload fails and starts over. On mobile or flaky connections, this makes large files impossible to upload reliably.

2. Server memory pressure: The server must hold the entire file in memory (or stream to disk) during the upload. With thousands of concurrent large uploads, this blows the server's RAM and disk I/O.

3. No parallelism: A single HTTP stream uses one TCP connection at bandwidth × RTT / 2 (bandwidth-delay product). Splitting into parallel chunks saturates available bandwidth much more efficiently.

Chunked upload design:
  1. Client splits file into 4 MB chunks (a sweet spot — large enough for efficient transfer, small enough to retry cheaply)
  2. Client computes SHA-256 of each chunk (content hash)
  3. Client asks server: "which of these chunk hashes have you already seen?" (block-level dedup check)
  4. Server returns: "I already have these 3 chunks" → client skips uploading them
  5. Client uploads only the missing chunks, potentially in parallel
  6. Client tells server: "assemble file from these chunk IDs in this order"
This design also gives us content-based deduplication for free — if two users upload the same file, the chunks are already stored and the second user's upload takes milliseconds.
💡 The chunk hash dedup check is the key insight that distinguishes a great answer. It's what makes Dropbox's "Dropbox Instant" feature work — same files upload instantly because chunks already exist.
🔄 Phase 2 — Delta Sync
PHASE 2 · DELTA SYNC
IR
INTERVIEWER
User edits a 500 MB PowerPoint and changes one slide. How does sync work efficiently?
ME
YOU
Without delta sync, editing one slide means re-uploading 500 MB — terrible. With delta sync, only the changed chunks are uploaded.

How the client detects changes:
The client maintains a local database of: {file_path → [chunk_hash_1, chunk_hash_2, ..., chunk_hash_N]}
When the file changes, the client re-chunks it with the same fixed 4 MB boundaries and recomputes hashes. It then diffs against the stored list:
  • Chunks whose hashes unchanged → skip (no upload needed)
  • Chunks whose hashes changed → upload only these
Editing one slide in a 500 MB PPTX changes perhaps 1-2 chunks (4-8 MB). The sync uploads 8 MB instead of 500 MB. 94× bandwidth savings.

Fixed vs content-aware chunking:
Fixed 4 MB boundaries have a problem: if you insert content in the middle of a file, all subsequent chunk boundaries shift, making all downstream chunks "changed" even though the content is the same. Content-defined chunking (Rabin fingerprinting) solves this by choosing boundaries based on the content itself — chunk boundaries stay stable across insertions. Dropbox and rsync use this. More complex to implement, but far better for text files that are edited frequently.
IR
INTERVIEWER
How does the sync client know a file has changed? Polling? File system events?
ME
YOU
The client uses the OS file system event API:
  • macOS: FSEvents API — kernel notifies the app when any file in the watched folder changes
  • Windows: ReadDirectoryChangesW — same concept
  • Linux: inotify — watches specific paths for modification events
This is event-driven, not polling. The OS notifies the sync client within milliseconds of a file change. The client then:
  1. Waits briefly (debounce 500ms) in case multiple writes happen in quick succession
  2. Re-chunks the changed file
  3. Diffs against stored chunk hashes
  4. Uploads only changed chunks
  5. Notifies the server of the new version
On mobile, the sync client runs as a background service. When the app is in the foreground or on WiFi, it syncs. Battery constraints mean mobile sync is less aggressive than desktop.
⚔️ Phase 3 — Conflict Resolution
PHASE 3 · CONFLICTS
IR
INTERVIEWER
User edits the same file on laptop and phone while offline. Both come back online. How do you handle the conflict?
ME
YOU
This is the hardest problem in sync systems. There's no perfect solution — only tradeoffs.

When does a conflict occur?
A conflict occurs when two clients modify the same file starting from the same base version. We detect this by comparing vector clocks or version numbers: if device A's version 5 diverges from device B's version 5, both were modified independently.

Three approaches:
1. Last-Write-Wins (Dropbox default):
The server has a timestamp. Whichever client's upload arrives first "wins" — the other client's changes are preserved as a conflicted copy: "document (John's conflicted copy 2026-03-23).docx". The user sees both files and manually merges.
Pro: Never loses data. Simple to implement.
Con: Forces manual merge on every conflict. Annoying UX.

2. Operational Transform (OT) / CRDT:
Transform concurrent operations to be commutative. Both edits are applied; the final result is the merge of both changes. Google Docs uses this for collaborative editing.
Pro: Seamless merge, no user intervention needed.
Con: Only works for text/structured documents, not binary files. Very complex to implement correctly.

3. Three-way merge:
Find the last common ancestor (LCA) version. Diff both modified versions against LCA. Auto-merge non-overlapping changes; flag overlapping changes as conflicts for user resolution.
Pro: Intelligent auto-merge for many cases.
Con: Requires LCA tracking, complex for binary files.

My recommendation for this system: Use Last-Write-Wins with conflicted copy for binary files (DOCX, XLSX, images). Use three-way merge for plain text files. This matches what Dropbox and Google Drive actually do.
🔔 Phase 4 — Cross-Device Sync via Long Polling or WebSocket
PHASE 4 · NOTIFICATION
IR
INTERVIEWER
How does Device B know to download an update when Device A uploads a file?
ME
YOU
Device B needs to be notified of changes without constant polling. Three options:

1. Short polling: Client polls GET /sync/changes?since={cursor} every 5 seconds. Simple but generates constant load even when nothing changes. 100M DAU × polling every 5s = 20M requests/sec constantly — wasteful.

2. Long polling (recommended for most cases): Client opens GET /sync/changes?since={cursor}. Server holds the connection open for up to 90s. If a change happens, server responds immediately. If 90s elapses without a change, server responds "no changes" and client immediately reconnects. Good balance: low latency, efficient for sparse changes.

3. WebSocket: Persistent bidirectional connection. Server pushes changes as they happen. Best latency, but: 100M concurrent WebSocket connections require a dedicated connection management tier (like Slack/WhatsApp). Heavy infra overhead. Justified when changes are frequent (collaborative editing), not for file sync where changes are rare.

My choice: Long polling for mobile, WebSocket for desktop web.
Mobile uses long polling (battery-friendly). Desktop web uses WebSocket (always-open app, better UX). Native desktop clients use long polling with OS-level connection management.
🔐 Phase 5 — Storage Architecture & Deduplication
PHASE 5 · STORAGE
IR
INTERVIEWER
How does content-based deduplication work across users? Does it raise privacy concerns?
ME
YOU
Content-based dedup: Each chunk is stored once using its SHA-256 hash as the key in S3. When a second user uploads the same file, their chunk list points to the same S3 objects — no new storage allocated.
S3 key: chunks/{sha256_hash}

The metadata database stores per-user file records that reference chunk IDs:
files(file_id, owner_id, name, version) → file_chunks(file_id, chunk_index, chunk_hash)

Privacy concern — Convergent Encryption:
Naïve dedup means: if you can hash the content of a copyrighted file, you can check if another user has that file without ever downloading it (hash collision attack). This is a real privacy concern.

Solution: Convergent Encryption. Before uploading, the client encrypts each chunk with a key derived from the chunk's own hash: key = HMAC(master_secret, chunk_hash). Two users encrypting the same file get the same ciphertext (because same hash → same key) — dedup still works. But a third party who knows the hash still can't decrypt the chunk without the master secret.

Cross-user dedup: Allowed for public content (files you've explicitly shared). For private files, dedup only within the same user account. This is the Dropbox model — cross-user dedup was controversial and they've walked it back.

🏗 High-Level Architecture

📤 UPLOAD FLOW (TWO-PHASE)
1
FS event fires: file modified. Client re-chunks + hashes.
2
POST /files/upload/init with chunk hashes. Server returns: which chunks to upload + pre-signed S3 URLs.
3
Client directly PUTs each missing chunk to S3 (parallel, bypassing app servers).
4
POST /files/upload/complete — server commits metadata, publishes file-event to Kafka.
5
Kafka → Change Notifier writes to Redis → wakes other devices' long-poll connections.
6
Other devices receive delta, download only changed chunks from CDN/S3.
🔄 CONFLICT RESOLUTION FLOW
1
Laptop (offline) edits file v3 → creates v4_laptop
2
Phone (offline) edits same file v3 → creates v4_phone
3
Laptop comes online first → uploads v4_laptop, server creates version 4
4
Phone comes online → uploads v4_phone, server detects base version mismatch (both derived from v3)
5
Server: v4_laptop wins (first to arrive). v4_phone saved as "report (conflicted copy 2026-03-23).docx"
6
User sees both files, manually merges or discards conflicted copy.

// ── 1. Upload Init — Dedup Check + Pre-signed URLs ────
@RestController
public class UploadController {

    @Autowired private ChunkManager         chunkMgr;
    @Autowired private FileMetadataService  metaSvc;
    @Autowired private QuotaService         quotaSvc;
    @Autowired private RedisTemplate<String,Object> redis;

    @PostMapping("/api/v1/files/upload/init")
    public UploadInitResponse initUpload(
            @RequestBody UploadInitRequest req,
            @AuthenticatedUser User user) {

        // 1. Check quota before starting
        if (!quotaSvc.checkAndReserve(user.getId(), req.getFileSize())) {
            throw new StorageQuotaExceededException();
        }

        // 2. Dedup check: which chunks already exist in S3?
        Set<String> existingHashes = chunkMgr.checkExisting(req.getChunkHashes());

        // 3. Generate pre-signed PUT URLs for missing chunks
        Map<Integer, URL> presignedUrls = new HashMap<>();
        List<Integer>    chunksNeeded  = new ArrayList<>();

        for (int i = 0; i < req.getChunkHashes().size(); i++) {
            String hash = req.getChunkHashes().get(i);
            if (!existingHashes.contains(hash)) {
                chunksNeeded.add(i);
                presignedUrls.put(i,
                    chunkMgr.generatePresignedUrl(hash, 7200)); // 2h TTL
            }
        }

        // 4. Create upload session in Redis
        String uploadId = UUID.randomUUID().toString();
        redis.opsForHash().putAll("upload:" + uploadId, Map.of(
            "file_name",    req.getFileName(),
            "parent_id",    req.getParentFolderId(),
            "user_id",      user.getId().toString(),
            "chunk_hashes", String.join(",", req.getChunkHashes()),
            "total_chunks", String.valueOf(req.getChunkHashes().size())
        ));
        redis.expire("upload:" + uploadId, Duration.ofHours(2));

        return new UploadInitResponse(uploadId, chunksNeeded, presignedUrls);
    }

    @PostMapping("/api/v1/files/upload/complete")
    public FileResponse completeUpload(
            @RequestBody UploadCompleteRequest req,
            @AuthenticatedUser User user) {

        // Retrieve upload session
        Map<Object,Object> session = redis.opsForHash()
            .entries("upload:" + req.getUploadId());
        if (session.isEmpty()) throw new UploadSessionExpiredException();

        List<String> chunkHashes = Arrays.asList(
            ((String) session.get("chunk_hashes")).split(","));

        // Determine if this is a new file or update
        String existingFileId = metaSvc.findByNameAndParent(
            (String) session.get("file_name"),
            (String) session.get("parent_id"),
            user.getId());

        if (existingFileId != null) {
            // Update existing file — check for conflict
            FileVersion latest = metaSvc.getLatestVersion(existingFileId);
            boolean conflict = !req.getBaseVersionId().equals(latest.getVersionId());

            if (conflict) {
                // Create conflicted copy instead
                String conflictName = makeConflictName(
                    (String) session.get("file_name"), user.getName());
                return metaSvc.createNewFile(conflictName,
                    (String) session.get("parent_id"),
                    user.getId(), chunkHashes, true); // isConflict=true
            }
            return metaSvc.createVersion(existingFileId, chunkHashes, user.getId());
        }
        return metaSvc.createNewFile(
            (String) session.get("file_name"),
            (String) session.get("parent_id"),
            user.getId(), chunkHashes, false);
    }
}

// ── 2. Sync Long-poll ─────────────────────────────────
@RestController
public class SyncController {

    @Autowired private SyncCursorManager cursorMgr;

    @GetMapping("/api/v1/sync/changes")
    public DeferredResult<DeltaResponse> getChanges(
            @RequestParam String cursor,
            @RequestParam(defaultValue = "90") int timeout,
            @AuthenticatedUser User user) {

        // Async result — suspends current thread, releases HTTP thread to pool
        DeferredResult<DeltaResponse> result =
            new DeferredResult<>((long) timeout * 1000);

        // Set timeout response
        result.onTimeout(() -> result.setResult(
            new DeltaResponse(false, cursor, List.of())));

        // Check for pending changes immediately
        DeltaResult pending = cursorMgr.getChanges(user.getId(), cursor);
        if (!pending.getEntries().isEmpty()) {
            result.setResult(new DeltaResponse(
                pending.hasMore(), pending.getNewCursor(), pending.getEntries()));
            return result;
        }

        // No changes yet — subscribe to Redis BRPOP for notification
        CompletableFuture.runAsync(() -> {
            try {
                // Block until Redis BRPOP delivers a change notification
                String changeKey = "changes:" + user.getId();
                var change = redis.opsForList()
                    .rightPop(changeKey, timeout, TimeUnit.SECONDS);

                if (change != null) {
                    DeltaResult delta = cursorMgr.getChanges(user.getId(), cursor);
                    result.setResult(new DeltaResponse(
                        delta.hasMore(), delta.getNewCursor(), delta.getEntries()));
                }
            } catch (Exception e) {
                result.setResult(new DeltaResponse(false, cursor, List.of()));
            }
        });

        return result;
    }
}

// ── 3. Client-side chunking (conceptual Java) ─────────
public class SyncClient {

    private static final int CHUNK_SIZE = 4 * 1024 * 1024; // 4 MB
    private final Map<Path, List<String>> chunkHashCache = new HashMap<>();

    public void onFileChanged(Path filePath) throws Exception {
        // 1. Chunk the file and compute hashes
        List<byte[]> chunks = chunkFile(filePath);
        List<String> hashes = chunks.stream()
            .map(this::sha256Hex)
            .collect(Collectors.toList());

        // 2. Delta: find only changed chunks vs cached hash list
        List<String> previousHashes = chunkHashCache.getOrDefault(
            filePath, List.of());
        Set<String> changedHashes = new HashSet<>(hashes);
        changedHashes.removeAll(new HashSet<>(previousHashes));

        // 3. Initialize upload session
        UploadInitResponse init = api.initUpload(
            filePath.getFileName().toString(),
            hashes,
            currentFolderId);

        // 4. Upload only missing chunks (parallel, 3 concurrent)
        ExecutorService pool = Executors.newFixedThreadPool(3);
        init.getChunksNeeded().forEach(idx ->
            pool.submit(() -> uploadChunkToS3(
                chunks.get(idx), init.getPresignedUrls().get(idx))));
        pool.shutdown();
        pool.awaitTermination(10, TimeUnit.MINUTES);

        // 5. Commit upload
        api.completeUpload(init.getUploadId());

        // 6. Update local chunk hash cache
        chunkHashCache.put(filePath, hashes);
    }

    private List<byte[]> chunkFile(Path path) throws Exception {
        List<byte[]> chunks = new ArrayList<>();
        try (FileInputStream fis = new FileInputStream(path.toFile())) {
            byte[] buffer = new byte[CHUNK_SIZE];
            int bytesRead;
            while ((bytesRead = fis.read(buffer)) > 0) {
                chunks.add(Arrays.copyOf(buffer, bytesRead));
            }
        }
        return chunks;
    }
}

import hashlib, os, math, concurrent.futures
from typing import List, Dict, Optional, Tuple

CHUNK_SIZE = 4 * 1024 * 1024  # 4 MB


# ── 1. Client-side Chunker ────────────────────────────
class FileChunker:
    """
    Splits files into 4 MB chunks and computes SHA-256 hashes.
    Maintains a local cache of chunk hashes per file for delta sync.
    """

    def __init__(self, chunk_size: int = CHUNK_SIZE):
        self.chunk_size  = chunk_size
        self._hash_cache: Dict[str, List[str]] = {}  # path → [hashes]

    def chunk_file(self, file_path: str) -> List[Tuple[int, bytes, str]]:
        """Returns list of (index, chunk_data, sha256_hash) tuples."""
        chunks = []
        with open(file_path, 'rb') as f:
            idx = 0
            while True:
                data = f.read(self.chunk_size)
                if not data:
                    break
                h = hashlib.sha256(data).hexdigest()
                chunks.append((idx, data, h))
                idx += 1
        return chunks

    def get_changed_chunks(self, file_path: str) -> List[Tuple[int, bytes, str]]:
        """
        Returns only chunks whose hashes differ from the cached version.
        This is the heart of delta sync — only upload what changed.
        """
        all_chunks    = self.chunk_file(file_path)
        prev_hashes   = self._hash_cache.get(file_path, [])
        changed       = []

        for idx, data, h in all_chunks:
            if idx >= len(prev_hashes) or prev_hashes[idx] != h:
                changed.append((idx, data, h))  # new or modified

        self._hash_cache[file_path] = [h for _, _, h in all_chunks]
        return changed

    def all_hashes(self, file_path: str) -> List[str]:
        """All chunk hashes for a file (for upload init)."""
        return [h for _, _, h in self.chunk_file(file_path)]


# ── 2. Upload Client ──────────────────────────────────
class DriveUploadClient:
    """
    Handles two-phase chunked upload:
    1. Init: declare upload, get pre-signed URLs for missing chunks
    2. Upload: PUT chunks directly to S3 in parallel
    3. Complete: commit metadata to API
    """

    def __init__(self, api_client, s3_session):
        self.api = api_client
        self.s3  = s3_session
        self.chunker = FileChunker()

    async def upload_file(self, file_path: str, parent_folder_id: str,
                             base_version_id: str = None) -> dict:
        """Upload or update a file via two-phase chunked upload."""

        file_name = os.path.basename(file_path)
        all_hashes = self.chunker.all_hashes(file_path)

        # Phase 1: Init — server tells us which chunks to upload
        init_resp = await self.api.post('/files/upload/init', {
            'file_name':      file_name,
            'file_size':      os.path.getsize(file_path),
            'parent_folder_id': parent_folder_id,
            'chunk_hashes':   all_hashes,
        })

        upload_id      = init_resp['upload_id']
        chunks_needed  = set(init_resp['chunks_needed'])
        presigned_urls = init_resp['presigned_urls']  # {idx: url}

        if not chunks_needed:
            print(f"⚡ Instant upload: all {len(all_hashes)} chunks already in S3!")
        else:
            print(f"Uploading {len(chunks_needed)}/{len(all_hashes)} chunks")

        # Phase 2: Upload only missing chunks to S3 (parallel)
        chunks = {idx: (data, h)
                  for idx, data, h in self.chunker.chunk_file(file_path)
                  if idx in chunks_needed}

        etags = {}
        with concurrent.futures.ThreadPoolExecutor(max_workers=4) as pool:
            futures = {
                pool.submit(self._upload_chunk_to_s3,
                            presigned_urls[str(idx)], data): idx
                for idx, (data, _) in chunks.items()
            }
            for future in concurrent.futures.as_completed(futures):
                idx = futures[future]
                etag = future.result()
                etags[idx] = etag

        # Phase 3: Complete — commit metadata
        resp = await self.api.post('/files/upload/complete', {
            'upload_id':       upload_id,
            'chunk_etags':     etags,
            'base_version_id': base_version_id,
        })
        return resp  # contains file_id, version_id

    def _upload_chunk_to_s3(self, presigned_url: str, data: bytes) -> str:
        """PUT chunk directly to S3. Returns ETag."""
        import requests
        resp = requests.put(presigned_url, data=data,
                            headers={'Content-Type': 'application/octet-stream'})
        resp.raise_for_status()
        return resp.headers.get('ETag', '')

import asyncio, aiohttp, watchdog.observers, watchdog.events

class SyncClient:
    """
    Watches local folder via OS file system events.
    Polls for remote changes via long-polling.
    """

    def __init__(self, watch_dir: str, api_client):
        self.watch_dir = watch_dir
        self.api       = api_client
        self.uploader  = DriveUploadClient(api_client, None)
        self.cursor    = "initial"
        self._debounce: Dict[str, asyncio.Task] = {}

    def start(self):
        # Watch local folder for changes (OS events)
        handler  = LocalChangeHandler(self)
        observer = watchdog.observers.Observer()
        observer.schedule(handler, self.watch_dir, recursive=True)
        observer.start()

        # Start long-poll loop for remote changes
        asyncio.run(self._poll_remote_changes())

    async def on_local_change(self, path: str):
        """Called when OS detects a local file change. Debounced."""
        if path in self._debounce:
            self._debounce[path].cancel()

        async def debounced_upload():
            await asyncio.sleep(0.5)  # wait 500ms for burst of writes to settle
            try:
                await self.uploader.upload_file(path, parent_folder_id="root")
                print(f"✅ Synced: {os.path.basename(path)}")
            except Exception as e:
                print(f"❌ Sync failed: {e}")

        self._debounce[path] = asyncio.create_task(debounced_upload())

    async def _poll_remote_changes(self):
        """Long-poll for remote changes, apply them locally."""
        async with aiohttp.ClientSession() as session:
            while True:
                try:
                    resp = await self.api.get('/sync/changes', params={
                        'cursor': self.cursor, 'timeout': 90
                    })
                    for entry in resp['entries']:
                        await self._apply_remote_change(entry)

                    self.cursor = resp['cursor']

                    if not resp['has_more']:
                        await asyncio.sleep(0.1)  # brief pause before reconnect

                except Exception as e:
                    print(f"Poll error: {e}")
                    await asyncio.sleep(5)  # backoff on error

    async def _apply_remote_change(self, entry: dict):
        """Download and apply a remote file change."""
        change_type = entry['type']
        local_path  = os.path.join(self.watch_dir, entry['path'])

        if change_type == 'deleted':
            if os.path.exists(local_path):
                os.remove(local_path)
        elif change_type in ('modified', 'added'):
            download_url = await self.api.get_download_url(entry['file_id'])
            await self._download_file(download_url, local_path)
        print(f"🔄 Applied remote {change_type}: {entry['path']}")
⚖️ Design Tradeoffs
File sync is one of the hardest distributed systems problems because it must handle concurrent offline modifications across devices while maintaining user trust that no data is ever lost.
📦 Chunk Size: 4 MB vs Smaller vs Larger
✅ 4 MB (standard)

Sweet spot for most workloads. S3 minimum multipart part size is 5 MB. 4 MB gives good parallelism (a 40 MB file = 10 chunks, 10 parallel streams). Retry cost on failure is modest. Dropbox uses 4 MB.

⚠️ 256 KB (too small)

Each chunk = one S3 API call. Uploading a 1 GB file: 4,096 API calls. At $0.005/1K S3 PUT requests, small chunks cost 20× more. Also: more network round trips = worse throughput on high-latency connections.

⚠️ 64 MB (too large)

If a 64 MB chunk upload fails at byte 63.9 MB, you retry 64 MB. On mobile (frequent connection drops), this is brutal. Also, delta sync granularity suffers — a 1-byte change in the middle of a file means re-uploading 64 MB.

⚔️ Conflict Resolution: Last-Write-Wins vs CRDT vs 3-Way Merge
✅ Last-Write-Wins + Conflicted Copy

First upload wins, second creates a conflicted copy. Simple to implement. No data loss. Users see both versions. Annoying but predictable. Standard for Dropbox. Works for all file types.

⚠️ CRDT / OT

Conflict-free replicated data types auto-merge concurrent changes. Works beautifully for text (Google Docs). But: only possible for structured/text data, not binary (images, XLSX, PPTX). Requires understanding of the file format internals. Not universally applicable.

💡 3-Way Merge

Find common ancestor, diff both branches against it, auto-merge non-overlapping changes. Git uses this for code. Works for text files. For binary, merging at the byte level is meaningless. Requires LCA tracking. Git LFS uses this pattern for large binary files.

🔔 Sync Notification: Long-Poll vs WebSocket vs SSE
✅ Long-Polling (90s timeout)

Simple, works through firewalls/proxies, no special infrastructure. Each poll connection holds for 90s max. At 100M active clients, 100M concurrent HTTP connections needed — manageable with async servers (Netty, Node.js). Dropbox uses long-polling.

⚠️ WebSocket

True push, lowest latency. But: 100M persistent WebSocket connections require a dedicated connection server tier. More complex infra. Only justified if sync latency below 1s is required. Google Drive uses WebSocket for collaborative editing, long-polling for Drive sync.

💡 Server-Sent Events (SSE)

Unidirectional push from server to client. Simpler than WebSocket (one-way only). Works over HTTP/1.1. Client reconnects automatically. Good alternative to long-polling when you don't need client→server messages on the sync channel. Not widely supported on mobile.

🔑 Chunking Algorithm: Fixed vs Content-Defined (Rabin Fingerprinting)
✅ Fixed 4 MB chunks (simpler)

Simple, deterministic. Good for appends (new chunks at end don't affect earlier chunks). For files that are only appended (log files, database exports), fixed chunking is perfect. Dropbox uses fixed chunking with good results.

⚠️ Content-Defined (Rabin)

Boundaries based on content hash rolling window. Insertions at position N don't shift all subsequent chunk boundaries. Far better delta sync for text files edited in the middle. rsync, bup, Perforce use Rabin. More CPU on client. Google Drive reportedly uses variable chunking.

💡 Which to use when

Fixed: video/binary files (appended, not edited); Initial implementation; Mobile clients (lower CPU). Content-defined: text/document files heavily edited; Desktop clients; Maximum dedup across edits. Many systems use fixed chunking first, migrate to content-defined later when edit patterns emerge.
