S3 (Simple Storage Service) stores objects (files) identified by bucket + key, at any scale, with 11 9s of durability. Design the storage backend: how data is stored, replicated, retrieved, and guaranteed never to be lost.

What Makes 11 Nines of Durability So Hard?

11 nines math: 99.999999999% durability for 10 billion objects = you expect to lose at most 1 object in 100,000 years. A single hard drive fails ~1–4% per year. To get 11 nines, data must survive: disk failure, server failure, rack failure, datacenter failure, and even region failure — simultaneously. This requires at minimum 3× replication across 3 geographically separate availability zones.

Chunked Storage for Large Objects

A 5 TB file cannot be stored on a single disk. S3 splits large objects into chunks (e.g., 64 MB each). Each chunk is stored independently across storage nodes. The metadata service tracks chunk_ids → storage node locations. On read: fetch chunks in parallel, reassemble on client.

PUT Object Flow

  • Client → API: PUT /bucket/key
  • API → Metadata: allocate chunk_ids for this object
  • API splits object → N chunks
  • Each chunk written to 3 storage nodes (different AZs)
  • API waits for 2/3 AZ confirms → ACK to client
  • Metadata updated: key → chunk_id list

GET Object Flow

  • Client → API: GET /bucket/key
  • API → Metadata: get chunk_ids for this key
  • API → Storage Nodes: fetch chunks (parallel)
  • Verify checksums (detect bit rot)
  • Reassemble and stream to client
  • Heal corrupt chunk from replica if detected
Durability & Replication

How S3 Achieves 11 Nines of Durability

1
Erasure Coding within each AZ
Instead of storing 3 full copies (300% storage overhead), S3 uses Reed-Solomon erasure coding. A 6+3 scheme: split data into 6 shards + 3 parity shards. Any 6 of 9 shards can reconstruct the original. Overhead: only 50% (vs 200% for 3× replication).
2
Cross-AZ synchronous replication
Each object is written to 3 availability zones before acknowledging success. AZs are physically separate datacenters in the same region, each with independent power and networking. If one AZ floods or burns down, 2 others have the data.
3
Background data integrity checks
S3 continuously reads stored data, verifies checksums, and automatically heals any corruption detected by re-reading from another replica and rewriting. This catches silent disk corruption ("bit rot") that would otherwise go undetected.
4
Cross-region replication (optional)
S3 Cross-Region Replication (CRR) asynchronously copies objects to another region. Protects against region-level disasters. Adds cost and write latency. Used for disaster recovery and compliance.

Erasure Coding Deep Dive

Reed-Solomon (6+3)

Divide 12 GB object into 6 × 2 GB data shards. Generate 3 × 2 GB parity shards using polynomial arithmetic. Store each of 9 shards on a different disk/node. Any 6 of 9 reconstruct original. Tolerates loss of any 3 shards.

Storage overhead: 9/6 = 1.5× (vs 3× for full replication)

Why Not Just 3× Replication?

3× replication is simpler but wastes 200% storage overhead. At petabyte scale, the cost difference is enormous. S3's actual durability comes from the combination of erasure coding + cross-AZ replication + continuous integrity verification, not from brute-force copying.

Consistency & Versioning

S3's Consistency Model

Strong Read-after-Write Consistency (2020+)

Since Dec 2020, S3 offers strong consistency for all operations. PUT → immediate GET returns the new object. LIST → reflects all completed PUTs. Previously, S3 had eventual consistency for overwrites and LIST operations — this caused real production bugs.

How Strong Consistency is Implemented

S3's metadata service (a distributed KV store) is the single source of truth. Any GET goes through the same metadata service that processed the PUT. No stale reads from replicas. The metadata service uses Paxos/Raft for consistency across its nodes.

Multipart Upload

For large objects (>100 MB), multipart upload allows parallel chunk uploads. Each part is uploaded independently and tracked. Only when all parts are uploaded does the final object become visible. Resumable on failure.

multipart_upload.py — parallel chunk uploadPython
import asyncio, boto3
s3 = boto3.client("s3")

async def multipart_upload(bucket: str, key: str, filepath: str,
                            chunk_size: int = 64 * 1024 * 1024):  # 64 MB chunks
    """Upload large file in parallel chunks."""
    # 1. Initiate multipart upload → get upload_id
    resp      = s3.create_multipart_upload(Bucket=bucket, Key=key)
    upload_id = resp["UploadId"]
    parts     = []

    try:
        # 2. Upload each chunk in parallel
        with open(filepath, "rb") as f:
            chunk_num = 1
            while True:
                chunk = f.read(chunk_size)
                if not chunk: break
                r = s3.upload_part(
                    Bucket=bucket, Key=key, PartNumber=chunk_num,
                    UploadId=upload_id, Body=chunk
                )
                parts.append({"PartNumber": chunk_num, "ETag": r["ETag"]})
                chunk_num += 1

        # 3. Complete → object becomes visible atomically
        s3.complete_multipart_upload(
            Bucket=bucket, Key=key, UploadId=upload_id,
            MultipartUpload={"Parts": parts}
        )
    except Exception:
        s3.abort_multipart_upload(Bucket=bucket, Key=key, UploadId=upload_id)
        raise
Code — S3-like Object Store Implementation
object_store.py — minimal S3-compatible object storePython
import os, hashlib, json, time
from fastapi import FastAPI, UploadFile, HTTPException
from fastapi.responses import StreamingResponse

app      = FastAPI()
DATA_DIR = "/data/objects"
META_DIR = "/data/metadata"

def _chunk_key(bucket: str, key: str) -> str:
    """Deterministic storage path — flat namespace avoids directory limits."""
    h = hashlib.sha256(f"{bucket}/{key}".encode()).hexdigest()
    return f"{h[:2]}/{h[2:4]}/{h}"   # 2-level prefix avoids inode hotspot

@app.put("/{bucket}/{key:path}")
async def put_object(bucket: str, key: str, file: UploadFile):
    """Upload object. Returns ETag (MD5 of content)."""
    content  = await file.read()
    etag     = hashlib.md5(content).hexdigest()
    ck       = _chunk_key(bucket, key)
    path     = f"{DATA_DIR}/{ck}"

    os.makedirs(os.path.dirname(path), exist_ok=True)
    with open(path, "wb") as f: f.write(content)

    # Store metadata (key → chunk path + etag + size)
    meta_path = f"{META_DIR}/{bucket}/{key}.json"
    os.makedirs(os.path.dirname(meta_path), exist_ok=True)
    with open(meta_path, "w") as f:
        json.dump({"chunk_key":ck, "etag":etag,
                   "size":len(content), "created_at":int(time.time())}, f)

    return {"ETag": etag, "size": len(content)}

@app.get("/{bucket}/{key:path}")
async def get_object(bucket: str, key: str):
    """Download object. Verify ETag on serve."""
    meta_path = f"{META_DIR}/{bucket}/{key}.json"
    if not os.path.exists(meta_path):
        raise HTTPException(404, "NoSuchKey")

    with open(meta_path) as f: meta = json.load(f)
    data_path = f"{DATA_DIR}/{meta['chunk_key']}"

    def stream_file():
        with open(data_path, "rb") as f:
            while chunk := f.read(65536): yield chunk

    return StreamingResponse(
        stream_file(),
        headers={"ETag": meta["etag"], "Content-Length": str(meta["size"])}
    )

Why Orders Are Hard

Distributed transactions

Order creation requires: reserve inventory + charge payment + create fulfillment record — all atomically. Across separate microservices. 2PC is slow and fragile. SAGA pattern is the solution.

Concurrency

1000 users ordering the last iPhone simultaneously. Inventory must be consistent — no overselling. But also no deadlocks that crash the checkout flow for everyone else.

Idempotency

Network failures mean requests may be retried. Charging a card twice is catastrophic. Every order operation must be idempotent — same input → same result, regardless of how many times it's called.

Multi-tier Architecture

The SAGA Pattern for Distributed Order Processing

Instead of 2-phase commit (synchronous, fragile), use choreography-based SAGA: each service does its local transaction and publishes an event. If any step fails, compensating transactions roll back previous steps.

1
Order Service: CREATE order record (PENDING)
Persist order to DB. Publish order.created event to Kafka. Return order_id to client immediately — no waiting for payment or inventory.
2
Inventory Service: Reserve stock
Consumes order.created. Decrements inventory (SELECT FOR UPDATE). Success → publish inventory.reserved. Failure (out of stock) → publish inventory.reserve_failed → triggers order cancellation.
3
Payment Service: Charge card
Consumes inventory.reserved. Charges payment method. Success → publish payment.charged. Failure → publish payment.failed → compensating: release inventory reservation.
4
Fulfillment Service: Create shipment
Consumes payment.charged. Creates fulfillment record, assigns to warehouse. Publishes fulfillment.scheduled.
5
Notification Service: Confirm to customer
Sends order confirmation email/SMS. Order is now in CONFIRMED state. Each subsequent state change (shipped, delivered) repeats this notification pattern.

Compensating Transactions (Rollback)

Payment fails after inventory reserved

Compensating transaction: publish inventory.release event → Inventory Service releases the reservation. Order moves to CANCELLED state. Customer notified.

Fulfillment fails after payment charged

Compensating transaction: refund payment → release inventory. These compensating transactions must also be idempotent — network failures may cause them to be retried.

SAGA vs 2PC: 2-Phase Commit holds distributed locks for the duration of the entire transaction — seconds with network roundtrips. SAGA has no distributed locks; each service commits immediately and compensates on failure. 10–100× better throughput for order processing.
Concurrency & Inventory Management

The Last-Item Race Condition

1000 users simultaneously try to buy the last iPhone. All read inventory = 1. All try to decrement. Without proper concurrency control: all succeed and you've sold 1000 iPhones you don't have.

Solution 1: Optimistic Locking (high concurrency, low conflict)

inventory_optimistic.py — version-based CASPython
async def reserve_inventory_optimistic(db, product_id: int, qty: int,
                                          max_retries: int = 3) -> bool:
    """Try to reserve qty units. Retry on conflict (optimistic lock)."""
    for attempt in range(max_retries):
        # 1. Read current state with version
        row = await db.fetchrow(
            "SELECT available_qty, version FROM inventory WHERE product_id=$1",
            product_id
        )
        if row["available_qty"] < qty:
            return False   # not enough stock

        # 2. Try to update with version check (CAS)
        updated = await db.execute(
            "UPDATE inventory SET available_qty=available_qty-$1, version=version+1 "
            "WHERE product_id=$2 AND version=$3 AND available_qty>=$1",
            qty, product_id, row["version"]
        )
        if updated:
            return True   # successfully reserved
        # Version mismatch = concurrent update; retry
    return False   # gave up after retries

Solution 2: Pessimistic Locking (high contention scenarios)

inventory_pessimistic.py — SELECT FOR UPDATEPython
async def reserve_inventory_pessimistic(db, product_id: int, qty: int) -> bool:
    """Lock row during reservation — guaranteed consistency, lower throughput."""
    async with db.transaction():
        row = await db.fetchrow(
            "SELECT available_qty FROM inventory WHERE product_id=$1 FOR UPDATE",
            product_id
        )
        if row["available_qty"] < qty:
            return False
        await db.execute(
            "UPDATE inventory SET available_qty=available_qty-$1 WHERE product_id=$2",
            qty, product_id
        )
        return True

Solution 3: Redis Atomic DECRBY (fastest)

Redis DECRBY for inventory — lock-free, O(1)Redis + Python
async def reserve_inventory_redis(redis, product_id: int, qty: int) -> bool:
    """
    Redis DECRBY is atomic — no locking needed.
    If result < 0: too many reservations, undo and return False.
    Use Lua script for atomicity of check + decrement.
    """
    script = """
    local current = redis.call('GET', KEYS[1])
    if not current or tonumber(current) < tonumber(ARGV[1]) then
        return 0
    end
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 1
    """
    result = await redis.eval(script, 1, f"inventory:{product_id}", qty)
    return bool(result)

    # DB is the persistent record

Idempotency Keys

Pattern: Client sends Idempotency-Key: uuid-v4 header. Server stores {idempotency_key → response} in Redis for 24h. If same key arrives again (retry): return stored response without re-executing. This prevents duplicate charges, duplicate orders, duplicate emails.

order_service.py — SAGA orchestrationPython
import uuid, time
from fastapi import FastAPI, Header, HTTPException

app = FastAPI()

@app.post("/orders")
async def create_order(
    cart: Cart,
    user_id: int,
    idempotency_key: str = Header(None)
):
    """
    Create order and kick off SAGA.
    Returns immediately — processing is async.
    """
    # 1. Check idempotency
    if idempotency_key:
        cached = await redis.get(f"idem:{idempotency_key}")
        if cached:
            return json.loads(cached)   # idempotent: return previous result

    # 2. Create order record (PENDING)
    order_id = str(uuid.uuid4())
    now      = int(time.time() * 1000)
    await db.execute(
        "INSERT INTO orders(id,user_id,status,cart_json,created_at) "
        "VALUES($1,$2,'PENDING',$3,$4)",
        order_id, user_id, cart.json(), now
    )

    # 3. Publish to Kafka → triggers SAGA chain
    await kafka.publish("order.created", {
        "order_id": order_id,
        "user_id":  user_id,
        "items":    cart.items,
        "idempotency_key": idempotency_key
    })

    response = {"order_id": order_id, "status": "PENDING"}

    # 4. Cache idempotent result
    if idempotency_key:
        await redis.setex(f"idem:{idempotency_key}", 86400, json.dumps(response))

    return response

LSM Trees Ground Up

Week 5 introduced LSM trees as a storage engine. Week 6 goes deeper: building one from scratch, measuring performance characteristics, and understanding how RocksDB — the world's most widely deployed LSM engine — tunes the fundamental trade-offs.

The Core Design Decision: Sequential vs Random I/O

B-Tree: Random I/O dominant

Every INSERT may trigger a page split → read page → modify → write page. Random write at an arbitrary disk offset. Modern NVMe: random write ~500K IOPS at 4KB. Sequential write: ~3M IOPS. B-tree uses random: 6× slower.

LSM: Sequential I/O by design

All writes go to MemTable (in-memory). When full, flush to disk in one sequential write. Disk sees only large sequential writes — no random seeks. Can sustain 500K–5M rows/sec on modern hardware.

Building the Minimal Viable LSM

minimal_lsm.py — complete minimal implementationPython
import os, json, bisect, time, threading
from pathlib import Path

TOMBSTONE = "__DELETED__"

class SkipListMemTable:
    """
    Simplified MemTable using sorted dict (conceptually a Skip List).
    Real implementations use Skip List for O(log N) concurrent reads.
    """
    def __init__(self, max_size: int = 1000):
        self._data    = {}        # in production: sorted structure
        self.max_size = max_size
        self.size     = 0

    def put(self, key: str, value: str) -> bool:
        self._data[key] = (value, time.time_ns())  # (value, timestamp)
        self.size += 1
        return self.size >= self.max_size   # True = needs flush

    def get(self, key: str):
        entry = self._data.get(key)
        return entry[0] if entry else None

    def to_sorted_pairs(self):
        """Return entries sorted by key (for SSTable write)."""
        return sorted(self._data.items())

class SSTableFile:
    """Immutable sorted file on disk with Bloom filter."""
    def __init__(self, path: str, entries: list):
        self.path     = path
        self.min_key  = entries[0][0]   if entries else None
        self.max_key  = entries[-1][0]  if entries else None
        self._bloom   = set(k for k, _ in entries)  # simplified bloom
        # Write sorted entries to disk
        with open(path, "w") as f:
            for key, (val, ts) in entries:
                f.write(f"{key}\t{val}\t{ts}\n")  # sequential write!

    def get(self, key: str) -> str | None:
        if key not in self._bloom: return None  # Bloom filter skip
        if key < self.min_key or key > self.max_key: return None
        with open(self.path) as f:
            for line in f:
                k, v, _ = line.strip().split("\t")
                if k == key: return v
                if k  > key: return None
        return None

class LSMEngine:
    def __init__(self, data_dir: str = "/tmp/lsm_engine"):
        self.dir      = Path(data_dir); self.dir.mkdir(parents=True, exist_ok=True)
        self.memtable = SkipListMemTable()
        self.sstables = []   # newest first
        self._lock    = threading.Lock()
        self._seq     = 0
        self._open_wal()

    def put(self, key: str, value: str):
        with self._lock:
            self._wal.write(f"PUT\t{key}\t{value}\n"); self._wal.flush()
            if self.memtable.put(key, value):
                self._flush()

    def get(self, key: str) -> str | None:
        val = self.memtable.get(key)
        if val is not None:
            return None if val == TOMBSTONE else val
        for sst in self.sstables:
            val = sst.get(key)
            if val is not None:
                return None if val == TOMBSTONE else val
        return None

    def _flush(self):
        entries  = self.memtable.to_sorted_pairs()
        self._seq += 1
        sst_path = str(self.dir / f"sst_{self._seq:06d}.sst")
        self.sstables.insert(0, SSTableFile(sst_path, entries))
        self.memtable = SkipListMemTable()   # fresh MemTable
        if len(self.sstables) > 5:            # compact threshold
            self._compact()
Performance Analysis

Amplification Factors — The Core Trade-offs

Write Amplification (WA)

Data written to disk ÷ data user wrote. A single key-value pair gets written to: WAL (1×), MemTable flush (1×), L0→L1 compaction (10×), L1→L2 (10×), … = 30–50× for leveled compaction.

High WA accelerates SSD wear. At 10× WA, a 1 TB SSD rated for 1 PB total writes becomes effectively 100 TB capacity.

Read Amplification (RA)

Reads required to satisfy one user read. Worst case (missing key): check MemTable + all SSTable Bloom filters + all SSTable data. With Bloom filters: typically 1–2× disk reads for existing keys, 0 disk reads for Bloom-filtered misses.

Space Amplification (SA)

Disk space used ÷ unique data. During compaction: old + new versions coexist. Worst case: 2× during active compaction. Size-tiered: can reach 2–10× SA. Leveled: lower SA but higher WA.

The RUM Conjecture

RUM Conjecture: You cannot simultaneously optimize for Read overhead (RA), Update overhead (WA), and Memory overhead (SA). Every storage engine makes a specific trade-off among the three. LSM optimizes for WA at the cost of RA and SA. B-tree optimizes for RA at the cost of WA.

Block Cache

Block cache is RocksDB's in-memory LRU cache for frequently accessed SSTable data blocks. Setting it to 50% of available RAM dramatically improves read performance — hot keys are served from memory instead of disk. Configure with: options.block_cache = NewLRUCache(4 * 1024 * 1024 * 1024LL) (4 GB).
Bloom Filters in LSM Trees

Bloom filters are the single most important optimization in an LSM engine. Without them, every read for a missing key must scan all SSTables — O(N) disk reads. With them: one hash computation per SSTable, almost never touching disk for misses.

Bloom Filter Per SSTable

Why One Per SSTable?

Each SSTable is immutable and has a fixed key set. Its Bloom filter is built at creation time and never needs updating. Loaded into memory when the SSTable is opened. A typical SSTable with 1M keys has a Bloom filter of ~1.2 MB (10 bits per key).

Read Path with Bloom

For a GET(key): For each SSTable (newest first):
1. Check Bloom filter → is "NO" → skip entire SSTable (no disk I/O)
2. If "MAYBE" → check SSTable's key range (metadata)
3. If in range → binary search index → fetch data block

Result: 99%+ of misses touch zero SSTables on disk.

Memory trade-off: For 1B keys, increasing from 10 to 20 bits per key = 1.25 GB extra RAM for the Bloom filter. This eliminates ~9 unnecessary disk reads per 1000 GETs. At 100K GETs/sec, that's 900 fewer disk reads/sec → worth it if memory is available.

Systems Using LSM Trees

RocksDB (Meta/Facebook)

Powers Kafka's log storage, TiKV (TiDB's storage engine), CockroachDB, MyRocks (MySQL with LSM), Yugabyte. The industry-standard LSM implementation.

Cassandra / HBase

Write-optimized NoSQL. Cassandra: social graph, IoT, time-series at Meta/Netflix. HBase: Hadoop-based big data analytics. Both use LSM with cross-node replication.

Kafka

Kafka's log is essentially a simple LSM: append-only segments (SSTables), compaction via log compaction, sequential I/O. Not a general KV store but uses the same fundamental design principles.

Outcome
  • LSM converts random writes to sequential writes — the single insight that makes it 10× faster than B-trees for write-heavy workloads. Every write is a MemTable insert + sequential disk flush. No random seeks.
  • The RUM Conjecture frames all storage engine trade-offs — Read / Update / Memory. No engine optimizes all three. LSM chooses write (update) at the cost of read and space. B-tree chooses read at the cost of write.
  • Bloom filters eliminate almost all unnecessary disk reads — for missing keys (99%+ of cache misses), Bloom filter answers in microseconds without touching disk. This is the biggest read optimization in LSM.
  • Write amplification is the LSM tax on SSDs — 30–50× WA for leveled compaction. At 10 TB/day writes, that's 300–500 TB/day of actual disk writes. SSD endurance must be budgeted for.
  • RocksDB is the reference implementation — used by Kafka, TiKV, CockroachDB, Meta's social graph. Understanding RocksDB tuning parameters (write_buffer_size, bloom_bits, compaction triggers) is production-critical knowledge.

Why Live Ads Are Hard

Real-time constraint

In VOD: pre-process the video offline, insert ads at transcoding time. In Live: no pre-processing possible. Must detect the ad opportunity, run an auction, fetch the ad creative, and inject into the stream — all while the stream is ongoing.

Scale

Super Bowl: 100M+ concurrent viewers. Every viewer needs a (potentially different) ad — targeted advertising. 100M simultaneous ad decisions in <500ms = 200M QPS on ad servers. Requires massive caching and pre-warming.

Viewer experience

A buffering pause during an ad break = viewer churns. The ad system must serve the ad faster than the viewer's buffer depletes. At typical 10s buffer: system has 10s to complete ad insert. But target is <3s to feel seamless.

Ad Insertion Architecture

SCTE-35 — The Ad Opportunity Signal

SCTE-35 is the broadcast industry standard for signaling ad insertion opportunities in a live video stream. The encoder/broadcaster inserts a SCTE-35 marker into the MPEG-TS stream at ad break points. The ad system watches for these markers.

1
SCTE-35 marker detected in stream
CDN or packager detects the SCTE-35 splice_insert or time_signal command. Includes: duration (e.g., 30 seconds), avail_num (break number), unique_program_id. Published to message bus immediately.
2
Ad Decision Service (ADS) called
Within 100ms of SCTE-35 detection, ADS receives the signal. Runs targeting logic: user demographics, content context, geo, device. Calls ad exchange for real-time bidding or uses pre-reserved ads.
3
Ad creative pre-fetched and transcoded
The winning ad creative must be available in the correct format (HLS, DASH), bitrate ladder, and segment timing. Pre-transcoding ad inventory is critical — on-demand transcoding adds 5–30 seconds of latency.
4
Server-Side Ad Insertion (SSAI)
The CDN/packager seamlessly splices the ad segments into the live stream manifest (HLS playlist). The viewer's player sees a continuous stream — no separate ad request, no CORS issues, no ad blocker avoidance.
5
Impression tracking & billing
The packager marks ad segments with tracking URLs. When player downloads the segment, the tracking server records the impression. Used for billing and frequency capping. Must be resilient to player crashes mid-ad.

SSAI vs Client-Side Ad Insertion (CSAI)

SSAI (Server-Side)

  • Ad blocker proof — same CDN domain as content
  • Seamless playback — single bitrate adaptive stream
  • No client-side SDK needed
  • Works on all devices (smart TVs, set-top boxes)
  • Easier to target CTV/OTT audiences
  • ❌ Complex server infrastructure

CSAI (Client-Side)

  • Simple: client loads ad from ad server separately
  • Flexible: A/B test ad experiences easily
  • ❌ Ad blockers can block it
  • ❌ Buffering between content and ad (URL switch)
  • ❌ Doesn't work well on CTV/smart TV SDK limitations
  • Standard for desktop web, mobile apps

Targeting Signals

  • User signals: age, gender, interests, watch history, subscription tier
  • Content signals: sport type, team, league, tournament, moment (goal scored)

@dataclass

async def make_ad_decision(opp: AdOpportunity) -> list[dict]:
    """
    Select ads for a break. Must complete in < 200ms.
    Returns ordered list of ad creatives to play.
    """
    deadline = time.time() + 0.2   # 200ms hard deadline

    # 1. Fetch user profile (Redis, ~1ms)
    user_profile = await asyncio.wait_for(
        get_user_profile(opp.user_id), timeout=0.05
    )

    # 2. Match eligible campaigns (pre-indexed in Redis)
    eligible = await match_campaigns(
        user_profile, opp.content_ctx, opp.device,
        max_duration=opp.break_dur
    )

    # 3. Run auction (Vickrey: 2nd price)
    winning_ads = run_vickrey_auction(eligible, break_dur=opp.break_dur)

    # 4. Verify creative is pre-transcoded and available
    available = [ad for ad in winning_ads
                 if await check_creative_ready(ad["creative_id"], opp.device)]

    # 5. Record impression (async, non-blocking)
    asyncio.create_task(record_ad_decision(opp, available))

    return available

def run_vickrey_auction(candidates: list[dict], break_dur: int) -> list[dict]:
    """
    2nd-price auction: winner pays 2nd highest bid.
    Fill break with highest-value non-conflicting ads.
    """
    candidates.sort(key=lambda x: x["bid_cpm"], reverse=True)
    selected, remaining_time = [], break_dur
    for ad in candidates:
        if ad["duration"] <= remaining_time:
            selected.append(ad)
            remaining_time -= ad["duration"]
        if remaining_time <= 0: break
    return selected

Frequency Capping

Problem: Without frequency capping, the same viewer sees the same ad 50 times in one session → ad fatigue → brand damage. Solution: Redis per-user impression counters with hourly/daily expiry: HINCRBY user:{id}:ad_freq {ad_id} 1. Check before selecting: if HGET user:{id}:ad_freq {ad_id} > cap → exclude from auction.

Geo-Blocking — Rights Management

Sports broadcasting rights are sold by territory. The Super Bowl may be licensed to NBCUniversal in the US, Sky in the UK, and Canal+ in France. Users outside licensed territory must be blocked.

1

3
Block or serve alternate content
If blocked: 403 with "Not available in your region" message, or redirect to blackout feed (e.g., local affiliate). Never serve blank screen — explain why.

DRM (Digital Rights Management)

What DRM Does

Encrypts video segments. Each player needs a license (decryption key) from a DRM license server. License issued only to authenticated, authorized users. Even if someone intercepts the encrypted segments, they can't play them without the key.

Multi-DRM

Different devices use different DRM systems: Widevine (Android, Chrome), FairPlay (Apple), PlayReady (Microsoft). Must support all three. DRM-as-a-service providers (Irdeto, EZDRM) handle license servers for all three from one API.

CDN Token Auth

CDN URLs are signed with a time-limited token. Without a valid token, the CDN returns 403. Prevents unauthorized users from accessing stream URLs. Token includes: user_id, content_id, expiry, signature. Renewed every 15 minutes.

GDPR & Privacy Consent

Consent flow for live streaming: Before showing targeted ads or processing viewing data for analytics: (1) Show consent banner on first visit. (2) If user consents: track viewing, serve targeted ads. (3) If user declines: serve non-targeted ads, no cross-session tracking. (4) Store consent decision in cookie + server-side DB. (5) Respect consent regardless of CDN edge node serving the viewer.
High Availability Architecture

Failure Points in Live Streaming

Encoder failure

Primary encoder dies mid-game. Solution: always run two encoders simultaneously pushing to ingest. Automatic failover from encoder A to B. Viewer sees a 3–5 second freeze, then stream resumes from backup.

Ingest server failure

Encoder pushes RTMP to ingest. Ingest crashes. Solution: push to two geographically separate ingest points. Redundant ingest servers at each point. If one ingest fails, encoder switches to backup within 30 seconds.

Transcode cluster failure

GPU cluster fails mid-stream. Solution: two independent transcode clusters process the same ingest. Primary feeds CDN. If primary fails, CDN switches to backup transcode output. Segment timestamps keep playback seamless.

CDN PoP failure

A CDN point-of-presence goes down. Solution: multi-CDN — Akamai + Cloudflare + AWS CloudFront simultaneously. DNS-based steering routes viewers away from failed CDN within 30–60 seconds (DNS TTL).

Origin failure

CDN origin (where segments are stored) fails. Solution: CDN edge nodes have 2–4 hours of segment cache. Viewers continue watching from cache while origin recovers. Grace period buys time for repair.

Network partition

AWS region goes down. Solution: redundant ingest and transcode in two regions (US-East + US-West). Traffic automatically shifts to healthy region via Anycast routing or Global Accelerator.

Segment Health Monitoring

stream_monitor.py — detect and failover on segment gapsPython
import asyncio, time

class StreamHealthMonitor:
    """
    Monitor segment generation rate. Switch to backup if primary stalls.
    Expected: 1 segment every 2 seconds (2s HLS segments).
    Trigger failover if: no new segment for > 4s (2 missed segments).
    """
    def __init__(self, segment_interval=2.0, failover_threshold=4.0):
        self.interval  = segment_interval
        self.threshold = failover_threshold
        self.last_seg  = time.time()
        self.active    = "primary"

    async def on_segment_received(self, origin: str, seg_num: int):
        self.last_seg = time.time()
        if origin != self.active:
            return   # ignore backup segments while primary is healthy

    async def watch(self):
        """Background loop: trigger failover if segments stop arriving."""
        while True:
            await asyncio.sleep(0.5)
            staleness = time.time() - self.last_seg
            if staleness > self.threshold:
                await self.failover()

    async def failover(self):
        if self.active == "primary":
            print("FAILOVER: primary → backup")
            self.active = "backup"
            # Update CDN origin to point to backup transcode output
            await update_cdn_origin("backup-origin.streaming.com")
            # Alert on-call team
            await page_oncall("LIVE STREAM FAILOVER: primary transcode down")
            self.last_seg = time.time()   # reset timer for backup monitoring
