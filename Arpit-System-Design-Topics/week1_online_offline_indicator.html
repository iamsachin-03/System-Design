Problem Statement

Imagine you're building a chat app. Before a user starts a conversation, they want to know who's online. The micro-question: Given a user ID, is that user online or offline right now?

Real-world Examples

WhatsApp

Shows "online" when active; "last seen X ago" when not. Leniently marks offline after inactivity. Privacy settings can hide last seen.

Discord

Online / Idle / DND / Offline with user override. Presence updates fan out to all mutual server members on change.

Slack

Workspace-scoped; auto-Away after 30 min. Green dot in sidebar. Integrates with calendar for auto-DND during meetings.

Why It's Hard at Scale

Write Amplification

5M users × 1 heartbeat / 10 s = 500,000 writes/sec. A direct DB write per heartbeat kills any relational database instantly.

Read Fan-out

Checking a 200-friend list = 200 status reads. 1M users doing this = 200M reads/s. Must batch with Redis MGET — one round-trip for all 200.

Offline Detection

Users don't always send explicit "logout." Network drops, crashes, app backgrounding — the system must infer offline from absence of signal.

Ghost Users & Flapping

TTL too long = ghost users online after crash. TTL too short = user appears offline on every network hiccup. Must tune the sweet spot.

Assignment scope: Design the full presence system. Show a user list grouped with online users first. Display "last seen X mins ago" for offline users. Status must update within 10 seconds of a user coming online.
Requirements

Structured across three levels — core product constraints, high-level system properties, and low-level micro guarantees.

Core Requirements

  • Update online status within ≤ 10 seconds of user coming online
  • Can be lenient marking offline (30–60 s acceptable)
  • Scale for 5 million concurrent active users
  • Accurate status, eventually consistent
  • Support batch query — list of user IDs at once
  • Show "last seen X ago" for offline users

High-Level Requirements

  • All components operate with high availability
  • Data must be durable — last_seen survives restarts
  • Define behavior during scale-up / scale-down
  • System must be cost-effective with justification
  • Capacity planning drives architecture decisions
  • Other services (notifications, chat) can query presence via API

Micro Requirements

  • Data must never go inconsistent
  • System must be free of deadlocks
  • Throughput must not be choked by locking
  • Heartbeat from client; silence = offline (TTL-based)
  • Explicit disconnect triggers immediate offline

Derivation

Write Rate

5,000,000 × (1 / 10s) = 500,000 writes/s to Redis. Each write: SET user:<id>:ts <epoch> EX 30 ≈ 40 bytes. Redis single node cap ≈ 1M ops/s. 3 shards → ~167K/s each. Comfortable headroom.

Memory

5M keys × 40 bytes = 200 MB. Fits on even a small Redis node (r6g.large = 16 GB). We shard for HA and write throughput, not memory pressure.

Read Load

Friend list = 200 users → MGET 200 keys = 1 round-trip. 1M users × 1 MGET / 5s → 200K MGET ops/s. Served from read replicas. Each MGET ≈ 200 key lookups.

DB Flush Load

Batch UPSERT every 60 s. Worst case: 5M rows / 60 s = ~83K rows/s in 10K-row chunks = ~8 queries/s. Trivial for MySQL. Kafka buffers prevent data loss.

Architecture

Stateless heartbeat servers write to Redis with TTL. Kafka async-flushes last_seen to MySQL. Status queries use MGET on read replicas.

Step-by-Step Flow

1
Client sends heartbeat every 10 s
POST /heartbeat {user_id:42} — signals "I'm alive." Sent on foreground activity and on app open.
2
Server writes to Redis with TTL=30
SET user:42:ts 1711180000000 EX 30 — key auto-expires if no refresh. No explicit logout needed.
3
Event published to Kafka (async)
{user_id, timestamp} → presence.heartbeats topic. Fire-and-forget; doesn't block the HTTP response.
4
Status query uses Redis MGET
MGET user:A:ts user:B:ts … user:Z:ts — one round-trip for 200 friends. Key exists → online; missing (TTL expired) → offline.
5
Flush Worker batch-writes last_seen to MySQL
Kafka consumer aggregates events, keeps max ts per user, bulk-UPSERTs every 60 s. MySQL stores durable last_seen for "X ago" display.
Design Document

Critical decisions: mental models for presence, connection pooling, caching issues at scale, async processing rationale, and chokepoints.

3 Mental Models for Presence Detection

Model 1 — Pull Polling

Client polls GET /status?id=X every few seconds. 5M users × 200 friends × poll/10s = 100M RPS just for reads.

❌ Not viable at scale

Model 2 — WebSocket Push

Server pushes status changes. Fan-out: User A goes online → push to all A's friends. 1K friends × 5M events = 5B pushes/s at peak. Needs stateful servers.

⚠️ Good for chat events, not polling

Model 3 — Heartbeat + TTL ✓

Client heartbeats → Redis key with TTL. Key missing = offline. Status via MGET. Stateless servers. No fan-out on write. Scales linearly.

✅ Chosen — industry standard

Connection Pool & DB Proxy Decisions

Redis — Client-side Sharding

Each heartbeat server uses a Redis Cluster client that routes keys internally using consistent hashing on user_id. No proxy needed; avoids a SPOF. Adding a shard → rehash ~⅓ of keys; use MIGRATE for zero-downtime resize.

MySQL — ProxySQL

Flush worker has 1000 app connections but MySQL accepts ~500 real ones. ProxySQL multiplexes N→M connections. Short batch-UPSERT transactions make connection reuse very efficient. Prevents connection exhaustion.

Caching Issues at Scale

IssueWhere It HitsMitigation
TTL too short (= heartbeat interval)Any missed beat → user goes offline; terrible mobile UXTTL = 3× heartbeat interval (30 s)
TTL too long (5 min)Ghost users — appear online 5 min after crashExplicit DEL on app background event
Redis restart stampedeAll 5M keys expire together → cache miss stormRedis AOF + RDB; Sentinel/Cluster failover
App-layer cache stalenessStatus Query caches MGET results 5 s → slight stalenessAcceptable — eventual consistency is requirement
Hot key (celebrity)Everyone queries Justin Bieber's status → hot shardRead replicas + short app-layer cache for VIP users

Async Processing — Why Kafka

Without Kafka: Heartbeat API → direct MySQL at 500K writes/s → InnoDB max ~100K/s → lock contention + timeouts → system collapses.

With Kafka: Heartbeat API → publish event in microseconds (fire-and-forget) → Kafka buffers → Flush Consumer batch-writes ~83K rows/s in chunks of 10K → decoupled, durable, replayable on crash.

Chokepoints

Short Term — Redis Write

500K writes/s per node hits Redis ceiling (~1M ops/s). Mitigated with 3 shards → 167K/s each. Monitor shard latency; auto-scale shards when p99 > 5 ms.

Long Term — Read Fan-out

1B users × 200-friend MGET = massive read amplification. Solution: Redis read replicas + 5 s in-memory cache on Status Query API servers. Short TTL keeps staleness acceptable.

heartbeat_service.pyPython
import time
from fastapi import FastAPI
import redis.asyncio as aioredis
from pydantic import BaseModel
from typing import List

app = FastAPI()
# Redis Cluster — client-side routing, no proxy needed
redis_client = aioredis.RedisCluster(host="redis-cluster", port=6379, decode_responses=True)

TTL  = 30            # 3× heartbeat interval — tolerates 2 missed beats
KEY  = "user:{}:ts"   # value = epoch ms of last heartbeat

class HBReq(BaseModel): user_id: int
class SReq(BaseModel):  user_ids: List[int]

@app.post("/heartbeat")
async def heartbeat(r: HBReq):
    """Client calls every 10 s. Refreshes TTL."""
    await redis_client.set(KEY.format(r.user_id), int(time.time()*1000), ex=TTL)
    return {"ok": True}

@app.post("/offline")
async def mark_offline(r: HBReq):
    """Explicit disconnect: app goes to background, tab closes."""
    await redis_client.delete(KEY.format(r.user_id))
    return {"ok": True}

@app.post("/status")
async def get_status(r: SReq):
    """MGET — one Redis round-trip for entire friend list."""
    keys   = [KEY.format(uid) for uid in r.user_ids]
    values = await redis_client.mget(keys)   # single pipeline call
    return {
        uid: {"online": ts is not None, "last_seen": int(ts) if ts else None}
        for uid, ts in zip(r.user_ids, values)
    }
flush_worker.py — Kafka consumer → MySQL batch UPSERTPython
import asyncio, time, json
from aiokafka import AIOKafkaConsumer
import aiomysql

async def flush_worker():
    consumer = AIOKafkaConsumer("presence.heartbeats", bootstrap_servers="kafka:9092",
        group_id="presence-flush", value_deserializer=lambda v: json.loads(v))
    await consumer.start()
    pool = await aiomysql.create_pool(host="mysql", db="chat", user="writer", password="***", maxsize=20)

    buf: dict[int,int] = {}   # user_id → max(last_seen ms)
    t0 = time.time()

    async for msg in consumer:
        uid, ts = msg.value["user_id"], msg.value["timestamp"]
        buf[uid] = max(buf.get(uid,0), ts)       # keep max ts per user
        if len(buf) >= 10_000 or time.time()-t0 >= 60:
            await _upsert(pool, buf); buf.clear(); t0=time.time()

async def _upsert(pool, buf):
    rows = [(uid,ts) for uid,ts in buf.items()]
    sql  = "INSERT INTO users(id,last_seen) VALUES(%s,%s) ON DUPLICATE KEY UPDATE last_seen=VALUES(last_seen)"
    async with pool.acquire() as conn:
        async with conn.cursor() as cur:
            await cur.executemany(sql, rows); await conn.commit()
HeartbeatController.javaJava
@RestController @RequestMapping("/api")
public class HeartbeatController {
    private static final int TTL = 30;
    private static final String K = "user:%d:ts";
    private final StringRedisTemplate redis;
    public HeartbeatController(StringRedisTemplate r) { this.redis=r; }

    @PostMapping("/heartbeat")
    public Map<String,Boolean> heartbeat(@RequestBody HBReq req) {
        redis.opsForValue().set(String.format(K, req.getUserId()),
            String.valueOf(Instant.now().toEpochMilli()), TTL, TimeUnit.SECONDS);
        return Map.of("ok",true);
    }

    @PostMapping("/offline")
    public Map<String,Boolean> offline(@RequestBody HBReq req) {
        redis.delete(String.format(K, req.getUserId()));
        return Map.of("ok",true);
    }

    @PostMapping("/status")
    public Map<Long,UserStatus> status(@RequestBody StatusReq req) {
        List<String> keys = req.getUserIds().stream()
            .map(id->String.format(K,id)).toList();
        List<String> vals = redis.opsForValue().multiGet(keys); // MGET
        Map<Long,UserStatus> result = new LinkedHashMap<>();
        List<Long> ids = req.getUserIds();
        for(int i=0;i<ids.size();i++){
            String v=vals.get(i);
            result.put(ids.get(i), v!=null?new UserStatus(true,Long.parseLong(v))
                                             :new UserStatus(false,null));
        }
        return result;
    }
}
PresenceFlushConsumer.javaJava
@Service
public class PresenceFlushConsumer {
    private final JdbcTemplate jdbc;
    private final ConcurrentHashMap<Long,Long> buf = new ConcurrentHashMap<>();
    private volatile long lastFlush = System.currentTimeMillis();

    @KafkaListener(topics="presence.heartbeats", groupId="presence-flush")
    public synchronized void consume(HeartbeatEvent e) {
        buf.merge(e.getUserId(), e.getTimestamp(), Math::max);
        if(buf.size()>=10_000 || System.currentTimeMillis()-lastFlush>=60_000L) flush();
    }

    private void flush() {
        List<Object[]> rows = buf.entrySet().stream()
            .map(e->new Object[]{e.getKey(),e.getValue()}).toList();
        jdbc.batchUpdate(
            "INSERT INTO users(id,last_seen) VALUES(?,?) ON DUPLICATE KEY UPDATE last_seen=VALUES(last_seen)", rows);
        buf.clear(); lastFlush=System.currentTimeMillis();
    }
}

Heartbeat + Redis TTL

✅ Pros

  • Automatic offline via TTL — handles crashes, drops, backgrounding without explicit logout
  • Lock-free — Redis SET is atomic; zero deadlock risk
  • O(1) per write; MGET for batch reads in one round-trip
  • Horizontally scalable via Redis Cluster; shard by user_id % N
  • Stateless API servers — scale up/down freely, no sticky sessions
  • Self-healing — crashed clients auto-expire without cleanup jobs

❌ Cons

  • Flaky networks cause flapping — user offline for ~30 s on brief disconnect
  • 500K writes/s to Redis requires proper sharding and monitoring
  • Redis is in-memory — needs RDB/AOF + Kafka for durability
  • No push on change — clients must poll /status to see updates
  • TTL misconfiguration has product consequences (ghosts or flapping)
  • Adds Kafka + flush worker infra for durability

Consistency vs Availability

Failure ScenarioSystem BehaviourAcceptable?
Redis primary crashReplica promoted in <30 s; brief stale reads possible✅ with replicas
Network partition (client)User appears offline after TTL expires (~30 s)✅ lenient is requirement
Heartbeat server crashIn-flight heartbeats lost; users offline after 30 s⚠️ use multi-AZ + LB
MySQL downRedis still serves presence; Kafka buffers last_seen events✅ graceful degradation
Kafka downlast_seen stops updating; Redis presence still works✅ presence unaffected

TTL Sweet Spot

TTL = 10 s

= heartbeat interval. One missed beat → offline. Terrible on mobile. Avoid.

TTL = 30 s ✓

= 3× interval. 2 missed beats forgiven. Goes offline 30 s after last heartbeat. Chosen.

TTL = 5 min

Very tolerant but creates ghost users. Crash → appear online 5 min. Avoid for chat presence.

⚠️ Keep in Mind

  • I/O in Node.js is asynchronous — never block the event loop in heartbeat handlers
  • Do NOT broadcast status changes to all users — fan-out only to the user's friend list. Broadcasting to 5M = catastrophic.
  • Trigger heartbeat on visibilitychange events on mobile, not just setInterval
  • Use exponential backoff if heartbeat fails — don't hammer the server on disconnect

    Outcome

    By completing this design you build concrete, transferable mental models — all of which recur throughout the remaining 7 weeks.

    • Presence detection patterns — heartbeat + TTL is the industry standard (WhatsApp, Slack, Discord). You now understand the why, not just the what.
    • Redis as first-class storage — not just a cache, but the primary store for high-throughput ephemeral state. TTL as a feature, not a workaround.
    • Connection pooling decisions — Redis client-side routing vs. proxy; why MySQL needs ProxySQL at 500+ concurrent app connections.
    • Decoupling writes with Kafka — async pipeline lets 500K writes/s land in Redis without touching MySQL. Fire-and-forget + durability via Kafka retention.
    • Lock-free concurrency — Redis SET is atomic. Systems designed so locking is structurally impossible eliminate entire bug classes.
    • Capacity planning as design driver — 500K writes/s → shard Redis; 200 MB footprint → no memory pressure. Numbers precede architecture, always.
    • Eventual consistency in practice — 30 s offline latency is a deliberate product call with real UX tradeoffs, not an engineering limitation.
    • MGET fan-out on read — collapsing 200 Redis GETs into 1 round-trip. The N+1 query pattern at scale.
    • Schema design for "last seen" — store epoch ms (not DATETIME), index on last_seen, async write avoids hot-row contention on UPDATE.
    • SocketIO broadcast trap — fanning out status changes to ALL users = O(5M) writes per event. Always scope push to friend lists.
