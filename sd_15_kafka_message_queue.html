✅ Core Requirements
CORE

Consume: Consumer groups read messages. Each consumer in a group reads exclusive partitions. Offsets track progress per group.

⚡ Why Kafka is Fast — 4 Design Decisions
The append-only log is Kafka's superpower. All writes are sequential appends. Sequential SSD writes: ~2 GB/s. Random writes: ~150 MB/s. This single design decision makes Kafka 10-15x faster than traditional queues using B-trees or hash tables.

1. Sequential I/O (Append-Only Log)

All writes are sequential appends to end of log file. Never update or delete in-place. Sequential disk I/O at ~2 GB/s vs ~150 MB/s random. Traditional queues (RabbitMQ) use B-trees: O(log N) random I/O per message.

2. OS Page Cache

Kafka writes directly to OS page cache (RAM), trusts the OS to flush to disk. Recent messages served entirely from page cache at RAM speed. No JVM heap = no GC pauses on message data.

3. Zero-Copy (sendfile syscall)

Traditional: disk → kernel buffer → user space → socket buffer → NIC (4 copies). Kafka: disk → kernel buffer → NIC (2 copies, zero user-space copies). Linux sendfile() syscall. 60% CPU reduction for reads.

4. Batching & Compression

Producer batches messages (linger.ms=5) and compresses with snappy/lz4/zstd. One network round-trip per batch instead of per message. 3-10x compression on JSON payloads. Broker decompresses only for consumers that need it.

👥 Consumer Groups — Parallelism Model
Key rule: partitions ≥ consumers per group for max parallelism. Each partition assigned to exactly one consumer in a group. Multiple groups read the same topic at different offsets independently.

Rule: consumers in group ≤ partitions (extra consumers sit idle)
3 partitions + 3 consumers = 1 partition/consumer → max parallelism
3 partitions + 4 consumers = 1 consumer idle → wasted resources
3 partitions + 2 consumers = 1 consumer reads 2 partitions → uneven but fine
To scale: increase partition count FIRST, then add consumers

🔐 ISR Replication & acks
ISR (In-Sync Replicas)
1
Producer sends to partition leader (only the leader accepts writes)
2
Leader appends to log, forwards to ISR followers
3
Followers fetch & append (async pipeline)
4
Once all ISR replicas appended: message "committed" — High Watermark advances
5
Consumers only read up to High Watermark (never see uncommitted messages)
6
On leader failure: controller picks first ISR member as new leader
acks Setting Tradeoffs
acks=0: No ACK. Fire & forget. Fastest, data loss.
acks=1: ACK from leader only. Fast. Loss if leader crashes before replication.
acks=-1/all: ACK from all ISR. Safe. Use for critical data.

Safety setting: RF=3 + acks=all + min.insync.replicas=2
If ISR drops to 1 (1 broker down), writes are rejected (NotEnoughReplicasException) rather than silently losing durability guarantee.

2

4

Producer calls beginTransaction()
2
Write to output topic + commit input offset atomically in one transaction

Consumer uses isolation.level=read_committed
4
Crash: transaction rolled back, no partial state

Phase 1 — Why Kafka is Fast
IR
INTERVIEWER
Kafka handles 1 million messages/sec on commodity hardware. How?
ME
YOU
Four design decisions each eliminate a traditional bottleneck:

1. Sequential I/O (append-only log): All writes are sequential appends. Sequential SSD writes achieve ~2 GB/s vs ~150 MB/s for random I/O. Kafka never updates or deletes messages in-place. 10-15x better write throughput vs traditional queues using B-trees.

2. OS page cache: Kafka writes directly to the OS page cache and trusts the OS to buffer it. Recent messages served from RAM without touching disk. No JVM heap for message data — no GC pressure.

3. Zero-copy (sendfile): Traditional path: disk → kernel buffer → user space → socket buffer → NIC (4 copies). Kafka uses Linux sendfile(): disk → kernel buffer → NIC (2 copies, skips user space entirely). 60% CPU reduction for consumer reads.

4. Batching & compression: Producer batches with linger.ms=5, compresses with snappy/lz4. One network round-trip per batch instead of per message. 3-10x compression on JSON. Combined effect: 1 write operation sends thousands of messages.
💡 "Sequential I/O + zero-copy sendfile" is the canonical answer. Name both and explain why. Many candidates say "Kafka is fast" but can't explain the mechanism. This is what separates you.
Phase 2 — Ordering Guarantees
IR
INTERVIEWER
"Kafka guarantees message ordering." Is that true? Under what conditions?
ME
YOU
Partially true — Kafka guarantees ordering within a partition, not globally.

Within a single partition: messages have totally ordered offsets. If A is written before B to partition 0, any consumer always reads A before B. Guaranteed.

Across partitions: no ordering. Message 1 to partition 0 (offset 100), message 2 to partition 1 (offset 50). Consumer A reads P0, consumer B reads P1. No global ordering.

Practical solution: Route all events for a specific entity to the same partition via partition key: producer.send("events", userId, message). All events for user X go to the same partition → fully ordered per user.

Critical gotcha: Increasing partition count after messages are routed breaks key-based ordering — the same key may hash to a different partition. Partition count is a one-way door. Size it correctly upfront.
IR
INTERVIEWER
With RF=3 and acks=all, what happens if ISR drops to 1?
ME
YOU
With acks=all, the producer waits for all in-sync replicas (ISR) to ACK. If two brokers are down, ISR shrinks to just the leader (size=1). acks=all now only requires the leader — you've silently lost durability.

Fix: min.insync.replicas=2
Refuse writes if ISR < 2. If ISR drops to 1, producers get NotEnoughReplicasException — explicit failure rather than silent data loss. Trade: availability decreases but durability holds.

Equation: RF ≥ min.insync.replicas + 1. With RF=3 and min.insync.replicas=2: tolerate 1 broker failure without data loss or blocking writes.
Phase 3 — At-Least-Once vs Exactly-Once
IR
INTERVIEWER
Consumer processes a payment message, crashes, restarts. It re-reads the same message. How do you handle this without charging the user twice?
ME
YOU
This is the at-least-once duplicate processing problem. Two solutions:

Option 1 — Idempotent consumer (preferred):
Make processing safe to run twice. For payments: use the Kafka message offset or a payment_id as an idempotency key. Before charging, check: "has this payment_id been processed?" Use INSERT ... ON CONFLICT DO NOTHING or check-then-act. If the operation is idempotent, running it twice produces the same result — the duplicate is harmless.

Option 2 — Kafka Transactions (exactly-once):
Consumer reads payment message, processes, writes result to output topic AND commits offset in one atomic transaction. beginTransaction() → send to output → sendOffsetsToTransaction() → commitTransaction(). If crash: both the output write and offset commit are rolled back. On retry: idempotent producer ID prevents duplicate output.

When to use each: Idempotent consumer is simpler, faster (zero overhead), and works for 95% of cases. Use Kafka transactions only when you CANNOT make the downstream operation idempotent (e.g., third-party API that doesn't support idempotency keys).
IR
INTERVIEWER
How many partitions should a new topic have?
ME
YOU
Formula: max(throughput_needed / throughput_per_partition, max_consumers)

Throughput-based: A single partition handles ~10-50 MB/s. If target is 1 GB/s: 1,000 MB ÷ 20 MB = 50 partitions minimum.

Parallelism-based: Partition count is the ceiling on consumer concurrency. If you ever want 30 consumers processing in parallel, need at least 30 partitions.

Practical rule: 10-100 partitions covers most use cases. Over-provisioning is safer — you can add consumers up to partition count without changing topic config. You can't reduce partition count. Increasing partitions breaks key-based ordering for existing data.

Operational cost of more partitions: Each partition = 1 file handle per broker replica. 100 partitions × RF=3 = 300 file handles per broker. More partitions = slower leader election on broker failure.
Phase 4 — Kafka vs RabbitMQ vs SQS
IR
INTERVIEWER
When would you choose RabbitMQ over Kafka?
ME
YOU
Choose Kafka when: high throughput (1M+/sec), replay needed (multiple consumer groups at different offsets), event sourcing / audit log, stream processing, fan-out (same event to analytics AND notifications AND ML independently).

Choose RabbitMQ when: complex routing rules (fanout exchanges, topic exchanges with wildcards), per-message TTL or priority queues, traditional task queue (produce → consume → delete), lower throughput (<100K/sec) where operational simplicity matters, AMQP protocol compatibility.

The key distinction: Kafka retains messages and lets consumers replay from any point. RabbitMQ deletes messages after consumption. Kafka is a distributed log that happens to work like a queue. RabbitMQ is a pure message broker.

Choose SQS when: AWS-native, zero operational burden, don't need replay, <256KB messages. Standard SQS: unordered, nearly unlimited throughput. FIFO SQS: 3000 msg/sec, ordering per message group. Cheapest option when you don't need Kafka's features.

🏗 Architecture
📰 Partition Log Internal Structure
Each partition is a directory on disk. Inside: a series of segment files (.log + .index). The sparse index enables O(log N) offset lookups without scanning the full log.
// On-disk layout for partition: /kafka-logs/orders-0/
//
// 00000000000000000000.log    segment 0 (offsets 0..999999)
// 00000000000000000000.index  sparse index for segment 0
// 00000000000001000000.log    segment 1 (offsets 1M..1.999M)
// 00000000000001000000.index
// 00000000000002000000.log    active segment (append here)
// 00000000000002000000.index
//
// MESSAGE RECORD FORMAT (per message in .log):
// offset(8) + timestamp(8) + key_len(4) + key(N) + val_len(4) + val(M) + crc(4)
// Overhead: ~29 bytes per message
//
// INDEX FORMAT (sparse, every ~4KB of log data):
// relative_offset(4) + file_position(4)
// e.g., offset 50000 is at byte 204800 in this segment's .log file
//
// LOOKUP for offset 50000:
// 1. Binary search segment files by base offset -> find segment 0
// 2. Binary search .index -> find byte_pos = 204800
// 3. Scan .log from byte 204800 until offset 50000 found (small linear scan)
// Total: O(log segments) + O(log index_entries) + O(tiny scan)

// 1. Producer Config
@Configuration
public class KafkaProducerConfig {
    @Bean
    public ProducerFactory<String, Object> producerFactory() {
        Map<String, Object> cfg = new HashMap<>();
        cfg.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka1:9092,kafka2:9092");
        cfg.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,   StringSerializer.class);
        cfg.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, JsonSerializer.class);
        cfg.put(ProducerConfig.ACKS_CONFIG,                   "all");    // wait all ISR
        cfg.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG,     true);     // dedup on retry
        cfg.put(ProducerConfig.LINGER_MS_CONFIG,              5);        // batch 5ms
        cfg.put(ProducerConfig.BATCH_SIZE_CONFIG,             65536);    // 64 KB batches
        cfg.put(ProducerConfig.COMPRESSION_TYPE_CONFIG,       "snappy"); // 5x on JSON
        cfg.put(ProducerConfig.RETRIES_CONFIG,                Integer.MAX_VALUE);
        return new DefaultKafkaProducerFactory<>(cfg);
    }
}

// 2. Producer Service — route by key for per-entity ordering
@Service
public class OrderEventProducer {
    @Autowired private KafkaTemplate<String, OrderEvent> kafka;

    public void publish(OrderEvent event) {
        // KEY = orderId → same order always same partition → ordered
        kafka.send("order-events", event.getOrderId(), event)
             .whenComplete((res, ex) -> {
                 if (ex != null) {
                     log.error("Publish failed: {}", ex.getMessage());
                 } else {
                     RecordMetadata m = res.getRecordMetadata();
                     log.debug("Delivered P{} offset {}", m.partition(), m.offset());
                 }
             });
    }
}

// 3. Consumer — manual offset commit (at-least-once safety)
@Service
public class OrderConsumer {
    @KafkaListener(topics = "order-events", groupId = "notification-svc", concurrency = "3")
    public void consume(@Payload List<OrderEvent> events, Acknowledgment ack) {
        try {
            for (OrderEvent e : events) {
                processIdempotently(e); // upsert by messageId — safe to retry
            }
            ack.acknowledge(); // commit offset AFTER success
        } catch (Exception ex) {
            log.error("Batch failed, will retry: {}", ex.getMessage());
            throw ex; // don't ack → Kafka redelivers → at-least-once
        }
    }

    private void processIdempotently(OrderEvent e) {
        // INSERT ... ON CONFLICT DO UPDATE — same result if run twice
        notifRepo.upsertByOrderId(e.getOrderId(), e.getStatus());
    }
}

// 4. Exactly-Once: consume + produce atomically
@Service
public class ExactlyOnceProcessor {
    @Autowired private KafkaTemplate<String, Object> txTemplate;

    @Transactional("kafkaTransactionManager")
    public void process(OrderEvent order, Acknowledgment ack) {
        Invoice invoice = generateInvoice(order);
        txTemplate.send("invoices", order.getOrderId(), invoice); // within txn
        ack.acknowledge(); // offset commit within same txn
        // If crash: BOTH produce AND offset commit roll back
        // On retry: idempotent producer ID prevents duplicate invoice
    }
}

from confluent_kafka import Producer, Consumer, KafkaError, KafkaException
import json, logging, signal

logger = logging.getLogger(__name__)

# 1. Producer
class KafkaProducer:
    def __init__(self, bootstrap_servers):
        self._p = Producer({
            'bootstrap.servers': bootstrap_servers,
            'acks': 'all',
            'enable.idempotence': True,
            'linger.ms': 5,
            'batch.size': 65536,
            'compression.type': 'snappy',
            'retries': 2147483647,
        })

    def send(self, topic, key, value):
        def on_delivery(err, msg):
            if err:
                logger.error("Delivery failed: %s", err)
            else:
                logger.debug("Delivered P%d @ %d", msg.partition(), msg.offset())

        self._p.produce(
            topic=topic,
            key=key.encode(),
            value=json.dumps(value).encode(),
            callback=on_delivery
        )
        self._p.poll(0)

    def flush(self):
        self._p.flush(30)

# 2. Consumer — manual commit (at-least-once)
class KafkaConsumer:
    def __init__(self, bootstrap_servers, group_id, topics):
        self._c = Consumer({
            'bootstrap.servers':  bootstrap_servers,
            'group.id':           group_id,
            'auto.offset.reset':  'earliest',
            'enable.auto.commit': False,          # manual = safe
            'isolation.level':    'read_committed', # only see committed txns
        })
        self._c.subscribe(topics)
        self.running = True

    def run(self):
        try:
            while self.running:
                msg = self._c.poll(1.0)
                if msg is None: continue
                if msg.error():
                    if msg.error().code() != KafkaError._PARTITION_EOF:
                        raise KafkaException(msg.error())
                    continue
                try:
                    key   = msg.key().decode() if msg.key() else None
                    value = json.loads(msg.value())
                    self._process(key, value)
                    # Commit ONLY after success — at-least-once guarantee
                    self._c.commit(message=msg, asynchronous=False)
                except Exception as e:
                    logger.error("Processing failed P%d@%d: %s",
                                 msg.partition(), msg.offset(), e)
                    # Don't commit: message will be redelivered on restart
                    self._send_to_dlq(msg)
                    self._c.commit(message=msg, asynchronous=False)
        finally:
            self._c.close()

    def _process(self, key, value):
        logger.info("Processing key=%s: %s", key, value)

    def _send_to_dlq(self, msg):
        # Publish to dead letter topic for manual inspection
        logger.warning("Sending to DLQ: %s", msg.topic())

# 3. Usage
if __name__ == "__main__":
    BOOTSTRAP = "kafka1:9092,kafka2:9092,kafka3:9092"

    producer = KafkaProducer(BOOTSTRAP)
    producer.send(
        topic = "order-events",
        key   = "order-12345",   # same key = same partition = ordered
        value = {"order_id": "12345", "status": "PAID", "amount": 99.99}
    )
    producer.flush()

    consumer = KafkaConsumer(BOOTSTRAP, "notification-svc", ["order-events"])
    signal.signal(signal.SIGTERM, lambda s, f: setattr(consumer, 'running', False))
    consumer.run()
⚖️ Design Tradeoffs
Kafka tradeoffs are nuanced. The right acks, partition count, and delivery semantics depend on whether you can tolerate data loss, duplicate processing, and added latency.
Producer acks: 0 vs 1 vs all
✅ acks=all (critical)

Payments, inventory, user events. Wait for all ISR replicas. Zero data loss. Latency: +5-20ms (replication). Must pair with min.insync.replicas=2. Production standard for anything you can't lose.

⚠ acks=1 (analytics)

Metrics, logs, click events. ACK from leader only. Fast. Risk: data loss if leader crashes before followers replicate (~ms window). 3-5x higher throughput than acks=all. Fine when some loss is tolerable.

❌ acks=0

Fire and forget. UDP-like. Maximum throughput. No retry on failure. Rarely justified — acks=1 with linger.ms batching achieves near-identical throughput with much better reliability.

Delivery Semantics: At-Most-Once vs At-Least-Once vs Exactly-Once
✅ At-Least-Once + Idempotent

Commit offset AFTER processing. If crash = redeliver. Make consumer idempotent: upsert by message ID. Simple, fast, zero overhead. Works for 95% of use cases. Start here always.

⚠ Exactly-Once (Transactions)

Atomic: produce + commit offset in one transaction. 10-30% throughput reduction. Complex to configure. Use when: downstream operation cannot be made idempotent and duplicate processing causes real money/data problems.

❌ At-Most-Once

Commit offset BEFORE processing. Crash after commit = message lost forever. Use only for metrics where losing samples is acceptable. Very rare — if you can tolerate loss, why use Kafka at all?

Partition Count: Too Few vs Just Right vs Too Many
✅ 10-100 (most use cases)

10-100 partitions covers most workloads. Each partition = 1 consumer thread max. 100 partitions x RF=3 = 300 file handles per broker. Safe range. Over-provision: start with 30 partitions even if you only need 5 initially.

❌ Too few (1-3)

Limits parallelism. Can't scale consumer count beyond partition count. Repartitioning breaks key-based ordering for existing data. Most common Kafka antipattern: creating topics with default 1 partition "for simplicity."

⚠ Too many (10,000+)

Seen at LinkedIn/Netflix. Each partition adds file handles, memory, replication overhead. 10K partitions x RF=3 = 30K file handles per broker. Slower leader election on failure. ZooKeeper-based Kafka choked; KRaft handles it better but overhead is real.

Kafka vs RabbitMQ vs SQS
✅ Kafka

Log-based, replay, 1M+/sec, multiple independent consumer groups, stream processing. Event sourcing, microservice event buses, analytics pipelines. High operational complexity. Choose when throughput and replay matter.

⚠ RabbitMQ

Flexible routing (fanout/topic/headers exchanges), per-message TTL and priority, delete-after-consume. ~50-100K msgs/sec. Lower complexity. Choose for complex routing logic, task queues, request-reply patterns.

📄 AWS SQS

Fully managed, zero ops, 14-day retention, 256KB max. Standard: unordered, unlimited scale. FIFO: 3000/sec, ordered per group. Choose for AWS-native apps that don't need Kafka's replay or throughput.
