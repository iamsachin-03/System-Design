🎙 Phase 1 — System Overview & Hard Problems
PHASE 1 · SCOPE
IR
INTERVIEWER
Design YouTube. Walk me through how you'd approach this.
ME
YOU
I'd split this into three distinct sub-problems because they have completely different characteristics:

1. Video Upload & Transcoding Pipeline — Write-heavy, CPU/GPU intensive, async. 500 hours uploaded per minute. The hard part is converting raw video into multiple streamable formats efficiently.

2. Video Streaming Delivery — Extremely read-heavy. 1B+ views/day. Latency-critical. The hard part is delivering large video files globally with <200ms start time and no buffering.

3. Metadata & Interactions — View counts, likes, comments, subscriptions. The hard part is accuracy at scale — a viral video can get 10M likes in an hour.

The most interesting problems are the transcoding pipeline and adaptive bitrate streaming — I'll focus there. The interactions problem we solved for Instagram.

Key observations:
  • This is massively read-heavy — upload rate vs view rate is like 1:100,000
  • The same video is served millions of times — perfect CDN cache efficiency
  • Video files are enormous — a 1-hour 1080p video = ~4GB. We can't serve this from application servers
  • Different users on different networks need different quality levels — adaptive bitrate is essential
💡 Structure your answer early. Splitting into sub-problems and identifying the hardest ones immediately shows senior-level thinking. Don't try to design everything at once.
📤 Phase 2 — Video Upload & Transcoding Pipeline
PHASE 2 · UPLOAD + TRANSCODE
IR
INTERVIEWER
Walk me through the upload flow. A creator uploads a 4K, 2GB video file.
ME
YOU
Upload has two phases — the file transfer itself, then the transcoding pipeline:

Phase 1 — File Transfer (Chunked Resumable Upload):
  1. Client hits POST /api/v1/videos with metadata (title, description, category). Server creates a video record (status: UPLOADING), returns video_id and a resumable upload session URL
  2. Client splits the 2GB file into chunks of 5–10MB and uploads each chunk via PUT to the session URL, with chunk index and total chunk count
  3. Server assembles chunks in S3 using S3 Multipart Upload — each chunk is stored independently, assembled at the end. Chunks are stored out-of-order and reassembled by index
  4. If network drops mid-upload: client queries GET /upload-session/{id}/progress, gets back which chunks are received, resumes from missing chunks. Nothing is re-uploaded unnecessarily.
  5. When all chunks received: S3 CompleteMultipartUpload, video status → PROCESSING

Phase 2 — Transcoding Pipeline (Async):
  1. S3 upload completion triggers an S3 Event → Kafka message (topic: video-uploaded)
  2. Transcoding Orchestrator consumes the event, creates separate transcoding jobs for each target resolution
  3. Jobs distributed to a fleet of Transcoding Workers (GPU-equipped instances) via a job queue
  4. Each worker transcodes the raw video to one resolution: 360p, 480p, 720p, 1080p, 2160p (4K)
  5. Also generates: 10 thumbnail candidates (1 per 10% of video), chapter markers, WebVTT captions
  6. Each completed rendition uploaded to S3 → CDN origin
  7. When lowest resolution (360p) is ready: video status → PUBLISHED — video becomes watchable immediately, higher resolutions become available as they complete
IR
INTERVIEWER
Why chunk the upload instead of streaming the file directly?
ME
YOU
Three key reasons:
  • Resumability: A 2GB upload on a mobile connection can take 30+ minutes. If it fails at 95%, without chunking the user re-uploads everything. With chunks, we resume from the last successful chunk.
  • Parallelism: Chunks can be uploaded in parallel (5–10 concurrent chunks), maximising bandwidth utilisation
  • Memory efficiency: Server never needs to hold the entire 2GB in memory — each chunk is streamed directly to S3 and immediately freed
S3 Multipart Upload was specifically designed for this pattern — it's the industry standard.
IR
INTERVIEWER
The transcoding is slow. How do you parallelize it?
ME
YOU
Two levels of parallelism:

Across resolutions: All 5 resolution jobs (360p, 480p, 720p, 1080p, 4K) start simultaneously on different workers — not sequentially. A 1-hour video transcoded to 5 resolutions in parallel takes ~15 minutes, not 75 minutes.

Within a single resolution (segment parallelism): Split the raw video into 2-minute segments and transcode each segment independently on different workers. A 2-hour video has 60 segments → 60 workers can transcode in parallel → complete in ~2 minutes instead of 120 minutes. Then stitch segments together.

Technology: FFmpeg on GPU instances (NVIDIA Tesla V100 or A100). GPU encoding (h.264 NVENC, h.265 HEVC) is 20–50× faster than CPU encoding. AWS Elemental MediaConvert or Google Transcoder API are managed alternatives.
▶️ Phase 3 — Adaptive Bitrate Streaming (HLS/DASH)
PHASE 3 · STREAMING PROTOCOL
IR
INTERVIEWER
How does the video actually stream? What protocol do you use and why?
ME
YOU
I'd use HLS (HTTP Live Streaming) — originally by Apple but now the industry standard for on-demand video (used by YouTube, Netflix, Twitch).

How HLS works:
  1. The transcoded video for each resolution is split into short segments (2–10 seconds each, stored as .ts or .fmp4 files)
  2. A Master Playlist (.m3u8 file) lists all available quality levels with their stream bandwidth and resolution
  3. Each quality level has a Media Playlist (.m3u8) listing all its segment URLs in order
  4. The video player downloads the Master Playlist, reads available qualities, picks the starting quality
  5. Player downloads 2–3 segments ahead (buffering), then plays from buffer
  6. Player measures download speed continuously → if speed drops, switches to lower quality for next segment → seamless quality adaptation

Why HLS over DASH?
HLS has better compatibility — it's natively supported on iOS/Safari (Apple devices) where DASH is not. For universal support, use HLS. DASH is technically more flexible but requires additional player code for Apple devices. YouTube actually uses DASH for Android/Desktop but HLS for iOS.
IR
INTERVIEWER
How does a segment request flow from the user's player to the video data?
ME
YOU
The segment delivery path is carefully optimised:
  1. Player makes HTTP GET request: https://cdn.youtube.com/{video_id}/720p/segment_042.ts
  2. Request hits the nearest CDN edge node (via Anycast routing) — could be <10ms away
  3. CDN cache hit (expected ~95% for popular videos): CDN serves the segment directly from its cache. Player receives data in <20ms.
  4. CDN cache miss: Edge forwards to CDN origin → S3 bucket → streams back. S3 returns the segment, CDN caches it for future requests from the same region.
The first viewer in a region experiences the cache-miss path. Every subsequent viewer in that region gets the cached version. For a viral video, 99.9%+ of views are served from CDN edge — S3 and application servers are barely touched.
IR
INTERVIEWER
What happens when a user seeks to 01:23:45 in a 3-hour video?
ME
YOU
This is where HLS segments shine. At 10-second segments, the user seeks to timestamp 5,025 seconds → that's segment #503 (5025 ÷ 10 = 502.5 → segment 503).

The player directly requests segment_503.ts via HTTP range request — no need to download segments 1–502. The URL is deterministic from the playlist. CDN serves segment 503 instantly if cached.

This is fundamentally different from how traditional media files work (where you'd need byte-range requests deep into a single huge file). HLS segments make seeking O(1) — just compute segment number, fetch that segment URL.
👁 Phase 4 — View Counts & Interactions at Scale
PHASE 4 · COUNTERS + INTERACTIONS
IR
INTERVIEWER
A video goes viral — 10 million views in 5 minutes. How do you count views accurately without killing your database?
ME
YOU
10M views in 5 minutes = 33,333 view increments per second on a single row. Direct DB writes would create a lock contention disaster.

Solution — Multi-tier counter aggregation:
  1. Client-side batching: Video player doesn't emit a view event immediately on play. It waits until the user has watched 30+ seconds (prevents accidental counts), then sends a view event to the View Service.
  2. Redis atomic counter: View Service does INCR view_count:{video_id} in Redis. Redis handles ~1M INCR/sec single-threaded without contention. In-memory, atomic, ultra-fast.
  3. Periodic flush: Every 30 seconds, a flush job reads the Redis counter delta and writes it to the persistent DB (Cassandra or MySQL): UPDATE videos SET view_count = view_count + {delta}
  4. Read from Redis: When displaying view count, read from Redis (real-time) not from DB. Redis value is always ahead of DB by at most 30 seconds — acceptable.
Anti-gaming: Track view events per (user_id/IP, video_id) in a Redis set with TTL. Duplicate views within 24 hours are deduplicated. This prevents view count inflation from repeat visits or bot attacks.
IR
INTERVIEWER
What about the recommendation system — how does it work at a high level?
ME
YOU
At a high level, YouTube's recommendation system is a two-stage pipeline:

Stage 1 — Candidate Generation: For a given user, quickly narrow 800M+ videos down to ~hundreds of candidates. Uses collaborative filtering (users who watched X also watched Y), content-based filtering (similar tags, categories, creators), and user's watch history. Fast, approximate — runs on a simplified model.

Stage 2 — Ranking: Takes ~hundreds of candidates, applies a deeper ML model (Deep Neural Network) that scores each candidate on predicted watch time, engagement likelihood, freshness, diversity. Returns top 20 ranked results.

For the system design, the key data store is a User Watch History service (Cassandra, partitioned by user_id) and a Video Embeddings service (vector database like Faiss or Pinecone for approximate nearest-neighbor search). A separate offline training pipeline (Spark) trains the models on interaction data from Kafka.
💾 Phase 5 — Storage Strategy & Cost Optimisation
PHASE 5 · STORAGE + COST
IR
INTERVIEWER
Storage must be enormous. How do you manage it cost-effectively?
ME
YOU
YouTube stores roughly 1 exabyte (1,000 PB) of video data. Storage cost optimisation is critical.

Tiered storage strategy:
  • Hot tier (S3 Standard): Videos uploaded in the last 30 days + any video with >1,000 views/day. Fast retrieval, higher cost. ~5% of all videos but ~80% of views.
  • Warm tier (S3 Standard-IA): Videos 30–180 days old, moderate view rate. Lower storage cost, small retrieval fee.
  • Cold tier (S3 Glacier): Videos >6 months old with very low view rate. Very cheap storage (~$0.004/GB/month). Retrieval takes minutes — acceptable for rarely-watched content.

Video deduplication: Many users upload identical videos (clips, re-uploads). MD5/perceptual hash of video content → if hash exists, don't store again. Save the storage reference instead. YouTube saved enormous space this way.

Codec efficiency: Encode older content with newer codecs (AV1, H.265) as they become mainstream. AV1 achieves ~50% better compression than H.264 at same quality — halving storage for re-encoded content.

    private static final String BUCKET = "yt-videos-raw";

    /** Phase 1: Create metadata + S3 multipart upload session */
    public UploadSessionResponse initiateUpload(InitiateUploadRequest req) {
        long videoId = snowflake.nextId();
        String s3Key = "raw/" + videoId + "/original";

        // Create S3 Multipart Upload
        CreateMultipartUploadResponse multipart = s3.createMultipartUpload(
            CreateMultipartUploadRequest.builder()
                .bucket(BUCKET).key(s3Key).build());

        int totalChunks = (int) Math.ceil(
            (double) req.getFileSizeBytes() / CHUNK_SIZE);

        // Persist video + session
        videoRepo.save(Video.builder()
            .videoId(videoId).channelId(req.getChannelId())
            .title(req.getTitle()).status("UPLOADING").build());

        sessionRepo.save(UploadSession.builder()
            .sessionId(UUID.randomUUID().toString())
            .videoId(videoId).totalChunks(totalChunks)
            .s3UploadId(multipart.uploadId())
            .expiresAt(Instant.now().plus(24, ChronoUnit.HOURS))
            .build());

        return new UploadSessionResponse(videoId, totalChunks, CHUNK_SIZE);
    }

    /** Phase 2: Upload a single chunk */
    public void uploadChunk(String sessionId, int chunkIdx,
                               byte[] data) {
        UploadSession session = sessionRepo.findById(sessionId)
            .orElseThrow(() -> new SessionNotFoundException(sessionId));

        // S3 part numbers are 1-indexed
        UploadPartResponse part = s3.uploadPart(
            UploadPartRequest.builder()
                .bucket(BUCKET)
                .key("raw/" + session.getVideoId() + "/original")
                .uploadId(session.getS3UploadId())
                .partNumber(chunkIdx + 1) // S3 parts are 1-indexed
                .build(),
            RequestBody.fromBytes(data));

        // Record this chunk as received
        session.addReceivedChunk(chunkIdx, part.eTag());
        sessionRepo.save(session);

        // All chunks received? Complete the multipart upload
        if (session.isComplete()) {
            completeUpload(session);
        }
    }

    private void completeUpload(UploadSession session) {
        List<CompletedPart> parts = session.getParts().stream()
            .map(p -> CompletedPart.builder()
                .partNumber(p.getIndex() + 1)
                .eTag(p.getEtag()).build())
            .collect(Collectors.toList());

        s3.completeMultipartUpload(CompleteMultipartUploadRequest.builder()
            .bucket(BUCKET)
            .key("raw/" + session.getVideoId() + "/original")
            .uploadId(session.getS3UploadId())
            .multipartUpload(CompletedMultipartUpload.builder()
                .parts(parts).build())
            .build());

        // Update video status + trigger pipeline
        videoRepo.updateStatus(session.getVideoId(), "PROCESSING");
        kafka.send("video-uploaded", new VideoEvent(session.getVideoId()));
    }
}

// ── 2. Transcoding Orchestrator ───────────────────────
@Service
public class TranscodingOrchestrator {

    private static final int[] RESOLUTIONS = {360, 480, 720, 1080, 2160};
    private static final int    SEGMENT_SEC  = 120; // 2-minute segments

    @Autowired private TranscodeJobQueue jobQueue;
    @Autowired private VideoRepository  videoRepo;
    @Autowired private S3Client         s3;

    @KafkaListener(topics = "video-uploaded")
    public void onVideoUploaded(VideoEvent event) {
        long videoId = event.getVideoId();
        Video video = videoRepo.findById(videoId);

        // Probe video duration to calculate segments
        int durationSec = probeDuration(video.getRawS3Key());
        int numSegments = (int) Math.ceil(
            (double) durationSec / SEGMENT_SEC);

        // Submit jobs: numSegments × numResolutions in parallel
        List<TranscodeJob> jobs = new ArrayList<>();
        for (int seg = 0; seg < numSegments; seg++) {
            for (int res : RESOLUTIONS) {
                jobs.add(TranscodeJob.builder()
                    .videoId(videoId)
                    .segmentIndex(seg)
                    .startSec(seg * SEGMENT_SEC)
                    .durationSec(Math.min(SEGMENT_SEC,
                                  durationSec - seg * SEGMENT_SEC))
                    .resolution(res)
                    .inputKey(video.getRawS3Key())
                    .outputKey(outputKey(videoId, res, seg))
                    .build());
            }
        }
        // Dispatch all jobs to worker queue
        jobQueue.submitAll(jobs);
    }

    /** Called by workers when a segment is done */
    public void onSegmentComplete(TranscodeJob job) {
        boolean resolutionComplete = jobQueue
            .markDone(job.getVideoId(), job.getResolution(), job.getSegmentIndex());

        if (resolutionComplete) {
            stitchSegmentsAndGeneratePlaylist(job.getVideoId(), job.getResolution());

            // 360p done? Publish video immediately
            if (job.getResolution() == 360) {
                videoRepo.updateStatus(job.getVideoId(), "PUBLISHED");
            }
        }
    }

    private void stitchSegmentsAndGeneratePlaylist(long videoId, int res) {
        // Generate .m3u8 media playlist listing all segment URLs
        StringBuilder playlist = new StringBuilder();
        playlist.append("#EXTM3U\n#EXT-X-VERSION:3\n");
        playlist.append("#EXT-X-TARGETDURATION:").append(SEGMENT_SEC).append("\n");

        int segCount = videoRepo.getSegmentCount(videoId);
        for (int i = 0; i < segCount; i++) {
            playlist.append("#EXTINF:").append(SEGMENT_SEC).append(",\n");
            playlist.append(cdnUrl(videoId, res, i)).append("\n");
        }
        playlist.append("#EXT-X-ENDLIST\n");

        // Upload playlist to S3
        String playlistKey = "processed/" + videoId + "/" + res + "p/playlist.m3u8";
        s3.putObject(PROCESSED_BUCKET, playlistKey,
                     playlist.toString().getBytes());

        // If all resolutions done, generate master playlist
        if (videoRepo.allResolutionsDone(videoId)) {
            generateMasterPlaylist(videoId);
        }
    }

    private void generateMasterPlaylist(long videoId) {
        String master =
            "#EXTM3U\n" +
            "#EXT-X-STREAM-INF:BANDWIDTH=500000,RESOLUTION=640x360\n" +
            cdnBase(videoId) + "/360p/playlist.m3u8\n" +
            "#EXT-X-STREAM-INF:BANDWIDTH=2500000,RESOLUTION=1280x720\n" +
            cdnBase(videoId) + "/720p/playlist.m3u8\n" +
            "#EXT-X-STREAM-INF:BANDWIDTH=5000000,RESOLUTION=1920x1080\n" +
            cdnBase(videoId) + "/1080p/playlist.m3u8\n";
        s3.putObject(PROCESSED_BUCKET,
            "processed/" + videoId + "/master.m3u8",
            master.getBytes());
    }
}

// ── 3. View Count Service (sharded Redis) ─────────────
@Service
public class ViewCountService {

    @Autowired private RedisTemplate<String,String> redis;
    private static final int SHARDS = 10;

    /**
     * Atomic view count increment using sharded counters.
     * Prevents single-key hot spot for viral videos.
     */
    public long incrementView(long videoId, long userId) {
        String today = LocalDate.now().toString();

        // Deduplication: one view per user per video per day
        String dedupKey = "views_dedup:" + videoId + ":" + today;
        Boolean isNewView = redis.opsForSet()
            .add(dedupKey, String.valueOf(userId)) == 1;
        redis.expire(dedupKey, Duration.ofHours(25)); // TTL slightly over 24h

        if (!isNewView) return getViewCount(videoId); // already counted

        // Shard by user ID to distribute hot key load
        int shard = (int) (userId % SHARDS);
        String countKey = "view_count:" + videoId + ":shard_" + shard;
        redis.opsForValue().increment(countKey);

        return getViewCount(videoId);
    }

    public long getViewCount(long videoId) {
        // Sum across all shards
        return LongStream.range(0, SHARDS)
            .mapToObj(s -> redis.opsForValue()
                .get("view_count:" + videoId + ":shard_" + s))
            .filter(Objects::nonNull)
            .mapToLong(Long::parseLong)
            .sum();
    }

    /** Scheduled: flush Redis counts to MySQL every 30 seconds */
    @Scheduled(fixedDelay = 30_000)
    public void flushToDatabase() {
        // Scan for all view_count keys, aggregate, update MySQL
        redis.keys("view_count:*:shard_0").forEach(key -> {
            long videoId = extractVideoId(key);
            long total = getViewCount(videoId);
            videoRepo.setViewCount(videoId, total); // SET not increment
        });
    }
}

# ── 1. Video Upload Service ───────────────────────────
class VideoUploadService:
    """
    Handles chunked resumable video upload to S3.
    Uses S3 Multipart Upload for reliability and parallelism.
    """
    CHUNK_SIZE   = 5 * 1024 * 1024  # 5 MB
    RAW_BUCKET   = "yt-videos-raw"

    def __init__(self, s3_client, video_repo, session_repo, kafka):
        self.s3       = s3_client
        self.videos   = video_repo
        self.sessions = session_repo
        self.kafka    = kafka

    def initiate_upload(self, channel_id: int, title: str,
                          file_size_bytes: int, description: str = "") -> dict:
        """Phase 1: Create video record + S3 multipart session"""
        video_id  = self._snowflake_id()
        s3_key    = f"raw/{video_id}/original"
        total_chunks = math.ceil(file_size_bytes / self.CHUNK_SIZE)

        # Start S3 Multipart Upload
        response = self.s3.create_multipart_upload(
            Bucket=self.RAW_BUCKET, Key=s3_key)
        s3_upload_id = response["UploadId"]

        # Persist video record
        self.videos.save({
            "video_id": video_id, "channel_id": channel_id,
            "title": title, "description": description,
            "status": "UPLOADING", "created_at": time.time()
        })

        session_id = str(uuid.uuid4())
        self.sessions.save({
            "session_id": session_id, "video_id": video_id,
            "s3_upload_id": s3_upload_id,
            "total_chunks": total_chunks, "received": {},
            "expires_at": time.time() + 86400
        })
        return {"session_id": session_id, "video_id": video_id,
                "total_chunks": total_chunks, "chunk_size": self.CHUNK_SIZE}

    def upload_chunk(self, session_id: str, chunk_idx: int,
                       data: bytes, md5_b64: str):
        """Phase 2: Upload one chunk. Idempotent — safe to retry."""
        session = self.sessions.get(session_id)
        if not session or time.time() > session["expires_at"]:
            raise ValueError("Session not found or expired")

        # Upload to S3 (part numbers are 1-indexed)
        part_resp = self.s3.upload_part(
            Bucket=self.RAW_BUCKET,
            Key=f"raw/{session['video_id']}/original",
            UploadId=session["s3_upload_id"],
            PartNumber=chunk_idx + 1,
            Body=data,
            ContentMD5=md5_b64
        )
        session["received"][chunk_idx] = part_resp["ETag"]
        self.sessions.update(session_id, {"received": session["received"]})

        # All chunks received? Complete upload.
        if len(session["received"]) == session["total_chunks"]:
            self._complete_upload(session)

    def _complete_upload(self, session: dict):
        parts = [
            {"PartNumber": idx + 1, "ETag": etag}
            for idx, etag in sorted(session["received"].items())
        ]
        self.s3.complete_multipart_upload(
            Bucket=self.RAW_BUCKET,
            Key=f"raw/{session['video_id']}/original",
            UploadId=session["s3_upload_id"],
            MultipartUpload={"Parts": parts}
        )
        self.videos.update_status(session["video_id"], "PROCESSING")
        self.kafka.send("video-uploaded",
            key=str(session["video_id"]),
            value={"video_id": session["video_id"]})


# ── 2. Transcoding Orchestrator ───────────────────────
class TranscodingOrchestrator:
    RESOLUTIONS  = [360, 480, 720, 1080, 2160]
    SEGMENT_SEC  = 120  # 2-minute segments for parallel processing
    PROC_BUCKET  = "yt-videos-processed"

    def __init__(self, job_queue, video_repo, s3):
        self.jobs   = job_queue
        self.videos = video_repo
        self.s3     = s3

    def on_video_uploaded(self, video_id: int):
        """Kafka consumer handler — splits video and dispatches jobs"""
        video    = self.videos.get(video_id)
        duration = self._probe_duration(video["raw_s3_key"])
        num_segs = math.ceil(duration / self.SEGMENT_SEC)

        # Create numSegments × numResolutions jobs in parallel
        all_jobs = [
            {
                "video_id":     video_id,
                "segment_idx":  seg,
                "start_sec":    seg * self.SEGMENT_SEC,
                "duration_sec": min(self.SEGMENT_SEC, duration - seg * self.SEGMENT_SEC),
                "resolution":   res,
                "input_key":    video["raw_s3_key"],
                "output_key":   f"processed/{video_id}/{res}p/seg_{seg:04d}.ts"
            }
            for seg in range(num_segs)
            for res in self.RESOLUTIONS
        ]
        self.jobs.submit_batch(all_jobs)

    def on_segment_complete(self, video_id: int, resolution: int, segment_idx: int):
        """Called by GPU workers when a segment finishes transcoding"""
        self.jobs.mark_done(video_id, resolution, segment_idx)
        resolution_done = self.jobs.is_resolution_complete(video_id, resolution)

        if resolution_done:
            self._generate_media_playlist(video_id, resolution)
            if resolution == 360:
                # 360p is ready — make video public IMMEDIATELY
                self.videos.update_status(video_id, "PUBLISHED")
                # Notify subscribers asynchronously

        if self.jobs.all_resolutions_complete(video_id):
            self._generate_master_playlist(video_id)

    def _generate_media_playlist(self, video_id: int, res: int):
        """Build HLS media playlist (.m3u8) for one resolution"""
        num_segs = self.jobs.get_segment_count(video_id)
        lines = ["#EXTM3U", "#EXT-X-VERSION:3",
                 f"#EXT-X-TARGETDURATION:{self.SEGMENT_SEC}"]
        for i in range(num_segs):
            lines.append(f"#EXTINF:{self.SEGMENT_SEC},")
            lines.append(f"https://cdn.yt.com/processed/{video_id}/{res}p/seg_{i:04d}.ts")
        lines.append("#EXT-X-ENDLIST")

        playlist_content = "\n".join(lines)
        self.s3.put_object(
            Bucket=self.PROC_BUCKET,
            Key=f"processed/{video_id}/{res}p/playlist.m3u8",
            Body=playlist_content.encode(),
            ContentType="application/vnd.apple.mpegurl"
        )

    def _generate_master_playlist(self, video_id: int):
        """Build HLS master playlist listing all available quality levels"""
        bandwidth_map = {360: 500000, 480: 1000000,
                         720: 2500000, 1080: 5000000, 2160: 20000000}
        res_map = {360:"640x360", 480:"854x480", 720:"1280x720",
                   1080:"1920x1080", 2160:"3840x2160"}

        lines = ["#EXTM3U"]
        for res in self.RESOLUTIONS:
            lines.append(
                f"#EXT-X-STREAM-INF:BANDWIDTH={bandwidth_map[res]},RESOLUTION={res_map[res]}")
            lines.append(f"https://cdn.yt.com/processed/{video_id}/{res}p/playlist.m3u8")

        self.s3.put_object(
            Bucket=self.PROC_BUCKET,
            Key=f"processed/{video_id}/master.m3u8",
            Body="\n".join(lines).encode(),
            ContentType="application/vnd.apple.mpegurl"
        )
        self.videos.update_hls_manifest(video_id,
            f"https://cdn.yt.com/processed/{video_id}/master.m3u8")


# ── 3. View Count Service ─────────────────────────────
class ViewCountService:
    """
    Sharded Redis counters for view counts.
    Prevents single hot key during viral spikes.
    Periodic flush to MySQL for persistence.
    """
    SHARDS = 10

    def __init__(self, redis_client, video_repo):
        self.redis  = redis_client
        self.videos = video_repo

    def record_view(self, video_id: int, user_id: int,
                      watched_sec: int) -> int:
        """Record a view if user watched ≥30 seconds. Deduplicates per day."""
        if watched_sec < 30:
            return self.get_view_count(video_id)  # not a real view

        today = date.today().isoformat()
        dedup_key = f"views_dedup:{video_id}:{today}"

        # SADD returns 1 if new member (first view today), 0 if already seen
        is_new = self.redis.sadd(dedup_key, str(user_id)) == 1
        self.redis.expire(dedup_key, 90000)  # 25h TTL

        if not is_new:
            return self.get_view_count(video_id)

        # Shard by user_id to distribute hot-key load
        shard    = user_id % self.SHARDS
        cnt_key  = f"view_count:{video_id}:shard_{shard}"
        self.redis.incr(cnt_key)
        return self.get_view_count(video_id)

    def get_view_count(self, video_id: int) -> int:
        """Sum all 10 shards for total view count"""
        total = 0
        pipe = self.redis.pipeline()
        for shard in range(self.SHARDS):
            pipe.get(f"view_count:{video_id}:shard_{shard}")
        results = pipe.execute()
        return sum(int(v) if v else 0 for v in results)

    def flush_to_database(self):
        """Scheduled every 30s: persist Redis counts to MySQL"""
        cursor = self.redis.scan_iter("view_count:*:shard_0")
        for key in cursor:
            video_id = int(key.decode().split(":")[1])
            total    = self.get_view_count(video_id)
            if total > 0:
                self.videos.set_view_count(video_id, total)
⚖️ Design Tradeoffs & Decisions
Video systems have unique tradeoffs around storage cost, encoding efficiency, and streaming latency. Always tie your choices back to the scale numbers.
📡 Streaming Protocol: HLS vs DASH vs Progressive Download
✅ HLS (HTTP Live Streaming)

Native iOS/Safari support. CDN-friendly (standard HTTP GETs). Adaptive bitrate built-in. Segment-based seeking is O(1). Wide device support. De-facto standard.

⚠️ DASH (Dynamic Adaptive Streaming)

More flexible codec support (VP9, AV1). Better for DRM. But: no native iOS support — needs JavaScript player shim. YouTube uses DASH for Chrome/Android but HLS for iOS. More complex.

❌ Progressive Download

Server streams a single large file via HTTP Range. Simple but: no adaptive quality (stuck at one bitrate), seeking requires downloading to that byte position, huge CDN cache files. Only practical for very short videos.

⚙️ Transcoding Architecture: Dedicated Fleet vs Serverless vs Managed Service
✅ Dedicated GPU Fleet

Maximum control over encoding parameters. GPU instances (NVIDIA A100) encode H.264 at 100× real-time. Auto-scales via Kubernetes. At YouTube scale (500 hrs/min), managed services are cost-prohibitive.

⚠️ Managed (AWS Elemental / GCP Transcoder)

Zero ops overhead. Pay per minute of transcoded video. Great for startups/medium scale. But: limited customisation, expensive at massive scale ($0.017/min × YouTube volume = billions/month).

💡 Segment-level parallelism

The key innovation: splitting video into 2-min segments before transcoding. A 2-hour video becomes 60 independent jobs. 60 GPU workers process in ~2 minutes instead of 120. Critical for YouTube's upload-to-playable SLA of 3 minutes.

💾 Storage Tiering: Hot vs Warm vs Cold
✅ Tiered Storage (S3 ILM)

Intelligent Lifecycle Management: new/popular videos on SSD (fast, expensive). Old/rare videos on Glacier (slow, very cheap at $0.004/GB/month vs $0.023/GB for SSD). 80% of storage is cold tier → enormous cost savings at EB scale.

❌ All on Hot Storage

Simple but cost-prohibitive. 1 exabyte × $0.023/GB = $23 billion/month in S3 costs alone. Obviously impossible. Tiering is not optional — it's existential.

💡 Glacier retrieval latency

Cold tier retrieval takes minutes (Glacier Instant is seconds). For rarely-watched old videos, a few minutes of initial buffering is acceptable — the video wasn't going to start playing instantly anyway. Hot videos are already in CDN edge cache.

👁 View Counts: Redis Sharded Counter vs DB Row vs Kafka Event Stream
✅ Sharded Redis Counter

INCR is atomic and <1ms. Shard across 10 keys per video to handle hot videos (33K INCR/sec for viral → 3.3K per shard). Periodic flush to MySQL. Near-real-time accuracy.

❌ Direct DB Row Update

33K row locks/sec on one row = deadlock disaster. Even with optimistic locking, contention kills performance. SQL databases can't handle hot-row update patterns at this rate.

⚠️ Kafka Event Stream

Every view event to Kafka, consumer aggregates counts. Accurate, auditable, replayable. But: higher complexity and ~seconds latency to reflect counts. Good for offline analytics, not for real-time display. Can combine: Redis for display, Kafka for analytics.

📤 Upload Strategy: Chunked Resumable vs Direct Upload vs Presigned URL
✅ Chunked + S3 Multipart

Files split to 5MB chunks. Parallel upload (5–10 concurrent). Resumable: only missing chunks re-sent on failure. MD5 per chunk for integrity. Server never holds full file in memory. Industry standard for large files.

❌ Single POST through server

Whole 2GB file through app server. Server is memory and bandwidth bottleneck. No resume on failure — full re-upload. At 71 videos/sec × 200MB avg = 14.2 GB/s through app servers. Impossible.

💡 vs Simple Presigned URL

Simple presigned URL (like Instagram) works for small files (<100MB). For videos up to 256GB, S3 Multipart is required — S3 doesn't support single-PUT for files over 5GB. Multipart also enables progress tracking and resumability that single presigned PUT doesn't.

🎬 Codec Choice: H.264 vs H.265 vs AV1
✅ H.264 (Primary) + AV1 (Growing)

H.264: universal device support, hardware decode everywhere. AV1: 30–50% better compression than H.264 at same quality — huge storage and bandwidth savings at YouTube scale. YouTube now serves AV1 on supported devices.

⚠️ H.265 (HEVC)

50% better than H.264 but: patent licensing costs, inconsistent hardware decode support on older devices, iOS support only recent. Between H.264's universality and AV1's royalty-free efficiency, H.265 is the middle child nobody loves.

💡 The tradeoff

Encode once in H.264 (universal), serve AV1 to modern browsers/devices. Dual encoding costs ~2× transcoding compute but pays back in bandwidth savings. At 300 Tbps CDN egress, even 30% savings = 90 Tbps less bandwidth — saves millions per day.
